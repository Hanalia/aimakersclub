## *AI 다크 팩토리 구축: 스스로 코드를 작성하는 코드베이스, 라이브 실습* 핵심 요약

- **AI 다크 팩토리란 인간 개입 없이 AI가 전체 소프트웨어 개발 라이프사이클(기획, 구현, 리뷰, 배포, 릴리즈 등)을 자율적으로 관리하는 시스템으로, 이번 라이브에서는 이를 실시간으로 구축하는 과정을 보여줌**
- **개념적 배경은 1980~90년대 중국의 무인 '다크 팩토리(공장)'와 Dan Shapiro가 제안한 소프트웨어 다크 팩토리 논의에서 비롯, AI 코드 작성 진화 단계를 5레벨로 소개**
- **레벨 0~3까지는 인간의 직접적 개입·검증이 필수지만, 레벨 4~5(다크 팩토리 단계)로 갈수록 AI의 자율성과 책임이 증대, 인간은 최소한의 가이드(미션, 목표, 규칙)만 제공**
- **다크 팩토리 실현을 위해 오픈소스 AI 코드 하니스(harness) 빌더인 Archon을 중심기술로 선정, 워크플로우 설계(이슈 분류, 개발, 검증, 수정, 회귀테스트 등)를 집중적으로 개발**
- **관리 대상 코드베이스는 AI가 스스로 버그를 수정·기능을 추가하는 YouTube RAG(Chat) 서비스로 설정, 실험 후 실시간 공개 예정**
- **StrongDM 사례를 참고, 이슈 입력→Triage→분기(수락/거부/수정)→병렬 개발→검증→배포까지 전과정을 Archon 워크플로우로 표준화**
- **테스트용 대형 언어모델(LLM)은 비용·성능 최적화 이유로 Miniax M2.7을 채택(Anthropic Opus/Sonnet 등은 요금·Rate Limit 문제), 실험설정·환경변수 매핑 등 구체적 구현 노하우 제시**
- **워크플로우 설계의 핵심은 '고브넌스 레이어(미션, 규칙)' 파일을 통해 AI가 인식할 수 있는 범위와 한계, 행동/품질 기준, 보호 규정(예: 자동 무한루프 방지·Rate Limit·휴먼 개입구간 등)을 명확히 하는 것**
- **AI Self-verification 한계(자기검열/편향 등)를 극복하기 위해 StrongDM의 'Hold Out 패턴'을 도입, 구현 전후 역할 분리 및 철저한 회귀/블랙박스 테스트 구조 설계**
- **라이브 전반에 걸쳐 실시간 코드/워크플로우 구현, 문서화, 문제 발생 및 수정 과정(코드베이스 의존 패키지 구축, 라벨 관리, 처리 병목, 비용 제약 등)을 상세하게 시연함**

## 세부 요약 - 주제별 정리

### AI 코딩 어시스턴트 발전 5단계와 다크팩토리의 의미

- Dan Shapiro가 제안한 AI 코딩 5단계(Spicy Autocomplete→Coding Intern→Junior Developer→Self-driving Engineer→Dark Factory)를 자동차 자율주행 수준에 비유해 설명
- 레벨 0: Enhanced Stack Overflow, AI는 참조 도구일 뿐 개발자가 직접 코딩
- 레벨 1: AI에게 보일러플레이트 등 신뢰 높은 반복 업무 일부 대행(자동속도조절 수준)
- 레벨 2: 페어 프로그래밍 형태, 복잡성 일부 위임, 키 컨트롤은 여전히 인간
- 레벨 3: AI가 대다수 코드 생성, 인간이 전수 리뷰·결재(현실적 신뢰성·생산성 균형점)
- 레벨 4: 장기 무감독 작업도 AI가 수행, 인간 검증은 최종 시점에 개입, 엔지니어링팀 급 확장
- 레벨 5: 완전 자율(스티어링휠 없는 차량), 다크팩토리 구현. 인간은 미션·목표 제시와 일부 최상위 콘솔 역할만 수행, 코드작성·테스트·배포·버그수정 모두 AI 자체 주도

### 실제 다크팩토리 운용 사례(StrongDM 등)와 실현 상의 도전

- StrongDM 등 실제 기업이 프로덕션 코드베이스를 '다크팩토리' 기반으로 운용하는 사례 공유
- 코드는 비공개이나, PRD 및 운영 스펙/계획 문서 등은 일부 공개. 이로부터 실질적인 설계 인사이트 도출
- 인간 개입 없이 Pull Request가 머지되는 구조, 특별한 'Hold Out' 패턴(테스트 주체 분리) 활용
- 공개 다크팩토리 구축은 아직 실험단계로, 신뢰성·편향·무한루프·보안 위험 등 실질적 한계 존재

### Archon의 역할과 다크팩토리 전체 아키텍처 설계 전략

- Archon: 오픈소스 AI 코드 하니스 빌더. 사용자 정의 워크플로우로 전체 코딩 프로세스 패키징 가능
- 개발자의 소프트웨어 개발 패턴을 표준 워크플로우로 훈련시켜 코드베이스 관리에 적용
- 병렬 워크플로우·작업격리(Work tree)·정형입출력(Structured Output)·CLI 통합 등 높은 범용성 제공
- 클라우드의 orchestrator(크론잡)이 워크플로우 트리거, 핵심 흐름은 ①Triage ②Implement ③Validate/Review ④Fix ⑤Regression Test 등 반복
- 각 과정은 GitHub 이슈/라벨/PR에 ICP 적용으로 완전히 체계화

### 거버넌스 레이어/미션·룰 문서 설계와 그 중요성

- ‘미션(Mission).md’ : 애플리케이션 목적, 범위(In/Out of Scope), 절대불변 규칙(코드베이스 삶과 죽음을 결정)
- ‘Factory Rules.md’: 이슈 관리 기준, 자동화 범위, 승인/반려 조건, 풀 리퀘스트 품질조건, 자동테스트/검증/자동머지/보호파일 지정, Rate Limit·비용 통제 등 모든 운영 규칙 명시
- 모든 워크플로우가 항상 이들 문서를 불러 내부 의사결정에 사용(큰 편향 방지, 반복 가능한 표준화, 오작동 방지 목적)
- 예: "500라인 이상 PR 금지", "회귀테스트 시 사용자 흐름 자동화", "거버넌스 문서 자체는 AI가 절대 수정 불가" 등 명확하고 구체적 기준 제시

### GitHub 워크플로우(이슈–라벨–PR–테스트) 자동화 정책과 안전장치 설계

- 이슈가 입력점(누구나 제출 가능), Archon의 Triage 워크플로우가 라벨링(‘Accepted’, ‘Rejected’, ‘Needs Human’ 등) 및 전처리
- 일괄 작업 시 병렬 워크플로우로 분기, 각 이슈별 독립 작업공간 할당
- 라벨 및 상태 기반으로 명확한 논리 흐름 유지(오류 2회 누적 시 'Needs Human' 라벨링·자동 Escalation 등)
- 무한이슈생성·글로벌 Rate Limit·비용 폭증 방지 위한 일별 신용한도 등 이중 장치 내장
- 각 이슈/PR 템플릿 역시 AI가 정보 부족시 자동 반려하도록 강제, 일관성·최소 입력 보장

### LLM(대형 언어모델) 선택/교체 전략과 비용 절감 환경설정법

- Anthropic (Opus, Sonnet) 모델은 성능 좋지만 요금·토큰 한도 문제로 대규모자율자동화에 불리
- Opus, Sonnet 비용(1M input 토큰 기준 1달러 이상), Miniax M2.7(30센트/1M input)로 두드러진 효율
- 실험용으로 Miniax M2.7로 운용, 환경변수 라우팅(함수별 모델변경), provider switch 구조화
- OpenRouter/GLM/Olama 등 다양한 모델 호환성 및 교체법(엔드포인트 설정, 환경변수 매핑 등) 구체 소개
- 새 모델 도입·API 변경 등도 30초 내 빠른 스위칭 가능

### AI 코드 자동작성–검증–머지 워크플로우의 구체적 설계 및 구현 과정

- Archon 워크플로우 별도 설계(명령어 집합, 구조화된 아웃풋, 의사결정 기준, Bash 스크립트 등 단계별 상세 명세)
- Triage: 이슈를 batch 처리, 라벨 할당, 수락/거부/우선순위 분기, 구조화된 JSON 반환
- Implementation: 기본 Archon의 ‘fix GitHub issue’ 워크플로우를 프로젝트에 맞게 수정/확장, 개발환경 세팅, 분류, 리서치, 계획, 구현, 1차 리뷰/테스트, PR 생성
- Label/상태 반복적 적용, 'Needs Human' 예외처리, 라벨 지정 주기별 Rate Limit 자동 제어 등

### AI 자체검증, ‘Hold Out 패턴’ 및 AI 편향 최소화 구조 구체 구현

- ‘Hold Out’ 패턴 도입: 구현/검증 워크플로우 분리, 각기 다른 세션·컨텍스트·작업폴더 할당(자기 검열/자기검증 편향 해소)
- 검증과정은 ‘이슈 내용 기준만으로’ diff 판단, implementation 상세내용 비공개(사전 지식 의존 차단)
- 블랙박스형 회귀테스트, 오토메이션 브라우저 활용 전체사용자 시나리오 정기적 점검(회귀 이슈 발견 시 즉시 새 이슈 생성, 자동 루프)
- 보다 나은 결과를 위해 타모델(Gemini, CodeEx 등) 병합 검증·다중모델 등 평가체계 실험중

### 실제 실험–테스트 및 문제점 해결 (라이브 코딩 세션의 주요 흐름)

- 실제 Archon 워크플로우 작성, triage→implement→validate 흐름을 각 단계마다 직접 코딩/반복 수정
- 로컬 개발환경(VPS/로컬/클라우드) 동기화, package/dependency 문제 해결, Bash/JQ 인코딩 문제 등 실제 오류 발생→수정과정 상세 시연
- Trieage 워크플로우 smoke test(라벨링/상태별 검증), 문제시 직접 상태 롤백/복구
- 일반/복잡 이슈 병렬 처리, 필요시 휴먼 구제(human escape), 자동 레이트리밋에 따른 크레딧 소진 방지

### 자동화 전과정의 위험/한계와 강력한 품질보증 체계의 필요성 강조

- 완전 자율 다크팩토리는 현 시점에서 생산코드에 적용시 신뢰성, 보안, 편향(사이코팬시), 사소한 오류의 누적 등 근본 문제 내포
- 자동회귀 검증/강력 테스트, 안전딥러닝, 변경/진화 이력의 관리, 거버넌스의 기본 인프라화, 구조적 라벨링·모델 스위칭 등 여러 전략 제시
- 마치 끓는 물에 개구리 비유: 작은 오류가 꾸준히 누적, 인간개입 없는 경우 대형 장애로 종국에 폭발 위험
- ‘실험’임을 읍소, 결과 신뢰성 확보 위한 인적 병행 개입(최종 리뷰) 권고 및 오픈소스 전체 공개 예고

### 오픈소스, 커뮤니티 안내 및 실험 생태계 확장 방향

- Archon, 다크팩토리, 유튜브 챗플랫폼 등 관련 소스코드는 점진적 오픈소스화 방침(한 주~한 달 내 전체공개)
- Dynamis 커뮤니티, Second Brain 강의, 유저 피드백/아이디어 실시간 소통 및 향후 확장 아이디어(차기 워크숍, 실시간 코딩, 모델 교체 실험 등) 공유
- 실제 질문/답변 통해 로컬모델 운용, 커뮤니티 활용팁, 경험자/초보자 차별화 학습법 등 실용적 정보 다수 제공

### 실습 결과 총평 및 다음 단계 예고, 참고자료 공유

- 2+시간 라이브 동안 이론, 설계, 코드 전반 실시간 구현 및 반복적 문제해결 과정 등 심층 공개
- 완전 무인 AI 코딩 팩토리의 가능성과 한계, 핵심 설계 기준, 실제 자동화 인프라(VPS, 워크플로우, CI, 실패대응 등) 전체 구조 구현
- 추가 실험, 더 정제된 실전 워크플로우, 테스팅, 커뮤니티 확장, 관련 워크숍(e.g., AI 변환 워크숍) 등 다음 일정 예고
- 강의/실습/오픈소스 등 모든 내용은 단계별로 공개 예정임을 마지막에 재차 강조
