## *멜버른 2026 AI 엔지니어 키노트 라이브스트림 Day 1* 핵심 요약

- 이 영상은 2026년 6월 멜버른에서 열린 AI Engineer 컨퍼런스의 키노트 및 주요 발표 내용을 담고 있음
- AI의 미래는 단순 모델 경쟁을 넘어 서비스, 데이터, 브랜드, 하드웨어, 에이전트 등 전체 생태계의 효율과 전략적 선택에 달렸음을 강조함
- AI로 생성되는 코드의 비중이 급등, 2024년 4~5%에서 2026년 연말 최대 40~50%까지 증가할 전망임
- 클로드 오퍼스 4.8이 GPT-5.5보다 뛰어남을 비롯해, 주요 모델의 지능지표 수치와 각 모델의 선택 기준에 대해 상세히 소개함
- 오픈소스(open-weights) 모델은 상용(프로프라이어터리) 모델 대비 3~9개월 뒤처지지만, 선택 가능성과 가격 효율성에서 여전히 의미 있음
- 인퍼런스 비용은 6~18개월 주기로 급락하며, 최신 중간급 모델로 10~100배 저렴한 비용/효율을 달성하는 사례 다수
- 코드·업무·챗봇·프리젠테이션 등 실사용 에이전트(Agent)들이 빠르게 실생활에 적용, 제품별 최적화 전략과 모델 스위칭의 중요성이 커짐
- Notion 등 기업사례를 통해 대규모 모델 “락인(lock-in)” 회피, 멀티모델 운용, 오픈웨이트 활용이 현명한 제품 전략임을 다양한 데이터와 예시로 설명함
- 소프트웨어 개발이 점점 commoditized(상품화)되며, 기존 개발자 역할 변화, 조직구조 혁신, 직원 역량 및 자기개발이 핵심 생존전략으로 부상하고 있음
- 메모리 아키텍처와 장기적 에이전트 운영, AI 기반 대규모 워크로드 최적화 등 차세대 AI–SaaS 생태계 운영 노하우도 공유됨

## 세부 요약 – 주제별 정리

### AI는 모델 자체를 넘어 서비스, 데이터, 하드웨어, 에이전트 등 전체 생태계로 확장됨을 강조함

- OpenAI, AI21 등은 AI를 '모델'이 아닌, '하네스(harness, 적용 기술)', 서비스와 데이터, 브랜드, 제품 전체로 간주
- AI 서비스의 경쟁력은 모델 성능 외에도 실사용 경험, 에이전트, 하드웨어, API, 가격·속도 등 다방면에서 결정됨
- AI 엔지니어는 '모델 외부'의 다양한 요소(제품화, 브랜드, 사용자 경험, 데이터 운영 등)에 능동적으로 대응해야 한다고 제시

### AI로 생성되는 코드 비중 급등이 소프트웨어 시장 변혁을 예고함

- GitHub 클라우드 코드 커밋 기준: 2024년 2월 전체 코드의 4~5% AI작성 → 연말에 10%, 2026년 말에는 40~50% 예상
- AI 기반 코드 작성 도구 및 에이전트가 소프트웨어 개발에서 점점 핵심 역할을 차지
- 소프트웨어 개발 및 유지비의 극적 절감, 개발자 역량 변화와 채용 기준의 변화 등을 촉진

### AI 모델의 기술·지능 발전은 여전히 가속 중이며, 지역별·모델별 경쟁 구도가 심화됨

- 2023년~2026년 주요 AI 랩(Lab, 연구소)들의 모델 출시 시계열 차트로 해당 기간 AI 경쟁이 오히려 가속화됨을 데이터로 입증
- 미국·중국이 선도하고 있으나, 프랑스(Mistral), 한국(자주 AI 이니셔티브), UAE 등이 신규 경쟁자로 부상
- 호주 등 일부 국가는 첨단 AI 개발력에서 아직 경쟁이 어려운 실정임

### 오픈 웨이트(open-weights) 모델은 상용 모델 대비 3~9개월 뒤처지나, 가격·유연성 측면에서 실전 가치가 높음

- 오픈 웨이트(오픈소스) AI는 상용 모델보다 지능 수치(Artificial Analysis Intelligence Index)에서 다소 낮으나, 여전히 "Opus 4.5 ~ GPT-5.2" 수준을 실전에서 제공
- 오픈모델은 파인튜닝, 독립 배포, 비용 절감 등 강점을 지녀 여전히 선택지로 가치가 있음
- 최근 KimK 2.6, DeepSeek V4 Pro 등 오픈 AI들도 실사용에서 경쟁력 강화

### 전체 인텔리전스·인퍼런스 비용은 하락 추세이나, 기업의 실제 AI 비용은 오히려 늘어나는 역설적 상황이 발생함

- GPT-4급 인텔리전스에 6~18개월 주기로 10~100배 더 저렴하게 접근 가능
- 예시: 예전 Opus 4.5에 썼던 비용으로 최신 KimK 2.6이나 다른 모델에서 10배 더 저렴하게 동일 작업 수행 가능
- 그럼에도 기업들은 빠른 모델 업그레이드 및 더 복잡한 작업·에이전트 활용으로, 실제 지출(코드 요금제 업그레이드 등)이 오히려 늘어남
- 주요 원인: 더 복잡·큰 모델 수요, reasoning token 증가, 반복·다회전 에이전트 증가, 서비스 모델 다양화, 작업 난이도 증가

### 에이전트(Agent)는 다수 카테고리에서 실사용 등급에 도달했고, 실제 생산성에 직접적 영향을 미치고 있음

- 7대 카테고리: 코드 생성, 일반 업무, 챗봇, 프리젠테이션, OCR, 데이터 분석, 고객 지원 등에서 실제 시장 적합성과 급성장 중
- 코딩 에이전트는 모델 성능과 하네스(Harness/도구) 성능 모두 중요
- 클라우드 코드, 코덱스, Cursor CLI 등 다양한 상용·오픈소스 코드 에이전트 도구 실제 활용 사례 다수 제시

### AI 제품 전략에서는 '모델 락인(lock-in)' 대신 멀티모델 운용과 유연성이 중요하다는 점을 제품 사례로 보여줌

- Notion의 사례: 단일 벤더(Anthropic, OpenAI 등)에 락인되면 신모델 출시에 빠르게 대응 불가, 제품 혁신에 제약 발생
- 대형 기업은 분량 할인 등 협상력이 있으나, 대부분의 B2B·SMB는 협상력 부족 및 불공정한 시장 구조 노출
- 멀티모델(auto model), 모듈별 최적 모델 스위칭, 오픈 웨이트·강화학습·사용자 다이나믹 모델 고도화 등 전략적 포트폴리오 운용 필요

### 모델 및 인퍼런스 선정에서는 단일 API 콜 가격이 아닌, 전체 태스크 기준 생산성·품질·비용 균형이 핵심임

- 모델/웹서치 API 비용과 전체 태스크에서의 실 효과(예: 토큰 사용량, 처리속도, 정확도 등)의 연관성을 실제 Notion 데이터로 입증
- 평가팀(evaluation team)과 자체 벤치마크 활용, 잦은 모델 스위칭, 심지어 고객에게 모델 선택 옵션도 제공
- 품질, 속도, 비용 중 use case별 최적 조합을 우선시하며, 오토모델로 자동화·최적화

### 오픈 웨이트 모델과 강화학습(RL)은 중간·반복성 작업에서 실전 투입이 현실화됨

- 예: Notion은 Kim 2.6이 자체 태스크에서 GPT-5.2에 버금가는 효율성 및 품질을 보임
- 에러/리트라이도 실비용에 영향을 주는 점 강조: 토큰 요금은 에러에도 부과됨
- 단, 첨단·복합적 추론이 요구되는 작업에는 여전히 상용 최신 모델이 적합

### 코드·에이전트 개발 및 아키텍처 측면에서 '메모리 아키텍처', '장기 지속성', '집단적(협업) 에이전트'가 중요 과제로 부상함

- 대표적 사례: GitHub Copilot을 만든 Igor Costa의 AutoHand는 기억력 향상을 위한 아키텍처를 실전 적용
- 인간의 기억 구조(세맨틱, 프로시저럴, 에피소드, 리플렉티브 메모리)에서 해법을 착안하여, CLI/에이전트의 context와 memory를 분리 설계
- 장기 실행·매우 많은 동시 에이전트 운용·집단 협업 시 드리프트·충돌·컨센서스 문제 발생
- "Agent Spawn" 논문 등 각 모델/에이전트가 자식(span) 형태로 메모리 일부를 상속 후, 태스크별 차별화
- 오픈소스 CLI/커맨더 등으로 30여개의 코드 에이전트와 협업, 실제 사용자는 매일 15만명 규모

### 대규모 AI·음성에이전트 실서비스에서는 속도, 결정성, 스테이트머신과 같은 전통적 컴퓨팅 방식의 복귀가 필요함

- 인도 시장 등 대규모 콜센터용 AI 음성 에이전트 구축 시, 기존 Python SDK 방식은 레이턴시 한계로 한계
- Rust 등 초저지연 언어로 콜플로우를 설계, 텍스트→DAG/State Machine 프로세싱, regex 등 전통적 방법과 결합해 대량 트래픽 실시간 처리 성공
- 전체 시스템 설계에서 LLM이 필요하지 않은 부분에는 LLM 호출 대신 전통적 컴퓨팅 논리 활용해 비용·신뢰성 극대화

### AI로 인한 소프트웨어 개발 및 비즈니스 경제 모델이 “최저임금보다 저렴한 개발” 등으로 혁명적으로 변화하고 있으며 자기개발 역량이 생존의 핵심임

- AI 에이전트/자동화로 누구나 소프트웨어 개발 가능해짐 → 개발자 직군의 상품화(commoditization) 본격화
- 과거 “경력, IDE, 기술스택, 소속 아이덴티티”가 점차 무의미해지고, 진정한 차별화는 AI/Agent 내재화 경험과 시스템 구축 역량으로 이동
- 기존 조직 구조, 매니지먼트 계층, 작업 프로세스도 AI 도입의 “J커브”를 거쳐 재설계 필요
- 기업의 변혁 속도와 “최소 인원·최고 효율”로 무장한 신생 AI 조직에 대기업이 따라잡지 못하는 현상 심화
- 일선 개발자 및 조직 리더는 지금 ‘궁금증 테스트’를 통해 지속 자기투자, AI/Agent 구현 이해도 강화 시급
