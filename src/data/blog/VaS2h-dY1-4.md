## *Kubernetes에서 acpx와 ACP로 에이전트 확장하기* 핵심 요약

- 영상은 OpenClaw에서 오픈소스 에이전트 프레임워크를 Kubernetes 상에서 운영 및 확장하는 실무 경험을 공유하며, ACP(Agent Client Protocol)와 acpx 툴의 개발 및 적용 과정을 소개함
- 발표자는 ChatGPT 이전부터 AI 코딩 하니스 개발에 몰두해왔으며, 실제로 Discord, MS Teams 등의 플랫폼에 ClaudeBot, Codex 등의 에이전트를 통합하여 업무에 활용함
- OpenClaw에서는 하루 평균 300~500건의 PR이 생성되고, 6만여 PR이 누적되는 등 활발한 커뮤니티 협업 환경을 운영 중임
- ACP는 에이전트와 클라이언트 간의 상호작용을 표준화하는 프로토콜로, 중복을 줄이고 플러그인 개발 생산성을 높이기 위해 등장함. MCP와 다르게 “도구 제공”이 아니라 “상호작용 표준화”에 초점이 있음
- acpx는 ACP를 CLI에서 사용할 수 있도록 하여, 다양한 에이전트와의 워크플로우 자동화를 돕는 일종의 스위스아미 나이프식 도구임
- PR 리뷰, 리팩터, 버그 재현, 자동화된 리뷰 루프 등 에이전트 중심의 소프트웨어 개발 워크플로우를 추상화하고 실제 적용 예시를 시연함
- 에이전트 워크플로우는 개인용과 엔터프라이즈용으로 구분되며, 특히 엔터프라이즈 환경에서 “온디맨드 disposable(일회성) 에이전트”의 필요성과 가치를 강조함
- Kubernetes 기반 에이전트 오케스트레이션을 위해 오픈소스 orchestrator 및 다양한 인프라(react UI, helm chart, Codex pod 등)가 활용됨
- 실제 사용 예로, Slack 이슈 발생 시 자동으로 에이전트를 배치하여 버그를 진단·해결하거나, 별도의 UI에서 관리하는 데모를 소개함
- 영상 제작자는 OpenClaw와 Text Cortex 모두에서 오픈소스 에이전트 인프라의 발전과 적용을 지원하고 있으며, 관련 오픈소스 참여도 권장함

## 세부 요약 - 주제별 정리

### 발표자는 ClaudeBot 도입 이후 에이전트·워크플로우 자동화에 몰입하며 핵심 개발자가 됨

- 발표자는 ChatGPT 등 대형 언어모델 이전부터 AI 코딩 하니스를 개발해왔음(초기엔 Codex/DaVinci Code 사용)
- Discord에서 ClaudeBot 최초 배포 당시, 개인·회사 클러스터 양쪽에 바로 설치하고 놀라움을 느낌
- 클라우드 기반 협업 환경과 오픈소스 프로젝트의 확장성이 큰 잠재력을 가짐을 인식
- 초창기에는 PR을 자동으로 합치거나 슬랙·디스코드·MS Teams 등 여러 플랫폼과 통합하는 과정에 주도적으로 참여, 최종적으로 OpenClaw 유저에서 Maintainer로 성장
- 에이전트 상호운용, 오케스트레이션 등 복잡한 기업 요구사항을 충족하기 위해 다양한 실험을 진행함

### 다양한 에이전트/하니스를 Discord와 연동해 여러 작업을 병렬 처리한 실 사례를 소개함

- 개인적으로 Discord 채널 1~5개에서 각각 Codex 및 OpenClaw 에이전트와 동시 작업(프로그래밍, 테스트 등)을 실행한 실제 경험 공유
- 작업 흐름: Discord 채널을 Codex와 ACP를 통해 바인딩(연결) → 프롬프트를 전달 → Codex의 작업 결과를 타 채널로 전달하는 식의 병렬 워크플로우 구성
- 예시: PDF 문서 변환 과제를 출국 직전 Discord에서 전달 -> 결과물을 다른 채널로 옮겨 수령
- AI 에이전트 도구의 자가 개발 및 적용에 적극적이며, 일상적인 사이드 프로젝트 구현에 충실히 활용함

### ACP는 에이전트-클라이언트 간 상호작용의 표준으로, 중복노력을 크게 줄임

- 기존 MCP는 “모델에게 도구를 제공”, ACP는 “에이전트와 클라이언트의 커뮤니케이션 표준화”
- VSCode, Zed(러스트 기반), Cloud Code 등 IDE 환경마다 각각의 Agent 플러그인 구축이 난무하나, ACP 표준 채택 시 한 번만 플러그인 구축하면 다수 환경에서 재사용 가능
- Zed 등 새로운 에디터와의 결합 사례도 강조하며, 중복노력을 줄여 생산성 증대
- 경쟁 프로토콜(Agent-Agent Protocol 등)도 존재하나, 실제로 클라이언트에 필요한 어댑터의 실존 여부에 따라 ACP 중심으로 채택
- 발표자는 당시 Codex/CloudCode용 어댑터를 지원한 ACP를 선택해 개발 진행

### acpx 도구는 ACP의 CLI 연동·자동화 워크플로우 구현을 목표로 개발됨

- OpenClaw의 새로운 기능 추가는 CLI를 통해 수행되며, 이를 ACP와 자연스럽게 연계해 “에이전트간 커맨드라인 통신”을 가능케 하는 클라이언트 툴로 acpx를 개발
- acpx의 목적: 다양한 플랫폼에 연결된 에이전트의 조합/인터페이스 자동화를 손쉽게 처리, 향후 ACP 관련 자동화 기능의 중심도구로 성장 중
- 실제로 개인 업무 뿐 아니라 전체 워크플로우의 통제와 추상화, 여러 개발환경 간 일관성·유연성 유지 역할

### PR 리뷰 프로세스의 반복적 작업을 에이전트/워크플로우로 완전히 자동화함

- OpenClaw에는 하루 평균 300~500, 누적 6만 PS 이상이 몰릴 정도로 피드백이 폭증 → 분류, 필터링, 리뷰 등 매뉴얼 작업의 자동화 필요가 커짐
- 자동화 방식: PR이 오면 “무엇인지” 케묻는 작업(자동화), “최상의 수정안인지” 판정, 단순/심층 리팩터, 충돌해결, 리뷰 반복루프 등 일련의 SOP(표준운영절차) 정의
- 기존에는 커뮤니티에서 발생한 모든 데이터(예: sloppy PR 등)를 수동으로 분류·참조했으나, acpx로 기계적 작업을 대체해 생산성 제고
- 코드 내에서 JSON 등의 구조화된 데이터로 워크플로우 로깅·분류 자동화, 사람이 스타트만 정하면 나머지는 에이전트가 전담처리

### 자동화는 '슈퍼피셜(피상적)·루틴' 작업에 집중하며, AI 루프의 가치와 한계를 인정함

- 엔지니어의 경험에 따르면, “에이전트 반복 루프”가 반드시 나쁜 코드(모든 과제 자동화, PR 자동 머지 등 ‘Slop’)만 양산하는 것은 아님
- 피상적 버그, 간단한 리팩터(예: 리뷰 지적사항 적용) 수준의 반복은 오히려 자동화에 최적화
- 코드 충돌 해결 등도 예전처럼 사람이 직접 할 필요 없이 거의 자동화
- 이 과정을 통해 ‘소프트웨어 개발 워크플로우 자체를 에이전트로 추상화’ – Routine, 반복 작업의 human-out-of-the-loop 실현 가능

### 엔터프라이즈 에이전트는 ‘온디맨드, Disposable, 멀티 인스턴스’ 지원이 절실함

- 일반적인 챗봇(Discord/Slack/Teams 등)은 한 번에 한 인스턴스(한 계정·이름·프로필)만 생성 가능, 여러 에이전트 동시 운영이 어려움
- 현행 챗 앱은 멀티 에이전트 프로비저닝, 동적 인스턴싱 미지원 → 수동 app manifest 작성, 반복적인 앱 추가 작업 필요
- 이상적인 형태: 자동으로 필요 작업 단위별 에이전트 인스턴스 생성, 개별 관리/통신 가능(UI 등) 구조로 진화 필요
- ChatGPT, Slack, Discord 등 차세대 협업 플랫폼에도 이러한 에이전트 모델이 반영돼야 엔터프라이즈용 에이전트의 잠재력이 최대화됨

### Kubernetes 기반 에이전트 오케스트레이션을 통해 확장성과 관리 효율성을 확보함

- 고도화된 에이전트 운영 위해, Kubernetes 상 파드로 각 에이전트(instance)를 띄움 → 고비용이지만 가장 강력한 방식
- Codex, OpenClaw, ACP, CloudCode, GitHub 등 다양한 도구가 하니스로 활용될 수 있음
- 데이터 싱크(동기화)는 rsync, Dropbox류 알고리즘 등 외부 솔루션 활용 가능
- Helm chart 기반 클러스터 배포, 리액트(UI) 앱 등으로 관리 인터페이스 구성
- Text Cortex/spritz 등 오픈소스 오케스트레이터 프로젝트를 활용, 다양한 기업 내부 배포 사례 실무 적용

### 실제 Slack 이슈 및 오류처리 시 에이전트 자동 배치·진단·해결을 실연함

- Slack에서 “운영 배포 후 신규 버그 발생” 등 에러보고 시, 에이전트를 자동 할당해 버그 진단 루프 실행
- 에이전트의 Slack 통합이 불완전해 별도 UI(React, WebUI 등)에서 대화·작업 진행 – 보완 필요지점 강조
- Codex web, Devin 등 시중 도구처럼 작동하지만 Kubernetes 풀파드로 구동, 독립적 강력성 확보
- 오픈소스이므로 helm chart로 손쉽게 배포 및 커스터마이즈 가능, 내부 자동화/이슈처리 시스템 구축에 바로 적용
- Slack wiring, 에이전트 상태관리, 에이전트간 상호교환성, 추상화 등 확장성/유연성 핵심

### ACP 표준은 다양한 환경에서 에이전트 플러그인/오케스트레이터 교체를 자유롭게 허용함

- acpx 기반 오케스트레이션은 특정 에이전트에 lock-in 되지 않음(언제든 교체 가능)
- 실제 기업 배포·개인 실험 등 다양한 시나리오에서 한 번 개발된 플러그인을 여러 곳에서 쉽게 회수/재사용 가능
- 표준화와 오케스트레이션의 결합 효과로, 대규모 사용자 요구에도 빠르고 안정적으로 대응

### 발표자는 OpenClaw와 Text Cortex에서 오픈소스 에이전트 인프라 확대에 중추적 역할을 하며, 관련 오픈소스 참여를 권장함

- 클러스터/이슈 자동화 등 실제 적용 사례와 함께, 원하는 경우 오픈소스 프로젝트 지원 및 협업을 제안
- Text Cortex 소속에서 ACP, acpx, 오케스트레이터 등 다수 프로젝트의 실질적 개발주도 및 확산 역할
- 발표 마지막에 소셜 링크·평판 안내, 직접 문의 독려 등 관련 생태계 참여를 확대하고 있음을 부연
