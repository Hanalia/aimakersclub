## *에이전트 인터페이스 구축: 크롬 개발자 도구 팀의 MCP 경험에서 얻은 교훈* 핵심 요약

- MCP 서버와 CLI 도구를 사용하는 개발자들 중 절반 정도가 실제로 배포해 사용한 경험이 있음
- 크롬 개발자 도구(DevTools)는 매일 수백만 웹 개발자들이 웹 페이지 디버깅, 에러 탐지, 성능 프로파일링 등에 활용하는 핵심 툴임
- 최근 DevTools를 에이전트(Agent) 환경에 특화시킨 버전을 개발, 다양한 MCP 지원 환경(예: Gemini CLI, Cloud Code 등)에서 작동하도록 설계함
- 대량의 성능 트레이스 데이터(50,000줄 이상의 JSON, 수 메가바이트)를 에이전트에 그대로 제공한 결과, 컨텍스트 윈도 초과로 인해 에이전트가 제대로 활용하지 못하는 문제 발생
- 이를 해결하기 위해, 단순 요약(markdown 및 시맨틱 서머리)을 반환하도록 수정, 핵심 정보만을 제공해 에이전트의 효율성 및 정확성 향상
- 에이전트와 인간 사용자는 목표는 같지만 인지적 병목점이 달라, 인터페이스 구성 및 평가 지표(토큰 당 성공 결과 등)를 개별적으로 최적화해야 함
- 기능적 효율성 지표로 ‘토큰당 성공 결과(tokens per successful outcome)’를 제시, 효율성·정확성 모두 측정 필요하며, 사용자 여정별로 비교해야 함
- DevTools의 실제 개발에서는 툴 카테고리화, 슬림 모드, 명령행 조합 등 세 가지 방식으로 ‘토큰 소모 최적화’ 진행
- 오류 발생 시, 에이전트가 스스로 복구(자체 치유)하도록 에러 메시지 개선, 진단 플레이북과 프로액티브 디투어 도입
- 툴 설명 품질 문제와 ‘발견성’ 딜레마 해결을 위해 명확한 목적 정의, 사용 가이드라인 강화, 과도한 기능 추가의 부작용 등 트레이드오프를 인식하며 최적화 작업 지속
- 자동연결(autoconnect) 등 보안 이슈는 사용자 동의 절차 반복 등 인위적 마찰을 의도적으로 설계해 신뢰성 확보
- 마지막으로, 효율성/오류 복구/발견성/신뢰성 등 에이전트 경험(Agent eXperience, AX)이 점점 더 중요해지고 있음을 강조하며 네 가지 핵심 교훈 제시

## 세부 요약 - 주제별 정리

### 크롬 DevTools를 에이전트를 위한 특화 버전으로 재설계한 배경과 활용 방식

- Chrome DevTools는 웹 개발자들이 주로 웹 페이지 디버깅, 에러 탐지, 성능 프로파일링 목적으로 매일 사용하는 툴로, 크롬에 내장되어 있음
- 최근 Chrome DevTools의 기능을 ‘에이전트’ 전용 MCP 버전으로 개발
- MCP란 Machine Control Protocol로, 다양한 에이전트 하니스 및 클라이언트(예: Gemini CLI, Cloud Code, Codex, OpenClaw 등)에서 연동 가능
- 시연 영상에서 Gemini CLI를 통해 MCP 서버를 설정하고, 크롬을 실행시킨 뒤 성능 트레이스를 분석해 웹페이지 최적화까지의 과정을 시연
- 웹 사이트 성능 문제를 자동으로 탐지하고 대응할 수 있으며, 설정 및 설치 안내는 QR 코드를 통해 별도 제공

### 대용량 성능 트레이스 데이터의 비효율적 활용 문제와 요약 기반 개선방안

- 초기에는 성능 프로파일링 결과를 에이전트에게 그대로 제공(예: 5만 줄 이상의 JSON, 수 MB 데이터)
- 대량 데이터를 직접 제공하면 에이전트 LLM의 컨텍스트 윈도우 한계를 금방 초과하게 되어 ‘덤프 존’(context overrun) 상태로 빠짐
- 대다수 에이전트 하니스에서 배포전에 테스트했으나, 데이터 투입만으로는 에이전트가 기대만큼 잘 동작하지 않음
- 해결책으로, 데이터 전체 대신 시맨틱 서머리와 markdown 요약을 제공(예: LCP, INP, CLS 등 주요 웹 성능 지표만 간추려 제시)
- 핵심 문장에만 집중하도록 하여 에이전트가 본질적인 정보만 활용하고, 효율·정확성 동시 향상 유도

### 에이전트와 인간 사용자의 인지 구조 차이에 따른 인터페이스 및 평가 전략

- 인간 개발자와 에이전트 모두 문제 탐지 및 해결이라는 공통 목표를 가짐
- 그러나 인간은 시각적 복잡성이 높은 정보 구조에 취약, 색상/레이아웃 등으로 신호를 찾음
- 에이전트는 대용량 데이터와 복잡한 정보 처리가 병목점이며, 컨텍스트 한계/토큰 비용 등이 주요 제한 요소
- 성능 측정 기준을 ‘효율성(토큰당 성공 결과)’과 ‘정확성(목표 달성 여부)’로 구분해 각각 별도로 관리해야 함 

### ‘토큰 당 성공 결과(tokens per successful outcome)’라는 새로운 효율성 지표 도입

- 인터페이스의 ‘연료 효율성’을 산출하는 지표로 토큰당 성공 결과를 활용
- 예: 같은 기능도 사용자 여정(웹 스크래핑 vs. 디버깅)에 따라 토큰 소모량가/복잡성이 다름. 따라서 지표는 여정별로 비교해야 의미 있음
- 덜 복잡한 작업은 토큰 소모가 적고, 복잡한 인터랙션은 토큰 소모 증가가 불가피하므로 단순 비교는 금물
- 실제 개발에서는 인터페이스 내 ‘네온 바’ 등 시각적 도구로 각각의 사용사례별 효율성을 표시, 짧은 바(효율 저하)는 집중 개선 대상으로 관리

### MCP 서버의 토큰 소모를 줄이기 위한 세 가지 구체적 전략

- 1) 툴 카테고리화: 특수 도구(예: 크롬 확장 디버깅) 등은 기본 메뉴에서 숨기고, 필요 시만 노출해 컨텍스트 부담 완화
- 2) 슬림 모드: 핵심 세 가지 기능(페이지 선택/이동/스크립트 평가)만 노출하는 극단적 경량화. 토큰 소모 최소화에 효과적이지만, 추가 기능이 필요할 경우 반복 호출 필요 등 트레이드오프 존재
- 3) 명령행 인터페이스: 여러 명령을 연계·파이프라인 조합해 후처리 가능(예: 접근성 트리 추출 후, grep·click 조합), 모델이 토큰을 덜 사용하고, 일부 처리는 로컬에서 실행

### 오류 발생 시 에이전트의 자체 복구 능력 강화를 위한 조치

- 모든 에이전트 오류가 토큰 재시도 등을 야기하므로 토큰 소모에 부정적 영향
- (1) 명확한 에러 메시지 설계: 자체 치유 지원(예: “History entry to navigate not found.”에 추가 설명 문구 삽입으로 에이전트의 자가 복구 유도)
- (2) 프로액티브 디투어: 훈련 데이터의 편향이나 잘못된 경로로 빠지는 경우, 올바른 툴로 선회시키는 기능(예: 퍼포먼스 트레이스 대신 Lighthouse audit으로 가지 않도록)
- (3) 진단 플레이북: 사용자가 DevTools MCP 설정에 어려움을 겪는 경우, troubleshooting ‘스킬’을 자동 호출해 문제 해결 지원
- 전 이 과정을 통해 제품(에이전트 하니스)의 복원력(Resilience) 강화

### 툴 설명의 명확성과 ‘발견성’ 딜레마 및 최적화 방식

- 초기에는 ‘debug webpage’처럼 모든 작업을 하나의 거대한 툴에 몰아넣었으나, 성능·발견성 저하로 25개 개별 툴로 분할
- 문제는 이 경우 또다시 ‘어떤 툴을 언제 써야 하는가?’라는 발견성 이슈가 발생
- MCP 툴 설명의 97%에서 ‘quality smell’(품질 문제) 발견된 논문 언급
- 툴 설명은 UI(인터페이스) 역할을 하므로 명확한 목적, 사용 가이드라인 필수(예: ‘이 툴은 웹 퍼포먼스 문제 및 핵심 지표 탐지에 사용됨. 관련 지표: LCP, INP, CLS’)
- 설명을 길게 하면 컨텍스트 부담 증가, 짧게 하면 정보 결핍/잘못된 활용이 증가하므로 지속적인 트레이드오프 관리 필요
- 모델 세대/규모별로도 최적 설명 방식이 달라 ‘최소 기능 설명’ 수준을 탐색 중

### 추가 기능(스킬/마법서)의 효과와 한계 및 트레이드오프 인식

- 복잡한 워크플로우일수록 ‘skills’ 기능을 추가해 자동화 효과 증진 가능
- 그러나 너무 많은 스킬은 컨텍스트 창 크기를 늘리고, 불필요한 스킬 호출로 부작용 발생(효율성 저하, 오남용)
- 기능·설명의 과도한 추가가 문제를 단순 이전시키며, 최적 균형점 지속 개선 중

### 신뢰성과 보안(Trust & Security)을 위하여 사용자 동의(Consent)의 마찰 deliberately 설계

- MCP 기반 Chrome DevTools for Agents는 자동연결(autoconnect) 기능 지원, 사용자가 에이전트와 화면 공유 가능
- 유저요청: “항상 허용” 옵션 제공 → UX 관점에서는 당연해 보이나, 에이전트 자동화 환경에선 보안 경계 유지 필요
- Simon Willison의 ‘lethal tri factor’ 블로그 언급(보안 리스크 심화)
- 실제 설계에서는 ‘매 세션 동의 절차’ 의무화 → 의도적 마찰을 통해 보안(신뢰 경계) 유지 선택
- 환경별 보안 정책: 1) 로컬개발(인간 중심, 제한적 권한), 2) CI(검증 환경, 컨테이너/분리 프로필 활용), 3) 완전 인터넷 공개(도메인 allowlist, prompt injection 방지 등), 각 환경별 분리된 보안 모델

### 요약과 주요 교훈 네 가지 정리

- 에이전트 역시 사용자의 한 형태로, Agent eXperience(AX)의 중요성 증대
- DevTools for Agents 개발 과정에서 얻은 네 가지 핵심 교훈 소개:
    1. ‘토큰당 성공 결과’로 연료(데이터) 효율성 정량 측정
    2. 오류를 복구 플레이북으로 전환해 자체 치유력 증진
    3. 목적 중심 툴 설명/설계로 의도를 정확히 반영
    4. 편의성보다 신뢰성을 우선, 보안 마찰 의도적 유지
- 결론적으로, 에이전트가 곧 사용자가 되는 시대에서 효율성, 복구, 발견성, 신뢰성 등 4대 품질 요소를 균형 있게 실현하는 것이 핵심임
