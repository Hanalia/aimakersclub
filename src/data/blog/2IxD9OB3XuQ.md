## *AI 에이전트의 지속적 학습: 실패에서 내구적 개선으로* 핵심 요약

- 영상 제목: **AI 에이전트의 지속적 학습: 실패에서 내구적 개선으로**
- Soheil Feizi(메릴랜드 대학교 컴퓨터공학 교수, RELAI 대표)가 AI 에이전트의 지속적 학습(Continual Learning)과 내구적 개선(Durable Improvements)의 구조 및 도전 과제, 해결 방안에 관해 발표
- AI 에이전트는 인간처럼 경험을 통해 “잊지 않고” 지속적으로 학습·개선하는 것이 궁극적 목표임을 강조
- 지속적 학습은 모델(Model), 하니스(Harness: 프롬프트·도구·코드 등 컨텍스트), 메모리(Memory) 등 에이전트의 여러 계층에서 이루어질 수 있음
- 피드백 획득(어디서, 어떻게 얻을지)과 그 피드백을 실제로 적용·최적화하는 실무상의 두 가지 큰 과제가 존재함
- 벤치마크·평가 지표가 명확한 개발 단계와 달리, 실제 서비스(Production)에서는 주로 세션 로그와 불분명한 피드백만이 존재하여 학습 환경으로 활용하기 어려움
- 이를 위해 로그+피드백 데이터를 ‘재실행 가능한 실험 환경(Replayable Learning Environment)’으로 변환하여, 테스트 가능하고 검증 가능한 학습 사이클을 만들 필요성 강조
- 모델 파인튜닝보다 하니스·메모리 계층에서의 변화가 효율적이고, 더 빠른 개선을 이끌 수 있는 사례 소개됨
- 최첨단 방법론으로서 ‘검증 가능 지속적 학습(Verifiable Continual Learning)’ 개념 및 구현 원칙(재실행성, 전체성, 평생성, 효율성) 정리
- RELAI의 솔루션은 일상적 로그·피드백 신호로부터 자동으로 테스트 환경을 만들고, 누적적으로 안정적인 개선을 제공함
- 실제 벤치마크(가상 고객지원 에이전트 시나리오)에서 1회 최적화 루프로 10% 정확도 증가(78%→87%→97%)를 달성한 사례 제시

## 세부 요약 - 주제별 정리

### 인간의 경험 기반 학습을 모방하는 지속적 학습이 AI 에이전트 개선의 핵심임

- 인간은 환경과 상호작용, 피드백 수집, 경험 누적으로 학습함
- AI 에이전트 역시 이러한 인간식 학습 과정을 모방해야 지속적으로 성능이 개선될 수 있음
- 단순히 데이터를 추가해 학습하는 것이 아니라, 실제 ‘경험’을 통한 개선 및 ‘망각 없는’ 누적이 이상적임
- 지속적 학습(Continual Learning)은 에이전트가 계속 실세계에서 경험하며 자가 수정하는 것을 의미

### 지속적 학습은 모델, 하니스, 메모리 등 다양한 계층에서 일어날 수 있음

- 모델 계층: LLM 등의 파라미터 자체를 변경 (예: SFT, RL 기반 학습)
- 하니스 계층: 프롬프트, 툴, 코드, 워크플로우 등 컨텍스트 제공 방식을 변경 (예: 프롬프트 엔지니어링, 도구 추가·수정, 워크플로우 개선 등)
- 메모리 계층: 세션별 또는 영구적 정보 저장. 실패·사실 저장, 스킬·경로 추출 및 재활용 (예: LETA, MemZero)
- 각 계층별로 적용 가능한 학습·최적화 접근법이 다르며, 비용·효율성·안전성 차이 존재
- 좋은 학습 엔진은 세 계층 중 가장 효율적이고 내구적인 변화만 최소한으로 적용해야 함

### 효과적 피드백 획득이 지속적 학습의 첫 번째 관건임

- 개발 단계에서는 명확한 벤치마크와 평가 도구를 사용해 에이전트의 성능 피드백을 쉽게 얻을 수 있음
- 실제 서비스(Production) 단계에서는 구체적 점수·피드백 없이 세션 로그만 존재하는 경우가 많음
- 로그에서 피드백을 얻는 두 가지 방법:
    - 자동화: 다른 LLM, 코드, 자체 분석 에이전트를 활용해 로그 비평 자동화 (스케일은 빠름, 정교함은 낮을 수 있음)
    - 전문가 수동 피드백: 도메인 전문가가 소수 로그를 직접 평가 (정확성 높음, 비용·시간 소요)
- 그러나 이 두 방식 모두 ‘실행 가능·테스트 가능한 학습환경’이 아니라는 근본적 한계 존재

### 단순 로그+피드백은 불완전하며, ‘재실행 가능 학습환경’ 전환이 필요함

- 에이전트의 경험(로그+피드백) 단편만으로는 개선시 일관성·재현성·확장성을 보장할 수 없음
- 이를 극복하려면 단순 피드백 자료를 “재실행 가능한 시뮬레이션 환경”으로 승격해야 함
    - 실제 도구(혹은 가짜로 시뮬레이션한 도구), 합성 유저, 명확한 성공 평가자(evaluator) 등을 포함해야 함
- 한 번의 실제 발생 사건을 확률적·패턴적 분포로 확장, 모든 후보 에이전트를 이 환경에서 재실행·평가 가능해야 함
- 이로써 개선 제안의 진위, 영향, 오작동 발생 여부까지 체계적으로 검증·테스트 가능함

### 피드백 적용은 모델, 하니스, 메모리 계층 별로 방법론적 차이가 명확함

- 모델 계층(Update Model Weights)
    - SFT(지도 학습 파인튜닝): 올바른 행동 경로 imitation, 라벨된 샘플 필요
    - RL 기반 사후학습: DPO, GRPO, RLV R 등 보상/선호 시그널 바탕의 파라미터 강화
    - LoRA 등 제한 파라미터 학습: 낮은 비용, 더 안전한 업데이트
    - 단점: 벤치마크와 명확한 evaluator 필요, 로그+피드백만으론 직접 적용 불가
- 하니스 계층(Update Harness)
    - 트레이스-투-하니스(trace-to-harness): 로그+피드백을 분석, 코드 에이전트에게 개선 제안 요청
    - 장점: 로그+피드백만으로 가능, 단점: 변경 효과 검증·일관성 파악 어려움, '숨은 회귀(regression)' 위험
    - 진화 알고리즘 기반(GEPPO, prompt search): 프롬프트 변이·서치 후 채택, testable하나 명확한 벤치마크 필요
- 메모리 계층(Update Memory)
    - 사실·경로 기록(LETA, MemZero): 빠르고 저렴, 직접적
    - 스킬 추출·재사용: 성공 경로 압축해 차후 유사상황서 재활용
    - 단점: 업데이트 효과성을 사전 검증할 수 없어, 비의도된 회귀 위험 있음

### ‘검증 가능 지속적 학습’은 실사용 AI 에이전트 개선의 필수적인 방향임

- 검증 가능 지속적 학습(Verifiable Continual Learning, VCL): 에이전트의 경험 기반 학습 중 각 변경이 실제 개선되는지, 과거 성공을 깨뜨리지 않는지 입증하는 프레임워크
- 3가지 실무 단계
    1. 실패를 재실행 가능한 테스크로 만들기(실행 가능한 테스트 생성)
    2. 변경 전후의 개선치(Delta) 측정
    3. 과거 성공 케이스들이 모두 여전히 통과하는지 회귀(Regression) 테스트

### VCL의 4가지 원칙(재실행성, 전체성, 평생성, 효율성)이 지속적 학습의 핵심임

- **재실행성(Replayability):** 모든 실패 및 개선을 시뮬레이션 환경에서 반복 테스트, 평가 기준 포함
- **전체성(Holisticness):** 단일 실패에 여러 원인·수정책이 가능, 그중 ‘최소한의 내구적 변화’만 적용 필요. (예: 기억 오류, 프롬프트 미최적화, 도구 오류, 워크플로우 오류, 모델 한계 등 동시에 고려)
- **평생성(Lifelongness):** 새 수정은 과거 성과 유지하며, 이전 성공 케이스에 회귀(regression) 발생하지 않아야 함
    - 회귀 인지 최적화(Regression-aware optimization) 도입: 회귀를 사후가 아닌 최적화 루프 내에서 실시간 감지 및 방지
    - 과제: 환경 종류(K)가 많아질수록 효율적 관리가 필수
- **효율성(Efficiency):** 각 계층별 업데이트 비용(메모리 < 하니스 < 모델) 및 최적화 과정 자체의 비용 최소화 필요

### RELAI의 솔루션은 2개의 명령만으로 검증 가능 지속적 학습 루프를 구현함

- 최초 1회 원-타임 셋업: 에이전트에 ‘러닝 하니스’ 생성 (주요 LLM/에이전트 프레임워크 모두 지원)
- 루프 실행: 로그·피드백·명령 등 신호로부터 자동 학습 환경 생성 → 최적화 명령 실행
- 루프 내 각 단계:
    - 신호(로그 등)→재실행 가능한 학습환경 변환(재실행성)
    - 루트 원인 분석 후 적합 계층에 수정 적용(전체성)
    - 회귀 인지 최적화(평생성)
    - 최소 비용 움직임 추구(효율성)
- 결과물: 어떤 변경이, 왜 이뤄졌으며, 어떤 테스트를 통과했는지 모두 검토 가능한, 안정적이고 해킹 가능한 새로운 에이전트 버전 제공

### 실제 벤치마크 실험에서 VCL 기반 솔루션의 효과가 실증됨

- 가상 고객지원 에이전트 시나리오로 반복 가능 벤치마크 환경 구축
- 단일 명령만으로 무례하거나 악의적 콜러 마주친 경우 등 다양한 상황별 학습환경 신속 생성
- 각 시나리오마다 시뮬레이터(페르소나, 의도, 툴), evaluator까지 자동 포함
- 시뮬레이션을 통해 에이전트의 현재 점수(예: 78%) 파악, 1회 최적화 루프로 87%, 추가 최적화로 97%까지 개선
- 실제 서비스 로그+피드백 케이스(예: 환불 규정 초과한 관용 방지)도 같은 방식으로 들어가서, 기존 성능 훼손 없이 누적적으로 개선
- 각 업데이트는 개별적으로 테스트되고, 개선점 측정, 기존 기능 회귀 없음 보장

### “에이전트 지속적 학습 = 모델 파인튜닝” 통념을 넘어, 하니스·메모리 계층이 중요함이 강조됨

- 다수의 실질적·유용한 개선은 모델 무거운 파인튜닝이 아닌 하니스 개선, 메모리 업데이트로 충분함
- 효율성·비용 측면에서 계층별 ‘최소 변경’ 전략이 바람직함

### 실제 서비스에서는 로그만으론 부족하고, 재실행 가능 학습환경 변환이 필수임

- 서비스 로그+단순 피드백은 반복 가능·재검증 가능한 완전한 학습환경이 아니므로 실질적 에이전트 교정이 어려움
- 인스트럭션이나 로그+피드백만으로도 자동적 재실행 환경 구축 필요(프로그래밍 방식 구현)

### 회귀 인지 지속적 개선은 실패 수정이 역으로 과거 기능을 깨뜨리지 않는지까지 관리하는 최첨단임

- 실제 학습주기에서 각 신규 개선(ex: 새로운 규정 반영)이 이전의 모든 성공 기능을 손상하지 않는지 실시간 테스트
- 모든 업데이트가 누적적, 안전하게 쌓이도록 설계

### 발표 요약 및 실질적 3대 메시지가 최종 정리됨

- 핵심 메시지 1: 에이전트 지속적 학습은 꼭 모델 파인튜닝만이 아님(하니스, 메모리 계층도 중요)
- 핵심 메시지 2: 실제 서비스(Production) 로그는 별도의 변환·재실행 가능한 학습환경으로 승격 필요
- 핵심 메시지 3: 프런티어는 ‘회귀 인지형’ 지속적 개선이며, 과거 성과를 훼손하지 않고 누적적으로 개선하는 것이 목표임
- VERIFIABLE CONTINUAL LEARNING의 4대 원칙(재실행성, 전체성, 평생성, 효율성) 재강조, RELAI 솔루션의 실전 적용 안내
