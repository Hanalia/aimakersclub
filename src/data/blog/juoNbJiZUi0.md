## *LLM 코드 생성 실패와 이를 방지하는 전략 — PostHog 위저드의 실제 적용 사례* 핵심 요약

- 본 영상은 PostHog의 ‘PostHog Wizard’를 개발한 다니로 캠포스(Danilo Campos)가 LLM 기반 코드 생성의 실패 요인들과, 실제 제품에 안전하게 배치하는 전략을 구체적으로 설명함
- PostHog Wizard는 매월 15,000명의 사용자가 활용하고, 통합 절차를 ‘2시간에서 8분’으로 단축시켜 사용자로부터 긍정적인 소셜 미디어 피드백을 얻는 중임
- LLM 모델의 ‘모델 부패(Model Rot)’ 문제—구식 지식 및 최신 정보 반영 실패—는 실시간 문서(Markdown) 패치 및 선택형 컨텍스트 삽입으로 해결함
- 모델이 프로젝트에서 부정확하거나 이상한 아키텍처/패턴을 생성할 위험은 ‘모델 에어플레인(Model Airplane)’이라는 최소 구현 예시 프로젝트를 컨텍스트로 제공해 해소함
- 다양한 통합 경로가 남발되지 않도록, 프로젝트 비즈니스 맥락 탐색→이벤트 식별→구체적 코드 생성의 ‘브레드크럼’(Breadcrumb) 단계 전략을 도입해 통제함
- 에이전트가 제대로 동작하지 않을 때, 매 실행 후 “성공을 위해 어떤 정보가 더 필요했는가?”를 에이전트에 직접 질문하여 설계 오류와 사람(운영자) 실수를 빠르게 파악·수정함
- 에이전트가 사용자 시스템에서 민감 정보(예: .env 파일)를 과도하게 읽거나 유출하는 보안 문제는 ‘키 존재 확인’과 ‘키값 업데이트’만 허용하는 안전제약 툴 설계로 대응함
- 전체 솔루션은 복잡한 코드보다는 90% 이상 평문 문서(Markdown/Prse)에 집중, LLM이 이 지침을 상황에 맞게 재구성·적용하도록 레시피/설계서를 제공하는 방식임
- 기술 스택으로는 Claude Agent SDK와 CLI, 토큰 관리를 위한 LLM Gateway, 통합형 컨텍스트 서비스 등이 실제 현장에서 운영 중임
- 영상 전반에 걸쳐 각 사례별 문제와 해결책, 실제 경험에서 얻은 교훈 및 시스템 운영 구조가 구체적으로 소개됨

## 세부 요약 - 주제별 정리

### 코드 생성 에이전트 도입 경험에서 반복되는 실패와 고통이 개선 동기를 제공함

- 발표자는 여러 차례 “로봇에게 코딩을 맡기다 실패해 상처받은 경험”이 있다고 자조적으로 소개함
- PostHog Wizard는 이런 고통을 줄이고, 사용자가 개발환경 세팅 시간을 현격히 단축하도록 설계됨
- 실제로 15,000명 규모의 사용자가 월간 활용하며, 높은 만족도를 보여주고 있음
- 세션 시작 6시간 이내에도 Bluesky·Twitter에 긍정적 피드백이 2건 게시됨
- 그러나 LLM이 항상 올바른 코드를 자동 생성하는 건 아니며, 다양한 실패 사례가 축적되어 있음
- 이런 반복되는 실패와 문제의 구체적 원인을 분석해 해결책을 마련하는 과정을 공유한다고 밝힘

### LLM의 모델 부패(Model Rot) 문제는 최신 문서를 컨텍스트 주입으로 해결함

- LLM은 대규모 자본과 시간이 투입돼 훈련되지만, 공개 시점의 웹/정보 상태가 반영되어 있음 (6~18개월 지연)
- 특히 빠르게 변화하는 소프트웨어 환경에서는 LLM이 ‘현실을 반영하지 못하는’ 문제가 발생
- 과거에는 LLM이 PostHog 통합 시 임의 API/키/패턴을 만들어내기도 했고, 이로 인해 실제와 맞지 않는 결과를 초래함
- PostHog Wizard는 “문서(Pure Markdown)”의 실시간 셋을 에이전트가 자유롭게 선택·삽입하는 구조를 개발
- 사용자는 통합하려는 항목을 선택하면, 에이전트는 해당되는 최신 마크다운 문서를 컨텍스트에 삽입
- 모델에게 최신 정보가 즉시 적용되어 현행화된 코드/통합이 이루어짐
- 이를 통해 ‘허구의 API·패턴 생성’ 문제를 효과적으로 억제함

### 잘못된 아키텍처와 무의미한 구현을 막기 위해 ‘모델 에어플레인’을 활용함

- LLM 코드 생성 결과, 실제 프로젝트와 동떨어진 구조/의존성이 발생할 위험 있음
- PostHog 팀은 ‘모델 에어플레인(Model Airplane)’이라 부르는 경량 샘플 프로젝트를 제작해 모범 통합 양식을 제공
- 다양한 프레임워크/언어별로 준비하되 실제 배포용 서비스가 아닌, 검증 목적의 최소 구현 버전임
- 예를 들어, 로그인 기능이 ‘비밀번호를 아무거나 입력해도 통과’되는 식으로 간이하게 설계
- 이 모델 에어플레인을 LLM 컨텍스트에 포함시켜 ‘정상적인 통합 패턴’을 참고하게끔 유도
- 실제 대규모 프로젝트가 아니므로 토큰 효율성도 높게 유지 가능함
- 결과적으로, 일관된 통합 구조를 주변화 없이 LLM이 반복해서 산출하는 효과 있음

### 다양한 통합 경로가 난립하는 문제는 비즈니스 이벤트 중심의 단계별 설계로 해결함

- 자동화된 LLM 통합은 15,000명 사용자만큼 다양한 경로/방식이 난립할 위험 있음
- 이로 인해 지원·유지보수에 심각한 부담이 유발됨 (너무 다양한 패턴)
- 문제 해결을 위해 ‘브레드크럼(Breadcrumb)’ 방식 단계별 설계를 도입
- 처음엔 LLM에게 ‘PostHog 통합’ 언급 없이, ‘비즈니스 가치 있는 파일 탐색’(로그인/Stripe 결제 등)을 지시
- 다음 단계에서 ‘해당 파일 내 주요 이벤트명 및 설명’만 목록화하도록 요구함 (아직 코드 생성 금지)
- 목록화가 끝나면, 비로소 ‘포스트호그 통합 작업’을 실시하며, 이 시점에서 최신 문서를 프레임워크/언어별로 로딩
- 이런 단계별 유도방식을 통해 LLM이 전형적인 통합 구조를 따르며, 이상한 방식을 선택하지 않도록 통제함

### 에이전트(LLM)가 실수하는 가장 큰 원인은 인간(운영자)의 망각과 실수에서 발생함

- 에이전트의 실행 환경이나 도구 설정, 지시사항은 인간(운영자)이 수시로 변경/업데이트함
- 예를 들어, 과거와 충돌되는 지시문, 없는 도구를 언급, 언어/프레임워크 오류 등
- 실제로 hundreds(수백)번의 실행 동안 ‘존재하지 않는 MCP 툴’을 사용하라고 잘못 지시한 사례도 존재
- JavaScript용 지시문을 Python 프로젝트에서 주는 등 인적 실수 다수
- 근본 원인은 운영자의 기억력 한계(커피·아침 여부에도 좌우), 파편화된 정보, 관리 소홀에 있다고 강조

### 실행마다 LLM에게 “무엇이 부족했는가?”를 질의하고 자동화로 설계 오류를 빠르게 추적함

- 모든 에이전트 실행 종료 즉시, “이번 실행에서 더 성공적이기 위해 어떤 점을 개선해야 했는가?”를 LLM에 질의
- 사용자 리서치의 일환이지만, 리서치 대상이 ‘로봇’이라는 점이 특징
- 이를 통해, “필요한 도구에 대한 접근 권한 결여”, “지시문의 모순”, “언어 착오” 등의 오류를 LLM이 직접 보고
- 운영자는 이 피드백을 바탕으로 설계·지시문을 빠르게 패치해 오류를 확산 전 조기에 차단함

### 사용자 시스템에서 에이전트가 민감한 파일(.env 등)을 과도하게 읽거나 유출하지 않도록 툴 허가를 엄격히 제한함

- 에이전트가 사용자의 환경에서 arbitrary(무제한)한 권한을 갖는 건 위험하며 신뢰를 저해함
- 초기 버전에서는 환경설정 파일(.env) 전체를 읽어 클라우드로 전송하는 등 보안상 취약점 존재
- 개선책으로, ‘키 존재확인(check existence)’, ‘키값 쓰기(write value)’ 기능만 가능하게 함
- 민감정보는 클라우드·로그로 전송하지 않고, 파일 전체도 읽지 않음
- 툴 권한의 엄격한 제한으로 사용자 시스템 보호와 보안사고 예방

### 코드 그 자체보다 ‘텍스트 지침서’(Markdown) 품질이 중요하며, LLM이 이를 재구성해 최적 결과를 도출함

- 현대 LLM 기반 워크플로에서는 실제 코드보다 ‘플레인 텍스트 프로즈, 문서, 지시문’이 더 큰 가치를 지님
- 실제 PostHog Wizard는 90% Markdown과 문서, 8%는 문서 활용 툴, 나머지 일부만 진짜 코딩 로직임
- 해당 문서가 잘 쓰였을수록, 이후에 더 뛰어난 모델이 등장하더라도 계속 가치를 발휘
- LLM/에이전트가 프롬프트, 지침서, 예시코드를 자유롭게 활용해 목표를 달성하도록 설계
- 개발자는 “에이전트를 과도하게 제한하는 것”보다, 충분한 정보와 컨텍스트 제공–순차적 공개(시퀀싱)가 더 중요하다고 강조
- 즉, 코딩량이 줄어들고 ‘텍스트-레시피 설계’가 핵심 자산이 되는 패러다임 전환을 시사

### 시스템 작동 구조: 모델에어플레인·문서·컨텍스트 서비스·CLI 등 연동

- ‘Skill 파일’이라는 컨텍스트 패키지가 존재, 각 Skill마다 문서/모델에어플레인 등을 포함
- 컨텍스트 서비스가 모든 모델에어플레인을 하나의 플랫 마크다운 파일로 병합, Skill에 통합
- LLM이 해당 Skill을 레퍼런스 삼아 일관된 통합작업 가능
- 실제 클라이언트 도구로는 Claude Agent SDK+CLI를 사용, PostHog 로그인시 무료 추론 기능 제공
- LLM Gateway로 토큰 소비 관리, Sierra·Clawed 코드 관리 등 실제 현장 운영 노하우 부연

### Q&A를 통해 시스템 설계 및 운영 방식이 구체적으로 보충 설명됨

- 컨텍스트 드라이브 방식 및 Skill 구조에 대한 질의응답이 있음
- 각 Skill 파일에 어떤 자료(문서, 모델에어플레인 등)가 포함되는지 구체적 안내
- Claud agent SDK를 래핑한 CLI 및 LLM Gateway, 인프라 운영 경험, 이슈 공유
- 현 단계에서 “서비스형(Codegen as a Service)” 사업 모델로서는 아직 시행착오와 개선진행 중임

### 발표 마무리: 반복된 실패에서 배우는 교훈과 새로운 패러다임 확산 메시지

- 전체 과정에서 얻은 경험(코딩이 아닌 텍스트 설계가 핵심)이 강조됨
- Q&A 이후 발표 마감, 코드 자동화의 실제적 현장 문제와 해결책, 교훈이 요약됨
- 시청자에게 “좋은 하루 되라”는 인사로 마무리
