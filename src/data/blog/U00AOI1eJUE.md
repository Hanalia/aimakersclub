## *공개 MCP 서버를 손상시키지 않고 유연하게 다루는 방법* 핵심 요약

- 영상 제목: *공개 MCP 서버를 손상시키지 않고 유연하게 다루는 방법 — 님로드 하워저, Baz*
- Baz(버즈)의 창립 엔지니어 님로드 하워저가 Playwright MCP 서버와 외부(서드파티) agentic 도구를 실제 예시와 코드로 다루며, 이상적이지 않은 성능을 극복하기 위한 5가지 핵심 개선 프레임워크를 소개함
- MCP 서버에서 제공하는 외부 도구는 기본적으로 범용적이고 설명이 얕으며, 이로 인해 비결정적(agentic) 에이전트는 예측 불가한 행동, 성능 저하, 보안 이슈까지 발생시킬 수 있음
- Baz의 주요 사례로 ‘spec reviewer’(요구 사항과 구현 비교 자동화) 제품을 예시로 들며, 현재 애플리케이션에 Playwright의 MCP 서버를 통합해 자동화 검사 흐름을 시연함
- 에이전트 자동화 검증(예: 버즈의 기능 설정 서랍 확인) 시도 초반, vanilla MCP 도구를 그대로 쓸 때 오작동(404 페이지, 잘못된 파일 경로, 스냅샷 실패 등)이 발생함을 시연함
- 5가지 베스트 프랙티스(외부 도구 큐레이션, 설명 래핑, 결정적 가드레일, 도구 조합, 결정적 함수 호출)를 단계별로 실제 코드와 함께 구현해 성능과 신뢰성을 크게 개선하는 과정을 상세히 소개
- 각 단계에서 필터링, 강조 설명, 경로 지정 등 매우 구체적이고 실질적인 방법들(예: 툴 이름/설명 매핑, 경로 유효성 체크, 에이전트 피드백 메시지 설계 등)이 활용됨
- 마지막으로 '로그인' 등 반복적·고정적인 작업의 경우, 에이전트에게 맡기기보다 결정적으로 직접 처리함으로써 전체 흐름의 견고함을 확보

## 세부 요약 - 주제별 정리

### Playwright MCP 서버와 외부 Agentic 도구는 그대로 사용 시 예측 불가 및 성능 저하 문제를 유발함

- Playwright의 MCP 서버는 수십 개의 서드파티 도구(브라우저 조작, 스냅샷, 클릭 등)를 제공하면서, 각 도구에 얕고 일반적인 설명만 포함함
- Baz가 예시로 든 'spec reviewer'(스펙 자동 점검기)는 티켓 관리시스템(Jira 등)에서 요구 사항·디자인을 읽고, MCP 서버 도구로 실제 구현 페이지를 자동 체크해 증거 스냅샷과 판정(verdict)을 남기는 기능임
- MCP 서버의 vanilla 도구만을 적용(베이스라인)했을 때, 에이전트가 비현실적인 주소로 이동하거나, 스냅샷에 실패하며, 잘못된 파일 경로를 사용하는 등 다양한 실패 케이스 발생
- 에이전트는 본질적으로 비결정적이라, 외부 도구의 설명이 부정확/일반적이면 예상외 동작, 보안 위험, 성능 저하가 쉽게 유발됨
- 멀티테넌트 시스템 등에서 외부 도구가 아키텍처 정보를 모르고 동작하면 데이터 유출 등 치명적 보안 이슈도 동시에 발생 가능
- 따라서 out-of-the-box 도구 사용만으로는 실무 적용에 한계가 많음을 체감할 수 있도록 사례 연계 시연

### 도구 필터링 및 큐레이션으로 컨텍스트 혼잡을 줄이고 성능을 개선할 수 있음

- MCP 서버의 기본 21개 도구 중 실제 사용에 불필요한 기능들(예: 브라우저 리사이징, 드래그, 임의 코드 실행 등)을 필터링
- Python 리스트 컴프리헨션 등으로 제외 목록을 생성, get_tools 함수에서 해당 도구만 제거 후 반환
- 실험 결과, 에이전트의 컨텍스트 창에 16개 도구만 남게 되어 선택지가 줄어듦 → 혼란도 감소, 에이전트가 더 목적 지향적으로 동작 가능
- 이는 일부 도구가 많을수록 혼란이 커질 수 있으므로, 프로젝트나 제품 특성에 맞추어 필요한 도구만 선택적으로 노출하는 것이 중요함을 강조

### 도구 설명 래핑을 통해 특정 도구 사용 방법을 명확히 안내하고 에이전트 행동을 유도함

- MCP 도구의 기본 설명은 "Press a key on the keyboard", "Close the page" 등 매우 일반적이고 얕음
- 프로젝트 특화된 래핑(wrapping) 클래스(tool wrapper)에서 도구 이름별로 상세 설명(enhanced description) 매핑을 직접 정의
- 예를 들어, ‘접근성 스냅샷’ 도구는 “현재 페이지의 모든 버튼과 메뉴 목록을 텍스트로 캡처하라” 식으로 구체적으로 수정
- '클릭' 도구는 "반드시 접근성 스냅샷 도구를 먼저 실행한 뒤 클릭하라" 등 사용 순서 및 주의사항도 설명에 포함, 경험에서 유도된 베스트 프랙티스를 직접 반영
- 설명 매핑은 딕셔너리 형태로 관리, 래핑 함수에서 원본 도구와 신규 설명 조합으로 새 도구 객체를 생성하여 반환
- 이로써, 에이전트가 도구를 언제·어떻게 사용할지 판단 시 구체적 힌트를 받게 되어, 실제 워크플로 성능/정확도가 현저히 개선됨

### 경로 제어 등 결정적 가드레일 도입으로 보안 및 미션 크리티컬 작업을 강력하게 보호함

- 에이전트 행동 중 보안/중요 업무(예: 스냅샷 이미지 저장 경로)는 비결정적 로직에 맡기기 위험
- take_screenshot 도구에 대해, 저장 경로(path/filename) 유효성(스크린샷 폴더 하위인지 등)을 직접 검사
- path_validation 클래스로 경로가 의도한 폴더 내에 상대적으로 위치하는지 check, 조건 불만족시 예외 발생
- 단, 에이전트가 실패 시 전체 프로세스가 멈추지 않도록 커스텀 에러 메시지를 에이전트에 반환, "접근 거부. 여기 저장해야 한다" 등 안내
- 에이전트는 정상 경로로 재시도하여 적절히 저장하게 유도함(보안, 데이터 격리, 사용자별 데이터 보호에 효과적)
- needle-in-the-haystack, loss-in-the-middle 등 LLM 에이전트의 한계로부터 실무적 신뢰성을 지키기 위한 강력 결정적 제어 코드

### 기존 도구로 신규 도구를 조합 생성해 더 풍부한 에이전트 행동을 설계할 수 있음

- 스냅샷 등 도구의 활용 맥락에 따라 특화 도구(예: evidence tool)로 만들어, 실제 업무에 적합하게 사용 가능
- 예시: 기존 take_screenshot 기능을 활용하지만, 신규 evidence tool은 "증거 수집"이라는 목적에 맞춰 별도 설명과 파일명 규칙(티켓 번호 포함) 등 지정 가능
- 신규 도구는 사용 의도와 조건, 추가 guardrail 등 더 세밀한 제어 가능 → 컨텍스트/목적별 도구 선택과 행동 유도 용이
- 코드상에서는 래핑 함수 내에서 기존 도구 유무 확인 후 신규 기능 조합(예: 스크린샷 도구 기반 evidence tool 추가)
- 에이전트는 “evidence” 맥락에서는 일반 스크린샷 대신 evidence tool을 특정 조건에서 사용하도록 실행망 설계 가능

### 반복적·고정적/복잡한 작업(예: 로그인)은 도구를 직접 결정적으로 불러 해결하는 것이 실용적임

- 예시: 로그인을 에이전트에게 시키면 실패율이 높음, 클라이언트/환경별 로그인 방식이 달라질 수 있음
- get_tools에서 MCP 서버의 모든 도구를 가져온 뒤, log_in_to_buzz와 같은 함수에서 직접 JWT 토큰 삽입 등 고정된 로직으로 로그인 처리
- 이후부터만 에이전트에 컨트롤을 넘겨 실제 검증, 워크플로 수행
- 실무에선 로그인·초기화, 민감한 환경 세팅 등 반복적이고 반드시 필요한 프로세스는 agentic workflow 밖에서 결정적으로 처리하는 것이 안정성과 통제력 확보에 효과적

### 각 개선 단계 이후 실제 워크플로가 성공적으로 동작하며 보안·정확성·효율 모두 개선됨을 실시간 코드로 검증함

- vanilla MCP 도구 사용 시: 잘못된 페이지 네비게이션, 스냅샷 저장 실패, 허위 판정 등 명백한 실패 사례 발생
- 5가지 프레임워크 적용 후 재실행: 설계(다크모드·라이트모드 차이 등) 불구, 원하는 drawer UI 확인, 티켓 번호 포함 스크린샷 저장, reasoning 기록 등 성공적 결과 산출
- 실 제품에서는 프론트엔드에서 pixel-perfect 검증까지 적용, 마진/패딩 등 세세한 시각적 요소까지 agentic 방식으로 검증 가능함을 언급

### 에이전틱 도구 통합 시, 성공적인 결과를 얻으려면 각 도구를 최대한 주어진 목표/상황에 맞게 적극적으로 가공하고 맞춤화해야 함

- 성능 향상, 보안 향상, 에이전트 로직의 “의도적” 통제 모두를 위해 도구 큐레이션·래핑·가드레일·조합·결정적 처리 등 각 방법을 목적에 맞게 혼합 적용
- 외부 MCP 도구는 물론 직접 작성한 통합 라이브러리, API 등 모든 서드파티 agentic 도구에 적용 가능
- 무조건적 자동화보다는, 각 목적별로 맞춰서 agentic 컨텍스트 설계 및 제어 밀도를 높이는 것이 실질적 효과를 가져옴

### MCP/agentic 도구 최적화의 향후 발전 가능성과 실무적 적용 팁을 간략 정리하며 마무리

- MCP 서버와 agentic 도구, 라이브러리 등은 out-of-the-box로 바로 쓸 때 한계가 명확함
- 도구를 깊이 있게 커스터마이즈, 목적에 맞게 엮으며 워크플로별로 정밀 제어하는 것이 차별화 포인트임을 실감
- 실무에서는 다양한 실험과 튜닝, 목적에 따른 맞춤형 개선이 필수
- 영상 제작자는 향후에도 적극적으로 관련 기술과 툴킷에 대해 공유하고 커뮤니티와 소통 의지 표명

### 실제 코드 예시 및 구조적 설명을 기반으로, 단계별 개선을 직관적으로 이해할 수 있게 시연/분석함

- 각 단계(V0 ~ V4, vanilla → curated → wrapped → guardrail → compose → deterministic)는 실제 Python 코드 구조와 함수, 클래스(class), 메서드 내용으로 상세하게 추적/설명
- 도구 처리의 상세 흐름(get_tools, tool_wrapper, path_validation, wrap_playwright_tools 등) 명확히 시현
- 모든 변경점에 대해 의도·방법·결과를 시각적으로 설명하며, 전 과정이 통합 프로젝트 맥락에서 유기적으로 진행
- 각종 helper 함수, 오류처리, 에이전트 메세지 설계 등 실전적이고 구체적인 테크닉 공유
