## *CI/CD는 죽었다, 에이전트는 지속적인 연산과 컴퓨팅이 필요하다* 핵심 요약

- 영상의 제목처럼, 기존 CI/CD(지속적 통합/배포)는 더 이상 오늘날의 '에이전틱 소프트웨어(agentic software)'와 대규모 코드 생성·배포 환경을 다루기에 부적합함을 강조함
- 에이전트 기반 개발(LLM, 오토노머스 소프트웨어 등)이 기존의 소수 PR(풀리퀘스트) 기반 인간 개발 주기 패러다임을 완전히 무너뜨리고 있음
- CICD 파이프라인은 인간 개발자가 작성한 몇 개의 PR을 느린 속도와 확실한 검증 절차(리뷰, 테스트, 배포)로 처리할 때 유효했으나, 대량의 에이전트 변화에는 한계를 드러냄
- 최근 GitHub의 커밋, 코드 추가/삭제량 등 주요 지표가 폭발적으로 증가함—대량의 짧은 생명체계 브랜치와 코드 충돌이 일상화됨
- 이에 따라, '캐시'와 같이 애플리케이션 수준의 오케스트레이션 레이어가 필요한데, 하드웨어와 소프트웨어의 긴밀한 공동 설계가 중요하다고 주장함
- 기존 PR 단위의 '순차적 검증 + 딜레이' 구조 대신, 명시적 목표(intents)와 계획(plan)을 바탕으로 에이전트가 자체 루프를 통해 빌드·내부검증·외부검증을 신속하게 반복하는 구조가 등장함
- 외부검증(리뷰)은 인간이 아니라 보안 LLM, API 컨포먼스 LLM 등 다양한 자동화 에이전트가 담당하게 되고, 인간은 '결과'의 통합적 평가만 담당한다는 새로운 개발 흐름이 등장함
- 중간 '프리머지(pre-merge) 큐' 등 병렬적 작업과 의사결정, 멀티버스(여러 커밋 기반 동시 브랜칭) 등을 예시로 기존 CICD의 구조적 변화 필연성을 구체적으로 설명함
- CI 원칙(불변성 검증, 컴플라이언스, 거버넌스 등)은 '루프 내 자동화'로, 오케스트레이션·관리·효율성을 최우선으로 하는 새로운 개발세계가 가까운 시일(몇 주~몇 달)에 도래한다고 전망
- Namespace·Ramp·Fall·Zed 등 실제 기업들의 변화를 통해, 이미 일부 개발 조직은 에이전트 기반의 'CI/CD 이후'로 진입하고 있다고 사례를 들며 발표를 마침

## 세부 요약 - 주제별 정리

### 기존 CI/CD 파이프라인은 급격히 진화하는 소프트웨어 개발 환경에 더 이상 적합하지 않다고 강조함

- Madison(NEA 파트너, Meta AI 출신)과 Hugo Santos(Namespace CEO, 前 Google 마이크로서비스 리더)가 영상 발표를 진행함
- 전통적 CI/CD는 "빌드 → 테스트 → 배포" 및 인간의 리뷰가 지속적으로 루프를 이루면서, 소수의 PR(풀리퀘스트)을 다루는 구조임
- 기존의 주기: 개발자가 코드를 작성 > PR 제출 > 동료의 코드 리뷰 > 자동화된 빌드/테스트/배포 > 피드백 및 반복
- 인간이 직접 PR을 올릴 때는 한 주에 한두 건의 변화가 일어나는 수준으로 병목이 적고 인프라 병목이 '인간의 속도'에 가려짐
- 현재는 에이전트/자동화 시스템이 인간을 대신해 다수(수백~수천)의 PR·브랜치를 짧은 시간에 쏟아냄
- 기존 CI/CD 파이프라인 구조로는 이렇게 폭증한 변화의 합병, 테스트, 배포를 감당할 수 없다는 점을 강조함

### 에이전트 기반 개발환경에서는 기존 PR·코드 리뷰 프로세스가 병목으로 작용함을 실증적으로 설명함

- 에이전트 기반 개발 환경에서는 N개의 PR, N개의 레포에서 동시 다발적으로 변화 발생
- 기계적 리뷰가 없거나 부족하면, 병렬 작업이 꼬이고, 브랜치 머지(merge) 충돌 등으로 혼란 초래
- GitHub의 최근 실시간 통계(커밋 수 및 코드 변경량(추가/삭제))에서 변화 폭발적 증가 확인
- 인간이 중심이던 루프에서는 작업량이 예측 가능했으나, 에이전트 기반 루프는 짧은 브랜치가 난립하며 병렬적 변화와 합병의 '불가능' 상황이 빈번
- Review bot 등의 자동화도 도입되고 있으나, 근본적인 구조개혁이 필요하다는 주장

### 대량의 코드 변화 속도에 대응하려면 캐시 및 오케스트레이션 계층이 핵심이 됨을 언급함

- 느린 빌드·테스트·배포 단계는 현재 개발자들이 자주 겪는 문제
- GitHub Actions 등 기존 CI/CD 인프라 위에 '캐시(cache)'를 얹어 오케스트레이션 레이어를 구축해야 속도 개선 가능
- 이 과정에서 하드웨어(HW)와 소프트웨어(SW)가 공동 설계(co-design)되는 것이 필수로 제시됨
- 캐시/오케스트레이션 계층은 '인그레스 셰이핑(ingress shaping)', '레이팅 리밋(rate limiting)' 등 네트워크적 제어 역할도 추가됨
- agentic identity(에이전틱 소프트웨어 개별 식별), 대규모 리트라이(retry) 등 확장적 운영 요소 필요성도 짚음

### 소프트웨어 개발의 중심이 '의도와 계획' 단위의 자동화 루프로 전환되고 있다는 실무 사례를 들음

- Namespace가 협력하는 Fall, Zed, Ramp 등 실제 기업들은 이미 이러한 패러다임 전환을 경험 중
- 'PR'이라는 업무 단위가 사라지고, 목표(intention)와 계획(plan)이 먼저 명확하게 정의됨(예: linear 티켓, Slack)
- 에이전트가 계획을 받아 루프(harness loop)를 시작—코드 체크아웃, 빌드, 테스트(내부검증), 피드백 등 각 단계 자동화
- 사람에게 "문제 없나요? 계속 진행할까요?" 등 간단 피드백만 받아가며, 나머지는 에이전트가 내부 루프에서 처리함
- 내부 검증 후, 여러 변경이 병렬적으로 '프리머지(pre-merge) 큐'에 쌓이고, 이후 통합과정(인간 외부승인)이 거쳐짐

### 외부 검증 단계에서 인간 리뷰 없이 자동화된 에이전트 심사가 대세가 될 것임을 제시함

- 코드 리뷰 단계를 보안 LLM, API 컨퍼먼스 LLM 등 특정 목적의 자동화 에이전트가 대신
- 예: 보안 체크, API 규격 준수 등 각종 퀄리티 보증을 LLM 기반으로 자동화
- 인간 리뷰어는 '기능 동작 영상', '자동화된 보안 보고서' 등 최종 산출물을 종합적으로 확인(의도-결과 비교)하는 역할로 변모
- "한 명의 개발자가 감당 불가"한 변경량(len=4배 PR 증가, 병렬변경 등)에 대응하기 위한 자동화의 필요성을 구체적 체감 경험으로 설명

### 프로세스 속도와 상태유지(statefulness), 멀티버스 브랜칭의 필요성을 구체적으로 강조함

- 내부 개발 루프에서 '완전히 상태가 유지되는(stateful)' 환경 중요—매번 새로 시작하면 매우 느려짐
- 동시다발적 병합이 필요한 시점에는, 마지막 커밋이 아닌 여러 후보 커밋(commits)상에서 병렬적 브랜칭(멀티버스)이 일반화됨
- 그 반영으로, 자원 사용량 폭증(동시 탐색 후보 큐 증가) 대처와, 불필요 작업 중복방지(캐시/증분 처리) 전략을 병행해야 효율성 유지
- 효율성은 "새로 시작하지 않는다", "중복된 작업 안 한다", "엔지니어가 로컬에서 작업하듯 증분 방식"으로 달성

### 병합(merge)·배포(ledger)에 있어 '시리얼라이저빌리티'와 '프리머지(pre-merge)' 큐 시스템 같은 데이터베이스적 방식이 적용됨을 주장함

- 많은 변경을 한꺼번에 serial하게 반영해야 하므로, high-performance DB의 serialization/ledger 패러다임(단일 원장, 동시잠금)이 코드 저장소에도 도입
- 병합 대기 중 변경 충돌 reconciliation(조정)이 자동화되어야 하며, 이 작업이 선결되지 않으면 chaos 초래
- 병합 전 대기 단계(pr-merge queue)에서는 여러 변화가 그룹화/의미 단위로 묶여 인간이 관리 가능한 형태로 제공됨

### 기존 CI 원칙(불변성·컴플라이언스·거버넌스)은 새 구조 내에서 더욱 자동화·내재화될 것임을 설명함

- 빌드/테스트 내부 검증, 시작점(commit)의 신뢰성, 외부코드 유입 차단 등 불변성(invariant) 검사는 루프 내 반복적으로 자동화
- 거버넌스는 오케스트레이션 레이어(에이전트 하네스, harness) 및 각종 정책검증으로 내재화됨
- CI가 완전히 사라지는 것은 아니나, 작업의 '중심축'이 루프와 자동화된 에이전트로 이동

### 미래 전망으로, 대폭 자동화된 개발 오케스트레이션 시스템이 몇 주~몇 달 내 본격 상용화될 것임을 주장함

- 속도: 코드 생성·검증·병합 전 과정이 인간 리뷰 없이 에이전트(LLM/자동화) 루프로 수 초~수 분 내 진행
- 효율·컴플라이언스·거버넌스 등 기존 CI 중심 철학은 한층 진화된 구조에서 계승
- Namespace 등 업체, Fall·Zed·Ramp 등 실제 적용 및 선도사례가 이미 등장
- "기존 시스템에 안주하면 죽는다"는 경고 및, 개발 생산성/효율성 혁신에 유연한 대응이 생존 핵심임을 강조하며 발표 마무리
