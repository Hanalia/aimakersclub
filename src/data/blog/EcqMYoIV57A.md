**채널명:** AI Engineer

## *더 많은 컨텍스트가 에이전트를 더 멍청하게 만드는 이유와 해결법* 핵심 요약

- 영상에서는 Nupur Sharma(Qodo)가 최근 에이전트의 발전 흐름과 그 과정에서 컨텍스트(문맥) 관리가 갖는 핵심 문제점을 다룸
- 에이전트 설계에서 단순히 컨텍스트 윈도우(모델 입력가능 길이)가 커진다고 결과의 질이 개선되지 않음을 강조함
- LLM(대형언어모델)은 주로 입력 초반과 말미의 정보에 집중하고, 중간 영역 컨텍스트를 무시하는 'U자 곡선' 패턴이 실험적으로 발견됨
- 복잡한 멀티에이전트 시스템이나 툴 추가는, 각 에이전트 간 충돌·오해를 증가시켜 결과의 신뢰성을 오히려 저하시킬 수 있음
- 단순한 '모든 컨텍스트 투입' 방식 대신, 컨텍스트 추출·최적화 전략(예: 계층적 요약·지식그래프·반복 검색 등)이 필요함
- 컨텍스트 엔진, 계층적 요약, 지식그래프 등 여러 방법의 장단점, 비용, 적용 한계 등 구체적으로 소개됨
- 최신 LLM은 '문제 해결 방식'에 대한 탐색에 치중해 불필요한 리서치 루프에 빠지는 오케스트레이션 패러독스가 발생함
- Qodo팀은 80/20 하이브리드 접근(80%는 자유 추론, 20%는 결정적 검증 단계)으로 리서치의 무한 루프와 품질 저하를 통제함
- 강한 멀티에이전트 시스템의 설계(전문 에이전트 + 판정 에이전트 구조)와 이를 실제 코드리뷰 등 실무에 적용한 사례 및 장단점 논의됨
- 조직의 요구(안내문서·PR 이력·규칙 등)에 따라 추천, 예외처리, 가중치 반영 등 피드백 체계를 도입해 실효성을 높임
- 최적화된 컨텍스트 전달 및 평가·피드백 구조 설계가 곧 지능형 에이전트의 실전 성능 차별화의 핵심임을 실증 사례와 함께 설명함

## 세부 요약 - 주제별 정리

### 에이전트 진화 과정에서 컨텍스트 입력 확대는 오히려 부작용을 낳음

- 초창기 에이전트는 4K 내외의 작은 컨텍스트 윈도우에 '사용자가 중요하다고 판단한 정보'만 입력
- 사용자가 오입력하거나 불충분한 정보를 줄 경우 출력 품질이 낮았지만, 결과의 예측가능성·결정성이 유지됨
- 컨텍스트 윈도우 확대 및 툴 활용, 멀티에이전트 워크플로우로 진화하며 "더 많은 데이터를 넣으면 성능이 더 좋아질 것"이라는 기대가 생김
- 실제로는 툴 수 및 에이전트 수 증가에 따라 정보 파악, 결과 통합 등에서 예기치 못한 오류와 혼란이 오히려 증가
- 각 에이전트가 모든 정보를 완전히 이해하지 못하거나, 목표 해석에 혼선이 생기는 경우가 빈번히 발생함

### LLM이 입력 초·말 컨텍스트에만 집중하는 U자 곡선 현상이 발견됨

- Qodo팀의 실험 결과, LLM(예: GPT 등)은 전체 컨텍스트 중 '시작 부분'과 '끝 부분' 정보만을 집중적으로 반영
- 중간에 위치한 컨텍스트 정보(예: Jira, MCPS 관련 문서 등)는 결과 도출에 실제로 거의 반영되지 않는 패턴 확인
- 이러한 현상 때문에 '모든 컨텍스트 던지기' 전략은 불필요한 비용을 초래할 뿐 실질적 효과가 미비
- 모델 평가 및 벤치마킹 과정에서 이 부족한 맥락 활용 문제가 재현됨

### 쏟아 붓는 입력 대신 맥락 선택적 최적화 전략이 중요함

- 단순히 "모델이 알아서 알아낼 것"이라는 접근은 한계가 명확하므로, 입력을 전략적으로 선별해야 함
- 컨텍스트 엔진(검색·랭킹), 계층적 요약(파일/폴더 단위 요약), 지식그래프(Key dependencies), 반복 검색 등 복합 전략 활용
- "전부 요약" 전략은 LLM 연산 비용이 높지만, 중요한 파일·폴더만 빠르게 파악할 수 있음
- 지식그래프는 논리적 의존성이 복잡한 대규모 코드베이스에 유리하나, 구축 초기에 개발자의 심층 입력이 필수
- 반복 검색(Iterative Retrieval)은 개발자 입력 부담이 적고, 효과적 인덱스 기반 검색이 가능함

### 컨텍스트 엔진, 요약, 그래프, 반복검색 방식의 장단점과 스케일 한계

- 컨텍스트 엔진은 대규모 코드베이스에서 검색·우선순위 결정에 효과적이나, 저장소 수 증가(600~700곳 시) 시 맵핑·검색 지연 발생
- 계층적 요약은 변화가 잦을 경우 요약 업데이트 비용(LM 연산량) 급증, 장기적 유지 부담 큼
- 지식그래프는 복잡한 함수/파일 참조, Repo 이슈 종속성 등 파악이 용이하지만, 일반 개발 파이프라인에는 적용 난이도 높음
- 반복 검색은 '도서관 카드' 개념으로, 주제별 인덱스만 들고 세부 내용은 필요시 검색해 비용을 절감
- 각 방법의 선택은 조직 규모·문제 구조·개발팀 특성 등과 맞물려 결정됨

### 셀프코렉션, Critic Node 등 후처리 기반 재검증 방식이 효율적임

- 에이전트가 1차로 작업 후, Critic Node(비판자 노드)가 해당 결과가 초반 목표와 부합하는지 검증
- 만약 부적합하다면 에이전트에 재작업(재시도)을 요청하여 맥락 손실 시에도 결과 품질을 높임
- 해당 방식은 초기 개발자 입력 부담은 적으며, 지연(latency)이 늘어나지만 전반적 품질 확보에 유리
- 반복 평가 및 피드백 구조를 통해 최종 출력물의 정합성을 높이는 구조

### LLM의 무한 리서치 루프(Orchestration Paradox) 문제와 임계점 관리 방안

- 최신 고성능 LLM들은 다수 방법 중 '최적의 방법'을 찾으려는 탐색에 집착, 문제를 해결하기보다 '방법 찾기'에 과도한 토큰 자원 소모
- 예시) Opus 모델 실전 적용 시 끊임없이 더 나은 솔루션을 찾아 루프에 빠짐
- Counter 메커니즘(4~5차 시도/5분 경과 등 임계점 도달 시 무조건 마지막 결과 사용) 등으로 이 무한루프를 차단한다

### 80/20 하이브리드 구조로 연구-결정 단계 분할 및 최적화

- 80% 동적 리서치: 자유 탐색, 창의적 문제해결에 최신 LLM 활용(Reasoning 모델)
- 20% 결정적 판정: 결과 요약·검증 등에는 단순 절차적 모델로 검증/요약(비딩적)
- 예시) 80%가 다양한 대안/탐색 진행, 20%는 Critic Node 등으로 최종 사용자 요구에 부합하는지 여부만 평가
- 사후 Counter 또는 Timeout 등으로 80% 영역의 무한 반복을 차단한다
- 이 방식은 성과 최적화와 비용 관리, 결과 신뢰성 제고에 효과적임

### 하나의 에이전트에 과도하게 많은 작업·컨텍스트를 부여하면 작업 손실 발생

- 큰 컨텍스트 윈도우를 이용해 "모든 업무를 한 에이전트에 몰아넣기"를 시도할 때, 입력이 넘쳐나면서 일부 작업(예: 4개 중 2개)만 잘 처리되고 나머지는 소실
- 역할별(이슈별) 전문 소규모 에이전트(Mixture of Experts) 체계가 더 나은 결과를 보임
- 각 미니 에이전트가 자기 분야(보안, 코드리뷰, 이슈 추적 등)에 특화된 분석·결과 제공

### 판정(판사) 에이전트 구조로 개별 결과를 통합·정합성 확보 가능

- 각기 다른 결과(예: 그리스 호텔 추천, 포르투갈 항공권 추천 등)가 상호 조합상 의미 없을 수 있음
- 판정(Judge) 에이전트가 여러 개별 결과를 종합·교차 검증해 사용자 목적에 부합하는 최종 결과 도출
- Qodo 실사례) PR 코드리뷰 단계에서 컨텍스트 수집기가 PR/툴 컨텍스트 통합 → 각 담당 에이전트로 분배 → 결과를 판정 에이전트가 종합·노이즈 제거

### 에이전트 간 통신·결과 공유: langchain 등 프레임워크 기반 프로세스

- 각 에이전트는 결과를 파일시스템 혹은 프롬프트 체인 형태로 다음 에이전트에 전달할 수 있음
- Qodo팀은 langchain 사용, 결과 수집 후 다시 프롬프트 생성 → 다음 에이전트에 입력하는 체계 운용
- 결과가 여러 개인 경우도 수집 에이전트가 통합·요약해 다음 작업에 연결

### 에이전트별 '캘리브레이션(보정)'과 조직 맞춤 피드백이 품질 결정

- 각 조직/개발팀의 특수성을 LLM이 스스로 알 수 없기 때문에, PR 이력·검토 내역·산업별 특화 가이드 제공 필요
- PR 히스토리: 과거 비슷한 변경점, 리뷰자가 조치한 케이스를 인덱싱하여 추천/가중치 부여
- 사용자/조직이 건넨 아키텍처 가이드, 컴플라이언스(예: ISO/SOC2) 정책, PR 히스토리 등 다양한 문서를 통해 컨텍스트 섬세하게 제공
- 룰(규정)로 제공되는 가이드라인은 강제 적용, 반복 미수용시 가중치 조정(학습 기반 피드백)
- 최종 판단 및 피드백 루프(에이전트 추천 → 실제 개발자 적용 여부 → 반영)로 품질 지속 개선

### 다양한 컨텍스트 입력원이 상호 보완적으로 활용되어야 진정한 맞춤형 에이전트가 탄생함

- PR 히스토리, 업무 가이드라인, 컴플라이언스 요구사항, 실사용자 피드백 등 복합적 맥락 입력의 중요성 강조
- 각 문서·규정·이력 정보 별도의 가중치 할당 및 유동적 적용으로 최종 결과물의 실무 적합성 확보
- 조직별로 문서 미제공 시 "범용 룰"만으로는 세밀한 실무 맞춤 추천에는 한계 인정
- 학습·추천·결과 조정 등 다양한 소스의 피드백 루프 설계가 진화형 에이전트 실무 적용의 핵심임
