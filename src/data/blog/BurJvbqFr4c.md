## *보안에 취약한 MCP 서버는 실제 서비스 환경에서 살아남을 수 없다* 핵심 요약

- 본 영상은 Lenses의 AI 엔지니어 팀인 Tun Shwe와 Jeremy Fronae가 진행하며, MCP(Machine-Consumable Protocol) 서버의 보안 문제와 실제 서비스(프로덕션) 환경에서의 생존 조건을 중심으로 다룸
- Lenses는 다양한 산업에 대규모로 적용되는 데이터 운영 패브릭 공급사로, 보안 및 거버넌스를 최우선으로 하는 대형 고객사를 보유하고 있음
- MCP 서버 설계 및 운영에서 사용자(인간)와 에이전트의 차이점이 크며, 부실한 설계는 곧 보안 취약점으로 이어진다고 강조
- 에이전트와 인간의 근본적인 상호작용 차이는 ‘발견(Discovery)’, ‘반복(Iteration)’, ‘문맥(Context)’이라는 세 가지 차원에서 드러나며, 각각이 고유의 보안 위협을 동반함
- MCP 서버의 보안 강화를 위한 5가지 핵심 원칙을 제시, 모든 원칙은 OAuth 같은 인증 구현에 앞서 반드시 고려되어야 함
- 표면적으로는 안정적인 로컬 개발 환경(standard IO 모드)에서 실제 조직 내 프로덕션 환경(HTTP 스트리밍 모드)으로 전환 시, 즉각적으로 복잡한 보안 이슈가 급증한다는 '보안 절벽(Security Cliff)' 문제를 지적
- 대부분의 MCP 서버가 긴 수명의 단일 API 키를 통해 인증하며, 이 구조는 대규모 사용자 환경에서 'confused deputy' 등 치명적인 보안 리스크를 초래
- OAuth 2.1, 동적 클라이언트 등록(DCR), CIMD(Client ID Metadata Document) 등 신규 표준을 활용한 현대적 접근법과 그 장점 및 한계를 상세하게 비교 분석
- 진정한 엔터프라이즈 수준의 MCP 보안을 위해 역할기반 접근 제어(RBAC), 데이터 마스킹, 로깅, 엔드 투 엔드 트레이싱 등 추가적인 조직적/기술적 조치의 필요성을 역설
- 영상 전반에 걸쳐 구체적인 인증 흐름, 공격 사례, RFC 번호 등 실전적 예시와 세부 지침을 제시함

## 세부 요약 - 주제별 정리

### 보안과 확장성을 동시에 요구하는 MCP 서버의 프로덕션 문제는 Lenses의 필드 경험에서 도출됨

- Tun Shwe와 Jeremy Fronae는 Lenses에서 AI 및 데이터 인프라 관련 엔지니어링 업무를 담당
- Lenses는 스트리밍 데이터 제공, 실시간 맥락(Context) 관리, 거버넌스와 보안 강화 등으로 다양한 대형 기업들의 실제 문제를 해결함
- MCP 서버의 오픈소스화 및 실전 경험을 통한 지속적 개선 중임을 언급하며, 사용자들에게 프로젝트 팔로우를 유도
- 본 세션의 주요 목적은 MCP 서버 또는 AI 에이전트용 인터페이스 설계에서 취약점을 사전에 방지하기 위한 구체적 사고법과 실천 지침 제공

### 에이전트와 인간 사용자는 API 사용 방식에서 근본적으로 다르며, 각 차원이 고유한 보안 위협을 내포함

- **발견(Discovery)**: 인간은 API 문서를 한 번 훑어보고 필요한 엔드포인트만 사용, 반면 에이전트는 연결할 때마다 모든 툴과 설명을 읽음
    - 매번 모든 툴 설명을 분석하므로 토큰 소비가 많고, 이 설명 자체가 ‘툴 포이즈닝(도구 설명에 은밀한 공격 코드 삽입)’의 공격면이 됨
    - OWASP MCP Top 10 중 ‘툴 포이즈닝’ 위협 직접 언급
- **반복(Iteration)**: 인간은 스크립트 실패 시 다시 실행하면 되지만, 에이전트는 매번 대화 이력 전체를 전송
    - 반복(리트라이) 시 이전에 출력된 민감 데이터까지 포함된 전체 기록이 노출될 수 있음 → 반복할수록 데이터 유출 위험 확대
- **문맥(Context)**: 인간은 광범위한 맥락을 기억, 에이전트는 약 20만 토큰 내에서만 맥락 구성
    - 문맥 과다 공급 시 'context injection', 'oversharing' 등으로 민감 데이터(PII, 내용, 계정정보 등)가 외부로 빠져나갈 위험
    - OWASP MCP Top 10의 ‘문맥 주입(Context Injection) 및 과공유’ 항목 참조 요구

### MCP 툴 및 정보 설계 및 노출 범위는 최소화해야 하며, ‘최소 공개, 최대 보안’ 원칙을 체계적으로 적용해야 함

- 모든 도구(툴)와 설명, 엔드포인트는 공격의 창구가 될 수 있으므로, 업무 목적에 맞는 최소 범위만 제공할 것
- 노출할수록 공격 가능한 면이 늘어나니, 가능한 한 ‘적게, 엄선하여’ 노출할 것을 제안(less is more)

### 안전한 MCP 서버를 위한 5가지 핵심 설계 원칙은 OAuth 등의 인증보다 우선 적용되어야 함

- 1) 결과 중심의 설계로 공격 표면을 줄임: 상세(세부) 기능은 묶어서 하나의 결과 중심 툴로 통합하여, 단일 권한 체크 및 로그만 관리하도록 함
    - 예: 주문 조회만 필요한 에이전트에게 사용자 삭제 등 불필요한 작업 공개 금지
- 2) 스키마(구조) 단위 입력값 제약: Enum/단순 dict 수준까지만 허용, Pydantic 등으로 타입 강제해 자유 입력과 중첩된 구조 금지
    - 임의 문자열/중첩 구조 허용 시 명령(프롬프트) 주입 취약점 대량 양산
- 3) 문서(Documentation)도 방어 목적의 레이어로 활용: 명확하고 완성도 높은 설명만 존재해야 하며, 인접 서버의 악성 설명문이 빈틈을 노릴 수 있음을 경계
- 4) 에이전트에 반환하는 정보는 당장 필요한 최소 데이터로 엄격하게 제한
    - PII, 내/외부 식별자, 자격증명 등은 필요없는 한 절대 반환하지 않음
- 5) 권한 범위를 ‘도구’ 및 ‘리소스’ 단위로 세분화해 세션 단위보다 촘촘히 통제
    - MCP read-only 등의 annotation 활용, 읽기만 필요시 MCP resource로 변환 권고
    - 불필요한 툴 제거=공격 벡터 제거

### 로컬 개발환경(standard IO 모드)은 안전하지만, 실제 서비스에선 보완 없는 상태로 전환 시 보안 위험이 급증함

- 로컬(standard IO) 환경: 단일 사용자, 인증 불필요, 안전한 샌드박스, 생산성↑, 단 멀티 유저/스케일 아웃은 불가
- 프로덕션(HTTP, 스트리밍): 다중 사용자, 네트워크 연결 필요, OAuth, 토큰 관리, TLS, Rate Limiting 등 복잡한 보안 설정 동시 필요(Gradual한 도입 불가)
- '보안 절벽(Security Cliff)' 현상: 로컬→프로덕션 이동 시 단숨에 해결해야 할 보안 문제가 대량 등장
- StackLock 사례: standard IO transport 환경에 20개의 동시 연결만 줘도 22개 중 20개 요청이 실패→확장성 심각 제한

### MCP 서버 인증 방식의 변화(긴 수명의 API 키부터 OAuth 2.1, DCR, CIMD까지)

- 과거/현행: 단일 API 키 방식(로컬 및 원격 서버 구분)
    - 키의 수명 길고, 조작 범위 넓으며, 거의 검증/회전되지 않음
    - 여러 시스템에 공유되며, API 혹은 MCP 서버가 제대로 검증하지 않는 경우 많음(confiused deputy, 통제 미비)
    - 전체 MCP 서버의 50% 이상이 여전히 이 구조 채택
- 현대식 전환: OAuth 2.1, DCR(Dynamic Client Registration), 토큰 교환(Token Exchange, RFC 8693)
    - 기존 OAuth의 "사전 등록된 클라이언트 ID" 패러다임은 MCP의 유동적인 다수 클라이언트/서버 구조에서 동작 불가
    - DCR: 각 클라이언트가 서버에 직접 등록하여 매번 신규 client_id 발급받음. 하지만 등록 이식성 불가, 피싱 공격 취약, Trust 문제 여전(누구든 등록 가능)
- RFC 번호와 관련 표준 문서(예: PKCE, RFC 8693 등) 준수 강조

### 동적 클라이언트 등록(DCR)의 장점과 한계를 분석하고, CIMD가 그 대안임을 구체적으로 소개

- DCR(동적 클라이언트 등록): MCP 서버가 각기 다른 툴(클라우드, 데스크탑, 커서, VS Code 등)로부터 동적 연결을 감당할 수 있게 해줌
    - 문제점: 여러 운영체제(Windows, macOS)에서 각각 등록 필요, 피싱 공격 취약, 메타데이터 신뢰 문제 발생
- CIMD(Client ID Metadata Document): 클라이언트 소유자가 공개 URL에 client_id 메타데이터 노출
    - 서버가 인가 시점에 직접 해당 URL에서 메타데이터 받아 진위 검증
    - 등록 DB 유지 불필요, client_id에 바인딩된 redirect_uri로 리플레이 공격 위험 감소
    - 2025년 11월부터 권고 표준으로 채택

### MCP의 엔터프라이즈급 보안을 위한 추가 요구사항(역할 기반 접근, 데이터 마스킹, 로깅, 추적)

- RBAC(역할 기반 접근 제어): OAuth scope 기반 권한 제어는 세션 단위에 불과, 실제론 도구/리소스 단위의 촘촘한 권한 관리 필요
- 데이터 마스킹: 이메일, 전화번호, 보험식별번호 등 민감정보는 에이전트가 읽기 전에 적절히 마스킹 처리 필요
- 로깅: 어떤 에이전트가 어떤 도구를 어떤 파라미터로, 어떤 데이터를 출력했는지 투명하게 기록
    - EU AI Act 등 각종 규제 대응과 감사 목적에 필수
- 엔드 투 엔드(End-to-End) 트레이싱: 요청 및 응답의 전 과정(클라이언트 요청→실행→데이터 수집→응답 생성)을 모두 추적 가능해야 조직 내 거버넌스 확보

### 프로덕션 환경 MCP 서버 구축에는 인증·인가·관찰(Observability)까지 조직적·기술적 전방위 적용이 필요함

- 단순 인증 외에 복수의 보안 및 감사 계층이 필수
- 위협 모델링과 실시간 감시(Observability)는 분산 시스템에서 사용하는 트레이싱 원칙을 에이전트형 AI에 적용해야 함
- 궁극적 목표는 MCP 서버-에이전트간 모든 상호작용의 ‘신뢰의 계단’을 구축하는 것임

### 영상 전반부와 후반부에 걸쳐 RFC, 와이어 플로우, 실전 구성도 등 실제 적용 예시와 기술적 레퍼런스가 다수 포함됨

- MCP 인증 플로우의 단계별 다이어그램, 토큰 발급 및 교환에 실제 HTTP 엔드포인트 명시
- 각 프로토콜 도입 방법, 취약점, 개선점, 실무상 발생하는 문제 및 해결법 등 구체적으로 제시
- OAuth, DCR, CIMD 각 방식의 장점과 취약점은 명확한 사례와 함께 비교됨

### MCP 서버 및 에이전트 인터페이스의 미래 방향성은 ‘안전한 설계’와 ‘신뢰 기반 연결’임을 강하게 시사함

- 영상 마무리에서 참석자에게 MCP 서버 프로젝트 팔로우 및 피드백을 요청
- 서버-에이전트 간 신뢰 계층 구축, 인터페이스와 설계의 동급 중요성, 보안 내재화를 지속적으로 헌신할 것을 강조함
