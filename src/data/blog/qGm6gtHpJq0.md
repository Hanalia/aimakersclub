## *Opus와 Kimi를 조합한 AI Dark Factory 실험 극한 도전* 핵심 요약

- "Dark Factory"란 사람이 코드 작성이나 리뷰에 한 줄도 관여하지 않고 AI가 ‘이슈 생성 → 코딩 → PR 리뷰 → 머지 → 배포’까지 모든 과정을 완전히 자동으로 처리하는 실험적 자체 진화 코드베이스임.
- 기존에는 수많은 토큰을 소모하는 대용량 LLM(예: Opus, GPT-4 등) 대신 Kimi K2.6이나 Minimax M2.7 같은 저렴한 모델을 "작업용"으로, 효율성 위주(비용/사용량 제한 회피)로 사용해왔음.
- 그러나 약점은 Opus 등 최상위 모델에 비해 완성도가 떨어지고, 마지막 PR 단계에서 항상 만족스럽지 못함.
- 오늘의 주요 도전 과제는 "Opus와 Kimi를 혼합하는 Archon 워크플로우"를 구축하여, 계획 단계는 Opus(깊은 추론/계획용), 그 이후 구현·검증은 Kimi(성능 효율화)로 나누어 토큰 사용 최적화와 결과 품질의 균형을 노림.
- Kimi K2.6은 Minimax에 비해 약간 더 우수하며(출력 품질 80~85/100 정도), Opus나 Sonnet 같은 플래그십에 미치지 못하나 비용 대비 우수함. 단, API 안정성은 다소 떨어짐.
- 실제 혼합 프로바이더 적용을 위해, 기존에는 환경변수로 Claude(Anthropic)를 Kimi로 라우팅하던 방식을 벗어나 Py(파이) 코딩 에이전트와 Archon Workflow의 노드별 모델 지정으로 대체.
- 워크플로우상 계획(plan)노드에만 Opus를 사용, 나머지 단계는 Kimi 할당. 이를 통해 주요 단계 성능과 전체 비용을 효과적으로 조율.
- 워크플로우 성능/안정성 확인을 위해 'smoke test' 등 다양한 실시간 리소스/토큰 상태 모니터링 및 실제 이슈 처리 데모.
- 지속적으로 발생하는 모델 요금제·API 사용 제한(Anthropic의 크레딧 월한 등)의 변화에 맞춰, 다양한 LLM·프로바이더 혼합 전략의 필요성을 강조.
- 앞으로 코드베이스 발전 방향: 노드별/단계별 최적 LLM 혼합의 벤치마킹, 다른 워크플로우(테스트/검증 단계 등)에도 확장 적용 예정.
- Dynamis 커뮤니티 및 chat.dynamist.ai AI 튜터 애플리케이션을 예시로 실험 결과를 실제로 즉시 확인 가능.

## 세부 요약 - 주제별 정리

### AI Dark Factory는 사람이 직접 개입하지 않는 완전 자율 코드 생산 공장을 실험함

- Dark Factory란 사람이 코드 한 줄도 직접 작성·검토하지 않는, 순수 AI만으로 코드베이스가 자기 진화하는 실험 프로젝트.
- 모든 이슈는 사용자가 제기하면 워크플로우가 자동으로 접수 순위/우선순위 결정을 하고, 각 이슈별로 Archon Workflow로 구현 진행.
- PR 생성 후에도 또다른 에이전트가 자동으로 검토 및 피드백 반영 → 최종 머지 및 프로덕션 자동 배포까지 중간에 사람 손길이 없음.
- 실험적으로 "Dark Factory Experiment"란 이름으로 실제 레포지토리로 운영하며, 실 사용 중 나타난 온갖 문제를 반복적으로 실시간 공개 시연함.
- 실용성 면에서는 현재(2024년 6월 기준) 인간 검수 없는 완전한 신뢰는 다소 무리, 실제 프로덕션엔 아직 한계가 있음을 인정.

### LLM API 사용량 제한과 비용 제약에 따라 "혼합 전략"은 필연적으로 대두됨

- 최근 Anthropic(Opus 등)·OpenAI 등 빅테크 LLM API들은 하드웨어 부족 등의 이유로 사용량(5시간/주간 제한 등)이 크게 강화됨.
- 저가 LLM(예: Kimi K2.6, Minimax M2.7 등)의 월 요금제(예: Kimi 4 Code Allegretto $40/월)로는 사용 제한에 거의 도달하지 않음. Anthropic은 자주 한계 도달.
- Opus 및 GPT 모델 사용의 토큰 단가가 높으므로, 탐색·계획·연구·실행 모든 단계에 플래그십 모델을 반복 호출시 현실적으로 비용 과다.
- 따라서 강력한 모델(예: Opus/GPT)은 복잡 추론(Planning) 목적만, 저렴한 모델(예: Kimy)은 반복적 작업(Implementation/Validation) 용도로 분업이 필요.

### 현재 Dark Factory의 워크플로우 구조와 LLM·프로바이더 변경 흐름 상세

- 메인 워크플로우 구성 요소:
    - 이슈 트리아지(접수/우선순위 조정)
    - 각 이슈별 Archon Workflow 실행(구현~PR생성~체크)
    - PR 검증용 별도 워크플로우(자율 검토/머지/배포)
    - 정기적 회귀테스트(앱 전체 기능 자동점검 및 신규 이슈 자동 생성)
- 워크플로우는 모두 엄청난 토큰 소모(폭넓은 검색/분석/문맥 유지) 특성.
- 과거에는 모든 노드를 하나의 모델/프로바이더(Kimi 또는 Minimax)를 글로벌 라우팅(환경변수로 Claude code→Kimi)을 통해 처리.
- 혼합 전략 실현을 위해 Py 코딩에이전트를 Archon 워크플로우와 결합, 노드별로 모델/프로바이더 지정(예: planning만 Opus/Claude, 나머진 Kimi/Pi)하는 방식으로 전환.

### LLM 혼합 전략의 영감: "계획·추론"에 Opus, "구현"에는 Kimi를 각각 최적 배치

- 두 가지 주류 혼합 전략 논의:
    - 전략①: 강한 모델(Opus)로 최적의 세부 계획 수립 → Kimi는 ‘따라서 구현’식 작업만 설계.
    - 전략②: Kimi 등 약한 모델로 대략 계획 후, 실제 구현은 Opus 등 강한 모델이 맡아 세부 오류 보정 구조.
- 저자는 ‘계획의 품질이 전체 결과를 크게 좌우한다’는 판단 하에 ‘planning 만 Opus, 나머지는 Kimi’ 방식을 채택.
- 실시간 실험을 통해 다른 혼합 구성이 실제 성능에 어떤 차이를 내는지 훗날 벤치마킹 예정.

### Py 코딩 에이전트와 Archon Workflow의 토폴로지/인증/키 관리는 혼합 구성의 핵심

- 기존 Claude 클라이언트 전체 라우팅(Claude->Kimi)은 혼합 환경(노드별 프로바이더)에서는 적용 불가.
- Py의 각 노드 별도 프로바이더 지정 기능을 활용: planning 노드만 Opus/Claude, 나머지 모두 Kimi/Py.
- VPS 환경 인증 관리:
    - Kimy API key를 Py provider용으로 별도 등록
    - Opus는 Anthropic 실구독 인증
    - 환경 변수는 글로벌 오버라이드 방식 대신 워크플로우 내 노드별 관리
- 인증·키 세팅 등 저수준 환경설정을 단계별 명확한 CLI 조작(SSH, ENV 파일 관리 등)으로 자동화.

### 실제 워크플로우 적용 과정에서 API 안정성·성능 이슈 및 비용/효율 체감

- Kimi API는 때때로 무응답/hang(에러 미발생, 그냥 대기 상태 지속) 문제. 대략 4회 경험.
- Minimax API는 Kimi보다 신뢰도 높으나, 나머지(Claude/OpenAI)는 드물게 발생.
- Anthropic(Claude)의 공식 가동률(uptime)은 98.66%로, 타 서비스(예: Supabase 99.9% 이상)에 비해 저조.
- 저가 LLM은 낮은 단가만큼 API가 덜 신뢰가능. 토큰 효율은 압도적이나 업무 연속성엔 리스크.
- Kimi K2.6은 Minimax M2.7보다 출력 품질이 더 낫지만 요금은 3배, 그만큼 가격대성능비 조율 필요.

### Archon 워크플로우 내 AI 상호 검증(Adversarial Judge) 및 검증 체계 이중화

- 이슈별 워크플로우 마지막에 “구현→1차 검증(컨텍스트 완전히 분리)”을 두어, 새로운 에이전트가 전체 과정을 몰라도 결과만 평가.
- 2차 단독 PR 검증 워크플로우에서는 GitHub 이슈와 최종 git diff만 주어지며, “매우 비판적으로” 리뷰할 것(편향 최소화)으로 프롬프트 설계.
- 검증 워크플로우 내 최대 반복 횟수 및 자동 머지, 강제 배포 절차 구축.
- 각 노드에 별도 에이전트 투입, 하위(서브) 에이전트의 리서치 후 요약 및 결과물 전달 구조.

### 실험용 파이프라인/컴포넌트 세팅 및 실제 이슈 생성·적용 전 과정 단계별 시연

- 로컬환경 IDE 및 VPS 연동, Py 기반 Archon 워크플로우 로컬 브랜치 생성·수정→VPS에 배포.
- 실제 'make the live tool activity indicator tool-aware and continuous' 등 프론트 실시간 피드백 개선 이슈를 생성, 워크플로우에 배치.
- 워크플로우상에서 Kimi가 리서치/분류/추출 단계, Opus가 계획 노드(Plan)에서 실질적으로 실행됨을 실시간 확인(UI 내 usage record, api call history 동시 체크).
- ‘smoke run’으로 혼합 프로바이더 실분기/토큰 추적, 정상동작 여부(계획→구현→검증 각 단계 API 호출 실시간 추적).

### 다양한 프로바이더(LMM, 번들), 환경 설정 및 비관습적 조합/서브모델 활용 방안 논의

- 현 워크플로우는 Kimi 및 Opus 혼합이지만, Minimax 등 추가 모델도 Py만 추가하면 간단히 통합 가능.
- 일부 구간은 아예 로컬 LMM(예: Qwen 3.6, 4090 GPU 활용)로 서브에이전트 대체 가능, 파일 리서치 등은 최적.
- 클라우드 기반 GPU 인스턴스(예: RunPod, Google Colab) 단가 계산 및 실질 비효율 설명
    - 4090 기준 GPU 서버 24/7 상시 가동시 월 $500 이상(69¢/h×24×30).
- 서브에이전트는 리서치/탐색용에 국한, 실제 코드 변경에는 요약된 결과만 전달하고 본체가 직접 PR 생성(정합성 문제 방지).

### Dark Factory 인프라 구조와 "세컨드 브레인"의 클라우드 운영 통합 실전 팁

- "세컨드 브레인" 에이전트를 VPS에도 상주시켜 Slack 등으로 연동, 로컬·클라우드 파일 시스템처럼 명령 일체화 가능(SSH, 각종 커맨드 자동생성/실행).
- 환경 인증 등 민감정보는 SSAH 및 직접 조작(프롬프트 기록 보안상 차단).
- Coding Agent가 실시간 클라우드 머신의 유저·IP·키 정보, 툴 설치현황, 상호연계 능동적으로 기억 및 활용 가능.
- Archon 워크플로우의 세부 경험/최적팁은 문서화(가이드/베스트프랙티스) 확장이 필요함을 언급.

### 혼합 워크플로우 확대/벤치마크 및 미래 방향성 논의

- 각 단계별로 어떤 모델(계획용/구현용/검증용)을 쓸 때 실제 결과 품질이 더 좋은지, 동일 이슈군에 병렬 테스트하여 평가하는 벤치마크 실험 예고.
- 미래에는 구현형 AI 워크플로우가 한 모델에 올인하는 시대는 끝, 하위/중간/상위 모델 혼합이 표준이 될 것이라 전망.
- 테스트·PR 검증·회귀테스트 등 다른 워크플로우로 혼합 구조 확장, 노드별 동적 모델 스위치 전략 실무 적용 준비.

### Dynamis 커뮤니티/AI 튜터 애플리케이션과 실험 피드백 즉시 체험 가능

- 참가자 누구나 chat.dynamist.ai 접속해 Dark Factory에서 진화된 에이전트 직접 사용 가능(무료, 단 일부 기능은 Dynamis 회원 한정).
- 애플리케이션은 Gemini 3 Flash 기반 LLM이 백엔드, Dynamis 커뮤니티·YouTube 전체 콘텐츠 RAG 파이프라인 검색/응답까지 실시간 지원.
- 실험 결과 반영·즉시 업데이트 검증 및 사용자 체감, 실제 제품용 예시로 활용가능.

### LLM 기반 Speech-to-Text(음성인식) 활용법·도구 추천 및 운영 팁

- AquaVoice(1년 이상 사용) 등 Speech-to-Text 도구로 장문의 생각·계획을 말로 효율적 이식, Cloud Code 등 다양한 에이전트 워크플로우와 연동.
- Type Whisper·Spokenly·WhisperFlow 등 대안 소개, 각자 활용도에 따라 선택 권장.
- Speech-to-text는 코딩·AI 워크플로우 외 일상 생각 정리에 매우 유용, 입력 속도가 타이핑보다 월등히 빠름.
- AquaVoice 딥컨텍스트 등 고급 기능은 현황인식 보조, 단, 실시간 하이라이팅 등은 미지원.

### 요금제·프로바이더 정책 변화(Anthropic의 변경 등) 및 운영 환경 전략

- 2024년 6월 15일부터 Anthropic 클라우드 구독 사용 변화 예정(Archon 등에서 직접 활용 불가, $200 마일리지 소진 후 차단).
- 이에 따라 좀 더 지속적인 사용 위해 Codex 등 대체 LLM 또는 Py+Kimi 같은 조합 실험 강화.
- 다양한 서브모델·웨이스트(교대/백업/벤치마킹 등) 포함 정책 빠른 적응 필요성.
