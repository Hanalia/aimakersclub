## *로컬 코드 인덱스를 통해 AI 코딩 토큰 94%를 절감한 방법* 핵심 요약

- 라즈와 파트너 포스는 AI 코딩 툴(Copilot, Cursor, Cloud Code 등) 사용 중 예상치 못한 월별 요금 급증을 경험함
- 요금 폭증의 핵심 원인이 '불필요하게 방대한 입력(context) 전송'임을 발견함: 필요 없는 코드 파일과 라인까지 전송돼 비용 발생
- 단건 쿼리에 약 45,000 토큰이 사용됐지만 실제 중요한 부분은 5,000 토큰에 불과했고, 나머지 40,000 토큰이 낭비됨
- 입력(Input)이 전체 AI 활용 비용의 90%를 차지하고, 출력(Output)은 약 10%만 차지함이 분석을 통해 확인됨
- 출력 길이를 줄여도 비용 절감 효과는 미미하며(최대 8% 이내), 대대적 비용 절감은 입력 토큰 절감에서 옴
- 이를 해결하기 위해 자신들만의 로컬 코드 검색·인덱스 레이어를 구축하여 사용
- 이 시스템은 코드 파일을 의미 단위(함수·클래스)로 쪼개고, 의미 기반 및 키워드 기반 이중 검색 방식으로 관련 코드만 추려냄
- 결과적으로 FastAPI 오픈소스 프로젝트에서 쿼리당 토큰 사용량을 약 94% 절감(83,000→4,900), 추가 압축 시 523 토큰까지 줄임
- 정확도 역시 90% 이상 유지했고, 1주일간 실험 시 1,240만 토큰(약 186달러) 절약, 절감량의 84%는 검색 레이어에서 발생
- 전체 시스템은 로컬에서 구동되고, 다양한 코딩 툴들과 공유 인덱스 및 컨텍스트 메모리 저장 기능을 지원함

## 세부 요약 - 주제별 정리

### AI 코딩 툴 사용 중 요금 폭증의 원인은 불필요한 입력 전송임을 발견함

- 라즈와 포스는 AI 코딩 툴을 일상적으로 사용하며, 월별 청구서의 급격한 변동을 경험함
- 프로젝트 및 사용 툴 자체는 동일했으나 작업량이 많아지면서 요금의 대부분이 예상 밖으로 증가함
- 조사 결과 AI의 '생각' 비용이 아니라, 툴이 너무 많은 코드 컨텍스트(불필요한 코드 파일 및 라인)를 송신하는 비용이 대부분임을 파악함
- 실제로 쿼리당 평균 약 45,000 토큰이 컨텍스트 전송에 사용되고, 실제로 모델이 필요한 코드는 약 5,000 토큰이었음
- 불필요한 40,000 토큰 비용을 매 쿼리마다 지불하는 셈이었음
- 이를 '피자를 주문할 때, 한 조각만 먹고 나머지 9판을 매번 돈 주고 버리는 것'에 비유함

### 입력 토큰 최적화가 AI 비용 절감의 핵심임을 수치로 증명함

- 90% 이상의 AI 비용이 입력(Input) 데이터(컨텍스트, 코드 파일 등) 송신에서 발생함
- 출력(Output, 즉 AI가 생성해 주는 코드 답변)는 전체 비용의 약 10%에 불과함
- 출력 압축(짧게 답변 생성)은 최대 75%까지 감축 가능하지만, 전체 비용 절감 효과는 8% 수준에 그침
- 반대로 입력 컨텍스트를 94% 감축했을 때는 전체 비용의 61%까지 절감했음을 수치로 명확히 제시함
- 결론적으로, AI 비용 절감의 핵심은 출력이 아닌 '입력 최적화'임을 강조함

### 출력 프롬프트, 모델 세팅 변경, 출력 압축 등 기존 접근법의 한계를 경험함

- 라즈와 포스는 처음에 프롬프트 짧게 쓰기, 관련 코드만 보여주기 등 간단한 프롬프트 최적화를 시도함
- 그러나 이미 대량의 불필요한 토큰이 입력 송신된 후라 실제 비용 절감 효과는 없음
- 모델 설정(예: max token, temperature 등) 변경 역시 출력에만 영향, 비용의 대부분인 입력에는 영향 없음
- 출력 압축(짧은 답변 요청)은 어느 정도 효과는 있었으나, 출력 원천 자체가 비용의 10%라 의미 있는 절감이 어려웠음

### 로컬 코드 인덱스와 검색 계층으로 입력 토큰을 획기적으로 줄임

- 자체적으로 '로컬 코드 검색 레이어'를 개발하여 코드베이스와 AI 툴 사이에 삽입함
- 더 이상 전체 파일을 AI에 넘기지 않고, 실제로 필요한 '적은 양'의 코드만 넘기도록 설계함
- 5단계로 동작:
    1. 코드 전체를 읽어 함수, 클래스 등 '의미 단위'로 쪼갬(임의적 분할이 아니라 논리적 구조 기준)
    2. 의미 기반(semantic) 검색과 키워드(lexical) 기반 검색을 동시에 시행, 결과를 결합
    3. 결과 코드에서도 핵심 정보(함수명, 설명 등)만 남겨 추가 압축(예: 50줄 함수를 5줄로)
    4. 서로 호출-연결되는 함수, 클래스 등을 추적해 관련 코드 블록 자동 반영
    5. 각 검색 결과에 '적합도 점수'를 부여, 임계점 이하 결과는 송신하지 않음
- 전 과정이 완전히 로컬에서 구동되며, 클라우드로 코드가 송신되지 않음을 강조

### 의미 기반과 키워드 기반 이중 검색 병행을 통해 검색 정확성과 효율을 높임

- 의미 기반 검색은 아이디어·기능적으로 연관된 코드를 잘 찾지만, 함수나 변수의 정확한 이름 일치에는 약함
- 키워드 기반 검색은 정확한 이름에 강점이 있으나, 유사기능(예: "login flow"와 "sign in")은 놓치는 약점이 있음
- 두 방식을 병행하자, 각각 25%(1/4) 정도 결과를 놓치던 약점이 합쳐져 놓치는 비율이 10%(1/10) 이내로 줄어듦
- 각각의 접근이 서로 상대의 약점을 효과적으로 보완함
- 검색 결과의 품질이 실제로 '정확한 관련 코드를 잘 찾는지'가 핵심임을 강조함

### 검색 결과 품질 향상을 위한 단순한 점수 공식 세팅이 효과적임을 확인함

- 검색된 코드 결과 항목이 반드시 '실제로 관련 있음'을 보장해야 함
- 초기에는 AI에게 자가 심사(judge its own results)를 시켰으나, 응답에 2~3초 대기 시간이 추가될 만큼 느렸음
- 점수 임계치(threshold) 조정을 단순히 고정시키면, 짧은 질문에는 불리하게 작동하는 등 한계가 있음
- 최종적으로 '간단한 가중치 공식'으로 품질을 계량화: 의미 일치 점수 50%, 키워드 일치 30%, 코드의 최근 변경도 20%
- 이 공식은 쿼리마다 동적으로 임계점을 조정할 수 있고, 0.4밀리초 내에 계산되어 AI 별도 호출이 필요 없었음
- 실험 결과, 복잡한 모델보다 단순한 수식 기반 접근이 더 빠르고 실질적 효용성이 높았음

### 실제 오픈소스 프로젝트 테스트에서 94% 토큰 절감과 90% 정확도를 달성함

- 실험 프로젝트: 오픈소스 FastAPI (53개 파일 사용)
- 20개의 실제 개발자 질문(예시) 기준 성능 측정:
    - 기존(최적화 미적용) 쿼리당 평균 83,000토큰 발생
    - 자체 도구 적용 시 쿼리당 4,900 토큰(94% 절감)
    - 추가 압축까지 더하면 523 토큰까지 감소
- 검색 결과의 '정확하게 원하는 코드 찾기' 성공률 90% 기록
- 해당 테스트 케이스 및 명령어를 공개, 누구든 직접 재현 가능함을 밝힘

### 대형 코딩 프로젝트에서는 입력 절감 효과 및 품질이 일부 한계에 부딪힘

- 파일이 많고(396개) 복잡도가 높은 대형 코드베이스에서는 recall(검색 정확도)이 급격히 저하됨
- 각 파일이 한 가지 목적일 때는 시스템이 잘 작동하지만, 여러 기능이 섞인 복잡한 파일 구조에서는 효과가 떨어짐
- 인덱싱과 탐색용 모델은 속도를 중시해 소형·경량 모델을 채택: 재인덱싱에 1초도 걸리지 않음
- 더 큰(고성능) 모델을 쓰면 recall은 늘 수 있으나, 현재는 속도를 위해 단순하고 빠른 방식을 택함
- 인프라나 복잡한 설계 대신, 단순 로컬 기반·작은 데이터베이스·2중 검색 같은 단순 전략이 더 현실적임을 경험함

### 여러 AI 코딩 툴 간 인덱스 공유와 메모리 일원화로 맥락 손실을 해소함

- Cloud code, Cursor, Copilot 등 여러 AI 코딩 툴을 병용할 경우, 모든 툴이 매번 별개로 코드를 파악해야 하는 비효율 발생
- 본 시스템은 하나의 '공유 인덱스'를 구축해, 모든 코딩 툴이 동일한 검색결과와 컨텍스트를 활용 가능하게 만듦
- "한 번 코드베이스를 설명하면, 그 지식이 모든 툴에 남아 있음"을 강조(컨텍스트가 다음 세션·다른 툴로에도 전달됨)
- 프로젝트 정보를 한번만 설명하면 모든 툴이 기억, 매번 반복 설명 필요 없음

### 실전 도입에서 1240만 토큰, 약 186달러 비용을 1주일 만에 절약함

- 실험 기준: 실제 프로젝트에서 247건 쿼리 수행, 1240만 토큰(약 186달러) 미사용
- 절감액 중 84%가 검색 레이어에서, 16%가 데이터 압축(출력 단축)에서 발생함
- 직접 비용절감량을 측정할 수 있도록, 도구는 '미전송 토큰 수'와 '모델 단가'를 곱해 실시간 리포트 제공
- 1주일간 본인 프로젝트에 적용해, 각자 비용절감 실적 확인 가능하다고 안내함

### 인풋 최적화가 모델 선택보다 더 근본적인 AI 비용 및 효율 전략임을 강조함

- Opus, Sonnet 등 어떤 AI 모델이 더 저렴·효과적이냐는 논쟁보다, 전체 비용의 70%가량은 입력 컨텍스트에 결정됨을 강조
- 'Better model'이 아니라 'Better input'이 진정한 비용 절감과 성능 핵심임을 주장함
- 시스템은 CCE라는 오픈소스 프로젝트로 무료 공개되어 있음(명령어 안내와 코드 링크 제공)
- 누구든 자신의 프로젝트에서 곧장 적용 및 비용 절감 효과를 검증 가능함
