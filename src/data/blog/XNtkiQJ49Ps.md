## *에이전트는 단순 채팅만으로 충분하지 않다* 핵심 요약

- 영상에서는 **복잡한 작업을 수행하는 에이전트는 단순 '채팅' 인터페이스만으로 한계가 있으며, 보다 고차원적이고 지속 가능한 협업 도구가 필요함**을 강조함
- 연설자 Jacob은 레고라(Legora) CTO로서 법률 분야를 위한 협업 AI 워크스페이스 기업을 운영 중임을 소개하며, **법률·코딩 등 다양한 분야의 복잡한 업무에서 에이전트 적용 사례**를 설명함
- 과거에는 '작업 그 자체'가 주요 과제였지만, 현재는 **작업의 계획·검토 등 '비기능 요구사항'이 큰 병목**이 되고 있음을 언급함
- Jason의 'Verifier's Rule' 개념을 인용, **문제가 쉽게 검증 가능한 일은 AI가 신속히 자동화 가능**하다고 설명
- 법률 등 복잡한 검증이 어려운 분야에서는 **기존 우수 사례와 비교(유사도 검사), 작업 분해, 가드레일 설정 등**을 통해 신뢰와 통제력을 높여야 함을 구체적으로 제시함
- **에이전트와 인간의 협업기반 효율성을 위해, 컨트롤(통제)과 트러스트(신뢰) 모두 향상 필요**하며, 각종 기술(스킬화, 의사결정 기록, 역할 분리 등) 적용 방안을 제시함
- '채팅' 인터페이스의 일차원적 한계를 지적하며, **문서·표 같은 '고대역폭' 지속적 협업 인터페이스의 필요성**을 강조함
- 실제 레고라의 제품 내에서 **하이라이트, 코멘트, 부분별 역할 할당, 테이블 리뷰**와 같이 구조적이고 직관적인 인터페이스를 구현한 사례를 소개함
- **채팅은 유연한 입력엔 좋은 인터페이스지만, 복잡한 에이전트와의 시스템적 협업에는 한계가 큼**을 결론적으로 주장함

## 세부 요약 - 주제별 정리

### 복잡한 에이전트는 단순한 채팅 인터페이스로는 실무적 한계에 부딪힘

- Jacob은 복잡한 에이전트로 계약서 초안 같은 과제를 처리해본 사례를 시작으로 설명함
- 에이전트는 다양한 서브 에이전트 실행, 웹검색, 파일 작성 등 긴 처리과정을 거침
- 작업 결과가 나오더라도, 특정 조항(예: 3조)이 잘못된 경우 피드백을 전달하면 '컴팩션' 단계(메모리 압축 등)에서 맥락 손실 발생
- 결과적으로 반복 수정 및 전체 맥락 오염·변화가 발생, 실무에서 '안 좋은 경험'을 초래함
- 이런 한계는 단일 채팅 기반 작업 흐름(1차원적 인터페이스) 때문임을 시사

### 레고라는 법률 분야에 특화된 협업형 AI 에이전트 기업으로 빠르게 성장 중임

- Jacob 자기소개: CTO, Legora(법률 시장용 수직 AI(Vertical AI) 기업) 리딩
- 1,000명 이상 고객, 50개 이상 시장 진출, 괄목할만한 속도의 기업 성장
- 최근 대규모 투자를 유치했고, 성장 속도는 “역사상 가장 빠른 수준”이라고 언급
- 런던에 엔지니어 채용 중이라는 메시지도 전달

### 기존에는 실제 '작업'이 어려웠으나, 오늘날은 계획·검토(Planning/Reviewing)가 병목으로 작용

- 몇 년 전까지만 해도 End-to-End 작업 수행의 난점은 '실제 수행 그 자체'였음
- 현재는 LLM 등 AI로 '작업 실행'은 매우 저렴·손쉬워짐
- 반대로 **계획 세우기, 요구사항 정의, 작업 결과 검토** 등의 '부수 프로세스'가 오히려 시간·비용의 병목이 됨
- 개발자들이 GitHub에서 대형 PR 리뷰 시 겪는 '고통' 사례와 유사
- 일부 AI 열성파는 에이전트끼리 자가 검토를 시도하지만, 실효성은 불분명

### 검증(Verification) 난이도에 따라 AI 자동화가 결정된다는 'Verifier's Rule'이 중요함

- Jason이 제안한 'Verifier's Rule' : “과업이 솔루블(해결 가능)하고 검증이 쉬우면, AI가 자동화한다”는 원칙
- 검증이 용이하면 RL 환경·후학습 등을 통해 문제를 효율적으로 해결
- 업무는 '계획-실행-검토' 세 단계로 나뉘며, 각 단계마다 검증 난이도/자동화 수준이 다름
- 예시: 계약서 내 정의어 확인은 검증도 쉽고 AI 솔루션 적용도 쉬움

### 법률, 코드 등 각 산업은 검증 가능성과 자동화 가능성에서 각기 다른 위치를 차지함

- 법률: 계약서 작성(작성은 쉽지만 법정에서 검증되어야 하므로 검증 불가, 자동화 난이도 높음), 소송 전략(객관적 정답 어려워 완전 자동화 불가)
- 예시로, 변호사 5명에게 소송전략을 물으면 5가지 답이 나옴: "객관적 진실 없어 검증 거의 불가"
- 소프트웨어 역시 일부(테스트 작성 등)는 자동화 쉬우나, 예를 들어 성공적인 소비자 앱 구현 방식은 정답·검증이 어려움
- 이런 배경에서, **인간·에이전트 협업의 필요성**이 강조됨

### 인간과 에이전트의 협업 효율성 확보는 '통제(컨트롤)'와 '신뢰(트러스트)' 향상이 핵심임

- 통제: 인간이 에이전트의 작업에 노하우를 얼마나 투입/주도할 수 있나의 문제
    - 예: 통제 낮으면 에이전트의 모든 행동 로그(trace)를 일일이 점검해야 함
    - 신뢰(trust)가 높아지면 '검토 필요성'이 줄어듦
- 신뢰를 높이기 위한 방법:
    - 작업 검증 난이도를 낮춤(예: 브라우저 접근권한, 테스트 주도 개발 등)
    - 법률 분야에서는 기존 '골든 계약서'와의 유사성 평가로 검증가능성 보조
    - 작업 분해(예: risk profile 선택은 인간, 정의어 linting 등은 에이전트 담당)
    - 가드레일(Guardrails): 에이전트의 작업 파일·폴더·웹사이트 접근을 제한해 예측 불가능한 행동 억제

### 작업 분해, 스킬화, 가드레일 등 구체적 기법으로 복잡 업무 자동화를 지원함

- 계약작성 예시: 리스크 프로파일링, 사례문서 선택 등 주요 의사결정은 인간 주도
- 정의어 확인, 문서 포맷 적용 등은 에이전트 자동화 가능
- Guardrail 적용 시: 지정된 파일·폴더(예: Claude Code에서 한정된 파일만 수정 권한 부여)로 에이전트 행동 제한
- 신뢰가 낮으면 어떤 행동도 사용자 승인 필요, 신뢰가 높으면 자유롭게 작업 진행(“YOLO 모드”)

### 작업 구조 시각화 및 계획·등급 구성을 통해 통제력 강화 필요

- 복잡 업무는 트리 구조(DAG: Directed Acyclic Graph)로 분해 가능
- 예: 노동계약서 검토 보고서 생성시 '회사 리서치 → 계약별 항목 검토 → 리포트 작성'의 트리
- 단, '루트 노드' 단위에서만 인간이 개입 가능하면 통제력 취약
- 통제 강화를 위한 방법: 가이드라인/스텝별 명시적 계획(Plan) 공유 및 승인
- 계획 단계에서는 인간이 원하는 '리뷰 항목/조항' 등을 미리 지정해 통제력 향상, 단 작업 규모 커지면 비효율 발생

### 스킬 기반 작업 진행 및 의사결정 기록(Decision Log)으로 예외상황 대처 및 효율적 협업 달성

- 'Skill'은 각 작업 노드별로 인간의 판단/노하우(예: 특정 조항 EU법 적용법) 구현 가능
- 일정 스킬을 기반으로 예외상황 등 다양한 변수·특수 케이스(Contingency) 처리 가능
- 단, 모든 케이스에 스킬 구성은 현실적으로 한계 있음
- 이 때 'Elicitation' 방식으로 에이전트가 모르는 상황마다 사용자에게 질의해서 의사결정 요청
- 작업 블로킹 방지 위해 임의 판단 후 '결정 로그'에 기록, 추후 리뷰·뒤집기 가능
- 업무 트리 규모가 커질수록 채팅 인터페이스는 비효율(수십~수백 가지 질문에 답해야 하는 식), 맥락 부재로 어렵다는 문제 지적

### 고대역폭·구조화된 인터페이스(문서·테이블 등)가 복잡 에이전트 협업에 최적임

- 효율적인 인간-에이전트 협업 UI로 **문서, 테이블 등 내구성(내재적 기록성)이 강한 인터페이스** 제시
- 실제 레고라 제품에서는, 사용자가 문서의 특정 조항(예: 3조)만 하이라이트/수정, 코멘트 기능, 에이전트/협업자 태깅, 부분별 에이전트 역할 분배 가능
- 'Tabular review(테이블 리뷰)' 인터페이스: 복수 계약서 리뷰 결과를 표 형태로 각 조항별 에이전트 검토 및 사용자 피드백 요청
- 사용자 입장에서 전체 작업 추적, 핵심 이슈 확인 및 판단 투입이 매우 용이해짐
- 리뷰 완료 후에는 후속 작업을 에이전트에 위임 가능

### 최근 주요 UI 경향은 채팅+구조적 인터페이스의 혼합 및 구조적 협업 도구로의 전환임

- 최신 실무에서는 선형·사후적 리뷰 UI와 다양한 실험이 활발히 진행 중
- Legora에서도 2주 내 새로운 UI 출시 예정이라고 언급
- 입력 방식 측면에서는 채팅 상자는 여전히 유연성·확장성 장점 있음(‘언어는 범용 인터페이스’)
- 그러나, 복잡 에이전트와의 생산적 협업은 **언어(채팅)만으로 한계**가 있음

### 에이전트는 인간과 달리 '언어'에만 한정되지 않고, 더 구조적이고 시각적인 협업이 가능함

- 인간은 주로 언어를 통해 협업하지만, 이는 정보전달 대역폭·맥락 전달 한계 존재
- 예: 조직도 설명을 언어로만 해야 하며, 직접 시각화 및 상호작용이 어려움
- 에이전트(및 협업 시스템)는 인간과 달리, 시각적·구조적 정보 활용에 제한 없음
- 따라서 **인간 수준의 언어 기반 협업을 넘어서는 새로운 협업 도구·인터페이스 필요성**을 역설하며 마무리
