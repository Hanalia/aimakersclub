## *원칙 중심의 에이전틱 엔지니어가 되는 완벽 가이드: AI를 활용해 무엇이든 개발하는 법* 핵심 요약

- 영상 제목은 “원칙 중심의 에이전틱 엔지니어가 되는 완벽 가이드: AI를 활용해 무엇이든 개발하는 법”임
- 호스트 Cole Medin이 AI 전문가 Leor Weinstein과 최근 라이브 워크샵에서 진행한 실전 강연을 정제해 제공함
- 목표는 복잡한 AI 코딩 프레임워크 대신, 실용적이면서도 기본에 충실한 3단계 시스템(아이디어화, 반복적 실행, 시스템 진화)을 소개하는 것
- “개발자는 직접 코드를 작성하기보다 기획·검증 등 레버리지가 높은 역할에 집중”해야 함을 강조함
- 예시는 Claude Code, Jira, 오픈소스 GitHub 리포지토리, 프롬프트 명령어 시스템 활용법 등 구체적 실습을 포함
- 초반 ‘아이디어화’ 단계에서는 대화를 통한 브레인스토밍→질문·명세 구체화→AI 룰/커맨드/스킬 파일화로 이어짐
- 두 번째 ‘PIV 루프’에서는 각 티켓(예: Jira 티켓, GitHub 이슈)의 개발을 반복적이고 체계적으로 처리하는 방법을 안내함
- 구현-계획-검증 과정에서 “팁: 반복 프롬프트는 반드시 커맨드/스킬로 템플릿화” 등 실용적 패턴을 소개함
- 마지막 ‘시스템 진화’ 단계에서는 반복 과정 중 발견된 문제를 룰·워크플로우·커맨드 개선으로 발전시키는 실시간 피드백 구조를 강조
- 모든 과정에 YAML/JSON 설정, 문서화, 자동화 도구 등의 실제 사례와 상세 절차·질문 지침이 포함되어 실무 적용성이 높음

## 세부 요약 - 주제별 정리

### AI 코딩은 복잡성을 피하고 본질에 집중해야 실질적 성과를 얻을 수 있음

- AI 코딩을 위한 툴과 프레임워크가 다양하지만, 대부분은 과도하게 복잡하게 설계되어 팀의 개발 프로세스에 쉽게 녹이기 힘듦
- 상용 및 오픈소스 프레임워크(예: BMAD, Cloudflow, GSD 등)는 고유한 ‘의견’이 많아 내부 프로세스에 맞추기 어렵고, 기존 개발 관행을 완전히 버릴 것을 요구함
- Cole은 최소한의 구조와 규칙을 바탕으로, 각 팀의 상황과 선호에 따라 확장·커스터마이즈할 수 있는 ‘단순하면서도 강력한’ 워크플로우를 제안
- AI 코딩 보조는 도구(GitHub Copilot, Claude Code 등)와 작업 관리툴(Jira, Linear, GitHub 등)을 갖추면 어떤 환경에서도 적용 가능함
- 반복되는 프롬프트/커맨드는 반드시 템플릿화하여 재사용성·정확성·표준화를 확보해야 한다고 강조

### 개발자와 프로덕트 매니저 모두를 위한 3단계 AI 코딩 원칙을 제시함

- 개발자는 이제 ‘코드 작성자’가 아니라, ‘계획’과 ‘검증’이라는 고차원적 임무 수행자로 역할이 이동함
- 전체 프로세스는 다음 3단계로 구분됨:
    - 1. **아이디어화(ideulation)**: 빌드하고 싶은 내용을 AI 에이전트와 대화하며 무작정 떠올리고, 질의응답을 통해 요구 명세를 구체적으로 작성
    - 2. **PIV 루프**: 각 작업 티켓(Jira, GitHub 이슈 등)의 실제 개발 과정을 반복·자동화하는 핵심 개발 루프
    - 3. **시스템 진화**: 반복적 개발 과정(PIV)에서 발견된 문제와 개선점을 룰/커맨드/워크플로우에 반영하며, AI 조력 시스템 자체를 발전시킴
- 프로덕트 매니저 역시 Sprint·김릴드 관리, PRD 작성 등 초반 업무에서 AI 활용이 중요하며, 워크숍 참가자의 팀 전체(PM 포함)를 교육시킴

### 아이디어화 단계에서는 AI와의 ‘브레인스톰+질문-답변’이 핵심임

- 개발 또는 새로운 기능을 착수할 때, AI 에이전트(예: Claude Code)와 자유롭게 대화하며 완전히 비구조적인 브레인스톰을 시작
- 현재 애플리케이션 상태, 수정/추가할 기능, 고치고 싶은 버그 등 구체적으로 설명하도록 유도
- 가장 중요한 점은 AI 에이전트가 사용자를 대상으로 “clarifying questions” 즉, 명확화 질문을 하도록 적극적으로 요청한다는 것
    - Claude Code에서는 Ask User Question 기능을 지원(멀티초이스), 다른 에이전트도 채팅 방식으로 대체 가능
- 이 프로세스는 20~30분 이상 질문·답변 반복이 가능하며, 요구 정의에서 잘못된 AI 가정이 생기는 것을 방지하는 데 필수
- PM, 개발자 모두 이 단계에서 구체적이고 명확한 요구명세 도출이 중요하며, 테스트·구현 단계의 상세 내용은 아직 다루지 않음

### 명확화된 요구를 기반으로 ‘AI 레이어’(커맨드/룰/스킬)를 구축해야 업무 자동화가 가능함

- PRD(제품 요구사항 문서), planning 등 반복적 워크플로우는 ‘명령어(Command)/스킬(Skill)/글로벌 룰(Global rule)’로 코드화
- 예: /plan, /create PRD, /create stories 등 명령어를 사용해 비정형 대화→명확한 명세→Jira 티켓 자동 분할 구현 가능
- 룰 예시: 코딩 스타일, 테스트 전략, 로깅 전략 등 팀의 모든 규칙을 AI가 반드시 따르도록 실체화
- 명령어와 스킬은 인자(argument) 전달을 지원하여, 프로젝트/에픽/문서명 등 실제 현장 조건에 맞게 동작 유도 가능
- “3번 이상 반복하는 프롬프트는 반드시 커맨드/스킬로 만들어라”는 실전 조언 제시

### 실제 PRD 및 티켓 생성 과정에서는 ‘명령어 기반 자동화’를 지향함

- AI 대화의 최종 산물(대화 내역, 요구 제안 등)은 /create PRD 커맨드 실행으로 완전한 PRD 마크다운 문서로 생성 가능
    - Executive Summary, Mission, Target Users, 범위 등 템플릿 구조를 갖춤
- PRD 산출 후 사람이 반드시 검토(HITL, human in the loop), 수정/보완 권장
- /create stories 커맨드: 완성된 PRD에서 개별 작업(스토리) 추출→Markdown 및 Jira 티켓으로 자동 생성
    - Jira MCP 서버를 통한 티켓/에픽 자동 생성, 자동 할당 및 추가 기술 노트/코멘트까지 지원
    - GitHub, Linear 등 타 작업툴도 동일 패턴/명령어로 대체 적용 가능
- 개발자는 생성된 티켓에서 작업을 선택, 다음 루프(구현 루프)로 진행

### PIV 루프는 각 티켓별 상세 계획 수립과 반복 개발의 자동화 구조임

- 티켓(혹은 이슈)을 선택하면 Layer1(전체 계획)-Layer2(작업별 세부계획)로 진행
    - Layer1: Sprint 전반의 큰 방향(기능/버그 우선순위) 결정, 기술 상세는 논의 안함
    - Layer2: 각 티켓에 맞춰 코드베이스 분석, 활용 가능한 문서/구성요소 탐색, 구체적 작업 개요 도입
- /prime 커맨드: 코드베이스와 이슈 연동, 최근 git 커밋 분석, 핵심 파일 자동 탐색, Jira 정보 연동 등으로 세션 초기화
    - 과도한 컨텍스트 로딩 방지(필요 내용만 로드) 등 최적화 가이드 제시
- 탐색 단계에서 Sub Agents 활용 권장: 코드를 분석하거나 웹 리서치를 병렬로 수행하고, 요약 결과만 메인 에이전트에 통합
    - 10만~30만 토큰 이상의 정보에서 요약 추출(1M 토큰 LLM 한계 내에서 맥락 과부하 방지)
- 탐색이 끝난 후 /plan 커맨드로 구현 계획 마크다운(변경/신규 파일, 작업 리스트, 검증 기준 등)을 자동 생성
    - “계획 문서가 완성되면 반드시 새 세션에서 구현 단계를 시작” → AI가 편향성/등장정보에 영향받지 않게 함

### 구현, 검증, 배포 작업은 AI 에이전트가 자동 처리하도록 커맨드로 분리해 관리함

- /implement(또는 /execute) 명령어: 앞서 생성된 plan.md 문서를 입력으로 받아 실제 코드 작성(새 브랜치, 파일 생성/수정, 자동 테스트 포함)
- 커맨드 내부에 새 브랜치 생성, Pull Request 오픈, 타입체크·린트·유닛테스트 등 전수 검증 및 검증 실패시 후속 피드백 가능
- 최종 검증(코드리뷰, 수동 테스팅 등)은 인간이 담당하며, 엄격히 실제 PRD/Plan과 코드 일치까지 체크
- 모든 관리(티켓 상태 변경, 관련 코멘트 추가, 파일 요약 첨부 등) 역시 AI MCP 서버가 자동으로 처리
- 예시 구현: QR 코드 생성 기능 추가, 앱서버와 프론트엔드 파이핑, 실시간 결과 표시, 브라우저 에이전트 호출로 자체 테스트/스크린샷 검증
- 실패·실수 발생 시 빠른 피드백/보완 ‘Follow-up 이슈’ 생성 및 처리

### 시스템 진화 단계에서는 각 피드백을 추상화하여 전체 워크플로우 개선으로 연결함

- PIV 루프 반복 중 버그·실수·누락 발견 시, 단순 패치에 그치지 않고 “왜 이런 일이 반복됐나?” 원인 규명
- AI 레이어(글로벌 룰, 커맨드, 스킬, 온디맨드 컨텍스트, PRD/Plan 템플릿)에서 수정·보완할 지점 도출
    - 예: 스타일 지침 누락→글로벌 룰/검증 워크플로우에 추가, 오류 재현→테스트 패턴 보강 등
- 모든 커맨드/룰/스킬은 소스컨트롤에 저장, Pull Request와 코드리뷰로 팀원 전체와 공유·협업 가능
- 단순 이슈 패치가 아닌, "한 번의 개선이 수십 시간의 개발 효율 증진으로 이어지는 고레버리지 구조"를 강조
- 커스터마이징 필요성과 진화 가능성: 명령/워크플로우는 각 팀·각 상황에 맞게 주기적 점진적 개선 필수

### 실습 시연을 통해 AI 코딩 보조 프로세스의 실제 적용 단계를 구체적으로 소개함

- 예시 애플리케이션: “퀵 폴 빌더”로, 설문 생성과 실시간 프레젠테이션 뷰 추가를 목표
- AI 대화로 브레인스톰/질문 → PRD 생성 → Jira 에픽과 티켓 자동 생성/분할 → 실시간 관리 체계를 데모로 진행
- /plan, /implement, /prime 등 명령어들이 어떤 구조와 인자(Arguments)를 가지는지 파일 예시/형식 직접 공개(GitHub 참조)
- 오류 발생 및 후속 작업(예: QR 코드 파이핑 누락→즉시 프런트엔드 연결 작업 추가 등) 시연
- 모든 과정은 클라우드 코드, Atlassian MCP 서버, Sub Agents/에이전트 브라우저 등 실제 툴 연동 방식으로 진행됨

### 작업 관리 및 자동화: Jira, GitHub, Linear 등 다양한 환경에 유연하게 대응할 수 있음

- Jira MCP 서버 JSON 설정 한 번으로 이슈/에픽 생성, 상태변경, 코멘트 자동화 모두 처리
- Claude Code 등 AI는 자체적으로 문서 검색, MCP 설정, 오류 디버깅, 설정 파일 자동 구성 등 과정도 안내 및 처리 가능
- 작업 관리 툴 변경시(run in GitHub, Linear 등)은 CLI 명령 및 MCP 서버만 변경하면 프로세스 전체는 동일 유지

### 반복 자동화가 가능한 시스템은 팀·조직 전체의 생산성와 품질을 동시에 끌어올릴 수 있음

- AI를 활용한 팀 전체의 일관적인 PRD/Plan/Task 템플릿, 코드 스타일, 검증 체계를 간편하게 표준화 및 확장 가능
- 팀원 모두(개발자·PM·QA 등)가 동일 커맨드와 템플릿을 참조하므로 지식 공유와 신속한 온보딩이 용이
- AI 에이전트는 코드 작성·이슈 분배·자동 검증·테스트·문서화·클러징까지 “엔드투엔드” 자동화 가능

### “비브코딩”이 아닌, 인간이 주도하는 설계와 검증을 통해 AI의 오류와 한계를 극복할 수 있음

- 단순 AI에 맡길 경우 요구 불일치, 예상 오류 발생→인간의 계획·검증·리뷰(“드라이버 시트에 앉는다”)가 핵심
- 각 단계 산출물(PRD, Plan, 코드 등)은 반드시 인간 검토·피드백·수정 과정을 포함해야 전체 품질이 유지됨
- AI의 최대 역량을 활용하면서도, 인간의 경험과 주도권을 지키는 균형적 프레임워크를 강조
