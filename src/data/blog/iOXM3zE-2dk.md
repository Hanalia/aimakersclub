## *에이전트 관찰성의 격차를 주의하라 (Mind the Gap in your Agent Observability)* 핵심 요약

- 영상의 주제는 "에이전트 관찰성(agent observability)에서의 격차(gap)"를 효과적으로 식별, 관리하고 개선하는 방법임
- 런던 지하철의 'Mind the Gap' 표어를 비유로 활용해 요구사항(플랫폼)과 에이전트(열차) 사이에 존재할 수 있는 간극, 지속적 평가·관찰의 필요성을 강조
- 에이전트는 본질적으로 비결정론적으로(non-deterministic) 동작하므로, 신뢰성과 일관성을 확보하려면 성능, 품질, 안전성 평가 및 모니터링이 필수임
- 개발 초기부터 운영 전 과정에 걸친 관찰성과 지속적 평가 체계를 구현함으로써, 실 사용 환경에서 요구와 실제 동작 사이의 격차를 줄이고 빠르게 최적화 가능
- Microsoft Foundry 플랫폼과 오픈 텔레메트리(OTEL) 기반 추적, 다양한 평가자(evaluator), 앱 인사이트(App Insights) 및 Azure Monitor 등 구체적 도구와 사례가 제시됨
- 단순한 프롬프트형 에이전트부터 툴 연동, 워크플로우 구조의 멀티에이전트 시스템까지 엔드투엔드 개발·관찰·최적화 가능
- 신규 에이전트 구축, 품질 및 안전성 확보, 악의적 프롬프트(적대적 입력) 방어(레드 팀 테스트) 등 실전 개발 과정을 실제 실습 레포와 코드 스페이스로 안내
- 최근에는 "Observe Skill" 등 코딩 에이전트가 인간의 개입(Human-in-the-loop)과 결합해 평가·최적화 루프를 자동화, 반복 실행, 최적판 롤백까지 지원함
- 관찰성, 평가, 레드팀, 모니터링, 버전 관리, 비용 최적화 등 통합 관점에서 체계적 실습 및 오픈소스 리소스를 제공
- Foundry 내 AI 에이전트 구축-배포-운영 전과정에 최적화된 워크플로우 및 확장 가능한 평가·안전·관찰 도구를 체험할 수 있는 실습형 워크숍 진행됨

## 세부 요약 - 주제별 정리

### 'Mind the Gap' 비유는 에이전트 관찰성의 본질적 이유와 개발 및 운영의 경계 문제를 잘 보여줌

- 런던과 뉴욕 지하철의 표어 'Mind the Gap'/'Watch the Gap'을 에이전트 관찰성과 연결, 변화하는 요구환경(플랫폼)과 동적으로 업그레이드되는 에이전트(열차) 사이에 항상 존재하는 격차(gap)를 비유
- 새로운 열차(에이전트)가 기존 플랫폼(요구사항)에 완벽히 적합할 수도, 큰 간극이 발생할 수도 있음
- 요구사항-에이전트의 불일치와 그에 따르는 위험성을 경고하는 “가드레일(guardrail)”로 작동
- 에이전트가 실사용자와 만날 때 일어나는 지속적 환경 변화에 대응, 위험성 통제와 품질 확보를 위해선 조기부터 관찰성과 평가 루프가 필요함
- 관찰성·모니터링은 일회성 혹은 개발 단계 끝이 아닌, 에이전트 라이프사이클 전 과정에서 지속적으로 필요

### 비결정론적 에이전트의 특성상 신뢰성과 품질, 안전성 확보에는 평가·관찰·최적화 루프가 핵심적임

- “에이전트는 비결정론적(non-deterministic)으로 동작한다”—동일 입력이라도 응답이나 툴 호출 결과가 변할 수 있음
- 생산 단계에서 불확실성과 품질 저하, 부적절한 행동(예: 툴 오용, 맥락 오해) 예방을 위한 3단계: 평가(Evaluate), 모니터링(Monitor), 최적화(Optimize)가 중요
- 평가 단계에선 성능, 품질(quality), 안전(safety) 등 다양한 내장 및 커스텀 평가자를 활용 가능
- 실시간 및 지속적 모니터링 체계 구축으로 에이전트의 환경·요구·사용자 피드백 변화에 신속 대응
- 방대한 에이전트 운영 및 멀티에이전트 시스템 확장 환경에서도 동일 원칙 적용

### Microsoft Foundry 플랫폼은 에이전트 구축-관찰-운영의 엔드투엔드 통합 환경 및 다양한 도구, 프로토콜을 제공함

- Foundry는 클라우드 에이전트 플랫폼으로, 에이전트 개발(자체 혹은 외부), 호스팅, 관찰, 모니터링, 통합관리가 한 곳에서 가능
- 오픈 텔레메트리(OTEL) 기반 추적 지원—여러 조직·플랫폼에서 생성된 다양한 에이전트들도 공통 추적으로 묶어 통합 관찰 가능
- Foundry 컨트롤 플레인과 Azure Monitor 연동(IT/운영 관점 통합), 기존 클라우드 및 데이터 스택과 연결 용이
- 추적(Tracing)을 통한 내부 동작, 툴/메시지 호출 단계, 비용/토큰 추적, 평가 결과 등 심층 데이터 시각화 지원
- 멀티에이전트 워크플로우, 에이전트 버전 관리, 보안(레드팀 등) 연동까지 폭넓은 확장성 보장

### 개발 초기부터 관찰성 및 평가 체계를 내장하면 추후 디버깅, 품질 개선, 운영관리에 큰 이점을 제공함

- 에이전트 개발 첫 단계(Foundry 포털)에서 프로젝트/에이전트 생성 → 모델 선택(예시: GPT-4, Anthropic, DeepSeek 등) → 툴 추가(예: Bing 웹검색)
- App Insights 연동을 통한 실시간 추적 활성화, 엔드투엔드 흐름의 투명한 관찰 가능
- Web portal에서 버튼 몇 번만 누르면 초기 품질/안전/비용/토큰 등 주요 평가 항목을 즉시 그래프로 확인 가능
- 초기 프로토타입 단계에서 평가와 추적을 동시에 시작하면, 최소 기능 검증(MVP) 이후 코드/SDK 기반 커스텀화 이전에 버전별 개선효과 확인·최적판 관리 용이
- 개발자와 IT 관리자의 협업 및 인프라 통합(조직별 비용처리, 코드스페이스의 기업 계정 연동)에도 용이

### 실습형 GitHub 레포지토리, 코드스페이스, Azure 무료 계정 등 실제 환경에서의 재연 및 확장 실습을 안내함

- 공식 GitHub 레포를 포크해 모든 브랜치(vs 메인만 포크X) 포함 확보—AI Engineer 채널에서 지속 업데이트 안내
- 코드스페이스(CodeSpaces)와 dev container 구조로, 별도 로컬 패키지 설치·환경셋업 없이 브라우저 기반 개발환경 즉시 구축
- 조직 레포 활용 시 코드스페이스 비용도 엔터프라이즈 계정 차감 제공(컴퓨트 무료 사용)
- 레포 내 단계별 실습 노트북(lab), 자동 개발 환경 구성 스크립트, 완성된 결과 예제(“베이킹쇼 구조”) 등 제공
- 워크숍 참가자 외에도 누구나 집에서 무료/저비용으로 반복 학습·확장 가능

### 실제 사례: 여행 에이전트(Contoso Travel 앱) 구축 절차를 통해 단일-툴-워크플로우-멀티에이전트 구조로 체계적 고도화를 시연함

- 목표: “호텔·차·비행기” 예약을 돕는 여행 에이전트 앱 개발—모델 선정, 데이터 유무, 툴 세팅 등 현실적 문제 직면
- 수백~수천 개의 공개 모델(예: Hugging Face 200만+, Azure 모델 카탈로그 1.1만+) 속에서 빠른 실행이 어려움
- 데이터 없는 상황에서의 평가·실험(“제로에서 에이전트로” 신속 프로토타이핑)을 위한 프레임워크 제공
- 기본 모델+지침+최소 툴 기반 단일 에이전트 → 기능별 툴 함수 추가 → 멀티에이전트(예: Hotel/Car/Flight 각각 수행) 및 워크플로우(scheduler/concierge로 통합)
- Portal 기반 빠른 실습 후 SDK/Jupyter notebook을 활용한 코드 기반 확장(커스텀 구성, 자동화, 디버깅, 트레이싱)

### 품질·안전성 평가자 내장 및 커스텀화로, 단순 모델 벤치마크를 넘어 실제 에이전트 행동 전체에 대한 정밀 평가·분석이 가능해짐

- Intent resolution(의도 파악), tool call(툴 호출 적합성), task adherence(과업 충실도) 등 워크플로우상의 각 단계별 세분 평가
- 내장 평가자(quality, risk, safety 및 agentic)는 플랫폼에 계속 추가/업데이트—커스텀 평가자도 코드로 작성해 적용 가능
- 다양한 평가 지표 선택 후, Portal 및 노트북 상에서 실험/비교/보고서 자동 생성
- 예: “오늘 런던의 날씨?” → 의도 파악→웹검색 호출→응답의 과업 충실도 등 단계별 평가, 각 단계에서 어디서 문제 발생했는지 추적

### 추적(Tracing), 모니터링, 로그 연동과 버전별 비교·롤백 등 운영·최적화 메커니즘에 대한 구체적 실전사례 제공

- OTEL 기반 추적으로 로컬/클라우드 모두 동시 추적(환경변수로 간단 활성화), 메시지·함수별 맞춤 속성 추가 및 디버깅
- Azure Monitor와 연동—AI 추적과 인프라 로그, 기타 서비스(예: Azure AI Search) 데이터까지 통합적으로 분석
- 각 에이전트/워크플로우 단계별 토큰 비용, 성능 이슈, 모델 변경 시 영향, 툴 호출 실패 등 복합 상황 신속 진단
- Portal 내 에이전트 버전 관리, 특정 버전 롤백, 코드/프롬프트 최적화→실시간 평가→회귀(성능 저하) 감지시 자동 롤백
- KQL 등 쿼리 없이도 ‘Observability Agent Helper’(AI 기반)로 자연어 로그 분석 및 트러블슈팅 지원

### 레드팀(Troubleshooting)·적대적 테스트(Adversarial Testing)로 실환경 공격·취약점에 대한 방어체계도 쉽게 실습할 수 있음

- “레드팀”은 모델/에이전트에 대한 공격적 시나리오로 내장 위험(violence, info leak, task adherence 등) 평가
- Open-source pirate 저장소 연동, 기존 행동 기반 안전성 평가에서 진보된 “에이전트 행동 자체의 적대 공격”까지 범위 확대
- 예: “prohibited actions(금지된 행동) taxonomy”에 기반, AI가 자율적으로 다양한 탈법·우회 프롬프트(leet, word twist, crescendo etc.)를 생성·공격
- 통과/실패 사례 자동 보고 및 취약점 상세 분석—실전 공격 대응 및 가드레일 검증 루틴 내장

### 최신 Observe Skill 등 코딩 에이전트 기반 자동화 루프가 관찰·최적화·버전관리를 획기적으로 단축시킴

- Copilot 등 코딩 에이전트와 Observe Skill의 결합: “도움을 요청”하면 자동으로 평가 데이터셋 생성, 배치 평가 실행, 결과 분석·해석·최적화 반복
- 인간 개입 하에(“Human-in-the-loop”), 문제 원인 분석→설정·프롬프트 최적화→재평가→최적판 별도 관리→자동 롤백(실패 감지시)의 전체 루프 자동화
- 단순 프롬프트 최적화뿐 아니라, 평가 기준/툴 변경, 모델 변경, 원클릭 확장·변환(Hosted Agent/LangGraph 등)까지 지원
- 실습형 워크플로우 완전 자동화: 각 단계별 이유(reasoning), 개선 내역, 버전별 지표 표 형태로 제공

### 실시간 질의응답 및 개발/운영 포털 내 ‘Ask AI’, Observability Agent Helper 등이 더욱 원활한 상호작용·분석을 지원함

- Foundry Portal 내 ‘Ask AI’: 현 프로젝트 상태, 에이전트 목록, 쿼터, 추적ID 기반 로그 조회 등 자연어 질의 가능
- Observability Agent Helper: 로그/트레이스 복잡 데이터를 코드/쿼리 없이도 요약, 실패 원인 추적 등 자동화
- 전체 실습은 Discord 연계 커뮤니티, 지속적 레포 업데이트, 장애보고 및 실전 결과 공유로 확장됨

### 결론적으로, 요구-실행 간 격차를 “지속적 관찰성” 기반으로 실시간 파악·고도화함으로써, 대규모·동적 AI 에이전트 환경의 품질·안전·최적화 과제를 통합적으로 해결할 수 있음을 강조함

- 개발-테스트-운영 전 과정에 ‘관찰→평가→최적화→롤백’ 루프를 내장하는 것이 대규모/멀티에이전트 환경에서의 품질 유지 비결
- 코딩 에이전트, 자동화된 평가/최적화 루프, 풍부한 도구·레포지토리, 오픈소스 협업 등으로 누구나 실전 구현/확장 가능
- 신속한 환경 변화(모델, 데이터, 요구, 악의적 공격)에도 유연히 대처할 수 있는 체계를 제시
- 최신 워크플로우, 실습 자원(레포, 포털, 노트북, Discord 등), 전문가 AMA 등으로 실무 실전력 함양 및 네트워킹 연계 가능
