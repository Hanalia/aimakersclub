## *망가지지 않았다: 소프트웨어 기본기는 AI 시대에 더욱 중요하다* 핵심 요약

- 이번 강연의 주요 메시지는 "소프트웨어 기본기는 AI 시대에 오히려 더 중요해졌다"라는 점임
- 기존의 ‘스펙-투-코드(specs to code)’ 운동 방식—AI가 명세를 코드로 자동 변환하는 접근—이 제대로 동작하지 않음을 현실 예시로 설명함
- 소프트웨어 복잡도 이론(John Ousterhout)과 소프트웨어 엔트로피(Pragmatic Programmer) 등 고전적 원칙이 지금 다시 중요해지고 있음을 강조함
- 복잡한 코드베이스는 AI의 효율적 활용과 빠른 변화 수용을 방해하며, 코드 품질이 곧 비용이 되는 시대임을 수치 없이 강력하게 주장함
- AI와의 효율적 협업을 위해 ‘그릴미(Grill Me)’, ‘유비쿼터스 랭귀지’ 등 핵심 스킬 및 오픈소스 툴을 수천 명 사용 사례와 함께 제시함
- 설계 개념 공유, 도메인 언어 통일, 테스트 중심 개발(TDD), 딥 모듈 구조 설계, 모듈 인터페이스 설계 등 실질적인 실천법 소개
- 앤트로피, 피드백 루프, DDD, Kent Beck의 시스템 설계 투자 등 다양한 고전 소프트웨어 공학 이론 실제 코드 개발 과정에 적용
- AI는 전술적(저수준) 역할을 하고, 인간 개발자의 전략적(고수준) 역할이 점점 더 필수적임을 강조함
- 발표자가 개발한 각종 스킬 모음집은 GitHub에서 제공됨(13,000개 이상 스타), 추가 학습 자료는 aihero.dev, 유튜브, 트위터 등에서 제공함

## 세부 요약 - 주제별 정리

### AI가 자동으로 코드를 만들어도 소프트웨어 기본기가 무너지면 코드 품질은 급격히 하락함

- 최근 AI 발전으로 ‘스펙을 작성하면 AI가 코드를 생성하는’ specs-to-code 방식이 확산되고 있음
- 경험상, AI로 스펙을 코드로 변환 후 반복 실행 시 코드 품질이 점점 더 나빠짐(“쓰레기 코드만 나오더라” 생생한 현장 경험 공유)
- 개발자가 코드 품질을 직접 챙기지 않으면 AI는 코드 구조를 점점 악화시키는 경향이 있음
- “코드는 싸다”라는 관념이 팽배하지만, 지속적 변화와 유지가 어려운 코드베이스는 오히려 더 비싸고 위험하다는 점을 지적
- 복잡한 코드베이스는 버그가 자주 발생하고 수정도 어려워 AI 효율을 떨어뜨림
- 결론: 지금 시대일수록 좋은 코드와 탄탄한 소프트웨어 설계 원칙이 더 요구됨

### 소프트웨어 복잡성 및 엔트로피 이론이 AI 시대에 더욱 설득력을 가짐

- John Ousterhout 『A Philosophy of Software Design』에서 복잡성: “이해와 수정이 어려운 코드베이스 그 자체가 나쁜 코드”
- Pragmatic Programmer의 ‘소프트웨어 엔트로피’ 개념: 변화 시 전체 설계 고려 없이 변화만 반복하면 시스템이 붕괴한다는 지론 소개
- specs-to-code 자동화 방식이 오히려 엔트로피를 가속화시키는 사례로 제시
- 변화가 거듭될 때 기본기가 없는 코드는 갈수록 품질이 저하되며 장기적으로 유지보수 비용이 상승함
- AI가 더 많은 코드를 생산해도, 설계를 투자하지 않으면 시스템 자체가 빠르게 망가짐

### AI가 요구사항을 오해하는 원인은 ‘공유된 설계 개념’ 부족에 있음

- AI가 개발자의 의도를 잘못 이해하고 엉뚱한 결과물을 자주 낸다는 사례 소개
- 이는 개발자와 AI가 ‘design concept(설계 개념)’을 공유하지 못하기 때문임(Frederick P. Brooks의 「The Design of Design」 인용)
- design concept: 제품 설계자들 사이에 존재하는, 문서로 완벽히 기록될 수 없는 ‘암묵적 이상 구조’
- 인간-인간, 인간-AI 모두 이 design concept 부재 시 협업 효율 급감함을 경험적으로 설명
- 이를 해결하기 위한 스킬/도구: “Grill me”—AI가 집요하게 개발자를 인터뷰하며 양측에 완전한 이해가 도달할 때까지 디자인 트리로 질문을 반복함

### ‘Grill Me’ 스킬은 요구사항 명확화와 AI 협업의 강력한 무기로 입증됨

- ‘Grill Me’ 스킬은 13,000개 이상의 스타를 받은 GitHub 오픈소스로, 대중적으로 확산됨
- 작동 방식: AI가 개발자에게 40~100개의 질문을 던져, 완전히 동일한 설계 인식을 공유함을 목표로 함
- 이를 통해 얻어진 대화 내역을 제품 명세(PRD)나 단위 이슈로 구체화해 활용 가능
- 기본 제공되는 'plan mode'보다 Grill Me를 사용했을 때 더 견고한 설계 개념을 구축할 수 있음
- 작은 변화에는 바로 이 대화 결과를 이슈로, 큰 프로젝트에는 PRD로 전환해 개발 흐름이 매우 명확해짐

### AI-개발자 협업에서 ‘유비쿼터스 랭귀지’ 정립이 중복/오해/장황함을 줄임

- AI의 답변이 장황하거나 개발 흐름과 동떨어질 때 발생하는 실제 사례 지적
- 이는 오랜 개발 경험에서 “도메인 전문가-개발자 언어 불일치”와 같음을 비유로 설명
- Domain Driven Design(DDD)의 핵심 원리 ‘Ubiquitous Language(통합 언어)’에서 해법 도출
- 공유된 용어 사전을 AI와 개발자 양측 모두가 참조(마크다운 표 형태)해, 대화와 코드 구현 때 용어 혼선을 최소화함
- 이를 위한 스킬 제공: 코드베이스에서 용어 추출→마크다운화→AI에 입력, 실제 계획 및 구현 과정에서 효율 증대 사례 제시

### AI가 제대로 동작하지 않을 때, 강력한 피드백 루프와 테스트 기반 개발이 필수임

- AI가 올바른 기능을 만들어도 작동하지 않는 사례 빈번
- 해결책 1: 타입스크립트 등 정적 타입, 자동화 테스트 등 ‘피드백 루프’ 도구 적극 도입 강조
- 실제로 AI는 피드백 루프를 능숙하게 활용하지 못해 ‘대규모 코드 생성 후 사후(事後) 검증’에 그침
- Pragmatic Programmer에서는 "피드백 속도가 곧 개발 속도 한계다"라고 설명
- 소단계 테스트와 구현(테스트 주도 개발, TDD)이 가장 효과적이며, AI도 이 흐름으로 안내해야 코드 품질 유지 가능

### 테스트 기반 개발(TDD)은 AI의 시행착오 빈도와 코드 불안정성을 현저히 줄여줌

- TDD 방식: 테스트 먼저 작성→통과→코드 리팩토링 및 설계 개선
- 테스트 설계의 어려움: 어떤 단위를 테스트할지, 무엇을 목(mock)할지, 어떤 행동을 검증할지 등 복잡한 의존성 문제 존재
- 좋은 코드베이스일수록 테스트 설계가 쉽고, AI가 더 나은 피드백을 받을 수 있음
- 결론적으로, 테스트가 편한 구조—즉, 설계가 잘 된 구조가 AI 시대에도 강점이 됨

### ‘딥 모듈’ 구조가 AI와 협업에 최적화된 코드베이스임을 강조함

- John Ousterhout가 제시한 ‘딥 모듈’이란: 복잡도를 모두 내부로 감추고, 단순 명확한 인터페이스만 외부에 드러낸 구조
- 흔한 얕은(샬로우) 모듈 방식은 수많은 자잘한 코드 덩어리들이 난립해 AI가 전체 흐름을 이해·활용·변경하기 어려움
- AI가 생성한 코드베이스가 종종 샬로우 모듈로 채워지는데, 이는 피드백 루프, 테스트, 변경 관리 모두에 장애가 됨
- 반대로 딥 모듈 구조는 외부 인터페이스만 명확히 설계하면 내부 구현은 AI가 비교적 자유롭게 생성·변경 가능
- 개선 방법: 연관 코드를 묶어 딥 모듈화, 외부 경계(인터페이스)만 테스트
- ‘Improve Codebase Architecture’라는 스킬로 자동 구조 개선 방법 제시

### 모듈 인터페이스에 집중하고 구현은 AI에 위임하면 인간 두뇌의 피로도를 대폭 줄일 수 있음

- 코드 생산성이 극적으로 높아져도, 인간 개발자는 복잡한 코드베이스에 더 쉽게 피로를 느낌
- 딥 모듈 구조 채택 시, 개발자는 모듈 경계와 인터페이스 설계에만 집중하고, 내부 구현은 AI가 담당 가능
- 중요한 시스템(예: 금융 등) 외, 다수 모듈은 이렇게 ‘외부 인터페이스’만 신경써도 문제 없음
- 이 방식은 인간과 AI 모두에게 인지적 부담을 줄이며, 코드 품질과 산출물 신뢰성 모두 높임
- 조건: 각 모듈의 목적, 인터페이스, 변경점을 항상 명확히 관리해야 함

### 소프트웨어 구조 설계에 대한 지속적 투자가 AI 시대 개발의 핵심이다

- 모든 계획, 요구사항 명세(PRD), 리팩토링 과정에서 ‘현재 코드베이스의 모듈 구조와 인터페이스’ 관찰 및 명시 필수
- Kent Beck의 "시스템 설계에 매일 투자하라"는 원칙을 현실 적용 사례로 인용
- 스펙-투-코드 방식은 설계 투자 자체를 포기하는 것과 같으며, 결과적으로 장기 유지보수와 확장성 악화 초래
- 핵심 결론: ‘코드는 싸지 않다’, 코드는 중요하며, 인간 개발자의 소프트웨어 설계 기본기가 앞으로 더욱 결정적임

### AI가 뛰어난 "전술적 프로그래머"라면 개발자는 "전략가"로서 기본기를 더욱 다져야 함

- AI는 부분 코드 작성·변경 등 실행단(전술) 역할엔 강점이나, 전체 구조와 장기적 전략 설계에는 한계가 있음
- 개발자는 모듈 정의, 설계, 경계 판단, 아키텍처 등 시스템 전체를 꿰뚫는 안목과 역량이 필수임
- 발표자가 개발한 다양한 스킬은 모두 GitHub(‘macpco-skills’), aihero.dev, 유튜브, 트위터 등에서 무료 혹은 오픈소스로 제공됨
- 핵심 메시지: AI 시대에도 소프트웨어 기본기에 투자하고 역량을 꾸준히 쌓아야 성장과 성공이 가능함
