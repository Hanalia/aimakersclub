## *지금 최상위 AI 에이전트 엔지니어를 구분짓는 하니스 엔지니어링* 핵심 요약

- **“하니스 엔지니어링”(Harness Engineering)은 대형 언어 모델(LLM) 위에 다양한 래퍼(wrapper)와 컨텍스트 및 업무 프로세스, 제어 레이어를 설계하는 기술로, 2024년 AI 분야의 핵심 트렌드로 부상**
- **기존 “컨텍스트 엔지니어링”의 직접적 진화로, 더 큰 차원에서는 여러 코드 에이전트 세션을 결합·조율하여 복합적 과업을 처리할 수 있게 하는 것이 차별점**
- **모든 코드 에이전트(코딩 도우미)는 모델(예: GPT, Claude)에 툴사가 랩핑한 1차 하니스(예: Claude Code, Codex 등)를 사용하고, 사용자가 직접 설계하는 최상위 ‘AI 레이어’를 통해 컨텍스트·규칙·스킬·후크 등을 자유롭게 탑재해 성능 강화**
- **AI 레이어의 6대 핵심 컴포넌트: 글로벌 규칙(global rules), 스킬(skills), MCP 서버, 코드 기반 검색(LSP/지식그래프), 후크(hooks), 서브에이전트(sub-agents)**
- **단일 코드 에이전트 세션에서의 하니스 설계와, 여러 세션·에이전트를 워크플로 형태로 연결(Orchestration)함으로써 대규모 과업도 효율적으로 자동 처리**
- **모델 실패 시 업그레이드만 기다리는 태도에서 벗어나, 하니스 엔지니어링은 ‘모든 실수는 곧 시스템 진화의 기회’라는 지속적 개선 마인드 강조(시스템 오너십)**
- **Google Cloud의 Agent CLI처럼, 다양한 오픈소스 도구 및 SDK가 손쉽고 자동화된 에이전트 개발·배포·모니터링을 지원, 실제 데모와 코드 예시 제공**
- **RA Loop, Archon(오픈소스 하니스 빌더) 등 코딩 에이전트 멀티세션 자동화 및 확장 사례와 구체적 워크플로 시연**
- **궁극적으로는 하니스 설계를 통해 각각의 에이전트 세션을 토큰 효율적·집중적으로 운용하고, 대규모 PRD까지 자동 분할·처리하는 미래형 에이전트 아키텍처 구축이 목표임**

## 세부 요약 - 주제별 정리

### 하니스 엔지니어링은 LLM의 한계를 뛰어넘는 시스템 래퍼 설계 기술임

- 하니스 엔지니어링은 대형 언어 모델(GPT, Claude 등)의 기본 ‘추론 능력’ 위에, 사용 목적에 맞춘 컨텍스트·규칙·프로세스 등을 래핑(감싼)함으로써 실질적 업무수행 능력을 획기적으로 확장
- 기존의 “컨텍스트 엔지니어링”은 에이전트가 동작하는 맥락을 잘 주입하는 데 초점이 있었음
- 하니스 엔지니어링은 이를 더 구조화하여, 업무 규칙·보안 후킹·스킬(작업 절차/기능)·지식베이스 연계 등 모든 레이어를 설계 가능하도록 한 단계 진화
- “하니스”의 정의: 에이전트(코딩 도우미) 행동을 결정짓는 전체적인 시스템 구조, 단순 프롬프트 입력 이상의 설계와 통제권을 갖게 됨

### 코드 에이전트 생태계는 본질적으로 모델+하니스 구조로 이루어짐

- 모든 에이전트는 **1) LLM(코어), 2) 툴사가 제공하는 1차 하니스(예: Claude Code, Codex, Pi 등), 3) 사용자가 설계하는 상위 하니스(즉, AI 레이어)**의 3중 구조로 구성
- Claude Code, Codex와 같은 제품 자체가 이미 1차 하니스(특정 기능과 컨텍스트 래핑) 형태
- 사용자는 ‘어떤 코드 에이전트(하니스)를 고른다’는 단계 자체도 하니스 설계에 포함됨(최적 도구 선정)
- 실제 개발자는 이보다 한 단계 위 ‘AI 레이어’(본인만의 규칙, 후크, 스킬, 서버, 지식 검색 등 통제권 보유)를 직접 구축해 진정한 성능/신뢰성 향상 가능

### AI 레이어에는 6가지 핵심 설계 요소가 포함됨

- **글로벌 규칙(Global rules)**: 에이전트가 반드시 따라야 할 규정, 관례, 제약조건 등(예: 코드 스타일, 파일명 규칙, 보안 정책)
- **스킬(Skills)**: 에이전트가 단계별로 수행하는 워크플로 정의(예: 기획→구현→검증을 분리된 스킬로 구성)
- **MCP 서버**: 외부 기능 호출, 협업 등 복잡한 연계·서버와의 통신 지원
- **코드 기반 검색(LSP, 지식 그래프 등)**: 코드베이스, 문서, 레퍼런스 등에 대한 접근성 확보
- **후크(Hooks)**: 에이전트 동작 전·후에 자동 트리거되는 검증/보안/클린업 코드. 예: 파일쓰기, 디렉터리 삭제 명령 전 검증, lint 자동 실행
- **서브-에이전트(Subagents)**: 특정 하위 작업을 분리·위임하여 병렬성·효율성 극대화

### 하니스 설계는 단일 세션에서 시작되고, 이는 컨텍스트 엔지니어링과 많은 부분 겹침

- 처음에는 단일 코딩 에이전트 세션에서 컨텍스트, 룰, 스킬, 후크 등을 적용하는 방식(기존 컨텍스트 엔지니어링의 확장)
- 예시: 하나의 작업을 위한 플랜 수립→구현→검증의 분리된 스킬 세션으로 토큰과 맥락의 집중도 극대화
- 각 스킬이 산출물을 내고, 산출물(markdown 문서 등)을 다음 세션에 직접 핸드오프(수동 연결)하여 단계별로 진행
- 이 단계에서도 후크를 통한 보안(실행 전 툴 사용 제한), 치명적 명령 방지 등이 가능

### 하니스 엔지니어링은 “통제와 오너십” 마인드셋이 필수 요소로 강조됨

- 일반적으로, 에이전트가 실수하면 “모델 업그레이드(예: GPT-6 나오면 좋아지겠지)”만 바라보는 경향
- 하니스 엔지니어는 시스템에 대한 피드백 루프를 적극 수용, 모든 실수·오류를 하나의 규칙/후크/스킬 업데이트 기회로 삼음
- “매번 실수마다 학습하고, 시스템을 진화시킴” → 지속적인 성능 개선 및 신뢰성 확보
- 시스템의 주도권을 인간이 가져가고, 에이전트의 업무 단계를 센서(후크·테스트·리뷰 에이전트 등)를 통해 자동 점검 및 발전

### Google Cloud Agent CLI 등 오픈소스 도구가 하니스 엔지니어링을 가속화함

- 구글 클라우드의 에이전트 CLI(개발전용 커맨드라인 툴킷)는 실제 에이전트 기획, 구현, 배포, 모니터링의 전과정을 명확/신속하게 지원
- 예시: “Agent CLI로 웹 검색 기능 에이전트 만들기” → 명확한 명령어/스킬 세트 기반으로 코드 자동 생성, 테스트, 배포(한줄 커맨드로 Google Cloud에 배포·실행)
- 에이전트는 고유의 식별자(ID) 확보, 라이브 테스트, 트레이스 기반 전체 처리과정 관찰 가능
- Agent CLI는 무료 오픈소스, 다른 모든 코딩 에이전트에서도 활용 가능(생산성 극대화)

### 실제 AI 레이어 구현 예시(코드 레포와 활용법 소개)

- 영상에서 공개하는 동반 깃허브 레포(github.com/colemedin/[...] 등)에는 전형적인 AI 레이어 구조와 구현 예제 포함
- 각 에이전트 작업은:
    1. 플랜 플로우(Plan Skill)로 기능 설계
    2. 구현 플로우(Implement Skill)로 코드 작성(이때 플랜 문서를 입력하여 맥락 제공)
    3. 검증 플로우(Validate Skill)로 결과물 자체 점검(텍스트 아웃풋, 유닛테스트, 타입체크, 린트 등 포함)
- 각 스킬 아티팩트를 수동으로 건네주는 기본 방식부터, 워크플로 배치 자동화까지 다양한 활용 가능

### 자동화된 멀티세션 오케스트레이션이 하니스 엔지니어링의 ‘진화’임

- 대규모 과업(PRD 등)을 한 번에 단일 코딩 에이전트에 넣으면 토큰 한계, 집중도 저하로 비효율/오류 초래
- 해결책: 개별 작업을 분할하여 각 에이전트 세션에 맡기고, 워크플로 흐름 전반을 ‘하니스’로 자동 조정(오케스트레이션)
- 예시 구조(워크플로):
    - 요구사항→플랜 세션→구현 세션→(병렬)코드 리뷰 세션(보안/정확성/단순성 등)
    - 모든 항목 통과 시 풀 리퀘스트 생성, 실패 시 자동 이터레이션/리트라이 진행
- 수동-반자동-풀오토메이션까지 확장 가능, 자동화된 하니스 구조의 도입 시 업무량·난이도/신뢰성 모두 크게 개선

### RA Loop, Ralph, Archon 등 대표적 하니스 엔지니어링 툴과 워크플로 데모

- Ralph Loop(제프리 헌틀리 개발): 파이썬/배시 스크립트 기반, 요구된 대작업(PRD 등)을 항목별로 쪼개고, 각 항목별로 클라우드코드, Codex 등의 세션을 자동 반복 호출하여 완성될 때까지 워크로그를 남기는 구조
- 조건: 모든 작업이 완료됐음(예: done.txt 파일 존재) 확인→전체 루프 종료
- 이 접근은 에이전트 다중 세션 활용을 극대화함과 동시에 감독자가 작업 상태를 명확히 추적 가능
- 오픈소스 하니스 빌더인 Archon 소개(직접 맞춤형 하니스 워크플로 편리 구축), 누구나 자유롭게 활용/확장 가능

### 하니스 엔지니어링의 본질과 실제 구현 요소 구분법 제시

- 단순한 ‘버즈워드’(흔한 유행어)에 그치지 않고, 실제로 컨텍스트 엔지니어링과 어떤 점이 구분되는지 예시와 토론으로 상세 설명
- 본질적인 차이: 컨텍스트 전달에 머무르지 않고, 여러 세션/에이전트를 직접 제어·자동화·감시·피드백하는 ‘시스템적’ 접근
- 영상 내내 실제 코드, 도구, 문서, 레포 활용 방법 등 구체적이며 실용적인 팁 제공
- 영상 마지막에서는 본 채널의 오픈소스 리소스 및 더 깊이 있는 설명·워크플로 예제 등 추가 자료 제공
