## *MCP = 메가 컨텍스트 문제 - Matt Carey* 핵심 요약

- **MCP(Mega Context Problem)는 LLM 기반 에이전트가 대규모 API와 상호작용할 때 초래되는 컨텍스트 한계 문제를 의미한다.**
- **Matt Carey(Cloudflare)는 MCP 및 에이전트 툴링 관련 최신 변천사와 실무상의 도전 과제를 상세하게 공유한다.**
- **과거에는 각 에이전트에 개별적으로 다양한 툴을 번들링했으나, 점차 표준화된 MCP 툴 셋을 외부적으로 제공하는 방식으로 변모했다.**
- **Cloudflare API 전체(약 2,600개의 엔드포인트)를 에이전트가 다루기 위해서는 최대 110만 토큰 이상의 컨텍스트가 소요되어, 사실상 기존 방법론이 한계에 부딪혔다.**
- **이 문제의 대응책으로 제품 기반 MCP 서비스 분할, 프로그레시브 디스커버리(progressive discovery), 툴 검색(tool search), CLI 활용, 코드 기반 상호작용 등 여러 접근이 시도됨.**
- **코드 기반 툴링(에이전트가 타입 정보만 받아 직접 코드를 작성하는 방식)은 강력하나, 신뢰할 수 없는 코드 실행에 대한 치명적 보안 리스크도 동반하여 기존 클라이언트에서 잘 도입되지 못함.**
- **Cloudflare Worker와 같은 격리(샌드박스) 환경에서 동적 코드를 안전하게 실행하는 방법이 해결책으로 대두, 복수의 클라우드 인프라 제공자(Deno, Pydantic Monty 등)도 유사 흐름 추종 중.**
- **향후 API/서비스 제공은 MCP와 같은 미들웨어 내장, 경량화된 SDK, 클라이언트의 프로그램화된 툴 호출 등으로 진화할 전망임을 강조한다.**

## 세부 요약 - 주제별 정리

### 에이전트 툴링의 초기 방식은 비표준 번들링에 치우쳤으나 점차 중앙화된 MCP 표준으로 전환됨

- Matt Carey는 Cloudflare에서 MCP 및 에이전트 관련 작업을 하고 있으며, API를 에이전트의 '손'처럼 쓰게 하는 것이 핵심 목표임을 밝힘.
- 초기에는 각 개발자가 사용하고자 하는 서비스(Gmail 등)에 대해 별도의 툴 셋을 직접 제작해 번들링하는 구조가 주류였음.
- 이런 번들링 방식은 매번 재작업이 요구되고, 같은 기능이라도 사용자마다 중복 구현한다는 비효율성이 존재했음.
- 2023년 4월경부터는 표준화된 MCP(Remote MCP 포함) 도입으로 서비스 제공자 측에서 ‘한 번’ MCP 툴 세트를 구축해 제공하고, 다수의 에이전트가 동일 툴을 공유해 쓸 수 있도록 구조가 크게 변화함.
- API 제공자는 이제 MCP, CLI, GraphQL, REST 등 다양한 접근 경로 중 하나로 MCP를 공식 서피스로 제공함.

### 대규모 API 전체를 에이전트에 노출할 경우 심각한 컨텍스트 윈도 한계에 직면함

- Cloudflare와 같이 방대한 API(예: 2,600+ 엔드포인트)에 툴 전부를 단순 나이브하게 에이전트에 노출하면, 컨텍스트 창이 110만~230만 토큰 수준으로 폭증.
- 대표적 OpenAPI 스펙의 경우 230만 토큰, 이를 MCP 툴 정의로 변환해도 110만 토큰, 현존 LLM 컨텍스트 윈도 최대치(수십만~100만 토큰)로는 도저히 불가능함.
- 이는 특정 회사 및 API에 국한된 문제가 아니라, 모든 범용 LLM에 내재하는 구조적 한계임(컨텍스트 오버플로우).

### 분산 MCP 서비스 도입 및 프로그레시브 디스커버리 도입 시도에도 한계가 존재함

- Cloudflare 내부적으로는 전체 API를 여러 개의 제품군별(MCP 서버: 최대 16개)로 분리해 부분적으로 에이전트에게 툴 노출을 시도함.
- 단점은 사용자가 목적에 맞는 MCP 서비스를 직접 선택해야 하며, 각 서버당 노출되는 API 엔드포인트가 제한적(예: 6개 노출, 전체는 30개)인 불완전 커버리지 현상이 빈번함.
- 일부 툴·엔드포인트가 누락될 수밖에 없고, 사용자 입장에서는 전체 API를 자유롭게 활용하는 것이 제한됨.

### CLI 접근 방식은 인기 있으나, 오로지 셸 접근이 가능해야 하는 제약이 존재함

- 에이전트가 명령형 인터페이스(CLI, 예: wrangler)를 활용하는 방안은 직관적이고 인기(특히 커맨드 자동 탐색, 헬프 등) 있으나, 반드시 셸 접근이 필요하다는 단점이 있음.
- 대표적으로 “데이터베이스 리스트” 등의 관리 작업이 wrangler CLI로 가능하고, 파라미터를 명확히 introspection하여 쓸 수 있음.
- 클라우드 코드 환경 등에서는 CLI 대신 툴 검색(검색기반 context 로드) 같은 구조도 선호됨.

### 툴 검색 도입은 토큰 낭비 최소화에 유리하나 완전한 문제해결은 아님

- Cloudflare의 cloud code 등에서는 ‘툴 검색(search tool)’ 방식을 도입: 사용자의 요청을 받아 키워드 매칭을 통해 연관된 툴(예: 6~8개)만 컨텍스트에 로드하여 토큰 소모를 최소화함.
- 실제 LLM은 최종적으로 2,100토큰 중 500토큰만을 실사용하며, 관련 없는 툴은 컨텍스트에서 미사용 상태로 남음.
- 이 방식은 기존 MCP 전체 로드 대비 효율적이나, 완전 자동화/확장성에는 한계가 있음.

### 타입 기반 코드 작성(코드 기반 툴링)은 에이전트-API 통합의 현존 최적 해법임을 보임

- Cloudflare는 2023년 여름, LLM에게 OpenAPI를 기반으로 자동 생성된 TypeScript 타입(입·출력 정의)만을 제공하고, 에이전트가 직접 코드를 작성하게 하는 ‘코드 모드(code mode)’ 방식을 공개함.
- 예: “list workers” 요청 → 타입 기반 SDK를 활용해 에이전트가 직접 코드(예: listWorker 함수)를 작성, 즉시 실행하여 결과 획득 가능.
- 배포, 권한 관리 등 고도화된 조작도 모두 타입 정보만으로 코딩 가능.

### 신뢰할 수 없는 코드 실행(유저/에이전트가 직접 작성)에는 보안 문제가 심각하게 대두됨

- 코드 기반 자동화 방안은 사실상 ‘임의 코드’를 곧바로 실행하는 구조이기에, 보안상 치명적인 리스크가 있음.
- 예: 에이전트가 악의적으로 파일시스템 접근이나 시크릿 노출, 무한 루프, 외부 네트워크로 시크릿 전송, 리소스 과다 소모, 크립토마이닝 등 다양한 악성 행위를 할 수 있음.
- 이 때문에 기존 클라이언트(에이전트)는 코드를 받아도 실행하지 못하거나, 전문 DSL, 샌드박스, VM, 수동 코드 리뷰 등 강력한 통제수단을 병행해온 역사가 있음.

### V8 격리환경 기반의 프로그램화된 샌드박스가 안전한 코드 실행을 가능하게 함

- Cloudflare Worker는 V8 기반 ‘동적 워커’를 통해, 외부에서 실행되는 코드(예: 에이전트가 생성한 JS 코드)를 완전하게 격리된 상태에서 안전하게 실행 가능하게 함.
- 예: process.env 접근 불가, 노드 호환성 설정에 따라 권한 차등 부여, 네트워크 접근 등도 세밀하게 가드레일(접근 제어) 가능.
- Cloudflare MCP를 활용하면 전체 API(2,600개 엔드포인트)에도 read-only/read-write 등 세분화된 접근 권한 하에서 동적 코드 실행이 가능함(예: DNS 조회, 워커 배포, 액세스 제어 등).
- 이 구조는 최소한의 서버 설정만으로 수십억 건 요청 처리도 무리 없이 확장 가능함.

### 동적 코드 실행 환경(WorkerD, Deno, Pydantic Monty 등)의 발전이 업계 전반에 걸쳐 가속화됨

- Cloudflare WorkerD, Deno(Deno Run), Pydantic Monty(파이썬 기반 동적 코드 인터프리터) 등 다양한 벤더/플랫폼이 안전한 코드 실행 샌드박스를 경쟁적으로 제공하기 시작함.
- 동적 워커 생성 후 간단한 알골리즘(피보나치 등)에서부터 외부 API 접근까지 모든 작업을 안전하게 자동화할 수 있음.
- 실무에서는 파이썬(Pydantic Monty)처럼 런타임 설치가 필요하거나, 실행이 불안정할 수도 있으나, 전반적으로 코드 직접생성→즉시실행 수요가 빠르게 확장 중임.

### 에이전트 클라이언트 측에서는 프로그램화된 툴 호출 및 미니 스크립트 활용이 핵심 트렌드가 될 것으로 예측됨

- 위험성이 낮아진 환경에서는 사용자/에이전트가 코드를 생성해 클라이언트 측에서 직접 ‘프로그램화된 툴 호출’을 할 수 있게 됨.
- 예: 에이전트가 자동으로 작성한 스크립트를 각종 작업(크론잡, 웹스크래핑 등)에 저장∙활용할 수 있으며, 스크립트가 깨질 때마다 자동 수정→재실행 가능.
- 클라이언트 환경이 점차 경량화되고, 별도의 MCP SDK 관리/상태관리 없이 stateless loop 구조로 대규모(1인당 100개 이상) 에이전트 운영체계로 발전할 전망임.

### MCP 서버/SDK는 미들웨어로 내장, 경량화, 프로토콜 자체가 프레임워크의 표준 옵션으로 자리잡을 것으로 전망됨

- MCP 서버와 SDK는 점점 더 최소화되고, 주요 웹 프레임워크(TypeScript 기반 풀스택 프레임워크 등)에서 native 옵션 또는 미들웨어로 내장될 흐름임.
- 서비스 제공자(API 개발자)는 단순하게 “MCP = true” 플래그만으로 모든 API 엔드포인트를 MCP 서피스로 통합·노출 가능하게 됨.
- 프로그래밍적으로 모든 엔드포인트가 MCP 및 타 툴링(MS tool 등)으로 동시에 서비스될 수 있음.
- 마지막 과제는 SDK 경량화 및 범용 번들화에 남아있으며, 2024년 말에는 대부분의 프레임워크 번들에 탑재될 것으로 전망.

### 대규모 API에서 MCP·코드 기반 인터페이스를 활용하여 확장성과 접근성을 동시에 달성할 수 있음을 강조함

- Cloudflare의 블로그(Code Mode)는 “1,000 토큰 내에 API 전체 노출” 등 대규모 API의 MCP/코드 기반 액세스 구조의 실익을 강조함.
- 코드 기반 툴링 방식은 접근성 개선, 작업 자동화, 사용자 경험 혁신 등 전방위적 파급 효과가 크다고 설명하며, 모든 API 제공자에게 이 방식을 적극 권장함.
