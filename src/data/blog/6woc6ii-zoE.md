## *AI가 스스로 코드를 배포하는 다크 팩토리 구축 (공개 실험)* 핵심 요약

- 본 영상은 AI 코딩 에이전트가 완전히 자동화된 시스템(다크 팩토리)에서 코드의 기획, 구현, 배포, 유지보수를 담당하는 공개 실험에 관한 내용임
- 다크 팩토리는 인간의 개입 없이 AI가 코드베이스를 관리 및 배포하는 개념으로, 기존에는 하드웨어(공장) 자동화 개념에서 유래함
- 영상에서는 AI 코딩 지원의 발전 단계를 “자가 운전 자동차” 단계와 비유하며, 레벨 0(스파이시 자동완성)부터 레벨 5(다크 팩토리)까지 세분화해 설명
- 현재의 AI 코딩은 대부분 인간 검토(휴먼 인 더 루프)가 필수적인 3~4단계에 머물러 있으나, 실험을 통해 완전 자동화 5단계로 도전하고자 함
- 실험 시스템은 Archon(오픈소스 워크플로우 엔진), Claude Code, Miniax M2.7 모델 등을 활용해 구축 중임
- 시스템의 아키텍처는 거버넌스 레이어(목표/규칙 관리), 워크플로우(이슈 선별~검증~배포), 에이전트/모델/자동화 스크립트로 구성
- 거버넌스 레이어에서 미션/공장 규칙/글로벌 규칙 파일(markdown)로 철저한 목표 및 금지사항, 코드 규약, 테스트 절차 명시
- 워크플로우는 깃허브 이슈를 트리거로 triage(이슈 선별), implement(구현), validate(검증)의 흐름을 자동 반복하며, 자동화된 레이블링과 별도 검증 에이전트로 신뢰성 확보 전략을 도입
- StrongDM, Spotify 등 기존 제작사 사례와 구별되는 점은 완전히 공개된 실험 환경과 실시간 참여, 오픈 리포 제공
- 최종적으로 시청자들은 해당 시스템의 실제 데모를 체험하고, 직접 이슈를 제기함으로써 다크 팩토리의 가능성과 한계를 직접 검증할 수 있도록 설계됨

## 세부 요약 - 주제별 정리

### AI 코딩 자동화 실험으로 '다크 팩토리'를 구축하며 완전 자동화를 시도함

- Cole Medin은 AI의 극한 활용을 목표로, 인간의 개입 없이 모든 코드를 자동 생성·배포하는 ‘다크 팩토리’ 시스템 공개 실험을 진행 중임
- 다크 팩토리란, 처음에는 물리적 공장에서 로봇들이 스스로 생산하는 'lights out manufacturing'에서 유래된 개념이며, 이후 Dan Shapiro가 코드베이스/AI에 확장 적용
- 이번 프로젝트는 실시간 스트리밍 및 레포 공개로, 커뮤니티와 함께 시스템을 구축하고 발전 과정을 투명하게 공유(참여형 실험)
- 사용 기술로는 Archon 워크플로, Claude Code(코드 생성 에이전트), Miniax M2.7(저렴하고 빠른 AI 모델) 등이 포함됨

### AI 코딩 도움의 발전 단계를 자율주행에 빗대어 0~5단계로 상세 설명함

- AI 도입 단계를 자율주행 자동차 등급처럼 소개
    - 0단계(스파이시 자동완성): Stack Overflow처럼 AI를 검색·참조 용도로 활용(두 손으로 직접 운전)
    - 1단계(코딩 인턴): 단순·보일러플레이트 코드를 AI로 생성(크루즈 컨트롤)
    - 2단계(주니어 개발자): 일부 반복/간단 작업을 AI에 위임, 인간이 여전히 핵심 제어(한 손 운전)
    - 3단계: AI가 코드 대부분을 생성, 인간이 계획-검토-수정(‘휴먼 인 더 루프’가 필수)
    - 4단계: 별도의 하니스(자동화 프레임워크)로 다양한 작업을 AI가 장시간(야간 등) 자체적으로 처리, 인간 최종 검토(‘수면 운전’ 가능)
    - 5단계(다크 팩토리): 인간 검토 없이 AI가 실행~배포까지 전 과정 자동화(스티어링 휠 없음)

### '다크 팩토리'란 인간 대신 AI가 코드 전 과정을 자동화해 실제 배포까지 책임지는 구조임

- 핵심 특징은 AI가 목표(plain English 설명, 명확한 규칙, 금지사항 등)는 인간이 정하지만, 작성·테스트·버그 수정·배포는 전적으로 AI가 담당
- 기존엔 강력한 자동화 강조 기업(StrongDM, Spotify 등)도 대부분 '휴먼 리뷰' 단계가 있었으나, 본 프로젝트는 검토조차 생략해 극한 자동화를 실험
- 시청자들도 실제 배포 어플리케이션 방문·이슈 제기 가능(공개 실험)

### StrongDM, Spotify 등 실제 다크 팩토리 사례와의 차별점 및 실전 운영에 대한 논의

- 2023년 초 StrongDM 사례: 수천 줄 규모 실서비스 코드를 AI가 완전히 작성/관리(수작업 코딩 금지 규정 도입)
- StrongDM은 내부 사양은 오픈소스 공개했으나, 완전 공개 실험은 아님
- Spotify는 배경 코딩 에이전트 도입 등 시도중이나, 완전 오픈화/실시간 참여성은 구현되지 않음
- Cole Medin의 실험은 완전히 공개적이며 누구나 GitHub 이슈를 제출해 플로우에 기여·감시할 수 있음

### 시스템의 거버넌스 레이어는 목표·범위·금지사항·아키텍처 규칙을 엄격히 정의함

- mission.md : 프로젝트 목표/적용 범위/이용자/핵심 기능 및 절대 허용하지 않는 기능(예: 다른 채널, 모바일앱, 결제, 모델변경 등)
- factory_rules.md : 이슈/PR 라벨링 기준, 구현 절차, 브랜칭, 검증 및 통합 방법, PR 크기 제한(큰 PR 방지로 품질 제어), 필수 테스트 기준
- claw.md(글로벌 규칙): 전체 코드베이스의 기술스택, 레이아웃, 실행-테스트 명령, 코딩/테스트 컨벤션 등 명세
- 이 세 파일은 모든 워크플로의 컨텍스트/context로 항상 로드되어, AI의 의사결정(이슈 triage~PR 생성 등) 명확하게 제한 및 안내

### Archon 오픈소스 하니스로 AI 코드 워크플로를 정형화·자동화함

- Archon: 오픈소스 AI 코드 프로세스 하니스/워크플로 엔진(개발자 직접 제작)
- 사용자는 구현 흐름(이슈 triage, 구현, 검증, 머지 등)을 YAML·스크립트로 정의해, 반복 가능하고 결정론적인 AI-자동화 실현 가능
- 각 단계(이슈 triage/구현/검증 등)는 Archon 워크플로 폴더에 독립적으로 관리, 반복 실행 및 병렬처리 지원

### 실험의 엔트리포인트는 '깃허브 이슈'이며, triage-구현-검증이 자동 루프됨

- 이슈는 사용자가 등록(또는 AI가 자체적으로 리그레션 이슈 생성) → triage 워크플로로 우선순위/수락여부 판단
- triage: mission.md, factory_rules.md 내용을 바탕으로 AI가 “accept/priority/reject” 등 자동 분류, 중복·범위외 이슈는 자동 reject
- 구현: 수락된 이슈별로 Archon 워크플로가 병렬 실행(과제가 여러 개여도 독립 처리)
- 구현 워크플로: 기능/버그분류→리서치·플랜→구현→PR생성→다음단계 전달
- 검증: 별도 Archon 워크플로에서 에이전트 분리(hold-out 패턴)로 bias 억제, 브라우저 자동화(E2E) 등 실제 사용 흐름을 테스트

### 검증 단계는 'hold-out 패턴'을 도입하여 편향/짧은 루프를 방지함

- 구현 에이전트와 검증 에이전트/워크플로가 완전히 분리되어 있음(동일 문맥·편향 영향 차단 목적)
- 검증 AI엔진은 ‘유저 여정’ 및 코드 diff만을 받아, 기능/버그/기준 적합여부를 독립적으로 판단하여 합격/불합격 결정
- StrongDM 사례 인용: LLM이 테스트를 맞춰서 왜곡하거나 문제를 은폐하는 사태 방지
- 검증 통과 시 해당 PR은 자동적으로 main 브랜치에 merge되어, 완전 자동화를 실현

### 전체 프로세스는 스케줄러(크론잡) 기반으로 무한 루프를 돌며, 인간 검토는 완전히 배제됨

- 시간 단위로 triage~구현~검증 워크플로가 계속 순환(자동화된 소규모 공장처럼 운영)
- 향후 리그레션(자동화된 전체 시스템 회귀 테스트 및 신규 이슈 자동생성) 프로세스도 구축 예정

### 실제 구축 예시(깃허브 레포, 워크플로, 이슈/PR 라벨링)와 GitHub/Archon 연동 사례 소개

- 실험 레포의 루트에 세가지 파일(mission.md, factory_rules.md, claw.md) 위치
- 이슈 및 pull request에는 운영 규칙에 따라 라벨이 할당, 승인/거부/우선순위 책정이 자동화됨
- 모든 워크플로(예: triage.yaml)는 GitHub/Archon 연동 스크립트와 함께 배치되어, 개발자는 실제 플로우 아티팩트를 직접 관찰 가능

### 전체 실험은 완전 공개이며, 누구나 직접 이슈 제기 및 다크 팩토리 실체를 체험할 수 있음

- 완성시점엔 배포된 어플리케이션에서 질문/이슈 제출(예: 본인의 유튜브 채널 전용 RAG 기반 튜터)
- 실험 참여를 통해 다크 팩토리의 문제점·한계·장점 모두 투명하게 체험 가능
- 생성형 AI, 코딩 자동화, 오케스트레이션 기술 등을 실제로 학습 및 검증할 수 있는 장으로 기능

### 실험 진행 상황과 향후 계획을 살펴보며 시스템 신뢰성 확보에 주안점을 두고 있음

- 현재 거의 완성 단계이며, 몇 주 내 공개 배포 및 실사용 테스트 시작 예정
- 추가적으로, 전체 시스템을 주1회 등 주기적으로 회귀 테스트하는 프로세스 구축 계획
- 실험 성공을 위해 아키텍처 품질에 각별히 신경 쓰고 있으며, 참여자와 함께 발전해나갈 예정임
