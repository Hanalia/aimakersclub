## *ReviewDebt: 모든 풀 리퀘스트를 점수화하는 실용적 프레임워크 — Sachin Gupta, 이베이* 핵심 요약

- 영상 제목은 "코딩 에이전트가 리뷰 부채(Review Debt)를 만들고 있다"로, AI가 코드 리뷰 부담을 양산하지만 현재 누구도 이를 측정하지 않고 있다는 점을 환기함.
- GitHub 2025년 보고서에 따르면 한 해 동안 커밋 수는 25% 증가했으나, 리뷰 코멘트 수는 27% 감소해 리뷰 활동이 코드 생산 속도를 따라가지 못함이 드러남.
- Faros AI 2026년 벤치마크에선 PR(풀리퀘스트) 리뷰 소요 시간이 441.5% 증가, 미리뷰 병합 PR이 31% 늘어남: AI가 생성한 코드에 대한 인간 리뷰가 불충분해지는 현상.
- PR 당 개발자 수, PR 크기, PR Throughput, AI 사용량 모두 실제로 증가했지만, 이들 수치는 '속도'만 말할 뿐 '신뢰'를 측정하지 않아 실질 의미가 제한적임.
- "리뷰 부채"란, 에이전트가 만든 코드와 인간이 실제로 리뷰·이해·신뢰하는 코드의 누적 격차. 이는 기술 부채보다 더 복합적이며, 인간의 주의(Attention)라는 '이자'가 누적되는 특성이 있음.
- 리뷰 부채는 ①차이 확장 및 결합도, ②테스트 증거 격차, ③디렉토리 및 오너십 범위, ④AI 작성 PR 인디케이터, ⑤증거 및 합리성 격차 등 5개 신호 패밀리를 통한 10개의 결정적(Deterministic) 체크로 수치화 가능.
- 실제 3개 공개 저장소와 524개의 PR을 분석한 결과, AI 작성 여부가 아닌 PR의 복잡도와 구조가 리뷰 부채를 좌우하며, AI 신호는 전체 리뷰 부채의 일부(New)가 아님이 드러남.
- 도구 및 프레임워크 활용 시 실제 팀 데이터로 측정값을 보정하고, 각 PR을 4개 밴드(Zero~High)로 분류해 구체적 행동 지침과 개선 피드백을 제공함.
- 리뷰 부채 관리 시작을 위해 최근 병합된 PR 대상으로 스코어링 시범 적용을 권장하며, 측정—공유—논의하는 문화로의 전환 필요성을 강조함.
- "승인+코멘트" 병합, 품질관리에서 처리, LGTM 등 리뷰 습관의 문제점과, 한 PR당 한 논리적 변경·의미 있는 테스트·명확한 설명 작성 등 실질적 실천 방안도 언급됨.

## 세부 요약 - 주제별 정리

### AI의 코드 생산 가속은 리뷰 활동과 괴리를 만들고 있음

- GitHub 2025년 10월 보고서: 전 세계 거의 모든 공개 PR 대상, 커밋 수는 25% 증가, PR 코멘트(리뷰)는 27% 감소.
- 코드 생산량이 증가하는 동안, 리뷰 활동은 오히려 감소해 두 지표가 정반대 방향으로 움직임.
- Faros AI 2026년 벤치마크: 일반적인 PR 리뷰 시간 441.5% 증가(이전 대비 5.4배 오래 걸림).
- 31% 더 많은 PR이 리뷰 없이 병합되는 현상 발생.
- AI가 PR 생성을 급격히 가속하지만, 인간 리뷰어가 이를 따라가지 못해 "리뷰 부채(review debt)"라는 격차가 생성됨.

### 기존 PR 생산성 지표는 실제 신뢰나 품질 향상과 무관할 수 있음을 설명함

- Faros AI와 DX 2026년 장기 연구: 22,000명 개발자/4,000개 팀, 400개 조직 데이터 기반.
- PR 수/개발자 16% 증가, PR 평균 크기 63% 증가(44줄→72줄).
- PR 오픈-병합 주기(cycle time)는 8% 단축, AI 사용 65% 증가.
- 하지만 이들 지표는 진정한 코딩 '효과'가 아닌 '허상 지표'로, PR 분할·크기 증가·리뷰 압박 없어진 결과일 뿐.
- 속도 지표는 "신뢰의 속도"(리뷰·고려·합의 속도)를 말해주지 않음.

### 엔지니어 팀에서 실제 발생하는 리뷰 부채의 증상과 문제점이 구체적으로 제시됨

- 리뷰어 피로 증가: 1년 전보다 훨씬 더 많은 리뷰를 부담.
- 미리뷰 PR의 야간 급병합, 마감 직전 심야 승인 등 품질 희생 사례 빈번.
- Test theater(테스트 극장): 실제 코드 '행위'만 검증, '예상 동작' 검증 미흡으로 버그가 테스트에 잠김.
- 아키텍처 분산: 동일 문제에 각기 다른 해결법을 적용, 구조적 일관성 부족.
- 사고(Incident) 발생 후 원인 PR로 역추적이 어려워지는 딜레이 현상.
- 이로 인한 비용과 문제점은 기존 대시보드에 잘 드러나지 않음.

### 리뷰 부채(Review Debt)의 정의와 누적(Compounding) 특성이 상세히 분석됨

- 리뷰 부채: 에이전트가 생산한 코드와 인간이 실제 리뷰·신뢰·이해한 코드 간 누적 격차.
- 기술 부채(TD)와 어감은 유사하지만, "관심(Attention)"이라는 이자가 복리로 붙는다는 금융 부채적 특성이 중요.
- 3가지 피드백 고리로 누적 가속:
    1. 에이전트는 리뷰되지 않은 코드를 학습, 다음 PR에 결함 전파.
    2. 리뷰어가 구조적 리뷰를 포기하고 문법/버그 위주로 축소 평가.
    3. 리더십은 생산성 증대를 기대, 리뷰/검증에 추가 인력을 투입하지 않음.
- 각각은 별개론 생존 가능하지만, 복합 작용 시 심각한 부채 폭증.

### 리뷰 부채 측정을 위한 5개 신호 패밀리와 결정적(Deterministic) 체크 개념이 구체적으로 제시됨

- 5대 신호 패밀리: (각각 2개, 총 10개의 체크)
    1. **Diff(차이) 크기 및 결합도**: 순 변경 줄, 파일 수, 한 모듈/여러 모듈 분산 여부.
    2. **테스트 증거 격차**: 추가된 테스트 줄/코드 줄 비율(테스트는 있으나 주요 동작 검증 놓치는 경우 많음).
    3. **디렉토리 및 오너십 확산**: 여러 팀 소유 파일 교차 시 리뷰 비용 급증.
    4. **AI 작성 PR 인디케이터**: co-authored-by footer, 브랜치명 패턴(codex/copilot 등), 커밋 메세지·PR 본문 포맷 등.
    5. **증거 및 합리성 격차**: PR이 '무엇'이 아닌 '왜'를 잘 설명하는지 여부.
- 모든 체크는 PR과 저장소 정보로 결정적(Deterministic) 계산 가능, LLM 판정 불필요(모델 변경에 따른 불안정성·검증 불가성 제거).

### 구체적 신호별 설명 및 실제 사례 데이터 분석이 포함됨

- **Diff 크기 및 결합도**: AI는 현상(fix at call site) 위주라 여러 파일에 분산, 인간은 루트 원인 수정 경향.
- **테스트 증거 격차**: AI는 산출코드 대비 테스트 줄 비율 낮음, 테스트 있어도 동작 그대로 검증만 할 때 많음(버그도 검증에 포함될 수 있음).
- **디렉토리/오너십 확산**: 여러 팀/디렉토리 걸치는 PR의 리뷰 비용·커뮤니케이션 부담이 조사됨.
- **AI 작성 인디케이터**: Co-authored footer가 강한 신호, 일부 저장소는 차단(Repo C).
- **증거·합리성 격차**: 제목/본문/커밋메시지에서 '왜'와 '무엇'의 정보량, 예시로 "Fix leaky test" vs. 원인, 증상, 해결방안, 벤치마크까지 서술된 경우 등 비교.

### 실제 PR 스코어링 및 프레임워크 적용 사례로 점수와 행동 피드백을 보여줌

- 자체 CLI 기반 툴을 이용, 3개 PR을 나란히 스코어링(0~100점; 구조별 차이 확인).
- 점수 0: 리뷰 부담 거의 없음(6분 소요 추정), 추가 행동 지침·경고 없음.
- 점수 60: '증거 필요', 검토 시간 86분. 구체 신호(‘AI 작성 소프트 인디케이터’ 등)와 리뷰어/작성자 각자 다음 행동 권장사항 제공.
- AI 작성된 PR이더라도 테스트 등 구조가 좋으면 점수 7 등 '낮은' 부담, 신호는 단순 정보(벌점 아님).
- 리뷰 부담은 복잡도·구조가 주도하며, AI 인디케이터는 부차적 영향.

### 524개 공개 PR 분석을 통해 리뷰 부채가 PR 볼륨과 구조 복잡도의 함수임을 입증함

- 3개 공개 레포, 524PR, 27~90일간 분석. AI 작성 PR 비율 5~20%로 안정적이나 전체 리뷰 부채는 볼륨/복잡도 따라 변화.
- 한 레포는 27일간 186시간(시니어 리뷰어 기준), 다른 레포는 43시간 소요, PR량 차이에 기인.
- 4건만이 'Needs evidence' 또는 'High' 밴드(구조적 변경/대규모 리팩터).
- 실제 데이터서 PR 복잡도와 구조가 리뷰 부담을 결정(AI 비율은 직접 영향 적음).
- 최고 부담 PR의 경우 단일 PR에 84시간 추정 리뷰 필요(점수 73).

### 리뷰 부채 해소를 위한 실질적 조치와 팀 적용 권고가 정리됨

- 한 PR당 한 논리적 변경만 담도록 제한; 작은 PR이라기보다 '의미 있는 변경 단위' 유지.
- 테스트는 반드시 변경사항과 함께 제출; 특히 "코드가 수행하는 동작"이 아니라 "코드가 수행해야 할 동작"에 대한 검증 필요.
- 한 팀 소유 디렉토리(오너십)에 국한되도록 범위 관리; 교차팀 작업은 분할하여 각기 독립 PR로.
- PR 본문의 '이유/배경'은 인간 작성자가 명확히 서술; PR 본문을 에이전트에게 위임하지 않기.
- AI 작성 PR도 동일한 리뷰 기준 적용, 신호(Amplifier)는 나머지 4가지 체크 기준이 약할 때만 영향력 발생.

### 프레임워크 도입 방법과 리뷰 부채 관리 절차가 구체적으로 제안됨

- 최근 200개 병합 PR에 대해 백필(Backfill)로 스코어 일괄 적용 후, 최고 점수 PR 검토.
- 기준선(Threshold) 설정: 점수 50 이상 PR에선 반드시 작성자의 코멘트(증거) 요구 등 규칙 도입.
- 모든 PR에 스코어 공개(서피스), 리뷰 방해가 아닌 가시성 제고.
- 팀별로 주간 단위 집계(aggregate), 리뷰 부채의 변화(기울기)를 관리자가 모니터링.
- 회의/회고 등에서 정기적으로 이 숫자를 토론, 측정→논의→개선 문화 구현.

### 측정 기반 논의 전환의 중요성과 장기적 거버넌스 체계 전환의 필요성이 제시됨

- 숫자 없는 '왜?' 중심 논의 탈피, 측정 기반 '무엇' 논의 촉진.
- AI 도입으로 산출량 증가와 함께 팬덤과 부채 적립을 동시에 고려해야 함.
- 2027년엔 코드 신뢰(거버넌스) 논의로 전환 예고: AI 작성 변경의 문제 발생 시 신뢰성·책임·감사추적의 근거가 필수.
- ReviewDebt 점수화가 "생산성 이득"과 "신뢰 기반 위험"간 균형, 거버넌스로의 연결고리를 제공.

### 실제 프레임워크 실천을 위한 체크리스트와 안티패턴 주의사항이 정리됨

- 20개 내외 최근 PR을 샘플로 5개 신호 패밀리 점수 적용해 실제 점수 측정 경험 권장.
- 리뷰 부채의 '증가 기울기(slope)' 주간 단위로 시각화, 팀 차원의 논의로 확장하는 것이 중요.
- "LGTM", "QA에서 확인", "코멘트와 동시에 승인" 등 최소한의 리뷰로 승인하는 관행 지양.
- 논의한 습관·원칙 준수와 실제 적용을 강조하며, PR 리뷰 문화의 구조적 변화를 촉구함.
