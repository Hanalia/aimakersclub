## *에이전트 평가(evals) 운영의 성숙 단계* 핵심 요약

- 이 영상은 Phil Hetzel(Braintrust 소속)이 에이전트 품질 평가(evals)를 실제 운영하며 경험하게 되는 성숙의 4단계와 각 단계의 특징·실천법을 심도있게 설명함
- 평가(evals)는 에이전트 품질 확보와 예측 불가한 실제 사용 상황 등 다양한 리스크(평판, 비용, 컴플라이언스 등) 방지에 필수적임을 강조
- 첫 번째 단계는 ‘vibe check’ 등 비공식적이고 인간 기반의 테스트(10개 내외 입력, 전문가의 평점·정당화)로 시작함을 인정, 이때 ‘정당화’ 데이터를 반드시 축적해야 함
- 두 번째 단계에서는 LLM(judge)나 코드 기반의 자동화된 평점 도입, 실제 UAT 혹은 운영 트레이스 데이터 수집 및 eval-dataset 구축으로 평가 신뢰도를 높임
- 평가 기법은 단위 테스트와 달리 모든 경우의 수를 테스트하지 않고, 주요 failure mode 위주로 설계 → 전문가의 암묵지(정당화, 실전 실패 사례 파악 등) 축적이 핵심임
- 도구(tool) 호출, 외부 시스템 연동 등 복잡도 증가에 따라 전체 트레이스 평가, 시스템 상태 보존·모킹(Mocking for APIs) 등 새로운 기술 과제가 등장함
- 외부 시스템 상태 재현을 위해 벡터DB의 버전·타임스탬프 쿼리, 모킹 등 고급 기법 활용 → 실제 운영 환경과 농후한 동등성 추구
- deterministic(코드 기반) 평가와 LLM judge 기반 평가 모두 혼용하되, LLM judge 역시 추가 eval로 신뢰성 보완 필요
- 평가 데이터셋 축적, 프로덕션 트레이스 재활용, 지속적 평가 자동화가 장기적 개선의 핵심 루프(‘flywheel’)임을 강조
- 향후 트렌드로 토픽 모델링 통한 자동 failure mode 발굴, eval provider CLI/클라우드 코드로 대규모 자동화 등이 대두됨

## 세부 요약 - 주제별 정리

### 발표자는 ‘에이전트 품질 및 평가’를 현업 경험 기반으로 체계적으로 소개함

- 발표자 Phil Hetzel은 Braintrust의 솔루션 엔지니어링 리더임
- 12년간 컨설팅(4년: KPMG, 8년: Slalom Consulting) 경력, Slalom에서는 글로벌 Databricks 유닛 리더였음
- 고객들이 생성형 AI 개념검증(PoC)은 잘하지만 실제 프로덕션 전환이 부족함을 경험함
- Braintrust를 직접 사용하며 에이전트 평가 관리의 필요성을 실감, 결국 입사
- Braintrust는 에이전트 품질에 초점을 두며, ‘evals’(운영 전 평가)와 ‘observability’(운영 중 모니터링)를 동일 문제로 접근함

### 평가(evals)는 에이전트 품질, 리스크 관리 및 지속적 개선의 출발점임

- 에이전트가 실제 사용 환경에서 기대하는 동작을 하는지, 실사용자 및 사용 파생 리스크를 점검하기 위해 평가가 필요함
- 평가 실패 시 생길 수 있는 주요 리스크: ① 고객에게 불친절·비효율로 브랜드 평판 위험 ② 과도한 운영 비용 ③ 규제·법무 이슈
- 평가는 디펜스(리스크 예방)와 오펜스(모델 개선의 계기, 변경효과 수량화) 양면 역할을 가짐

### 평가 방식은 단위 테스트와 달리 ‘모든 경우의 수’가 아닌 ‘핵심 실패 모드’에 집중해야 함

- 유닛 테스트는 가능한 모든 코너케이스를 당연히 작성하지만, 에이전트 평가는 무한에 가깝기 때문에 불가능
- 현실적으로, 각 에이전트 내 위험(failure mode)을 전문가/sme와 정의 후 이에 대한 케이스에 집중한 평가 설계를 권장
- 평가결과는 때로 완전히 정확하지 않아도 되며(특히 LLM judge 등 활용시), 방향성 있는 개선 여부가 더 중요함

### ‘평가 프로세스’는 ‘테스크-데이터셋-평가함수’ 구조로 이루어진다

- 평가 구성요소 ① Task: 테스트할 에이전트나 프롬프트 ② Data set: 케이스 (입력 예시들의 집합) ③ Scoring function: task의 결과물 품질을 재는 함수(객관식/LLM judge/커스텀 코드)
- 구조적으로 접근해야 각 단계별로 복잡도를 올리거나 자동화할 수 있음

### 첫 단계는 비공식적 ‘vibe check’이지만, 주관적 평가 근거와 정당화 기록이 반드시 필요함

- 초보자는 10개 내외의 입력에 단순히 좋아요/싫어요(thumbs up/down) 평가가 흔함(비공식적 vibe check)
- 다만 반드시, 평가자가 선택 이유/정당화(justification)을 기록해 추후 암묵지 전수 및 자동화 근거로 활용해야 함
- 이 과정에서 실제 도메인 전문가의 참여가 가장 바람직
- 플랫폼(BraintTrust 등)도 사용자별 커스텀 애노테이션 UI 제공이 유리함(일반화된 UI는 비효율적)

### 정당화/평가 내역을 구조화·자동화하면, failure mode 파악과 LLM judge/코드 평가로 확장 가능해짐

- 인간 평가의 ‘정당화’ 데이터는 코드나 클라우드 코드, Codex 등과 연계해, 왜 실패했는지 유형별 failure mode 도출로 발전
- 이후 엔지니어가 아닌 사람 없이도, failure mode 별 자동 평가 가능(LLM judge·deterministic code 혼합)
- LLM judge(모델 평가를 또 다른 LLM이 담당) 방식은 기존에도 오랫동안 활용되었고, 확장성/효율성이 높음
- 단, LLM judge의 평가 신뢰성은 반드시 별도 eval이나 사람 판정과 비교·보완해야 함(‘Judge를 평가하는 평가’ 필요)
- Eval 데이터셋엔, 실제 서비스 운영(trace) 혹은 UAT 데이터를 활용하는 것이 가장 실제성 높음. 테스트 환경이 아닌 운영 환경 반복·시뮬레이션 개념으로 전환 필요

### 트레이스 기반 데이터 축적 및 평가 자동화는 품질 개선의 선순환을 만든다

- ‘Flywheel’ 프로세스: ① 운영(프로덕션)에서 트레이스 수집 → ② failure mode 분류/자동화 평가 → ③ 데이터셋에 추가 → ④ 오프라인에서 재평가/시나리오 개선 → ⑤ 다시 실서비스 개선
- 이 선순환된 루프가 실제 에이전트 품질 개선의 핵심 동력으로 작동

### 평가 복잡도가 올라가면 도구, 외부 시스템 연동 등 새로운 과제가 생긴다

- 단순 모델-결과 평가를 넘어, 실제로 데이터베이스(RDB 등)·외부 시스템과 연계되는 에이전트 평가에서는 두 가지 문제가 발생
    - ① 에이전트 전체 트레이스 캡처·분석이 필요
    - ② 외부 시스템의 당시(타임스탬프 시점) 상태와 동등성을 오프라인 환경에서 재현해야 함
- 이를 위해, Mock API·스테이징 등으로 실제 환경 유사도 확보, 혹은 트레이스 내 상태(State)를 캡처해 Task에 주입하는 방식을 사용
- 크루드(CRUD) 툴 평가 시, 운영 데이터 덮어쓰지 않도록 세심한 설계 필요

### 외부 시스템 상태 동기화·재현에는 버전팅, 타임스탬프 쿼리 등 고급 기법이 필요하다

- 예: 벡터DB·DB 등에서 트레이스 기록 시점 기준 시간별 버전 쿼리
- 상태(state)를 트레이스에 대거 담아, 실제 환경 동등성 확보
- 복잡성과 비용이 늘지만, 실제 품질 평가의 신뢰도를 높이는 필수적 단계

### LLM judge와 deterministic grading(코드) 혼용 시, 각각의 장단점을 살려야 함

- 일부 평가는 정량적(코드로 확정, 예: 도구 호출횟수 초과, 토큰 과다 사용 등) 방식이 신뢰도 높음
- 반면 주관적/언어적 평가는 LLM judge의 스케일·유연성이 우수
- 업무 특성/평가 목적에 따라 적절히 혼용해야 하며, LLM judge 결과는 별도 ground-truth 데이터셋과 비교 검증 필요
- "Eval the eval as an eval"(평가자를 다시 평가) 원칙 강조

### 향후 논의되는 첨단기법: 주제 모델링, 대규모 자동화 평가, 생산성 극대화 등

- 토픽 모델링 등 자동화 기술로 프로덕션에서의 새로운 failure mode 발굴 시도
- Eval Provider CLI, 클라우드 코드 등으로 평가 배치·자동화 확장을 가속화
- 복잡도, 운영 신뢰도, 업무 효율성 모두가 중요 지표로 부상

### 질의응답 및 마무리: 평가 기준의 주관성과 자동화 기법 혼용을 적극적으로 권장함

- 질문: Deterministic grader(코드 기반) 선호 분위기에 대한 의견?
- 답변: 일부 평가는 주관적일 수밖에 없고 그것이 에이전트/AI의 매력. 하지만 LLM judge도 충분히 유효, 단 룰만 맹신해서는 안 되며, 인간 기준과 align되는 재평가 필수
- LLM judge는 discrete output을 내므로 ground truth 대비 비교·검증이 상대적으로 수월
- 마지막까지 참가자 부스 방문 권유로 실무 팁·후속 논의 장려
