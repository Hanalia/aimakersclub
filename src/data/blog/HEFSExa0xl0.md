## *코딩 에이전트에게 스프레드시트를 가르치는 4개월의 여정* 핵심 요약

- 본 영상의 제목은 "*코딩 에이전트에게 스프레드시트를 가르치는 4개월의 여정*"입니다.
- 발표자는 Nuno Campos(Witan Labs)이며, 지난 4개월간 코딩 에이전트가 스프레드시트 작업을 고도화하는 과정과 그 과정에서 얻은 교훈을 공유합니다.
- 목표는 코딩 에이전트가 Python·JavaScript 수준으로 스프레드시트를 자유롭게 다루게 하는 것이었습니다.
- 초기 내부 벤치마크(재무 분석 문제)에서 정답률 50%에서 시작해, 다양한 시도와 개선을 거쳐 결국 92%까지 달성했습니다.
- 여러 에이전트 구조, 스프레드시트 데이터 표현 방식(SQL, XML, CSV/TSV, HTML 등) 등을 실험했으나, 대부분 단독으로는 한계가 있었고 조합적 활용에 의미가 있었습니다.
- 결정적 진전은 15개의 별도 도구 대신, JavaScript 기반의 REPL(Node.js 환경) 하나로 모든 기능을 통합하면서부터 일어났으며, 에이전트가 직접 스크립트를 조립·활용하는 환경을 제공했습니다.
- 피드백 루프(계산 엔진 및 렌더 엔진)를 통한 결과 검증이 중요한 성능 향상 요소였으며, 도메인 지식 강화를 위한 프롬프트 개선 역시 성과에 기여했습니다.
- 평가(에이전트 성능 측정)는 초기에 LLM 판단에 의존했으나, 후반에는 입력·출력 정답셋을 활용한 결정적(deterministic) 평가 방법 확립을 통해 신뢰성을 높였습니다.
- 총평으로, 최고의 인터페이스 선정, 효과적인 피드백 루프 구축, 도메인 지식 강조, 평가체계 고도화가 다양한 영역에 걸쳐 일반화될 수 있는 교훈임을 강조합니다.


## 세부 요약 - 주제별 정리

### 스프레드시트를 AI가 다루는 것은 생각보다 훨씬 어려웠음

- 스프레드시트는 비정형적·시각적 구조가 강해 인간에게는 매우 직관적이나, LLM에게는 구조와 맥락을 파악하기 어렵게 함.
- 예를 들어 “revenue” 항목만 해도 gross/net 등 다양한 형태·기간(분기/연도)·입력/수식 구분 등이 복합적으로 존재.
- LLM은 사용자가 원하는 데이터를 제대로 판별·추출하는 데에서 충분히 맥락을 파악하지 못함.
- 이로 인해 단순 언어 처리 이상의 정교한 접근법이 요구됨.

### 초기에 시도한 에이전트(멀티 에이전트) 구조는 유연성 부족 등의 한계가 드러남

- 작업을 세 에이전트로 분배: 핵심은 5단계(결과 정의→계획→실행→검증 등)로 움직이는 에디트 에이전트.
- 이 구조는 에러의 유형을 바꾸는 데는 도움됐으나, 각 단계(특히 discovery)가 앞에 한번만 실행되어 이후 재탐색/수정 어려움.
- 개별 에이전트 간 맥락(context) 공유가 제한되어 종합적 작업에 비효율.
- 결과적으로 지나치게 경직되고 재사용성이 낮아 결국 폐기된 방법임.

### 스프레드시트 데이터 표현방식(구조화)의 모든 조합을 실험했으나, 단독 활용에는 한계가 있었음

- SQL: LLM에 친숙하며 구조적 데이터 다루기에 좋을 것으로 기대. 그러나 스프레드시트의 잡다한 문제를 해결하는 데에 한계.
- XML: 실제 Excel 파일의 내재 구조이나, 실제 작업에선 쓸모가 없었음.
- CSV/TSV: 특정 시트·영역에선 유용했으나, 스프레드시트 전체 맥락 이해에는 부족.
- HTML: 레이아웃 및 시각적 요소 표현에 도움되어, 후술하는 렌더 엔진 구축으로 연결.
- 결정적으로, 오직 하나의 방식으로만 접근하는 것은 한계가 있었으며, 조합적으로 활용 시 장점이 발현됨.

### 핵심 개선: 다수의 도구를 하나의 JavaScript REPL로 통합하여 작업을 비약적으로 효율화함

- 초기에는 15개의 개별 도구로 스프레드시트 탐색·처리, 도구마다 순서대로 실행하다보니 속도 저하·타임아웃 빈번.
- 자바스크립트 REPL(Node.js)을 단일 인터페이스로 구축, 기존 도구들은 각기 자바스크립트 함수로 변환됨.
- 코딩 에이전트는 REPL 내에서 자신만의 스크립트를 조합, 필요한 작업을 동시실행(one-call) 가능.
- 실제 작업 속도, 타임아웃 감소, 에이전트의 유연성 모두 극적으로 개선됨.
- 자바스크립트는 LLM이 친숙하며 샌드박스 실행이 용이, 실제 데이터 처리 및 엔진은 C# 등 최적언어로 백엔드 구현.

### REPL 구조와 코드 모드의 차이, 그리고 상태 유지의 중요성

- '코드 모드'는 여러 기능을 한 코드 안에 묶어 실행할 수 있음(Anthropic API, Cloudflare 등에서 도입 사례).
- REPL(REPL loop)은 코드 모드보다 한 발 더 나아가 '상태유지(persistent state)' 가능.
- 즉 한 번 정의한 변수, 계산 결과 등을 다음 명령에서도 유지·재사용할 수 있음.
- 코드 모드에서는 50줄 가까운 긴 스크립트 빈발→REPL 도입 후엔 더 짧게, 명령-추론-실행의 루프 효율화.
- 결과적으로, 에이전트는 더 빠른 피드백 사이클을 통해 답변 정확도 및 처리 속도를 높임.

### REPL 통합 구조는 도구 확장 및 유지보수, 프롬프트 디자인 모두 용이함

- 기존 방식에서는 새로운 기능 추가 시마다 별도 도구(툴) 구현·스키마 등록 등 번거로움 존재.
- 통합 REPL에서는 얼마든지 자바스크립트 함수 추가만으로 에이전트 도구 확장 가능.
- 타입 정의(TypeScript type definitions) 프롬프트 삽입만으로 에이전트에 새 행위 부여가 매우 간단.
- 이는 에이전트 인터페이스의 발전 가능성 및 유지보수 효율성을 극대화함.

### 피드백(검증) 루프: 고성능 계산 엔진과 렌더 엔진이 정답율 및 신뢰성에 절대적으로 중요함

- 코딩 에이전트(Claude, Codex 등)도 컴파일/린트/테스트 등 반복 검증 루프를 통해 성능이 극적으로 오른다는 사실에 착안.
- 스프레드시트 분야에서는 '수식 연산 엔진'(Excel 호환성 보장)과 '렌더 엔진'(스프레드시트의 시각화 포함)이 역할 수행.
- 렌더 엔진은 영역별 시각화(레이아웃·서식 등)를 이미지로 제공, 진짜 스프레드시트처럼 검증 가능.
- 수식 처리 능력이 엔진 신뢰도에 직접 영향: 수식 50%만 지원 엔진 사용시, 에이전트가 실제로는 맞는 답을 내도 엔진 채점에서 실패하는 등 문제 야기.
- 엔진 완성도와 피드백 루프의 선순환이 정답률 상승의 기본 조건.

### 벤치마크 점수, 타임아웃 현황, 구체적 개선 사례

- REPL 도입 전 목표 벤치마크 정확도: 50→ REPL 도입 후 74%, 이후 세부개선 누적으로 92% 달성.
- 추가 개선 추진 사항 예시: 퍼지 검색(fuzzy search), 수식 추적/의존성 파악 함수, 시스템 프롬프트 강화, 버그 수정 등.
- 전체 작업당 5분 제한을 두었으나 REPL 적용 후 사실상 모든 타임아웃 문제 사라짐(효율성 대폭 상승).

### 도메인 지식을 프롬프트에 명확히 반영하는 것이 보편적 효과를 발휘함

- LLM은 'revenue', 'ARR' 등 기본 정의를 알고 있지만, 해당 작업 맥락에서 집중해야 할 개념·용어를 프롬프트에 명확히 제한해주는 것이 중요.
- 도메인 지식 강화 프롬프트는 툴, REPL, 단독 기능 등 어떤 인터페이스와도 범용적으로 작동함.
- 작업 맥락에 따른 '주제 한정'이 오히려 에이전트의 집중도 및 정확도 상승을 가져옴.

### 정밀한 평가체계 구축은 성능 향상에 있어 매우 중요한 기반 역할을 함

- 초기엔 LLM 판별(LLM-as-a-judge)로 평가했으나, 가끔 평가결과 변동이 에이전트 때문인지 평가기 때문인지 구분 어려움 존재.
- 입출력 값이 명확한 '골든 시트' 기반 블랙박스 평가 방식을 도입: 동일 입력값 투입시 에이전트 산출물과 정답산출물이 일치하는지 비교.
- 결정적(deterministic) 평가를 가능하게 하는 인프라와 테스트셋의 중요성 강조.
- 평가체계의 발전이 새로운 표현 형식(SQL, CSV 등)이나 구조 변화 실험에도 객관적 성과 검증을 가능하게 함.

### 에이전트 행동 오류 중 상당수는 LLM 성능 문제가 아니라 인프라/도구/프롬프트 결함에서 비롯됨

- 현장에서는 추론 실패로 보이지만, 실제로는 코드나 툴의 버그, 프롬프트 예시 오류, 다운스트림 도구 실패 등으로 인한 문제 빈발.
- LLM은 주어진 환경과 예시를 “매우 충실히” 따라하는 특성이 강함.
- 따라서 각종 트레이스 확인, 로그 분석, 기저 인프라 점검이 필수적.
- 진짜 에이전트의 한계와 주변 환경의 문제를 구분해서 해결 접근 필요함.

### 실험을 통해 다른 도메인에도 확장 가능한 일반화 교훈이 도출됨

- 반복적 툴 호출(연속/병렬)은 비효율적·응집력 낮은 '나쁜 스크립트 언어'이므로, 직접 스크립트 환경(REPL, 코드 모드 등) 제공이 날카로운 해법임.
- 피드백 루프(계산/렌더 엔진 등)의 중요성: 없다면 직접 엔진 구현도 고려 필요.
- 최적 인터페이스 설계가 성능의 열쇠이지만, AI 모델 능력 발전에 따라 재설계는 언제든 필요.
- 사소해 보이는 단계적 계획(think before you act)의 효과 과소평가 금지.
- 도메인 지식 강조 프롬프트, 체계적이고 결정적인 평가체계, 그리고 충분한 트레이스 확인이 지속적 성능 개선의 기본임.
