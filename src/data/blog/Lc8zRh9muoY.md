**채널명:** AI Engineer

## *실서비스에서 에이전트가 재현 불가한 버그를 내면, 디버깅은 어떻게 해야 하는가?* 핵심 요약

- 영상은 실제 프로덕션 환경에서 AI 에이전트가 오류를 일으켰을 때 “재현 불가” 문제가 어떻게 발생하는지, 그리고 이에 대응하는 실질적인 디버깅 및 테스트 방법을 다룸
- 단순히 프롬프트를 다시 활용해 재현하는 것은 실제 문제 발생 시 흔히 실패하며, 한 번의 치명적 오류는 재현이 불가해 디버깅이 매우 어려워진다고 강조
- “온도(temperature)를 0으로 설정해 결정론적 운영을 꿈꾸는 것은 잘못된 접근”임을 수치와 최신 LLM 구조(예: MOE 아키텍처, GPU 연산 비결정성)로 설명
- 문제의 핵심은 “모델 결정론”을 추구하는 것이 아니라 “실제로 발생한 실행(run)을 관찰하고 복기 가능한 형태로 기록하는 것”이라는 점을 주장
- 비트 단위 결정론(입력 = 출력)과 리플레이어빌리티(실행 세션 복기)는 다르며, 실생성 내용을 모두 시스템적으로 추적하여 검증 환경으로 활용하는 것이 중요함을 강조
- ‘Boundary’라는 어노테이션 기법으로, 각 에이전트 워크플로우의 노드별 입출력과 메타정보(모델 버전, 코드 버전 등)를 모두 기록하는 구조와 예시(주식 매도 에이전트 문제)를 소개
- 실제 오류를 내고 동작을 기록한 “트레이스(trace)”를 활용, 코드 수정 후 재생(run replay)과 테스트 자동화(assertion)까지 가능한 흐름을 시연함
- 결정론적/행동적 테스트(Behavioral Testing)의 구분과, Chronicle·Boundary 도구가 결정론적 테스트 자동화에서 가지는 의미를 설명
- 주된 시사점: “비트 단위 결정론 추구는 의미 없으며, 관측-기록-재생 기반의 실질적 디버깅&검증 루프가 현명한 접근”임을 끝으로 조언
- 영상 말미에는 Chronicle 코드 및 문서화 리소스를 QR코드로 안내하며, 실제 도구 활용을 독려함

## 세부 요약 - 주제별 정리

### 실제 프로덕션에서는 에이전트가 동일한 버그를 재현하지 못할 수 있어 디버깅이 무력화됨

- 실서비스에서 AI 에이전트가 잘못된 행동(예: 잘못된 툴 호출, 잘못된 결과 작성)을 했을 때 팀은 원인 파악에 착수함
- 일반적 엔지니어의 반응은 원본 프롬프트를 로그에서 뽑아, 같은 모델에 전달하여 재현해보는 것임
- 그러나, 재실행할 때마다 제대로 동작하며 오류가 재현되지 않는 경우가 빈번함
- 즉, 실제 프로덕션에서 한 번 발생한 버그는 “재현 불가” 상황에 빠지며, 이 경우 디버깅이나 서비스 개선이 불가능해짐

### 온도를 0으로 낮추거나 입력 동일성을 기대해도 결정론적 동작을 보장할 수 없음

- 많은 팀들은 결정론적 동작을 얻기 위해 모델의 temperature 값을 0으로 설정함
- 하지만 temperature 0 상황에서도 동일한 프롬프트, 동일한 코드 상태에서도 서로 다른 결과가 나옴
- Reddit, Hacker News 등에서 실제 사례 및 실험 데이터를 보면, 동일 프롬프트 1,000회 반복에서도 다양한 출력이 존재
- 원인은 GPU 비결정적 연산(floating point 연산, 연관법칙 불일치, 배치 그룹핑), 그리고 MOE 구조의 라우팅임
- 즉, 결정론적 샘플링과 시스템 전반의 결정론성(system determinism)은 별개의 문제임

### LLM의 비결정성은 하드웨어와 아키텍처적 특성이 원인임

- floating point(부동소수점) 연산의 비연관성(덧셈 순서에 따라 결과 변동)이 결과 토큰에 영향을 미침
- GPU의 행렬 연산은 동일한 연산을 단독 실행해도 수치상 아주 작은 차이로 인해 결과가 달라질 수 있음
- MOE(Mixture of Experts) 아키텍처에서는, 한 배치에 함께 묶이는 요청에 따라 라우팅 결과가 달라지고, 각 expert별로 용량 제한이 달라 결정에 영향을 줌
- 최종적으로, 모델의 ‘동일 입력=동일 출력’이라는 가정이 깨지며, 시스템의 비결정성이 근본적으로 출현함

### 결정론적 결과 재현이 아니라 “실제 실행(run)의 상태 변화” 기록이 본질적임

- “매번 동일한 토큰이 나오게 하는 것”이 목표가 아니라, “시스템이 정확히 동일한 상태 변화를 실행”한 것이 중요한 기준임
- 많은 엔지니어들이 ‘모델을 결정론적으로 만들자’에 집착하지만, 정작 중요한 것은 “재생불가 실행을 어떻게 디버깅하고 검증할 것인가”임
- 현실적으로 비트 단위 결정론(입력=출력=같음)은 불가능/불필요하고, 대신 “실행 리플레이어빌리티(Replayability)”가 실무에 중요함

### “비트 단위 결정론”과 “리플레이어빌리티”는 엄연히 다른 개념임

- 비트 단위 결정론: 동일한 입력에 동일한 출력을 생성(이것은 controllability, 주로 연구 혹은 특수 하드웨어 환경에서나 가능)
- 리플레이어빌리티: 실제 실행 트레이스를 남겨 식별/검증 가능한 상태(즉, observability) 확보
- 실생성 과정의 무작위성이 창의성을 제공하므로, 이를 억제할 이유 없음
- API 환경에선 비트 결정론이 실질적으로 불가능함

### 에이전트 워크플로우 경계(boundary)를 기록해 전체 실행 상태를 추적할 수 있음

- 네트워크 패킷 단위가 아니라, 워크플로우의 메서드 단위(툴 호출, LLM 콜, RAG 조회 등)에서 입출력과 메타데이터를 모두 기록해야 함
- ‘Boundary’라는 어노테이션으로, 모든 워크플로우 노드별 입출력과 상태(모델 버전, 빌드 ID 등)를 자동으로 캡처
- 이는 향후 버그 재현 및 검증 자동화(test automation)에 활용할 수 있도록 트레이스(trace)를 생성

### 실제 주식 매도 에이전트 사고 시나리오로 트레이스와 리플레이를 시연함

- 사례: 사용자가 “$1,000 어치 주식 판매”를 요청했으나 LLM이 ‘1000’을 단순 수량(quantity)으로 해석하여, 1,000주(주당 $190 가치, 총 $190,000) 매도 주문을 잘못 실행하는 사고 발생
- API 호출 당시 200 OK 응답, 실패 신호 없음, 대시보드도 정상(문제 탐지 불가)
- 사고 발생 후, Boundary 어노테이션으로 각 노드(툴/LLM)별 입력, 출력, 모델 및 코드 버전, 샘플링 설정 등 상세 트레이스 확보 가능
- 각 노드별 상세 JSON 데이터, 메타데이터를 확인하여 오류 위치(툴 콜 생성 과정의 LLM 오류)까지 신속하게 추적

### 트레이스 저장-재생과 스텁 주입(stubbing)으로 코드 변경 후 안전하게 자동 테스트 가능

- 스텁과 트레이스 리플레이를 결합해, 문제 위치만 실제로 재실행하고 나머지를 기록된 대로 복제하여 전체 오류 맥락을 재현
- 예시: 코드 수정 및 가드레일 추가 후, LLM 노드는 스텁(stub) 처리하고, 툴만 실제 실행하여 실행 결과 검사(assertion) 수행
- 이렇게 함으로써, 코드 수정이 실제로 문제를 해결했는지 트레이스 기반으로 검증 가능

### Chronicle·Boundary로 결정론적 테스트 자동화를 구현하고, 행동적 테스트와의 차이도 설명

- 결정론적(Deterministic) 테스트: 툴 호출, 가드레일 등 결과가 동일한 노드에 대해 트레이스 활용 자동화 가능
- 행동적(Behavioral) 테스트: 전체 에이전트 경로, 생성 응답의 어조·의도 등 추상적·주관적 평가 필요(예: LLM을 심판자로 활용)
- Chronicle은 LLM 노드 출력을 스텁으로 대체하여, 불확실성을 제거하고 테스트 반복 시 비용 감소 효과까지 구현

### 실무에서 얻을 수 있는 5가지 주요 원칙 및 실질적 조언을 정리함

- API를 통해 비트 단위 결정론 추구는 무의미함을 인지
- 세션 가변요소(LLM 버전, 빌드 ID, RAG 청크 등)를 모두 로깅할 것
- 단순 프롬프트가 아니라, 전체 실행 “봉투(Envelope)”를 캡처해야 함
- 리플레이 기능과 트레이스 활용으로 실질적 디버깅·고장 수정·테스트 연계 가능
- 완전한 결정론 대신, 생성적 다양성(agency)을 유지하는 것이 중요함

### Chronicle 코드 및 추가 리소스로 실제 실무 적용을 제안하며 영상 마무리

- 영상 말미에 Chronicle 오픈소스 코드 및 문서 링크를 QR코드로 안내
- 실제 도구 활용을 독려하며, 현재의 트레이스 기록이 미래 온콜(on-call) 사고대응을 개선할 수 있음을 강조
