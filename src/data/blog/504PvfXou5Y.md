## *BDD, ADR, PRD, WTF: 인간과 AI 모두를 위한 의사결정 기록 방법* 핵심 요약

- 이 영상은 Michal Cichra(Safe Intelligence)가 인간과 AI 모두를 위해 의사결정 과정을 효과적으로 기록하고, 일관된 소프트웨어 및 에이전트 개발을 이끌어내는 실제적 방법론(BDD, PRD, ADR 등)을 구체적으로 설명함
- 조직 내 맥락 상실(예: 구성원의 이직, LLM의 제한된 컨텍스트 창 등)에 따른 일관성 문제를 해결하기 위해 ADR(Architecture Decision Record), PRD(Product Requirements Document), BDD(Behavior-Driven Development)의 중요성을 강조
- ADR은 ‘왜’와 ‘어떻게’의 근거를 남겨 아키텍처 결정이 사라지지 않고, 코드, 레퍼런스 문서, 코드 스니펫 등 다양한 방식으로 기록하며, 도구와 린터를 통해 규칙이 실질적으로 강제됨
- PRD는 개별 기능의 목적(‘왜’)과 해결하는 문제(‘무엇’), 유저가 앱을 사용하는 여정(‘어떻게’)을 간결하게 정리하여 개발자와 에이전트 모두가 의도를 이해할 수 있게 함
- BDD는 명세 중심 개발의 한계를 극복하고, 인간 언어로 제품의 행위를 기술·검증하며 PRD 및 핵심 사용자 여정과 직접 연결되어 자동 테스트와 문서화 모두에 유용함 (Cucumber 도구 활용)
- 일관된 UI 생성을 위해 디자인 시스템과 패턴 라이브러리를 명확히 정의하고, 각 UI 컴포넌트의 속성/규칙/상태/프리뷰를 체계적으로 기록 및 강제함
- 실제 규칙 준수 유지를 위해 git hook, CI, linters 등 자동화 도구로 사전·사후 점검·피드백–수정 루프를 구축하여 인간과 에이전트 모두 일관된 실행을 보장
- 코드 구조 분리, DB 접근 차단, N+1 쿼리 방지 등 구체적인 아키텍처 규칙을 정하고 이를 직접적인 코드 및 문서상에서 강제함
- 루프(작업-커밋-피드백-수정 과정)는 제품, UI, 백엔드 등 작업 유형별로 필요한 체크와 우선순위만 달라질 뿐, 동일하게 적용되어 일관성 유지에 기여
- 컨텍스트 부족 문제(특히 AI)는 남은 기록과 반복 조회를 통해 실질적으로 극복 가능하며, 관련 모든 결정을 기록하여 자율적 에이전트 운용의 기반을 마련함

## 세부 요약 - 주제별 정리

### 영상 시작과 인간 및 AI 공통의 맥락 상실 문제 진단

- Michal Cichra는 Safe Intelligence 팀과 신규 에이전트 테스트 제품 ‘Spec 27’을 출시했음을 밝히며, Microsoft, Red Hat 등에서 10년 이상 제품 개발 경험에서 얻은 일관성 문제 및 의사결정 기록의 중요성을 소개
- 인간 역시 ‘왜 이런 기능이 도입되었나?’, ‘해당 코드 구조의 이유는?’ 등 본질적 질문을 자주 잊고, 팀 초기 구성원이 떠나면 맥락 전승이 단절됨
- AI(LLM) 역시 제한된 컨텍스트 창 및 장기 기억 부재로 인해 정보 망실 문제가 인간과 매우 유사하게 발생함
- 유명 ‘원숭이 실험’(바나나와 사다리, 차가운 물)에 비유하여, 맥락을 모르는 채 규칙만 반복하는 조직/AI의 한계를 설명

### ADR(Architecture Decision Record)은 아키텍처 의사결정의 맥락을 정확히 남기고 강제해야만 팀과 에이전트가 일관성을 지킬 수 있음

- ADR은 “왜 이렇게 설계했나”와 “어떻게 강제·실행하는가”를 기록하는 간단 명료한 텍스트 문서임
- 예시: N+1 쿼리 방지를 위해 코드 레이어를 분리, 각 모듈 import를 린팅해 구조 강제, ORM 객체가 아닌 plain 데이터 반환 등 구체적 규칙을 문서화
- 각 ADR은 실제 코드, 폴더, 파일 등 적용 대상과 enforcement(강제 방법)를 명확히 명시함
- “형식은 중요치 않다, 핵심은 의사결정의 맥락과 근거가 반드시 남는 것”이라 강조
- 린터 등 소프트웨어 도구를 통해 실제 개발 및 배포 과정에서 해당 규칙이 자동으로 체크·강제됨
- 에이전트(혹은 개발자)는 각 행위의 이유와 해결책을 문서와 도구를 통해 즉시 확인 및 피드백 가능함

### PRD는 기능 도입의 배경·문제·목표·유저 여정을 알기 쉽게 기록하여 변화에도 목적·맥락을 잃지 않게 한다

- PRD(Product Requirements Document)는 기능별로 “왜 필요한가(목적)–무엇을 해결하는가(문제)–유저는 어떻게 이 기능을 만나는가(여정)”를 간단하게 정리
- 방대하고 상세한 문서일 필요 없이, 이후 6주~6개월 후에도 본질을 되살릴 수 있을 정도의 명확한 서술만 있으면 충분함
- 이 문서는 인간은 물론, 에이전트(LLM)가 이유, 배경, 목표를 빠르게 파악하고 재사용·참조하게 함

### BDD(Behavior-Driven Development)는 명세의 가독성과 자동화 테스트 모두를 해결하는 중간 계층으로 매우 유용하다

- 명세(스펙) 기반 개발의 가장 큰 문제: 실제 코드가 명세대로 정확히 동작하는지 검증·확신이 어렵고, 테스트 코드 역시 이해·리뷰가 난해함
- BDD는 인간 언어로 사용 시나리오와 기능 기대 행동을 적고, 이를 실행가능하게 만든 구조(예: Gherkin 포맷의 Cucumber)를 제공
- BDD는 각 시나리오를 PRD, 주요 유저 여정과 직결시켜 자동화와 문서화를 동시에 달성함
- 이러한 형태는 개발자/에이전트 누구나 읽고 이해할 수 있으며, 테스트 자동화와 리드미(문서) 역할을 병행함
- BDD의 각 스텝(rule, spec)은 자동으로 파싱되어 코드에 반영·검증 가능하고, 인간/AI가 똑같이 참조할 수 있음
- 규격·표현방식은 조직 목적에 따라 자유롭게 설계 가능함

### 디자인 시스템과 패턴 라이브러리 정의는 인간과 에이전트 모두에게 일관된 UI 생성을 보장한다

- UI의 일관성 확보는 AI 시대에도 여전히 디자인 시스템과 패턴 라이브러리가 유일한 답임
- 예시: “프라이머리 버튼은 파란색·특정 모양·하나만 노출 허용” 등 구체적 규칙을 문서화 및 자동으로 강제함
- 각 UI 컴포넌트별 상태 및 변형(여러 버튼 색상, 상태별 미리보기 등)을 명확히 프리뷰 및 스니펫으로 정리
- 문서, 미리보기, 컴포넌트 정의 등을 통해 인간/에이전트가 신규 UI/UX를 만들 때 항상 기준에 부합하도록 유도함
- 컴포넌트는 작은 단위에서 조립→재사용 구조로 관리해야 하고, 그렇지 않으면 코드처럼 UI도 ‘혼돈’에 빠짐

### 루프(작업·검증·피드백) 자동화는 팀/에이전트 모두가 규칙을 실제로 지키게 만드는 핵심 동작 원리이다

- “이 규칙들을 실제로 어떻게 일관성 있게 지킬 수 있을까?”에 답하기 위해, git hook, CI, linter 등 자동화 도구의 조합으로 작업–검증–수정 루프(하네스, reinforcement loop 등)를 구축함
- 에이전트의 목표는 pull request(PR) 제출이고, 그 과정에서 사전(safety gate)·사후 검증이 예외 없이 자동 진행됨
- git hook으로 pre-defined(사전에 정의된) 테스크 실행, CI에서도 동일한 체크 재실행, 패스하지 못하면 작업이 reject(거부)됨
- 강제 항목: 코드 포매팅, 타입 체크, 코드 중복, 아키텍처 준수, 문서 규칙 등 스타일 논쟁 소지가 거의 없이 자동 일괄 처리
- 규칙 위반시 commit 단계에서 원인 문서로 직접 연결되어 빠른 피드백 → 재수정→다시 커밋→반복 루프

### 실질적 코드·아키텍처 규칙 강제를 통해 구조 붕괴 및 버그 근원을 선제적으로 차단한다

- 예시: end-to-end BDD 테스트 스위트는 DB 접근을 엄격히 차단, DB 접근이 필요한 모듈 import를 금지해 실질적 코드 구조 분리 강제
- 렌더링 템플릿에서도 DB 접근 금지, N+1 쿼리를 실질적으로 불가능하게 설계
- 이러한 강제적 구조 분할은 문제 발생 ‘사후 해결’이 아니라 ‘사전 예방’에 초점
- 핵심: 사후에 문제를 계속 발견해 고치려 들지 말고, 애초에 근본적 구조로 확실히 원천 차단해야 함

### 작업 유형별 우선순위 조절로 같은 루프 안에서 최적의 피드백/검증을 달성한다

- 동일 루프 구조(작업, push, 피드백, 반복)는 일관되게 적용되나, 제품, UI, 백엔드, 테스트 등 각 작업 유형별로 우선 체크/생략 항목을 적절하게 조정해 진행
- ADR–에이전트 연계: ADR 문서 키워드 자동 탐색 및 코드를 자동 highlight
- PRD–에이전트 연계: PRD 관련 문서 자동 연결
- UI 작업 시에는 일부 체크 생략, 브라우저에서의 빠른 반복 테스트에 초점
- 테스트 실행시는 커버리지와 파일 변화 기반으로 필요한 최소 테스트만 선택적 실행
- 에이전트의 의사결정 과정도 기록·저장해 사후 검토 가능

### 에이전트와 인간 모두, 컨텍스트 누락의 위협은 반복적인 참조와 기록 누적으로 실질적 영향이 크지 않음을 밝힌다

- “많은 컨텍스트가 빠지지 않을까?”라는 우려에 대해, 최근 6개월 실전 세션 20~50건 사례에서 실제로 중요한 맥락만 반복적으로 살아남았음을 확인
- 에이전트가 중요 정보 및 결정문서를 반복적으로 찾을 수 있게 기록 누적 구조가 갖춰져있으면, context loss(문맥 손실)는 치명적 문제가 아님을 시사
- 장기적으로 중요한 결정, 규칙들은 분실 없이 항상 참조 반영 가능하게 됨

### 결론: 제품과 규칙, 행동, UI, 문서 기록을 통해 인간과 AI 모두가 일관성 있고 품질 높은 개발을 지속할 수 있게 됨

- 각종 결정(ADR), 기능 기획(PRD), 행동 시나리오(BDD), 디자인 시스템 정의가 모두 자동화 루프와 결합될 때 조직·에이전트 모두 일관성 유지 가능
- BDD/Cucumber 활용으로 명세가 읽기 쉽고, 코드/문서/자동테스트로 다시 연결됨
- 디자인 시스템(글로벌 규칙)→컴포넌트→프리뷰→자동 리유즈 구조가 UI 질과 일관성을 보장
- 모든 규칙은 자동화된 루프·린터·git hook·CI 등으로 최대한 규범적, 논쟁 없는 방식으로 enforce 
- “May the spec be with you”라는 인사와 함께 영상을 마무리함
