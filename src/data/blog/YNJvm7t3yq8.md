**채널명:** AI Engineer

## *왜 당신의 AI UX는 제대로 작동하지 않는가 (그리고 그것은 모델의 문제가 아님)* 핵심 요약

- 영상은 대부분의 AI 챗앱이 사용하는 ‘직접 HTTP 스트리밍(SSE, 서버 전송 이벤트)’ 패턴이 AI UX 품질 및 확장성에 구조적 한계를 초래한다고 주장
- 대표적인 한계로 연결 불안정(모바일 데이터/WiFi 전환 등), 다중 기기/탭 사용 시 실시간 세션 연속성 부재, 사용자와 에이전트 간 상호작용 제약(양방향 제어 불가) 등이 제시됨
- Ably의 Mike Christensen은 매월 20억 개 장치, 300억 건 연결, 2조 트랜잭션을 처리하는 대규모 실시간 메시징 경험과 40여 개 회사의 AI 엔지니어 팀과의 협업 경험을 공유함
- 견고한 AI 제품을 만드는 핵심 역량 3가지는 ‘견고한 데이터 스트림(연결 복원력)’, ‘표면간(디바이스/탭) 연속성’, ‘실시간(라이브) 제어(bidirectional control)’임
- 기존 SSE 기반 직접 스트리밍 모델은 모든 데이터 및 컨트롤 흐름이 단일 연결에 묶여 다중 기기나 중단/재개, 실시간 명령(중단, 취소)이 불가능하거나 매우 복잡함
- 이를 극복하기 위해 최근 ‘Durable Session(내구성있는 세션)’ 패턴이 부상하며, 에이전트와 클라이언트를 decouple하여 상태와 이벤트를 ‘세션’에 저장, 누구나 어느 때나 동일 진입점에서 접근
- Durable Session의 backend 구조는 Pub/Sub(발행/구독), 메시지 채널 기반이며, 각 클라이언트와 에이전트가 독립적으로 세션 채널에 접속·구독·이벤트 기록 가능함
- Ably는 이를 지원하는 ‘Ably AI Transport’ SDK를 제공, 자동 복구·동기화·양방향 제어·동시 작업(멀티에이전트) 등 복잡한 내구 세션 로직을 간단히 구현
- 데모 예시에서는 다양한 기기/탭, 동시 다중 작업, 인간 상담사와의 전환 등 복잡한 AI 채팅 시나리오에서도 세션 일관성·연속성·제어성이 자연스럽게 구현됨
- 영상은 최적의 AI UX 구현을 위해 기술적 세션 설계 패러다임의 변화를 강하게 강조함

## 세부 요약 - 주제별 정리

### AI 챗앱의 기본 UX 구조는 단일 연결 모형에 의존하며 본질적 한계가 존재함

- 대다수 AI 챗앱은 유저가 메시지를 보내면, 백엔드 에이전트가 실시간으로 응답을 스트리밍(예: ChatGPT, Claude 등)
- 개발에서 가장 보편적인 방식은 direct HTTP streaming, 대표적으로 SSE(Server-Sent Events)가 기본 옵션임
- 이 방식은 클라이언트가 백엔드 에이전트에 직접, 영구적 포인트 투 포인트(1:1) 연결을 맺고 에이전트로부터 스트림을 받음
- 코드적으로 구현이 쉽고 간단하게 데모가 가능함
- 문제는 이 연결 모델이 "하나의 클라이언트가 하나의 에이전트와 하나의 연결"에 묶여 다중 디바이스, 세션 복원 등 사용자 경험을 확장하기 힘듦

### 대규모 서비스 운영 경험이 보여주는 AI 라이브 UX의 실제 요구와 한계

- Ably는 실시간 메신징 플랫폼으로, 매월 20억 개 장치, 2조 API 오퍼레이션, 300억 연결을 지원
- 최근 1년 간 10개 산업군, 40여 개 기업의 AI 에이전트/어시스턴트 서비스 팀과 협업
- AI기능이 탑재된 상품들은 본질적으로 이용자 경험이 달라지고 새로운 상호작용 모델이 요구됨
- 실제 제작사들은 신뢰성 있는 운영(프로덕션 환경)에서 아래 3가지 역량이 가장 중요하다고 응답

### 신뢰도 높은 AI UX를 위한 3대 핵심 역량이 강조됨

- 1. **Resilient Delivery(견고한 데이터 스트림)**
    - 이동 중 네트워크 전환(예: WiFi→5G, 새로고침 등) 상황에서 연결이 끊기더라도 데이터 스트림이 보존되고, 재접속 시 정확히 이어받기
- 2. **Cross-surface Continuity(표면간 연속성)**
    - 여러 디바이스(폰, 태블릿, 브라우저 탭) 간에 동일한 대화 세션·라이브 작업 경험이 동기화되어야 함
- 3. **Live Control(실시간 제어)**
    - 단순 요청-응답을 넘어서, 작업이 진행 중일 때 추가 요청, 명령(중단, 수정 등), 진행상황 확인 등 양방향 상호작용이 가능해야 함

### 직접 HTTP 스트리밍 아키텍처는 이 세 가지를 구현하는 데 근본적 장애요인이 됨

- 연결 및 스트림의 생존 여부가 단일 클라이언트와 에이전트의 연결 상태에 1:1로 묶임
- 클라이언트 연결이 끊기면 스트림이 즉시 사라짐(모바일, 새로고침 등)
- 다른 디바이스/탭에서는 진행 중이던 대화 상태, 응답, 활동을 볼 수 없고 이어받을 수도 없음
- 스트림이 사적인 채널이기에 다중 클라이언트가 에이전트와 동시 상호작용하는 것이 불가능함

### Resumable Stream(데이터 스트림 복원) 요구사항이 별도 복잡 로직을 요구함

- 예시: 스트리밍 중 클라이언트 연결이 끊겼을 때, LLM은 계속 데이터를 만들어내지만 이를 받아줄 클라이언트가 없음
- 복원(Resume) 기능을 제공하려면
    - 1. 각 이벤트를 순서대로 임시저장(Redis 등 메모리 스토리지)
    - 2. 시퀀스 넘버 관리 필요
    - 3. 재접속 시 missed event만 추출·재전송하는 Resume Handler(핸들러) 구현 필요
- 결과적으로 각 연결 별 맞춤 로직, 복잡한 구현이 필수 → 에이전트/AI 모델 개발에 집중 불가

### SSE 기반 스트리밍에서는 Resume와 Cancel(작업 중단)이 근본적으로 병행 불가함

- SSE(Server-Sent Event)는 Server→Client 일방향 채널
- 예: 스트리밍 중 “중단(Stop)” 버튼을 눌러도, 클라이언트는 연결을 끊는 것 외에 구현적으로 취소 의사를 에이전트에 전달할 방법이 없음
- 이로 인해 에이전트 입장에서는 연결 종료를 Resume(클라이언트가 곧 재접속 예정)로 볼지, Cancel(진짜 취소)로 볼지 모호함
- Vercel AI SDK 등도 SSE에서 Resume와 Cancel은 동시에 지원 불가(문서에 명시)
- 즉, 진정한 양방향 제어(bidirectional control)는 지원 불가

### WebSocket으로 교체해도 다중 기기/탭에서는 여전히 한계가 남아 있음

- WebSocket은 양방향 통신이지만, 요청을 보낸(연결을 맺은) 탭만 응답/스트림을 볼 수 있음
- 동일 세션을 다른 탭/디바이스에서 열면, 진행중인 스트림·상태를 볼 수 없음 (“Second tab doesn't see anything”)
- 다수의 기기/탭에서 에이전트에 제어·질문하려면 여전히 별도 공유 채널/세션 관리가 필요

### Durable Session은 클라이언트와 에이전트 계층을 분리해 위의 문제를 해결함

- 에이전트와 클라이언트 사이에 ‘공유 자원’으로서의 Durable Session 계층을 둠
- 이벤트를 직접 세션에 기록, 클라이언트는 언제든 세션에 접속해 이어받기/제어/상호작용 가능
- 연결 유실·다중 기기/탭·Live Control을 별도 복잡도 없이 모두 지원
- 모든 참가자가 항상 세션에 동기화된 상태로 접근 가능

### 여러 참가자, 에이전트가 협업하는 시나리오도 구조적으로 간결하게 구현 가능함

- Orchestrator(오케스트레이터)와 Sub-agent(서브에이전트) 등 다양한 AI 또는 사람 참가자들의 세부 작업 경과를 하나의 세션 채널에 직접 기록
- 기존 방식은 중앙 오케스트레이터가 모든 granular update를 모아 중계하나, durable session에서는 각 에이전트가 독립적으로 작업 이벤트를 세션에 기록
- 클라이언트는 세션만 구독하면 전체 진행상황, 각 에이전트의 실시간 작업 경과, 다른 참가자 활동까지 한 번에 볼 수 있음
- 아키텍처가 단순해지고 동시 다중 작업(멀티태스킹)도 손쉽게 구현

### Pub/Sub(발행/구독), Channel(채널) 기반 구조가 Durable Session 구현의 핵심 엔진임

- Ably의 Channel은 발행자(Producer: Agent, Client)와 구독자(Subscriber: Client, Agent) 간의 직접적 연결 없는 공유 자원
- 개별적으로 주소지정 가능(채널 네임으로 자유롭게 접속)
- 메시지·이벤트는 단일 연결/디바이스·참가자 생애주기를 초월해 세션에서 영구 저장
- 클라이언트 연결이 끊겨도 상태를 잃지 않고, 복원(Resume) 시 정확히 이어받기 가능
- 다중 클라이언트·멀티디바이스·멀티에이전트의 요소가 섞여도 데이터, 명령, 상태의 동기화가 자연스럽게 됨

### Ably AI Transport SDK는 모든 복잡한 Durable Session 로직을 일괄적으로 제공함

- Ably AI Transport는 Durable Session 패턴 구현을 지원하는 “플러그&플레이” SDK
- AI 모델, 이벤트 포맷, 프레임워크에 독립적으로 적용
- 이벤트(예. LLM 생성 텍스트 청크) 실시간 스트림·결과 자동화, 자동 복구(resume), 멀티플렉싱(동시 다중 활동), 양방향 제어, 멀티 클라이언트 팬아웃 지원
- 부가로, 푸시 알림(비동기 작업 완료시), 실시간 협업 데이터 오브젝트 공유/구독 등 추가 API 제공

### 실제 데모에서 보는 내구성 세션 모델 적용 효과

- 전자상거래 AI 지원 챗봇 예제에서, 하나의 세션에서 여러 탭/기기·에이전트·사용자가 동시·비동기 작업 가능
- 다양한 기기/탭에서 세션 행동이 즉시 동기화(예: 한 탭에서 상품 주문, 다른 탭에서 취소), 새로고침·네트워크 종료 후에도 상태 보존·복구
- 한 탭에서 “전문 에이전트” 태스크 시작→다른 탭에서 취소, 실시간 진행상황 동기화
- 심지어 대화 중 인간 상담사로 참여 전환해, 이전 AI-클라이언트 모든 이력 및 상태가 지원자에게 자연스럽게 전파
- 세션 기반 접근을 통해 복잡한 UX 동기화, 라이브 제어, 멀티 참가자 전환을 추가 로직 없이 쉽게 구현

### 결론: AI UX 품질의 핵심은 모델이 아니라 세션 상태 관리 아키텍처의 혁신에 달려 있음

- AI 챗 경험의 본질적 한계를 해결하려면, SSE 등 직접 연결 기반 방식을 넘어 "Durable Session–Pub/Sub–Channel" 패턴으로 구조 패러다임을 전환해야 함
- 세션 관리가 견고하고 유연해야 멀티 플랫폼, 네트워크 불안정, 실시간 상호작용 등 복합적인 현대적 AI 사용자 경험 요구에 부응 가능
