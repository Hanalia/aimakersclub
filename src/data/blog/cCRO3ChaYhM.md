**채널명:** AI Engineer

## *인간과 머신을 위한 플랫폼: 에이전트 시대를 위한 엔지니어링* 핵심 요약

- 본 강연자는 글로벌 핀테크 기업 Banking Circle의 클라우드 네이티브 기술팀 리더 후안 에로사( Juan Herreros Elorza)이며, 연 매출 1조 유로와 700여 금융기관 대상 서비스 제공 실적을 바탕으로 플랫폼 엔지니어링 경험을 공유함
- 사내 엔지니어 250여 명이 다양한 시스템(코어 뱅킹, API, 데이터, 인프라, 결제 연동 등)을 개발하며, 이를 공동으로 활용할 수 있도록 최신 플랫폼 ‘아틀라스(Atlas)’를 구축
- 아틀라스는 쿠버네티스 기반 컴퓨팅, 스토리지/DB/비밀 관리 등 인프라 자원, 메시징, 관측성(Observability) 등을 서비스형 플랫폼으로 제공
- 신규 개발자의 온보딩 및 배포 과정에서 기존 수작업, 팀 간 의존, 문서의 불충분함 등 현실적 문제와, 이 상황이 LLM·코딩 에이전트 도입 시 더욱 심각하게 드러남을 사례로 설명
- 개발자가 아닌 AI 에이전트의 경우, 기존 '사람 간 소통 중심' 프로세스는 거의 불가능하므로, 완전 자동/셀프서비스 플랫폼의 필요성 대두
- 플랫폼은 셀프서비스, API 기반, 로컬 우선(Local First) 원칙을 따라야 하며, 프로세스 자동화·API 제공·인증/권한 관리·에러 신속 검증 등이 필수적임
- 문서 제공 방식을 인간뿐 아니라 AI 에이전트도 소비할 수 있도록 강화해야 하며, 코드 옆 또는 중앙 집중형으로 관리하고, API를 통한 문서 제공·에이전트 전용 마크다운 등을 적용
- 내부 플랫폼에는 개발자 및 에이전트의 적극적 기여(Contributions)를 장려하되, 보안·컴플라이언스·정책적 가드레일 구축 필요
- 도입 성과를 측정하기 위해 Dora 지표(배포 빈도, 변경 주기, MTTR 등), 안정성, 지원 요청량, 개발자 만족도, 플랫폼 전용 지표 등을 활용
- AI 바람을 계기로 API 우선, 셀프서비스, 문서, 관측성 등의 ‘오래된 가장 좋은 실천 방법’을 정당화·추진할 수 있는 기회를 제안

## 세부 요약 - 주제별 정리

### Banking Circle은 글로벌 지급결제 및 뱅킹 서비스로 대규모 인프라와 엔지니어링 플랫폼이 필수적임

- 주최자는 Banking Circle의 클라우드네이티브 기술팀 리더로, 회사는 글로벌 크로스보더 결제(연 1조 유로)와 계좌·유동성 운영 등 금융업무를 제공
- 700개 이상의 규제된 금융기관을 대상으로 서비스 제공
- 사내 250명 이상이 API, 핵심 뱅킹 시스템, 데이터 과학 및 엔지니어링, 결제 정산 등 여러 기술 영역에 걸쳐 개발 중
- 이런 다수 엔지니어와 시스템 통합을 위해 효율적 플랫폼이 필요함을 인식

### ‘아틀라스(Atlas)’ 플랫폼은 컴퓨팅·인프라·메시지·관측성 등 다양한 서비스를 추상화하여 제공함

- 아틀라스 플랫폼은 4대 축으로 구성
    - 쿠버네티스 기반 컴퓨팅(애플리케이션 배포/운영)
    - 인프라 자원(블롭스토리지, 데이터베이스, 비밀 관리 등) 프로비저닝
    - 각종 시스템간 메시지(메시징 플랫폼) 연결
    - 관측성(로그, 메트릭, 트레이스 등)으로 전체 상황 모니터링
- 여러 팀이 중복, 복잡성을 피하고 본연의 시스템 개발에 집중하도록 지원

### 기존 개발 프로세스는 팀 간 의존, 경험 공유, 매뉴얼 문제로 온전한 자동화를 방해함

- 신규 입사 개발자가 첫 애플리케이션을 작성할 때 배포 방식을 동료에게 문의
- 기존 파이프라인을 복사하지만 애플리케이션 특이 요건, 에러 발생, 원인 미파악 등으로 좌절 경험
- 인프라팀의 지원이 필요하나 우선 순위나 인력 부족으로 지연
- 문서 부족, 팀 내 경험 공유, 의사소통 경로 등으로 일관성·자동화 결여

### AI 에이전트가 등장하면서 기존 ‘사람 의존’ 프로세스의 한계가 뚜렷하게 드러남

- 최근 LLM, 코딩 에이전트 등 AI 기반 업무 도입이 활발
- 인간은 직접 상담, 문의, 문서 확인 등 비정형 과정을 거칠 수 있으나, 에이전트는 이를 대체하기 어렵거나 불가능
- 예: 에이전트가 슬랙, 팀즈, 음성봇을 사용할 수도 있지만 근본적으로 '명확히 정의된 인터페이스'가 필요
- 이로 인해 기존 업무상의 "편법"이나 "비공식 절차"가 더는 적용불가함

### 셀프서비스와 자동화, API 기반, 쉬운 사용성 확보가 필수적임

- 리소스 요청, 배포 등 모든 과정은 사람 개입 없이 개발자(혹은 에이전트) 스스로 수행 가능해야 한다고 강조
- 셀프서비스는 사용성이 직관적이고 추상적인 단계를 최소화해야 진정한 셀프서비스가 됨(자동화·사람제외·단순화)
- API 기반 플랫폼으로 설계하면 에이전트가 기능(프로비저닝, 배포, 검증 등)을 직관적이고 반복적으로 활용 가능
- 인증 및 권한 관리, 스키마 검증, 탐색성 등도 API 기반의 장점으로 언급

### 로컬 환경 중심(Local-first)·프론트로 실패(Shift-left) 설계가 중요함

- 에이전트가 로컬에서 동작하는 특성(예: 개발자 PC, 서버)에 맞추어, ‘최대한 빨리 실패(fail fast)’할 수 있어야 효율적임
- 미리 API 호출로 검증, 파이프라인 진입 전 에러 감지 등 shift-left 전략을 권장
- 명확한 작업 정의(Task description)와 '성공의 기준'을 에이전트에 설명해야 자동화 루프 완성 가능

### 에이전트가 사용 가능한 문서 구조와 API 접근성은 플랫폼 생산성에 직결됨

- 기존 인적 문서는 AI 에이전트가 읽기 어렵고, 코드에서 떨어진 위치에 있으면 맥락 파악이 불가
- 코드 근처(소규모 저장소) 혹은 중앙 집중(API 노출 문서)에서 문서화 필요
- API 혹은 맞춤형 엔드포인트로 문서의 '해당 부분'만 효과적으로 제공해야 함
- 에이전트 전용 마크다운(agents.md, instructions.mmd 등) 활용법도 소개 - 예: 빌드, 테스트, 배포, 검증 방법을 상세히 기술

### 조직 단위의 표준 프로세스·스킬 정의는 에이전트 활용성을 높임

- 반복적 작업, 관습(convention), 표준적인 상호작용 방법은 ‘스킬(skill)’이라는 마크다운 문서로 정리 가능
- 구체적 프로젝트용 agents.md와 범용 스킬용 마크다운 구분하여 활용 가능
- 에이전트가 스킬을 참조하여 올바른 방법으로 작업 수행하게 지원

### 플랫폼 기여(Contributions)는 에이전트 도입으로 문턱이 낮아지나, 철저한 가드레일 구축이 필요함

- AI 에이전트 도입 시, 내부 개발플랫폼에 다양한 기여자가 등장(사용성 증가, 변화 활성화)
- 기여는 조직 전체 효율성 증가에 이롭지만, 보안·컴플라이언스·유지보수 이슈 등 고려해야 함
- 신뢰성 확보를 위한 정책(Policies), 문서 기반 규칙(마크다운 가이드라인) 등 다층적 가드레일 필수

### 성공적 AI 플랫폼 구축 여부는 명확한 측정 지표로 검증해야 함

- 변경 도입 전후 성과를 수치로 측정 권장(플랫폼 개선 효과 파악)
- 주목할 지표로 Dora Metrics(변경 배포 빈도, 평균 복구 시간, 리드타임 등), 안정성(에러율, 트래픽), 지원요청 건수, 개발자/에이전트 만족도(Developer Experience Survey) 등 제시
- Self-service 기능이 실제로 지원요청 감소, 배포 효율 증가 등으로 이어지는지 확인 필요

### AI 대중화 흐름은 ‘베스트 프랙티스’ 도입의 변명과 기회가 됨

- 조직 내 API 우선, 셀프서비스, 문서화 등 기존 베스트 프랙티스 도입 시 저항이 있었더라도, AI 도입 열풍을 계기로 추진 논리를 확보 가능
- 모든 구성원이 AI에 주목하는 현시점에서, 실질적 업무 환경 개선을 위한 혁신적 변화 촉진 가능
- 발표자는 "지금이 바로 예전부터 권장됐던 베스트 프랙티스 실천의 적기"임을 재차 강조
