## *자율 경제를 위한 안전한 결제 인프라 구축* 핵심 요약

- **이 영상은 Stripe의 수석 소프트웨어 엔지니어인 Steve Kaliski가 자율 경제(autonomous economy)에서 로봇이 안전하게 돈을 쓰고, 사업자가 로봇으로부터 돈을 받을 수 있도록 하는 안전한 결제 인프라 구축 방안을 설명함**
- **Stripe는 인간뿐 아니라 로봇(에이전트)도 신용카드를 사용할 수 있는 발급 시스템(issuing system)을 개발하고, 최근 2년 동안 로봇의 결제 참여를 위해 다양한 기능을 연구 및 구축함**
- **'발견'과 '탐색'은 비결정적(non-deterministic)이어야 하나, '자격증명', '결제', '체크아웃' 과정은 반드시 결정적(deterministic)이어야 한다는 점이 핵심 주장임**
- **결제 인프라에서 발생할 수 있는 주요 문제(잘못된 주소, 잘못된 상품·금액, 잘못된 자격증명 사용 등)를 소개하며, Stripe 및 파트너들이 이를 해결하는 구조적인 접근 방식을 제시**
- **Stripe의 기술적 솔루션으로 '공유 결제 토큰(Shared Payment Tokens)', '머신 결제 프로토콜(Machine Payments Protocol)', '에이전트-커머스 프로토콜(Agent-Commerce Protocol)' 등 세 가지를 중점적으로 시연 및 설명**
- **공유 결제 토큰을 통해 각 결제단위별 제한(가맹점·한도·기간 등)을 엄격히 설정하고, 결제 대행사는 거래 내역을 감사하고 실제 결제 정보(브랜드, 카드번호 뒷자리 등)도 계속 파악할 수 있음**
- **머신 결제 프로토콜은 API 기반 상호작용에서 402코드를 사용하여 결제 필요를 선언하고, 토큰을 통해 자격증명 및 데이터 연동을 실시간으로 처리함**
- **에이전트-커머스 프로토콜(ACP)은 OpenAI와 협업하여, 제품 카탈로그·체크아웃·개별 결제 절차를 표준화된 API로 구조화함으로써 로봇이 인간처럼 웹사이트를 탐색하지 않아도 안전하게 구매가 이뤄지도록 함**
- **이런 시스템들은 결제 및 상거래 프로세스를 안전하게 자동화하고 비즈니스와 에이전트 모두 결정론적인(예측 가능한) 트랜잭션을 보장**
- **결론적으로, 탐색 단계는 비결정적이어도 되지만 실제 거래 및 인증, 결제, 체크아웃 과정은 항상 결정적으로 설계해야 함을 강조함**

## 세부 요약 - 주제별 정리

### Stripe는 인간뿐 아니라 로봇의 경제 행위도 동일하게 지원하려고 함

- Stripe는 기존에 인간 사용자를 대상으로 한 신용카드(실물/가상) 발급 기능(issuing)을 제공해 왔음
- 최근 2년간은 로봇(에이전트)이 독립적인 경제 주체로 결제에 참여할 수 있도록 인프라를 연구 및 개발
- 새로운 결제 인프라는 인간 중심에서 로봇·에이전트 중심으로 확장되는 시장 상황에 대응함
- 로봇도 클라우드 코드/코덱스 등 다양한 툴에서 사실상 ‘돈’을 지출함(토큰·구독 등 화폐/재화로 환전)
- Stripe의 핵심 관심은 “어떻게 하면 더 다양한 결제수단과 패턴, 비즈니스 인터랙션을 지원할 수 있을지”에 있음

### 비결정성과 결정성의 경계 설정이 핵심이며, 결제 과정에선 반드시 결정성이 필요함

- LLM, 에이전트 등은 비결정적인(예측불가) 방식으로 데이터를 탐색함
- 하지만 결제, 자격 인증, 체크아웃 등의 근본 인프라 영역은 반드시 결정성(Determinism)이 필요
- 발견/탐색(예: 어떤 상품을 살지 골라주는 LLM, 추천)은 비결정성의 이점을 갖지만,
- 자격증명 입력, 실제 결제(과금)에서는 결과가 반드시 예측 가능해야 함(안전 및 책임 확보)
- Stripe의 혁신은 “탐색과 트랜잭션의 경계를 명확히 분리”하여 안전한 인프라를 만드는 데 있음

### 로봇이 결제 과정에서 겪을 수 있는 4가지 주요 오류 상황을 체계적으로 분류함

- 잘못된 가맹점(도메인)에서 결제: 예를 들어 “ama.zon.com” 등 피싱/가짜 사이트 문제
- 잘못된 상품 선택: 예시로 ‘보라색 티셔츠’를 주문하려다 잘못하면 ‘오렌지색 티셔츠’ 또는 지나치게 비싼 물품 구입 가능
- 잘못된 금액 결제: 통화 변환, 세금, 단가 오독 등 금액 추출 오류로 잘못된 대금 결제
- 잘못된 자격증명(카드 등) 사용: 신용카드 정보가 유출되어 엉뚱한 곳에 전달될 수 있음
- 기존 방식(로봇이 브라우저로 인간처럼 클릭해 결제)은 느리고, 오류가 많으며, 결과 추적이 어려움

### Stripe는 카드번호 직접 노출 방식 대신 API 기반 결제 프레임워크로 안전성을 높임

- Stripe는 인간 사용자용 UI(좌측 대시보드)와 달리 로봇·에이전트는 코드를 통한 API 대화 방식을 선호함
- 바람직한 결제 플로우는 다음과 같은 조건을 충족해야 함:
    - 가맹점 바인딩(merchant binding)
    - 결제 한도 및 정책 강제(Spend policy enforcement)
    - API 기반, 코드·프로그램화 가능
    - 검증 가능한 신원(Verifiable identity) 제공
- Stripe의 목표는 브라우저 자동화 대신 API·토큰 기반 검증 및 결제 구조를 구축하는 것임

### Stripe의 첫 번째 혁신: ‘공유 결제 토큰’으로 안전하고 유연한 결제 권한을 관리함

- 신용카드만이 결제수단이 아니며, 다양한 형태(은행, 간편결제 등)의 결제를 지원해야 함
- 단순히 번호를 넘기는 방식은 결제 한도·기간·대상 가맹점 등 제한을 적용할 수 없음
- ‘공유 결제 토큰’(Shared Payment Token) 도입을 통해
    - 특정 카드(예: Visa)를 특정 금액($25), 특정 기간(예: 30일), 특정 가맹점 등으로 목적·범위 제한 가능
    - 토큰은 Stripe가 강제로 제한 및 관리 (예: $25 한도 설정시 이를 초과할 경우 결제 차단)
    - 토큰 사용 시 카드 이미지(브랜드, 뒷자리 번호 등) 등 기존 리스크 판단 정보도 가맹점에 전달
- 실제 데모에선 구매 요청 금액($50)이 토큰 제한($25)을 초과하면 결제 거절 됨
- 금액 조정($25 이하로) 후에는 결제 승인, 전 과정이 Stripe 시스템에서 실시간으로 검증됨
- 이 방식은 수백 개의 결제수단에도 적용 가능하며, 사업자-고객 간 위험범위를 최소화함

### 머신 결제 프로토콜(Machine Payments Protocol)로 API 기반 자동 결제 및 인증 체계를 구축함

- 로봇은 웹 API 엔드포인트를 활용해 타 시스템(툴)에 접근(실행)함
- 일부 엔드포인트는 접근 시 결제가 필요하며, HTTP 402 코드로 결제 필요 신호를 반환함
- Shared payment token을 첨부해 해당 결제 요청을 즉시 처리할 수 있음
- 데모에서 로봇(서버)이 API 호출 시 결제가 필요함을 안내받고, 해당 토큰을 사용하여 1센트(USD) 결제를 실행
- 결제는 Tempo 블록체인 등 외부 네트워크를 활용하며, 거래 내역은 체인에 기록되고 Stripe가 거래 뷰를 동기화함
- 이 과정을 통해 API 도구 이용, 권한 부여, 결제 등 모든 흐름이 실시간·자동화됨

### 에이전트-커머스 프로토콜(ACP)로 로봇 친화적 체크아웃/구매 시스템을 표준화함

- OpenAI와 공동 개발한 ACP(Agent-Commerce Protocol)는 자동화된 로봇 상거래를 위한 API/오브젝트 표준임
- 제품 카탈로그(상품 이미지, 설명, 가격 등)를 JSON 구조로 노출 → 로봇이 직접 UI를 해석하지 않고 선택 가능
- 장바구니 생성, 수량/결제/배송옵션 변경 등 모든 절차를 API 기반으로 양방향 상태동기화로 처리
- 체크아웃 시 세금, 배송비 등 각종 디테일을 구조적으로 전달 → 구매 품목/혜택/금액 불확실성 최소화
- 실제 Stripe Press(도서 쇼핑몰)에 데모 적용
    - 오른쪽 API 로그로 실시간 구매 단계, 선택 상품, 장바구니 상태, 배송 옵션 등을 주고받음
- 궁극적으로 결제 수단(신용카드, 암호화폐 등) 무관하게 API로 통합 처리, 사업자가 거래 원칙 및 위험관리를 계속 담당 가능

### 탐색(추천·발견)과 결제(인증·체크아웃)는 명확히 분리해 리스크를 최소화함

- 상품 발견, 추천, 선택은 비결정적(non-deterministic)이어도 무방함(LLM, 웹 크롤링 등 활용)
- 결제, 결제 인증, 체크아웃 단계에서는 반드시 결정적(deterministic)이어야 하며
    - 인증된 이해관계자(에이전트-비즈니스 간)
    - 협의된 조건(한도, 기간 등)
    - 실시간 구조적 검증
    - 소규모(제한된) 위험관리를 달성해야 함
- ‘비결정적 플래너+결정적 제약+검증 가능한 당사자’ 조합이 안전한 자동 결제 실현의 핵심

### Stripe의 머신 결제 시스템은 반복 결제(구독)와 고정 예산 관리도 유연하게 지원함

- 월 $25 한도를 예로, 특정 모델/서비스에 반복적으로 자동 결제 허용도 설정 가능
- OAuth의 access/refresh 플로우처럼, 반복 사용에 맞춰 추가사용 요청 및 승인을 API로 처리
- 무제한(고한도) 예산도 동일 메커니즘으로 관리 가능, 여전히 가맹점 단위로 범위 제한 및 통합 관리 가능

### Stripe Projects 등 Stripe의 B2B 솔루션은 이러한 결제 기반 위에서 동작함

- Stripe Projects는 세션에서 소개한 Shared Payment Token, API 등 결제 프리미티브의 래퍼(wrapper) 역할
- SaaS 사업자, B2B 플랫폼이 손쉽게 로봇 친화적인(Agent-friendly) 결제·상품관리 시스템 구축 가능

### 결제·거래 데이터의 블록체인 기반 처리와 Stripe의 데이터 통합 구조를 병행 운영함

- Tempo, Base 등 다양한 블록체인 네트워크 프로토콜을 Stripe가 지원함
- 트랜잭션 데이터는 블록체인에 저장되며, Stripe의 관리 시스템에서도 실시간 ‘제품 뷰’로 동기화
- 데이터 소유권, 기록 방식 등이 네트워크마다 상이할 수 있음

### Stripe의 로봇 결제 인프라는 실사용 데이터와 볼륨을 점차 늘리고 있음

- 도입 2년차, 정확한 결제 건수/볼륨 등은 공개하지 않지만 Stripe 내에서도 확대에 자신감을 가짐
- 다양한 비즈니스가 로봇 또는 에이전트 기반 결제 지원 확대에 관심이 높아지고 있음
