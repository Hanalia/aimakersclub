## *컨텍스트 엔지니어링을 위한 에이전틱 서치* 핵심 요약

- **영상 제목:** 컨텍스트 엔지니어링을 위한 에이전틱(Agentic) 서치
- 발표자는 Elastic(Elastic Search의 개발사) 소속의 Leonie Monigatti로, 검색과 검색 데이터 활용 경험 중심으로 주제 소개
- 컨텍스트 엔지니어링의 핵심은 "수많은 컨텍스트 소스 중 LLM에 입력될 정보(컨텍스트 윈도우)를 어떻게 선정할 것인가"에 있음
- 기존 RAG(Retrieval-Augmented Generation)는 고정된 검색 파이프라인으로, 모든 쿼리에 동일한 검색 방식 적용 → 유연성 부족 및 엉뚱한 정보 검색 문제
- 에이전트 기반 RAG(Agentic RAG)에서는 검색 도구 호출 여부, 횟수, 검색 쿼리 재작성 등 LLM이 능동적으로 판단 가능해짐
- 실제 컨텍스트 소스(로컬 파일, Skill, 데이터베이스, 웹, 메모리 등) 종류와 각 소스에 특화된 검색 도구(예: 파일 검색, Skill 로딩, SQL, 웹검색, 메모리 도구 등)를 상세히 나열
- 강력한 범용 도구(예: Shell tool, Bash tool, Exec tool)는 CLI를 통한 파일 시스템/DB/웹 서치, 스크립트 실행 등 다방면 활용 가능하나, 오히려 복잡도와 위험도 동반
- 실습에서는 Elastic Search 기반의 시맨틱 검색, ESQL 쿼리 실행 도구, Bash/Shell을 활용한 로컬 파일 시스템 검색, 그리고 외부 CLI 연동 예시(예: GINA Grap 등)까지 단계별로 보여줌
- 각 검색 도구의 설계, 파라미터 복잡성, Tool Description의 중요성, Context Management, Agent Skill 활용법, 하이브리드/전문화 도구의 실제 적용 및 장단점 분석
- 도구 선택/구성 시 "전문화 도구(낮은 진입장벽, 쉬운 사용)"와 "범용 도구(높은 확장력, 복잡 쿼리 대응)"의 균형 필요함을 강조하며 실제 서비스 설계 가이드 제시
- 발표 마지막 Q&A에서 다양한 실제 활용 노하우(모델 성능, 에러율, 도구 조합, 컨텍스트 관리, 서브에이전트, 롱텀 세션 등) 공유

## 세부 요약 - 주제별 정리

### 발표자는 컨텍스트 엔지니어링의 핵심이 실제로는 80%가 ‘에이전틱 서치’임을 주장함

- Leonie Monigatti(Elastic) 발표로, 트위터에서는 검색(Retrieval) 주제에 대해 활발히 발언
- 컨텍스트 엔지니어링은 "다양한 소스 중 LLM이 활용할 컨텍스트를 효과적으로 추출하여 입력하는 기술"로 정의
- 기존 논의에서 컨텍스트 선택(화살표)의 역할이 과소평가되고 있으며, 실제로 이를 가능하게 하는 것은 '검색 도구'임
- 발표자는 "컨텍스트 엔지니어링의 80%는 좋은 검색(Agentic Search)이 좌우한다"고 주장하며, 검색 도구의 설계와 운영이 핵심임을 강조

### 기존 RAG(고정형 리트리벌 파이프라인)는 구조적 한계를 지님

- 초창기 RAG는 사용자의 메시지를 거의 그대로 벡터 검색 쿼리로 사용해 DB에서 컨텍스트를 가져옴
- 필요 여부와 관계없이 항상 검색을 수행, 불필요하거나 혼란을 야기하는 정보까지 집어넣는 문제가 발생
- 멀티홉 검색이 필요한 경우(복잡 쿼리, 추가 검색 필요성 노출)에도 적응이 어렵고, 단 한번만 검색하는 구조의 한계 노출
- 결과적으로 고정 파이프라인은 유연성이 부족하고, 실제 쿼리/상황에 적합한 동적 대응이 어렵다는 문제 설명

### 에이전틱 RAG에서는 LLM이 능동적으로 검색 도구 호출/활용 여부를 판단할 수 있음

- 에이전틱 RAG는 에이전트가 직접 "검색 도구"를 호출할지 여부, 몇 번 검색 실행, 쿼리 수정 등이 가능함
- 따라서 불필요한 검색 수행, 적절하지 않은 컨텍스트 선택, 추가 검색 필요성 등 기존 RAG의 한계 극복
- 단, 이 구조에서도 하나의 컨텍스트 소스(예: 하나의 DB)에만 의존하는 문제는 여전함

### 실제 컨텍스트 소스는 매우 다양하며, 각각에 특화된 검색 도구가 필요함

- 로컬 파일 시스템(코딩 프로젝트, 파일 탐색), 작업용 임시 메모리(Scratch pad, Plan.md), Skill(함수 등), DB, 웹, 롱텀 메모리 등 다양한 소스를 예시로 제시
- 각 소스에 맞는 특화 검색 도구가 존재: 파일 탐색/search tool, Skill 로딩, SQL/Custom 쿼리 도구, 웹 검색 도구, 전용 메모리 도구 등
- 최근에는 Shell Tool(Langchain, Anthropic 등에서 이름 다름, OpenClaw에선 Exec tool)을 통해 터미널 명령어 실행 기능까지 확장
- Shell tool 활용시 CLI를 통한 파일 시스템/DB조작/웹검색, 스크립트 자율 구현 등 가능
- 각 소스와 도구의 연계, CLI 기반 실행 등 실제 활용 구조도 함께 설명

### 범용 Shell/Bash/Exec Tool의 등장은 검색 유연성을 극대화하지만, 위험성과 복잡성도 동반함

- Shell tool로 LLM에 터미널 접근 권한을 제공시 ls, grep 등으로 파일 실시간 탐색/검색, 데이터베이스 명령, 외부 API(curl 등)와 인터랙션 가능
- 다양한 커스텀 CLI 연동(예: GINA Grap 등)까지 활용 가능
- 단, "좋은 검색" 그 자체가 굉장히 어렵기에 다양한 검색 방식/도구(벡터 검색, 키워드 검색, 인덱싱, 임베딩 유형 등) 혼용 필요
- Shell tool 등 범용 도구 남용시 에이전트가 정확한 쿼리 파라미터/명령을 생성하지 못하는 실패 케이스가 많아짐
- 복잡 쿼리, 시행착오 반복, 보안 위험(파일 삭제 등) 우려까지 존재하며, 실서비스에는 Sandbox 환경 특징 및 제약 필요

### 도구 설계시 Tool Description(설명) 및 파라미터 설계가 성공적 검색을 좌우함

- Tool Description(도구 설명)은 에이전트가 얼마나 정확히 도구를 사용하느냐의 핵심 요소임에도 불구하고 현실에서는 매우 소홀하게 다뤄짐
- 단순 1~2문장 설명이 아닌, 사용 트리거, 비사용 조건, 도구 간 관계(예: 먼저 Skill 호출 필요) 등 충분한 맥락 제공이 중요
- 파라미터가 단순(예: ID로 조회)할수록 LLM이 실수 없이 도구 활용 가능성이 높음
- 반면 여러 필터 또는 다양한 옵션(Top-K, 복수 조건 등)이 추가될수록 LLM이 적합한 파라미터 조합을 생성하는 실패가 많아짐
- 특히 ESQL 등 복잡 쿼리 전체를 작성해야 하는 범용 도구의 경우, 에러 핸들링, 자기 교정 구조, 상세한 설명/Skill 연계 등이 필수임

### Elastic Search 데이터 예제로 시맨틱 검색 도구 구현 및 한계 시연

- 실습 데이터: Elastic Search DB에 저장된 컨퍼런스 세션 정보(제목, 설명, 메타데이터)
- LangChain 활용: GPT 5.4 Nano 모델 기반 에이전트, 심플 시스템 프롬프트, 시맨틱 검색용 벡터 임베딩(GENOME Embeddings v5) 설정
- Python 함수(검색 도구) → LangChain의 @tool 데코레이터로 Agent Tool로 변환(함수명=도구명, Docstring=도구 설명)
- "regulatory constraints" 등 쿼리 수행 결과, 적절한 토크(강연) 검색에 성공
- 하지만 같은 방식으로 "GPA" 등 쿼리 수행시, 관련 없는 세션이나 오답(토크나이징 오류 등) 다수 발생
- 키워드 기반 쿼리, 개별 필터 적용 등이 힘든 한계 확인

### 범용 쿼리 실행 도구(ESQL 쿼리 입력) 및 에이전트 Skill 활용으로 복잡 검색 과제 극복

- Elastic Search의 쿼리 언어 ESQL 전체를 입력받아 임의의 쿼리 실행을 허용하는 범용 도구로 교체
- 도구 설계시 오류 발생시 Try-Except 통해 LLM에 에러 응답 전달, 자기 교정(search query 재입력) 로직 구현
- SQL/ESQL 문법 차이(와일드카드 %, *) 같은 문제 혼잣말로 설명, 필요한 경우 Skill(에이전트 용 문법/예시/주석 등 첨부) 활용
- LangChain의 Skill Middleware 등 활용, Skill Injection 시점/조건(프로그레시브 디스클로저), Relationship 추가(항상 Skill 먼저 활용 후 도구 실행)
- 시스템 프롬프트/설명서 강화로 에이전트가 올바른 쿼리로 수정하여 원하는 토크(‘GPA’ 세션 등) 성공적으로 검색
- 결과적으로 집계, 필터 등 복잡 쿼리도 직접 실행이 가능해짐(예: ‘4월 8일 세션 수 카운트’ 쿼리→ 27세션 즉시 집계 등)

### 로컬 파일 시스템 검색(Shell Tool 활용), Bash 명령에 대한 LLM의 강점 및 한계 시연

- "모든 에이전트엔 Shell Tool과 파일 시스템만 있으면 충분한가?" 주제 실습
- 폴더별 세션 데이터(메타/제목/설명 포함) 구조화, 시스템 프롬프트에 파일 시스템 구조 명시
- Shell tool 허용시 LLM은 직접 ls, grep 명령으로 폴더 구조 확인, 키워드 검색, 파일 읽기 일련 과정을 자율적으로 진행
- 특정 세션 검색시, 폴더 구조 파악 후 grep으로 키워드 검색, 필요시 파일 내용까지 읽어 결과를 생성
- 시맨틱 검색 대체를 위해 Bash 명령으로 연속적 유의어 체이닝(‘regulate’, ‘compliance’, ‘constraint’, ‘GDPR’, ‘governance’ 등 여러 용어 grep 연결) 시도
- 실제로 유의미하게 연결된 세션 결과를 도출하지만, 너무 브루트포스식, 비효율적(동물 히어로 영화 검색 등은 현실적으론 불가능)임을 시연

### 외부 CLI(예: GINA Grap)의 연동으로 파일 시스템 내 시맨틱/하이브리드/멀티벡터 검색도 에이전트 레벨로 구현 가능함을 시연

- GINA Grap 등 현대적 시맨틱 파일 검색 CLI를 Shell tool과 연동하면, LLM이 shell에서 그랩을 호출하여 파워풀한 검색을 수행 가능
- 시스템 프롬프트에 CLI명령 예시, 시멘틱/정확매치 둘 다 사용할 조건 명시
- LLM이 ‘regulatory constraints’ 등 시맨틱 쿼리에서 곧바로 gina grap 호출 → 실질적으로 첫 시도만에 주요 세션 다수 정확 검색 성공
- 다양한 CLI(라이트온 co-grap, LlamaIndex의 SamTools 등)도 동일 방식으로 확장 가능함을 실습

### 도구 선택/조합시 "전문화 도구(낮은 진입장벽, 효율적)와 범용 도구(높은 확장력)"의 균형과 로그 기반 최적화 절차를 제안

- 하나의 만능 도구에 의존하는 것은 적합하지 않음
- "전문화 도구"는 파라미터가 간단하고 LLM이 실수 없이 사용하기 쉬우며 빠르고 효율적(예: 단순 시맨틱 검색, ID 기준 조회)
- "범용 도구"는 복잡 쿼리, 예기치 못한 검색까지도 대응 가능(단, 시행착오/시도 수 ↑, 효율 ↓)
- Elastic에서는 UXR에서 'Low floor(쉬운 진입, 빠른 사용)', 'High ceiling(확장, 복잡 과제 처리)'의 조화를 중시하며, 에이전트에 두 종류 도구 병행 제공 권장
- 유저 쿼리 패턴/행동 미확정 단계에서는 범용 도구 우선 제공, 쿼리 로그 분석 후 자주 쓰이는 복잡/전문 쿼리에 맞는 특화 도구 추가 및 조합
- 로그상 도구 호출 횟수 4~5회 이상 등 비효율적 행동 발견시 도구 구조 점검, 인터페이스 개선, 전문화 도구 도입

### Q&A: 모델 크기, 에러율, 도구 선택, 하이브리드 도구, Threshold, 서브에이전트, 스킬 관리 등 실무 질문 답변

- 더 강력한 LLM(대형 모델) 사용시 범용 도구의 파라미터 에러율이 현저히 감소
- RAG 대비 에이전틱 RAG는 검색 정확성/유연성은 높으나 지연(latency)이 높아, 두 체계를 하이브리드로 분기 적용하는 방법도 논의 중
- 쿼리 용어나 문법 상 차이(SQL vs ESQL 등)는 설명 강화/Skill 보완/자동 변환 도구 등으로 보완 가능 (단순한 대체로 충분치 않을 수 있음)
- Shell tool과 EBSQL(또는 파일 검색) 등 도구를 혼합하여, 서로 결과 검증하여 신뢰도 상승시킨 사례(Vercel 블로그 벤치마크, Bash+DB tool 조합 정확도 ↑)
- 시맨틱 검색 결과 Relevant Threshold(상관 기준치) 조절시, LLM이 결과 판별 능력 있지만, 긴 대화(롱텀 세션)에서 불필요한 정보 누적으로 혼란 가중될 수 있음
- 서브에이전트(Sub Agent) 활용 경험 묻는 질문에, 직접적 사례는 없지만, CloudCode 등에서는 특정 도메인 질문에 특화 서브에이전트 활용 중
- Agent Skill Progressive Disclosure(점진적 공개) 및, 긴 세션 내 컨텍스트 윈도우 최적화 방법: Skill 메타만 system prompt에 유지, 필요시에만 추가 정보 로딩, 불필요시 오프로드(파일스토어 기반), 컨텍스트 리사이클 등 권장
- 전체적으로 실제 Agent 설계시, 도구군 설계, 모니터링 로깅, 컨텍스트 관리정책, Skill Progressive Disclosure 등 구체적 실전 설계법 및 노하우 공유
