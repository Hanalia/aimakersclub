## *Google DeepMind가 대규모로 에이전트를 운영하는 방법* 핵심 요약

- Google DeepMind 엔지니어 KP Sawhney와 Ian Ballantyne가 에이전틱(agentic) 소프트웨어의 내부 구조와 대규모 운영 방안에 대해 패널 형식으로 설명
- 에이전틱 작업을 수행하는 자체 IDE 플랫폼 ‘Antigravity’의 데모와, 여러 에이전트를 동시에 프로젝트별로 실행 관리하는 시스템을 시연
- Antigravity는 내장 에이전트 매니저, 플래닝 시스템, 인간 피드백 인터페이스, DOM 검사 및 실행 결과 리포트, 스크린샷·영상 캡처 등 다양한 기능을 포함
- 대규모 코드베이스(모노레포) 내에서 코드, 리서치, 계획 등 다양한 작업을 효율적으로 분담하고, 파일 시스템 협업 방식 확장 논의
- 내·외부적으로 우수한 ‘Skill(기능)’을 엄격하게 선별 및 관리하는 시스템 도입. 대형 기업 특성상 스킬 확산 관리가 중요함을 강조
- 에이전트의 확장(스케일링) 시 토큰 소모, 연산 리소스 한계 등 비용 및 자원 관리가 핵심 과제
- 사용자의 에이전트 인스턴스 과다 생성 방지 등, 사용자별 쿼터 관리와 전체 시스템 안정성 확보 방안 설명
- 내부 관제 시스템, 자동화된 트래픽 모니터링, 에이전트 경로(trajectory) 추적 및 진단 도구 활용
- 다양한 모델(예: Gemma 4, Gemini 모델 등)을 에이전트 워크플로 내에서 유연·자동으로 전환해 자원 소진 시에도 업무 연속성 유지
- 에이전트 기반의 코드 자동 검토, 다양한 수준의 자동화된 리뷰 절차 및 평가 인프라 마련

## 세부 요약 - 주제별 정리

### Antigravity는 여러 에이전트 관리 및 계획·피드백·실행 자동화를 지원하는 통합 IDE 플랫폼임

- Antigravity는 Visual Studio와 유사한 인터페이스지만, 내부에 에이전트 매니저와 계획·실행·분석 기능을 포함
- 사용자는 프로젝트 별로 여러 개의 에이전트를 한 번에 실행하고 관리할 수 있음
- 파일별로 사양(spec)을 주면 에이전트가 분석하여 내부 툴을 활용, 자동으로 실행 계획을 수립·수행
- DOM 구조를 직접 검사하고 웹 페이지 기능을 추가하거나 수정하며, 필요한 경우 브라우저를 띄워 직접 조작
- 사용자가 ‘human-in-the-loop’ 식으로 에이전트가 제안한 계획을 직접 수정·피드백 가능. 필요 시 중단도 가능
- 작업 완료 시 결과 리포트, 스크린샷·영상 캡처, 작업 중 노트(스크래치패드) 등 다양한 산출물 제공

### DeepMind 내부에서는 대용량 코드베이스와 다양한 용도에 맞춘 에이전트 기반 시스템을 확장 중임

- DeepMind는 거대한 모노레포 환경에서 Antigravity의 ‘에이전트 하니스(harness)’를 다양한 워크플로에 적용
- 단순 코딩 자동화뿐 아니라 리서치, 데이터 파이프라인 등에도 적용하며, 커다란 텍스트 블롭 대신 공유 파일 시스템 협업 구조 채택 모색
- Deep Research Agent 등 일부 최신 기능은 API로도 공개됨. 내부에서는 다양한 부서가 맞춤형 기능을 실험·적용
- 향후 더 폭넓은 활용 및 확장 방향(예: Deep Research의 협동적 요소, 워크스페이스 단위 협업 등)을 논의 중

### 우수한 스킬의 선별 및 관리가 대규모 조직 내 에이전트 운용 효율성의 열쇠임을 강조함

- 에이전트의 “스킬 라이브러리”가 업무 효율에 핵심적이나, 복잡한 조직에서는 품질·용도 편차로 확산 관리가 필요
- DeepMind는 “최고의 스킬만 살아남는” 방식으로 엄격하게 스킬을 선별(진화론적 관리)함
- 직무 별, 업무별로 전문성이 높은 구성원이 직접 스킬을 기여
- 사용자는 검증된 스킬과 툴을 CLI, Inspector 등 다양한인터페이스에서 활용 가능

### 에이전트 확장 시 토큰/연산 자원 관리 문제와 사용자별 쿼터 제약이 현실적 난제로 부각됨

- 에이전틱 시스템은 “토큰 헝거(token hungry)” 특성으로, 대규모 병렬 작업시 트래픽/비용 관리가 최우선 현안임
- 팀/사용자 별로 토큰, GPU/TPU 자원 할당량(쿼터)을 엄격히 관리하여, 시스템 과부하·비효율 방지
- 파워 유저가 수십~수백 개 인스턴스를 남용할 경우, 관리자가 작업 중단을 요청하는 실시간 관제 시스템 운용
- Anthropic의 open-crawl 차단, 구독 모델의 한계 사례 인용 등, 토큰/연산 경제성 논의

### 다양한 AI 모델을 상황에 맞게 조합·전환하여 연산 자원과 효율성을 극대화함

- Gemma 4(로컬 GPU/TPU 기반, 쿼터 무제한)와 상위 Gemini 모델(유료)에 대한 ‘믹스앤매치’ 전략 제안
- 특정 에이전트 단계는 저가형 모델을, 일부 고차원 작업엔 고성능 모델을 적용해 비용 최적화
- 자원 소진(토큰/쿼터 한도 도달) 시, harness가 자동으로 모델을 전환하며 사용자의 워크플로를 중단없이 이어감
- 향후에는 ‘신용 한도’ 초과를 사용자가 명확히 인지하고, 작업 오류나 지연을 최소화하는 UX 개발에 집중

### 내부 관제 도구와 에이전트 진단·평가 시스템을 구축하여 운영 안정성과 원인분석을 지원함

- 커스텀 웹앱 형태의 내부 관제 UI를 통해 에이전트가 수행하는 모든 쿼리·트랜잭션·예측 요청 추적 가능
- 계층별 모니터링/디버깅, 필요시 원시 predict 요청까지 드릴다운이 가능함
- ‘에이전트 trajectory store’를 도입하여, 작업 흐름 내 반복(loop) 및 비정상 현상 추적·진단
- 대규모 자원 사용은 24/7 실시간 모니터링 팀(SRE)이 그래프·스파이크 감시 및 즉각적 개입 실시

### 에이전트 기반 코드 자동 리뷰, PA별 검수 프롬프트, Jewels 등 코드 품질/리뷰 자동화 인프라 마련

- 언어별 auto review 모델(스타일가이드, 과거 우수 예시로 파인튜닝), PA/제품별 맞춤 리뷰 프롬프트 체계화
- PR 제출 시, 별도 명령없이 에이전트가 자동 리뷰 및 코멘트를 제공
- Jewels 등 웹 기반 자동 리뷰 인터페이스로, 대규모 코드 생성 시대에 맞춘 관리 체계 확립
- GitHub 등 외부 PR에서도 연동하여 자동 코드 검수 및 품질 향상 지원
- 에이전트가 PR을 포함해, 리뷰 자체도 자동화하려는 시도가 확대

### 대규모 에이전트 시스템의 평가, 샌드박싱, 데이터셋 구축에는 여전히 많은 과제가 남아있음

- 샌드박스 환경 구성 및 복잡계 문제세트 평가에 드는 엔지니어링 비용이 높음
- 공개된 오픈소스 데이터셋을 활용하나, 특정 스킬의 벤치마킹을 위한 데이터 구축은 스킬 개발자 책임이 큼
- 일부는 에이전트 자체가 평가용 벤치마크·테스트를 설계하는 실험도 진행(메타적 접근)
- 평가 시스템의 자동화·효율화 필요성이 지속적으로 제기

### 사용자·팀 규모 확장 시, 에이전트 인스턴스 관리와 모델 사용 정책 변화가 예상됨

- 워크플로 우선 순위, 모델별 정책(프로·일반·로컬 등)에 따라 자원 분배 및 자동 우회가 일반화될 것
- 에이전트가 개인/조직 구분 없이 자원 한도에 도달할 경우 자동 모델 전환·대체가 기본 전략
- “작업 중단 없이 연속성 유지”를 최우선 UX로, ‘사용자 친화적’ 자원 관리 인터페이스 강화 예정

### 인간 ‘슈퍼바이저’ 역할과 에이전트-에이전트 커뮤니케이션이 미래 확장의 핵심이라고 내다봄

- 다수 에이전트가 동일 프로젝트 내 평행적으로 부분 작업 분담하는 구조는 아직 완전히 정형화되지 않음
- 미래에는, 인간이 디지털 조립라인의 감독자(supervisor) 역할로, 에이전트간 커뮤니케이션·분업·감독이 자연스러운 시스템 방향을 제시
- 에이전트-에이전트 조율/의사소통 효율화 방안 및 인간 설계 참여의 중요성을 강조

### 커뮤니티·외부 개발자 생태계와의 협력 및 향후 공개 계획을 조심스럽게 언급함

- 내부 핵심 인프라 일부는 커스텀 기반이나, 공개용 인터페이스(Gemini, API 등)로 점진적 확장 계획
- 커뮤니티의 사용 패턴과 요구에 따라, harness와 모델, CLI·Skill 생태계의 동반 지원 방침
- 향후 발표(Google I/O 등)에서 구체 사례 및 확장 계획 공개 예정
