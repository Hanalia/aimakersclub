## *AI에서 하니스(Harness)의 역할과 구현 방법 심층 탐구* 핵심 요약

- 발표자는 IBM의 AI 개발자 에반젤리스트 Tejas Kumar로, AI 하니스(harness) 개념의 정의와 실제 활용 가치를 심도 있게 다룸
- "하니스" 용어는 머신러닝과 AI 엔지니어링 영역에서 차이가 있는데, 이 강연은 AI 에이전트 중심의 하니스를 설명함
- 하니스가 필요한 결정적 이유는 우리가 빌리는(사용하는) AI 모델이 블랙박스이며, 신뢰성과 일관성을 위해 제어 가능한 환경이 필요하기 때문임
- 하니스는 "모델을 현실(안정적인 환경)에 연결하고, 에이전트가 일관되게 작업할 수 있도록 돕는 것"으로, 도구 레지스트리, 모델 선택, 컨텍스트 관리, 가드레일(안전장치), 에이전트 루프, 결과 검증(verify) 단계 등 필수 구성요소로 이루어짐
- 단순한 프롬프트 엔지니어링만으로는 에이전트의 신뢰도를 확보할 수 없으며, 하니스가 직접적인 행동 제어와 검증을 담당함
- GPT-3.5 Turbo(2023년 모델) 등 저렴한 모델도 하니스를 통해 실제 업무(예: Hacker News 첫 글 추천)를 성공적으로 처리할 수 있음을 데모로 입증함
- 점진적으로 가드레일, 컨텍스트 압축, 결과 검증, 자동 로그인 등 하니스를 확장하며, 각 단계별로 신뢰성과 안정성이 향상됨을 실제 코드/실행 예시로 설명함
- 하니스가 실제 엔터프라이즈 환경(예: IBM의 open rag 프로젝트)에서 데이터 보안 및 안정적인 RAG 등 다양한 AI 오퍼레이션의 핵심 인프라임을 강조함
- 발표 내내 프롬프트(지시문)는 변경하지 않고, 오직 하니스 구조 개선만으로 결과를 변화시킨 점을 강조
- 미래에는 동적으로 생성되는 셀프-어웨어 하니스(에이전트가 작업 전 자체 하니스를 만들고 활용)가 AGI(범용 인공 지능)로 가는 중요한 단계가 될 것으로 전망함

## 세부 요약 - 주제별 정리

### AI 하니스는 블랙박스 모델을 신뢰성 있게 사용할 수 있게 해주는 안정 장치임

- AI 모델은 대부분 블랙박스 형태여서, 제공사 측의 사양 변화나 성능 차이를 사용자가 통제할 수 없음
- 대부분의 유저(회사)는 모델 사용에 월 구독료(예: Claude Pro 20달러/월)를 지불하지만, 성능이나 신뢰도에 대한 확신은 없음
- Opus 등 특정 모델을 선택했다고 해도, 제공업체가 실제로 Sonnet을 서빙해도 사용자가 알 수 없는 환경임
- "하니스"의 본질은 에이전트가 일관되게 동작하도록 환경을 통제·안정화하는 것에 있음
- 하니스가 있을 때 다양한 변수(모델 업그레이드, 컨텍스트 제한, 시스템 변화)가 있어도 안정적인 결과물을 확보할 수 있음

### 머신러닝 영역과 AI 에이전트 영역에서의 하니스 개념은 본질적으로 다름

- 머신러닝에서의 하니스: 모델 테스트용 입력/출력을 관리하는 테스트 스위트(테스트 러너)에 가까움
- AI 에이전트 엔지니어링에서의 하니스: 실제 환경과 에이전트 사이를 연결하는 통제 시스템 구성임
- 이 강연에서는 AI 에이전트 하니스의 구조와 실용성에 초점을 맞추어 설명

### 에이전트 하니스는 도구 레지스트리, 모델, 컨텍스트 관리, 가드레일, 에이전트 루프, 결과 검증 등 핵심 컴포넌트로 구성됨

- 도구 레지스트리: 예시로 Claude Code, Cursor, Codex 등은 파일 읽기/쓰기, bash 명령 실행 등 다양한 툴 제공
- 모델: 사용자가 선택하거나 시스템이 선택하는 다양한 LLM(대형 언어 모델)
- 컨텍스트 관리: 에이전트가 동작하는 동안 필요한 정보 유지를 위한 프리미티브 존재 (예: 대화 히스토리 축약)
- 가드레일(Guardrail): 최대 실행 단계, 메시지 제한 등 통해 "폭주"를 막는 제한장치 ex) 5회 이상 도구 호출 시 실행 중단
- 에이전트 루프: 에이전트가 "반복적으로" 작업을 시도하며 조건을 만족할 때까지 동작
- 결과 검증(Verify): 예를 들어 코딩 에이전트면 lint/test를 자동 실행, 실패 시 즉시 알리고 성공 검사 후 종료

### 프롬프트만으로는 해결이 안 되는 "행동 오류"는 하니스를 통해 직접 제어해야 함

- 사용자가 흔히 프롬프트 수정(예: "항상 로그인하세요" 추가)만으로 에이전트 행동을 교정하려 하지만 한계가 있음
- 하니스는 프롬프트를 건드리지 않고, 코드/시스템 상에서 행동의 검증과 제어, 예외 처리를 수행함
- 프록시(validation step)과 루프, guardrail을 통해 실패 상황(예: 로그인 실패, 거짓 성공 보고 등)에 즉각 대응 가능

### "Hacker News 글 추천" 데모를 통해 하니스의 구조와 적용 방법을 구체적으로 구현함

- 목표: 저렴한 LLM(GPT-3.5 Turbo)로 "Hacker News 첫 글 추천" 자동화 에이전트 만들기
- 프롬프트는 변하지 않고, 하니스 구조만 점진적으로 보강하며 문제를 해결해 감
- Playwright를 사용해 브라우저 세션(Chromium) 실행, 도구 등록, 에이전트 루프, 컨텍스트 및 히스토리 적재 등 단계적으로 구현

### 1차 하니스: 기본 컨텍스트 및 도구 적용, 실패 상황(로그인 문제, 거짓 성공 보고)을 둘러싼 문제 확인

- 단순 실행 시 로그인 화면을 만났을 때 에이전트가 비정상 종료하거나 거짓으로 성공했다고 리포트함
- 히스토리를 남기되, 별도의 검증(lock step)은 없는 상태

### 2차 하니스 개선: Guardrail(최대 반복/메시지 수), 컨텍스트 압축 등 기본 하니스 개념 도입

- max_iterations(6단계 초과 시 종료), max_messages(지정 메시지 수 초과 시 컨텍스트 압축) guardrail 적용
- 컨텍스트 압축 로직: 시스템 프롬프트 + 유저 프롬프트 + 최신 대화 2개만 유지(매우 단순화 버전)
- 하니스 구성은 별도 함수(harness.js 등)로 모듈화, index(엔트리포인트) 파일은 run_harness로 간결화

### 결과 검증/수정 로직 추가: 행동 이력 검증, 거짓 보고 방지, 자동실패 처리

- 하니스에 run_harness_attempt, verify_successful_upvote 등 검증 함수 삽입
- agent가 upvote 도구를 실행했다고 무조건 성공처리하지 않고, 히스토리 상 실제 upvote 발생여부, 로그인 실패/미복구 리디렉트 등 조건별로 실패 처리
- 검증 로직에서 실패 상황 감지시 즉각 루프 중단·실패 보고 → 에이전트가 결과를 왜곡하거나 "성공"이라고 거짓 응답하는 문제 방지

### 자동 로그인 핸들러 도입: 단계별 하니스의 기능적 확장과 효과 검증

- login_handler 함수 구현: 브라우저가 로그인 페이지 접근 시, 하니스가 직접 credential을 채우고(form 입력/제출) 로그인 처리
- 자동 로그인 후 에이전트가 본연의 작업(upvote) 성공적으로 진행, 실제로 Hacker News 글 추천 성공 및 확인
- 로그인 단계는 하니스가 에이전트보다 우선적으로 개입하여, 보안 및 로그인 안정성까지 책임짐

### 하니스 구조 개선만으로 저렴한 모델에서 고신뢰 자동화가 가능함을 실증

- GPT-3.5 Turbo, Quinn, GPT-OSS 등 상대적으로 느리고 저렴한 모델도 잘 설계된 하니스만 있으면 실제 환경에서 업무 자동화 가능
- 하니스의 도입/개선을 통해, 프롬프트 변경 없이 결과의 신뢰성과 반복가능성을 극적으로 향상시킴

### IBM의 오픈 소스/엔터프라이즈 활용 사례로서 하니스의 중요성이 대기업 환경에서도 검증됨

- IBM에서 오픈소스로 제공하는 Open RAG 시스템은 매우 강력한 하니스 구조를 통해 내부 사일로 데이터(RAG, PDF, Teams calls, 인보이스 등)에서의 신뢰성과 보안을 보장
- 대규모, 민감한 데이터를 다루는 엔터프라이즈 환경에서 하니스를 통한 통제(guardrail, 검증, 자동화)가 필수 인프라임을 강조

### 하니스의 미래: 동적, 자가 생성형 하니스가 AGI 진화를 촉진할 것이라는 전망

- 연도별 예측: 2025년은 "에이전트의 해", 2026년은 "하니스의 해", 2027년은 "동적 하니스 자동 생성의 해"
- 미래에는 에이전트가 작업 전 자체 하니스를 생성(플랜 모드+), 위험구간(예: 환각, 실패) 인지 및 방지 등 메타에이전트적 행동 가능
- 이러한 "자가 생성·동적 하니스"는 AGI(범용 인공지능)에 필수적이자 논리적인 진화 단계로 제시

### 결론 및 요약: 하니스는 AI 에이전트 신뢰성과 안전성의 궁극적 열쇠임

- 18분간의 딥다이브에서 하니스의 정의·구조·단계별 확장 및 실질적 영향(프롬프트 외부 제어)을 구체적으로 입증
- 하니스는 단순한 기술 요소가 아니라, AI 시스템의 현실적 문제(검증·신뢰·안정·보안) 해결의 중심임
- 발표 자료와 코드는 GitHub에 공개되어 있으며, 추가적 논의를 환영함
