## *원샷으로는 충분하지 않다 — Granola의 Mehedi Hassan* 핵심 요약

- 이 영상에서는 Mehedi Hassan이 Granola(크로놀라)에서 제품 엔지니어로서 경험한 AI 기능 개발 및 제품 개선 사례를 공유한다.
- Granola는 실시간 전사 및 맞춤형 회의 요약을 제공하는 데스크톱 기반 회의 노트 앱이다.
- 주 사용자층의 다양한 요구(예: 세일즈, 엔지니어, HR)에 맞춰 한 가지 prompt나 LLM(대규모 언어모델)로 모든 사용자를 만족시키기 어렵다는 점을 강조한다.
- LLM 기반 기능(예: 회의 질의 응답, 웹 검색 추가 등)을 단순히 적용하는 것만으로는 실사용자에게 충분한 품질을 제공할 수 없음을 지적한다.
- 기본적인 AI 기능 도입 후 실제 배포 단계에서 사용자들의 피드백에 기반하여 지속적으로 세밀한 개선 및 커스터마이징이 필요함을 구체적으로 설명한다.
- LLM 기능에서는 개발자가 제어할 수 없는 외부 요인(예: 웹 검색 API의 변화, 토큰 비용 급증 등)들이 실제로 장애요인임을 사례와 함께 소개한다.
- Granola 내부에 LLM 툴콜 트레이싱 시스템을 구축해 ‘블랙박스’ 같은 AI 동작 과정을 투명하게 추적하고, 다양한 부서와 직군이 직접 원인 파악 및 개선에 활용할 수 있게 했음을 밝힌다.
- 데스크톱 앱의 테스트와 다양한 기능 실험에 속도를 더하기 위해 Electron 앱의 렌더러를 웹 환경에 맞춰 분리하고, CI-프리뷰와 자동화된 LLM 기반 테스트까지 도입한 개발 인프라 향상 사례도 상세히 언급한다.
- 마지막으로, AI/LLM 기능 개발에서 중요한 것은 ‘한 번에 완벽한 결과를 노리는 것’이 아니라, 사용자와의 피드백 루프 속에서 꾸준하게 품질을 개선하는 프로세스 자체임을 강조한다.
- Q&A에서는 Electron과 Tauri 플랫폼 간 성능 비교에 대한 짧은 의견도 덧붙인다.

## 세부 요약 - 주제별 정리

### Granola는 실시간 전사 및 다양한 커스터마이징이 강점인 회의 노트 앱임을 시연과 함께 소개함

- Mehedi Hassan은 Granola(크로놀라)의 제품 엔지니어로, jQuery 시절부터 다양한 프런트엔드 변화를 겪었음을 밝힘.
- Granola는 사용자가 회의 도중 전사 및 마이크 오디오 접근이 실시간으로 가능하도록 설계됨.
- 회의 종료 후 자동으로 수준 높은 요약 노트를 제공하며, 사용자가 직접 추가 노트도 작성 가능.
- 데모에서는 ‘20% 오버랩’ 등 사용자가 개별적으로 강조한 부분까지 합쳐 최종 노트를 자동 생성하는 과정을 보여줌.
- HR, 세일즈, 엔지니어 등 직군별로 원하는 회의 요약의 형태가 다르며, Granola의 철학은 어떤 역할의 사용자에게도 ‘방해되지 않으면서’ 최고의 회의 기록을 제공하는 데 있음.

### LLM 기반 기능을 도입하면 단순 배포가 아닌 실제 사용 경험에 집중해야 함을 설명함

- 기본적인 LLM 기반 챗봇 기능(예: 회의 질의 응답)을 ‘원샷’ 방식으로 간단히 개발해 배포했다고 가정.
- 실제 사용자는 예기치 않은 요구와 문제(도시 목록 요청 시 느린 검색, 맞춤 이메일 미작성, 코치 관련 문맥 오류 등)를 대거 경험함.
- ‘일반적인’ 챗봇은 사용자 개개인의 맥락적 요구를 제대로 충족시키지 못하는 한계가 존재함.

### 웹 검색 등 외부 도구 연동은 단순 ‘라인 한 줄’이 아닌 수많은 파생 문제를 내포함

- 대부분의 LLM 제공자는 웹 검색 도구 추가가 ‘코드 한 줄’이면 된다고 설명하지만, 실제로는 여러 문제가 발생함.
- 복잡한 질의에서는 토큰 사용량과 비용이 급증. 예를 들어 채팅 하나에 10펜스 이상의 비용이 발생할 수 있음.
- 수백만 유저에게 서비스를 제공하게 되면 이 비용은 현실적으로 감당하기 어려움.
- 웹 검색 품질은 외부 공급자의 내부 정책이나 업데이트 변경에 즉각적으로 영향을 받음.
- Granola 팀은 한동안 만족스럽게 쓰던 모델이 하루 만에 컨트롤 불가의 품질 저하를 겪은 경험을 설명.
- 이런 불확실성을 해결하려면 ‘그냥 붙이기’가 아닌 회사가 충분히 제어할 수 있는 체계를 구축해야 함.
- 실제로 대형 수십억 달러 검색 기업이 존재한다는 사실 자체가, 이 기능이 결코 단순하지 않음을 뒷받침.

### 사용자 지향적 요약은 다양한 직군의 니즈를 충족하는 맞춤형 설계가 필요함을 강조함

- 동일한 요약이라도 세일즈 직군은 거래 상황에, 엔지니어는 액션아이템이나 이슈에, HR은 전혀 다른 관점에 집중.
- 한 가지 프롬프트/요약 메커니즘이 모든 사용자에게 적합하지 않음을 구체적으로 예시로 설명.
- LLM의 동작은 블랙박스처럼 불투명하기 때문에, 원하는 대로 동작하게 만들기 위해서는 내부 과정을 깊이 파악할 필요가 있음.

### Granola는 LLM 툴콜 트레이싱 시스템을 자체 구축해 ‘블랙박스’ 문제를 극복했음을 상세히 밝힘

- 팀 내부적으로 LLM의 모든 툴콜(검색·추론·비용 등) 실행 흐름을 완전히 가시적으로 관찰할 수 있도록 tracing tool을 개발함.
- 시스템은 각 툴콜의 사유, 사용 도구, 원인, 비용 등을 데이터베이스와 특화된 UI에 통합 저장·표현함.
- 이 시스템은 엔지니어뿐 아니라 제품 기획, 데이터, 고객지원(CX) 등 다양한 부서에서 효과적으로 활용됨.
- 복잡한 로그 분석 없이도 UI에서 바로 실패 원인, 문제 지점 등을 직관적으로 파악 가능함.
- 창업자까지도 자체 툴로 LLM agent의 실행 루프 전체를 따라가며 문제분석을 직접 수행함.
- 이를 통해 정확히 무엇이, 어떤 맥락에서 잘못되었는지 빠르게 원인을 진단하고, 반복적으로 개선할 수 있는 기반을 마련함.

### 예전에는 SaaS에 의존했던 기능을 LLM 활용으로 ‘직접’ 빠르게 구축할 수 있게 됨을 언급함

- SaaS(소프트웨어 서비스)에서 제공하는 툴을 활용하는 것에 비해, 직접 맞춤형 tracing system을 개발하는 데 시간이 단축됨.
- 간단한 경우 AI SDK 래핑과 DB 저장, front-end 개발 위주로 툴을 빠르게 구현할 수 있음을 예시로 제시.
- OpenTelemetry 같은 타 데이터 수집 플랫폼도 응용 가능하나, Granola는 목적에 맞춘 간소화를 선택함.

### 내부 CI·QA 시스템 혁신으로 다양한 기능 실험과 개발속도를 크게 높임

- Granola는 Electron 기반 데스크탑 앱으로, 한 번에 한 인스턴스만 실행 가능해 동시 피처 실험 및 테스팅에 제약이 있었음.
- 기존에는 개발자가 일일이 로컬 환경 세팅·의존성 설치·직접 테스트를 해야 했고, 협업 테스트가 비효율적이었음.
- Electron 앱 프론트엔드를 웹 쉘로 분리해 온라인 배포가 가능하도록 구조를 바꿈.
- 이제 PR(풀리퀘스트)마다 CI 기반 프리뷰 링크가 자동 생성되어 누구나 쉽게 동시 테스트 가능.
- LLM을 활용한 자동화 테스트 예시: PR 오픈 시 ‘Cursor’가 자동으로 테스트 후 스크린샷을 PR에 첨부.
- Electron 아키텍처 내 ‘main’(시스템 API 담당)과 ‘renderer’(프론트엔드) 분리를 자세히 설명.
- IPC(System API)와 React 관련 API(router, session 등)를 웹 표준으로 대체하여 웹과 데스크탑 환경 모두에서 동일하게 테스트/실행이 가능해짐.

### 다양한 피처 실험 및 반복 테스트를 통해 실제 사용 경험을 최적화할 수 있었음을 사례로 설명함

- 구조 전환 이후에는 단일 Electron 앱이 아닌, 웹 앱으로도 여러 피처와 UI 버전을 동시에 실험 가능해짐.
- 실제로 Figma 등 목업툴에서 보는 것이 아닌, 진짜 환경에서 체험하며 사용자 경험을 다각도로 검증·개선.
- 결과적으로 더 많은 제품 버전을 빠르게 시도·채택함으로써 최종 사용자에게 최적화된 결과물을 제공할 수 있었음.

### AI 기능 개발의 핵심은 ‘원샷(one-shot)’ 개발이 아닌 치밀한 피드백 루프와 반복적 개선임을 강조함

- LLM 도입으로 예전보다 빠르고 다양한 시도가 가능하다는 장점은 분명히 존재함.
- 그러나 단순히 ‘한 번에 잘 작동하는 시스템’을 목표로 하는 방식은 현실적으로 한계가 있다고 명확히 지적.
- ‘테니스 게임’을 하듯, LLM 기능과 사용자 간의 끊임없는 상호 피드백을 통한 프로덕트 발전이 핵심임을 강조.
- 최종적으로, 신중한 개선과 점진적인 프로덕트 개발만이 마치 ‘마법’처럼 사용자를 감동시키는 실제 경험을 만들어냄.

### Electron과 Tauri 플랫폼 간 전환 여부에 대한 단답형 Q&A 및 입장을 간단히 밝힘

- 청중 질문으로 ‘Electron에서 Tauri로 리플랫폼할 생각이 있나?’에 대해 답변.
- Electron은 팀의 니즈(특히 API 변화, 신뢰성, 주요 성능 관점)와 잘 맞는다고 언급.
- Tauri도 실험해봤지만 특별한 성능 향상을 느끼지 못했다고 솔직히 밝힘.
- 현재로서는 전환 계획이 없으나, 관련 논의는 내부적으로 여러 차례 이뤄졌음을 언급.
