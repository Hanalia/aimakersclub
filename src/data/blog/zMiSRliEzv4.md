## *제품이 스스로 PR을 생성하는 '셀프 드라이빙' 파이프라인: 신호에서 PR까지* 핵심 요약

- 영상 제목은 "*제품이 스스로 PR을 생성하는 '셀프 드라이빙' 파이프라인: 신호에서 PR까지*"입니다.
- PostHog의 Josh Snyder가 제품의 관찰 데이터(Observability)를 바탕으로 자동으로 Pull Request(이하 PR)를 생성·배포하는 파이프라인 구축 과정을 소개합니다.
- 기존의 제품 관찰 방식은 데이터 수집 후, 사람이 대시보드에서 문제를 확인·분석·수정하는 등 느린 프로세스를 거쳐야 했습니다.
- 새로운 파이프라인은 각종 제품 신호(에러, 세션 리플레이, 로그 등)를 실시간 수집·정제·그루핑 후, AI 에이전트가 이를 분석해 바로 PR을 생성·관리합니다.
- 신호는 정규화 및 임베딩 후, 기존 임베딩 기반 군집화의 한계를 개선하여 LLM 기반 쿼리-임베딩 방식으로 효과적으로 문제를 묶어냅니다.
- 연구 에이전트는 문제 요약, 우선순위 산정, 담당자 식별까지 자동화하며, Linear, Notion 같은 도구와 연계해 정확도를 높여줍니다.
- PR 생성 단계에서는 샌드박스 환경에서 실제 코드를 수정하고, 지속적 통합(CI) 체크와 재시도 과정을 반복해 최적의 수정 PR을 만듭니다.
- 도입 과정에서 임베딩 모델의 한계, 신호 분류·그루핑 방법, 에이전트 활용/비용 최적화, 적극적인 프로덕션 데이터 평가 등 여러 시행착오와 교훈이 있었습니다.
- 향후 목표는 ‘제품이 스스로 개선되는 자동화’로, 에이전트가 피드백을 받아 매 번 더 나은 PR을 생성하고, 개발자는 반복적 버그나 제품 실험에 신경 쓸 필요 없이 주요 기능 개발에만 집중할 수 있게 하는 것입니다.
- 현재 이 파이프라인은 알파(Alpha) 단계로, 앞으로 점진적으로 확장될 예정입니다.

## 세부 요약 - 주제별 정리

### PostHog는 대규모 제품 데이터 관찰·분석 툴로 출발하여 다양한 기능을 제공함

- PostHog는 원래 제품 분석(Analytics) 회사로 시작했으나, 세션 리플레이, 웹 분석, 에러 트래킹, 실험 등 다양한 기능을 확장하여 제공합니다.
- PostHog 툴을 제품에 연동하면, 다양한 소스에서 방대한 양의 데이터를 실시간 수집합니다.
- 이 데이터는 주로 사용자가 직접 대시보드에서 조회·분석하는 목적으로 제공되고 있었습니다.
- Josh Snyder는 본 발표가 PostHog 홍보 목적이 아니며, 데이터 자체와 기술 파이프라인 공유가 목적임을 강조합니다.

### 기존 제품 문제 해결 과정은 느리고 비효율적인 수작업 중심임

- 현재 관찰 시스템은 신호가 발생하면 대시보드의 메트릭 변화로 확인 가능합니다.
- 담당자는 보통 몇 시간 또는 며칠 뒤 대시보드 변화를 확인 후, 문제를 찾고, 문제의 긴급도를 평가해 이슈 트래킹 시스템(예: Linear)에 등록합니다.
- 이어 개발자는 적절한 시점이 되면 PR을 생성해 코드 수정 후 리뷰와 배포를 진행합니다.
- 전체 프로세스는 신호 발생부터 문제 인지·수정까지 최소 수 시간에서 수일이 소요되며, 많은 소프트웨어 엔지니어링의 반복적 노동이 이뤄집니다.

### 미래형 파이프라인은 신호 발생부터 PR 생성·배포까지 자동화하여 비약적으로 빠르게 전환함

- 지금 추진 중인 파이프라인에서는 신호가 발생하면 즉시 백그라운드 분석 에이전트가 문제 원인을 파악합니다.
- 사람이 대시보드나 에러, 로그를 볼 필요 없이, 에이전트가 바로 GitHub에 자동 PR을 생성하고, 필요에 따라 리뷰 또는 즉시 배포(Feature flag 활용)까지 진행합니다.
- 최종적으로 개발자는 '준비된 PR'만 검토하면 되는, 전체 트러블슈팅 과정의 자동화를 목표로 합니다.

### 파이프라인은 수집(ingest)→그루핑(group)→연구(research)→행동성(actionability)→실행(execution)의 단계로 구성됨

- 관찰 신호 수집 단계에서는 월간 수조 개(trillions)의 이벤트/신호를 다양한 타입(에러, 로그, 세션 등)에서 받아냅니다.
- 외부 공격자가 생성한 악의적 신호도 필터링해야 하므로, LLM 기반 분류기로 최상단에서 유해 신호 걸러냅니다.
- 이후, 신호 데이터의 구조(에러는 스택트레이스, 로그는 JSON 등)를 통합해 정규화하고, 각 신호에 중요도(Weight)와 임베딩 벡터를 부여함으로써 차후 군집화 작업을 수월하게 만듭니다.

### 단순 임베딩 군집화 방식은 다양한 신호 유형을 효과적으로 묶지 못해 LLM 기반 쿼리-임베딩 방식으로 개선함

- 최초 시도한, 오프더셸프 임베딩 모델 기반 '신호 간 임베딩 거리' 방식은 구조적 유사성 위주로만 군집화되어, 예를 들면 에러끼리, Slack 메시지끼리만 그룹화되는 문제가 발생했습니다.
- 이 방식은 '동일한 문제'를 다양한 신호(에러, 세션 리플레이, 고객 메시지 등)로 포착해야 하는 실제 요구에 부합하지 못했습니다.
- 개선책으로, 각 신호별 LLM에서 "이 신호가 무엇에 관한 것인지"를 묻고, 생성된 쿼리를 임베딩으로 변환해 관련성 있는 신호를 효과적으로 그룹화합니다.
- 신호 그룹 별로 Weight를 평가해 일정 임계값을 넘으면 'Report'로 생성하고, 연구 에이전트가 후속 작업을 이어받게 됩니다.

### 연구 에이전트는 Claude agent SDK를 활용해 문제 분석·우선순위 산정·담당자 추출을 자동화함

- 연구 에이전트는 Modal 기반 샌드박스 환경에서 실행되어, MCP 서버를 통해 필요한 추가 데이터(로그 등)를 실시간 확장적으로 수집할 수 있습니다.
- Linear, Notion 등 외부 MCP 연동으로 더 정확한 문제 추적과 문맥 파악이 가능해졌습니다.
- 연구 에이전트는 "문제 요약", "우선순위(priority)", "PR 리뷰어(예: Git blame 결과 기반)" 등 작업 결과를 산출합니다.

### ‘행동성(actionability)’ 판별 단계에서 작업 가능/불가능 상황을 분기함

- 보고된 문제(Report)가 '바로 행동 가능'하지 않은 경우, 추가 데이터 수집이 필요할 시 신호 풀로 재투입됩니다.
- 특정 이슈가 제품 운영자만이 판단 가능하다면, 별도 인박스로 전달해 수동 리뷰에 넘깁니다.
- 진단이 명확하고 신호 출처가 구체적(예: Sentry·에러 트래킹)일수록 에이전트가 쉽게 코드를 수정할 수 있습니다.
- Slack/세션 리플레이 기반 이슈의 경우 문제 원인이 모호하여 에이전트가 바로 PR을 만들기는 다소 어렵습니다.

### 실제 코드 수정(PR 생성) 단계는 샌드박스 기반 PR 자동 반복 실행으로 신뢰성 확보함

- 샌드박스에서는 해당 리포지터리를 클론한 뒤, 동일하게 Claude agent SDK를 활용해 코드를 수정합니다.
- PR 생성 후 CI(지속적 통합) 결과에 따라 자동 재실행 및 snapshot-snapshot 기반 상태 유지를 반복합니다.
- 리뷰어의 코멘트 등도 동적으로 반영하여, "green PR"(성공적으로 머지 가능한 PR) 상태까지 자동화합니다.
- 덕분에 개발자는 아침에 출근해 수많은 CI 실패나 코멘트 대응이 아니라, 바로 머지 가능한 PR 세트를 받아볼 수 있습니다.

### 파이프라인 개발 중 얻은 주요 교훈은 임베딩, 평가체계, 데이터 다양성, 비용최적화 등이 있음

- 실제 제품 데이터가 아닌 자체 샘플로만 실험하면 정확한 성능 평가(Eval)가 어려워, 다양한 고객 실제 데이터를 대상으로 해야 함을 깨달았습니다.
- 임베딩 모델은 구조적 유사성에만 치우칠 수 있으므로, 데이터 통일화 또는 LLM 기반 쿼리 전환이 중요합니다.
- 문제가 구체적이지 않으면, AI 에이전트가 무작위에 가까운(노이즈성) PR을 양산할 위험이 있어, 타당성 판단 단계가 필수적임을 확인했습니다.
- 비용(토큰 사용)에 너무 집착해 에이전트 활용을 미루면 학습 기회를 놓칠 수 있으며, 에이전트로 대량 실험 후 그 결과 유사성을 추출해 최적화(한방 LLM 호출 등)하는 것이 바람직함을 알게 되었습니다.

### 향후 목표는 에이전트가 반복 피드백을 학습하여 실제 '스스로 개선되는 제품'을 구현하는 것임

- 개발자는 재미없는 버그 수정·실험이 아니라 본질적인 기능 개발·혁신에 집중할 수 있게 하는 것이 근본 취지임을 강조합니다.
- 단순한 PR 자동화에 머무르지 않고, 배포 후 결과(feedback)를 에이전트가 학습하여 다음 이슈 대응 능력을 스스로 개선하는 방향이 차기 핵심 과제입니다.
- CI 실패, PR 거절, 실제 서비스 실패·성공 등 모든 결과를 피드백 삼아 파이프라인 품질을 지속적으로 높일 계획입니다.
- 실험("버튼 클릭")도 자동화하여, 간단한 변경은 에이전트가 feature flag 후 자동 배포 및 필요 시 롤백까지 전담하는 완전 자동화 수준을 지향합니다.

### 현재 파이프라인은 알파(Alpha) 단계로 점진적으로 확대될 예정임

- 이 파이프라인은 현재 알파 단계로, 앞으로 몇 개월에 걸쳐 차차 사용자가 확대될 예정임을 안내합니다.
- 개발자 커뮤니티에는 "폭넓은 데이터, 다양한 신호가 관찰되는 제품에서는 에이전트의 활용 가능성이 매우 높다"며 적극적인 실험을 권장합니다.
