## *Claude 코드와 OpenClaw의 창작자들은 더 이상 에이전트를 직접 프롬프트하지 않는가?* 핵심 요약

- AI 코딩 보조 도구(Claude Code, OpenClaw 등) 분야에서 ‘loop engineering(루프 엔지니어링)’이라는 새로운 패러다임이 등장하고 있음
- ‘루프 엔지니어링’이란 개발자가 직접 프롬프트를 작성하지 않고, 루프(반복 구조)를 짜서 에이전트가 자율적으로 작업을 수행하게 하는 방식을 의미
- Peter Steinberger(OpenClaw 제작자), Boris Cherney(Claude Code 리드)가 이 방식을 강조하며 실제 워크플로우에 적용하고 있음
- 루프 엔지니어링의 기본은 간단하며, Claude Code의 /loop, /goal, /routines 기능으로 구현 가능함
- 루프 구조를 통해 에이전트가 일정 주기로 외부 시스템(GitHub 등)을 확인하고 작업을 자동화함
- 그러나 이 방식은 토큰 사용량이 매우 많아 비용이 급증하며, 하나의 세션에서만 루프를 반복할 경우 LLM의 컨텍스트 오버플로우, 신뢰성 저하 등 실용상 문제가 있음
- 영상에서는 루프 엔지니어링의 단점(비용, 신뢰성, 확장성 등)을 극복하기 위한 실제 워크플로우 관리 방안(Arkon, 대시보드 시스템 등)을 제안함
- AI 에이전트 orchestration(조정)에서 인간의 개입(human-in-the-loop)과 분산 세션, 비용 최적화, 내구성 확보의 중요성을 강조
- 루프 엔지니어링은 일종의 도구적 개념으로, 견고한 환경(“harness engineering”)의 일부로 흡수되어야 실용적으로 운용 가능함을 시사

## 세부 요약 - 주제별 정리

### 루프 엔지니어링이 AI 코딩 분야의 새로운 화두로 부상함

- 최근 AI 코딩 에이전트 분야에서 ‘loop engineering(루프 엔지니어링)’이라는 용어가 주목받고 있음
- OpenClaw의 창작자 Peter Steinberger, Claude Code 리드 Boris Cherney 등이 트위터 및 팟캐스트에서 이 접근을 여러 차례 언급
- 루프 엔지니어링은 더 이상 개별 프롬프트에 집착하지 않고 루프 구조를 활용해 장시간 반복적·자율적으로 에이전트를 작동시키는 개념임
- 영상 제작자는 과도한 과장(Buzzword)이라고 평가하면서도, 일부 실제 활용 방법과 유용한 교훈이 있음을 인정
- 루프 엔지니어링이 ‘AI 코딩 도구의 차세대 패러다임’처럼 과대포장되는 분위기도 언급

### 루프 엔지니어링이란 무엇이고, Claude Code에서 어떻게 구현되는가

- Claude Code에는 /loop, /goal, /routines 등 루프 엔지니어링을 지원하는 다양한 기능이 마련되어 있음
- /loop : 일정 주기(예: 5분마다)로 GitHub 이슈 등 외부 입력을 체크, 발생한 이벤트를 자동으로 처리
- /goal : 완료 조건(criteria)을 명시해 에이전트가 목표 달성까지 반복적으로 작업 수행하게 만듦
- /routines : 팀 내 반복적인 작업(예: 매시간 스펙 문서 검토 등)을 스케줄링 하여 정기적으로 처리
- 이러한 기능들로 AI 에이전트가 대량의 작업을 직렬/병렬적으로 처리 가능함
- 네이티브로 “루프 스킬” 로드를 통한 자동 워크플로우 셋업 가능(예: plan.md의 할일을 순서대로 체크)
- 실제 워크플로우 진행 과정에서 에이전트가 각 단계별로 체크리스트를 처리하고, 작업 완료 후 종합 결과를 사용자에게 리포트
- Boris Cherney는 I don’t prompt Claude anymore, I write loops and the loops do the work”라며 루프 구조 중심 전략을 대표적 업무 방식으로 제시

### 루프 엔지니어링의 핵심 한계 1: 이상적인 결과를 보장하지는 않음

- 루프 엔지니어링이 AI 코딩의 ‘최적 솔루션’이라는 주장에 의문을 표함
- 예시로 Boris Cherney는 Daisy라는 AI가 수만 개의 에이전트를 동시에 관리한다고 주장하였으나, 실제로는 확장성·실용성에 한계가 있음
- 복잡한 시스템일수록 Claude Code 내 버그 및 신뢰성 문제 다수 발생
- 루프 설계는 증명·실험·아이디어 탐색 단계에서는 유용하나, 모든 실제 프로젝트에 일괄 적용하기에는 부적합함

### 루프 엔지니어링의 핵심 한계 2: 토큰 사용량 폭증에 따른 비용 문제

- 에이전트 orchestrator가 여러 worker 인스턴스를 병렬로 관리하고 반복 작업을 수행할 때 토큰 소비량이 극심하게 증가
- 영상의 예시 대시보드에서는 단순 애플리케이션 구동에도 100만 토큰 이상의 비용 소요(상세 토큰 사용 궤적도 공개)
- 초기 사양(spec) 입력부터, orchestrator의 분기(logic), worker 스핀오프, 결과 검증 등 각 단계마다 토큰 대량 소모 발생
- 문맥 전달, 결과 집계, 다음 라운드 준비 등 분산 시스템 특성상 어쩔 수 없이 고비용 구조를 초래함
- 단일 대형모델만 사용하는 경우 더욱 비용이 급증함

### 루프 엔지니어링의 핵심 한계 3: 세션 컨텍스트 오버플로우와 분산 처리의 부재

- Claude Code의 /loop 등으로 루프를 반복할 때, 모든 작업이 동일한 인스턴스/세션에서 이어짐
- 장기 루프 반복시 LLM 컨텍스트에 불필요한 정보가 누적·팽창하여 성능 저하, 논리 오류, 처리 불능 등 부작용 위험
- 세션 간 분산 및 역할 분리 없이 진행되는 구조가 확장성과 신뢰성을 저해함

### Arkon 및 대시보드를 활용해 루프 엔지니어링의 문제를 극복하는 실제 워크플로우 구성

- 영상 제작자는 자신의 툴인 “Arkon”을 활용해 대규모 AI 코딩 워크플로우를 체계적으로 관리(홍보성 언급 최소화)
- Arkon은 다수의 코딩 에이전트 세션을 오케스트레이션(조정)하는 ‘하니스 빌더’로, 대규모 작업을 효율적으로 분배·실행함
- 예시 워크플로우: 계획→구현→코드 리뷰/테스트 등으로 분할하여 각 단계마다 별도의 세션(모델) 활용
- 각 노드별(단계별)로 모델 선택 가능(예: 첫 분류 단계는 Kimmy K2.7 등 소형 모델, 구현은 Claude Code, 리뷰는 Codex 등 활용)
- 이렇게 단계를 모델별·세션별로 구분하면서 비용 최적화 및 분산 운영 실현
- 각 단계의 진행 중간에도 인간 개입(Human-in-the-loop) 가능: 실시간 확인, 검증, 워크플로우 일시 중지/재개 등의 유연한 통제 지원
- 워크트리(work tree) 및 독립 DB(Neon 기반 Postgres 등)와 연계해 세션별 결과와 상태를 안전하게 분리·보관—복구 및 재개 가능

### 다양한 AI 모델 및 워크플로우를 조합해 비용 절감과 신뢰성 확보를 실현함

- Arkon을 통해 워크플로우 각 단계에 맞춰 다양한 AI 모델(Claude, Codex, Kimmy K2.7, MiniMax M3 등)을 혼합 사용
- “단일 대형 모델 중심” Claude Code 루프와 달리, 각 역할에 맞는 모델 배분으로 비용 대폭 절감
- 각 단계가 별도 세션에서 실행되므로 컨텍스트 팽창 문제 최소화
- 모든 워크플로우 상태를 DB에 저장(Neon Postgres 활용)—장기 작업 중에도 내구성(Durability), 신뢰성 극대화

### 병렬 처리와 분리된 세션 운영 방식으로 대규모 자동화 작업을 안전하게 실행

- 워크트리(Work tree) 분리/격리를 통해 여러 에이전트가 동시에 병렬로 작업하더라도 충돌·상호 방해 없이 독립적으로 운영
- 예시로 하나의 orchestrator가 여러 GitHub 이슈 처리, PR 생성, 코드 리뷰까지 각 단계별 개별 워크플로우를 여러 세션에 파견
- 중간중간 진행상황(status update) 요청시, orchestrator로부터 전체 진행 현황 리포트 수령 가능
- 인간 개입이 필요한 단계에선 즉시 워크플로우가 멈추고 승인/수정 후 이어서 진행 가능

### 대시보드 및 관찰성을 통해 루프 엔지니어링의 실제 효과와 신뢰성을 개선함

- 독립된 대시보드를 구축해 모든 루프 실행 정보를 외부 DB에 기록 및 시각화
- 오케스트레이터는 DB에 저장된 상태를 기반으로 다음 작업 분배 및 로직 결정
- 예시: 간단한 칸반 보드 웹앱을 루프로 구현—초기 단계에서 오케스트레이터가 자동으로 작업 분할, 개별 워커 분배 및 토큰 사용량 관찰
- 대시보드 내에서 각 라운드별 진행상황, 토큰 사용, 이벤트 로그를 상세하게 확인(실시간 관찰성/투명성 제고)
- 인간의 승인 하에 다음 라운드로 진행 가능—단순 데모 수준을 넘어 신뢰성 있는 반복 자동화 촉진

### Arkon 및 대시보드 기반 루프 엔지니어링 환경을 클라우드로 손쉽게 배포할 수 있음

- Retool 등 플랫폼을 활용해 대시보드 앱을 클라우드로 즉시 배포, 동료들과 URL로 공유 가능
- React 기반 프론트엔드 생성→Retool에서 zip 파일 임포트→API 연동으로 백엔드와 통신→Neon(혹은 각종 Postgres) DB와 연결
- 사용자/권한 관리, 워크플로우 승인, 중지/재개 등 각종 Control Panel 기능 손쉽게 적용
- 실시간 코드 수정, 피드백, 팀 협업 환경 제공—현업 적용 및 생산성 극대화 지원

### 결론: 루프 엔지니어링은 안정적 하니스(“harness engineering”) 내에서 부분적으로 흡수·운용해야 실용성이 보장됨

- 루프 엔지니어링 방법론 자체는 흥미롭고 자동화 범위를 넓힐 수 있지만, 견고한 워크플로우 관리(harness engineering)와 병행되어야 함
- 세션 분리, 분산 처리, 비용 최적화, 인간 개입(휴먼 인더 루프), DB 내구성 등 추가 체계가 뒷받침되어야 실전형 시스템으로 운용 가능
- 루프 엔지니어링만으로는 고비용, 신뢰성 저하 등의 한계가 뚜렷하며, 제대로 된 하니스와 결합해야 효과적임을 강조
- 영상의 모든 데모 및 워크플로우 예제를 통해 실제 적용법과 주의사항, 장단점이 구체적으로 제시됨
