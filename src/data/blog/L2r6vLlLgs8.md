## *AI로 AI를 관리하다 — Laurence Jones, Incident* 핵심 요약

- Incident.io의 창립 엔지니어 Laurence Jones가 AI 기반 사고 대응 플랫폼의 복잡성을 AI로 관리하는 방법을 구체적으로 소개함
- Incident.io는 Netflix, Etsy, Skyscanner 등 대형 고객사에서 사용 중인 자동화된 사고 대응 솔루션을 제공하며, 목표는 프로덕션 조사와 문제 해결 자동화임
- 시스템 내부의 AI 복잡성이 인간이 직접 디버그하기 어려운 수준에 이르러, AI를 활용한 관리 및 분석 도구 개발이 불가피해짐
- 수백~수천 개의 텔레메트리 쿼리, 로그·메트릭·트레이스·과거 사고 데이터의 교차 참조 등 복잡한 AI 주도 조사 보고서가 자동 생성됨
- 인간이 이 보고서가 유용하고 정확한지 평가하는 데 1시간 이상 소요되어, 시스템 성능을 대규모로 진단하는 데 AI의 도움이 필요해짐
- Evals(평가)는 프롬프트 단위 테스트로 활용되며, YAML 파일에 실 사례 기반 평가 데이터와 평가 기준을 관리함
- 실사용 환경에서 발생한 대형 사례(실제 사고 데이터)를 eval로 손쉽게 추출·축적할 수 있는 기능과, eval 관리를 위한 전용 CLI 툴을 개발함
- 프롬프트 단위뿐 아니라 복잡한 다중 에이전트·툴 체계의 문제 진단을 위해, 모든 UI 트레이스·프롬프트 구조를 파일 시스템화하여 Claude Code 등 AI 에이전트가 활용 가능하도록 패키징함
- 대규모 고객 계정 및 사고 데이터 분석을 자동화하기 위해, Markdown 플로우 기반 반복적 분석 러너북과 병렬 에이전트 분석 파이프라인, 결과 보고 자동화를 통해 효율적 시스템 개선 루프를 구축함
- 핵심 전략은 내부 디버깅/분석 도구를 에이전트 친화적으로 만들어 AI 도움을 극대화하는 데 있음

## 세부 요약 - 주제별 정리

### Incident.io는 프로덕션 사고 대응을 전면 자동화하려는 목표를 추진함

- Incident.io는 사고 발생 시 자동으로 알리고, 사고 관리를 지원하는 AI 기반 사고 대응 서비스임
- Netflix, Etsy, Skyscanner 등 대형 고객사들이 실사용 중임을 구체적으로 언급
- 단순 사고 대응을 넘어 "프로덕션 조사 전면 자동화"를 목표로 삼고 있음
- 큰 사고부터 사소한 서비스 이슈까지 모두 자동적으로 원인분석 및 대응 조언을 제공하는 시스템 구축을 지향함
- 제품 도입 1.5~2년간 AI 시스템 구축 경험 기반의 실제적 복잡성 및 문제 사례 설명

### AI 기반 복잡성 증가는 인간의 직접 디버깅을 점점 더 어렵게 만듦

- 내부 시스템은 수백~수천 개 프롬프트와 다양한 AI 에이전트 구조로 구성됨
- 시스템이 생성하는 사고 조사 보고서는 수백 개의 로그, 메트릭, 트레이스, 과거 사고 데이터를 교차 분석함
- 결과적으로 "문제 진단 및 수정 제안"을 자동 산출하여, 사용자는 해당 정보를 그대로 활용 가능함
- 인간이 결과 보고서의 품질을 직접 평가하려면 보통 1시간 이상이 걸림(전체 사고 맥락을 따라가야 하므로)
- 다양한 고객사 환경의 다양한 사고를 대규모로 효율 진단하려면, 인간만으로는 역부족임을 실감
- 프롬프트, 에이전트, 툴이 중첩된 다계층 구성으로 인해, 문제의 정확한 발생 위치 및 원인 파악이 어려움

### Evals(프롬프트 평가)는 AI 제품 품질 확보를 위한 핵심 유닛 테스트로 자리잡음

- Eval(평가)은 "AI 프롬프트 유닛 테스트"로서, 특정 입력에 대해 원하는 출력을 산출하는지 자동 검증
- 프롬프트와 그에 대한 평가 기준(grading criteria, 예: 의미 유지, 스타일 변환 성공) 및 여러 실제 test case를 YAML 파일로 정의함
- 예시: "pirate speak" 변환 프롬프트의 경우, '해적 말투 여부'와 '의미 보존'을 평가 기준으로 명확하게 설정
- 모든 AI 작업(프롬프트 생성/수정 등)은 Go 언어 기반으로 incident.io 내에서 진행됨
- 신규 프롬프트 생성 또는 수정 시, 반드시 해당 프롬프트에 연결된 eval suite를 통과해야 merge 가능

### 실사용(프로덕션) 사고 데이터를 eval 사례로 즉시 추출이 가능하도록 자동화함

- 현실적인, 대형 실제 사고 사례는 매우 복잡하고, 이를 eval test suite에 반영하기가 어렵다는 문제 인식
- 한 사고의 전체 맥락(장문의 YAML 데이터)이 그대로 test case에 들어가게 되면 유지보수성 급격 저하
- 이를 위해 "프로덕션에서 AI 이상 행동이 감지되면, 즉시 eval 사례로 추출해 코드베이스에 더하는 버튼"을 자체적으로 개발
- Eval 단위 테스트의 품질을 높이는 한편, 크고 복잡한 사례가 test suite를 오염시키는 문제도 발생

### 대형 eval suite 유지·편집을 위한 CLI 기반 eval 툴을 자체 개발함

- eval suite가 방대해지면 코드 기반 에이전트가 전체를 인식·편집할 수 없게 됨(컨텍스트 크기 초과 등)
- 이를 해결하기 위해 'eval tool' CLI 유틸리티를 제작, 에이전트가 eval을 검색/편집/대체/추가하도록 설계
- 사람뿐만 아니라 "코딩 에이전트"가 프롬프트-평가 사이클에 직접 참여해, 실패 케이스 자동 생성→프롬프트 수정→재검증까지 자동화
- 예시: 코딩 에이전트가 프롬프트 개선→실패 eval 케이스 생성·통과 여부 확인→전체 suite 회귀 확인→프롬프트 간소화→최종 반영

### AI 기반 에이전트가 eval을 직접 활용하여 프롬프트 수정 및 품질 검증을 자동화함

- 실제로 Claude Code 등에서 eval tool을 불러와, 프롬프트-평가-수정 사이클을 자동으로 밟을 수 있도록 구현함
- 예시: 자연어 질의→로그 질의 프롬프트 개선 중, 코딩 에이전트가 eval 직접 추가·테스트·반복적 검증 실시
- 최종적으로 "패스율(pass rate)이 특정 기준(예시: 95%)을 넘는 시점"까지 반복 개선 가능
- 이 방식으로 "프롬프트 변경 전후, 다른 eval 케이스가 깨지는지 회귀 체크" 및 "프롬프트 불필요 복잡화 시 자체 간소화"를 일괄 처리

### 복잡한 AI 시스템 내 다수 프롬프트 및 에이전트 조합 구조가 문제 진단을 애매하게 만듦

- 대부분의 AI 시스템(챗봇·자동 조사 등)은 수십~수백 개 프롬프트와 에이전트, 툴 구성요소로 묶여 동작
- 예시: Incident.io의 챗봇 구조 트리를 도식화하면, 수많은 에이전트와 프롬프트가 계층적으로 얽힘
- 개별 프롬프트 문제를 발견해도, 전체 시스템 어디의 문제인지 확인/수정 포인트 도출이 어려움
- 다단계 트레이스/결과물은 각 단계에서 수백 개의 프롬프트 및 툴 콜을 포함, 단일 오류가 전체 결론 왜곡(예: 원인분석 오류)으로 확대 가능
- 기존에 사람이 살피는 UI 트레이스는 인간에게만 최적화되어, 에이전트(코딩 AI)가 활용하기 어려움

### UI 트레이스, 프롬프트, 툴 호출 내역 모두를 파일 시스템에 패키징하여 AI가 탐색·분석 가능하게 함

- Anthropic의 Claude Code 등 최신 코딩 에이전트가 파일 시스템 기반 탐색에 특화되어 있는 점을 참고
- 각 AI 시스템별 전체 트레이스/프롬프트/결과 데이터와 코드베이스를 ASCII 텍스트 파일 패키지로 변환·다운로드
- Claude Code 등 에이전트가 이 파일 시스템에 접근, 전체 트레이스를 self-documenting한 형태로 분석/수정 제안 가능
- 텍스트 기반 패키지는 데이터 크기 등 제약이 적어, 방대한 인터랙션 히스토리·트레이스까지 일괄 활용 가능
- 예시: 문제 발생 인터랙션을 패키징→Claude Code에서 문제 원인, 수정 대상 지점 등 분석 자동화 가능
- UI 중심 → 파일 시스템 중심 패키징 전환으로, 에이전트 친화적 내부 디버깅 체계 확립

### 대규모 고객 데이터 분석을 위해 병렬 에이전트와 반복 분석 파이프라인을 구축함

- 매일 수백~수천 건의 조사(인베스티게이션)를 여러 고객사 계정에서 자동 수행
- 백테스트(backtest)로 배치 실행→정확도 숫자(예: RCA 86%)만으로는 시스템 개선 근거가 빈약함
- 모든 조사 데이터를 한꺼번에 파일 시스템에 다운→코딩 에이전트가 Markdown 러너북 기반 분석 파이프라인 가동
- "scrapbook" 레포지토리 내 러너북: 분석 플로우 및 절차를 마크다운으로 구조화하여 반복적이고 일관된 결과 도출 가능
- 병렬(예: 25개) 에이전트를 띄워 각 데이터·사건별로 개별 분석/클러스터링→동일 유형의 실패/오류 집단 파악
- 분석 중간 결과와 전체 결과를 파일 시스템에 저장, 분석 진행상황을 언제든지 이어가기 쉬움
- 분석 결과를 코드베이스와 연동, 문제 지점을 코드에서 자동 지목 및 개선 제안까지 이어짐

### AI가 분석·개선한 내용을 PR(코드 변경 요청) 단계까지 자율적으로 연계함

- 백테스트·분석 파이프라인 결함 검출→코드베이스 직접 수정→eval로 변경 효과 즉시 검증
- PR 생성→운영 배포→프로덕션 실측 테스트까지 전 과정이 AI 에이전트를 중심으로 실행됨
- 반복적인 일관성·재현성 높은 시스템 개선 사이클이 구축됨

### 내부 분석·디버깅 도구의 에이전트 친화성 확보가 미래 AI 시스템 운용의 핵심임을 강조함

- 복잡한 AI 시스템에서 "AI로 AI를 관리·디버그"하는 것이 불가피해지는 현상 설명
- 내부 디버깅, 분석, 유지보수 도구 역시 AI 코딩 에이전트와의 궁합 최적화가 중요
- 파일 시스템 기반 정보 패키징이 매우 효과적이며, 컨텍스트 크기·분석 효율 모두 탁월함
- 반복적 분석/운영엔 AI 러너북(마크다운 문서 기반)이 개발 생산성·정확도에 큰 기여
- 궁극적으로 반복 가능한 개선 루프를 자동화함으로써 운영 리소스 절감과 품질 고도화 동시 달성

### 최종 채용 안내 및 발표 마무리

- Incident.io의 런던 본사에서 신규 엔지니어 채용 중임을 알리며, AI SRE 제품에 관심 있는 이들의 지원을 독려
- 세션을 마치며 질의응답 및 감사 인사로 발표 종료
