## *한 명의 개발자, 두 다스의 에이전트, 0의 정렬: 협업적 AI 엔지니어링의 필요성* 핵심 요약

- 본 강연의 제목은 "한 명의 개발자, 두 다스의 에이전트, 0의 정렬"이며, 소프트웨어 개발 현장에서 협업적 AI 엔지니어링이 필요한 이유를 다룸
- 최근 AI 코딩 에이전트가 개인 생산성을 폭발적으로 향상시키는 것처럼 보이지만, 팀 단위 정렬(alignment)이 오히려 더 중요한 병목으로 부상
- 코딩 구현 자체는 저렴하고 빠르며 고품질로 해결되고 있지만, "무엇을 만드는 것이 옳은가"에 대한 팀 내 합의가 어려워짐
- 기존 협업 도구(예: GitHub, Slack, Jira, Linear)는 개별적 코딩 에이전트 환경에 맞지 않아, 대량의 에이전트 산출물을 비효율적으로 다룸
- 구현 단계가 너무 빨라지면서, Plan-Implement-Review 구도의 ‘정렬 접점’이 사라지고, 피드백은 늦게 도착하여 낭비와 충돌이 빈번
- 비효율적 정렬로 인해 불필요한 기능 개발, 중복 작업, 충돌(PR merge conflict), 맥락 없는 코드 리뷰가 증가
- 현장에서는 계획, 맥락, 의사결정, 개발이 실시간 순환적으로 이뤄지는 새로운 도구가 필요하며, 이를 위해 GitHub Next 팀이 ‘ACE’(Agent Collaboration Environment)를 개발 중
- ACE는 실시간 멀티플레이형 세션, 에이전트·팀원 동시작업, 컨텍스트 공유, 실시간 코드 실행 및 계획 협업 등 기존 도구와 본질적으로 다른 협업 패러다임을 제시
- 에이전트와 팀원 모두가 입력한 히스토리·대화까지 공유함으로써, 모든 맥락이 살아있는 공간에서 소프트웨어 개발이 이루어짐
- 빠르고 싼 소프트웨어 환경에서 "품질"이 핵심 차별 요소임을 강조하며, 에이전트는 생각 없는 무의미한 생산이 아닌, 깊이 있는 집단적 사고와 정렬을 가능하게 해야 한다고 주장

## 세부 요약 - 주제별 정리

### 대량의 코딩 에이전트가 개발자 '개인' 생산성만 늘려도 협업적 병목은 해소되지 않음

- 현재 AI 코딩 에이전트들은 개발자 한 명이 여러 에이전트를 병렬로 돌려 "한 명이 여러 명 몫"의 소프트웨어를 만들 수 있다는 미래상을 강조
- 하지만 이 접근은 소프트웨어가 본질적으로 팀의 스포츠(Team Sport)라는 점, 즉 '무엇을 왜 만드는가'에 팀 단위 합의가 필요하다는 현실을 간과
- "9명의 여성이 한 달 만에 아기를 낳을 수 없다"는 비유처럼, 개인 생산성 부스팅만으로는 의사소통·조율의 본질적 문제를 해결하지 못함
- 에이전트가 실제 업무를 자동화·가속화하더라도, 커뮤니케이션 병목과 올바른 목표선정 없이 성과는 제한적

### 구현은 쉬워졌지만, '무엇을 만들지'에 대한 팀 내 합의가 결정적 병목으로 부상함

- 코딩 구현은 점점 더 빨라지고, 비용이 저렴해지고, 품질까지 향상되는 추세임(Implementation is a rapidly solved problem)
- 현시점에서 "어떻게 만들지"가 아니라 "무엇을, 왜 만들지"가 진짜로 어려운 문제로 떠오름
- 제작비가 싸질수록, 기회비용(opportunity cost)이 실제 비용이 되고, 무엇에 역량을 투입할지 신중한 팀 합의가 필수
- 팀 내 모든 구성원이 ‘올바른 것’을 만드는지, 에너지 배분이 적절한지, 임팩트가 최대화되는지 고민해야 함

### 기존 협업 도구(깃허브, 슬랙, 지라 등)는 대규모 에이전트 개발에 적합하지 않음

- 현재 업계 표준 툴(GitHub, Slack, Jira, Linear 등)은 ‘에이전트 기반 개발’ 시대를 염두에 두고 설계되지 않음
- 이들 툴은 예전 ‘느린’ 소프트웨어 개발 프로세스에 최적화되어 있음
- 대량의 에이전트 산출물을 이질적 구조로 흘려보내야 하며, 대부분의 개발자조차 PR(풀리퀘)과 이슈(issues)가 미래 동적 개발의 해답이라고 생각하지 않음
- GitHub 내 Next 팀 역시, 차세대 개발 과정을 위한 새로운 구조와 플랫폼이 필요하다고 내부적으로 인식

### Plan-Build-Review의 접점이 사라지며 정렬의 기회마저 급격히 줄어듦

- 전통적 개발 프로세스에선 '계획-구현-리뷰'의 각 단계마다 팀원간 의견 교환, 맥락 공유, 피드백, 오류 예방 등 정렬(alignment)이 자연스럽게 일어남
- 코딩 자동화·에이전트 적용 이후엔, 이 초기 접점들이 사라지고, 대부분의 리뷰/피드백이 "너무 늦게" 이루어짐
- 이슈 등록 후 에이전트가 PR을 올릴 때까지 몇 분밖에 걸리지 않아, 상세 계획 논의나 중간점검이 증발
- 에이전트의 ‘local plan’ 모드는 아예 팀과 공유되지 않은 채 독단적으로 진행되는 경우가 많아, 더욱 정렬의 기회가 취약
- 결과적으로, 모든 책임·정렬이 마지막 PR 단계에 집중되어 있지만, PR 자체는 원래 이런 용도로 설계된 것이 아니라 비효율 심화

### 정렬 부재로 인한 조직·협업 고질 문제들이 심각하게 확대됨

- 정렬 없는 빠른 개발은 ‘쓸모없는 기능(Feature nobody asked for)’, ‘진짜 문제 미해결’, ‘완성 후 폐기’, ‘심각한 머지 컨플릭트’ 등 각종 낭비를 유발
- 에이전트/개발자가 동일한 파일을 중첩적으로 다루거나, 우연히 동일 과제를 병행하게 되면 충돌·중복작업이 빈번
- 결국 PR이 쌓이고 맥락을 파악 못한 팀원은 리뷰조차 어려움

### 효과적인 에이전트 기반 협업 환경엔 실시간 정렬과 맥락 공유를 결합한 새로운 도구가 필요함

- 앞으로의 툴은 ‘계획’, ‘맥락 수집’, ‘의사결정’, ‘개발’이 하나의 도구·프로세스 내에서 실시간 융합적으로 이루어져야 함
- 그 이유는 정렬의 많은 맥락, 즉 비즈니스·재무·의사결정·담당자·제품 비전·유저 인사이트·프로젝트 이력 등 대부분이 코드가 아닌 ‘사람의 머릿속’에 있기 때문
- 이런 맥락은 에이전트가 스스로 탐색할 수 없는 정보이기 때문에, 자연스럽고 간소화된 방식으로 사람이 초기에 공유할 수 있는 환경 필요

### GitHub Next 팀은 ACE(Agent Collaboration Environment)라는 실험적 협업 환경을 개발함

- ACE는 팀원과 코딩 에이전트가 동일 세션에서 실시간 협업할 수 있도록 설계, 기존 툴 파편화를 최소화
- 세션 단위로 업무를 진행하며, 각 세션은 독립된 마이크로 VM(클라우드 샌드박스 컴퓨터, 별도 깃 브랜치)에 의해 실행됨
- 각 세션에서 팀원과 에이전트가 모두 채팅, 프롬프트, 터미널 명령·코드 실행, 코드 프리뷰, 변경사항(diff) 확인을 실시간으로 공유

### ACE의 세션 구조는 실시간 협업·컨텍스트 공유·병렬 개발을 융합함

- 슬랙 채널/깃허브/코파일럿 클라우드 환경의 특징을 합친 형태로 설계
- 세션은 Slack처럼 조직적이고, 각 세션에서 팀원들은 상호 실시간 대화와 피드백, 그리고 코딩 작업(터미널, 코드 실행, 프리뷰 등)을 동시에 진행
- 변경사항은 자동 커밋, 히스토리와 리뷰 지원으로 자연스러운 추적 및 논의가 가능
- 팀원 간 코딩 세션 간 이동이 자유로워, 작업 내용·에이전트 프롬프트 히스토리까지 맥락을 한 눈에 파악

### ACE는 에이전트와 팀원 모두가 ‘활동 맥락’에서 즉시 논의·협업·지시하는 멀티플레이 환경임

- 예시 데모로 Hacker News 앱의 테마 컬러 추가와 변경, 팀원(Nate 등)의 실시간 코멘트/대안 제안/에이전트 직접 프롬프트 등 시연
- 팀원과 에이전트가 동등하게 세션 내 기능 확장·수정 요청, 대화 기반 명령(“@ace, do it”) 등을 할 수 있음
- 디자이너, PM, CS 등 다양한 역할이 실시간으로 기능 진화 과정을 관찰·참여·지시할 수 있음
- 결국 PR 생성, VS Code 실시간 공동 편집 등 표준 개발 워크플로우와 완벽히 연동

### ACE의 대시보드는 개인과 팀의 생산맥락, 진행상황, 협업 히스토리를 집약 제공함

- 개인의 진행 중 작업, 미완성 PR, 최근 완료 이슈/PR 정보 및 팀원별 주요 작업 현황(예: 팀 pulse 영역에서 Nate의 lobby 채널 출시, David의 토큰 이슈 fix 등)을 요약해 제공
- 에이전트가 소셜·업무 맥락을 파악해 아침마다 재정비를 돕거나, 결정사항 안내, 기존 작성자 호출 등 ‘팀 정렬’ 지원
- 고속·대량 개발 환경에서 협업 맥락 파악이 힘든 문제를 해결할 수 있도록 ‘정보 패브릭’을 제공하는 것이 목표

### 빠르고 싼 코드 시대, 품질과 장인정신이 팀을 차별화하며 에이전트는 깊은 집단적 사고를 지원해야 함

- 과거엔 “시간 부족” 때문에 설계/리서치/아키텍처 숙고를 희생하며 만족스럽지 못한 소프트웨어 출시가 빈번
- 에이전트 혁신으로 ‘구현’에 소비되는 시간과 에너지가 감소, 그만큼 더 깊은 조사·고민·실험적 탐색에 자원을 투자할 기회가 생김
- 무의미한 대량 생산이 아닌, 소수의 뛰어난 결과물을 더 잘 만들어내는 방향을 지향해야 함
- 품질이 새로운 차별점이 되므로, 도구(ACE 등)는 팀이 “깊이 생각하는 작업 환경”을 조성해줘야 하며, 강력한 팀 정렬을 지원하는 것이 핵심

### ACE는 개발·에이전트·팀원 간 자연스러운 연결을 통해 미래 개발 패러다임을 실증적으로 탐색 중임

- ACE는 현재 내부 테스트 및 수천 명 대상 기술 프리뷰 준비, ace-session 링크·연동 등으로 외부 시스템과도 호환
- 개별적인 단말기 기반 솔루션이 아닌, 항상 연결된 클라우드·멀티플레이 환경으로 팀원을 완전히 연결
- 향후 모바일 등 다양한 디바이스에서도 실시간 개발 및 협업이 가능하도록 설계 중
- 구조적으로 코드, 맥락, 의사결정, 계획, 피드백 등 개발 전 주기를 하나의 라이빙 환경에서 통합
