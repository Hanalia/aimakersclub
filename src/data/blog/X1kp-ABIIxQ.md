## *여러분의 에이전트 아키텍처는 6개월의 반감기를 갖고 있습니다 — Dan Farrelly, CTO, Inngest* 핵심 요약

- Dan Farrelly(Inngest CTO)는 AI 에이전트 아키텍처가 불과 6개월마다 대대적인 개편 또는 수정이 불가피할 만큼 빠르게 변화한다는 현실을 짚으며, 의도적이고 계층화된 설계의 필요성을 강조함
- 기존에 작성한 에이전트 코드의 상당 부분이 곧바로 구식이 되기 쉬우며, 우연히 남아있는 코드와 의식적으로 설계된 부분을 구별해야 함
- AI 에이전트 아키텍처를 절차(Execution), 컨텍스트(Context), 컴퓨트(Compute) 세 가지 계층으로 분리하는 것이 중요하다고 주장
- 각 계층의 반감기는 서로 다르며, 프롬프트는 수주, 모델은 수개월, 실행(Execution)은 수년간 지속 가능하게 설계할 수 있음
- 많은 팀들이 계층 분리 없이 모든 요소를 결합해 기술 부채를 키우며, 이는 변화에 약하고 관리가 어려운 시스템을 야기함
- 안정적이고 장기적으로 유지 가능한 아키텍처의 핵심은 "Execution Layer"에 투자하고 올바른 추상화와 내구성을 확보하는 것임
- 실행 계층은 에이전트 전체의 수명 주기, 오류 복구(Resumability), 다양한 실행 패턴 지원, 관측 가능성(Observability)를 담당해야 함
- 샌드박스는 코드 실행·파일 조작 등 유연성을 제공하나, 내구성 보장이나 스테이트 저장 등에는 적합하지 않으므로 실행 계층에서 이를 관리해야 함
- 현대적 에이전트 아키텍처는 백그라운드 에이전트, 반복(loop) 시스템 등 장기·비동기 운영이 필수인 만큼, 적절한 실행 계층 없이는 운용·관측·디버깅이 힘듦
- Inngest는 이 실행 계층에 특화된 시스템으로, 다양한 컨텍스트·모델·샌드박스와 연동되며, 내구적 스텝, 추적, 점수화·이벤트 연결 등 실행 계층의 주요 기능을 지원함

## 세부 요약 - 주제별 정리

### 6개월마다 에이전트 아키텍처를 다시 써야 하는 현실이 도래함

- Dan Farrelly는 Inngest CTO이자 공동 창업자로, AI 인프라와 에이전트 자체를 실제로 개발하면서 겪은 빠른 변화 속도를 강조함
- 지난 6개월 이전에 릴리즈한 에이전트 코드가 지금도 동일하게 동작하는지를 질문하며, 코드를 의도적으로 설계했는지, 아니면 우연히 살아남은 것인지 파악해야 함
- 새로운 언어모델(LMM), 프레임워크, 도구 호출 표준, 패턴 등이 빠른 속도로 등장하면서 기존 아키텍처가 곧바로 맞지 않게 됨
- 이는 불평이 아닌 현실이며, 개발자들은 이러한 변화의 빠르기를 인식하고 수용해야 함

### 에이전트 아키텍처는 실행, 컨텍스트, 컴퓨트 3계층으로 사고해야 함

- 대부분의 아키텍처 논의는 구체적 컴포넌트, 플로우, 구성요소에 집중하지만, 실제로는 다음 3가지 계층적 관점이 중요함
    - Execution(실행): 플로우, 상태, 내구성, 재시도 등, 시스템의 '두뇌' 역할
    - Context(컨텍스트): 모델, 프롬프트, 도구, 메모리 등, 지식 계층이며 가장 빠르게 변함
    - Compute(컴퓨트): 샌드박스, 런타임, 브라우저 등, 실제 연산이 이뤄지는 '손'의 역할
- 이 계층화된 구조를 기반으로 아키텍처를 설계해야 장기적인 유지보수와 변화 대응이 쉬워짐

### 각 계층의 반감기는 다르며, 이것이 결합될 때 기술 부채가 커짐

- 과학에서 '반감기'가 물질이 절반으로 감소하는 시간인 것처럼, 각 계층도 소모되는 속도가 다름
    - 프롬프트: 수일 또는 일주일 정도 유지
    - 모델: 운이 좋아야 수개월
    - Execution(실행): 올바르면 수년간 유지 가능
- 실제로는 모든 계층이 강하게 결합되고, 한 계층의 반감기가 다른 계층까지 영향을 주어 변화가 일어날 때마다 전체 시스템을 커버해야 함
- 이로 인해 추적과 유지보수가 힘들어지고, 기술 부채가 쌓임

### 레이어를 분리하는 아키텍처적 사고가 장기 생존성을 보장함

- 프레임워크나 하네스(harness)를 써도, 추상화 수준이 너무 높거나 혹은 계층이 분리되지 않은 경우가 많음
- 오케스트레이션이 체인이나 프레임워크 내부에 묻혀 실제 동작을 파악하기 어려움
- 상태(state)를 샌드박스나 프롬프트 논리와 엮으면, 어떤 요소를 바꾸려고 해도 전체를 다시 써야만 함
- 변화는 불가피하다는 점을 인정하고, 계층 중 장수할 수 있는 실행 계층(execution layer)에 집중하는 것이 최적임

### 실행 계층이란 시스템 운영의 신뢰성과 관리성을 책임지는 뇌에 해당함

- 실행 계층은 코드 실행의 신뢰성, 작업 플로우 관리, 언제 어떻게 각 작업을 완료할지 제어하는 시스템임
- 이 계층이 제대로 설계된다면, 모델·컨텍스트·샌드박스를 자유롭게 교체해도 전체 틀을 바꿀 필요가 없음
- 플래닝, 모델 호출, 코드 실행, 서브에이전트 호출, 반복, 재시도, 조정 등 에이전트 수명주기를 총괄해야 함
- 뛰어난 실행 계층은 변화에 대한 내성이 강하고, 전체 구조의 견고함을 높임

### 실행 계층의 주요 요건 ①: 장애 복구(resumability)와 외부 상태 관리

- 장기 실행 에이전트는 언제든 LLM 호출, 도구 호출 등에서 장애가 발생할 수 있음
- 실패 시, 작업을 처음부터 다시 시작하지 않고 실패 지점에서 재시작(resume)하는 것이 반드시 필요함
- 이를 위해 3시간 이상 실행되는 작업의 상태는 메모리나 디스크가 아닌, 외부·내구적 스토리지에 보관해야 함
- 수동 체크포인팅이나 로그 기반 복구 등은 추상화 누수로 이어져 다른 계층에 부담을 줌

### 실행 계층의 주요 요건 ②: 다양한 호출/실행 패턴(크론, 이벤트, 비동기 등) 지원 필수

- 크론(예약 실행), API 트리거, 이벤트, 인간 개입(human-in-the-loop), 서브에이전트/동적 워크플로우 등 다양한 실행 패턴을 지원해야 함
- 동기/비동기, 지연 실행(delayed invocation) 등 실행의 유연성이 전체 시스템을 좌우함
- 이런 패턴 지원이 없으면 큐, 워커, 폴링, 백오프, 스케줄링 등이 하네스 로직에 난삽하게 섞여 복잡해짐

### 실행 계층의 주요 요건 ③: 전 과정에 대한 관측 가능성(Observability)이 확보되어야 함

- 단순히 LLM 호출·도구 호출 결과 뿐 아니라, 데이터베이스 오류, 권한 이슈, 트리거·성능 등 전체 세션의 흐름을 추적할 수 있어야 함
- 전체 트레이스(Trace)를 확보하지 못하면 원인 파악이나 개선이 매우 어려우며, 반복적인 디버깅도 어려움

### 샌드박스는 필수적이나, 상태나 내구성 책임은 오히려 위험함

- 샌드박스는 코드 실행, 브라우저 조작, 디스크 작업 등에 필수적이나, 본질상 무상태(ephemeral), 일시적임
- 상태나 스냅샷, 내구성 관리 등을 샌드박스에 맡기는 것은 잘못된 설계(anti-pattern)임
- 상태와 내구성, 문맥 부여는 반드시 실행 계층에서 책임져야 함
- 샌드박스는 손(Hand), 실행 계층은 두뇌(Brain)로 명확히 역할 분리해야 함

### 장기·비동기·복합 에이전트 트렌드는 더욱 강력한 실행 계층을 요청함

- 최근 부상하는 아키텍처(예: 백그라운드 에이전트, 동적 워크플로우, 자동 반복 루프, 에이전트 팩토리 등)는 모두 비동기적, 장기 실행, 위임 등 복합적 특성을 가짐
- 이런 환경에서는 단 한 번의 장애도 중대한 문제를 유발할 수 있고, 관측·추적·디버깅이 핵심 과제가 됨
- 각 패턴을 조합·혼합해서 유연하게 활용하려면 강력한 실행 계층이 필수임

### 실행 계층 중심의 코드 예시: 루프·중재·리뷰어 함수로 구성

- 크론(cron) 기반 건강진단 루프: 30분마다 주요 시스템 지표를 확보 후, LLM에게 전달해 시스템 건강 여부 및 추가 조치 판단
- 건강 이상 시, 중재(Triage) 에이전트 트리거: 상세 지표 취합, LLM 조사, 반복 문제 탐색, 코드 복제 및 커미트 분석 등 복합 작업 수행
- 리뷰어(Reviewer) 함수: 주기적으로 과거 기록 평가, 프롬프트 및 지표 개선, 시스템의 실제 성능 판단 및 개선사항 제안
- 각 함수가 실행 계층에서 연결·감시되어야 하며, 하위 에이전트 호출, 히스토리 로그, 서브 워크플로우 등 전체 흐름 관리가 필수

### 실행 계층이 사용자 행동, 결과, 세션 등 전반의 허브이자 평가지점(스코어링)이 됨

- 실행 계층은 사용자 입력, 실행 결과, 세션 전체의 행동 데이터가 모두 흐르는 중심(hub) 역할을 함
- 이 계층에서 모든 데이터를 집계·관찰·스코어링(평가)할 수 있으므로, 실질적인 어플리케이션 개선 순환의 핵심 위치가 됨
- 각 세션 또는 단계마다 자동 점수화·이벤트 첨부가 가능해, 함수적 개선이나 outcome 기반 성능 평가 등에 용이
- 예: 트라이에이지 성공시 PR 오픈, 리서치 결과의 저장 및 적합성 평가 등 실제 실행 결과를 기준으로 한 구체적 평가 가능

### Inngest가 제공하는 실행 계층 인프라 및 기능 소개

- Inngest는 AI 에이전트의 내구적 실행 계층(durable execution layer) 제공을 목표로 하는 플랫폼임
- 다양한 컨텍스트 계층(모델, 프레임워크, 도구), 다양한 컴퓨트 계층(샌드박스, 브라우저)과 쉽게 연동 가능
- 주요 기능: 내구적 실행 스텝, 이벤트 트리거, 스케줄링, 에이전트 간 조정, 전체 세션 트레이스, 인프라 자동 관리, 실행 후 자동 스코어링 등
- 각종 데이터와 이벤트를 한 세션에 연결해 outcome 기반 평가 및 장기적 시스템 개선에 적합

### 변화의 속도를 수용하되, 실행 계층 추상화에 집중하는 것이 향후 경쟁력의 핵심임

- 6개월, 3개월 단위로 끊임없이 바뀌는 AI 에이전트 환경에서 오직 실행 계층만이 장기적 내구성을 가질 수 있음
- 설계의 시작점으로 실행 계층부터 추상화하고, 주변(컨텍스트, 컴퓨트)은 빠른 진화에 맞춰 교체·적응하도록 구성해야 함
- 정교한 실행 계층 위에서만, 빠른 변화에도 흔들리지 않는 AI 에이전트 시스템을 구축할 수 있음
