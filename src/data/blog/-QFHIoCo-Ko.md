## *계획부터 프로덕션까지 AI 코딩 워크플로우 전 과정 실전 가이드* 핵심 요약

- 본 영상은 Matt Pocock이 AI를 활용한 소프트웨어 개발의 전 과정을 실제 사례(교육관리 플랫폼)로 시연하며, LLM(대형언어모델) 기반 코딩의 한계와 전략적 활용법을 상세히 설명함
- LLM에는 ‘스마트존’(최적의 추론능력, 컨텍스트가 적을 때)과 ‘덤존’(컨텍스트가 많아질수록 성능 저하)이라는 독특한 제약이 존재하며, 이에 따라 작업은 작고 독립적인 단위로 구분해야 효율적임을 강조
- 초기 기획은 ‘그릴미 스킬’(Grill Me Skill)을 사용해 AI로부터 집요한 질문 세례를 받아 클라이언트 의도와 요구 사항(예: 게이미피케이션 기능)을 명확히 정렬하는 방식으로 진행됨
- 대규모 요구사항은 PRD(제품 요구사항 문서)로 체계화되어 명확한 ‘도착지’를 정의하고 이후 단계의 토대가 됨
- PRD는 칸반(Kanban) 보드 방식으로 세분화되며, AI로 실행 가능한 독립적 이슈(가로가 아닌 세로 슬라이스, 즉 트레이서 불릿/버티컬 슬라이스)로 분할·정렬함으로써 병렬 개발 및 신속한 피드백을 가능하게 함
- 구현 단계부터 인간은 루프에서 벗어나고, AI가 AFK(away from keyboard, 무관여) 방식으로 계획된 이슈를 처리하되, 이때 TDD(테스트주도개발)를 통한 피드백 루프를 중시
- AI의 코드 구현 후 자동 코드 리뷰, QA(품질검증), 코드 리뷰 등에서 인간의 리뷰가 여전히 중요하며, 특히 코드베이스의 구조(딥 모듈 vs. 얕은 모듈)가 품질에 결정적 영향을 미침
- 지속적으로 코드베이스를 개선(모듈 깊이화 등)/AI의 역할과 한계를 명확히 하며, 결국 효율적 병렬화 및 자동화 기반 팀 개발 체계를 지향함
- 세션 전체가 다수의 실질적 예시, 스크립트·도구(Sand Castle 등), 실제 실습 리포, 명확한 설계 및 QA 전략 등 구체적 스텝과 실천 팁으로 구성됨
- AI 기반 개발 워크플로우 전체의 체계적 자동화·병렬화와 동시에, 인간 개발자의 코드 및 방향성 통제가 여전히 핵심임을 반복적으로 강조

## 세부 요약 - 주제별 정리

### AI 코딩의 새로운 패러다임은 소프트웨어 엔지니어링의 기본 원칙과 긴밀히 연결됨

- AI 시대가 완전히 새로운 패러다임처럼 보이지만, 실제로는 기존 소프트웨어 엔지니어링의 핵심 원칙(분할정복, 작은 단위 업무, 구체적 요구 명확화 등)이 여전히 중요함
- AI가 바꿔놓는 것은 ‘자동화/병렬화의 수준’이지만, 인간-기계 협업의 기본 로직은 변하지 않음을 언급
- 영상 초반 직접 질문: "AI로 코딩해본 적 있는가, 매일 사용하는가, AI 경험에서 좌절한 적 있는가" 등으로 참여도 유도 및 문제의식 공유

### LLM의 '스마트존'과 '덤존' 개념은 AI 효율 극대화의 핵심 전략적 지점임

- Dex Hy(휴먼레이어 대표) 제안: LLM은 대화 초기, 컨텍스트 토큰이 적을 때(=스마트존)에 가장 지능적으로 작동
- 컨텍스트 토큰이 추가될수록(축구 리그 참가 팀이 늘어날 때 경기 수가 급증하듯) 추론 효율이 급격히 떨어져 덤존에 진입
- 실제 경험상 100K 토큰 이후부터 LLM은 의미없는 결정/오판을 내리기 시작
- 따라서, 작업 단위를 "스마트존"에 맞게 작고 분리된 청크로 쪼개야 하며, AI가 한번에 처리할 범위를 넘지 않도록 설계해야 함

### '컨텍스트 리셋'과 '컴팩팅' 등 세션 흐름 관리가 작업 품질에 지대한 영향을 미침

- LLM은 영화 ‘메멘토’ 주인공처럼 금세 기억을 잃으므로, 세션 내 컨텍스트와 그 크기 관리가 매우 중요
- 시스템 프롬프트 → 탐색(코드베이스 엑스플로러) → 구현 → 테스트/피드백 루프가 반복되는 전형적 구조
- 컨텍스트를 ‘컴팩트’(요약 이력으로 압축)하거나 완전히 클리어할 수 있는데, 연사자는 ‘완전 리셋’ 방식 선호
- 각 작업 단계의 토큰 사용량, 컨텍스트 크기 실시간 체크(Pocock 본인 웹사이트 AI Hero의 유틸 활용 등)가 필수라 강조

### 그릴미 스킬(tool)은 AI-인간간 설계 개념 공유를 위한 강력한 도구임

- 초기 ‘사양→코드’식 작업(specs-to-code)은 코드 품질/정렬에 한계가 있으므로, 실질적 ‘공유 설계개념(Design Concept)’의 형성이 핵심임을 설명
- Grill Me Skill: “계획 모든 면을 집요하게 인터뷰하고, 설계 트리의 각 브랜치를 따라가며 종속성/전제를 명확히 할 때까지 질문한다”
- 실제 사용 예: Slack 클라이언트 요청 메시지(‘게이미피케이션 추가’)를 투입, AI가 포인트 시스템/레벨/레트로액티브/보상 등 세부절차를 집요하게 질문→연속 커뮤니케이션을 통해 완전 정렬
- 최소 수~수십, 최대 100여개에 달하는 질문과 권장답변으로 의도를 완전히 해부·구체화
- 이에 따라 개발자, PO(제품책임자), 도메인 전문가 등, 실무자 모두 공유 이해 도달 가능

### 섭에이전트(sub-agent) 및 대화 세션/프롬프트 설계의 실질적 활용방식

- 서브에이전트는 메인 AI로부터 작업을 위임받아 별도 컨텍스트에서 세부탐색·요약을 실행/결과만 전달, 주요 요건·설계에 ‘병렬 탐색+정제’ 적용 가능
- 각종 프롬프트/스킬 구성(→예: grill me, write a PRD, break a PRD into issues 등)은 리포 내 예시 코드/문서에 포함, 누구나 복습/확장 가능함

### 설계 개념을 바탕으로 목표를 명확히 하는 PRD(제품 요구사항 문서) 작성법

- 그릴미 세션 결과는 ‘도달점’ 문서(=PRD, Product Requirements Document)로 요약
- PRD는 문제 기술/해결안/user story/시행 결정사항/테스트 기준 등을 하나의 구조적 자산으로 보관
- 연사는 “LLM이 요약에 매우 강해, 설계 개념이 잘 공유된 이후에는 PRD 본문 자체는 굳이 직접 리뷰하지는 않음”이라 함
- 실질적 PRD/이슈-tracking은 본인 GitHub 리포의 740여 개 이슈 예시, 로컬 디렉터리 내 관리 등으로 히스토리 보관

### 수평 분할(phase→phase) 플랜보다, 세로 트레이서 불릿(Vertical Slices) 분할이 AI-개발 모두 효율적임

- PRD를 칸반 보드 기반(세로 이슈로 분할/블록킹 관계 정의)으로 쪼개면, 각 이슈별 병렬작업 및 빠른 피드백이 용이
- AI는 본능적으로 ‘수평(layer별) 코딩’을 선호(예: DB→API→프론트엔드 순), 이 경우 마지막까지 전체 흐름 피드백 불가
- 버티컬 슬라이스(트레이서 불릿): 하나의 얇은 이슈가 여러 레이어(DB, API, 프론트엔드, 테스트 등) 전체를 얇게 관통하여, 각 이슈마다 실질적 사용자 피드백·QA 가능
- 칸반 보드의 이슈 간 블록킹(선후관계) 정보로 다수 에이전트/개발자가 동시 작업 가능, Directed Acyclic Graph 형태의 유연한 플랜 관리 가능

### AI 코딩/구현 단계는 AFK 방식으로 자동화되고, 인간은 이후 QA/코드리뷰를 통해 통제권을 행사함

- 미리 정의한 이슈셋(로컬 markdown, 깃이슈 등)을 한 번에 컨텍스트로 전달, AI는 once.sh 스크립트(깃 최근 커밋 참조 등)로 자동화된 구현 시작
- 구체적 구현 스크립트(예: once.sh, Ralph loop 등)는 배포, 샌드박스화(Docker)로 리스크 최소화
- 우선순위는 크리티컬 버그→인프라→버티컬 슬라이스→빠른개선/리팩터 순으로 작업 큐 관리

### AI의 TDD(테스트주도개발) 기반 피드백 루프 활용은 코드 품질 확보의 필수 요소임

- TDD(레드–그린–리팩터): AI가 먼저 실패하는 테스트 생성→코드 작성→테스트 통과→리팩터 순으로 처리
- Test coverage 대폭 증가(예: 284개 테스트)
- LLM이 테스트를 ‘속이려’ 하는 경우도 있으나, TDD 절차 내재화 후 품질 보장 수준 향상
- 피드백 루프(테스트, 타입체크 등)는 AI의 추론력 한계와 코드 품질 ceiling을 결정하며, 고도화될수록 좋은 결과 도출

### 코드베이스 구조(‘깊은 모듈’ vs. ‘얕은 모듈’)와 AI 생산성의 상관관계 및 관리법

- 얕은 모듈(작고 과다 분리, 복잡한 의존성): AI/인간 모두에 비효율, 테스트 및 코드베이스 이해도 저하
- 깊은 모듈(명확한 인터페이스, 내부 동작은 Black Box): 모듈별 QA, 효율적 병렬 작업, 코드베이스 구조 파악 및 통제 용이
- ‘모듈 인터페이스’는 인간이 설계, 내부 구현은 AI가 담당하는 등의 업무 분리 TIP
- 코드베이스 아키텍처 개선 스킬(‘improve codebase architecture’)로 구조 개선 및 테스트합성 강화 가능

### 대규모 자동화·병렬화 프레임워크(Sand Castle 등)를 통한 실질적 워크플로우 시연

- 자체 구현 TypeScript 라이브러리 Sand Castle: 다중 워크트리/도커 샌드박스/병렬 에이전트/병합·리뷰 루프 등 완비
- 플래너→병렬 이슈 분배→각 샌드박스 내 구현/리뷰→머지 루프를 코드 예시로 상세 설명
- 코딩 표준 Push(자동 리뷰)·Pull(구현자 판단) 전략 등 세밀한 제어 방법도 공개

### AI로 코드·문서 자동화가 진행되어도, 인적 QA·코드리뷰(특히 Taste와 트렌드 반영)가 필수적임

- 헤드리스 자동화(아이디어~프로토타입~QA)에만 의존할 때 Taste·기능 완결성·실용성 저하 우려
- 문서(과거 PRD 등) 보관의 Dock Rot(문서-코드 괴리/오염) 리스크, 대부분 직접 폐기 또는 닫힘 처리
- 병렬화·자동화가 진전될수록 실제 코드 핸들링, 프론트엔드 UI 검수 등은 반드시 인간이 개입하여 품질/제품방향을 유지해야 함

### 협업 및 사례Q&A: 실질적 팀 적용, 코드리뷰 부담, 아키텍처 정책, 표준 적용 방안 등 실전 사례 중심으로 논의

- 실제 팀 현업 적용시 아이디어~도착지(설계, 프로토타입, 팀내 공유, 의견수렴) 부분은 팀 전원 참여, AI는 병렬화/자동화에 집중
- 코드리뷰 부담 증가는 불가피하나, Pull Request의 단위 최소화(Ticket별 QA 등)로 대응
- 코딩 표준, 라이브러리 사전 정의법(프롬프트 Push/Pull), 리뷰 자동화 구조, 인적 Taste 개입 등 구체적 방법 제시
- 다양한 실무 Q&A(프론트엔드 워크플로우, 문서관리, PRD 최적화 필요성, 코드 스탠다드 적용, 모듈/이슈계획 최적화 등)에 대해 직접 경험 기반 솔루션 공개
