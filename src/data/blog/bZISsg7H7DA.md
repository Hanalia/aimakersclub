## *당신의 에이전트에게 저장 버튼이 필요하다 - Hamza Tahir, ZenML* 핵심 요약

- 영상은 에이전트 실행 중의 상태와 코드, 결정 과정을 “저장”하여 언제든 재현·분석할 수 있는 ‘저장(세이브) 버튼’의 필요성을 제기함
- 기존의 트레이스(trace)는 에이전트 실행의 입력과 출력 등 일부 정보만 저장하며, 실시간 상태와 코드, 파일 시스템, 변수 등 전체 실행 맥락은 잃어버림
- 만약 저장 버튼이 있다면 모델 교체, 툴 반환값 변경, 오류 시나리오 등 다양한 “what if” 재실행 시나리오를 신속하고 체계적으로 실행 가능
- 이러한 기능을 지원하기 위해 ZenML에서 새로 출시한 Kitaru 도구가 소개되며, 실행 중 각 체크포인트의 코드/환경/결과를 저장하고 원하는 지점부터 반복 실행이 가능함을 시연
- 예시로 고객 지원 챗봇의 실행을 저장·재실행하여 모델을 교체하거나 툴 결과를 모킹(mocking)하고, 결과를 비교(diff)해 비용 및 성능을 체계적으로 검증할 수 있음을 보여줌
- DoorDash 등 업계 사례에서는 이러한 시뮬레이션 환경을 통해 수백 건의 고객 봇 실행을 5분 만에 분석하고, 환각(hallucination) 발생률을 90% 감소시키는 등 효과를 입증
- Kitaru는 개별 실행뿐만 아니라, 비용 기준 등으로 추린 다수의 실행(cohort)을 한 번에 재생/비교·분석하는 기능 역시 제공
- BrainTrust 연구에서 설명하듯 단순히 저렴한 모델로 대체하는 “1회성” 실험은 의미 없고, 대규모 리플레이와 데이터 집단 분석이 필수라는 점을 강조
- 요약하면, “실제 환경에서의 에이전트 실행 상태 저장→즉시 재실행→결과 비교 및 분석→업데이트”의 실행 루프가 AI 시스템 개발/운영의 핵심 플레이북임을 역설

## 세부 요약 - 주제별 정리

### 에이전트 실행 과정에는 과거 문서 저장과 같은 ‘저장 버튼’이 본질적으로 결여되어 있음

- 1980년대부터 문서 편집에는 ctrl+s/command+s 등 저장 버튼이 보편화되어 문서 작업 상태를 언제든 기록·복원
- 반면, AI 에이전트 및 LLM 기반 서비스의 실행(런타임)에서는 이런 저장 기능이 부재
- 기존에 사용되는 trace는 툴 호출·입출력 등의 일부 텔레메트리 정보만 기록하며, 전체 상태/코드/변수 등 전체 맥락 보존은 불가능
- trace 정보는 일반적으로 코드 실행 환경과는 별도 위치에 “읽기 전용”으로 보관됨

### 전체 상태 저장(체크포인트)은 에이전트의다양한 “what if” 실험을 가능하게 함

- 실행 중인 에이전트의 모든 상태를 저장하면, 과거 상태로 돌아가서 다양한 질문(would it have been cheaper? faster? etc.)에 실험적으로 답할 수 있음
- 예시: 오픈소스 모델로 교체하면? 툴 반환값을 오버라이드(mocking) 하면? 에러 시나리오를 가정하면?
- 저장된 상태가 없다면 이러한 실험은 불가능하거나 대단히 번거로웁고 단편적

### 에이전트 프레임워크 위에 “지속적 런타임+체크포인트” 계층이 새롭게 요구됨

- 최근에는 기존 오케스트레이션 프레임워크 위에 “런타임+체크포인트” 계층을 두는 아키텍처적 시도가 등장
- 이를 통해 trace 데이터뿐 아니라 실행 코드, 환경(Docker, sandbox 등), 파일 시스템, 변수 등 모든 상태를 체크포인트 단위로 스냅샷 저장
- 이렇게 하면 실시간으로 언제든 재생, 실험, 배포, 평가의 루프가 가능해짐
- 본 영상을 통해 시연된 대표 도구로 ZenML에서 만든 Kitaru가 소개됨

### 실사용 예시: 고객 지원 챗봇의 실행을 세밀하게 저장·재실행하며 다양한 실험이 쉬워짐

- 고객 클레임 처리 및 환불을 지원하는 에이전트 실행을 Kitaru로 상태 체크포인트화
- 각 툴 호출 시점의 코드, 환경(Docker image 등), 입·출력 데이터, 실행 소요 시간까지 모두 스냅샷 형태로 저장
- 사용자는 특정 체크포인트에서 실행을 분기(fork)하여 모델만 교체, 툴 정책(lookup policy)만 모킹하는 등 다양한 what-if 실험을 즉시 수행 가능
- 예: 기존 LLM을 ‘GPT-5 Nano’로 바꿨을 때 후속 결과와 비용, 소요 시간 등을 한눈에 분석
- 또 다른 실험으로 lookup policy 툴을 코드 내 다른 함수로 오버라이드해 에이전트의 판단 변화를 즉시 확인

### “diff” 기능을 활용해 원 실행과 다양한 실험 결과를 구조적으로 비교·분석할 수 있음

- Kitaru의 diff 기능을 통해, 최초 실행(original run)과 여러 리플레이(실험) 결과를 나란히 UI상에서 비교
- 각 체크포인트별로 무슨 결정과 변화가 있었는지, 출력 아티팩트의 차이, 실행 경로상 분기점이 어디였는지 명확하게 시각화
- 세 번째 실험의 경우 정책 변경으로 인해 최종 답변이 “safe to answer”로 바뀌는 등 실제 비즈니스 의사결정에 즉각적 영향 확인 가능
- 각 실행의 토큰 소비량, input/output 데이터, 실행 로그 등 세부 자료도 동일 화면에서 투명하게 파악

### 개별 실행뿐 아니라 대규모 batch(코호트) 실험도 모델/툴 교체 한 번으로 자동화 가능

- 비용 등 다양한 기준으로 여러 번의 실행을 묶어, 일괄적으로 모델 교체·툴 변경을 적용한 리플레이 가능(예: 모든 비싼 실행에 저가 모델 적용)
- batch 실험 결과는 JSON 리포트로 추출하고, Kitaru MCP 서버로 자동 분석
- 여러 실험 결과를 구조적으로 요약/분석하여 의사결정에 도움
- 대규모 데이터(수천건 이상)는 자동 분석 에이전트 및 MCP 서버의 활약이 중요하며, 런타임 상태를 쿼리·접근할 수 있는 구조가 필수

### DoorDash 사례를 통해 시뮬레이션·재실행 기반 실험의 실효성이 입증됨

- DoorDash는 2023년 6월 1일 공식 블로그를 통해 이런 시뮬레이션 환경을 도입한 사례를 공개
- 수시간 소요되던 고객 서비스 봇의 수백 건 실험을 5분으로 단축
- 실행 기반 시뮬레이션으로 환각(잘못된 답변) 90% 감축, 실제 환경(observer)와 2% 이내 차이의 정확한 평가
- 이는 저장된 실행 상태 기반 리플레이가 실제 문제 해결 및 시스템 개선에 크게 기여함을 방증

### 단순 모델·툴 교체만으로 비용 절감을 기대하는 것은 “가짜 경제”임을 경고

- BrainTrust 연구(참조) 결과, 비용(토큰 등)만 보고 저가 모델로 교체하는 것은 소위 “false economy”일 수 있음
- 예: 더 저렴하지만 결과 자체가 떨어져 지원 요청이 제대로 해결되지 않는다면 본질적 가치는 하락
- Towell Bench 등 벤치마크에서 어떤 모델이 60% 기준을 통과해도, 실제 일관된 결과는 25% 이하
- 결론: “단편 실험”은 오류가 크고, “대규모 코호트 분석”만이 실제 서비스 개선에 기여

### 저장/체크포인트 기반 실험 분석 루프가 AI 실사용(Production) 환경의 필수적 플레이북임

- 실제 실행(실사용 데이터 기반)에 기반한 코호트 선정(비용, 리스크, 속도 등), 대규모 실험 및 반복적 리플레이/분석 루프 구축이 중요
- “한두 번”의 단편적 실험은 무의미하며, 반드시 자동화/대량 분석 체계를 갖춘 에이전트 도구가 필요
- 실질적 플레이북: 실서비스에서의 실행→저장→코호트 선정→리플레이→결과 분석(diff)→필요시 배포 및 업그레이드
- 가능하면(코호트 실행/분석은) 자동화하고, 최종 결정만 인간이 검토하는 구조가 이상적

### Kitaru 오픈소스 도구를 사용시 이러한 저장-리플레이 실험 루프를 무료로 구현 가능

- 영상에서 시연한 Kitaru는 ZenML 팀이 최근 공개
- 오픈소스로 누구나 무료 사용 가능하며, 실행별 상태 체크포인트, 리플레이, diff, batch(코호트) 분석 등 필요한 주요 기능을 제공
- GitHub 등 저장소에서 소스 직접 확인·피드백 가능

### 결론적으로 저장 버튼/체크포인트-기반 에이전트 개발이 AI 시스템의 품질·비용 최적화에 핵심적임을 재차 강조

- 효과적인 “저장-리플레이-분석(What if 루프)” 시스템이 현업 AI/에이전트 개발자에게 실질적으로 중요하다는 메시지로 마무리
- 저장, 체크포인트 방식의 실행 관리가 실제 운영 변수 조정·최적화의 핵심 수단임을 다시 한 번 환기
