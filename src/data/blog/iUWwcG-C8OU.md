## *왜 회사의 질문에 아무도 명확히 답하지 못하는가?* 핵심 요약

- 영상 제목은 *왜 회사의 질문에 아무도 명확히 답하지 못하는가?*로, WorkOS의 제품 책임자 Garrett Galow가 사내 정보 접근 문제와 이를 해결하기 위한 자체 툴 ‘Studio’의 개발 과정을 설명함
- 전통적으로 구성원들은 비즈니스 관련 질문(마케팅 성과, 고객 사용 정보 등)에 답을 얻기 위해 엔지니어나 데이터 담당자에게 반복적으로 요청해야 하는 비효율적 프로세스를 거침
- 각종 대시보드나 레포트 툴은 구축이 고정적·제약적이어서 새로운 질문마다 재구성 혹은 추가 개발이 필요함
- 이를 해결하기 위해 WorkOS 내부에서는 ‘Studio’라는 자체 도구를 개발해 누구나 원하는 질문을 쉽게 데이터로부터 답변받고, 필요하다면 대시보드(위젯)까지 생성 가능하도록 함
- Studio는 슬랙 봇 또는 대시보드로 질문을 받아 Opus 기반 LLM, Lane Graph 에이전트, 데이터 통합 프록시 등을 통해 자동 쿼리와 결과 제공까지 일괄 처리함
- 주요 데이터 소스로는 Snowflake, Linear, Notion을 활용하며, 가이드라인·컨텍스트 설정으로 데이터 해석 오류를 줄이고 정확도를 높임
- Studio를 활용해 마케팅 콘텐츠 별 신규 팀 생성 추적, 내부 보안 도구(Radar) 경고 내역 조회 등 다양한 사례를 손쉽게 다룰 수 있음
- 반복적 질문은 ‘위젯’으로 자동화되어 팀원들이 셀프 서비스 형태로 재활용하고 공유할 수 있게 설계됨
- LLM 쿼리 생성의 신뢰성, 컨텍스트 주입, 입력값 처리, 접근권한 관리 등 여러 안정성 및 보안적 과제와 그 해결 방안에 대해 구체적으로 설명함
- 위젯 생성 이후에는 LLM을 반복적으로 호출하지 않아 운영비 절감, 신뢰성 보장 등 실질 이점 존재

## 세부 요약 - 주제별 정리

### 전통적인 데이터 질의 프로세스는 매우 비효율적이며 반복적인 커뮤니케이션 문제가 발생함

- 사내 각 팀(마케팅, 지원 등)은 "고객이 우리 제품을 어떻게 사용하는가?"와 같은 질문에 빈번히 직면함
- 기존에는 기술적으로 SQL 등 데이터 접근 역량이 없는 인원이 엔지니어나 데이터 담당자에게 질문을 위임함
- 비즈니스 질문의 맥락과 목적을 반복적으로 설명해야 하고, 답변이 부족하거나 부정확하면 추가 질의가 이어짐
- 주로 Slack 등의 커뮤니케이션 채널을 통해 답변이 이뤄지지만, 일회성, 비효율성, 반복성 문제로 지식 축적·공유가 어려움
- 각 부서가 요구하는 새로운 정보는 기존 리포트·대시보드로 커버되지 않는 경우가 태반임
- 수기로 쿼리 처리, 결과 공유, 미세 조정 등 생산성 저하 유발

### WorkOS는 사내용 데이터 셀프 서비스 툴 Studio를 개발하여 문제를 해결하고자 함

- Studio는 WorkOS가 자체 개발한 내재화 툴로, 누구나 쉽게 비즈니스 관련 질문을 데이터로부터 답변받을 수 있도록 고안됨
- 실질적으로 지원 및 마케팅 팀 등 비기술 인력이 별도의 엔지니어 도움 없이 답을 얻음
- 반복되는 복잡한 대시보드와 레포트 수요를 위젯 형태의 자동화 가능한 형태로 전환
- 업무 프로세스의 자동화 및 생산성 향상을 도모

### Studio가 질문 처리부터 대시보드(위젯) 생성까지 전 과정을 자동화함

- 사용자는 Studio에 자연어로 질문을 입력할 수 있음(예: "어떤 콘텐츠가 신규 팀 생성에 가장 효과적인가?")
- Studio는 연결된 여러 데이터 소스(Snowflake, Linear, Notion)에 접근해 필요한 정보를 추출함
- 데이터를 추출해 단순 답변뿐만 아니라, 시간 구간별/카테고리별 추이 등 상시 참조 가능한 테이블·위젯·대시보드도 자동 구축함
- 위젯은 JavaScript 기반의 샌드박스 코드로, API 요청과 UI 포함한 완전한 셀프 서비스 도구로서 동작함
- 위젯 내 카테고리 필터, 기간별 조회 등 실시간 재조회와 다양한 설정이 가능

### Studio 아키텍처는 LLM(오퍼스)과 다양한 통합 기술 스택 및 컨텍스트 가이드라인으로 구성됨

- 사용자는 Slack에서 Studio 봇 또는 웹 대시보드를 통해 질문을 접수함
- 내부 API가 질문을 파싱하여 Lane Graph라는 LLM 에이전트에 전달하고, 이 에이전트가 Snowflake, Linear, Notion 등 데이터 소스에 적합한 쿼리를 생성·실행함
- 통합 프록시 및 가이드 레이어가 각각의 데이터 소스와 상호작용 규칙, 쿼리 방식 등을 상세히 정의함
- "고객" 엔티티 같은 내부 테이블 구조 파악, 다중 조인 등 복잡한 쿼리 문법을 미리 LLM에 컨텍스트로 삽입하여 정확도와 효율 높임
- 쿼리 실행 결과는 Convex라는 상태 저장 서비스에 보관됨으로써 세션 간 결과 유지 및 재활용 가능

### Studio를 통한 실사례: 마케팅 성과 측정 및 자주 보는 리포트의 위젯화

- 마케팅 팀은 팟캐스트 광고, 구글 광고, 블로그 포스트 등 여러 채널로 유입된 사용자가 어떤 콘텐츠를 통해 제품에 가입했는지 궁금해함
- Studio에 "어떤 페이지 트래픽이 신규 팀 생성으로 이어지는가?"와 같은 질문을 입력
- 자동으로 Snowflake의 관련 데이터 추출, 어떤 블로그/문서/페이지가 효율적인지 파악
- '테이블 위젯'을 생성해 시간대별 신규 팀 생성, 콘텐츠 유형별 전환 효과 등 반복적 리포트를 자동화
- 해당 위젯은 동료와 공유하거나, 정기적인 미팅에서 재활용 가능

### 내부 보안 솔루션 Radar의 활용 내역도 Studio로 질문·분석 가능

- WorkOS 내부 보안 제품인 Radar는 봇, 악성 트래픽 차단 기능 제공
- 고객 또는 내부 지원팀이 "왜 이 사용자가 Radar에 의해 차단됐는가?"와 같은 질문을 자주 함
- 과거에는 복잡한 SQL 쿼리 작성, 로그 분석 필요 → Studio에서는 간단히 위젯을 통해 이메일 등 검색 입력만으로 최근 로그인, 차단/비차단 이력 등 결과 즉시 제공
- 지원팀은 본인 혹은 고객 질의 시 별도 개발자 도움 없이 즉시 셀프 서비스로 조회 가능
- 재사용성 높은 위젯 구축 시, 내부에서 자유롭게 공유 및 커스터마이징 가능함

### Studio의 설계 핵심: 시퀀싱, 컨텍스트 계층화, 검증 프로세스로 안정성과 정확도를 강화함

- **시퀀싱**: 새로운 질문이 들어오면 각종 사전점검(pre-flight check)을 실행(도구 연결 상태, 맥락 충분성 등 확인)
    - 필요한 컨텍스트가 부족할 때는 안내문 및 추가 질문을 통해 정보 확보
    - 쿼리 호출 시점에 데이터베이스 스키마, 내부 규칙 등 상세 컨텍스트를 주입
    - 예를 들어, Snowflake 연동 시 고객-환경-리소스 연결관계 등 복잡한 연결 정보를 실행시 컨텍스트로 삽입해 오류 가능성 줄임
- **계층화**: 기본 프롬프트, 조직별 가이드, 각 도구별 컨텍스트 블록 등을 계층별로 구성하여 다양한 상황에 맞게 조정
    - 제품 정보는 LLM 학습 데이터와 차이가 많아 최신화 어렵기 때문에, LLM 지식 대신 항상 1차적 실제 데이터&문서 참조하도록 가이드
- **검증(Validation)**: 쿼리 작성 시 단순 문법 오류뿐 아니라 실제 데이터 반환 여부, 실질 유의성까지 미리 검증
    - 위젯 코드 반영 전 자동 쿼리 결과를 사전 실행·확인
    - 개발 환경 및 실제 운영환경 모두에서 동일한 평가(eval) 체계를 적용, 일관성 확보

### LLM의 쿼리 정확도 보장을 위한 콘텍스트 제공 방법 및 품질 검증 노하우

- Snowflake 내부에서 복잡한 다중 조인이 필요할 경우, 반복적으로 쓰이는 쿼리 조인 블록을 정형화해 Studio에 프롬프트 형태로 전달
- LLM은 전체 스키마를 자동 해석할 수 있으나, 제대로 된 결과를 위해 추상적 설명보다는 명시적 컨텍스트 주입이 효과적임
- Radar 데이터 테이블(로그인 시도, 탐지 등)에 대한 구체적 연결법·조건(예: status 컬럼 필터, deleted 엔티티 제외 등) 또한 콘텍스트에 포함
- LLM 쿼리는 surprising하게도 RAG 방식 없이도 정확성이 높으며, 정확한 명명 규칙, 상태값, 비삭제엔티티 제한 등의 지침 공유로 오류 최소화

### 위젯 생성 이후에는 LLM을 반복 호출하지 않아 신뢰성과 효율이 높고, 사용자 입력값도 유연하게 처리됨

- 위젯은 생성된 시점 이후 LLM의 재참여 없이도 독립적으로 동작하며, 입력값(검색어 등)은 변수로 받아 각종 재조회가 가능
- 쿼리·API·UI가 하나의 샌드박스 코드로 동작해 실시간 반복 사용에 적합
- 사용자마다 코드/스크립트 수정 없이 다양한 파라미터로 활용 가능

### 멀티 소스 데이터 조합, 위젯식 코드 구현 등 Studio의 확장성과 선언적 구조

- Studio는 Snowflake, Linear, Notion 등 복수 데이터 소스를 동시에 조회, 조합하여 하나의 데이터셋 혹은 대시보드로 통합 출력 가능
- 위젯 자체가 자바스크립트 코드 형태로, 각 데이터 소스 API를 직접 호출·결합함
- 새로운 데이터 통합이 필요할 때마다 조직별 커넥터 및 접근 권한을 쉽게 추가 설정할 수 있도록 설계 중(WorkOS Pipes 등 자사 솔루션 활용)

### 데이터 접근 권한 및 보안 정책, 비용 관리 등 운영상 과제와 대응책

- 현재는 각 사용자가 본인 계정으로 Snowflake/Linear/Notion 등 도구에 직접 연결(사용자 단위 인티그레이션)
- 장기적으론 조직 단위(Org connector)로 1회 연결 후 역할/직책/권한별로 접근 제어를 확장 중
- 예: 기본적으로는 read-only, 특정 역할(관리자 등)에만 쓰기/편집 권한 허용 등 세부 설정 예정
- 위젯 생성 시에는 LLM 비용만 발생, 실시간 쿼리에는 추가비용 없음(운영 비용 효율)
- Opus LLM은 타 모델 대비 월등히 뛰어난 품질로 비용효율을 기꺼이 감수함

### 질의응답 정리: 데이터 거버넌스·정확성, 멀티 소스 결합, 운영 안정성 등에 대한 실무적 답변 제공

- 쿼리의 정확성과 내부적 감사(Audit) 필요성에 대해 명확한 콘텍스트와 현업 규칙(비삭제, 활성화 상태 등) 주입이 오류 방지에 효과적이라 언급
- 멀티 소스 결합, 사용자 입력값 인자화, 자바스크립트 기반 API 호출 등 유연한 구조로 어떤 데이터 조합도 실현 가능
- 실시간 쿼리 재실행시 LLM 호출이 없어 신뢰성과 비용 측면에서 강점
- 다양한 시나리오에서 실제 운영 성공 사례와 시행착오를 구체적으로 언급함
