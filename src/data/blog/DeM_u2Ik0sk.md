## **AIE Miami Day 2: Cerebras, OpenCode, Cursor, Arize AI 등 주요 세션 종합 요약**

- **영상 제목(한글): *AIE 마이애미 데이 2: Cerebras, OpenCode, Cursor, Arize AI 등과 함께하는 AI 엔지니어 컨퍼런스 현장***
- AI 엔지니어 마이애미 컨퍼런스 2일차의 주요 테마는 "코딩 에이전트", "메모리", "멀티모달", "에이전트 스택" 등 AI 엔지니어링 최신 흐름과 실전 적용 사례임.
- G2I(주최사)의 에이전틱 코딩 워크플로우와, 초보/시니어/주니어 엔지니어의 흡수·내재화 경험에 대한 사례 발표가 인상적임.
- "코딩 에이전트"는 더 이상 프로토타입이 아니라, 실제로 비즈니스 자동화·소프트웨어 개발 현장에서 실질적 핵심 역할로 자리잡아가는 중임.
- Cerebras와 OpenAI의 Codex Spark는 초당 1,200토큰 생성속도(20배 향상)를 통해 기존 LLM의 지연(latency debt) 한계를 크게 돌파, 차세대 개발 워크플로우 패러다임 변화를 이끎.
- 현장 데모에서는 로컬 NPU(Mobile Neural Processing Unit) 상에 Latent Diffusion 모델을 완전히 오프라인으로 구동해, 스마트폰 실시간 이미지 생성까지 성공함. 이를 통해 AI 에이전트-엣지 컴퓨팅 결합 가능성 확인.
- 코드 생성 워크플로우에서는 전문화된 '서브에이전트', '특화모델'을 다수 병렬로 활용하여, 작업비용 최적화와 속도/효율 극대화를 도모하는 최신 실전 전략(case study)들이 공유됨.
- "컨텍스트 설계(context engineering)"를 위한 '지식 그래프', '컨텍스트 그래프', '하이브리드 검색(벡터+그래프)' 기법 등이 높은 정확도·추적성·감사성 확보에 중요한 역할로 대두됨.
- 코드 에이전트가 소프트웨어 자체가 되어가고 있으며, '소프트웨어 먹는 세상'에서 '코딩 에이전트가 소프트웨어를 먹는 중'이라는 새로운 패러다임이 강하게 제시됨.
- CLI(커맨드라인 인터페이스) 기반과 MCP(Machine Communication Protocol) 기반 에이전트 각 방식을 자체 평가(evaluation)하여, 어떤 상황에 어떤 툴 접점이 더 효율적인지 실제 데이터를 통해 비교 분석.
- 최종적으로, 다양한 실전 앱 및 워크플로우의 "빠른 실험→피드백→간단/구체성 기반 구조화→새로운 모델 접목→릴런칭" 등 구체적인 AI 앱 개발 & 운영 노하우와 생생한 사례가 대거 공유됨.

## **세부 요약 - 주제별 정리**

### G2I의 에이전틱 코딩 워크플로우 내재화가 성공적 에이전트 활용의 열쇠임

- G2I 프로젝트에서는 팀원 전체가 동일한 "에이전틱 프레임워크"로 온보딩되어 도큐먼트 기반 단계별 핸드오프(슬래시 브리프→사스 스펙→코드→테스트→리뷰→PR)로 에이전트와 효율적으로 협업.
- 초보 엔지니어는 프레임워크 내 '입력 제약'으로 자신도 모르는 실수를 방지, 숙련 엔지니어는 오히려 자신의 개발력 증폭 효과 얻게 됨.
- 각 참여 엔지니어(Ava/Lucy/Antoine/Dale/Andy)들의 초기 두려움·불신→반복 실습→프롬프트 설계/리뷰/데이터 손수집 내재화→자율적 도구 제작 단계까지의 심리·실무 변화를 사례로 소개.
- 주니어(Andy)는 시니어 멘토링+에이전틱 워크플로우 병행으로, 단 3개월만에 주요기여자로 성장함(팀 내 유일 주니어).
- 내재화 성공 핵심은 '문서 중심 디버깅-리뷰-피드백 구조'와 '실패·실수·혼돈'을 개방적으로 학습자산으로 끌어안는 문화임.
- 다른 환경에서도 G2I식 프레임워크(RPI, Beam Aad 등)은 '출력 보다는 입력구조를, 검토가능성/위임능력 훈련'에 초점을 두는 것에서 차별성을 둠.
- 결론적으로, 프레임워크 자체 우열보다 '실질적 워크플로우 내재화에 의한 성장'이 본질임을 강조.

### 빠른 코드 생성(초고속 LLM) 등장과 그 영향력이 대역폭/인터랙션 모델을 혁신시킴

- Sarah Chiang(Cerebras): 모델 사이즈와 컨텍스트 윈도우가 22년~24년 4배(입력), 3배(출력) 성장, 개발자 실사용량 급증(100조 토큰 단위 조사를 통해 입증됨).
- reasoning token(중간 추론 토큰, step-by-step reasoning)이 전체 출력에서 차지하는 비중도 빠르게 늘어남.
- 코덱스 스파크(Codex Spark, Cerebras × OpenAI)는 초당 1,200 토큰 생성, 기존 공개 모델 대비 20배 속도, 완전히 새로운 속도 체제로 규정됨(정확히 y축이 늘어서 기존 차트에 안 들어갈 정도).
- 하드웨어적 혁신: 기존 GPU 대비서 메모리 이동이 전체 지연의 50~80% 차지 → Cerebras는 대용량 메모리를 on-chip(SRAM)화, 모든 코어가 직접 데이터 접근 가능.
- 상업적 트렌드: Google TPU(엔비디아 GPU 대비 4배), Anthropic·OpenAI의 하드웨어 투자, Grock LPU, Cerebras-WaferScale-HW 등 대규모 자본 투입의 증거.
- 미래: 하드웨어/소프트웨어/모델/추론(Operation) 전 계층의 "inference stack 최적화" 연쇄혁신으로 코드 생성-리뷰 속도가 인간 인지·행동 속도를 이미 초과, 인간이 병목으로 전환됨.
- 개발자는 입력부터 출력까지 전체 반복 주기(cost, 패턴, 대기시간) 혁신에 따라, 생산성의 새로운 패러다임 하에 진입 중임.

### 모바일 NPU 기반 로컬 디퓨전 모델(이미지 생성)이 오프라인 AI 에이전트의 신기원을 보임

- 레카리노프스키(Kalinoswki, Kstack): 스마트폰 내장 NPU에 Latent Diffusion 모델 직접 설치 운용, 임베딧·프롬프트 없이 ‘센서 데이터→수치변환→라이트 텐서/디노이즈’ 방식 설계.
- 외부 API, 네트워크, 클라우드 완전 차단(비행기 모드 데모), 센서 입력(조도, 가속도 등) 직접 latent space로 변환 후 실시간 이미지 생성 데모 성공.
- 600ms 내 고속 추론, 삼성 개인 스마트폰 단독 구동, 높은 안정성+낮은 발열로 실용적.
- 생성 결과로 이미지 외에도 심플한 애니메이션 등 다양한 활용 가능.
- 중요한 점은 디퓨전 내부 수치/랜덤성·자연 영감·경량화 구현까지 모두 메타학습/휴대 예시로, 향후 모바일 엣지 AI 결합 코어 사례 제시.
- ONNX·Android native·fallback 등 다각도 안전장치+최적화 루틴 실제 도입 경험 공유.

### 서브에이전트/특화모델 도입과 코드 생성 업무의 분산·최적화가 현행 실전 전략임

- Tis(Morph): 코드 생성 에이전트 워크플로우에선, '서브에이전트'(작업별 별도 context window)·'특화모델'(코드 검색/컴팩션/패치 등)을 결합하여 속도/비용/성능 최적화.
- 대규모 '프론티어 모델'(Opus 등)는 계획·추론·코딩 담당, 나머지는 경량 특화모델이 대량의 반복·보조 작업(코드검색, diff 적용 등) 처리, 실제 인프라 compute cost·처리량 지각변동.
- 인공지능 수용량 한계(특히 GPU 리소스 부족)에 따라, 업무 맞춤 '최소 계산량/최대 효율' 전략이 대세.
- 코드서치 모델 예시: 80K input, 200 output에서 병렬(최대 12개) API call, cost down/속도향상 실현.
- 서브에이전트용 RL 세팅 시 메인 에이전트의 assumption과 서브에이전트 실제 성능 에러(역할 오해 등) 최소화에 주력.
- Speculative decoding, Prefill/Decode 분리·전용 칩, custom kernel 등 속도혁신 기법 실제 적용.
- 실제 프로덕션 협력사 다수 대상으로 실험, 핵심은 '모든 서비스가 결국은 코딩에이전트'로 수렴, doubling speed는 conversion rate(성과지표)도 2배로 높아짐이 일관된 관찰.

### 코딩 에이전트가 소프트웨어를 "먹고 있는" 소프트웨어 패러다임 대전환 논의

- Rick Blelock(Agentuity): 2026년 현재, 코딩에이전트는 더이상 프로토타입 수준의 자동화가 아니라, 조직 실무·SMB·비엔지니어 집단까지 "소프트웨어 생산 및 운영의 원자단위"로 자리잡음.
- 최초 AI에이전트(autogen, 2023)는 루프·슬롯 중복, 잘못된 결정, 실패 반복 등 한계 있었으나, 실체적 진화(데본, 오픈클로드 등 실사용 사례)로 다양화.
- OpenClaw 같은 제품/오픈·노코드 범용 플랫폼이 등장, 비개발자도 실제로 비즈니스 업무를 위한 매크로/자동화 파이프라인 만들고 생산성 대폭 상승.
- 예시: HubSpot 대체, 창업자들이 에이전트만으로 조직 관리/데이터 파이프라인 구축, 작고 빠른 시장에서 스타트업/개인 사업자들이 업무 흐름 자체를 coding agent에 위임.
- 기승전결: "개발자만 에이전트 쓰는 시대"→"비개발자도 coding agent로 실질적 업무 파이프라인 직접 구축하는 시대".
- 도구/클라우드 인프라 역시 코드에이전트 기반에 맞춰 점진적 수렴 중, '에이전트 플랫폼/클라우드/서비스' 로 이행 예상.

### "지식 그래프 기반 컨텍스트 설계"가 AI 에이전트 정확도·투명성 혁신의 열쇠로 부상

- Nia(Mlin, Neo4j, GraphAgg): 단순 텍스트 벡터/램 기반 검색(naive/hybrid RAG)은 문맥 유사성은 높이지만, "구조적 연결(Structural Similarity)" 정보 결여되어 실무적 맹점 발생(예: 제재리스트 검증, 법률, 컴플라이언스...).
- MIT NANDA(2025) 보고서, 기업 95% AI 시범PJT가 실질적 이익 미비, 40%는 2027년까지 폐기 예상→핵심 원인 '컨텍스트 단절, 관계 정보 부재'.
- 실제 사례: 37%(Base Model 정확도)→54%(도메인 파인튜닝)→91%(그래프+RAG 결합)로 도달, "그래프-자동화 컨텍스트 없인 실전 deploy 어려움" 실증.
- 지식/컨텍스트 그래프는 "결정 추적성(decision trace)", "Causal audit", "사전 정책·과거 사건·주체간 관계 등" 전 과정을 시각/질의/감사 가능, 에이전트 생성 결과물의 투명성·신뢰성·법적 책임 가능성 구현.
- 머신러닝 기반 벡터 임베딩+그래프 임베딩(hybrid search) 동시 적용, 자료구조/쿼리/API 등 오픈소스와 실제 시연으로 공식화.
- Context Graph 컨셉(2023~), 연결/조건/결과·이유(why)를 모두 기록·질의, 다중 에이전트/복합적 의사결정/규제 대응 등에서 핵심 인프라로 자리매김 중.

### 실전 AI 개발(Production App) AI 앱, '속도-UI-변화 빠른 실험'이 성공적 확산 공통 공식임

- Hassan(TogetherAI): 연간 10~15종 AI 앱 제작 경험, 실습 주기·빠른 피드백·최신 모델 접목·UI에 집중한 반복 배포가 핵심.
- 전체적인 AI 앱 아키텍처는 4단계(입력→모델→DB→유저)로 매우 단순화하여 빠르게 실험.
- 오픈소스 AI코딩툴(Codeex, Cursor, Cloud Code 등) 실제 조합/로테이션(생성/리뷰/가이드), UI/UX 최적화에 전체 에너지 80% 집중.
- 최신 모델(이미지, 오디오, 코딩 등)이 릴리즈 되면 즉시 접목하여 앱 성능·기능 leapfrog 추진, 빠른 버전2~n 릴런칭으로 네트워크효과+유저 반응 강하게 견인.
- 핵심 교훈: AI 시대 앱 성공 조건은 "단순한 구조/작은 유닛/빠른 실험/자주 릴런칭/근거있는 피드백+직접적 성장"임을 수치/실험으로 입증.

### 멀티모달·오픈소스·플렉시블 모델 선택권 강조가 코드 AI 툴 미래 경쟁력임

### 커맨드라인(CLI) vs MCP(Machine Communication Protocol) 기반 에이전트, 실제 데이터로 성능 및 한계 비교

- Lori Voss(Arize AI): 'CLI만 써라' vs 'MCP가 진리' 트위터 논쟁에 실제 에이전트 평가(evaluation, eval) 실험.
- 실험: GitHub 공식 MCP 서버 vs gh CLI 스킬(markdown 기반) vs baseline(아무 스킬 없이 transmite), Opus 4.6 단일 모델, 25개 과제, 4단계 난이도(읽기/쓰기/복합 트리)를 각각 5회 반복.
- 결과: 정확도 별 차이 없음(모두 80~100%), 하지만 MCP는 CLI 대비, 실행시간(최대 5배 이상), 비용(최대 6배), 호출수(최대 12배↑)에서 열세.
- CLI스킬도 별거 없이 baseline(아무 스킬 없음)과 차이 미미. 단, 유명한 gh CLI는 이미 LLM 훈련데이터에 익숙해 추가 지식 없이도 에이전트가 잘 사용. 반면 조직별 자체 CLI/new API는 스킬·docs·맥락 주입 안 하면 사용 실패 확률 높음.
- CLI 선택: 로컬/개발자 중심/구성·확장 자유/파이프라인 연결임. MCP: 원격/소비자/복잡한 인증·접근제어 필요/에이전트→에이전트 협동 작업 등에서 강점.
- 결론: 트위터 논쟁 무의미, 실제 현장에선 두 방법 병행 적용이 실전 해법.

### 실제 AI 제품의 성공을 가르는 것은 '행동 런타임' 및 베이스 인프라 설계임

- Lena Hall(Akapi): 리치 미니 로봇×LLM 기반 실제 데모(실시간 스트리밍 오디오, 영상 인식, 오디오 입출력 담당한 로봇과 멀티레이어 시스템) 선보임.
- 행동 런타임 설계의 본질은, '모델은 의도(intent)만, 실제 행동은 런타임이 결정', 기능경계(behavior-runtime) 명확하게 구분, 하드웨어 안전성까지 책임.
- 실시간 응답, 인터럽트·큐 동시 처리·타이밍·성격(voice/policy)·아이들 상태 등 모든 동작 레이어 설계가 제품 퀄리티와 직결.
- 많은 AI 제품이 실제 모델보다 런타임의 미성숙으로 '이상한' 사용자 경험 초래. 모델 타령 전, 런타임과 실무오케스트레이션, 행동코디네이션 계층부터 점검해야.
- 엔지니어 고유의 시스템 설계·운영 역량이 AI 시대에도 여전히 핵심. "모델 인프라보다 행동계층(company, coordination)이 곧 제품임" 강조.

### "메모리 문제"는 결국, 도구/인프라 추가가 아닌 '에이전틱 내재화적 접근'으로 전환 중

- Alvin Payne(Alt Rival): 대용량 대화/컨텍스트 다루며, 전통 RAG(벡터/파이프라인)보다 "에이전틱 메모리(동적 메모리 발견법, DMD: Dynamic Memory Discovery)"로 industry 벤치마크(롱멤) 100% 달성.
- 접근: 벡터DB 등 인프라 요소 모두 걷어내고, 파일시스템 기록, 에이전트가 직접 디스크 탐색(프리미티브 툴 콜), 메타정보 업데이트, 반복 실행.
- 실제 100% 기준점조차 실전 투입 시 어려움 다수(ex. 시계열 추론, 엔티티 중복/혼동, 원리적 정보 삭제 전략 등은 여전히 어려운 활자).
- 실제 '상태/큐레이션/라이프사이클'로 분할하고, 생산성·실전성·성능 효율화를 위해선 raw–derived 당장 조건 및 필요에 따라 구조를 조정.
- 결론은, 에이전트 빌딩=곧 메모리 시스템 설계, 오늘날 LLM 성능은 창의적 프롬프트·인프라 추가 이전에 "올바른 정보를 올바른 때 올바른 창에 넣기" 기술이 관건임을 강조.

### 실질적인 에이전트 '스킬' 개발과 복수 실험·실수 내재화가 차별화의 핵심

- Eric(Code Rabbit): 🍀 '빌리언달러 SaaS 만들어줘'처럼 완벽지시식 프롬핑 대신, 반복 실험·실패·교정루프 직접 수동화-자동화하여 스킬화하는 게 진정한 성장임을 체험 사례+데모(음악/스토리/웹)로 공유.
- 실제로 강력한 스킬은 단 1회성(one-shot)이 아닌, 수십회 이상의 실패/교정 루프에서 나온다. 구체 실패원인 발견·명시화·교정 피드백을 지속적으로 기록해야 함.
- 새로운 모델 나올 때마다 “하위 호환성”을 위해 일부러 더 약한 모델에서부터 스킬 튜닝, RL 커버리지 최대화.
- "Continuous learning"라는 거창한 목표보단, “Skill 파일에 ‘교훈’을 직접 써넣는 방식”으로도 충분히 ‘연속적 메모리’ 효과 달성.
- 실제 오픈 평가 프롬프트, 스킬 자가생성 스킬, 자동화 루프 등 다양한 실제 활용법 소개.
- '진짜 차별화는, 에이전트나 모델 본연이 제공 못하는 반복된 오류극복 경험'임을, 실험–피드백–교정–내재화 루프 반복을 통해 드러냄.

### 현대 IDE(통합 개발 환경)는 AI 퍼스트 시점에서 구조적 재설계가 필요함

- David(Cursor): 과거 Emacs/Vim–VS Code 시대, 확장가능성·옵션과표·키맵 사용자화 중심→오늘날 탭완성 등 AI코드=즉시화→이제 탭 사용조차 줄어들고, 전통 설정/커스터마이즈의 생산성 실효성 급격히 감소.
- 실제로, 동영상 재생, 이미지 렌더링/메모, 복합 UI(mermaid, knowledge graph 등) 등 rich interface가 본질적,
- 데스크탑 vs 클라우드 워크플로우 비중, 클라우드에서 더 많은 에이전트 병렬 운영, IDE는 손수 코딩보단 "검토/조정" 위주 플랫폼으로 변모 중.
- 실험: Cursor 2.0→3.0 본질은 기존 VS Code 아키텍처 족쇄로 운영한계→UI 자체를 완전히 재설계, VS Code와 병렬 운영·점진적 적용 방식으로 혁신 친화성 확보.
- 유저 피드백(모바일/Xcode 연계/에이전트간 핸드오프/코드리뷰 UI 등) 수렴이 곧 제품 경쟁력 원천.
- 핵심: "IDE=코드작성 도구"서 "AI–코드–다중에이전트 협업–검증–특수UI" 허브로 완전 진화중임.
