## *프론티어 결과, 온디바이스 - RL 네이버스, Arize* 핵심 요약

- 영상에서는 웹, AI, UI 분야 전문가 Rachel Lee Neighbors가 중앙집중형 대형 모델(LLM) 대신 개별 장치에서 동작하는 소형·특화 모델로 AI 비용을 줄이고 효율을 높이는 방법을 구체적으로 시연한다.
- 대형 LLM 활용 시 발생하는 보안 위험(외부 서버 데이터 노출), 응답 지연(4초 넘으면 사용자 몰입 저해), 예측 불가한 API 비용, 오프라인 사용 불가, 생산성 저하 등이 지적됨.
- 최근 LLM 가격은 하락 추세지만, 실제 추론(인퍼런스) 총비용은 늘어나는 경우가 많음; 특히 에이전트 지향형 워크플로우에서 토큰 소비가 급증함.
- 소형 언어 모델(Small Language Models, SLM) 및 특화(task-specific) 모델은 파라미터 수, 저장공간, 전력 소모가 훨씬 작으며, 온디바이스 배포로 보안·비용·latency 문제를 대폭 완화 가능.
- 예시 모델: MobileNet, YOLO, MediaPipe(비전), Whisper, Wave2Vec2(오디오), Gemma, Qwen(텍스트 등), Llama 등 다수 소형 모델이 실제로 1~5GB 내외 디바이스에 탑재 가능.
- “프로토타입은 대형 모델로, 실제 배포는 작은 모델로(Prototype Big, Deploy Small)” 개발 프레임워크를 직접 설명—Google과 협력하여 구축한 모델 선택 기준과 워크플로우 공유.
- 실제 SNS 대화 요약 기능 개발 예시를 기반으로, 골든 데이터셋 구축, 기준 정의, SLM/LLM 및 주요 모델군 성능 벤치마킹 등 구체적 절차를 소개함(Arize 오픈소스 ‘Phoenix’ 활용).
- 벤치마크 결과, Llama 3.2가 속도와 정확도 모두에서 균형이 가장 뛰어남(약 90% 정확도, latency 1초), ‘few-shot prompt’ 적용 시 추가 성능향상 확인.
- 프롬프트 엔지니어링(few-shot, chain of thought 등)과 후처리(포스트프로세싱)를 통해 SLM 성능을 LLM과 근접하게 끌어올릴 수 있음을 실증함.
- “최소한의 크기로, 충분한(good enough) 모델(SAGE: Small And Good Enough)” 선정, 실무 적용, 지속적 평가(리그레션 테스팅) 권장.

## 세부 요약 - 주제별 정리

### LLM을 사용하면 비용, 보안, 사용자 경험, 생산성 모두 손해를 볼 수 있음

- LLM(GPT-5/Claude 등) 등 퍼블릭·프론티어 모델 사용 시, 데이터와 프롬프트가 원격 서버로 전송됨.
- 외부 서버 노출에 따른 데이터 유출, 미인가 저장 및 3자 접근 위험 있음(실제 비즈니스 민감 정보 유출 사례 발생).
- 지연(latency) 문제: 연구 결과 VR 환경의 첨단 LLM 챗봇 응답 지연이 4초를 넘으면 사용자가 “실제처럼” 느끼지 못함.
- 대부분 LLM 호출은 4초 이상 걸릴 수 있어 사용자 경험 저해.
- 인퍼런스 비용은 사용자가 통제하기 어려움(API call·토큰·에이전시 레벨의 중첩 비용 발생).
- 오프라인 환경/네트워크 이상 시 LLM 완전 불가; 생산성 저하, 현장성, 독립성 제한
- 토큰 단가 하락 속도보다 쿼리량/사용량 급증(특히 에이전트형 워크플로우)으로 실 사용자 월 비용 상승.

### “정말로 LLM이 필요한가?”라는 질문에서 온디바이스/소형 모델 활용이 시작됨

- 대형 LLM의 모든 능력이 실제 사용 목표에 항상 필요한 것은 아님.
- 특정 과제(예: 대화 요약, 감정 탐지 등)는 훨씬 적은 파라미터 수로 처리 가능.
- 대체 옵션은 “특화 모델” (예: 카메라 → MobileNet/YOLO/MediaPipe, 마이크 → Whisper/Wave2Vec2, 텍스트→Gemma, Qwen 등).
- SLM(Small Language Model)은 수백만~수십억 파라미터(1~5GB)에 불과하여 모바일·PC에 쉽게 탑재 가능.
- 대부분 8비트·4비트 양자화로 메모리/디스크 사용량을 1/4까지 축소—1B 파라미터=FP16 약 2GB.

### 소형 언어 모델(SLM)은 실제 제품에 바로 투입 가능하며 에너지 효율도 대단히 높음

- NVIDIA 등 여러 리포트에서 SLM은 에이전트 기반 AI의 미래로 지목됨.
- 2025년 논문 기준 SLM, 기존 LLM 수준 과제에 충분히 활용 가능, 에너지 소모는 LLM 대비 25% 수준.
- 각종 실험에서 과제 달성률(정확도), 처리 속도, 실제 배포 용이성 등에서 우수함.
- SLM/특화 모델은 오프라인/온디바이스 배포가 가능—보안(로컬 데이터), 비용(서버/API 無), latency(라운드트립 無) 모두 향상.
- 실제 사용 예시로 Pixel Pro/PXL 10 Pro에 탑재된 온디바이스 모델, SLM 기반 채팅 앱 등 다수 소개.

### 모델 선정은 ‘크거나 최신’이 아닌, 목적에 “충분히” 맞는 최소 모델 선정이 핵심임

- 크고 최신 모델일수록 리소스 소모, 불필요한 데이터, 낮은 효율 가능성.
- 예) 소셜미디어 대화요약 용도라면 전체 인류 지식(LLM)이 필요 없으며, 맞춤형 SLM이 더 적합.
- 수억~수십억 파라미터의 작은 모델이 대화요약 등 실전 미션에서 뛰어난 성과 가능.
- 구글과 협력 구축한 “프로토타입은 큰 모델로, 배포는 작은 모델로(Prototype Big, Deploy Small)” 철학 적극 강조.

### 실제 과제—SNS 대화 요약 기능: “Mima” 사례로 워크플로우와 성능 측정 과정을 상세히 공개함

- SNS 다중 스레드 요약 기능 개발.
- 대형 모델(Claude)로 프로토타입 구현 → 입력·출력셋(스레드, 요약본 등) 수집, 골든 데이터셋(JSONL, 총 14건+2버전=28) 구축.
- 평가 기준(정확성·JSON구조·참조 일치·사실성·요약 길이·P50/95 latency) 정의.
- 각 소형 모델(Quen 2.5/3, Llama 3.2, Gemma 4 등)과 대형 모델(Claude Sonnet) 간 캐퍼빌리티 평가(Arize ‘Phoenix’ 활용).
- 비용 측정: 늘어나는 API 사용량 감안, 실제 LLM(Claude)기준 일평균 1달러씩 소진됨, 소형 모델 온디바이스 추론 시 비용 “0”.
- 디스크/메모리 부하: Qwen(1.5–1.7B, 1GB), Llama 3.2(3B, 2GB), Gemma 4(5B, 3.1GB) 등 실제 사용자 단 말단 디바이스 적용 가능 수준.

### 실제 벤치마킹 결과 Llama 3.2가 가장 뛰어난 정확도–속도–리소스 효율을 보임

- Quen 2.5: P50 latency 약 1초로 최속이나, 정밀도가 낮아 실제 요약 품질 하락.
- Gemma 4: 업계 추천 많으나, 평균 latency 8초로 사용성 저하.
- Claude Sonnet: 정확도 최상, latency(3초)로 기준점 역할.
- Llama 3.2(메타): 정확도 약90%, latency 1초 내외, 더욱 소셜 대화 요약 용도에 특화되어 실제 결과가 Claude와 “구분 불가” 수준.
- 주요 기준(정확성, 구조 일치, latency 등) 평균, 모델별 다중 수행 후 산출.

### “SAGE 모델”—충분히 작고 좋은(작고, 충분히 좋은) 모델이 최적 선택임

- 가장 작은 모델로 시작해, ‘충분히 좋은’ 응답이 나올 때까지 테스트 후 그 모델을 선정(Small And Good Enough, SAGE).
- 현업에서는 자의적으로 유명모델/신형모델만 따를 경우, 의외의 성능 저하/사용 경험 훼손이 발생할 수 있음을 사례로 설명.

### “few-shot prompt” 등 프롬프트 엔지니어링으로 SLM 성능을 추가로 개선할 수 있음

- SLM의 LLM과의 성능 차이는 프롬프트 구성(입력 방식, 예시 제공, 명령어 엄격성 등) 조정으로 상당히 메울 수 있음.
- 프롬프트 바리에이션(기존, 숫자 메시지화, few-shot 예시, 규칙 강조, chain-of-thought 등) 5종 실험 실시.
- few-shot prompt(예시 제시)가 길이·정확성·구조 일치 등에서 가장 큰 개선 효과를 보임(정확도 91.7%, 사실성 92.9%, latency 단 200ms 증가).
- 너무 엄격한 명령어(규칙 강조)는 오히려 성능을 저해(모델이 거부반응).
- chain-of-thought(사고과정 노출)은 길이 판별에 긍정, latency 약간 증가(600ms).

### 후처리(post-processing)로 구조/길이/참조 등 형식·일관성 문제를 완전히 해소할 수 있음

- 구조 일치성(JSON·참조) 및 길이 이슈는 간단한 후처리로 보정 가능.
- 예: 요약 길이 초과시 자르기(truncate), 참조 수/스레드 일치 체크 등은 “코드 레벨”에서 쉽게 자동화.
- 실제 Bar: few-shot 프롬프트+후처리 적용시 SLM(Llama 3.2)가 LLM(Claude)의 기준값을 모두 충족 혹은 초과 달성함.

### 성능 평가는 단편적 지표뿐 아니라 직접 예/출력 비교, 지속성(regression test)이 중요함

- “factual consistency” 등 LLM, SLM간 약 10% 차이는 실제 평가관 기준 또는 엄격판정에서 발생할 수 있음(Claude로 기준 잡을시 Claude output 판정이 bias 일으킬 수 있음).
- 모델/프롬프트 업그레이드시 구조화·일관성·hallucination 방지 위해 지속적/자동화 평가(리그레션) 필수; CI/CD 테스트와 유사한 자동화 프로세스 권장.

### 크롬, 스마트폰 등 사용자 디바이스에 이미 내장된 SLM 활용으로 배포 부담, PII, 환경 영향 등 추가 개선 기대

- 예시: Chrome의 Prompt API는 “Gemini Nano”에 직접 연결, 별도 모델 배포 필요 없음.
- 온디바이스 SLM 활용시 사용자 데이터 완전 로컬 보관, PII(개인정보 보호) 보장, 서버 부하–에너지–환경 영향까지 줄일 수 있음.
- “Prototype big, deploy small” 전략으로, 프로토타입(LLM)—골든데이터셋—SLM 테스트—최적 SAGE 선정까지 체계화할 것.

### 누구에게나 적용 가능한 실행 제안 및 도구: 사용 중인 LLM 호출을 SLM으로 변환해 직접 실험해볼 것

- 현재 서비스/제품/프로젝트 내 LLM 호출 중 실제로 SLM으로 대체 가능한 부분이 없는지 점검할 것(비교시 하루 1달러 절감 등 실질적 인퍼런스 비용 절감 발생).
- 소형/특화 모델이 이미 디바이스 내장(Chrome/Gemini Nano 등)되어 있을 수 있으므로 활용 전략 모색 권고.
- 개발자/엔지니어는 프로토타입과 프로덕션 분리, 변환·테스트·평가 자동화 절차(Phoenix 도구·웹 프레임워크 등)로 실무 전환 독려.
- 본 영상 내 언급된 실험/벤치마킹 도구: Arize Phoenix(무료, 오픈소스), SNS 요약앱 “Mima”, 개인 블로그(nearestneighbors.com) 및 도구 링크 제공.
