## *RL 에이전트를 활용한 ETL 파이프라인 장애 감지 및 복구 자동화* 핵심 요약

- 이 영상은 Anna Marie Benzon이 RL(강화학습) 기반 에이전트를 이용하여 ETL(Extract, Transform, Load) 파이프라인 장애를 자동으로 감지하고 복구하는 시스템 설계를 소개한다.
- 전통적으로 ETL 장애 대응은 엔지니어가 로그, 스키마, 데이터 등을 직접 확인하고, 진단 및 안전한 복구 방식을 결정하는 고비용·고지연 프로세스에 의존해왔다.
- 영상에서 제시하는 RL 기반 시스템은 반복적이고 인지 가능한 장애를 자동 복구하고, 불확실·고위험 장애 사례만 엔지니어에게 이관(에스컬레이션)한다.
- AWS Glue, EventBridge, Lambda, CloudWatch 등 클라우드 인프라와 연동된 구조를 통해, 장애 이벤트 포착–증거 수집–상태 분석–복구 방안 선택–안전성 검토–작동–결과 검증의 자동화된 실행 흐름을 구현했다.
- 시스템의 핵심은 "무엇이 정상/비정상인가"를 결정하는 확정적 규칙(Anomaly Rule), RL을 통한 상황별 행동 선택(Q-Learning), 그리고 정책 바깥의 안전성 오버라이드 계층.
- 엔지니어가 신뢰할 수 있도록, 각 단계별 의사결정 근거(예: 리스크 점수, 데이터 품질), 안전 가드, 취한 액션, 실패시 이관 등이 기록·검증되도록 설계되었다.
- 공개된 벤치마크 환경에서 룰 기반 이상 감지기는 precision 1.0, recall 0.8, F1 0.889을 달성했으며, RL 자동화 시스템은 평균 2.5일 소요되는 장애 복구를 약 5.24분으로 단축(MTTR 약 99.85% 감소)했다.
- RL 정책은 실제성과 설명력(Inspection 가능한 Action Table) 모두 확보하며, 정책·안전성 구성의 차이를 상세히 검증했다.
- 장애 다양성, 온라인 정책 검증/롤백, 지속적 모니터링 등 실제 배포를 위한 한계와 미래 단계를 명확히 제시했다.
- 구현 코드, 벤치마크, 실험 스크립트 등은 Github를 통해 오픈소스로 제공된다.


## 세부 요약 - 주제별 정리

### 전통적 ETL 장애 복구 프로세스는 고비용·고지연 문제가 있다

- 엔지니어가 매번 대시보드를 확인하고, 로그와 스키마, 데이터 원본 상태를 점검한 뒤 장애 원인을 분석
- 같은 패턴의 질문, 반복되는 원인(데이터 형식 변동, 스키마 불일치 등)이 밤낮 없이 발생
- 장애 자체는 사소할지라도, 진단–복구–검증 전체 과정이 상당한 시간적·심리적 비용을 유발
- 평균적으로 한 장애당 약 2.5 워킹데이(근무일)가 소요되는 것으로 모델링됨
- 수동 장애 복구는 엔지니어 집중력 소모, 운영 리스크 증대, 생산성 저하 등 부정적 파급 효과 발생

### RL 기반 장애 복구 시스템은 반복적 장애는 자동처리하고 고위험만 엔지니어에게 이관한다

- 목표는 “패턴이 있는, 반복적 장애”는 자동 인식 및 안전 복구하고, 새로운(신규), 불확실, 고위험 장애만 에스컬레이션
- 장애 인식–분류–데이터 품질/리스크 평가–상태(State) 생성–강화학습 결정 엔진–행동(Action) 제안–안전 계층 검토–실행 및 결과 검증의 end-to-end 프로세스 자동화
- 결과적으로 장애의 약 75%는 완전 자동 복구, 나머지만 엔지니어 직접 개입 필요

### AWS와 연동된 자동화 파이프라인으로 장애에서 복구까지 빠르게 단계화한다

- AWS Glue: 작업(Job) 실패 시 이벤트 발생
- Amazon EventBridge: 장애 이벤트 포착
- AWS Lambda: RL 에이전트가 탑재된 람다 함수 기동
- Lambda–CloudWatch(로그) & Glue 카탈로그(스키마 메타데이터)로부터 증거(evidence) 확보
- 수집된 신호(로그, 스키마, 데이터 품질 등) 기반으로 RL 에이전트가 상태(State)를 도출, 적합한 Action을 선택
- Action을 실행하기 전 반드시 Safety 계층에서 정책 외 컨디션 확인(리스크 평가, 규칙 위반, 임계치 초과 시 오버라이드)
- 성공 시 S3 아티팩트 기록·결과물 검증, 실패/실행 불가 시 즉시 이관 및 기록
- 전체 실행 이력, 판단 근거, 장애 상황은 모두 감사를 위해 저장

### 장애 종류, 신호, 상태 정의–각각의 컴포넌트 역할이 명확히 분리된다

- 장애 유형: 조기 도착, 데이터 타입 불일치, 스키마 변화, 실행 오류 등 다양한 실제 ETL 에러 유형을 커버
- 각 신호(로그, 스키마 변화, 데이터 품질, 리스크 등)가 명확히 분리된 컴포넌트에서 추출됨
- 터미니스틱(확정적) 룰: 예) 스키마 필드 소실, 데이터 타입 변경, 결측률 임계치 초과 등은 사람이 규칙을 직접 정의
- ML 컴포넌트와 룰 베이스 분리를 엄격히 준수 — 기준이 확실한 것은 ML이 아닌 룰로, 나머지만 RL
- Q-Learning 자체는 상태(State)는 장애 유형, 리스크 점수, 데이터 품질 상태 등 최소한의 정보로 구성됨

### RL 에이전트의 Q-Learning 정책과 안전성 계층의 결합이 핵심 아키텍처다

- Q-Learning은 소규모 상태-행동 공간에서 탁월(Compact State/Action space, Q-Table로 직관적 검증 가능)
- Action set: Retry(재시도), 커버(스키마 롤백/현행화), Quarantine(격리), Escalate(이관), Log 등 6가지
- ‘Action’은 단일스텝 결정 문제로 모델링 (장애…단일 행동 선택)
- RL 정책은 경험(Outcome) 기반 행동 선호도를 학습하고, 엔지니어가 검증 가능하도록 설계됨
- 하지만 RL 정책이 현실에서 모든 상황 판단 권한을 가지지 않음 — 안전성 계층이 의무적으로 개입하여, 임계치 상회/미지원 액션/불확실 장애 상황 시 정책 무시하고 Escalate 수행

### 데이터 품질·리스크 평가 도구가 실제 신호 해석 및 복구 정책 판단에 결정적 역할을 한다

- 스키마 프로파일러: 스키마 구조, 데이터 타입, 중첩(네스팅), 통계량 등 자동 추출
- 드리프트 디텍터: 베이스라인 대비 필드 추가/삭제/타입 변화 감지
- 데이터 품질 분석기: 완전성, 유효성, 일관성 지표 산출
- 에러 분류기: 로그 패턴을 사전 정의된 장애 카테고리로 매핑
- 리스크 스코어러: 위 신호 종합해 운영상 리스크 레벨 결정
- 이 모든 컴포넌트는 확정적(straightforward) 데이터 기반, 명시적 규칙으로 작동 → 검증/설명/감사 용이
- 일부 클래스(이상탐지 등)는 추가적으로 ML/학습 기반으로 확장 가능하게 설계

### 장애 복구 시스템은 벤치마크에서 높은 정확도와 엄청난 복구 시간 단축을 입증했다

- 공개된 벤치마크 환경: 사내 또는 공개된 클라이언트 시뮬레이션 데이터, 스키마, 로그, 인시던트 시나리오 총망라
- 룰 기반 이상감지기: Precision 1.0, Recall 0.8, F1 Score 0.889로 신호 플래깅
- RL 가이드의 워크플로우: 30회 테스트에서 평균 복구시간 5.24분(MTTR), 수동기준 2.5 워킹데이 → 약 99.85% 단축
- RL 정책을 핸드코딩 정책과 비교 — 동등한 성공률 및 설명력 유지, 랜덤 정책 대비 15.63%p 높은 성능
- Safety Override 활성화 시 Escalation(엔지니어에게 이관) 비율 정상적으로 증가, 이는 운영상 신뢰성 의미
- “성공률(Non-Escalation)”만 보고 최적화하면 오히려 위험 — 경계 밖은 반드시 명확히 에스컬레이션해야 실제 안전

### 복구 정책의 실행 가능성과 운영 환경 반영 여부까지 엄격히 검증된다

- 실제 장애 복구는 “정책상 안전해도, 인프라나 권한상 실행 안 되는 경우”가 있음
- 시스템은 해당 상황도 명시적으로 인지·보고·기록한 뒤 즉시 수동 개입 요청
- “모든 정책–실행 가능성–안전성 컨트롤” 3계층 명확한 분리가 신뢰성의 핵심으로 강조됨
- 이관(Escalation)은 포기 아님: “증거/권한 바운더리를 넘은 상황임을 올바로 인식→ 엔지니어에게 넘김”이 자동화 목표임

### 파이프라인의 실제 성능은 테스팅 구조와 오픈소스 정책을 통해 재현성과 확장성을 확보한다

- 실험, 정책, 벤치마크 코드는 깃허브에 공개되어, 다양한 컨트롤 그룹, 시드(seed), 반복 시험에 활용 가능
- 클라우드 환경(예: AWS Lambda/Glue 등) 뿐 아니라, 동일 패턴의 ETL/데이터 파이프라인에도 확장 적용 가능
- 테스팅 환경에서는 실제 클라이언트 데이터·문서·인프라ID 등이 아닌 익명화된 시뮬레이션을 이용해 보안 유지
- 벤치마크 스크립트 실행 온라인 독립적 재현 가능, 퍼블릭 코드를 기반으로 실험/확장 의도

### 실무 적용에서는 “최소 모델–명확 상태–강한 안전성–재현성”이 셀프힐링 시스템의 본질이다

- 무조건 큰/복잡한 모델이 자동화의 정답이 아님
- 각 디시전(행동)마다 명확한 상태정의와, 감찰가능한(Observation possible) 구조, 그리고 의심이 생기면 무조건 멈춤(Don’t override safety)이 실제 운영상 실효성
- RL 정책업데이트가 권한 경계를 임의로 바꿔서는 안 됨(독립된 safety layer 필요)
- 인시던트 이력 누적/복잡성 증가 시, 수작업 정책 관리와 RL의 이점 비대칭 발생 — RL의 실질적 가치가 상승

### 완전자동화의 목적은 “인간 판단의 제거”가 아니라 “인간의 판단 자원을 더 중요한 장애에 집중시키는 것”이다

- 반복적 장애(예: 유형 반복 오류)는 자동화, 교대근무자의 동일문제 재확인/조치 작업 최소화
- 근본적으로 의사결정을 필요로 하는 신규/고위험/불확실 장애만 수동 개입하도록 의도
- 각 장애조치 단계 및 근거(이관 원인, 상태, 리스크 등)는 명시적 기록·감찰로 엔지니어가 언제든 확인 가능
- 궁극적으로 엔지니어의 시간과 집중력을 고위험·비정형 사례에 몰입할 수 있도록 지원

### 구현 코드, 벤치마크, 가이드라인 등 지속적 오픈소스화로 참여와 확장성을 독려한다

- RL 정책, Q-Table, Safety Layer, 벤치마크 실행 스크립트, 시뮬레이션 데이터 등 코드 일체는 Github 공개
- 데이터 품질·신호·리스크·상태정의·안전 경계 등에 대한 피드백과 참여를 적극 요청
- Agent 신뢰성, 데이터 품질 자동평가, 인시던트 자동화 등 관련 개발자 커뮤니티와의 협업/확장을 강조
