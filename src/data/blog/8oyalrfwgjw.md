## *대규모 코드베이스를 위한 재귀 언어 모델(RLM): Superagentic AI의 샤시 발표* 핵심 요약

- 이 영상은 MIT 등에서 발표된 RLM(Recursive Language Model, 재귀 언어 모델) 개념을 소개하며, 대규모 코드베이스에서의 활용 방안과 실제 구현 사례를 구체적으로 다룸
- 기존 코딩 에이전트가 작은 저장소에서는 잘 작동하지만, 대규모, 모노 리포(mono repo)에서는 ‘컨텍스트 관리’ 문제가 발생하여 성능이 저하됨을 지적
- 일반적 해결책(검색 도구 활용, 시맨틱/로컬 서치, 요약 및 메모리 저장)이 갖는 한계를 언급하고, RLM은 전용 실행 환경을 통한 ‘외부화된 컨텍스트 관리’로 이를 타개함
- RLM의 핵심은 코드베이스 전체를 데이터로 보고, 모델이 직접 코드를 짜서 필요한 컨텍스트만 추출(슬라이스)해 LLM 맥락창에 전달하는 프로그래머블 접근임
- 리드 엔지니어가 방대한 코드베이스를 분석하듯 모델도 지속적으로 코드를 작성하고, 필요시 LLM 질의를 통해 외부 지식도 재귀적으로 확보
- Superagentic AI에서 개발한 오픈소스 구현체 ‘RLM code’를 안내하며, MIT 원 논문과 코드 모두 실습 및 확장이 가능함을 강조
- 실제 실습(데모)에서는 도커 샌드박스에서 RLM code가 어떻게 루프를 돌며, 코드 작성, 맥락 추출, LLM 질의, 최종 답변 반환 과정을 거치는지 보여줌
- 실험적 CLI(커맨드라인)와 TUI(터미널 인터페이스) 하네스를 갖추고, 사용자가 독자적 실험 및 관찰이 용이함을 시연
- 관측 데이터, 세션, 토큰 사용량, 실행 경로 등 전 과정이 JSONL 등 형식으로 추적 및 분석 가능함을 보여줌
- RLM 개념은 루트코즈 분석, 신입 온보딩, 미지의 저장소 분석 등 실용적 AI 코딩 업무에서 활용할 수 있음을 강조하며, 최근 여러 상용/클라우드 에이전트 프레임워크에서도 활발히 채용되고 있음을 실사례로 소개

## 세부 요약 - 주제별 정리

### 기존 코딩 에이전트는 대규모 코드베이스에서 컨텍스트 관리 문제로 성능이 하락함

- 작은 코드 저장소, 패키지 수준에서는 코딩 에이전트의 자동화 능력이 뛰어남
- 하지만 모노 리포 등 대규모 코드베이스에서는 맥락(Context) 창이 크게 요구되어 에이전트의 성능이 떨어짐
- 코드를 모두 한 번에 LLM에 제공할 수 없어, 어떤 정보를 줄 것인지 선택하는 전략이 핵심
- 맥락 관련 문제는 저장소가 크고 복잡할수록 심각해짐

### 여러 해결책(검색, 시맨틱 서치, 요약, 메모리 저장)이 있으나 한계가 분명함

- grep 등 파일 검색 도구를 통한 소스 탐색이 대표적인 기존 접근
- 시맨틱(의미론적) 서치 및 로컬 서치 방식도 도입됨: 의미적 유사성을 기반으로 컨텍스트 추출
- 맥락을 압축하거나 요약해서 넣는 방법, 특정 메모리 솔루션(캐시 저장)도 현업에서 사용됨
- 그러나 이들 모두 ‘맥락 창 한계’, ‘정확도 저하’ 등 한계를 극복하지 못함

### RLM은 컨텍스트 관리를 프로그래머블 환경으로 외부화해서 문제를 해결함

- RLM 논문의 핵심: 컨텍스트 관리를 LLM 내부가 아니라 ‘프로그래머블 실행 환경’으로 외부화
- 전체 코드베이스를 ‘데이터’ 취급, 별도의 환경(예: REPL, 샌드박스) 내에서 컨텍스트를 직접 추출
- 모델이 코드를 작성해서(예: 검색, 슬라이스, 변형) 컨텍스트를 동적으로 생성해 입력창에 전달
- 컨텍스트 선별이 동적이며, 필요할 때마다 코드를 짜고, 답변·정보가 부족할 땐 재귀적으로 외부 LLM에 질의 수행

### 선임 개발자의 대규모 코드 분석 방식과 RLM의 프로세스가 유사함

- 신임(혹은 선임) 개발자가 대규모 신규 코드베이스에 투입됐을 때의 과정을 비유함
- 전체 코드를 상세히 읽지 않고, 구조나 의존성, 주요 문서, 필요한 첨언 등을 노트(메모)로 정리
- 미지 부분은 또 다른 엔지니어나 ‘전문가’에게 문의(LLM 질의)
- 이 흐름과 유사하게 RLM도 코드 분석·노트·외부 질의·최종 정리(노트)를 반복

### RLM의 재귀적 루핑 구조는 LLM 질의와 코드 실행/관측의 복합 루프임

- REPL 환경에서 코드 작성 → 관측(Observation) → LLM 질의(필요정보 재귀적 요청)의 루프 진행
- 루프는 필요한 컨텍스트가 충족될 때까지 반복, 최종적으로 ‘깨끗한 결과(노트/분석)’ 산출
- 루프 종료 조건(필요 정보 확보)까지 RLM이 계속해서 재귀적 질문 및 코딩 수행

### 코드베이스 맥락(텍스트 외에 구조, 의존성 등) 특성 상 RLM에 최적임

- 코드베이스는 단순한 ‘글’이 아닌, 디렉토리 구조, 테스트, 설정 파일, import 등 복합 데이터임
- LLM이 단순 텍스트만 이해하도록 하는 것보다, 코드 전체를 이해·추론할 수 있도록 외부화 환경 접근 필요
- 그러한 특성 때문에 대규모 코드베이스를 RLM 시나리오 사례로 선정

### Superagentic AI가 만든 ‘RLM code’ 오픈소스는 개념 실습 및 확장에 용이함

- MIT 원 논문(RLM) 기반 레퍼런스 구현체인 ‘RLM code’를 Superagentic AI에서 개발·공개
- Playground(연구 실험장)와 문서, 깃헙 저장소 제공(오픈소스)
- 구현 예시로 MIT 공식 ‘RLM’과 ‘RLM minimal’ 저장소, Omar(오마르)가 개발한 DSpy 프레임워크 내 RLM 참고 안내
- ‘RLM’은 패턴이자 컨셉, 구체 구현은 누구나 자유롭게 가능

### RLM code는 여러 모델/플랫폼과 결합 및 관찰 통합이 자유로움

- RLM code는 로컬 모델, 클라우드 모델(Gemini 등) 모두 지원
- 자신의 관찰(Observability) 프레임워크와 자유롭게 결합 가능
- 패턴 개방형 구조로 다양한 프레임워크(Google ad, padenti 등)에서도 RLM 패턴 적용 가능

### 실습 데모: 저장소의 코드 분석, 맥락 추출, LLM 질의, 최종 대답까지의 실제 흐름을 시연함

- 도커 샌드박스 환경 내에서 RLM code의 실행 전체 과정을 데모로 시연
- 저장소 체크아웃, 에디터 내 파일 확인, 데모 타겟(실제 코드, readme 등) 제공
- 커맨드라인에서 명령어 입력, 실행 로그 확인, 토큰 사용량, 세부 트레이스, 툴 호출 횟수 등 실시간 모니터링
- 실험적 CLI와 TUI(터미널용 인터페이스) 하네스에서도 동일 개념 실험 가능

### CLI, TUI, 관찰 가능성 등 RLM code의 활용성과 실시간 추적성이 뛰어남

- Gemini 모델과의 연결, doctor 명령을 통한 상태 체크, budget 옵션 통한 비용 관리 등 기능 제공
- 실행 시 재귀 깊이, 스텝, 보상 등 전체 실행 궤적 및 이벤트 시간 상세 추적
- 모든 과정은 JSONL 등 머신리더블 형식으로 남아, 외부 관찰 솔루션과 연결 쉽게 추적 가능

### 실제 AI 엔지니어링 업무에서 RLM 개념은 온보딩, 루트코즈 분석 등 다방면에 적용이 가능함

- 대규모 저장소 온보딩, 원인 분석 등 실무 프로세스에 RLM 하네스를 직접 설계 적용 가능
- 전체 계획, 코딩, 관측, 서브콜, 예산, 최종출력의 전체 궤적 설계가 핵심
- 최근 다수 상용/클라우드 에이전트(코덱스 하네스, Gemini managed agent 등)는 실제로 RLM 개념을 내부적으로 적극 도입 중
- 예: 코덱스 하네스가 파이썬 REPL에서 컨텍스트 수집용 코드를 작성하는 등 RLM이 내장되어 있음
- Anthropic의 클라우드 코드 엔지니어 등도 X(구 트위터)에서 RLM 개념을 적용했음을 밝혔다고 언급
- 여러 에이전트가 서로 독립 샌드박스에서 재귀적으로 동작, 결과 취합하는 다이내믹 워크플로우 역시 RLM 확장 구조임
