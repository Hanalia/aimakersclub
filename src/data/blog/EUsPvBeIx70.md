## *LLM이 50만 개 센서 이름에서 혼란을 겪으며 드러난 시맨틱 블라인드니스 문제* 핵심 요약

- 본 영상은 Phaidra의 AI 연구 엔지니어 라훌 싱(Raahul Singh)과 밴치 레브스틱(Vanč Levstik)이 대형 데이터센터(1GW급)에서 50만 개가 넘는 센서 이름을 LLM에 제공했을 때 발생한 "시맨틱 블라인드니스(semantic blindness)" 현상을 중심으로 실제 문제와 해법을 논의함
- 데이터센터의 각 고객마다 일관되지 않은 장비 네이밍 규칙을 사용하며, 장비 수가 기하급수적으로 늘수록 LLM 기반 솔루션의 한계와 오류가 두드러짐
- LLM은 장비명을 문맥 창에 모두 입력할 때, 빈번한 토큰 반복 및 유사 이름 탓에 페널티 및 응답 중단 등 심각한 처리 오류를 일으킴 (예: 40만 GPU 이상일 때 정확도 30% 수준으로 감소)
- 단순 분할(샤딩) 및 벡터 임베딩 기반 검색(RAG)도 유사 이름 간 분간 불가, 팬텀(존재하지 않는) 장비 발생, 실제 장비 누락 등 현실적 시스템에 적용시 치명적 문제를 초래함
- 새로운 해법은 공장/데이터센터의 계층적 구조(트리 구조)를 활용해, 트리의 깊이에 따라 요약 정보만 LLM에 입력하고, 검색·필터링·집합 연산 등은 코드 및 결정적 방식(Software 1.0)으로 처리하도록 설계함
- LLM은 검색이 아닌 플래닝(계획)과 애매한 질의 해석에만 집중, 최종 집합 산출은 전용 코드(프리인덱스 트리 등)로 실행해 대규모 시스템에서도 100% 정확도와 일정한 연산 비용을 달성함
- 실제 시험에서 기존 방식은 64GPU 대비 46만GPU에서 정확도가 80%→30%로 급락, 토큰 사용량은 1억 1,600만 개까지 폭증한 반면, 새 방식은 토큰 39만 개 내외, 정확도 100%로 유지됨
- 결론적으로 대규모, 계층적, 반복 패턴 구조의 산업 데이터 처리에는 LLM+결정적 코드(Software 3.0+1.0) 융합이 필수임을 강조

## 세부 요약 - 주제별 정리

### LLM 기반 설계가 50만 개 센서 이름 앞에서 실패하며 시맨틱 블라인드니스를 드러냄

- Phaidra팀은 AI 공장(데이터센터 등)의 운용 지원을 위해 LLM을 활용하여 장비/센서 데이터를 질의·분석하는 시스템을 개발함
- 장비 이름이 고객마다 제각각이고 명확하지 않아, 예시로 "GPU_rack_1_datahall4"부터 "CH3X7" 같은 해석 불가 코드까지 혼재
- 데모 시스템에서는 소규모 데이터 및 단일 목적이어서 LLM이 이름을 문제없이 처리하지만, 1GW급(40만~50만 GPU 이상) 규모에서는 LLM 컨텍스트 한계 및 처리 오류가 심각하게 드러남
- 예) “Data hall 1의 GPU 리스트” 질의에 대해 LLM이 실제 장비 중 누락/추가/엉뚱한 조합을 반환함
- 시스템 요구상, 단 하나의 장비라도 누락·허위 반환하면 신뢰가 즉시 무너지는 치명적 리스크가 존재

### 벡터 서치·임베딩 기반 접근은 유사 네이밍 구조에서 검색 실패를 반복함

- 벡터 임베딩 구조나 RAG(Retrieval Augmented Generation) 방식도 실무 적용에 실패
- 장비 이름이 매우 근접(“Chiller 6” vs “Chiller 7”)하거나 코드 명명(“CDO11” vs “CDO12”)이기 때문에 벡터 거리 기준으로는 분류가 불가
- LLM은 동일 토큰 반복 시 ‘빈도 페널티(frequency penalty)’로 인해 출력을 중단(guardrail 발동)하거나 반복 출력을 거부함
- 예) “i7 데이터홀의 GPU 이름 100개 나열 질의” 시, LLM이 중별 없이 출력을 멈춤
- 따라서 대규모 장비 리스트의 이름 검색에는 벡터 기반 서치·LLM 단독 전략 모두 실무에서 실패

### “분할 정복”과 같은 나이브한 해법들이 실질적 오류와 환각 문제를 유발함

- 샤딩(sharding·분할)하여 LLM 여러 개를 병렬로 돌리는 방식도 테스트
- 현실에서는 recall(누락 없이 모두 탐색) 성능이 매우 저조, LLM이 존재하지 않는 장비를 “환각(phantom equipment)”으로 생성하기도 함
- 일부 실제 장비는 응답에서 원인 없이 누락됨
- 이런 누락/환각은 미션크리티컬(생산·운영) 시스템에서 결정적 결함이 되며, 연쇄적 시스템 장애 유발 가능성이 높음

### 계층적 트리 구조에서 트리의 깊이만큼 요약하면 LLM 컨텍스트는 일정하게 유지됨을 발견함

- AI 공장 구조는 트리형 계층(데이터센터 > 데이터홀 > 통로 > 랙 > GPU 등)으로 자연스러운 구조화가 가능
- 트리 깊이는 비교적 천천히 증가(예: 4~5단계)하지만, 각 레벨에서 지엽적으로 수많은 장비가 추가됨
- 실질적 요약 정보 입력 시, 장비 수가 64개이든 46만개이든 요약 컨텍스트의 크기는 거의 동일
- LLM에 트리 각 경로만 입력하면 전체 망 구조 이해 및 사용자 질의 해석이 문제없이 가능

### LLM은 플래닝에는 강하지만 대규모 검색·집합 연산에는 부적합함을 실험으로 확인함

- LLM에 검색이 아닌 “플래닝(계획)”만 맡기는 것이 성능과 정확도 향상에 키 역할
- 예) “데이터홀 11에서 온도가 높은 GPU 모두 알려줘” 질의 → LLM이 검색 직접 수행이 아니라 “데이터홀 11의 GPU 중 온도 조건 필터” 형태의 구조적 플랜 반환
- 이후 집합 연산(데이터홀/장비 필터, 교집합 등)은 결정적 코딩으로 처리
- 덕분에 모호한 사용자 질의에도 최종 결과 집합 생성이 정확함

### 집합 연산·프리인덱스 트리 등 결정적 코딩으로 필터링·검색을 대폭 최적화함

- 백엔드에서 “프리 인덱스 트리(pre-indexed tree)”와 집합(set) 연산에 기반하여 장비 하위 집합 추출 및 중복 제거 처리
- 예) 데이터홀 11의 모든 GPU 리스트와 온도 기준 충족 장비 리스트의 교집합 산출
- 집합 연산 기반이므로 recall 100% 보장 및 LLM 표현 한계와 무관

### 이름 패턴 검색 시 LLM이 전체 이름을 읽지 않아도 패턴 추출로 비용을 상수로 고정함

- 질의가 애매하거나 패턴 기반 필요 시 LLM에 “이름 패턴”만 추출시켜 코드단에서 패턴 일치 탐색 수행
- LLM이 전체 장비명 리스트를 입력받을 필요가 없으므로, 처리 비용은 장비 수 무관(거의 상수)
- 대규모 장비 확장에도 시스템 처리비용/시간이 선형 증가하지 않고 일정하게 운영 가능

### 전체 처리 과정은 질의→플래너 LLM→결정적 결과 산출로 2~3단계로 짧게 최적화됨

- 사용자 질의(예: “특정 그룹 장비 상태”)가 입력되면, 플래너 LLM이 검색 플랜/패턴 생성
- 이후 결정적(set 연산 등) 코드가 해당 패턴 플랜에 따라 최종 장비 리스트 산출
- Agent Loop 없이 단 2~3단계로 처리, 일관된 처리 시간과 비용 유지

### 실제 운영 환경 평가에서 새 시스템이 정확도·비용 모두 과거 전략을 압도적으로 능가함

- 동일 LLM·동일 데이터에서 구旧 방식 대비 일관 테스트(1건당 3회) 실시
- 기존 방법: 64GPU→정확도 80%에서, 46만 GPU로 확장시 정확도 30%로 급락, 토큰 소모 1억1,600만 개
- 새 방법: 64GPU, 46만GPU 모두 100% 정확도, 토큰 소모 단 39만 개 수준으로 비용 300배 이상 절감
- 실제 6개 프러덕션 센터, 66개 케이스 테스트에서 단 한 건의 오류도 발생하지 않음

### 대규모 데이터 구조에서는 플렉서블 LLM(Software 3.0)과 결정적 코드(Software 1.0)의 병용이 핵심임을 결론지음

- 카르파티(Andrej Karpathy)의 “Software 1.0(결정적 코드) vs 3.0(LLM 기반 프롬프트)” 개념 인용
- 1.0은 예측 가능성·재현성 강점, 3.0은 유연성·미지의 질의 처리 강점
- 대규모·계층적 구조에서는 LLM이 직접 탐색·집합연산을 맡기보다, 구조화·룰 기반 처리를 반드시 코드(1.0)에 이양
- 결론: 구축 초기엔 3.0(LMM) 중심, 실제 운영에선 분석/검색/집합 연산 등은 점진적으로 1.0(코드)으로 옮기며 신뢰성과 유연성, 비용 효율성 모두 확보
- “모델이 판단해야 하는 부분만 LLM에 남기고, 나머지는 반드시 코드로 처리하라”는 실무적 교훈 정리
