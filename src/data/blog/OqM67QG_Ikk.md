## *fork()에서 Fleet까지: 에이전트 샌드박스 클라우드 설계* 핵심 요약

- OpenAI의 RL(강화학습) 및 에이전트 인프라팀 소속 Abhishek Bhardwaj가 ‘에이전트 샌드박스 클라우드’의 필요성 및 설계 원리에 대해 설명
- 챗GPT와 같은 대형 언어 모델이 기존의 데이터로만은 해결할 수 없는 수학·코드 문제에 대해 코드 실행(tool calling) 능력이 필요함을 밝힘
- 코드 및 도구를 실행하는 환경에서 보안과 신뢰성, 대규모 확장성이 핵심 과제로 제시됨
- 가장 단순한 방법인 fork-exec 방식은 성능은 뛰어나지만, 커널 침해 등 보안 취약성과 "noisy neighbor" 이슈(자원 남용) 발생
- 컨테이너(Docker 등)는 네임스페이스와 cgroups로 최소한의 격리 및 자원 제어 제공하지만 커널 취약점 공격은 여전히 가능
- gVisor(어플리케이션 커널)는 사용자 공간에서 시스템 콜을 구현해 보안 수준을 높이지만, 체인된 이중 공격 가능성이 남아있음
- 하드웨어 가상화 기반 microVM(QEMU, crosvm, Firecracker, Cloud Hypervisor 등)은 하드웨어 레벨에서 호스트 격리를 강하게 보장함
- microVM의 도입으로 강력한 보안, 신속한 부팅, 낮은 오버헤드, Rust 기반 메모리 보안, 디바이스 수준 추가 격리가 가능해짐
- 디스크 스냅샷 및 영속적 저장 방식 도입으로 작업 중인 샌드박스의 상태 보존·복원, 장기 작업, 장애 복구 및 탐색적 학습이 실현됨
- 여러 노드·지역에 걸친 오케스트레이션은 스냅샷 레이어 최적 분배, 워밍풀 및 메모리 스냅샷 활용 등으로 신뢰성과 초저지연 성과에 기여함

## 세부 요약 - 주제별 정리

### 챗GPT류 대규모 언어 모델이 verifiable reward 문제를 잘 해결하려면 코드 실행 능력이 필요함

- 챗GPT는 인터넷 전체 데이터로 사전 학습돼 간단한 질의(예: 3+3=6)에 정답을 잘 냄
- 그러나 "딸기에 몇 시간이 있나요?" 같은 흔치 않은 질문엔 실패하기도 함
- 수학·코드 등 정답 검증이 가능한 문제(verifiable reward)는 단순 데이터만으로는 한계
- 모델이 “코드 실행” 기능을 가질 때, 코드와 수학 문제의 정확도가 크게 향상됨
- 학습 과정에서 모델이 코드 실행을 요청하면, 루프가 응답을 평가(정·오답)해 가중치를 변경
- 이로써 모델은 필요할 때 코드 실행이나 도구 호출을 스스로 학습하게 됨

### 제품에서 샌드박스를 통한 코드 실행은 보안 및 안전한 격리가 불가결함

- 훈련 단계와 달리 제품 단계(예: ChatGPT, Code Interpreter)에서는 별도의 함수 루프 없이 코드 실행
- 사용자의 노트북 또는 클라우드 노드에서 실행할 때 악의적 또는 우발적 공격(루트 권한 취득, 커널 취약점 악용 등)에 노출
- 각 샌드박스는 다른 사용자의 데이터에 접근하면 안 됨(데이터 탈취 방지 필요)
- 따라서, “신뢰할 수 없는 코드”를 격리된 환경에서 실행해야 하며, 이는 샌드박스의 핵심 목적임

### 연구와 제품에서 샌드박스의 필요 조건 및 우선순위가 상이함

- 연구에서는 다수의 훈련 루프 실행, 높은 처리량(throughput)이 중요
- 제품에서는 빠른 응답(낮은 레이턴시)과 안정성(reliability), 사용자의 이탈 방지, GPU 토큰의 효율적 사용이 핵심
- 양쪽 모두에 신뢰성과 보안이 필수이며, 인프라가 무너지면 전체 시스템에 심각한 손실

### 리눅스의 전통적 fork-exec 방식은 보안과 확장성에서 심각한 한계를 가짐

- 가장 단순한 도구 실행 모델: API 서버가 tool call마다 fork, exec로 프로세스 생성·실행
- 강점: “네이티브 성능”(fastest)
- 문제: 프로세스가 커널에 직접 접근 → 커널 공격 가능(루트 획득, 커널 익스플로잇 등)
- 루프나 fork 폭탄(while loop로 무한 fork)으로 “bad neighbor” 문제(노드 자원 고갈)
- 보안·안정성에 치명적이며 대규모 서비스엔 적합하지 않음

### 컨테이너(네임스페이스, cgroups)는 일부 자원 격리 제공하나 커널 공격엔 취약함

- 리눅스 컨테이너(Docker 등)는 네임스페이스로 자원(프로세스, 파일시스템, 네트워크 등) 격리
- cgroups로 CPU, 메모리 등 자원 한계 설정 가능하여 noisy neighbor 문제 완화
- 실질적으로 컨테이너 내부 프로세스도 호스트 커널을 직접 사용하는 구조
- seccomp 등으로 호출 가능한 시스템 콜 필터링 가능, 공격면 감소 가능성 있음
- 단, 어떤 콜을 필요로 할지 예측 어렵고 기능 제한·학습 루프에 비효율 야기
- 커널 공격/루트 권한 탈취 가능성 기본적으로 남아있어 보안이 미흡

### gVisor는 사용자 공간 커널로 공격체인을 어렵게 하지만 탈출 가능성은 여전함

- gVisor는 유저모드 커널(Go로 작성된 Sentry)이 호스트 커널 API를 에뮬레이트
- 시스템 콜은 gVisor “Sentry”가 먼저 처리, 실제 커널로 직접 접근 차단
- 호스트 커널까지 탈출하려면 Sentry→Gopher(파일시스템 데몬)→커널로 2단계 체인 공격 필요
- 보안성이 좀 더 올라가지만 여전히 최종적으로 커널 탈취 가능성 존재
- 공격 난이도는 증가하지만 대형 모델에겐 잠재적 위험 요소 남음

### 하드웨어 가상화 기반 microVM은 최상의 격리 보안성과 확장성을 제공함

- 리눅스 KVM 등 하드웨어 가상화는 CPU 수준에서 완전 분리된 게스트/호스트 모드 제공
- 게스트 VM(커널·유저 모두 ring 0/3)은 자체적 루트 권한·커널 익스플로잇 시도 가능하나, VMX 비루트 모드라 호스트에는 영향 없음
- 호스트·하이퍼바이저는 ring 0 ‘루트’ 모드, 호스트 공격 불가(게스트 루트 탈취 무의미)
- 성능 오버헤드 있음(CPU 컨텍스트, 시스템 콜 등에서 슬로우)
- Rust 기반 MicroVM(crosvm, Firecracker, Cloud Hypervisor 등)은 QEMU 대비 빠른 부팅, 작은 메모리, 메모리 안전성을 제공
- 각 디바이스도 독립·격리, 특정 디바이스만 권한 제한적으로 허용(공격 표면 최소화)

### MicroVM이 도입되면서 빠른 생성, 메모리 안전성, 간편한 보안 확장성이 가능해짐

- QEMU는 수많은 아키텍처·디바이스 지원(부피 크고 공격 표면 넓음)
- crosvm(구글), Firecracker(아마존 Lambda·서버리스에서 사용), Cloud Hypervisor(여러사 공통)
- MicroVM 자체는 “마이크로”로, 게스트 내부가 아니라 VMM의 경량화·빠른 부팅(수 밀리/초)·메모리 절감 때문
- 각 MicroVM은 API와 Unix Domain Socket 통해 프로그래밍적으로 제어 가능
- VSOC, IP 스택 등 다양한 게스트-호스트 통신 방식 지원

### MicroVM 기반에도 여전히 일부 트레이드오프와 한계가 존재함

- 보안: 커널 공격 코드가 KVM 및 장치 계층까지 체인 익스플로잇 해야 공격 성공, 공격 난이도 급상승(그러나 불가능하지는 않음)
- 성능: 게스트-호스트 컨텍스트 전환, 메모리 reclaim 등에서 오버헤드 존재
- 메모리 공유: balloon driver 등 필요, 재할당/회수가 즉각적이지 않음
- GPU 접근: 고수준 그래픽만 지원(vertio-GPU), 메탈(GPU 직접) 접근은 1개 샌드박스에만 독점(VFIO)
- 시스템 트릭으로 성능 보완 가능하지만, 보안 침해는 시스템 신뢰성을 한순간에 무너뜨림

### 영속적 디스크 스냅샷은 샌드박스 유지, 복구, 장기 탐색·학습에 결정적 역할을 함

- 기존엔 에이전트에게 디스크 없는 컴퓨터만 제공(실행 종료·노드 다운시 상태 소멸)
- 점점 복잡해진 작업(예: 프레젠테이션, GitHub 저장소 생성 등)과 장기 실행 필요성 증가
- 노드/클러스터 장애 시 스냅샷·복구 통해 데이터 손실 방지 및 비용 절감
- Checkpointing(체크포인트)은 장기·병렬 탐색, 롤아웃, 백트랙, 몬테카를로식 연구, 업무 복원 등에 필수

### 디스크 체크포인트와 복구는 효율성과 확장성, 대규모 서비스의 기반이 됨

- 인크리멘탈 스냅샷: 이전 스냅샷 대비 변경분만 저장(데이터 용량·속도 최적)
- Snapshot API의 저지연, 신속한 create/restore가 서비스 품질 좌우
- 언제나 저장(always-on) 방식/명시적 저장(explicit-save) 방식 모두 함수 API로 구현 가능
- 파일 시스템 레벨/블록 레벨 중 실제 객체 스토리지(S3, GCS 등)와 연동되는 블록 기반 방식을 선호
- Copy-on-write, XFS 등으로 제로카피·블록 변경 추적, fiemap 사용, background 업로드로 저지연 실현
- Tiered cache(클러스터내 캐시 → 오브젝트 스토리지 백업) 구조로 대용량/고성능 구현

### 샌드박스 상태 스냅샷을 활용한 분산 오케스트레이션이 신뢰성·지연 최소화에 기여함

- 하나의 노드만 쓰면 장애 시 전체 소멸하므로, 분산 클러스터/지역 배포 필요
- 클러스터별 노드를 자동 스케줄링하며, 최적 지역·로드 기반 할당
- 생성 시간 단축: 미리 MicroVM을 프리웜(warm pool)하거나, 메모리 스냅샷 활용하여 즉시(Millisecond) 생성
- 워밍풀은 자원 낭비 위험, 메모리 스냅샷·하이브리드 방식 병행해 최적화
- 스냅샷 레이어별 최적 노드로 분배(이미 캐시된 레이어 많은 곳 선택)해 복구 성능·신뢰성 극대화

### 결론: 진보적 샌드박스 인프라 설계가 미래의 대규모 에이전트 서비스의 핵심이 됨

- 샌드박스는 연구·제품 모두에서 확장성, 보안, 신뢰성, 사용자 경험 보장을 위한 필수 인프라
- 성급한 컨테이너/초기 해법보다 곧바로 MicroVM 중심 아키텍처 도입 권장
- 첨단 스냅샷, 디스크 영속, 오케스트레이션 조합이 대규모 AI 에이전트 환경의 핵심 기반임
