## *에이전트 하니스가 바이브 코딩을 부활시키는가?* 핵심 요약

- **GPT와 같은 대형 언어 모델(LLM)의 원초적 성능 증가는 최근 들어 둔화되고 있음**
- **Gemini 3, Claude Opus 4.5 등 벤치마크 성능은 지속적으로 상승 중이나, 눈에 띄는 혁신은 모델 자체보다는 그 ‘위의 레이어’에서 일어남**
- **이러한 LLM 위에 쌓이는 새로운 계층(에이전트 하니스 등) 개발이 향후 발전의 핵심으로 부상**
- **특히 에이전트 하니스와 같은 래퍼(wrapper) 및 월드 모델(world model)이 차세대 혁신의 기회임**
- **개발자들은 LLM 자체를 발전시키는 대신, 이 위에 커스터마이즈 가능한 하니스 및 시스템을 구축해 직접 최적화할 수 있음**
- **이런 하니스 최적화 덕에 개발자는 본격적으로 코딩의 99%를 코딩 에이전트에게 위임할 수 있을 것으로 기대**
- **장기 실행 에이전트(long-running agent) 및 다양한 영역에 적용 가능성 언급**
- **‘바이브 코딩(vibe coding)’ 개념이 재조명되겠지만, 미래에는 사람이 코드를 직접 작성하는 비중이 급격히 감소할 것**
- **2026년에는 시스템과 하니스 설계에 집중하고, 실질적인 코드 대부분은 에이전트가 작성하는 환경을 예상**

## 세부 요약 - 주제별 정리

### LLM의 원초적 성능 증가는 더 이상 가파르지 않음

- 최근 몇 년간 대형 언어 모델(LLM)의 성능이 폭발적으로 향상됐으나, 이제는 그 속도가 현저히 완만해짐
- 기존에는 LLM 자체의 강력한 성능 향상이 주목받았으나, 이는 이제 ‘뉴 노멀’이 되어 더 이상 극적인 변화가 드러나지 않음
- "There's this harsh truth... the raw power of LLMs is just simply not exploding anymore like it was the last couple of years"

### 최근 LLM 벤치마크(예: Gemini 3, Claude Opus 4.5)는 꾸준히 향상 중이나, 혁신의 무게추가 이동함

- Gemini 3, Claude Opus 4.5 등 주요 신형 LLM이 최근 출시되어 벤치마크 성적을 올리고 있음
- 하지만 이 모델들의 등장도 이전 세대만큼의 충격적 혁신을 가져오지는 못함
- 혁신의 중심이 모델 자체보다, 그 위에 쌓이는 추가 계층 및 래퍼 쪽으로 옮겨감

### LLM 위에 구축되는 ‘레이어’가 지속적인 혁신의 동력이 되고 있음

- 본질적으로 LLM을 둘러싼 구조(레이어), 즉 개발자가 직접 구축할 수 있는 래퍼(wrapper)가 중요한 경쟁력으로 부상
- 개발자나 사용자는 이제 LLM 자체가 아닌 그 위에 쌓는 기능성과 프레임워크에 집중
- 예시로 월드 모델(world model) 및 영상 주제인 에이전트 하니스(agent harness) 등이 언급됨
- 이 레이어의 구축과 최적화가 진정한 차별화 지점이 되고 있음

### 개발자는 LLM 위의 하니스와 시스템을 최적화하는 역할로 전환

- 기존처럼 모델을 직접 미세 조정하기보다, 그 위에 작동하는 하니스와 시스템을 설계하는 것이 개발자의 주 역할로 이동
- "this layer on top of LLMs is what we get to create and optimize for ourselves"
- 하니스 개발을 통해 각자의 용도에 맞춰 LLM 기반 에이전트의 퍼포먼스를 극대화할 수 있음

### 에이전트 하니스가 차세대 코드 생산 자동화의 핵심 열쇠가 됨

- 영상의 중심 주제는 ‘에이전트 하니스’(agent harnesses)임
- 이 하니스(래퍼)를 통해 LLM 기반 에이전트가 훨씬 더 복잡하고 장기적인 작업을 관리할 수 있게 됨
- “This is the next unlock for the next level of capability that we have for our agents”
- 하니스 구조를 개선함으로써 시스템 전체의 ‘잠금 해제’(unlock)를 실현하고 새로운 가능성을 열어둠

### 하니스 개발은 거의 모든 형태의 장기 실행 에이전트로 확장 가능

- 하니스-기반 접근법은 코딩 에이전트뿐 아니라, 이미 언급한 다양한 ‘장기 실행 에이전트’에도 적용 가능
- 예를 들어, 데이터 처리, 유지보수, 대화형 서비스와 같이 지속적으로 운영되는 AI 시스템에 효과적으로 활용될 수 있음
- “applying this to really any kind of long-running agent like I’ve been talking about as well”

### 바이브 코딩이 부활하지만, 실질적인 코드는 에이전트가 작성하는 시대로 변모함

- 바이브 코딩(Vibe Coding)은 과거 개발자가 코드를 ‘느낌으로’ 빠르게 작성하던 문화를 지칭
- 미래에는 이러한 문화가 부활하는 동시에, 코딩의 큰 부분을 에이전트가 자동으로 담당하게 됨
- "Vibe coding, it's going to sort of become viable, but also look a lot different because we are engineering the system, engineering the harness, but we aren't going to be writing most of the code in the very near future"
- 코드를 엔지니어링하는 방식이 시스템과 하니스 설계 중심으로 바뀌며, ‘코드 자체’보다 ‘메타 코딩’으로 역할이 이동

### 2026년에는 시스템 설계 중심, 코딩은 에이전트에게 99% 이상 위임되는 미래를 예상함

- 영상 제작자는 2026년을 기준으로 시스템/하니스 설계자와 에이전트(코드 자동화 도구) 간 역할이 명확히 분화될 것으로 전망
- "we aren't going to be writing most of the code in the very near future"
- 실제 코드의 대다수(“99% or more”)는 사람의 손을 거치지 않고, 자동화된 에이전트에 의해 생산될 것임을 시사
- 여전히 인간의 창의력과 시스템 아키텍처 최적화가 중요하지만, 코딩 생산업무는 에이전트에 대폭 이전됨
