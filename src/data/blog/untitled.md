## *포스트맨의 Git 네이티브 지원이 API 구축 방식을 혁신한다* 핵심 요약

- API 작업은 기존에는 여러 툴을 오가며 번거롭고 분산된 워크플로우가 필수였음
- 특히 AI 에이전트를 개발할 때, API 엔드포인트, 스펙, 테스트, 컬렉션 등이 각각 다른 위치와 툴에 관리되어 어려움이 많았음
- 코드와 API 스펙, 컬렉션, 테스트 등이 한 곳에 통합·버전관리되지 않아 생산성 저해 요인으로 작용함
- 포스트맨(Postman)은 최신 업데이트를 통해 ‘AI 네이티브 API 플랫폼’으로 변모함
- 이 업데이트로 포스트맨은 ‘Git 네이티브’ 도구가 되어 모든 API 리소스를 코드베이스 내에서 직접 YAML 파일로 유지·관리할 수 있게 됨
- 로컬 환경과 클라우드 환경 간 2-way 동기화가 지원되고, 변경사항이 git diff로 명확히 추적됨
- API 스펙, 테스트, 목(mock), 컬렉션 등이 코드 수준에서 버전관리돼 개발 및 협업 효율이 크게 향상됨
- 평문 영어로 요구만 하면, 코드 분석·라우트 추출·스펙/테스트/목 생성 등 API 라이프사이클 전체를 AI 기반으로 자동화함
- AI 모델(Claude, GBT, 자체 호스팅 등), 다양한 외부 툴(GitHub, Sentry, Linear 등)과도 연동되어 폭넓게 활용 가능함
- 동일한 테스트 컬렉션이 로컬, CI 파이프라인, GitHub Actions 등 어디서든 재사용 가능해 환경 간 중복 노력이 사라짐

## 세부 요약 - 주제별 정리

### 기존 API 개발은 코드·테스트·문서가 분리되어 비효율적 문제를 야기함

- 기존에는 API 엔드포인트를 IDE나 코드 에디터에서 직접 생성해야 했음
- 컬렉션(collections) 및 테스트(test)는 별도 도구에서 추가로 관리해야 했고, 이는 추가적인 전환 비용을 발생시킴
- CI(지속적 통합) 환경에서 테스트를 동작시키려면 별도의 설정과 호환성 문제를 신경써야 했음
- API 스펙(specs), 컬렉션, 코드가 각각 다른 위치에 존재해 관리와 버전 컨트롤이 일원화되지 않음
- 이런 분산은 협업 시 혼선을 유발하며 버그 추적이 어렵고, 변경 이력을 한눈에 파악하기 힘들게 만듦

### 포스트맨의 최신 업데이트는 ‘AI 네이티브’, ‘Git 네이티브’ API 개발 환경을 제공함

- 포스트맨은 최근 대대적인 업데이트를 적용하여 AI·Git 네이티브 API 플랫폼으로 발전함
- API 작업의 전 과정이 코드와 동일한 버전관리 흐름에 통합됨
- 개발자는 기존처럼 다양한 툴을 오가는 대신 한곳에서 모든 리소스를 관리 가능함
- ‘Git 네이티브’ 요소가 주요 혁신 포인트로 강조됨

### 모든 API 리소스를 YAML 파일로 코드베이스에 직접 관리할 수 있게 됨

- 컬렉션, 스펙, 목, 테스트 등 모든 포스트맨 리소스를 코드저장소에 YAML 파일 형태로 저장 가능
- YAML 파일 기반 구성이므로 가독성이 높고 유지관리가 용이함
- 로컬 코드를 직접 편집하거나, 포스트맨 데스크톱 앱에서도 동일하게 접근·작업할 수 있음
- 코드 및 API 리소스가 물리적으로 한 폴더에 집결되어 협업 및 백업, 배포가 편리해짐

### 로컬 작업과 포스트맨 클라우드 간 2-way 동기화로 실시간 협업이 가능함

- 로컬 환경에서 작업하던 내용을 클라우드와 실시간으로 동기화할 수 있음
- 한곳에서 변경 시 자동으로 양방향(2-way) 싱크, 버전 컨트롤, 동시 협업이 가능
- 실제 예시로 엔드포인트 테스트에 변경을 가하고, git diff로 바로 변경 내역을 확인 가능
- 개발자는 코드베이스처럼 명확하게 API 리소스의 변경이력을 추적할 수 있음

### API 라이프사이클의 모든 변경사항이 git 풀이 가능한 커밋 단위로 기록됨

- API 관련 작업이 모두 파일 단위로 디스크에 저장되어 git diff, git commit 등의 버전관리와 정밀하게 연동됨
- API 스펙, 테스트, 목, 컬렉션 등 각각의 변경사항이 코드 변경처럼 히스토리화됨
- 동일한 버전 관리 체계를 갖게 되어, 코드 리뷰 및 팀 협업 시 불일치나 충돌 문제를 효과적으로 예방 가능

### 평문 영어 입력만으로 AI가 전체 API 작업을 자동화해주는 ‘에이전트 모드’가 도입됨

- 개발자가 단지 “이 플라스크(Flask) 앱에 컬렉션을 생성해줘” 정도의 영어로 요구사항을 작성하면 됨
- 포스트맨 에이전트 모드가 코드베이스를 자동 스캔하여 라우트 정보, 엔드포인트 목록을 추출함
- 추출된 정보로 API 스펙, 자동화 테스트, 목(mock) 서버까지 자동으로 생성·구축함
- API 좌측 컬렉션 구조와 테스트 코드가 자동 동기화됨

### AI 모델 선택과 외부 도구 연동으로 다양한 기업 환경에 유연하게 통합됨

- Claude, GBT, 자체 호스팅 AI까지 선택적으로 연동 가능
- MCP 서버도 지원되어 기업 맞춤형 확장성이 높음
- GitHub, Sentry, Linear 등 실제 개발 생산성 도구와 컨텍스트 통합이 가능함
- 기존 사용하던 협업·이슈 관리·버그 추적 툴의 맥락을 반영한 API 자동화가 실현됨

### 개발 환경과 CI 파이프라인을 구분 없이 동일 테스트 리소스를 운영할 수 있게 됨

- 로컬 머신에서 사용하는 동일한 포스트맨 컬렉션, 테스트가 CI 환경(GitHub Actions 등)에서도 반복 사용할 수 있음
- pre-commit hook, 자동화 배포 파이프라인 등 다양한 단계에서 코드와 API 테스트가 일치
- 새로운 환경별로 코드를 수정하거나, 테스트를 재작성할 필요가 없음
- 모든 환경에서 일관된 품질 유지와 테스트 자동화가 용이해짐

### API 라이프사이클 전체가 파일 기반·AI 자동화로 ‘코드처럼’ 관리되는 패러다임 전환이 강조됨

- API 스펙, 컬렉션, 테스트 등 모든 구성요소가 이제는 단순한 폴더와 텍스트 파일(YAML 등)로 한데 관리됨
- 변경 및 확장이 git 커밋, diff 단위로 추적 및 협업 가능
- API 자동화 및 테스트가 영어 지시, 코드 분석만으로 구현되어 개발 생산성이 크게 향상됨
- Release, QA, 협업, 운영 자동화까지 API 구축의 전과정이 ‘코드 수준’ 관리 체계로 통합됨

### 포스트맨의 최신 변화는 다양한 플랫폼과 연계하여 누구나 쉽게 API 플랫폼을 구축하도록 지원함

- 포스트맨을 API 플랫폼으로서 더욱 강력하게 사용할 수 있게 됨을 강조
- 영상 설명에 포스트맨과 예시 Github 리포지토리 링크를 추가 제공함
- 실제 사용 경험을 바탕으로 ‘강력 추천’을 마지막에 언급함 (단, 구체적 설명 위주)
