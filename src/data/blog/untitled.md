## *모두가 놓치고 있는 AI 에이전트의 핵심 - 메모리와 확장 가능한 도구* 핵심 요약

- 대부분의 AI 에이전트와 MCP(Multi-Channel Platform) 서버는 한 명의 사용자만을 위해 구축되곤 함
- 대다수 개발자가 에이전트 설계 시 도구(툴) 인증 정보를 코드에 직접 하드코딩함으로써 한 개 계정만 연동이 가능해짐(Gmail, Slack, Asana 등)
- 여러 사용자가 각자의 서비스를 활용할 때, 에이전트가 사용자별 계정에 안전하게 액세스할 수 있도록 만드는 것이 중요한 과제임
- 영상은 Arcade라는 플랫폼이 이 문제를 OAuth 기반 인증 절차로 해결함을 시연
- Arcade는 에이전트가 사용자 계정 접근 권한을 동적으로 요청·획득할 수 있게 해줌
- 한 번 인증된 자격증명은 저장되어 이후 반복 요청 필요 없음, 보안(scope)도 준수함
- Arcade는 LangGraph와 함께 통합 작동하여 에이전트 인증·작업 흐름을 시각적으로 구현
- 실제 데모에서 Asana 계정 연동 및 사용자 승인 과정을 상세히 보여줌
- 이 과정을 통해 수천, 수백만 명 사용자가 각각 자신의 계정으로 동일한 에이전트를 활용할 수 있게 됨
- 장기 메모리 등과 결합 시 극도로 개인화된 대규모 확장형 AI 에이전트 구현 가능성을 강조함

## 세부 요약 - 주제별 정리

### 기존 AI 에이전트 대부분은 한 명의 사용자만 대상으로 설계되는 문제가 있음

- 대부분의 AI 에이전트 및 MCP 서버는 오직 하나의 사용자, 즉 개발자 자신의 계정만을 대상으로 작동하도록 만들어짐
- 이는 소규모 개인 실험·활용에는 문제가 없으나, 대규모 서비스로 성장하려 할 때 심각한 한계로 작용함
- 여러 사용자가 각각 자신의 계정(예: Gmail, Slack, Asana 등)을 사용하려면 근본적으로 설계가 달라져야 함을 강조

### 도구 액세스 권한을 하드코딩하는 기존 방식은 확장성에 취약함

- 일반적으로 에이전트의 도구 연동(툴 integration)은 코드를 직접 수정해 각 서비스의 인증 정보를 입력(하드코딩)하는 방식
- 이 방식에서는 한 번에 하나의 계정(Gmail, Slack, Asana 등)만 사용할 수 있어 공동 활용이 불가능함
- 새로운 사용자가 에이전트를 활용해 자신의 계정으로 작업하려면 인증정보 재설정 등 복잡한 처리가 필요

### 사용자별 계정 접근을 안전하게 동적으로 처리하는 것이 필수임

- 여러 사용자가 자기 계정을 활용할 수 있으려면, 에이전트가 개별 사용자 계정의 권한을 동적으로 요청·획득해야 함
- 즉, 에이전트는 "이 사용자가 이 서비스(Gmail, Slack 등)에서 내 계정으로 작동하도록 허용"하는 인증 절차를 매번 거쳐야 함
- 이 과정이 보안적으로 안전하고 사용자-에이전트 사이에 명확한 권한 범위(scope)가 설정되어야 함

### Arcade 플랫폼은 OAuth 기반 인증으로 문제를 간단하고 안전하게 해결함

- Arcade는 에이전트가 필요 시점에 동적으로 각 사용자의 서비스 계정에 대해 OAuth 인증을 요청할 수 있게 함  
    - 예: 에이전트가 최초로 Asana에서 데이터를 요청할 때 사용자에게 권한 승인을 요청
- 사용자가 인증을 승인하면 Arcade가 인증 토큰 등 자격증명을 안전하게 저장함(재활용 가능)
- 보안 및 권한 범위(scope)도 명확히 관리되어 불필요한 권한 노출 방지

### LangGraph와의 통합을 통해 에이전트 인증 및 작업 흐름이 직관적으로 구성됨

- Arcade는 LangGraph라는 시각화 툴과 연동되어, 인증 절차와 에이전트 동작 흐름을 한눈에 볼 수 있게 구성됨
- "에이전트가 어떤 서비스 작업을 요청할 때, 먼저 계정 인증 확인 → 필요시 인증 요청 → 작업 실행"이라는 표준 워크플로우가 만들어짐
- 이를 통해 복잡한 인증·작업 체계를 쉽게 빌드 및 유지 관리 가능

### 실제 Asana 연동 데모를 통해 인증 및 데이터 접근 과정을 구체적으로 보여줌

- 발표자는 Asana 서비스와 연동된 에이전트를 예시로 들어 실제 동작 과정을 시연
    - 사용자가 "SaaS 아이디어 프로젝트의 할 일(task)이 뭐야?"라는 질문을 던짐
    - 에이전트가 Asana 계정 접근 권한이 없을 경우 인증 플로우로 진입, 사용자가 접근 허용
    - 인증이 완료되자, 워크플로우가 자동으로 이어져 Asana에서 할 일을 읽어와 반환
- 인증 플로우와 결과물 확인을 영상 하단에 별도 데모 영상을 링크해 제공한다고 안내

### 수천, 수만 사용자에 대해 에이전트가 개인별 계정 작업을 처리할 수 있음이 확인됨

- 위 방식을 적용하면 에이전트는 사용자 수만큼 개별 계정 접근 및 작업(예: 이메일 읽기, 프로젝트 생성)이 가능해짐
- 인증정보 저장 및 재활용, 권한별 범위 설정 등으로 보안성과 확장성이 동시 확보됨
- 기존 하드코딩 방식보다 훨씬 손쉽고 효율적으로 대규모 서비스를 구축할 수 있음

### 장기 메모리와 결합하면 극도로 개인화된 확장형 에이전트 실현이 가능함

- Arcade 기반 인증 및 LangGraph 워크플로우 외에, 장기 메모리(long-term memory) 기술을 결합하면
- 사용자별 사용 패턴·이력·특성 데이터까지 누적·활용하는 초개인화 에이전트 구현이 가능해짐
- 이로써 자동화·대화·업무대행 등 다양한 분야에서 대규모로 맞춤형 AI 서비스가 현실화될 수 있음을 시사함
