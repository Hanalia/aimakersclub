## **Postman, Git 본연의 기능을 갖추며 API 개발을 혁신적으로 변화시킴** 핵심 요약

- Postman이 최근 대규모 업데이트를 통해 ‘Git-native’ 기능을 도입하여 API 개발 환경을 근본적으로 개선함
- 기존 API 개발 프로세스에서는 엔드포인트 정의, 컬렉션 및 테스트 작성, CI 연동 등이 각각 다른 도구와 위치에 분리되어 비효율적이었음
- 많은 개발자와 AI 에이전트가 다양한 API 엔드포인트를 다루지만, 명세(스펙), 테스트, 코드가 한 곳에 통합·버전관리되지 않아 통합 관리가 어려웠음
- Postman 업데이트로 모든 리소스(컬렉션, 스펙, 목(mock), 테스트 등)를 로컬 코드베이스에서 YAML 파일로 직접 관리 가능해짐
- 데스크톱 앱에서도 로컬 파일들을 그대로 다룰 수 있고, 클라우드와의 양방향 싱크로 환경을 자유롭게 전환 및 연동할 수 있음
- 로컬에서 변경한 테스트 코드가 git diff를 통해 명확히 추적되고, 전체 변경 이력·버전관리가 기존 코드와 동일하게 실행됨
- 모든 Postman 리소스가 파일화되어, Postman 에이전트 모드에서 API 라이프사이클 전체를 텍스트 기반(영어 명령문)으로 자동화 가능
- 실제 데모에서는 영어로 “Flask 앱을 위한 컬렉션 생성”을 지시하자, 코드베이스 분석(라우트 탐색), 명세 자동생성, 테스트/목 자동설정이 연동되어 깃 커밋으로 즉시 기록됨
- AI 모델 종류(Claude, GPT, 자체 호스팅 등) 및 서로 다른 상용 도구(MCP 서버, Github, Sentry, Linear 등)와의 연동 지원
- 동일한 컬렉션 & 테스트가 로컬 및 CI 파이프라인 환경에서 모두 재사용(사전 커밋 훅, GitHub Actions 등 적용), 환경별 별도 수정 없이 검증 가능함

## 세부 요약 - 주제별 정리

### **기존 API 개발 프로세스는 엔드포인트, 스펙, 테스트, CI가 분리되어 있어 비효율적임**

- 전통적으로 API를 만들 때 개발자는 IDE나 코딩 에이전트 등에서 엔드포인트를 직접 작성함
- 컬렉션과 테스트 작성은 별도의 도구(예: Postman)에 의존하며, 이 과정이 분리되어 있음
- CI 환경에서 테스트 자동화를 별도로 설정해야 하며, 컬렉션·스펙·코드가 한 곳에서 연결되거나 일관적으로 버전관리되지 않음
- 결과적으로 명세, 테스트, 코드가 서로 분산되어 있어서 추적·관리가 어렵고, 워크플로우가 혼란스러움
- 특히 AI 에이전트 개발 등으로 다수의 API 엔드포인트를 매번 효율적으로 관리하고 싶지만 구조적으로 어려움이 존재함

### **Postman의 새로운 Git-native 업데이트가 기존의 모든 문제를 해소함**

- Postman이 최근 ‘Git-native’ 기능을 도입, 모든 주요 리소스를 코드베이스 내에 직접 저장/관리하게 함
- 컬렉션, 명세, 목(mock), 테스트 등 모든 리소스가 관리·확장 가능한 YAML 파일로 로컬에 존재하게 됨
- 이로써 코드, 스펙, 테스트의 전체 라이프사이클과 버전 관리가 동일 경로 내에서 깔끔하게 통합됨
- 기존 10년 넘게 Postman을 사용해온 사용자는 이번 업데이트를 “AI 네이티브 API 플랫폼화”되는 수준의 혁신으로 평가함

### **로컬 코드베이스와 Postman 데스크톱 앱에서 모든 리소스를 통합 관리할 수 있음**

- 모든 리소스(YAML 파일)는 로컬 코드베이스에 직접 저장되며, 관리자 입장에서 유지·관리 용이
- Postman 데스크톱 앱으로도 동일 리소스를 연계, UI에서 코드 기반 리소스를 바로 확인·수정 가능
- 로컬 환경뿐 아니라 Postman 클라우드와 실시간 양방향 동기화(2-Way Sync)를 통해 변경사항이 항상 일치함

### **git 기반 버전관리로 스펙과 테스트 변경 이력을 명확하게 추적할 수 있음**

- 리소스 파일이 코드베이스에 직접 위치하기 때문에 git diff 등의 명령어로 모든 변경점을 확인할 수 있음
- 예시로 엔드포인트 테스트 파일을 수정한 뒤 git diff를 실행, 세부 변경사항이 즉시 추적 가능함
- 테스트·스펙의 변화가 코드와 똑같이 커밋/푸시되어, 협업 시 이력이 명확히 남음
- 코드 베이스와 동일한 방식의 git version control이 적용되어 작업 흐름에 자연스럽게 녹아들 수 있음

### **모든 Postman 리소스가 파일로 존재하므로 전체 API 라이프사이클 자동화가 가능함**

- 모든 리소스(스펙, 테스트, 컬렉션, 목 등)가 파일화되어 Postman의 에이전트 모드에서 인터랙션 가능
- 영어로 “필요한 것을 자연스럽게 서술”하면, Postman이 이를 해석하여 리소스 자동 생성·동기화 진행
- 예: “이 Flask 앱을 위한 컬렉션 생성” 요청 시, Postman이 코드베이스 전체를 스캔해 라우트, 스펙, 테스트, 목까지 자동 세팅
- 주고받은 모든 조작이 git 커밋으로 기록되어 코드리뷰 및 변경 관리에 최적화됨

### **AI 모델을 직접 선택하고 주요 협업 툴과도 끊김 없이 연동됨**

- 컬렉션·테스트 자동 생성 시 사용할 AI 모델(Claude, GBT, 자체 호스팅 등)을 직접 지정할 수 있음
- MCP 서버, Github, Sentry, Linear 등 실무에서 사용하는 다양한 협업·에러트래킹 도구들과도 정보 연동이 가능함
- 리소스 자동화 및 생성과정에 연구 환경이나 운영 환경의 다양한 맥락이 반영됨

### **동일한 컬렉션과 테스트 코드가 로컬·CI 등 모든 환경에서 일관되게 동작함**

- 한 번 만들어진 Postman 컬렉션/테스트 파일이 로컬에서 동일하게 실행
- 똑같은 리소스가 CI 파이프라인에서(사전 커밋 훅, GitHub Actions 등) 그대로 재사용될 수 있음
- 환경별로 별도 작성이나 수정 없이 일관된 테스트/스펙 관리가 실현됨
- 실수 방지 및 반복 작업 감소에 큰 효과가 있음

### **실제 데모로 손쉬운 변경·동기화·자동화 과정을 시연함**

- Postman 데스크톱 또는 클라우드 환경에서 변경사항을 생성→에디터에서 git diff→변경 확인으로 이어지는 과정을 실시간으로 보여줌
- 영어 프롬프트로 “컬렉션 생성” 지시 → 코드베이스 분석→자동 리소스 생성까지 전체 흐름이 자동화됨
- 모든 과정이 git 커밋 단위로 남아, 안전하고 검증된 개발 절차를 지원함

### **Postman의 이번 변화가 API 개발, 테스트, 유지 관리 수준을 한 단계 올림**

- 컬렉션 작성, 스펙 관리, 테스트/목 세팅, 통합 버전관리, 환경 간 테스트 일관화 등 수많은 문제를 통합적으로 해결
- AI와 코드 자동화의 결합 및 오픈된 파일 관리 방식으로 실용성과 확장성이 크게 강화됨
- 영상 제작자는 개인적으로 Postman 업데이트의 긍정적 경험을 강조하며, 관련 리포지터리 및 가이드 링크를 설명란에 제공할 것임
