## *클로드 코드를 활용해 더 안전한 OpenClaw 대체제를 만들다* 핵심 요약

- 최근 대규모 인기를 얻고 있는 오픈소스 AI 비서 프로젝트 OpenClaw가 소개됨
- OpenClaw는 사용자의 작업 내역과 선호도를 기억하며 '진짜 나를 이해하는' 첫 번째 AI 에이전트로 평가받음
- GitHub 저장소가 185,000개의 스타를 돌파하는 기록적인 인기를 얻음
- 기존의 인기 오픈소스 워크플로우 자동화 도구인 N8N의 기록까지 추월함
- OpenClaw는 그 강력함과 인기에 비해 심각한 보안 이슈가 존재함(에이전트의 권한, 코드베이스의 취약점 등)
- 이러한 문제와 개인화 니즈에 따라 제작자가 맞춤형 OpenClaw 버전을 직접 제작함
- 불과 며칠 만에 약 수천 라인의 파이썬 코드와 마크다운만으로 핵심 기능을 모두 재현함
- 제작자는 누구나 쉽게 OpenClaw의 강점을 참고해 자신만의 에이전트를 만들어보길 권함
- 오픈소스 AI 비서로부터 영감을 받되, 직접 구축·제어함으로써 보안과 맞춤성을 높일 것을 강조함

## 세부 요약 - 주제별 정리

### OpenClaw는 '나를 진짜로 이해하는' 최초의 AI 비서라 평가받음

- OpenClaw는 현재 개인용 AI 비서로 폭발적인 관심을 받고 있음
- 많은 사람들이 "처음으로 정말 내 취향과 작업 맥락을 이해하는 에이전트"라며 호평
- 시간이 지날수록 사용자별 메모리(작업 내역, 선호도 등)가 쌓이면서 점점 맞춤화됨
- 이런 매력 덕분에 다양한 사용자층에서 인기를 끌고 있음

### OpenClaw의 GitHub 저장소는 오픈소스 역사상 이례적 성장을 기록함

- OpenClaw의 GitHub 저장소는 185,000개 이상의 스타를 받으며 급성장
- 영상 제작자는 "이 정도의 별을 받은 저장소는 본 적이 없다"고 언급
- 기존에 유명했던 N8N(오픈소스 자동화 툴)보다도 스타 수가 많아졌다고 진단
- 오픈소스 커뮤니티에서 가장 뜨거운 프로젝트 중 하나로 부상

### OpenClaw에는 에이전트 권한과 코드 자체 문제로 보안 이슈가 상존함

- OpenClaw는 기본적으로 강력한 권한을 에이전트에게 위임
- 사용자의 개인 정보 및 시스템 접근 권한에 대한 위험성 제기
- 코드베이스(기반 코드구조)에도 보안 취약점이 다수 존재함이 지적됨
- 개발자/사용자 모두 보안 문제를 인지하고 있음

### 맞춤형, 안전한 대체 AI 비서를 만들고자 제작자가 직접 개발에 나섬

- 보안 이슈와 자신의 개인적 필요(커스터마이징 욕구)로 인해 직접 제작을 결심
- 기존 오픈소스 에이전트의 기능 아이디어와 구현방식을 참고함
- 자신만의 맞춤형 버전을 개발하여 OpenClaw의 핵심적 '마법'을 모두 재현했다 밝힘

### OpenClaw 수준의 기능은 비교적 적은 코드량과 시간으로 복제 가능함을 증명함

- 수천 행 수준의 파이썬 코드와 마크다운만으로 핵심 기능 구현에 성공
- "몇 일" 만에 OpenClaw의 거의 모든 주요 기능을 re-implement함
- 개발 난이도가 생각보다 낮다는 점을 강조

### 누구든 오픈소스 AI 비서의 아이디어를 참고해서 직접 제작할 수 있음을 권함

- 제작자는 시청자에게 "이렇게 직접 만들어보는 것을 권장한다"고 말함
- 그 이유로, 직접 만들어야만 진짜로 기술을 이해하고 제어가 가능하다고 설명
- 커스터마이징, 보안, 확장성 등에서 기존 공개 버전보다 자신만의 솔루션으로 발전시킬 수 있음을 시사
- 오픈소스 프로젝트의 장점을 취하되, 남이 만든 코드에만 의존하지 말고 스스로 구현해보는 태도를 제안

### 직접 구축하면 개인의 통제와 보안 측면에서 더 큰 이점을 얻을 수 있음을 강조함

- 본인의 사용 용도에 꼭 맞게 기능을 조정·제한할 수 있음
- 보안상 취약점이나 과도한 권한을 직접 통제 가능
- 완전히 이해 가능한 코드베이스를 갖춤으로써 신뢰성 확보
- 맞춤화‧학습이 용이함

### 오픈소스 AI 비서 혁신의 가치와 한계 속에서 '직접 구현'이 대세임을 시사함

- 오픈소스 AI 에이전트는 혁신적이지만, 대중적 인기에 가려진 구조적 보안 등의 현실 문제도 존재
- 커뮤니티와 대중의 아이디어를 적극 활용하되, 핵심 기능을 배워 직접 구축하는 것이 더 가치 있음을 1차 경험담으로 전달
- 사용자와 개발자 모두에게 '주도적으로 문제를 해결하고 발전시켜나가라'는 실용적 메시지를 전달함
