## *AI 에이전트의 확장성을 위해 ‘메모리’와 ‘동적 툴 접근’이 반드시 필요함* 핵심 요약

- 대부분의 AI 에이전트와 MCP 서버는 단일 사용자(즉, 개발자 본인)만을 위해 설계되어 있음
- 이런 구조는 소규모 테스트 단계에서는 괜찮지만, 수천~수백만 명의 실제 사용자를 위해 확장하는 데 실질적인 한계 발생
- 전통적으로는 각 툴(예: Gmail, Slack, Asana)과 하드코딩된 자격 증명을 연결해 에이전트를 구축함
- 이러한 방식은 오직 하나의 계정(예: 단일 Gmail)만 제어할 수 있어, 사용자별 맞춤화와 다수 사용자 지원에 취약함
- 여러 사용자를 지원하려면, 각 사용자의 개별 서비스 계정에 대해 에이전트가 동적으로 접근 권한을 받아야 함
- Arcade 플랫폼은 OAUTH 방식으로 자격 증명 수집 및 저장, 그리고 확장성 있는 사용자 인증 문제를 근본적으로 해결함
- Arcade는 LangGraph와 통합되어 에이전트 권한 부여 과정을 시각적 워크플로로 관리하고 쉽게 사용할 수 있도록 지원함
- 실제 데모에서는 Asana 작업 목록을 예시로, 사용자가 직접 권한을 부여하면 에이전트가 지정 계정에 접근해 작업을 수행함
- 이 패턴을 채택하면 어떤 사용자라도 자신의 계정으로 에이전트가 행동하도록 만들 수 있어, 대규모, 하이퍼퍼스널라이즈드 AI 에이전트 구축이 가능해짐
- 이러한 확장성 구조에서, 장기 메모리와 결합될 경우 매우 개인화된 에이전트를 웹 서비스 형태로 대량 제공할 수 있음

## 세부 요약 - 주제별 정리

### AI 에이전트와 MCP 서버의 기존 구조는 단일 사용자만을 대상으로 한다는 한계를 가짐

- 현재 대부분의 AI 에이전트 및 MCP(Multi-channel Platform) 서버는 한 명의 사용자(즉, 개발자 본인)만을 위한 용도로 설계됨
- 이 구조는 소규모 프로토타입이나 실험적 목적에는 문제없으나, 실제 서비스를 운영할 때 확장성에서 본질적인 제약이 있음
- 수천~수백만 명의 실제 사용자를 지원할 필요가 있을 때 기존 구조는 비효율적임

### 기존 에이전트 개발 방식은 계정 연결과 자격 증명 관리에서 구조적 문제를 가진다

- 에이전트 구축시 Visual Workflow Builder인 N8N을 예로 들어 설명
- 각 툴(Gmail, Slack, Asana 등)마다 하드코딩된 크리덴셜(자격 증명)을 연결하는 방식이 다수
- 단일 Gmail 계정을 연결했다면, 해당 에이전트는 오랜 기간 그 계정만 쓸 수 있음
- 사용자별로 다른 Gmail, Slack, Asana 계정을 쓰게할 수 없음
- 즉, 에이전트가 ‘각 사용자 고유의 서비스 계정’으로 동작할 수 없다는 점이 문제

### 여러 사용자가 자기 계정으로 에이전트를 운영할 수 있게 하려면 ‘동적 인증 플로우’가 필요하다

- “만약 당신의 에이전트가 각기 다른 사용자의 Gmail, Slack, Asana 계정에서 동작하기를 원한다면 어떻게 해야 할까?”
- 해법은 에이전트가 동적으로 각 사용자별 서비스 계정 인증을 요청하고, 권한을 받아야 한다는 점
- 기존 하드코딩 방식은 이런 동적 인증 및 대규모 확장에 불리함

### Arcade는 OAUTH 기반 인증 관리로 확장성 한계를 해결한다

- Arcade라는 플랫폼은 AI 에이전트가 사용자의 서비스 계정에 접근하도록 “OAUTH 방식으로 인증 요청”이 가능하게 해줌
- 즉, 서비스 계정 연결을 위해 에이전트가 사용자를 대신해 권한을 요청
- 보안(Scopes)과 자격 증명 저장이 Arcade 측에서 처리됨 – 한 번 승인받으면 에이전트가 반복적으로 자격 증명을 재요청할 필요 없음

### LangGraph 통합으로 에이전트 인증 및 권한 관리가 시각적 워크플로우로 단순화된다

- Arcade는 LangGraph라는 툴과 통합되어 인증 및 권한 부여 플로우 전체를 시각적 워크플로 형태로 관리 가능
- 개발자는 이런 워크플로를 직접 설계하여 인증 흐름 및 에이전트 동작을 쉽게 구현할 수 있음

### 실제 에이전트 구동 시 ‘인증 후 도구 실행’ 패턴을 따라야 한다

- 예: 에이전트가 사용자 Asana 계정의 할 일을 조회, 생성하는 시나리오
- 에이전트가 ‘특정 사용자의 특정 서비스(Asana 등)에 접근 권한이 있는지’ 먼저 확인
- 처음 사용하는 서비스라면 권한 획득을 위한 인증 플로우가 자동 실행됨
- 인증이 끝나면, 도구가 실제로 호출되고 에이전트가 필요한 작업(예: Asana 프로젝트 내 할 일 확인 등)을 수행

### 실제 데모에서는 Asana 계정 연결 및 활용 예시가 제공된다

- 영상에서는 “SaaS 아이디어 프로젝트의 할 일을 알려줘”라는 예시 쿼리를 에이전트에 입력
- 해당 요청을 처리하려면 에이전트가 사용자의 Asana 계정 접근 권한을 필요로 함
- 사용자가 승인 플로우(OAUTH 인증)를 거치면, 에이전트가 LangGraph 워크플로를 재개하고, Asana에서 할 일 목록을 읽어옴

### 이 구조를 도입하면 대규모 및 하이퍼퍼스널라이즈드 AI 에이전트 서비스가 가능해진다

- 어떠한 수의 사용자라도 각자의 계정에서 AI 에이전트가 작업을 실행하도록 만드는 것이 가능
- 이 방식에 장기 메모리(long-term memory) 기능을 결합할 경우, 각 사용자 맞춤형 초개인화 AI 에이전트를 대량으로 운영할 수 있음
- Arcade와 LangGraph의 결합은 확장성과 사용자별 맞춤화라는 두 마리 토끼를 동시에 잡을 수 있는 구조적 해법임
