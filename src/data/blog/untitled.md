## *MCP는 보안에 취약하다 - 그 이유와 해결 방법* 핵심 요약

- 영상은 MCP(Machine Control Plane) 서버의 구조적 보안 위험성을 짚으며, 실제로 여러 치명적인 문제가 있음을 지적함
- 가장 심각한 3가지 문제는 민감한 데이터 노출, 과도한 접근 권한 제공, 사용 내역의 추적과 감사(audit)가 불가능하다는 점임
- 예시로 GitHub MCP 서버 환경을 보여주며, 개인 액세스 토큰을 평문으로 저장해야 하는 구조적 약점을 강조함
- OAuth 인증도 불충분함을 언급하며, 여전히 장기 토큰을 위험하게 저장할 수밖에 없는 상황을 비판함
- MCP 클라이언트에서는 세밀한 권한 부여가 불가능하고, 권한 제한 없이 거의 모든 기능에 접근할 수 있어 위험이 큼
- 이 문제를 해결하기 위해서는 엔터프라이즈급 플랫폼에 의한 보안 구성, 감사 기능, 세분화된 권한 관리가 필수라고 주장함
- Teleport 플랫폼을 사용하면 각 MCP 서버가 보호·감사 가능한 리소스로 관리되고, 연결 시마다 계정 맞춤형 'just-in-time' 토큰으로 안전하게 인증됨
- Teleport 기반의 MCP 환경에서는 토큰·암호 등 비밀 정보가 클라이언트에 저장되지 않으면서도 동일한 사용 편의성을 보장함
- 플랫폼의 Audit Log로 모든 MCP 호출 기록을 서버·툴·파라미터 단위까지 감사·추적 가능함
- Zero Trust 접근관리, 역할 기반 권한 설정 등도 직관적으로 가능함을 보여주며 Teleport를 적극적으로 추천함

## 세부 요약 - 주제별 정리

### MCP 서버는 민감한 데이터 노출과 과도한 권한 부여로 보안 취약점이 발생함

- MCP(Machine Control Plane) 서버가 인기지만, 잠재적 보안 위협이 크다는 우려가 계속 제기됨
- 가장 중요한 위험 세 가지는 다음과 같음: (1) 민감 데이터 노출, (2) 불필요하게 많은 접근 권한, (3) 서버 사용 내역의 추적 및 감사 부재
- 대다수 MCP 서버는 필요 이상으로 넓은 액세스 권한을 제공함
- 사용자는 자신이 실제로 사용하지 않는 기능에도 접근이 가능해 위험도가 증가
- 보안상 최소 권한 원칙(least privilege principle)이 지켜지지 않아 사고 가능성이 높음

### GitHub MCP 실사례로 평문 저장과 심각한 권한 문제를 보여줌

- 실제 GitHub MCP 환경 설정 화면을 보여주며 부적절한 보안 관행을 지적
- 개인 액세스 토큰(PAT, Personal Access Token)을 평문으로 config 파일에 저장해야 해 정보를 노출할 위험이 큼
- 영상에서 직접 토큰을 'redact(삭제)'한 화면을 시연하며, 원칙적으로 이런 과정을 할 필요가 없어야 함을 언급
- OAuth 인증 방식을 써도 '긴 수명의 토큰'을 로컬 머신에 저장하는 위험은 여전함
- 인증 방법만 바뀔 뿐, 비밀정보 저장 및 권한 제한의 근본 문제가 해결되지 않음

### MCP 클라이언트의 권한 관리 한계가 보안 문제를 심화시킴

- MCP 클라이언트는 거의 모든 기능에 대한 권한을 갖고 있어, 세부 권한 조정이 불가
- 예를 들어, 동료에게 MCP 서버 접근 권한을 주고 싶어도 '읽기 전용' 등 세부 권한 제한이 불가능함
- 조금 더 제한된 권한만을 할당하고자 해도 별도의 방법이 없음
- 영상 제작자는 이에 대한 대안을 찾으려 여러 경로를 조사했으나 적절한 일반 솔루션을 발견하지 못했다고 밝힘

### 궁극적 해결책은 엔터프라이즈급 ‘보안 래퍼 플랫폼’이라고 제안함

- MCP 보안 취약점의 핵심 원인은, 서버 및 클라이언트 단위의 관리 한계에 있음
- 이를 극복하는 방법은, MCP 위에 엔터프라이즈급 보안 플랫폼을 ‘래핑(Wrapping)’하는 것이라고 강조
- 이런 플랫폼은 안전한 설정 환경, 감사 가능성(auditability), 세분화된 권한 관리(granular permissioning)를 제공해야 함

### Teleport 플랫폼은 MCP 보안 문제를 포괄적으로 해결함

- Teleport는 영상에서 소개한 보안 플랫폼으로, MCP 서버들을 보호 및 감사되는 리소스로 변환함
- Teleport에 연결할 때마다 계정에 맞는 ‘just-in-time 토큰(즉시 생성·소멸하는 토큰)’을 발급받아 인증이 이뤄짐
- 그 결과, MCP 클라이언트에는 비밀 토큰·암호가 더 이상 저장되지 않음
- MCP 기반 서버들을 애플리케이션에 연결하는 과정도 간편하며, 기존 MCP 기능과 동일한 편의성을 유지함
- 실질적으로 사용자는 Teleport의 보안 경유(proxy)를 통해만 MCP 요청을 하게 됨

### Audit Log 기능으로 MCP 활동을 세밀하게 추적 및 분석할 수 있음

- Teleport의 Audit Log(감사 로그) 영역에서는 모든 MCP 호출 내역을 확인 가능함
- 서버별, 도구별, 개별 파라미터 단위까지 상세히 기록되어 있음
- 예를 들어, 영상 속 예시인 ‘Claw Desktop’에서 이뤄진 모든 호출 내역을 실시간으로 조회할 수 있음
- 로그 데이터를 분석하는 전용 에이전트(agents) 기능도 탑재되어 있어 이상 징후(outliers) 탐지가 가능

### 역할 기반 접근 제어와 Zero Trust 모델로 보안성을 한층 강화함

- Teleport에서는 Zero Trust(제로 트러스트) 기반 역할 관리를 직관적으로 진행할 수 있음
- 예: MCP 서버에 대한 접근 권한을 역할별로 세분화하여 여러 사용자에게 다르게 할당 가능
- 인터페이스 상에서 간단히 권한 대상 툴이나 서버를 추가·삭제할 수 있음

### Teleport 도입으로 기존 MCP 운영 방식은 ‘보안성과 사용 편의성’을 모두 확보할 수 있음

- Teleport 도입 시, MCP 서버와 클라이언트가 기존과 비교해 보안은 대폭 강화되면서도, 사용자 경험의 불편함은 없음
- MCP 서버 추가 등 설정 변화도 여전히 간소하게 이루어짐
- MCP 기반 GitHub 서버 사용 예시도 Teleport 프록시 경유 방식에서 동일하게 작동함

### 엔터프라이즈급 MCP 운영 환경에서 Teleport는 실질적 대안임을 강조함

- 모든 MCP 요청 트래픽을 Teleport 기반 프록시로 통제함으로써 보안과 감사 체계가 통합적으로 확보됨
- 영상 내내 Teleport 기능 시연 장면을 첨부하여 실제 도입 가능성과 즉시 효과를 강조함
- 종합적 관점에서 ‘MCP의 근본 문제는 엔터프라이즈 플랫폼 도입’으로 해결 가능한 실증 케이스로 제시함
