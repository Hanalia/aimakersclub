## *Postman이 Git 네이티브화되며 API 빌드 방식이 완전히 바뀐다* 핵심 요약

- 지금까지 API 작업은 불필요하게 복잡하고, 특히 AI 에이전트 개발 시 API 엔드포인트를 빈번하게 다루게 되어 워크플로우가 엉망이 되기 쉬웠음
- 기존에는 API 엔드포인트 생성, 테스트 및 컬렉션 관리가 각각 다른 도구와 위치에서 이뤄져, 사양(specs), 코드, 컬렉션, 테스트가 서로 분리되고 버전관리도 통합되지 않음
- Postman은 최근 대규모 업데이트로 ‘AI 네이티브’ API 플랫폼이 되었고, ‘Git 네이티브’ 기능까지 지원하여 작업 방식을 혁신함
- Postman의 컬렉션, 스펙, 목(mock), 테스트 등 모든 리소스 파일을 이제 단순한 YAML 파일로 로컬 코드베이스에서 직접 관리·버전관리 가능
- Postman 데스크톱 앱에서도 이 리소스들을 직접 다룰 수 있고, 클라우드와 2-way 동기화도 지원함
- Git diff·commit으로 API 변경 사항 전체를 추적 및 리뷰 가능해져, 코드와 사양·테스트를 통합적으로 관리·협업 가능
- 모든 리소스가 로컬 디스크의 파일 구조로 존재하므로, Postman의 에이전트 모드는 API 생애주기 전체를 파일 중심으로 자동화·조율 가능함
- AI 기능을 활용하여 플레인 잉글리시(영어 설명)로 API 컬렉션 생성 등 다양한 작업이 가능하며, 코드베이스 스캔 후 라우트·스펙·테스트·목 자동 생성 및 동기화
- 변경사항은 git 커밋으로 추가되어 리뷰 및 협업이 용이해지고, 원하는 AI 모델(cloud, GPT, self-hosted 등)도 선택 가능
- GitHub, Sentry, Linear 등 외부 개발 도구의 컨텍스트도 통합하여 호환성 강화
- 로컬 기기에서 실행되는 컬렉션 테스트가 동일하게 CI 파이프라인(프리 커밋 훅, GitHub Actions 등)에서도 동작해, 환경별 테스트 중복 구현 불필요
- 영상 제작자는 Postman 공식 링크 및 예시용 GitHub 레포지토리를 설명란에 안내

## 세부 요약 - 주제별 정리

### 지금까지 API 개발은 워크플로우가 복잡하고 통합 관리가 어려웠음

- API 작업은 본질적으로 반복적이고 다양한 도구를 오가야 했음
- 엔드포인트 생성은 IDE나 코딩 에이전트에서 이뤄짐
- 컬렉션(예: 요청 집합) 및 테스트 생성은 별도의 툴(Postman 등)에서 진행됨
- CI(Continuous Integration) 환경에서 테스트 연동 역시 별도 처리 필요
- 코드, 사양, 컬렉션, 테스트 등은 서로 분리되어, 변동 사항 추적 및 협업이 번거로움

### Postman의 최신 업데이트로 AI 및 Git 네이티브 환경이 도입됨

- 영상 제작자는 Postman을 10년 정도 사용해온 경험을 바탕으로 언급
- 최근 Postman은 ‘AI 네이티브 API 플랫폼’으로 커다란 변화를 발표함
- ‘Git 네이티브’로 API 개발 리소스를 코드베이스와 완전히 통합할 수 있게 됨
- 이로써 API 생애주기 전 과정이 표준 개발 프로세스(git) 위에서 운영 가능해짐

### 모든 Postman 리소스가 로컬 코드베이스에 YAML 파일로 저장·관리됨

- 컬렉션, 스펙, 목(mock), 테스트 등 Postman의 다양한 리소스를 YAML 기반 파일 형태로 관리
- 이 파일들은 코드베이스 내부에 저장되어 유지·버전관리·협업이 용이함
- YAML 파일은 친숙하고 관리가 쉽다는 점이 강조됨
- Postman 데스크톱 앱 위에서도 이 파일 기반 스펙들을 다룰 수 있음

### Postman 데스크톱 앱과 클라우드 간의 2-way 동기화로 협업성이 대폭 향상됨

- 로컬에서 리소스를 수정하면 Postman 클라우드와 실시간 양방향 동기화 지원
- 사용 예시로, 한 엔드포인트 테스트에 변화를 준 뒤 git diff로 바로 변경 내역 확인 가능
- API 스펙과 테스트가 코드와 마찬가지로 변경사항 추적 및 리뷰 대상이 됨
- 이는 다수 개발자가 동시에 작업하는 프로젝트에서 안정적인 협업 기반을 제공함

### git diff 및 커밋 기반의 변경 이력 관리로 API 개발 투명성과 협업이 강화됨

- 모든 리소스 변화가 git diff로 시각화 및 명확하게 파악 가능
- 변경 사항을 커밋하여 팀원과 리뷰 절차 수행
- API 관련 변경이 코드 변경 흐름과 완전히 통일된 과정에서 이뤄져 효율성과 투명성이 높아짐
- 스펙, 테스팅, 코드의 변경 로그를 한 눈에 파악 가능

### 로컬 파일 구조 덕분에 Postman 에이전트가 전체 API 라이프사이클을 자동화·조율 가능해짐

- 모든 리소스가 로컬 파일로 존재하므로, Postman 에이전트(Agent Mode)가 전체 API 생애주기를 통제함
- 에이전트는 다양한 작업(예: 컬렉션 생성, 스펙·테스트·목 자동화, 라우트 탐색 등)을 처리함

### 플레인 잉글리시(일반 영어) 설명만으로 API 컬렉션 자동 생성 등 AI 기능이 작동함

- 사용자가 “이 Flask 앱을 위한 컬렉션 생성” 등 간단한 영어 문장을 입력
- Postman이 코드베이스 스캔 후 라우트를 발견, 스펙 파일 생성, 테스트 및 목 작성 등 일련의 작업을 자동으로 처리
- AI를 활용한 자동화 덕분에 반복적이고 번거로운 작업이 최소화됨
- 생성 결과물도 git 커밋 단위로 명확히 관리됨

### AI 엔진 종류 선택 및 다양한 개발 환경 지원이 가능해짐

- Postman 자동화 과정에서 Claude, GPT, 자체 호스팅 모델 등 AI 엔진을 선택해 활용 가능
- MCP 서버 등 다양한 API 백엔드 환경도 지원
- 개발자는 필요에 따라 가장 적합한 AI와 서버 환경을 선택해 활용 가능

### GitHub, Sentry, Linear 등 기존 개발툴 컨텍스트와 통합 지원으로 워크플로우 심리스화

- Postman은 GitHub, Sentry, Linear 등 외부 도구의 컨텍스트를 연동해 기존 워크플로우와 무리 없이 결합
- 실제 개발 과정에 Postman 자동화 결과물이 자연스럽게 녹아듦
- 기존 개발, 이슈 관리, 모니터링 등과의 통합 기반을 갖춤

### 테스트·컬렉션 코드가 로컬과 CI 환경에서 완전히 동일하게 동작함

- 머신에서 실행하는 컬렉션·테스트와 CI 파이프라인(GitHub Actions, pre-commit hooks 등)에서 도는 것이 완전히 동일
- 환경마다 테스트 코드를 별도 작성하거나 이식할 필요가 사라짐
- 일관된 테스트 및 품질 검증 환경을 구축할 수 있음

### 영상 제작자는 Postman 및 예제용 GitHub 레포지토리 안내를 통해 실습 접근성을 높임

- 영상 마지막에 Postman 공식 링크, 본인이 만든 GitHub 예시 저장소 링크 제공 안내
- 사용자가 즉시 참조해 실습해 볼 수 있도록 배려
