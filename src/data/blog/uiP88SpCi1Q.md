## *에이전트가 토큰을 낭비하고 있다는 사실을 모르는 이유와 AWS에서의 비용 절감 5가지 방법* 핵심 요약

- AWS의 시니어 개발자 에릭 한체트가 에이전트(Agent) 사용 시 토큰 비용을 절감하는 5가지 실질적인 방법을 제시함
- **시스템 프롬프트 캐싱**: 첫 호출 이후 시스템 프롬프트를 캐싱해 이후 호출마다 적은 토큰만 전송
- **난이도별 모델 분기 사용**: 작업 난이도에 따라 저렴한 모델(Claude Haiku 등)과 고성능 모델(Claude Sonnet 등)로 라우팅
- **툴 결과값 요약 및 저장**: 반복적으로 사용하는 대형 툴 결과값을 요약하거나 클라우드/로컬에 저장해 반복 전송 방지
- **툴 루프 횟수 제한**: 무한 반복을 방지하고 불필요한 비용 발생을 막기 위해 툴 호출의 최대 반복 횟수 설정
- **대화 이력 슬라이딩 윈도우 관리**: 다중 턴 대화에서는 최근 대화만 보내고 과거 기록은 요약해 전송, 불필요한 토큰 낭비 줄임
- 각 항목마다 AWS ‘Strands agents’ 예시 코드 및 실제 적용 API 설명 포함
- 오브저버빌리티 도구로 에이전트 성능 및 토큰 사용량 모니터링 권장
- 각 전략의 트레이드오프(예: 이력 요약 시 정보 손실)도 간단히 언급
- 빠른 토크 형식으로, 실무적이고 구체적인 적용 예시 강조

## 세부 요약 - 주제별 정리

### 시스템 프롬프트를 캐싱하여 반복 전송으로 인한 토큰 낭비를 줄일 수 있음

- 에이전트에 시스템 프롬프트를 전달할 때, 첫 호출 이후에는 캐싱을 이용해 재사용이 가능함
- AWS의 Strands agents 등 다양한 프레임워크에서 `cache_prompt=default`와 같은 옵션으로 적용
- 최초 호출 시 전체 시스템 프롬프트를 보내고, 이후에는 최소한의 정보만을 재사용하여 전송
- 툴 프롬프트와 메시지도 동일 방식으로 캐싱 가능, 지속적인 반복 호출에서 특히 유리
- 코드 예시 제시
- 캐싱을 통해 첫 호출 이외에는 불필요하게 많은 토큰을 LLM에 보내지 않도록 방지할 수 있음

### 작업의 난이도에 따라 적합한 모델을 선택하여 비용 효율을 극대화할 수 있음

- 모든 작업에 고가의 Frontier 모델을 사용할 필요가 없음을 강조
- 작업 난이도가 낮을 때는 저렴한 Claude Haiku 등 경량 모델, 복잡한 작업에는 Claude Sonnet 등 고성능 모델 선택 가능
- `if`문 등 코드 분기로 난이도에 따라 모델을 자동으로 선택하도록 설계 가능
- “어떤 모델을 쓸지” 자체를 별도의 저렴한 모델에게 맡기게 하는 방법도 있음
- 실제 코드 예시로 조건 분기 및 모델 라우팅 구체적으로 설명
- 이 전략은 다수의 에이전트/서비스에서 비용-성능 밸런스 개선에 효과적임
- “가장 비싼 모델을 모든 유즈케이스에 다 쓰지 말라”는 점 반복 언급

### 대용량 툴 결과값을 요약하거나 저장하여 반복적 컨텍스트 제공 비용을 크게 절감할 수 있음

- 툴(예: API/tool plugin) 결과가 매우 크고, 이를 매번 전체적으로 에이전트 컨텍스트에 포함시키면 토큰 소비가 급증함
- Strands agents 등에서는 결과값을 로컬 또는 클라우드 저장 후, 요약본(summarization)만 재사용하는 API 제공
- 코드 예시로 수동 구현 및 API 자동화 방식 모두 언급
- 툴 결과값이 반복해서 참조될 때, 전체 값이 아닌 요약값을 에이전트에 포함하도록 처리
- 이를 통해 반복 컨텍스트 업데이트로 인한 토큰 낭비를 방지할 수 있음
- 또한, 불필요한 툴 재호출도 줄일 수 있어 전체 성능 및 비용 효율 동시 개선

### 에이전트 툴 호출/루프의 반복 횟수를 제한함으로써 무한 루프 및 과도한 토큰 사용을 막을 수 있음

- 에이전트가 툴 호출 과정에서 반복 실행될 때, 최대 반복 횟수를 명시적으로 제한하지 않으면 무한 루프에 빠질 위험 존재
- 실제 경험 사례로, 툴이 10~20회 이상 반복 호출되는 경우도 있었음
- `max_iterations` 등 파라미터로 반복 횟수를 제한
- 무한 루프 발생 시, 불필요한 토큰 및 호출 비용이 기하급수적으로 늘어남
- 실서비스 적용 전, 툴 루프 관찰을 위한 오브저버빌리티 도구 사용 추천
- 각 툴별 실제 호출 횟수, 실행 시간, 루프 패턴 등을 정밀하게 모니터링해서 효율적으로 운영

### 오브저버빌리티 도구를 활용해 에이전트 내 각 툴 호출의 비용과 반복률을 사전/사후 점검해야 함

- 실서비스 배포 전, 각 툴 호출(툴콜)의 사용량, 실행시간, 반복 호출 횟수 등을 오브저버빌리티 툴로 분석
- 이를 통해 특정 툴이 과도하게 반복되고 있거나, 비효율적으로 동작하는지 파악 가능
- 분석 결과에 따라 반복 횟수 제한값 튜닝, 툴 설계 변경 등 조치 가능
- 제한 없이 배포하면 뜻하지 않은 토큰 낭비 및 비즈니스 손실 우려가 큼

### 다중 턴 대화에서는 대화 이력을 슬라이딩 윈도우 방식으로 관리하여 토큰 사용을 최적화할 수 있음

- 멀티턴 에이전트(대화형 챗봇 등)에서는 이전 대화 이력을 매번 전체 LLM에 주입할 경우 토큰 사용량이 급증함
- Strands agents의 `Sliding Window Conversation Manager`를 통해 최근 10건 대화만을 유지/전송
- 사용자가 직접 슬라이딩 윈도우 크기(예: 5, 10, 20) 조정 가능
- 긴 대화 기록이 누적될수록 수백~수천 개의 토큰이 낭비될 수 있음
- 전체 기록을 모두 사용하지 않고, 일정 수 이내만을 트림함으로써 토큰 사용을 최소화함
- 단, 슬라이딩 윈도우 사용 시 과거 대화 맥락이 소실되는 트레이드오프 발생

### 대화 이력 요약 기법 활용으로 슬라이딩 윈도우의 한계를 극복하고 중요한 맥락을 유지할 수 있음

- 긴 대화로 인해 과거 대화 기록이 잘리더라도, 주요 정보를 요약(summarize)해서 최신 컨텍스트에 포함하는 방식 적용
- 원본 이력을 모두 보내는 대신, 요약본을 컨텍스트 윈도우 앞부분에 포함시켜 맥락 유지
- 이 방식은 정보 손실 최소화와 토큰 절약의 균형을 잡아줌
- 대화형 서비스, 고객 지원 봇 등에서 실무적으로 많이 활용됨

### 결론적으로 5가지 전략을 동시 적용하면 복합적으로 토큰 비용을 크게 절감할 수 있음

- 에릭은 5가지 전략(시스템 프롬프트 캐싱, 난이도별 모델 분기, 툴 결과값 요약/저장, 툴 루프 횟수 제한, 대화 이력 트림/요약) 요약 정리
- 각 방법은 AWS Strands agents 등 주요 프레임워크에서 직접 적용 가능함을 시연
- 실무에서 모든 항목을 조합하여 적용 시, 불필요한 토큰 낭비를 대폭 줄이고, 성능 저하 및 비용 부담을 최소화할 수 있음
- 영상 마지막에 에릭의 LinkedIn, 블로그(programwitheric.com), 소셜 네트워크 정보 안내

### 프레임워크 및 실제 서비스에 적용되는 구체적 코드/API 예제가 학습에 직접적인 도움이 됨

- 각 전략 설명 시, Strands agents 활용법/파라미터 이름/직접적 코드 형태로 예제 제공
- 실제 모델명(Claude Haiku, Claude Sonnet 등) 언급
- 적용 API 설명 및 비공식 수동 구현 방법까지 함께 안내, 즉시 실무에 도입 가능한 구조
- 실무적이고 구체적 예시 위주로 빠르게 강의하며, 관련 질문 및 토론을 위해 에릭이 자신의 채널 공유

### 실제 토큰 비용 최적화는 소프트웨어 설계, 모델 선택과 에이전트 클라우드 운영 등 종합적 접근이 필요함을 강조함

- 단일 측면(예: 캐싱만 적용)에 의존하지 않고 다방면 개선이 중요하다고 언급
- 비용-성능-운영 편의성의 균형을 맞추기 위한 구체적 방법이 모든 단계에서 안내됨
- 에릭 본인의 실무 경험이 충분히 녹아있으며, 여러 단계를 반복 조정하는 것이 실제 효율화에 필수임
