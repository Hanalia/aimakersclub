## *에이전트는 눈가리개를 한 채 작업한다 — Johan Lajili, Poolside AI* 핵심 요약

- 영상은 Poolside AI의 Johan Lajili가 "AI 에이전트의 작업 신뢰성 및 테스트 자동화"에 대해 발표한 내용임
- 서로 다른 사용자들이 AI 코딩 에이전트의 효과에 대해 극과 극의 평가(완전 자동/무능) 하는 현상의 원인을 분석함
- Greenfield(신규 구축)와 Brownfield(레거시 통합) 환경에서 에이전트가 겪는 차이와 난점을 직관, 피드백 루프 관점에서 설명
- Brownfield 환경에선 예기치 못한 장애물(죽은 코드, 미인식 부분) 때문에 에이전트가 문제를 효율적으로 처리하기 어려움
- AI 에이전트가 실제로 새로운 코드·기능을 완전히 검증하지 못하고 "제대로 동작한다고 믿는 상태"에 머물러 신뢰성에 의심이 생김
- 이 문제를 해소하고자 Poolside에서는 에이전트가 효율적으로 애플리케이션을 테스트할 수 있게 해주는 CLI 툴 ‘Spoolside’를 개발함
- Spoolside는 로그 추출, 서비스 재시작, 인터페이스 조작, 이미지 업로드·분석, 사용자 커맨드 순차 실행 등 다양한 자동화를 지원함
- 본 툴은 자가 학습, 디버깅 및 신뢰성 확보의 첫 단계로, “AI가 자기 자신을 테스트 가능한 환경을 제공하는 것이 우리의 새로운 역할”이라고 강조
- 에이전트에 '자기점검용 마스크'를 씌우는 것이 장기적으로 더 빠르고 신뢰성 높은 개발을 만든다는 비유를 통해 미래 개발에 대한 통찰 제시
- 미래에는 전통적 제품 엔지니어에서 벗어나 ‘AIX(에이아이 경험) 엔지니어’로서, AI가 잘 활용되고 자기검증이 가능한 환경을 조성해야 함을 제안

## 세부 요약 - 주제별 정리

### 사용자별로 AI 코딩 에이전트의 효과에 대한 평가가 극명하게 엇갈리는 현상은 실제 사용 맥락의 차이 때문이다

- AI와 코딩 에이전트에 대해 온라인 커뮤니티(레딧, 트위터 등)에서 상반된 평가가 존재함
    - 일부는 “AI 덕분에 코드에 손 안 대도 모든 게 된다”는 긍정적 평가
    - 다른 일부는 “실제 앱에 적용하니 쓸모없고 무용지물”이라며 부정적 평가
- 흔히 이 차이를 “선전(홍보) 목적”이나 “불성실한 사용” 등으로 치환하지만 이는 표면적 해석임
- 실제로는 작업 상황(신규 앱 개발 vs. 기존 레거시 시스템 통합)이 영향을 미침

### 신규(greenfield)와 레거시(brownfield) 환경에서 에이전트의 효율은 피드백 루프의 품질에 따라 크게 갈린다

- Greenfield(신규 구축)에서는 에이전트의 코드 작성 직관이 거의 항상 맞음
    - 컴포넌트, 서비스 등 새로 짜면 예상대로 잘 작동함
- 반면 Brownfield(기존 시스템)에서는 예측 못한 문제(죽은 코드, 미사용 컴포넌트, 예외 상황) 다수 존재
    - 에이전트가 코드베이스의 모든 지점을 다 파악하기 힘드므로, 작업 중 장애나 오류가 쉽게 발생
- 핵심 차이는 “피드백 루프 구조”에 있음

### 피드백 루프의 빈약함이 에이전트의 결과물 신뢰도 저하로 이어진다

- 에이전트는 자신의 한계 내에서 “완료”라고 판단해도 실제로 잘못된 결과가 많음
- 예를 들어 “인증 플로우를 구현했다”고 선언하지만, 실제로 다양한 테스트를 거치지 않았을 가능성 높음
- AI에 비판적인 사용자는 이런 실패 사례를 곧장 불신/무능/헛소리로 해석함
- 신뢰의 단절은 사용자의 “AI 피드백 루프 참여 방식”에 따라 다름
    - 비판적 사용자: 즉시 포기
    - 개선적 사용자: "에이전트에게 로그 다시 점검/재시도 요청" 등 피드백을 줌

### 기존 도구의 한계를 넘어서는 에이전트용 테스트 자동화 CLI ‘Spoolside’의 개발로 신뢰성을 높이고자 함

- Poolside에서는 이 문제를 해결하고자 Spoolside라는 CLI 도구 제작
    - 실제로는 소비자용 제품이 아니라 엔지니어 맞춤형 실험적 도구임을 언급
- 기존에 발표된 G stack 등은 웹페이지 스크린샷, 스냅샷(상태 요약) 등을 지원했으나 한계가 있음
- Spoolside는 VS Code 확장 등 웹페이지가 아닌 다양한 환경에서 동작 가능
- 주요 기능
    - 백엔드/프론트엔드 로그 자동 추출
    - 개별 서비스 재시작
    - 고수준 명령(메뉴 접근, 페이지 이동, 메시지 전송) 수행
    - 이미지 업로드 및 분석(예: 스토리북 실행 처리)
    - 커맨드/테스트 동작을 일련의 시퀀스로 쌓아 효율적 자동화 구현

### '자기 자신을 재현 가능한 상황에서 테스트'하게 만드는 것만이 AI 에이전트 신뢰의 선결 조건이다

- 에이전트는 버그 수정 요청 시 “직관만으로” 처리 시도하는 경향이 많음
- 실제로 해당 버그 상태를 스스로 재현해야 신뢰할 수 있음
    - 예시: “충분히 직관이 좋아서 코드만 고치면 될 것 같아” → 실제로는 실패 확률 높음
- 인간 개발자는 “버그가 실제로 일어남”을 확인하지 않으면 결과를 신뢰하지 않듯, AI 역시 재현력을 우선 요구

### 엔지니어의 새로운 역할은 AI가 작업하기 쉽게 도구·환경을 ‘스스로 구축’하는 것임을 강조함

- 향후 개발자의 역할 변화: 제품 자체 코딩보다 “AI가 잘 작동하도록, 테스트와 피드백 루프가 쉬운 도구·환경 만들기”가 중요
- 코드 베이스 구조화, 문서화, 지식 베이스 개선, 맞춤형 테스트 CLI/스킬/MCP 구축 등이 예시
- 각자 문제·환경별로 최적화된 솔루션을 고민해야 함

### 개발 현장에서 ‘속도를 쫓는 AI’가 오류를 증폭시키지 않도록 하는 안전장치가 반드시 필요하다

- 'AI의 속도'만으로는 곧장 품질 보장 불가
    - 빠르게 복합 오류/실패가 쌓일 수 있음
- 작업 검증, 결과 제시, 자체 테스트 등 반복 검증 플로우 필수
- 2025년 이후에는 ‘제품 엔지니어’에서 ‘AIX 엔지니어’로의 역할 전환이 요구됨을 시사

### '에이전트에게 산소 마스크를 우선 씌워주라'는 비유로, 자기검증 비중을 높일 것을 당부함

- 항공기 안전교육의 “자녀보다 먼저 마스크 쓰라”는 지침과 유사한 논리
- 새로운 기능 개발보다 AI 자기 점검·테스트 환경 구축이 더 중요한 투자임
- 초기엔 속도가 늦어지더라도, 장기적·반복적 작업에서 생산성과 신뢰도가 급격히 향상될 것임을 강조

### 자동화된 테스트는 인간식 테스트 경험을 최대한 모방할 때 더 효과적임을 지적함

- 질문 응답에서, 정형화된 자동 테스트만으로는 한계가 있음을 밝힘
    - 너무 경직되어 있거나 결과 예측이 어려움
- 실제 인간이 앱을 테스트하는 과정을 흉내 내는 것이 더 바람직함
    - 예: 화면 UI의 이상 현상, 반복적으로 발생하는 패턴, 로그 상의 특이점 분석 등
- 테스트 루프에서, AI가 스스로 문제를 발견하도록 유도하는 것이 엔지니어의 역할임을 거듭 강조

### 각 제품·서비스별로 테스트·피드백 자동화 방식이 달라야 하며, 엔지니어가 이에 최적화된 환경을 고민해야 함

- 예시: 유니티 게임이라면 3D월드의 아스키(ASCII) 추상화 인터페이스 제공 고려 등
- 다양한 권한, 로그인 방식 등도 AI가 효율적으로 다룰 수 있게 설계 필요
- 각자 엔지니어가 제품 특성에 맞는 “AI 자기검증/피드백 도구” 구조를 엄밀히 고민하는 것이 필수가 됨
