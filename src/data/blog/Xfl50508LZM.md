## 세부 요약 - 주제별 정리

### 에이전트 평가의 필요성과 기존 AI 평가 방식의 한계가 실무에 문제를 일으킴

- AI 시스템(특히 LLM 기반 에이전트)은 고전 SW테스트와 달리 단일 입력에도 다수의 정답과 비결정적 출력을 보임
- 기존 단순 프롬프트 테스트/‘분위기(vibes)’ 체크/소규모 수작업 리뷰는 경계상황·오류·변경 후 회귀 미탐지 등 한계를 드러냄
    - 프롬프트나 모델이 변경되면, 예상외 입력·악의적 입력 등에서 쉽게 문제 발생
    - 기존 유닛 테스트(문자열 매칭 등)는 현황에 맞지 않아 LLM/에이전트 환경에선 불가능한 측면이 있음
    - 실무선 본격 배포를 위한 평가 체계가 필수
- 실 사례(Descript, Anthropic 등)도 ‘분위기 → 공식 eval suite’ 전환 과정을 필연적으로 경험
- 에이전트형 AI의 경우 다단계 경로, 도구 호출, 여러 하위 에이전트가 연쇄적 의존을 이루어 오류 전파 및 복구가 복잡

### 로그(Trace)와 스팬(Span) 데이터 구조, 오픈인퍼런스 표준으로 평가 데이터 자동 수집이 가능함

- AI 평가의 핵심 단위: 트레이스(Trace, 프로그램 실행 로그), 스팬(Span, 실행내 개별 단계)
    - LLM 호출, 도구 호출, 전체 에이전트 턴 등이 모두 스팬 단위로 캡처됨(중첩 가능)
    - 각 스팬: 입력/출력/실행 시간/토큰 수 등 다양한 메타데이터 포함 (대개 JSON 형태)
- instrumentation→tracing→eval 전 과정 자동화 가능
    - open-inference 표준의 등장으로 주요 에이전트 프레임워크(OpenAI, Gemini 등 포함)에서 자동 로그 수집 및 Phoenix 연동 지원
    - Arize Phoenix(오픈소스)/Fenix Cloud 서비스에서 API키와 2줄 코드로 로그 자동 집적
    - 실습 노트북: cloud agent SDK pip 설치/간단한 콜로 Instrumentation 후 로그 수집, UI로 각 스팬/에이전트 턴 및 상세 데이터 점검 가능

### 코드 기반 eval, LLM judge, 인간 평가 3가지 평가 방식의 구조와 적용시점, 상호작용 방식 규명

- 코드(eval): 단순 명확한 기준(포맷·문자열 유무·길이·필수단어 등)을 결정적으로 판단, 빠르고 저렴, 재현성 높음
    - 예: 출력에 질문한 주가가 포함되어있는가?, 500토큰 이하인가?
    - 단점: 복잡·비결정적·의미적 평가에는 부적합, brittle(사소한 변경에도 실패)
- LLM judge: 더욱 강력한 LLM이 출력 결과의 의미적 품질, 사실 적합성, 논리 일관성, 어투 등을 평가
    - 장점: 인간 수준에 근접한 의미판단 가능(예: 톤, 정책 준수 등)
    - 단점: 비결정적·비용 소모(시간·비용)·자체 오류(프롬프트 엔지니어링이 필요), 평가 루브릭 품질에 민감
- 인간 평가: 소규모 골든셋 데이터 생성에 활용, 확장성 한계(피로에 따른 오판율, 50% 정도 일부 사례 보고), 루브릭 해석 일치도 문제
- 실제 실무에서는 세 전략이 보완적으로 활용됨(Format: 코드eval, 의미평가: LLM, 신규 오류: 인간)
- 평가 목적에 따라
    - 회귀eval(성능보존 여부)
    - 기능(능력)eval(새로운 요구치 달성여부)로 구분, 반복 순환

### 재무 분석 에이전트 예제 실습 및 트레이스 분석으로 문제 카테고리화, 실패 원인 탐지 절차

- 예제: 주식 심볼 입력+주목점(재무성과/성장전망 등)→연구 및 요약 보고서 작성하는 이중 에이전트
    - CLAUDE Haiku LLM, Web Search 도구 사용
- 실제 12개의 다양한 조건(기업, 분석포인트, 비교 분석 등)에 대해 실행, 자동 트레이스·스팬 데이터 수집
- 실패/특이 사례
    - Apple: 파일.write 등 연구 결과 값 출력이 아닌 파일 쓰기 시도(프롬프트 요구 불명확)
    - Amazon: AWS 중심 설명으로 Amazon 누락
    - Rivian: 비상장 기업에 대해 그럴듯한 수치(62,000~67,000대 공급) 환각 의심
    - 여러 레코드에서 추천/액션어빌리티 누락, 잘못된 경로 등 반복 확인
- 트레이스 정독→실패 유형별 카테고리화(CC: categorization), 루트원인 분류가 eval 설계 전제

### 평가 루브릭(루브릭 개발의 5대 원칙과 예시)과 ‘액션어빌리티’ 커스텀 평가의 실제 구현법

- LLM judge 평가 루브릭 설계 5대 요소
    1. 판정자의 역할 명확화(“당신은 전문가 X…”)
    2. 판정 기준 구체화(명확/관찰가능한 체크리스트: 예- “구체적 투자 권고 포함”, “단순 과거 요약만 금지”)
    3. 입력/출력 데이터 명확 라벨링(예: <begin data>...<end data>)
    4. 예시 제공(최소 한 개의 긍정/부정 예시 포함, 체인오브쏘트 유도)
    5. 출력 형식 제한(“1단어로”, 이진/3중 분류 등, 점수화 비추천)
- “액션어빌리티(실제 투자 권고 존재)” 커스텀 평가 루브릭: 위 5요소로 실제 작성, 예시 prompt 제시
- 평가함수는 prompt + LLM + scoring체계(name/criteria/choices)로 추상화
    - 코드 예시: classification_evaluator 활용
    - 실질적 분류/설명/필터링 결과를 UI와 코드 모두에서 확인 가능
- 루브릭 변경/이전 관리, 예시를 포함한 신중한 설계, Input과 Output 설계의 미묘한 변경이 결과에 미치는 영향 사례 중심 해설

### ‘Faithfulness/Correctness’ 내장 평가와 실제 한계 지적, 평가 선택의 중요성을 실습적으로 제시

- 내장 eval(예: Correctness: 정보 정확성·완결성) — 롱텀 트레이닝 지식의 시점 차이로 실패(2026 전망은 모델이 모름)
- Faithfulness(출력 결과가 주어진 출처/근거에 충실했는지) 평가 설계: 다단계 실행 결과(예: 연구단계의 결과)를 context로 제공해야 제대로 평가 가능
- 실제 결과: 내장 Correctness는 실패, Faithfulness는 100% 성공
- 평가 루브릭의 “적합한 선택”이 eval의 품질 및 실무 개선에 결정적으로 중요함을 강조

### 평가 결과의 메타 평가: LLM judge 성능 검증/Golden Set/정량적 검증 체계 확립

- 평가의 평가(메타평가): LLM judge가 인간 판단(골든셋)과 얼마나 일치하는지 확인 필요
    - 판정자(LMM)의 성능은 분류기(classifier)와 동일하게 Precision/Recall 등 ML척도(정밀도/재현율)로 정량화
    - 예: 샘플 결과(Precision 1.0, Recall 0.33 등), 표본 수 부족 시 실제 판단 불가 — 실무선 최소 수십~수백 샘플 필요
    - Golden Data Set 구축: 인간이 예시 평가(동일 프롬프트 하에 엄격한 기준), 분할(75:25) 테스트/트레이닝 구분 추천
- Pitfall 및 다양한 편향(위치, 길이, 자신감, 모델 자기선호 등) 주의사항 및 교차모델(e.g. 판단용은 OpenAI, 실행용은 Claude로) 권장
- 인간간 합의도(interrator reliability)가 낮은 현실 — LLM 평가가 인간 합의율만큼 도달하면 충분히 실용적

### 평가 루프 전반: 실험 데이터 집합화, 반복 실험, 프롬프트 개선의 계량적 효과 측정법

- Phoenix UI의 dataset/experiment 모드를 통해 실패사례 집합 생성→prompt 개선→동일 데이터를 통한 개선효과 검증 루프 구축
    - 실습 예시: 13개 실행 중 actionability 실패 사례만 모아 개선된 prompt에 재적용→실제 개선(실패 6→0) 수치로 즉시 확인
    - 실험(Experiments): 동일 입력/동일 평가자/변경요소(예, prompt)만 다루도록 설계 → 의미있는 controlled comparison 성립
- 실험 및 배포 단계별 표본 수(예: 출시 기준 200~400개, 신뢰구간 설정), 실효적 비용·효과(몇% 정확도 향상 위해 어느 정도 표본 확보 투자?), 다양한 실험 전략(전체 인프라 재플레이, subset 실험 등) 설명

### 고급 평가 전략(쌍비교, 신뢰도 점수, 패스@K, 복수판정, 비용최적화 등)과 실무 운영 프레임워크 소개

- 쌍비교 평가(pairwise): 단일답 점수 1~10보다, 두 예시 중 우수작 비교가 신뢰성 높음 — AB 테스트·모델업그레이드 검증에 적합
- 신뢰도 스코어: pass@K(여러 번 실행 중 최소 1회 성공)/pass^K(K회 전부 성공)로 용도에 따라 달리 적용(예: 코딩 보조, 고객문의)
- 실무운용: Phoenix Fenix(오픈소스), Arize AX(엔터프라이즈/컴플라이언스), session-aware 트레이싱, 비용/정확도 최적 밸런스(Table: cost normalized accuracy), 운영중 지속 모니터링, 드리프트 자동 탐지 등 포함

### 프롬프트/모델/평가 버전 관리와 조직 조직 내 비기술 전문가의 eval 참여의 중요성 강조

- 프롬프트(및 평가루브릭) 버전 관리, 과거 버전별 성능 기록 습관, 변경효과 추적 필요
- 평가 항목/기준 설정시 도메인 전문가·PM·실사용자(비개발자) 적극 참여 권장 — 실질적 ‘좋은 결과’ 정의와 피드백
- 변화 관리(버전별 회귀/능력 평가), prompt-driven development, eval-driven development의 실전 적용 사례(예: Anthropic의 Claude Code 자동 hill-climb)

### 데이터 플라이휠, 차별화된 평가 모음이 장기적으로 경쟁우위를 좌우함

- 데이터 플라이휠: 평가-개선 루프 돌릴수록 골든세트와 평가 스위트 증가, 실패유형 이해 및 개선 누적, 경쟁사와의 차별적 mote 구축
- 실제 시장의 신속한 모델업그레이드(test regression suite) 대응력 제공→리스크 최소화/성능보장 신속화

### 실무적 Q&A로 구성원별 추가 전략, 확장 케이스(메타eval, multi-agent 등) 심화 논의
