## *Claude Opus로 기획하고, Kimi K2.6으로 구현한다? 라이브 혼합 프로바이더 벤치마크* 핵심 요약

- 이 영상은 AI 개발 자동화 프레임워크 ‘Archon’을 활용해 Anthropic Claude Opus와 Kimi K2.6 등 다양한 AI 모델을 결합하여 효율적인 소프트웨어 개발 파이프라인을 구성하고, 실제로 Github 이슈를 해결하는 다수의 ‘워크플로우’를 동시 벤치마킹하는 라이브 실험 과정을 상세히 보여준다.
- 실험 목적은 강력하고 비싼 모델(Claude Opus)을 전체 프로세스에 쓰지 않고 “기획” 단계에만 쓰고, 구현·검증 등에는 Kimi 등 저렴하고 토큰 효율이 높은 모델을 사용했을 때 품질과 효율의 최적 조합이 무엇인지를 과학적으로 분석하는 데 있다.
- 총 4가지 워크플로우 조합(KK: Kimi-Kimi, KO: Kimi-Opus, OK: Opus-Kimi, OO: Opus-Opus)이 설계되었으며, Github 이슈별로 각각의 조합이 배정되어 동일 과업을 동시에 12개의 프로세스로 처리하는 방식이 사용된다.
- 각 워크플로우는 동일한 구조로, Github 이슈로부터 정보를 수집(Explore) → 기획서(Plan) 작성 → 구현(Implement) → 검증(Self-review) → PR(Pull request)생성 및 벤치마킹 평가까지 전 단계를 자동화한다.
- 벤치마킹 평가는 7가지 세부 기준(문제해결, 근본 원인, 범위관리, 미묘한 정확성, 코드 품질, 테스트 추가, 기획-구현 일치도)으로 PR마다 70점 만점으로 점수를 산출한다.
- 실험 결과, Opus를 ‘기획’ 단계에 집중 투입하고 구현은 Kimi 등 저렴한 모델로 대체해도 품질이 거의 손실되지 않아 토큰 효율을 극대화할 수 있고, 오히려 모든 단계에 Opus를 쓰는 것과 비슷한 결과를 얻었다.
- 반대로 구현 단계에만 Opus를 투입할 경우 품질 향상이 거의 없거나 오히려 떨어지는 경향도 관찰되어, 기획의 중요성이 다시 한 번 부각되었다.
- 실험 중 각 모델별 요금, 토큰 소모 제한(rate limit), API 신뢰도, 성능 저하 사례, 기타 오픈 소스 모델(Anti-gravity, Gemini 3.5 Flash, GLM, MIMO 등) 탐색 및 후기 모음 등 AI 워크플로우 실제 운용자가 가지는 고민과 경험적 인사이트가 다채롭게 소개된다.
- 결론적으로 AI 프로바이더(모델)를 혼합 운용해 기획-구현 단계별 최적 배분을 구성하면 비용·속도·품질을 모두 확보할 수 있는 실질적 방법론을 확인할 수 있다.

## 세부 요약 - 주제별 정리

### Archon을 활용해 여러 AI 모델을 혼합하는 자동화 코딩 워크플로우 실험이 진행됨

- Archon은 ‘오픈소스 AI 코드 자동화 프레임워크’로, 각 단계별로 원하는 LLM(대형/소형 언어모델) 프로바이더를 혼합해 사용할 수 있음
- 주인공 Cole은 Anthropic Opus(기획, 고차원적 추론에 강함)과 Kimi K2.6(구현, 경제적이고 토큰 효율 높음)을 조합 운용하는 실험을 설계
- 코딩 에이전트로는 Opus는 Claude Code, Kimi는 Pi를 사용
- 실험용 저장소인 Dark Factory에서는 개발 단계에 인간의 직접적 관여 없이, 에이전트와 자동화 워크플로우로 코드베이스 자체가 진화하도록 설계됨
- Archon 워크플로우마다 각 단계별로 사용할 모델을 쉽게 노드 단위로 지정 및 교체 가능하다는 장점이 있음

### 4개 유형의 워크플로우 조합(KK, KO, OK, OO)으로 실험 설계가 이루어짐

- KK: Kimi로 기획과 구현 모두 수행
- KO: 기획은 Kimi, 구현은 Opus
- OK: 기획은 Opus, 구현은 Kimi
- OO: Opus로 기획과 구현 모두 수행
- 각 워크플로우는 Github 이슈 하나당 한 차례 실행, 총 3개 이슈에 각각 4개씩(총 12개 워크플로우)이 병렬로 동작함
- 워크플로우 단계는 [이슈 컨텍스트 수집] → [코드베이스 탐색(Explore)] → [기획 문서 작성(Plan)] → [구현(Implement)] → [(자기)검증 및 PR 생성]의 동일한 구조
- 테스트 이슈 선정으로는 난이도가 다른 3개 이슈(매우 간단한 컨트롤 케이스 1개, 복잡한 추론 요구 이슈 2개)가 미리 선발됨

### 벤치마킹 평가 항목과 자동화 룰이 체계적으로 구성되어 있음

- 워크플로우로 생성된 각 PR은 ‘7가지 기준’(Issue Solve, Root Cause, Scope Discipline, Subtle Correctness, Code Quality, Tests Added, Plan-Implementation Fidelity)으로 평가됨
    - 각 항목 점수는 1~10점. 총점 70점
- 예시로 “구현이 이슈를 실제로 해결했는가?”, “증상만 가리거나 구조적으로 풀었는가?”, “범위가 불필요하게 넓어지지 않았는가?”, “예외 상황 건드림없이 미묘한 정확성까지 확보했는가?” 등
- 평가 결과는 JSON 구조 산출물로 Archon이 자동 수집 및 집계하며, 평가 워크플로우 자체도 자동화됨

### 토큰 한도와 구독 요금, 모델간 API 사용성 및 실무적 한계상황이 실험 중 상세히 공유됨

- Opus, Kimi 각각 구독 형태 및 실제 사용시 토큰 사용량/제한(5시간/주간 한도 등), 비용($40/월 Kimi, Anthropic Opus 상대적으로 비쌈), OpenRouter 활용 가능성 등 언급
- 실시간 실험 도중 충돌/딜레이/에러(예: Kimmy API hang, 툴 에딧 실패, Cloud Opus subprocess crash 등)이 자주 발생, 신뢰도와 회피전략(재시도 논리, 노드/워크플로우 단위 재실행 등) 도입 상황 설명
- Rate limit 관리, 모델간 사용량 비교, 실제로 여러 구독 필요함 등 실전형 사용자 관점 팁 제공

### Archon 워크플로우 단계별 동작과 데이터/컨텍스트 전달이 구체적으로 해설됨

- 각 워크플로우의 입력은 항상 GitHub Issue이며, 이슈를 기반으로 레포지토리 파일 및 관련 정보 탐색
- 탐색 결과를 마크다운 문서 형태로 ‘계획 단계’에 전달, 구체적인 수정 계획과 파일단위 작업 리스트 산출
- Node간 데이터 전달은 output 필드 및 아티팩트 디렉토리(문서 저장 후 다음 노드가 읽음) 활용
- 구현 노드는 결과를 다시 PR로 제출, 평가자동화 노드는 PR diff와 계획(Plan)대비 일치 여부 점검 등 구조화됨

### Opus(고성능 모델)를 ‘기획’에 집중해서 쓰는 것이 비용-품질 두 마리 토끼를 잡는 핵심임이 실험으로 입증됨

- 모든 단계에 Opus를 쓴(OO) 경우와, Opus를 기획에만 써서 나머지 단계는 Kimi로 대체한(OK) 경우의 점수 차이가 거의 없음(67~68점대)
- 반대로 구현만 Opus를 쓸 경우(KO), 또는 Kimi만 쓴 경우(KK)는 오히려 점수 하락 또는 효율성 저하 발생
- 즉, ‘계획(Plan)’ 시에 강력한 추론력을 가진 모델을 사용하면, 실제 구현은 상대적으로 약한 모델도 문제 없음
- 실험자의 경험상 “쓰레기 계획이면 아무리 강력한 모델이 구현해도 좋지 못한 결과(Garbage in, Garbage out)” 강조

### 실제 라이브 실험에서 각 모델 구독 및 사용성,원활하지 않은 점(에러, 한계 등)이 소상히 드러남

- 실험 중 Kimmy API hang, tool edit 다중 실패, 과도한 컨텍스트 로딩, Opus subprocess crash 등 문제 발생 및 조치 과정 실시간 공유
- Anti-gravity(Gemini 3.5 Flash 기반) 툴의 디자인·프론트엔드 생성력은 우수하나 정보 정확도, 이유력, 토큰 한도 등 치명적 결함 지적
- Anti-gravity, Claude, Kimmy 각종 사용제한(e.g., plan당 토큰/주/시간 제한), 구독 요금, 실제 사용량 계산 등 구체적 실습 데이터 노출
- 일부 도구(anti-gravity)는 이탈 경험(신뢰도 저하, 한계 초과 알림 등), 실시간 채팅에서 다른 유저 경험/불편함과도 교차 검증

### 오픈소스 및 다양한 AI 모델(Anti-gravity, Gemini 3.5, Miniax, GLM, MIMO 등)에 대한 탐구와 실제 조합 운용 경험이 제시됨

- Gemini 3.5 Flash의 프론트엔드 생성력, 추론력·정보 신뢰 부족, 요금/토큰 이슈 등 실질적 장단점 진단
- Open Router를 통한 다양한 오픈소스 LLM 실험(Quen, GLM, DeepSeek, Mimo 등) 탐색, 추후 차후 조합 실험 예정 언급
- Pi, Cloud Code, Cursor 등 코딩 에이전트와의 연결/호환성, Node 단위마다 지정, Provider간 세션 정책 주의점 등 실무 팁 추가 설명

### 워크플로우/자동화 실험의 유연성, 향후 확장 가능성과 추가 실험 계획이 공유됨

- Archon 프레임워크로 DAG(Direct Acyclic Graph) 기반 자유로운 워크플로우 설계, 다양한 모델간 핫스왑 가능, 이슈 검증 자동화 등의 강점
- 트리거 관리(예: 30분 간격 트리거, 자동 triage/우선순위/난이도 판단 후 워크플로우 실행) 자동화 구성 소개
- 실험 확장 방안: 더 복잡한 이슈, 더 다양한 모델, 더 세분화된 계획-구현 단계 배분, 도메인 확대 등

### 채팅을 통한 실시간 커뮤니티 참여, 현업 개발자 팁, 도구 사용법과 경험적 인사이트가 풍부하게 공유됨

- 실험 진행 중 실시간으로 시청자와 의견 교환, 각종 AI·코딩 플랫폼 활용팁, 각종 모델의 현행 요금제, 사용 한계, 구독 관리 등 실전 정보 소개
- 구독 기간별 제한 초기화, API 문서 추출 요령, rate limit 관리 · 대체 구독 자동 전환 아이디어 등 현업 팁
- 오픈소스/무료 서비스(예: 9router)로 다수 AI 프로바이더 혼합 연결 및 무제한 AI 코딩 클라우드 등 참신한 서비스 제안 공유

### 최종 평가: 강력한 모델의 기획 단계 집중 투입이 AI 자동화 코딩 워크플로우의 토큰 효율-품질-속도 최적화를 가능케 함

- 12개 워크플로우의 70점 만점 평가결과 ‘Opus 기획 집중’ 조합(OK)이 거의 최고점, 모든 단계 Opus(OO)와 유사점수
- 각 케이스별 평균/세부점수 표 제시 및 요약
    - 예: OK: 평균 67점, OO: 68점, KO: 57점, KK: 60점 등(예시, 실제 영상 내 확인 필요)
- 결론 요약: “Reasoning(추론)은 ‘기획 단계’에서 결정적, 구현단계는 저가·저토큰 모델로도 충분히 대체 가능”
- 현업에서 Archon 같은 워크플로우 자동화 프레임워크를 활용해 프로바이더를 혼합하면 실질적 시간·비용·품질 최적화를 구현할 수 있음을 입증
