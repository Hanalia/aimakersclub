## *12,000줄의 코드를 200줄의 스킬로 대체하기 — Cursor의 혁신적인 리팩터링 경험* 핵심 요약

- Cursor 애플리케이션 내 특정 '워드 트리' 기능을 12,000줄이 넘던 코드에서 단 200줄 미만의 마크다운 기반 '스킬'로 대체하는 과정과 원리, 효과를 상세히 소개
- 기존 구현에서는 워드 트리 별 분기 관리, 에이전트 분리, 격리, 평가, 클린업 등 다양한 복잡성으로 인해 코드와 유지보수 부담이 매우 컸음
- 새로운 구현은 Cursor가 가진 ‘에이전트 스킬’ 및 ‘서브 에이전트’ 두 가지 원시 동작만으로 워드 트리 기능 및 베스트이벤트(여러 모델 간 비교)도 간단히 재현
- /worktree, /bestevent와 같은 간결한 슬래시 커맨드로 기존 기능을 완전히 대체하며, 각각은 주로 40줄 내외의 마크다운 프로프트로 동작
- 동일한 태스크를 여러 AI 모델(예: GPT, Grok, Opus 등)에 병렬 수행시킨 후, 결과를 비교/평가하고 원하는 결과를 조합할 수 있는 등 사용자인터페이스도 진보
- 코드량은 약 15,000줄이 삭제되어 유지보수 부담이 급감하고, 사용자 역시 훨씬 간단히 워크트리 전환 및 병렬 작업 활용 가능
- 멀티 저장소 환경(예: 프론트엔드/백엔드 별도 repo 동시 작업) 등에서의 워크트리 병렬화도 자연스럽게 지원하게 됨
- 다만, 에이전트가 워드 트리 내에서 지속적으로 작업 위치를 잘 지키지 못하는 이슈, 기능의 발견성 저하 등이 단점으로 지적되며, 개선을 위한 평가(evals) 및 RL 학습도 병행 중
- 향후에는 Git을 쓰지 않는 환경에서도 병렬화 가능한 새로운 원시 동작(primitives)까지 개발 계획이 있음

## 세부 요약 - 주제별 정리

### 복잡한 워크트리(Word Tree) 기능은 수많은 코드와 높은 복잡도를 초래했음

- 기존 Cursor 2.0(2023년 10월 출시)부터 도입된 워크트리(work tree) 기능은 깃(Git)의 워크트리를 활용, 분리된 작업공간에서 에이전트가 동시에 병렬작업 수행 가능하도록 설계됨
- 에이전트는 각 워크트리 별로 파일을 독립적으로 조작하며 실제 동일 파일이라도 서로 다른 트리에서는 다른 상태를 가질 수 있음
- 사용자는 여러 에이전트로 동시에 서로 다른 태스크 혹은 같은 태스크를 병렬 수행시키고, 변화(diff)도 병렬로 비교
- “베스트이벤트(Best Event)”라는 기능으로 서로 다른 모델(예:GPT, Grok 등)에 같은 태스크를 부여, 결과를 경쟁시킬 수 있음
- 이러한 기능 지원을 위해, 코드상에서 워크트리 생성/관리, 에이전트 격리, 권한 제한, 셋업 스크립트 실행, 평가 시스템, 커스텀 클린업 로직 등 수많은 로직(1만2천~1만5천 줄)이 필요했음
- 사용 빈도는 높지 않은 ‘고급 유저’ 대상 기능이었지만, 개발팀의 유지보수 부담은 매우 컸음

### 모든 기능을 마크다운 기반 스킬과 서브에이전트로 극적으로 단순화함

- 복잡했던 베스트이벤트 및 워크트리 기능을 ‘에이전트 스킬(skill)’과 ‘서브에이전트(sub agent)’ 두 개의 기본 동작만으로 재구현할 수 있음을 발견
- 마크다운 기반 짧은 프로프트 텍스트(약 40줄)로 에이전트를 특정 워크트리 안에만 머물도록 지시
- /worktree 커맨드: 입력된 태스크(예: "웹사이트 footer의 오타 수정")를 새로운 워크트리에서 고립 실행
- setup script 지원, OS별(Linux/Mac/Windows) 호환, 워크트리 커맨드의 경계 유지는 ‘강한 프롬프트’로 해결
- /bestevent 커맨드: 지정된 여러 모델을 각각 서브에이전트로 워크트리에 넣어 같은 태스크를 병렬 실행, 결과를 한 표로 정리·비교·코멘트 제공
- 합쳐서 200줄 미만의 마크다운 명령만으로 워크트리 & 베스트이벤트 기능 완결

### /worktree 및 /bestevent 커맨드는 “명령형 스킬”로 동작하여 즉각적이고 쉽게 배포됨

- 새로운 기능은 “스킬(skill)”이 아니라 “커맨드(command)” 형식으로 구현되나, 작동 원리는 거의 동일
- 커맨드는 서버에서 직접 프롬프트를 관리·버전업 가능
- 사용자는 커서 업데이트 없이 최신 프롬프트, 개선된 동작을 즉각 활용 가능
- 주요 명령: /worktree (고립 워크트리에서 작업), /bestevent (여러 모델 비교), /applyworktree, /deleteworktree 등

### 베스트이벤트 및 병렬 수행이 한층 쉬워졌으며 결과 조합 및 상세 피드백도 가능해짐

- 시연 예시: "Kimmy", "Grock", "Composer", "GPT", "Opus" 등 여러 모델 지정, 동일 작업 병렬 실행 후 표로 결과 비교
- 에이전트들은 각각 독립 워크트리에서 결과 생성, 합계정(parent) 에이전트가 차이점, 특징/장단점, 추천 등을 자동 표 형식 제공
- 사용자가 “Opus의 이 부분과 GPT의 이 부분을 합쳐줘”라 요청하면 결과를 조합 가능 (이전에는 모델 하나만 선택 가능했음)
- 멀티 저장소 환경(예: 프론트/백엔드 별도 repo)에서도 각각 워크트리를 자동 생성, PR도 자동 분리 가능

### 대규모 코드 베이스가 단 200줄 이내로 대폭 감축됨에 따라 유지보수 효율성 대폭 향상

- 기존 코드(약 12~15천 줄) 완전 삭제, 신형 스킬/커맨드에선 마크다운 프롬프트 200줄 이내로 동일 기능 대부분 구현
- 유지보수 부담 감소, 향후 수정 및 개선이 월등히 용이해짐
- 고급 사용자(’파워유저’) 대상이기에, 기능 복잡성보다는 효율과 단순성이 중요하게 작용

### 사용자 경험이 개선되고, 중간 전환 및 다양한 작업 흐름도 자연스럽게 지원 가능해짐

- 기존엔 UI의 드롭다운·설정 등으로 워크트리 사용이 번거로웠으나, 이제는 아무 채팅/작업 흐름 중 언제든 슬래시 커맨드로 전환 가능
- 여러 저장소 작업 병렬화, 채팅 진행중 미드 플로우에서 워크트리로 전환 등 다양한 워크플로우 지원 범위 확장

### 기존 방식 대비 약간의 단점 및 개선점도 존재함

- “에이전트가 워크트리 내에 국한되어 작업할 보장이 약함” → 모델의 프롬프트, 시스템 리마인더에 의존하는 구조로 모델이 가끔 작업 위치를 잊거나 일탈하는 케이스 존재
- 체감 상 작업 속도가 다소 느린 느낌 (작업 생성 및 진행 과정을 더 노출하기 때문, 실제 속도 저하는 아님)
- 기능이 UI 내 드롭다운 등에서 더이상 노출되지 않아, 모르는 사람은 발견·학습이 어려워짐(하지만 고급 기능임을 감안, 양해 가능)

### 에이전트 일탈 방지 및 성능 개선을 위해 평가(evals)와 RL훈련이 적극 도입되고 있음

- 에이전트가 워크트리 작업 경계를 잘 지키는지 평가하는 eval 작성
    - 주요 평가: 워크트리 내 정상작업 여부, 메인 체크아웃에 잘못 작업 여부 탐지
    - Braintrust 등 도구로 평가 자동화 진행
- 모델별 차이 발견: 작은 사이즈의 haiku 모델이 훨씬 잦은 일탈, Composer·Grok 등은 안정적
- 향후 RL 및 프롬프트 개선, 더 어려운 시나리오/롱세션에서도 견고한 유지력 확보를 목표

### 향후 Composer 3.0/4.0/5.0 등 점진적 고도화를 통해 자체 모델의 코딩 에이전트 품질을 강화할 예정

- RL 파이프라인에 워크트리 환경에서의 태스크 추가
- 서드파티 모델은 직접적으로 개선 불가하나, 각 랩/벤더에 피드백 지속 중
- 계속된 평가-프롬프트 개선-학습의 선순환 구조 마련 계획

### 다음 단계로는 ‘에이전틱 UI’ 및 비-Git 환경에서도 병렬 처리가 가능한 새로운 패러다임을 준비중임

- Cursor 3.0(최근 공개): 더욱 ‘에이전트 중심’의 코딩 인터페이스
- 추후 Work Tree의 보다 ‘본격적’이고 네이티브한 구현(드롭다운/전용 UI 등)도 새 에이전트 창에 적용 계획
- 동시에, git worktree의 한계(성능, 저장공간, Git 强의존)를 넘어, Git 없는 환경과 더 빠르고 가벼운 병렬화 동작도 연구중
- 향후 발표에서 추가해설 및 신기능 공개 예정

### 결론적으로, 마크다운 스킬 기반 리팩터링으로 기능 품질을 유지하며 개발 효율성과 사용자 경험 모두 크게 증진됨

- 규모, 구조, 유지보수, 기능성, 확장성 모두 균형 있게 한 단계 도약
- 고급 사용자를 위한 강력한 병렬화 기능 제공 및 평가/학습 주도 개선이 선순환 중임
