## *모든 하네스는 결국 클로(Claw)가 된다 — Sam Bhagwat, Mastra* 핵심 요약

- **Sam Bhagwat(Mastra 공동 창업자, TypeScript 에이전트 프레임워크 개발자)가 “모든 하네스는 결국 클로(Claw)가 된다”라는 주제로 에이전트, 하네스, 클로의 발전 방향과 미래를 제시함**
- **에이전트 프레임워크의 발전, 로컬 하네스에서 클라우드 기반의 항상 켜진 하네스로의 전환 등 최근 18개월간 AI 실전 적용의 변화상을 구체적으로 설명**
- **에이전트(Agent)와 LLM(범용 언어모델)의 차이, 그리고 에이전트에서 하네스(harness), 클로(claw)로의 기술적 스펙트럼 진화 단계를 제시**
- **견고함(durability), 자동화, 병렬 처리, 스킬 동적 할당, 세션 지속 등 하네스의 핵심 특성을 실제 예시를 통해 언급**
- **최근 3개월간 관찰된 “항상 켜진 클라우드 하네스”의 확산 현상 및 기업 내부와 협업 환경에서의 활용 사례 강조**
- **클로란 “이니셔티브(initiative)”와 “학습(learning)”이 결합된 자율적인 인공 에이전트로, 지속적 학습과 프로세스 자동화, 백그라운드 실행/지속성 등 고도화된 특징을 가짐**
- **스마트폰 플랫폼(안드로이드, iOS) 사례를 들어 결국 극소수의 강력한 클로만이 시장에 남게 되는 개체 제한 현상(스테인버거 법칙, Steinberger’s law)이 생길 것이라 전망**
- **사용자의 뇌 용량, 경제적 가치, 사용 빈도에 의해 살아남는 클로가 결정됨을 설명하며, 빠르게 변하는 시장에서 경쟁력을 유지하는 것이 중요하다고 조언**
- **에이전트 및 하네스 개발자는 사용자의 진짜 필요에 부합하는 강력하고 지속 가능한 클로를 만들기 위해 시장 변화에 민감해야 함을 시사**

## 세부 요약 - 주제별 정리

### Sam Bhagwat가 제시하는 “모든 하네스는 결국 클로가 된다”는 미래 비전을 통해 산업의 흐름을 기술함

- 발표자는 Mastra(타입스크립트 에이전트 프레임워크) 공동 창업자이자 “Principles of Building AI agents” 저자 Sam Bhagwat임
- 최근 18개월간 다수 AI 에이전트의 실제 생산환경 운영 경험을 바탕으로 업계의 큰 변화상을 진단
- 본 발표에서는 단순한 현재의 모습이 아니라 에이전트, 하네스, 클로의 미래적 진화에 초점
- “하네스(harness)”가 점점 더 자율적이고 강력해져 결국 “클로(claw)”로 진화한다는 핵심 논지를 제시

### 에이전트(Agent)란 무엇이며, LLM과 구별되는 특성을 요약함

- 에이전트는 단순한 LLM 호출(one-shot inference)과 다르게
    - 에이전트 루프(agent loop), 툴 호출(tool call), 메모리 사용, 실패 과제 재시도, 컨텍스트 관리 등 자율동작 구조로 이루어짐
- Dex Horty(발표자 친구)의 아이디어를 빌려, 에이전트의 상태 및 행동성(action), 내구성, 집요함(doggedness) 등이 품질로 제시됨
- 한 번에 툴을 사용하는데 그치지 않고, 여러 단계를 능동적으로 처리, 맥락을 기억하며 실패 시 복구 가능함

### 로컬 하네스에서 클라우드 하네스로의 전환이 최근 AI 에이전트 아키텍처의 가장 큰 변화임

- 초기에는 개발자가 로컬에서 돌리는 하네스가 주류였으나 클라우드 기반으로 “항상 켜진(always-on)” 하네스가 급증
- 클라우드 하네스는 Slack, 모바일 앱, 터널링 등 여러 채널에서 유저와 상호작용이 가능함
- 내부용 혹은 상용 하네스를 활용해 조직 구성원들이 협업하며 에이전트에 지시를 내림
- 클라우드는 고도의 병렬 처리와 자원 확장이 가능, 분산 시스템 아키텍처가 필수적임
- 코드 작성 시 로컬이나 git work tree가 아니라, 직접 Pull Request(PR) 생성 후 GitHub 등에 푸시함

### 하네스의 주요 기술적 특징은 견고함, 병렬성, 동적 에이전트 생성, 세션 지속성, TUI 등임

- 하네스는 일시적(one-off)이 아니라 장기 실행(수시간~수일) 가능한 견고함(durability) 강조
- 예시로
    - 네트워크 연결이 끊겨도 스트림이 복구되는 케이스
    - 클라우드 코드에서의 플래닝(planning) 모드
    - 병렬 하위 에이전트(sub-agent) 분기 실행
    - TUI(Text-based User Interface)와 슬래시 명령 등으로 조작 가능
    - 동적으로 필요한 에이전트를 즉석에서 생성
    - 백그라운드에서 Bash 태스크 실행, 맥락저장(autocompact), 세션/스레드 저장 및 이어받기 지원
    - 큐잉, 세션 별 툴 사용 권한 관리, 중단 및 조정, 실시간 제어 등이 제공됨
- 예시 도구: cloud code, codecs, cloud harness 등

### 클라우드 하네스의 확장은 사용자와 조직의 새로운 협업/자동화 방식으로 이어짐

- Slack 내에서 팀원들이 함께 명령을 내리거나, 각자의 사용 데이터를 합치는 등 실제 조직 내 활용 본격화
- 모바일 앱(예: 로컬 머신으로 터널링)의 등장이 클라우드/로컬 자원의 결합을 가능케 함
- 클라우드 샌드박스 내에서 실행돼 병렬성과 확장성 증가, 분산 아키텍처의 중요성이 상승함
- 개발 방식이 PR 중심 자동화로 전환중(rewrite 작업이 git에 직접 적용/푸시됨)

### 클로(Claw)는 이니셔티브를 갖고 학습하며 사용자로부터 자유롭게 행동 범위를 확장하는 자동화된 에이전트임

- 클로는 단순히 명령을 받는 하네스가 아니라 스스로 ‘이니셔티브(initiative)’와 ‘학습(learning)’ 능력을 가짐
- 예시: CAI 어시스턴트가 이메일의 긴급성을 파악해 직접 사용자에게 메시지
- 외부 피드에 계속적으로 반응(Heartbeat), 다양한 채널(문자/SNS/메신저 등)로 명령을 수신
- 메모리 역시 단순 파일 대신 더 접근성 높은 저장소에 보존
- 데몬(daemon), 게이트웨이 등 백그라운드 서비스 형태로 동작
- “지속적 학습(continual learning)”으로, 실행 트레이스를 기반으로 스스로 개선/자동화
- 자동 스킬 생성/코드 수정 등 다양한 자율성 구현 사례 존재
- 아직 업계에서는 최적화된 구현 방식 논의가 계속됨

### 산업체 및 Mastra 관점에서 클로화(claw-ification)를 요구하는 수요와 동향 관찰

- OpenClaw, Hermes agent 등 외부 도구의 고급 기능에 대한 수요가 클로 확장을 촉진
- 많은 유저와 개발자가 더 강력하지만 제어력 있는 클로를 원함(일방적 제한 없는 커스터마이즈 선호)
- Mastra도 이러한 기능 제공에 집중하며, 사용자/개발자가 손쉽게 클로급 에이전트를 만드는 도구 확대 목표

### 하네스의 진화는 “도파민 카지노”와 같은 즉각적 보상 처럼 점점 더 엔드유저 주도적/자동화적으로 변화함

- 사용자는 Slack, 텍스트 등 다양한 수단으로 하네스에 명령을 내리고, 밤새 태스크를 실행시키고 싶어함
- “도파민 카지노(즉각적으로 보상을 얻는 즐거움)” 개념이 활성화, 에이전트가 코드를 즉석 변환/실행하는 등 반복 이용 유도
- Dex Horty가 소개한 개념 빌려옴

### 플랫폼 진화 사례(스마트폰)처럼, 미래에는 소수의 강력한 클로만이 시장에서 살아남을 것임

- 2010년대 iOS, Android 앱 생태계에서 수많은 기능별 앱이 나왔지만, 각 카테고리/서비스별로 실제로 살아남은 것은 1~2개임(Uber, Lyft 등)
- 경제적 가치가 높거나, 사용 빈도가 높지 않으면(예: Thumbtack) 결국 도태됨
- 사용자는 기억할 수 있는 서비스/클로의 수가 극히 제한적임(“뇌 용량의 한계”)
- 따라서 앞으로도 수많은 하네스/클로가 등장하겠지만, 자주 쓰이거나, 경제적으로 매우 유의미한 소수만 시장에 남을 것

### 시장 내 Shakeout(거품 제거, 합종연횡 등 빠른 구조조정) 현상이 반드시 발생할 것을 경고함

- 지금은 혁신, 다양성, 실험이 넘치는 시기이나, 곧 몇몇 카테고리/기업 주도로 정리될 Shakeout이 불가피함
- 2020년대 후반, 또 한 번 큰 변동과 경쟁이 예상됨
- 사용 빈도·경제성 등 시장논리로 인해 소수만 살아남고 대다수는 잊혀지는 현상

### 에이전트/하네스/클로 개발자는 사용자 요구와 변화 속도를 주시하며 경쟁력을 갖춰야 함

- 시장 변화 속도가 매우 빨라, 사용자가 더 강력한 기능을 원하면 쉽게 대체될 수 있음
- 이벤트·컨퍼런스 참여 등 최신 동향 파악이 중요(3~4배 빨라진 변화율)
- 사용자 요구와 트렌드를 지속적으로 반영하고, 진짜 필요와 차별화에 집중해야 살아남음

### 결론: 점진적 진화에서 클로화로 향하는 속도를 인식하고, 진정 강력한 에이전트를 개발하라는 메시지로 마무리함

- LLM에서 에이전트, 하네스, 그리고 클로로 이어지는 “에이전틱 스펙트럼”에서 상향 이동이 가속화됨을 강조
- “Steinberger's law(스테인버거 법칙): 모든 하네스는 결국 클로가 된다”는 공식 제안
- 기술적, 경제적, 심리적 동기가 복합적으로 작용해 하네스의 클로화가 미래 표준임을 시사
- Sam Bhagwat는 Mastra 프레임워크와 저서(Principles of Building AI agents)를 강조하며, 앞으로의 에이전트 개발 문화와 혁신을 촉구하며 발표를 마무리
