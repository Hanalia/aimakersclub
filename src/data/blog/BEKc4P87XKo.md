## *AI와의 협업, 단순 사용을 넘어서는 에이전틱 엔지니어링* 핵심 요약

- ‘에이전틱 엔지니어링’(Agentic Engineering)은 AI를 단순히 사용하는 것이 아니라, 실제로 협력하는 패러다임으로 전환할 필요성이 있음을 강조함
- 2020년대 초는 코드 자동 완성(AI가 함수 시그니처, 문장 등 일부를 예측) 기능이 주축이었지만, 2025년 즈음 AI 모델은 작업 분석∙실행∙테스트∙PR 생성까지 ‘협력자’ 수준으로 발전함
- AI 에이전트는 방대한 지식과 빠른 코드 작성 능력을 제공하지만, 인간 고유의 비즈니스 맥락이나 판단력은 결여되어 있음
- 에이전트와 효과적으로 협업하기 위해서는 무엇을 AI에게 맡기고 무엇은 직접 해야 하는지 판단하여 작업을 ‘디렉팅’하는 역량이 매우 중요함
- ‘컨텍스트 엔지니어링’(Context Engineering)이 핵심이며, 입력창에 들어가는 맥락 데이터의 양과 질, 세션 관리가 작업 결과에 결정적임
- 컨텍스트 과다/부적절 혼입은 LLM 품질 저하와 효율 저하로 이어지므로, 작업별 ‘새 세션 시작’과 맥락 요약/구성을 적극 활용해야 함
- 효과적인 워크플로우는 ‘연구→계획→구현’의 루프를 따르며, 각 단계마다 업무 사전정의와 컨텍스트 정제가 필요함
- Kilo Code 등 AI 코딩 에이전트 플랫폼은 다양한 ‘모드(ask/architect/code)’ 및 agents.md, skills.md 등 문서화를 통한 맞춤형 에이전트 설계를 지원함
- MCP 서버 등 툴 확장 시 무분별한 정보 로딩을 피하고, 실제 필요한 도구만 활성화해야 효율적임
- 에이전틱 엔지니어링의 진정한 가치는 반복적 실습과 피드백 속에서 AI의 한계와 강점을 체득하며, 인간 엔지니어의 창의성을 고도화하는 데 있음

## 세부 요약 - 주제별 정리

### AI와의 협업 방식으로서 에이전틱 엔지니어링이 요구됨

- 영상 초입에서 "지금 당신은 어떻게 AI를 활용하고 있는가?"라는 질문을 던져, 많은 엔지니어들이 구체적으로 설명하지 못함을 지적함
- 현재 약 90%의 엔지니어가 AI 도구를 접해본 경험이 있으며, 그 중 절반은 정기적으로 사용 중임
- 단순히 코드 완성 속도를 높이는 '도구' 역할이 아니라, 더 적극적인 파트너십 형태가 요구됨
- 에이전틱 엔지니어링은 “AI를 단순히 쓰는 것”이 아니라 “AI와 함께 일하는 것”으로 새로운 사고방식의 전환임

### 2020~2026년 AI 코딩 에이전트 변화가 ‘협력자’ 패러다임을 열어줌

- 2020년대 초 AI 모델은 일부 코드(함수 시그니처 등) 자동 완성에 특화되어 있었음
- 2022년 GitHub Copilot 등장으로 함수 전체 생성 및 챗 기반 개발이 크게 확산됨
- 2025~2026년에는 AI가 작업을 실제로 분해∙실행(코드 변경, 테스트, PR생성)까지 수행
- AI가 더 이상 단순한 ‘자동 완성기’가 아니라 멀티태스킹 협동 엔지니어로 인식
- Flask 창시자 Armin의 말 인용: “우리는 이제 도구를 쓰는 것이 아니라 기계와 함께 일한다”라는 전환점 제시

### AI 코딩 에이전트는 열의 넘치고 빠르지만, 맥락과 판단력은 부족함

- AI 에이전트는 무한한 학습력, 다수 언어∙프레임워크 지식, 빠른 재작성 능력을 가짐
- 인간 특유의 ‘비즈니스 맥락’, 특정 설계 결정의 배경, 조직 고유의 상황 등은 알 수 없음
- 예시: 기술적으로 올바르지만 실제 맥락에서는 부적합한 코드를 자신만만하게 제출할 수 있음
- Armin은 AI 도입 후 일의 30% 효율 향상을 경험, 맡길 일과 직접 챙길 일의 구별이 관건임

### AI를 단순 사용이 아닌 ‘디렉팅’이 진짜 협업을 만듬

- AI의 제안이나 결과를 일방적으로 수용하지 않고, 직접 방향·업무 분배를 조정해야 진짜 생산성이 발생함
- 에이전트가 주니어 개발자라면, 인간이 매니저/리더처럼 가이드·검증·맥락제공 역할을 해야 함
- 실전에서는 AI로 넘길 업무 선정, 직접 챙겨야 할 사항 관리가 핵심
- 이 방식을 ‘Agentic Engineering’이라 명확히 정의함

### 컨텍스트 엔지니어링은 성공적 AI 협업의 핵심 스킬임

- Andrej Karpathy 인용: “컨텍스트 엔지니어링은 예술이자 과학”
- LLM의 컨텍스트 윈도우(입력 토큰)는 비용이 들고, 오히려 과다시 품질이 저하될 수 있음 (50% 채워지면 성능 급락 지적)
- 불필요한 내용 또는 구식 코멘트, 무관 작업 섞임 등은 ‘bad context’를 유발
- 작업 진행 중 잘못된 방향성을 감지하면, 즉시 새 세션으로 시작해 문맥을 재정비하는 것이 중요
- 여러 MCP 서버(확장 도구) 상시 활성화 시, 컨텍스트 오염/비용 증가 → 상황별 엄선 필요

### 기록과 문서 관리로 에이전트 컨텍스트를 최적화할 수 있음

- 현명한 컨텍스트 관리는 세션 내외부에 정보를 기록·보관(스크래치 패드, memory files, agents.md 등)하여, 필요한 순간만 불러오게 함
- 업무 세부 단계에 꼭 필요한 파일/코드/명령어만 컨텍스트로 넣고, 불필요한 것은 최대한 배제
- 예시: 특정 파일 언급 또는 @mentions 활용, MCP 서버 비활성화 등 필요 데이터 선별
- 긴 세션 진행시 적절한 시점에 컨텍스트 요약/압축, 요약본 검사 및 세션 재시작 등 실천 강조
- 작업별 세션 분할(parallel agents)로 맥락 혼합 방지와 업무분담 효율 증가

### 실제 매니저 경험을 통해 컨텍스트 관리의 중요성을 강조함

- 연설자는 헬스케어 소프트웨어기업의 초보 엔지니어링 매니저 시절, 인턴에게 Balsamic(와이어프레임 툴)로 설계도 전달 사례 설명
- 인턴들이 와이어프레임 속 ‘Comic Sans 폰트’, ‘이모지 아이콘’까지 그대로 구현한 결과, 사전 컨텍스트 명시 부재의 문제점을 체험
- 이 경험을 통해 Junior 엔지니어(또는 AI)에게는 “역할·포인트·문제의식”을 명확히 제공해야 올바른 결과 도출 가능함을 확신하게 됨

### 실제 업무 루프는 ‘연구→계획→구현’ 단계로 나눔이 효과적임

- 대다수 사용자는 곧장 코드 구현 요청, LLM은 ‘잘못된 전제’를 대규모 코드 생성으로 연결할 위험 큼
- Dexory 인용: “잘못된 연구는 수백줄의 잘못된 코드로 이어질 수 있다”
- ‘Ask Mode’(리서치) → ‘Architect Mode’(계획) → ‘Code Mode’(구현) 등 목적별 모드 활용
- 리서치(Ask Mode): 시스템 현황 분석, 적용 파일/패턴 탐색, 데이터 흐름 파악, 에지케이스 도출 등만 수행
- 계획(Architect Mode): 실제 생성/수정/이동 파일 페이름 포함, 테스트 전략·범위·비범위까지 명확화된 'plan file' 작성
- 실행(Code Mode): 새로운 세션/컨텍스트로 계획된 구현만 집중, 잦은 커밋과 철저한 리뷰 병행
- 인간의 사고(연구·계획) 단계가 가장 큰 가치와 효율을 창출함을 반복 강조함

### 에이전트 설정 및 모드/문서화로 워크플로우를 개인화 가능함

- 모드별(ask, code, architect 등) 역할 기반 설정
- agents.md(참조/필수 규칙, 명령, 빌드/테스트 필요조건 등)와 skills.md(자주 쓰는 반복 워크플로우) 구분·활용
- 규칙·와크플로우·권한 행위(파일 접근, 테스트 실행, 자동승인 범위 등) 설정을 통해 에이전트 자율성 조정 가능
- 실제 사용하며 설정을 세분화·조정/전환하는 ‘지속적 학습’ 모델 필요

### Kilo Code 등 실제 에이전트 플랫폼에서의 고급 활용법 제안

- 파일, 커밋, 터미널 출력을 빠르게 컨텍스트에 추가할 수 있는 @mention, /command 기능
- VS Code에서 코드 블록 선택-우클릭-Add to Kilo Code로 바로 컨텍스트 편집 가능
- 자동완성 기능(코드∙프롬프트) 및 세션마다 컨텍스트 압축/새작업 시작 지원
- IDE 외에도 CLI, 모바일, Slack 등 다양한 환경에서 에이전트 접근성 확장
- “AI는 협력자–언제 어디서나 ” 경험이 표준화되는 트렌드 언급

### MCP 서버 등 컨텍스트 확장 도구 활용 시 주의점과 방법 제시

- 모델 컨텍스트 프로토콜(MCP) 서버는 외부 도구 연결(예: GitHub API, 최신 프레임워크 문서 등) 지원
- 남용 시 컨텍스트 오염과 토큰 낭비 문제 발생 → 필요 MCP만 일시적 활성화 권장
- 엔터프라이즈 환경의 내부 API와 연계 시: OpenAPI/Swagger가 있으면 그대로, 없으면 마크다운화/참고 URL 등으로 관리, 복잡시 자체 MCP 구축 고려
- 에이전트가 작업한 결과를 PR로 분리, 인간 엔지니어가 리뷰하는 방식 권장

### 반복적 실습과 피드백 루프가 에이전틱 엔지니어링 역량을 완성시킴

- 결국 중요한 것은 특정 도구를 꾸준히 실전 적용하며 '신뢰할 업무 vs 직접 검증할 업무' 경계 감각 형성
- 리서치-계획-구현-피드백 루프를 통해 AI 협동의 품질과 효율을 끌어올릴 수 있음
- 반복 학습을 통해 많은 시니어 개발자들이 “프로그래밍의 재미를 되찾았다”는 평가를 하고 있음
- AI 에이전트가 반복적 단순 업무를 대신함으로써 인간 엔지니어가 더 본질적∙어려운 문제 해결에 집중할 수 있도록 유도함
