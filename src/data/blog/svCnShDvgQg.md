## *지속 가능한 에이전트 구현을 위한 두 가지 경로: 리플레이 방식과 스냅샷 방식* 핵심 요약

- 영상은 장기 실행과 오류 복구가 가능한, 즉 '지속 가능(durable)'한 AI 에이전트를 실제 프로덕션 환경에 배포하기 위한 두 가지 근본적 방식(리플레이 vs. 스냅샷)과 그 구조적 변화를 다룸
- 과거 CGI(1993년), LAMP(PHP) 스택, Ruby on Rails, Node.js, 서버리스 등 30년간 백엔드 인프라는 'stateless(무상태) 컴퓨트' 기반으로 발전해 왔음을 역사적으로 설명함
- 무상태 컴퓨트의 원칙(요청+DB=응답)과 'shared nothing architecture' 구조가 클라우드 시대 웹 백엔드의 표준이었음을 강조함
- 비동기 작업 및 멀티스텝(side effects) 워크플로우 복잡도 증가로 '리플레이(replay)' 기반 '지속 가능한 실행(durable execution)' 엔진(즉, 워크플로우 엔진)이 널리 채택되기 시작함
- 리플레이 방식은 각 작업을 '스텝' 단위로 래핑하여 실패 시 이미 성공한 스텝을 건너뛰고 이어서 실행하도록 해, 신용카드 중복 결제 등 오류를 방지함
- LLM 기반 에이전트 시대엔 코드가 LLM을 제어하는 대신 LLM이 코드를 오케스트레이션하며, 에이전트 세션 구조(지속적인 상호작용)로 변화함
- 리플레이 모델은 긴 에이전트 세션에서는 로그 성장/성능/버전 관리 측면에서 한계가 명확해지며, 최대 실행 시간도 몇 시간 수준이지만 빠르게 증가(4~7개월마다 두 배)하고 있음
- 에이전트의 상태를 '컨텍스트(append-only 로그, 시스템/유저/툴 메시지 등)'와 '실행상태(머신의 스냅샷)'로 분리해 각각 내구성을 확보하는 '스냅샷(snapshot)과 복원(restore)' 방식이 등장
- 실제 파일시스템·메모리·패키지 설치 등 실행 상태 유지가 필요한 상황에서 '머신 스냅샷 및 복원'이 리플레이보다 실용적임을 Firecracker·CRIU 등 기술 사례와 수치로 증명함
- 트리거.dev에서 개발한 오픈소스 툴(Firecracker VM 기반 snap/restore CLI, FC Run/FCrun)은 수백 밀리초 단위의 빠른 복원과 수십 MB로 압축된 스냅샷 등 고성능을 달성해, 차세대 에이전트 백엔드 인프라의 핵심이 될 것임을 시사함

## 세부 요약 - 주제별 정리

### 백엔드 인프라의 30년 역사는 무상태(stateless) 컴퓨트 구조의 확산으로 요약됨

- 최초의 동적 웹 백엔드는 1993년의 CGI(Common Gateway Interface)로, 각 HTTP 요청마다 새로운 프로세스를 포크하여 처리하는 완전 무상태 구조였음
- 이어 등장한 PHP 기반 LAMP 스택(Linux, Apache, MySQL, PHP)은 PHP 프로세스 재사용을 허용했음에도, 요청과 DB의 상태만으로 응답을 생성하는 '공유 없음(shared nothing)' 원칙을 계승
- 뒤이은 Ruby on Rails, Node.js, 서버리스 등 모든 현대 백엔드 기술이 무상태 컴퓨트 및 데이터베이스에 상태를 저장하는 모델을 따름
- 이 패러다임은 '요청과 DB의 값만 있으면 어떤 인스턴스에서도 결과를 재생성할 수 있음'을 강조하며, 클라우드와 대규모 트래픽 환경에 적합한 표준 구조로 자리잡음

### 외부효과(비동기/멀티스텝)의 등장으로 워크플로우와 내구적 실행 엔진이 필수화됨

- 이메일 발송, 신용카드 결제, 이미지 리사이즈 등 단순 비동기 태스크가 점차 다수의 스텝과 외부 작업이 연결된 복잡한 워크플로우(예: 주문 처리)로 진화함
- 멀티스텝 과정에서 일부 단계(예: 영수증 전송)가 실패하면, 전체 프로세스를 다시 실행할 경우 중복 결제 등 오류 발생 위험이 존재
- 이를 해결하기 위해서 약 10~15년 전부터 '워크플로우/지속적 실행(durable execution) 엔진'이 도입됨 (예: 각 과정이 스텝별로 캐싱됨)
- 실패 후 재실행 시 이미 성공한 스텝은 건너뛰고, 실패 지점부터 재실행되어 데이터 무결성과 중복 작업 방지가 쉬워짐

### 리플레이(replay) 모델은 멀티스텝 백엔드의 표준 내구성 확보 방식으로 자리잡음

- 리플레이 모델에서는 모든 결정적(동일 입력에 항상 동일 출력을 내는) 연산만을 스텝 안에 두고, 각 스텝 실행 이력(저널)을 남겨서 내구성과 감사(audit) 트레일 확보
- 복구(recovery) 시점에 원하는 스텝에서 재실행 가능하고, 중간에 사람의 입력이나 외부 이벤트를 기다린 뒤 이어서 처리하는 것도 가능함
- 다만, 모든 코드를 리플레이 시스템에 맞춰 작성해야 하는 제약(결정성, 구조적 rigid함)이 존재하고, 저널의 버전 관리/마이그레이션도 쉽지 않음

### LLM 등장 후, 에이전트 워크플로우의 본질과 내구성 요구가 급격히 바뀌고 있음

- LLM 도입 초기에는 워크플로우의 일부 한정된 스텝(예: 텍스트 분류)만 LLM API로 처리하는 형태라 기존 리플레이 파이프라인과의 호환성이 높았음
- 툴 콜링(tool calling)이 등장하면서, 에이전트 루프 구조(LLM이 자체적으로 여러 코드를 수행/지시하고 세션을 주도하는 구조)로 전환
- 이 구조에선 더이상 '트랜잭션(짧고, 시작과 끝이 분명한)'이 아니라 '세션(장기적, 유저가 중단할 때까지 지속)' 개념이 중요
- 사용할수록 각 LLM 호출·툴 호출이 리플레이 로그에 계속 기록되어, 세션이 길어질수록 로그 성장, 효율성 저하, 관리 복잡성 등 리플레이 모델의 한계가 드러남
- 실제로 현재 에이전트가 '실제로 의미 있게 지속될 수 있는 시간'의 한계는 불과 수 시간 단위이지만, AI·에이전트 발전과 함께 4~7개월마다 두 배로 늘어나는 추세임(수치 제시)

### 에이전트 내구성은 '컨텍스트 로그'와 '머신 실행 상태' 두 가지로 나뉘어 별개로 관리되어야 함

- 에이전트의 상태는 (1)컨텍스트 로그(시스템 메시지, 유저·툴 메시지 등 세션 히스토리)와 (2)실행 상태(코드·파일시스템·메모리 등 머신 환경)로 구분됨
- 컨텍스트 로그는 append-only(덧셈만 허용) 구조로, DB/오브젝트 스토리지/분산 파일시스템 등 표준화된 저장소에 보관 가능
- 이 방식은 코드 버전이 바뀌거나 머신이 다운되어도, 동일 컨텍스트로부터 이어 실행(continuation)이 가능함
- 실행 상태는 예를 들면, GitHub 레포지토리 클론, 패키지 설치, 데이터셋 메모리 적재, 개발 서버 구동 등의 상황처럼, 단순 로그로 복구 불가능한 환경을 포함함

### 실행 환경 내구성은 '스냅샷-복원(snapshot-restore)' 방식이 리플레이보다 실용적이고 비용 효율적임

- 리플레이로는 복잡한 머신 상태(파일시스템, 오픈 파일, 세션 메모리, 설치된 패키지 등) 재현이 어렵고 비효율적임
- 사용성 예시: 대화 도중 유저가 일시 중단(점심 시간 등) 시, 머신 전체를 계속 켜 둘 수도 없고, 로그인 세션, 패키지 상태, 실행 서버 등까지 커버해야 함
- 스냅샷-복원은 머신 전체 상태를 저장 후, 새 메시지 도착 시 해당 상태로 복구해 지속성을 보장
- 실제 운영 시 휴면/실행시 마다 스냅샷을 생성 및 복원해, '중단점'에서 손실 없이 이어서 작업 가능

### IBM 메인프레임, CRIU, Firecracker 등 스냅샷/복원 기술 발전사가 소개됨

- 1966년 IBM 메인프레임에선 이미 장시간 작업 실패에 대비해 '체크포인트-리스토어' 개념이 도입되어 있었음 (비용·효율성 목적)
- 2011년 등장한 CRIU(Checkpoint/Restore In Userspace)는 리눅스 프로세스를 유저 공간에서 스냅샷하고 메모리상에서 복원하는 솔루션임. 병렬주입(parasite) 및 프로세스 상태 덤프→복원 패턴
- 트리거.dev에서는 2024년 CRIU 기반 백만 건 이상 스냅샷/복원을 프로덕션에 적용, 프로세스 투명성 및 컨테이너 런타임(도커 등) 호환성 확보함

### 기존 스냅샷/복원(특히 CRIU)의 한계 및 최적화 방법이 명확히 다뤄짐

- CRIU는 개별 프로세스 상태만 스냅샷 가능. FFMpeg, 크롬 등 복합적 멀티 프로세스/멀티파일 환경에선 한계
- 스냅샷 시 파일시스템의 '오픈 파일'만 포함, 작업 중 새로 생긴 파일/닫힌 파일은 복원 불가
- 컨테이너 호환성 확보는 좋으나, 저장/전송/복원 시 도커 레지스트리 등과 연동시 느려짐

### Firecracker 마이크로 VM 기반 스냅샷은 머신 전체 상태를 빠르게, 저렴하게 복구 가능함

- Firecracker microVM 도입으로 단일 프로세스가 아닌 VM 전체를 스냅샷-복원 가능
- 단점: 머신 용량(예: 512MB)을 그대로 스냅샷/복원할 경우, 저장/네트워크 비용이 큼
- 최적화: 시커블 압축(필요할 때만 해당 메모리 페이지 복원), 레이어링(공통 레이어 재사용) 등을 도입해 스냅샷 크기를 14MB 이하로 압축(퍼포먼스·저장비용 균형 가능)
- 실제 스냅샷 약 1초, 복원 약 수백 밀리초 수준으로, 프로덕션 환경에 적합한 속도를 달성함(즉, 지연 지표 제시)

### FC Run(Firecracker 기반 CLI 툴) 공개 예고와 성능 지표를 통해 실용성 입증

- 트리거.dev가 곧 오픈소스로 공개 예정인 FC Run (혹은 FCrun)은 Firecracker VM을 도커 커맨드와 유사하게 관리 가능
- 사용자는 Alpine 등 원하는 이미지를 초고속으로 실행, 스냅샷, 복사, 복원 등 가능
- 벤치마크 결과: VM 부팅-인터넷 접속 대기 시간(TTI) 기준 분당 15,000회 VM 기동 가능, 실시간 영상 처리도 가능한 수준(30 FPS)
- 타임 투 스냅샷/복원이 각각 1초 이내/수백 ms 수준으로, 대규모 에이전트 백엔드 시스템에 적합함

### 컨텍스트 로그와 스냅샷-복원을 결합함으로써 미래 에이전트 인프라의 내구성과 상태 관리가 완성됨

- 정리: 에이전트 백엔드의 내구성 확보는 (1)컨텍스트 로그(append-only)와 (2)실행상태 스냅샷의 이중 전략이 필수
- 이 구조는 코드 버전, 세션 중단, 머신 장애 등 다양한 상황에서도 신뢰성 있게 업무를 이어갈 수 있게 해줌
- 이는 30년간 표준이던 무상태 컴퓨트의 패러다임이, 에이전트 및 LLM의 요구에 따라 '상태 지향 컴퓨트(stateful compute)' 시대로 변화함을 상징
- 미래의 생산성 있고, 오류에 강한 에이전트 시스템을 위한 핵심 인프라 전략임
