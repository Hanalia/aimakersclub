## *에이전트형 AI 시스템의 프로덕션 평가: Meta Superintelligence Labs의 니샨트 굽타 발표* 핵심 요약

- 영상은 Meta Superintelligence Labs의 소프트웨어 엔지니어링 테크 리드 니샨트 굽타가 에이전트형 AI 시스템의 프로덕션 평가(evaluation)의 진화를 다루는 프레젠테이션임
- 기존 챗봇 기반 AI 평가(정답 출력 여부)와 달리, 최신 에이전트 시스템은 계획 수립, 도구 호출, 정보 검색, 워크플로우 실행 등 복잡한 행동을 수행함
- 벤치마크 점수가 높더라도 실제 프로덕션 환경에서는 툴 실패, API 장애, 사용자 다양성, 장기 실행 등 다양한 변수로 인해 예측 불가성이 높음
- 에이전트형 시스템의 실패 유형은 기억/검색/안전 등 인프라 하위 계층부터, 추론 오류, 도구 실행 오류, 다중 에이전트 조정 실패 등으로 다각화됨
- SRE(서비스 신뢰성 엔지니어) 관점의 신뢰성, 가용성, 레이턴시, 비용, 복구율 등이 평가의 핵심 지표가 됨
- 효과적인 평가는 정답 여부가 아닌, 실제 시나리오 및 워크플로우 기반으로 시스템의 종단 행동을 측정해야 함
- 오프라인 벤치마크와 실제 프로덕션 평가, 시나리오 기반 평가의 피라미드 구조를 통해 신뢰도 높은 시스템 운영이 가능함
- 인간 평가자는 단순 예비 시스템이 아니라, 정답성, 신뢰성, 안전성 등 자동화 평가가 놓치는 신호를 제공하는 필수 구성 요소임
- 에이전트 시스템은 지속적으로 변화(모델, 프롬프트, 도구, 사용자 등)하기 때문에 드리프트 감지와 연속 평가 및 관찰(Observability)이 필수임
- 현재 업계는 평가를 별도 단계가 아닌, 컨트롤 플레인과 실행 플레인에 내재화된 상시 인프라로 통합하는 방향으로 진화 중임

## 세부 요약 - 주제별 정리

### 에이전트형 AI 시스템 평가의 본질적 변화가 필요함

- 기존 AI 시스템 평가는 대부분 벤치마크 점수(예: 90% vs 92%)에 집중되어 있었음
- 최신 에이전트형 AI는 답변 생성뿐만 아니라 계획, 도구 호출, 정보 검색, 워크플로우 실행, 실환경과 상호작용 등 복합적 기능을 수행함
- 따라서 "모델이 올바른 답을 생성했는가?" 대신 "시스템이 전체적으로 올바르게 동작했는가?"가 평가의 핵심 질문이 됨
- 프로덕션 환경에서의 신뢰성(reliability) 평가가 오프라인 벤치마크와 확연히 다름을 강조함

### 벤치마크와 프로덕션 환경의 괴리가 크며, 신뢰성 측정이 실제로 더 중요해짐

- 오프라인 벤치마크는 모델의 능력을 측정하지만, 실제 서비스에서는 시스템의 동작과 신뢰성(예: API 장애 등 전반적 동작)이 더 중요함
- 벤치마크 점수가 높아도 프로덕션에서는 툴 장애, 문맥 변화, 사용자 다양성 등으로 인해 신뢰도가 낮을 수 있음
- 시간이 지날수록 벤치마크 성과와 실제 서비스 성과 사이의 격차가 커짐
- 실제로 많은 팀들이 높은 점수를 달성했지만, 실 서비스에서는 불안정한 시스템을 경험함

### 에이전트 시스템은 단일 답변의 정확성보다 전체 워크플로우와 행동을 평가해야 함

- 에이전트형 시스템의 행동 평가 요소: 계획 품질, 도구 사용, 워크플로우 실행, 실패 복구, 의사 결정
- 평가방식이 "답변" 중심(output-centric)에서 "행동" 중심(behavior-centric), 즉 "워크플로우" 검증으로 이동함
- 이를 위해 기존 평가 아키텍처와 근본적으로 다른 구조가 필요함

### 에이전트형 시스템에서 실패 모드는 계층적으로 더욱 복잡해짐

- 기존 LLM의 주요 실패인 환각(hallucination)은 프로덕션 기준에서는 단지 한 계층에 불과함
- 바닥(인프라)에는 기억(memory), 검색(retrieval), 안전(safety) 실패, 중간에는 추론 및 계획 오류, 잘못된 도구 실행, 최상위에는 다중 에이전트 조정 실패 등이 있음
- 모델 출력만으로 평가지표를 삼으면 실제 위험요인을 놓치기 쉬움

### SRE(서비스 신뢰성 엔지니어) 관점의 평가 프레임워크를 채택해야 함

- 연구자 중심의 정답률(accuracy) 측정보다 SRE(생산 엔지니어)의 평가 지표(신뢰성, 가용성, 레이턴시, 비용, 복구율 등)로 접근해야 함
- 목표는 높은 벤치마크 점수가 아니라, 예측 가능하고 신뢰할 수 있는 시스템 결과가 됨
- 정답률(accuracy)은 입력지표 역할, 신뢰성이 핵심 목적 척도가 됨

### 벤치마크, 시나리오 기반 평가, 프로덕션 텔레메트리의 계층적 운영이 필요함

- 벤치마크는 반복성·확장성 측면에서 유의미하나 실제 서비스 운영가치는 제한적임
- 중간 단계로 시나리오 기반 평가(실제 워크플로우 시뮬레이션)가 현실성 높은 평가 방식으로 중요해짐
- 실 서비스 단계(상위 계층)에서는 실사용 데이터(프로덕션 텔레메트리)가 가장 중요한 평가 신호(source)가 됨

### 에이전트 평가의 중심은 프롬프트가 아닌 시나리오 기반이어야 함

- 오프라인 프롬프트 평가에서 실제 워크플로우(고객지원, 코드 생성, 리서치 등) 내에서의 에이전트 태스크 완수율, 도구 실행 정확성, 계획 품질, 리소스 사용량 등 다양한 실적을 계량
- 프롬프트가 아니라 현실적 시나리오에서 평가하고, 고도화·대규모화 될수록 리소스 사용량이 기하급수적으로 증가함
- 핵심 교훈: 에이전트 평가는 프롬프트 기반이 아닌 시나리오 중심으로 전환해야 함

### 실 서비스(프로덕션) 상의 모든 인터랙션 로그가 평가 데이터로 활용됨

- 실 서비스에 진입하면 모든 상호작용이 평가 신호로 전환됨
- 실행 기록, 사용자 결과, 에스컬레이션(인간 개입), 실패, 피드백 등 다양한 신호를 수집
- 프로덕션 데이터가 그 어떤 오프라인 데이터보다 방대하고 대표성 높은 평가 소스가 됨

### 인간 평가자는 단순 백업이 아니라, 자동 평가 사각지대를 메우는 결정적 역할을 함

- 많은 조직에서 인간을 장애 시 백업용으로 보지만, 실제로는 평가자이자 신호 제공자로 더 중요함
- 인간은 자동화 시스템이 측정하지 못하는 정답성, 신뢰성, 신용도, 안전성을 직접 피드백함
- 가장 성공적인 시스템은 자동 평가와 표적 인간 검토를 결합함

### 지속적 드리프트와 평가/관찰의 상시 통합이 신뢰성을 확보함

- 에이전트 시스템은 모델 업데이트, 프롬프트·도구 변경, 사용자 행동 변화 등 계속해서 관리되지 않는 변화가 발생함
- 신뢰성 저하가 서서히 진행되면서 문제(성공률 하락, 에스컬레이션 증가, 도구 실패 등)는 누적됨
- 지속적인 모니터링·관찰(Observability)와 연속 평가가 필수적임
- 평가·관찰을 위해서는 모델 추론 경로, 도구 호출, 메모리 접근, 실행 타임라인, 상태 전이 등 취합이 필수
- 전통적 로그만으로는 부족, 세분화된 trace(분산 추적)가 요구됨

### 평가 사이클은 시점별 테스트가 아니라 상시 인프라로 내재화되어야 함

- 평가는 배포 전 테스트(일회성)이 아니라, 배포 후에도 끊임없이 동작하는 서비스가 됨
- 인간이 엣지 케이스 리뷰 → 피드백 반영 → 오프라인 시나리오 검증 → 데이터셋 개선 → 다시 검토하는 루프가 반복됨
- 평가가 QA의 일부 또는 별도 도구가 아니라, 실제 운영 인프라에 통합됨

### 주요 비즈니스 메트릭은 신뢰성·복구율·운영비용 등 종합적 지표 중심으로 전환됨

- 태스크 완료율(Task Completion): 실질 서비스 제공 가치 측정
- 도구 성공률(Tool Success): 운용 신뢰성 지표
- 에스컬레이션률(Escalation Rate): 인간 개입 부담
- 안전 규정 위반(Safety Violations): 리스크 노출도
- 레이턴시(Latency): 사용자 경험 품질
- 비용(Cost): 확장성과 운영성 결정
- 복구율(Recovery Rate): 시스템 회복 탄력성
- 단순 정확도(Accuracy)는 중요하지 않다는 것이 아니라, 비즈니스 성과의 전체 요건을 대변하지 않음

### 평가·관찰 체계는 기존의 보조 프로세스에서 컨트롤 플레인에 통합되어 서비스 기반 인프라로 정착됨

- 평가 시스템이 개별 도구, 테스트 단계가 아니라 컨트롤 플레인의 핵심 운영 영역으로 정착
- 컨트롤 플레인은 연속적으로 시스템 관찰, 텔레메트리 수집, 시뮬레이션, 인간 리뷰를 조율
- 실행 플레인은 실제 워크플로우 실행 담당
- 평가와 거버넌스를 담당하는 운영구조가 차세대 에이전트형 AI 시스템의 표준 패턴이 됨

### 결론: 모든 AI 조직들이 실환경 신뢰성 평가와 평가 인프라 내재화로 전환해야 함

- 벤치마크는 여전히 필요하지만 충분조건이 아님
- 에이전트 시스템 평가는 반드시 전체 워크플로우(프로세스, 행동) 중심이어야 함
- 실사용 데이터(프로덕션 텔레메트리)가 가장 강력한 평가 신호원임
- 실제 신뢰성이 모델의 정확도보다 더 중요함
- 평가는 더 이상 테스트나 QA가 아니라, 핵심 인프라로 업그레이드되어야 함
