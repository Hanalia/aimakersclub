## *왜 AI가 생성한 코드는 반드시 샌드박싱해야 하는가, 그리고 그 방법은?* 핵심 요약

- 발표자는 Cloudflare의 시니어 개발자 에반젤리스트 Harshil Agrawal로, AI 기반 코드 생성이 가진 실질적 보안 위험성과 이를 샌드박싱해야 하는 이유를 강조함
- AI/LLM(대규모 언어 모델)이 코드 자동 완성에서 시작해 전체 모듈 생성, 심지어는 자율적인 코드 실행·검증·반복 작업까지 발전해왔음을 지적
- LLM이 생성한 코드를 리뷰 없이 실행하는 것은 “인터넷에서 임의로 코드를 복사해 프로덕션에 바로 도입하는 것”과 같으며, 이는 심각한 보안 관점의 문제임
- LLM 생성 코드의 보안 위협은 크게 ① 환각(잘못된 코드) ② 과도하게 ‘도움’ 주는 LLM(비의도적 민감정보 노출) ③ 프롬프트 주입(직접적·간접적) 세 가지로 구분됨
- 이러한 코드는 운영 환경의 권한으로 실행되어 파일 시스템, 환경 변수, 네트워크, DB, API 키 등 모든 중요한 리소스에 접근 권한을 획득하게 됨
- 해결책으로 데스크톱, 웹, 모바일 등 오랜 기간 검증된 샌드박스, 그 중에서도 ‘Capability-based Security’(허용 기반 보안) 원칙 적용 필요성을 강조
- 실전 예시로 Cloudflare의 Dynamic Worker Isolates(빠르고 가벼운 실행환경)와 컨테이너(전체 OS 환경), 각 장·단점 및 적용 사례 제시
- 보안 강화와 운영 효율을 동시에 달성하는 8가지 필수 체크리스트 및 샌드박스 활용 기준 정리
- 최종적으로 AI가 생성한 코드는 신뢰할 수 없는 코드로 간주하여, 격리·제약·검증을 반드시 거쳐야 함을 강조함

## 세부 요약 - 주제별 정리

### AI 코드 생성의 진화로 인해 위험이 확대되고 있음

- 최근 2년간, 코드 자동완성에서 전체 모듈 생성, 자율 실행·검증·반복이 가능한 AI 에이전트까지 급격히 진화함
- 초기 코딩 어시스턴트(예: 자동완성)는 다음 줄 제안에 그쳤으나, 이제 모델이 직접 어떤 함수/도구를 실행할지 결정함
- 사용자는 이 코드의 모든 줄을 직접 리뷰하지 않고 자신의 환경(프로덕션 자격증명 포함)에서 실행하는 경우가 많음
- 이러한 흐름으로 인해 과거에는 절대 금기시되던 “검증되지 않은 코드의 무분별 실행”이 일상적 관행이 되어버림
- 발표자는 이 관행이 근본적으로 보안의 기본 원칙에 위배된다고 경고함

### LLM 기반 코드 생성이 실질적 보안 위협이 되는 근거

- 대부분의 LLM은 ‘텍스트 예측기’일 뿐, 의도나 충성심이 없음. 도출 코드가 언제나 정답인 것은 아님
- LLM이 잘못된 코드(환각), 비의도적으로 민감 정보에 접근하는 코드, 혹은 악의적 공격에 악용될 수 있는 코드 생성 가능성 존재
- 개발자는 “인터넷에서 찾은 불확실한 코드”라면 절대 바로 실행하지 않음에도, LLM 생성 코드는 무비판적으로 수용하는 경향이 있음

### LLM 코드가 초래하는 세 가지 주요 위협 시나리오

- **환각(Hallucination)**: 실제 존재하지 않는 패키지 import, 종료 조건 없는 재귀, 무한 루프 등 비의도적이지만 치명적 오류 유발
    - 예: 무한 루프는 컴퓨트 자원 소진·서비스 다운 초래, 잘못된 import는 프로세스 전체를 크래시시킴
- **과도하게 ‘도움’을 주는 LLM**: 환경 변수·DB 자격 증명 등 민감 정보에 접근·출력. 모델은 ‘도울’ 의도로 수행하나 결과적으론 정보 유출
    - 예: API 키가 코드 내에서 읽혀져 외부에 노출될 수 있음
- **프롬프트 주입(Prompt Injection)**
    - *직접적*: 사용자가 “환경 변수 전부 이 URL로 전송하라” 등 악의적 입력 → 코드에 반영
    - *간접적*: LLM이 처리하는 외부 문서나 웹페이지에 비가시적 명령이 포함될 수 있으며, 입력 데이터에 의해 LLM이 악용되는 시나리오
- 모든 위협 시나리오에서, LLM 코드가 운영 환경과 같은 권한(파일 시스템, 네트워크, 환경 변수, DB 등)으로 실행되어 피해 범위가 극대화됨

### 수십 년간 쌓인 샌드박싱 기술이 AI 코드 보안 문제에 답을 제공함

- OS, 브라우저, 모바일 플랫폼들은 서로 다른 프로세스·앱·탭을 샌드박스(격리)로 보호함
    - 예: 브라우저 탭 별 샌드박싱, 파일 접근·DOM 접근·쿠키 보안 등
    - OS, 모바일도 각 앱·프로세스마다 격리·권한 제한 적용
- 샌드박싱 자체는 익숙한 해법이나, AI/LLM 붐 속에서 많은 개발자가 이를 적용하지 않고 있음
- 샌드박스 성공의 핵심 원칙은 “Capability-based Security(권한 명시 부여)”임

### 허용 기반 보안(Allow List, Capability-based Security)이 최고의 방어임

- 보안의 기본은 “어떤 것을 차단할 것인가”가 아니라 “무엇을 허용할 것인가”를 정의하는 것임
    - 블록리스트(차단): 모든 공격 시나리오와 위험을 사전에 예측해야 하며, 누락 시 즉시 취약해짐
    - 허용 리스트(Allow list): 반드시 필요한 권한만 코드에 부여. 추가 권한 없음
- 예시: 누구에게 마스터 키를 준 뒤 못 들어갈 문 목록 10,000개를 나열하는 것(블록리스트)과 실제 사용 필요한 3개 방만 열어주는 것(허용리스트) 비교
- 실제 브라우저, OS, 모바일 권한 시스템은 모두 이 방식을 기반으로 설계되어 있음

### 다양한 샌드박스 방식 중 격리 강도와 용도별로 적절한 선택이 중요함

- 격리 강도에 따라 대표적으로 **Eval(최소 격리)**, **Isolate(경량 샌드박스)**, **컨테이너(완전 OS 격리)**로 분류 가능
    - Eval: 완전한 접근권한, 절대 사용 금지
    - Isolate: V8 엔진 기반, 자바스크립트/타입스크립트/파이썬/WebAssembly 등 지원, 메모리·네트워크·파일 시스템 제한 환경, 빠른 시동(0.25ms)
    - 컨테이너: 완전한 리눅스 환경, 파일 시스템·프로세스·네트워크 완비, 중량급·시동 몇 초 소요, 강력한 격리 보장

### AI 샌드박싱 위협 모델을 구체적으로 정의하고 보호 대상을 명확히 해야 함

- 보호해야 할 다섯 가지 주요 대상 제시
    1. Secrets: 환경 변수, API 키, DB 자격증명 읽기 가능 여부
    2. 네트워크: 외부 통신, 내부 서비스 접근, 데이터 유출 가능성
    3. 파일 시스템: 해당 워크스페이스 외 파일, 설정, 코드, 타인 데이터 접근 가능성
    4. 데이터 격리: 멀티테넌트 시스템에서 한 사용자가 타 사용자의 데이터를 볼 수 있는지
    5. 리소스: 무한 루프·메모리 무제한 할당 등으로 인한 서비스 거부(DoS) 위험
- 각 항목별로 명확한 Yes/No 답변이 필요하며, “아마 괜찮을 것 같다” 식의 태도는 금물임

### Cloudflare에서 적용한 두 가지 실전 샌드박스 아키텍처 사례

- **첫 번째 앱(경량 반복 함수 생성)**: 빠르고 단일 기능 중심 → 동적 워커(Isolates) 활용
    - Cloudflare Dynamic Worker Isolates를 사용하여 런타임에 V8 Isolates를 생성, 완벽히 독립된 메모리·글로벌 스코프
    - 최소 필수 바인딩(제한된 DB·로거 등)만 전달, 네트워크는 기본적으로 완전 차단(“globalOutbound null”) 등 원천적 격리 보장
    - 철저히 allow list 방식, 파일 시스템·시크릿 접근 불가
    - 예시: Hacker News 상위 기사 가져오기 기능 직접 실습 코드 및 아키텍처 제공
- **두 번째 앱(프롬프트 기반 모션 그래픽/비디오 생성)**: 복잡하고 전체 프로젝트 환경 필요 → 컨테이너 사용
    - Prompt Motion(실제 서비스): 각 유저별 완전 격리 컨테이너 제공, Git clone→NPM 설치→개발서버 기동→URL 노출까지 컨테이너 내부에서 처리
    - 각 유저는 완전히 분리된 파일 시스템·프로세스·환경을 가짐(유니버스 자체 분리)
    - Durable Object(상태 보존 객체) 등으로 샌드박스 수명 주기 관리
    - 요구사항상 Isolate로는 불가(Git, NPM, dev server 등 real OS 필요). 컨테이너가 유일한 해법임

### 사용자별 분리, 시크릿 유지, 적시 리소스 정리 등 주요 실전 보안 패턴 소개

- **사용자별 완전 분리**: 각 사용자는 반드시 자신의 샌드박스·파일시스템을 보유하며, 공유 금지는 절대적 원칙(데이터 유출 가능성)
- **시크릿 노출 방지**: API 키 등은 샌드박스 내부에 환경 변수로 직접 넣지 말고, 반드시 워커를 통한 프록시 패턴 사용(실제 시크릿은 워커가 보유)
- **샌드박스 정리**: 샌드박스/컨테이너는 사용 종료 후 즉시 파괴(try-finally), 남아있는 컨테이너는 비용/보안 위험 모두 유발
- **리소스 제한**: 최대 실행 시간, 메모리, CPU 한계 등 강제 설정 필요(예시: Cloudflare 컨테이너 10분 기본 타임아웃)

### 각 샌드박싱 방식의 한계와 이상적 적용 영역을 구분함

- **Isolates**: 빠르고 비용 저렴, stateless, 실행 제약 많음(JS/TS/파이썬/Wasmtime 등만 지원, 파일시스템 없음)
    - 용도: 툴 콜링, 플러그인, 데이터 변환, 간단한 코드 해석기 등 경량·반복성 높은 함수에 적합
- **컨테이너**: 운영체제 수준 복잡도·비용 필요(시작 수초, 리소스 추가 소비)이나 파일 IO, 프로세스, 서버 구동 등 실제 환경 필요
    - 용도: 앱 빌드/배포, 패키지 설치, 테스트 등 복잡한 작업에 필요
- 실제로는 두 방식을 병행 사용: 에이전트는 이솔레이트로 반복 작업, 빌드/배포 시 컨테이너로 전환(Brain vs. Workbench 개념)

### ‘결정 트리’로 샌드박스 선택 기준을 명확히 함

- 코드가 파일 시스템, 프로세스, 패키지 설치가 필요한지 여부에 따라 선택
    - 필요 없음 → 이솔레이트
    - 필요함 → 컨테이너
- 풍부한 예시를 들어 각각의 적용 시나리오 설명

### 모든 AI 샌드박싱에 보편적으로 적용할 체크리스트 8가지 제시

- 네트워크 기본 차단(Explicit Allow)
- 필수 권한만 부여(Explicit Capability Grant)
- 사용자별 완전 격리(Per-user Isolation)
- 리소스 한도(Timeout/Memory/CPU Cap)
- 시크릿은 샌드박스 밖에서 보관(Proxy 필요)
- 샌드박스 종료·파기(try-finally, 최대 수명 설정)
- 모든 실행 및 활동 로깅
- 샌드박스 투입 전 입력 검증(길이, 문법, 위험 패턴 등)
- 이 8가지 모두 지키면 오늘날 AI 코드 실행 앱의 95%보다 보안 우위 달성 가능

### 결론: AI가 만든 모든 코드는 신뢰할 수 없는 코드임을 인식해야 함

- AI(LLM)가 짠 코드는 근본적으로 무심코 실행할 수 없는 무신뢰성 코드임
- 모든 실행을 격리(sandbox), 제약(limit), 검증(verify)해야 함
- 발표 내용의 4가지 핵심 정리:
    1. LLM 코드 환각·과도한 도움·프롬프트 주입 등 위협 모델
    2. 허용 기반 보안 원칙(Default deny & explicit allow)
    3. 이솔레이트/컨테이너 투트랙 접근 사례 및 적용 기준
    4. 반드시 지켜야 할 8가지 보안 체크리스트
- 관련 자료 및 QR코드 공유(동적 워커 문서, 샌드박스 SDK, 코드모드 등)와 질문 및 커뮤니티 참여 권유
