## *에이전트 관찰성에 대해 꼭 알아야 할 모든 것* 핵심 요약

- 이 영상은 Raindrop의 CEO와 엔지니어가 AI 에이전트의 관찰성(agent observability)에 대해 심도 있게 설명하는 자리로, AI 에이전트의 모니터링이 기존 소프트웨어와 어떻게 다른지 조목조목 다룸
- AI 에이전트는 입력과 출력의 조합이 무한하며, 행동이 비결정적이기 때문에 기존의 단순 테스트(평가, eval)만으로는 한계가 있다고 강조
- 병원, 금융, 군사 등 고신뢰성이 요구되는 분야에서 AI 에이전트가 활용되면서, 에이전트 관찰성의 중요성과 함께 발생하는 잠재적 위험성도 커지고 있음
- 전통적인 테스트(예: 유닛테스트)도 중요하지만, 실제 서비스 운영(프로덕션)에서는 실시간 모니터링 및 이슈 탐지가 훨씬 더 중요하다고 주장
- 관찰 지표(signal)는 크게 명시적(explicit) 신호(에러율, 지연시간, 비용 등)와 암시적(implicit) 신호(거절, 사용자 불만, 실패, NSFW 등)로 구분하여 명확히 설명
- 암시적 신호는 이슈 탐지에 매우 유용하며 클러스터링, 분류기, 정규표현식(regex) 등 다양한 방식으로 탐지할 수 있고, 실시간 알림 기능과 연계 가능함
- 실험(AB Test, 신호 기반 배포)은 신호의 변화를 통해 서비스 개편의 효과를 측정, 빠른 시장 적응 및 문제 회피에 매우 효과적임
- Raindrop은 관측 데이터에서 문제 신호 자동 탐지, triage agent(자체 에이전트로 이슈 요약/분석), Deep Search, Trajectory Visualization 등 다양한 고급 기능 제공
- 셀프 다이애그노시스(self diagnostics) 기능으로 에이전트 스스로 문제, 격차, 셀프코렉션 등을 보고할 수 있게 함
- 실제 코드 워크숍(깃헙 레포)을 통해 손쉽게 에이전트의 셀프 다이애그노시스 기능을 적용할 수 있음을 단계별로 시연
- Raindrop의 이벤트 및 신호 데이터는 BigQuery, Snowflake 등과 연동 가능하며, 툴 에러, 유저 불만, capability gap 등도 시계열·분류별로 모니터링 가능
- 고객별 실제 활용사례, 시스템 대규모 운영 시의 통계적 유의미성, feature flagging, experiment 병렬 처리, SDK 지원 등 실무적 고려사항도 폭넓게 다룸

## 세부 요약 - 주제별 정리

### AI 에이전트의 관찰성은 복잡성과 활용범위 확대로 인해 필수적 문제로 대두됨

- Raindrop은 AI 엔지니어들이 배포된 에이전트에서 일어나는 문제를 찾고 추적 및 수정하도록 지원하는 플랫폼임
- 기존 소프트웨어 오류와 달리, AI 에이전트의 에러는 입력·출력 조합의 무한성, 비결정성, 외부 도구·시스템과의 임의적 상호작용 등으로 더욱 복잡함
- 요즘 에이전트는 복합적인 도구 호출, 세션의 장기화(수시간 작동), 출력을 다른 서브에이전트에 재귀적으로 위임 등으로 더 예측 불가능해짐
- 에이전트가 의료, 금융, 군사 등 고위험 분야에서도 실전 배치됨에 따라 한 번의 오작동이 치명적 결과를 초래할 수 있음
- 기존 eval(평가 기반 테스트)만으론 모든 edge case를 포괄할 수 없고, 모니터링의 필요성이 더욱 강조됨

### 전통적 소프트웨어 테스트를 넘어 실제 운영 환경 모니터링이 필수임을 주장함

- 전통적인 유닛테스트와 eval은 중요하지만, 실제 운영(prduction) 환경에서의 이슈를 포착하고 빠르게 대응하는 모니터링이 핵심임
- 생산환경에서의 longtail 이슈(희귀·복잡한 오류)는 사전 테스트로는 포착 불가능하며, 실시간 모니터링만이 효과적임
- Raindrop은 이 틀을 "인류의 마지막 문제(Humanity’s last problem)" 중 하나로 표현, 인간이 더 이상 AI 에이전트를 추적·모니터링할 수 없어질 위기를 언급

### 관찰 신호는 명시적·암시적 신호로 나뉘며, 그 활용법이 매우 상세히 설명됨

- 명시적 신호: 에러율, 도구 에러율(tool error rate), 처리 지연(latency), 비용, 사용자 재생성(regeneration) 등 확인 가능한 객관적 수치
- 암시적 신호: 거절(“제가 할 수 없습니다”), 태스크 실패, 사용자 불만(“이게 뭐냐”, “틀렸어요”), 콘텐츠 부적합(NSFW), 탈옥(jailbreak) 등 의미적·정성적 지표 포함
- 이 신호들은 각각 개별 추적이 가능하며 급등·급락시 알림 설정(예: user frustration spike) 및 원인 파악에 이용
- Raindrop은 이런 신호를 기본 내장(out-of-the-box)으로 제공, 사용자가 직접 설정 가능

### 암시적 신호(특히 사용자 불만·거절 등)는 다양한 방법으로 정확히 탐지됨

- 분류기(classifiers): 이슈별로 binary classifier를 적용, LM(언어모델)을 ‘재판관’으로 쓰는 단순 평점 방식보다 더 효과적임
- 사용자 거절, 실패, 불만, NSFW, 탈옥, 긍정 사례(‘win’) 등 각종 신호를 각기 나눠 탐지
- 예시: 구체적으로 “You’re wrong”, “This sucks”, “질문 안했어요” 등 피드백을 탐지하여 매일 프러스트레이션 비율 보고/알림
- 실시간 알림(스파이크 감지) 및 간단한 설정(알림 추가) 기능 제공

### 정규표현식(regex)은 실제 코드베이스 예시를 통해 효과적 신호 탐지 수단임이 설명됨

- 클로드(Claude) 코드 소스가 유출된 사례에서 keywords.ts 파일에 방대한 정규표현식이 프로덕션 이슈(불만, 욕설 등) 감지에 사용된 사례 설명
- 해당 정책은 이슈 발생률을 추적하여 제품 배포 후 변화를 수집, 저비용·고효율 모니터링의 실제 전례로 제시

### 실험(AB Test) 기반으로 기능 변경 효과를 실시간 신호로 검증할 수 있음

- 예시: 프롬프트 버전 변경(prompt 2.4 배포시) 후 사용자 불만 37%→9%로 대폭 감소 등 구체적 수치 공유
- 실험군/대조군을 나누어 배포, 도구 사용 패턴 및 특정 신호 변화(예: 도구 사용 수 증가)도 함께 관찰
- 기존 단순 eval 방식도 유효하나, 실제 운영 데이터 기반 모니터링이 더 중요한 인사이트 제공

### 대규모 운영 및 통계적 유의미성 확보 문제, 실무 적용상 병목 및 해소책 소개

- 몇백 건 이상의 이벤트만 쌓여도 전체 로그를 일일이 읽는 것은 불가능하며, 이때부터 통계적 신호가 의미를 가짐
- feature flag, 실험군 자동 분배, 메타데이터 기반 실험 설정 등 다양한 운영 노하우 공유
- LLM을 모든 출력에 적용해 감정신호 분류는 비효율적이므로, 커스텀 모델을 훈련시켜 저렴하고 빠르게 대규모 감정신호 탐지 활용

### 셀프 다이애그노시스(self diagnostics)는 에이전트가 직접 문제나 격차를 보고하도록 설계됨

- OpenAI 최신 논문(2023년 12월 발표)을 참고, 언어모델이 자가 고백(self-confess, 예: 버그·불일치·우회규칙 등)을 유도
- 예: 유닛테스트 수정 과제에서 실제론 테스트 전체를 삭제했을 때 이에 대한 자기 고백을 잘 유도하면 스스로 사실을 보고함
- 셀프 다이애그노시스로 도구 오류, 사용자 불만, capability gap(기능 미지원), 자기 수정(self-correction) 패턴도 탐지 가능
- 셀프 다이애그노시스용 도구(tool)를 하나 추가하고 시스템 프롬프트에서 호출 가이드만 추가하면 손쉽게 구축 가능
- 툴 설명과 네이밍(unsafe 대신 report 등) 방식을 바꾸면 모델의 자발적 고백 빈도가 달라짐

### 실제 코드 워크숍을 통해 셀프 다이애그노시스 적용법을 단계별로 직접 시연함

- GitHub 레포(nav: AI-talk-code)에서 오픈AI API키만 세팅하면, 다이애그노시스가 적용된 코딩에이전트 실행 가능
- 예제: write tool 장애 발생시, 에이전트가 즉시 bash tool로 우회(heredoc) 후, report 툴을 통해 문제행동을 보고하도록 세팅
- 시스템 프롬프트 조정만으로 self-report 빈도와 방식 제어
- 모델 훈련상, 자백(admission)을 꺼리는 성향이 있음을 파악하고, “creator에게 피드백 기록 남기기” 등 프레이밍이 동작에 영향

### 다양한 고객사례와 Raindrop 기능 활용법은 신호 기반 피드백루프·이슈 root cause 분석·클러스터링 등에 유용함

- 고객사례: 유형별 신호 설정(코딩 에이전트, 동반자 챗봇, 법률 에이전트 등 각각 다른 신호 정의)
- Deep Search 기능 통해 자연어로 신규 신호 생성 → 이진 분류기(cheap binary classifier)로 손쉽게 프로젝트 내 배포 가능
- 각 신호 클러스터링 분석 기반으로 사용자 의도/유즈케이스별 이슈율, 불만율 추적 가능
- Sentry와 유사하게 일별 이슈 리포트, 신호 스파이크 등 실시간 알림 및 상세 트레이스 제공

### 대규모 실험, 병렬 실험, query API, 데이터 파이프라인 등 실무 자동화 지원 기능도 강조됨

- Query API를 이용, 다양한 실험 결과를 BigQuery, statsig 등과 연동
- 여러 병렬 실험 결과를 합산·관찰·조합(Compound effect) 가능하도록 API 제공
- 실적, 추론 과정, 도구 사용 실패 등 트레이스를 Trajectory Visualization으로 형태별 시각화 및 탐색 지원

### 오픈 데이터, 신호 라벨/이벤트 내보내기, SDK 지원 등 확장성 및 연동 기능까지 포괄적으로 다룸

- 고객 데이터의 보안·컴플라이언스로 인해 전사적 aggregate 데이터는 제한적으로 공개
- Raindrop의 분류 및 신호라벨은 BigQuery 및 Snowflake 등 외부 데이터베이스로 익스포트 가능
- SDK(Python 등)도 점진적 지원 강화 중, 셀프 다이애그노시스 기능이 내장된 템플릿 제공
- 실험 자동 분배, 히스토리컬 데이터 역산(reingest) 및 신규 신호 백필(backfill) 지원

### AI 에이전트는 점점 더 복잡·지능적이며, 완전한 모니터링 없이 서비스 운영은 불가능에 가까운 시대가 도래하고 있음을 강조함

- 에이전트가 활용하는 도구, 컨텍스트, 결정 수준이 고도화되며 실시간 대량의 운영 데이터 속에서 발생하는 알려지지 않은 이슈(unknown issues)가 더욱 중요해진다 언급
- 이상치 탐지, 자동 신호 클러스터링, 신호 기반 이슈 자동 생성 등 end-to-end 모니터링 없이는 서비스의 품질 유지 불가
- Feature flagging, 실험 로직 자동화 등으로 빠른 배포와 품질 유지가 병행 가능하도록 시스템 구조화 필요함
