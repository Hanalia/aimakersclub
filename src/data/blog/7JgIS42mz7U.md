## *AI 버그 아포칼립스가 오고 있다. 이제 무엇을 해야 할까?* 핵심 요약

- Jack Cable(코리더 Corridor 공동창업자, 전 미국 CISA 고문)가 AI가 주도하는 소프트웨어 취약점 발견 및 악용(버그 대폭발) 현상을 설명함
- 최근 AI 코딩 도구(Copilot, Cursor 등)의 도입이 급속히 확대되어, 2023년 기준 개발자 84%가 AI 코딩 툴 사용, 기업 30~40%가 AI 코드 보조 도구 사용을 장려함
- AI가 취약점 자동 발견·악용 능력을 빠르게 발전시키며, 오픈소스 라이브러리 등 전체 소프트웨어 생태계의 공격 표면이 확장되고 있음
- 공격자는 AI를 사용해 전체 공격 과정(취약점 발견~공격 자동화)을 자동화해 방어자보다 더 효율적으로 공격할 수 있게 됨
- 반면, AI가 발견하는 취약점 다수는 이미 수십 년간 알려져 온 버그(대표적으로 메모리 안전 문제 등)로, 완전히 새로운 종류는 거의 없음
- Rust·Go와 같은 메모리 안전 언어로의 전환은 60~70%의 보안 취약점 예방이 가능하며, 실제로 Android 등에서 메모리 취약점 비중이 2019년 75%→2022년 30%로 감소함
- Opus 46 등 최첨단 LLM도 스스로 취약점을 생성 또는 놓치며, AI 코드 작성 시 20~40%에서 취약점이 삽입된다는 연구(Backsbench)에 의함
- AI 개발·코드 작성 자동화가 가속화되며, 인간 중심의 코드 리뷰에서 AI 기반 자동 리뷰로 전환이 진행 중임
- 보안팀은 생산성 유지와 보안(가드레일) 확보를 병행하는 도구 구축 및 관리가 필수적임
- 정책적으로는 (1) 신규 코드 보안 설계 강화, (2) 오픈소스 기반 소프트웨어의 보안 강화를 위한 구조적 재작성, (3) 미국산 오픈웨이트 AI 모델 생태계 활성화 등이 추천됨

## 세부 요약 - 주제별 정리

### AI의 능력 향상으로 소프트웨어 취약점 탐지·악용이 자동화되는 현상이 나타남

- Frontier AI 모델이 소프트웨어 취약점 탐지 및 악용 면에서 빠르게 발전함
- 기존에는 어려웠던 자동화된 공격 절차(취약점 탐지, 익스플로잇 등)가 이제는 AI 모델에 의해 동시다발적으로 실행 가능
- 공격자들은 이제 AI의 도움을 받아 전체 공격 체인을 자동화할 수 있음
- 이로 인해 ‘AI 버그 아포칼립스(취약점 대폭발)’라는 용어가 등장함
- 취약점이 주로 오픈소스 라이브러리 등 핵심 인프라 중심으로 발견되고, 세계 소프트웨어가 광범위하게 영향받음

### 코딩 에이전트와 AI 개발 툴 채택이 기록적으로 가속화되고 있음

- 2023년 Stack Overflow 통계를 인용해, 개발자 84%가 AI 코딩 툴(Copilot, Cursor 등)을 사용
- 회사의 30~40%가 AI 개발 보조도구 사용을 공식적으로 허용/권장함
- 최근 추세는 단순 자동완성(autocomplete) 수준에서, Slack 등 협업 툴 기반 비동기/자율적 코딩 에이전트가 실시간 여러 작업을 처리하는 형태로 고도화 중
- 생산성, 코드 품질, 대응 속도를 높이기 위해 AI 도구에 대한 의존도가 폭증하고 있음

### 공격 표면 확장과 AI의 공격 능력 상승으로, 방어 전략도 변화가 필요해짐

- AI 도입 확대와 더불어, 공격당할 수 있는 코드의 총량(attack surface)이 기하급수적으로 늘어남
- AI는 취약점 탐지 능력 자체가 고도화되어, 숙련된 해커 이상의 성능을 발휘
- 방어자 입장에서도 AI를 적극적으로 활용해 방어 체계 강화가 필수적임
- AI가 도입됨에 따라 소프트웨어 개발 및 보안의 패러다임이 근본적으로 변화함

### 대부분의 취약점 유형은 수십 년 전부터 알려진 고전적 버그임을 인식해야 함

- Anthropic의 Mythos 모델 등 SOTA(최첨단) AI도 발견하는 대부분의 취약점은 이미 오래전부터 반복되는 클래스(CWE 기준)임
- MITRE, CISA 등이 선정한 자주 악용되는 취약점 Top 리스트 대다수도 모두 고전적이며, 이론적으로는 충분히 예방 가능한 것들임
- 예: 버퍼 오버플로우(biffer overflow), 인젝션 등은 30년 전부터 밝혀진 결함
- Rust, Go 같은 메모리 안전(Memory Safe) 언어로의 전환만으로도 이러한 취약점을 근본적으로 차단할 수 있음

### 메모리 안전 언어 도입은 취약점 대폭 감소로 이어진 실증적 결과가 존재함

- CVE 등 실제 데이터에 따르면, 메모리 안전 언어(Rust, Go 등) 사용시 메모리 관련 취약점의 60~70% 예방 가능
- 구글, 마이크로소프트, 아마존, 리눅스 커널 등 주요 업체 및 오픈소스 커뮤니티에서도 Rust 중심 재작성 움직임 확산
- Android OS의 예: 2019년 메모리 안전 취약점 비중 약 75% → 2022년 30%로 급감(새로운 코드만 Rust 등으로 작성하더라도)
- 단발성 패치보다는 ‘근본적 재작성’과 ‘구조적 기술 전환’이 보안 향상에 더 큰 효과

### AI 스스로도 보안 취약점을 지속해서 도입할 수 있음을 주의해야 함

- 최신 LLM(예: Opus 46)도 소스코드 생성 과정에서 취약점을 종종 도입
- Opus 46이 생성한 스마트컨트랙트 취약점으로 수백만 달러 해킹 사례 발생
- ETH Zurich, UC Berkeley가 개발한 Backsbench 연구 결과: AI가 코드 작성시 20~40% 확률로 취약점 삽입
- 이는 AI가 인간이 만든 불완전한 코드 데이터로 학습했기 때문이기도 하며, 소프트웨어 비즈니스 로직 등 맥락(Context)에 대한 이해 부족도 원인
- 최근 취약점 트렌드는 ‘단순 버그’보다 복잡한 권한/인증 관련 맥락성 오류에 치우침

### AI 개발 에이전트가 점차 완전 자동화 및 AI 코드 리뷰로 전환되는 현상이 예상됨

- 코드 자동화 개발은, 과거 자동완성 → 단일 기능 에이전트 → 완전 자율형 코드 에이전트 순으로 수직 발전
- 수 시간 동안 대규모 코드 변화를 혼자 수행하는 AI 에이전트가 속속 등장
- IH Corridor 내부 전망: 6~12개월 내 대부분의 코드가 인간이 아니라 AI에 의해 코드 리뷰될 것으로 예측
- 기존 개발 파이프라인은 AI 기반으로 재구성되며, 생산성 극대화와 보안 확보가 동시 요구됨

### 보안팀의 역할은 ‘생산성-보안 균형’ 및 선제적 가드레일 도구 마련으로 변화함

- 기업 보안팀의 고민이 ‘AI 도입 여부’에서 ‘어떻게 안전하게 사용할 것인가’로 완전히 전환됨
- 가드레일, 사전 검사, 실시간 모니터링 도구 등 모범 사례와 기술적 관리 수단이 필요
- Corridor 사가 지향하는 접근: PR 머지 전 사전 취약점 차단, AI 코드 사용 가시성 증대가 관건
- 궁극적으로 보안팀이 성장의 병목이 되지 않으면서도 최소한의 저지선을 보장해야 함

### 정책 차원에서 미국 오픈웨이트 모델 육성 및 구조적 코드 개선이 필수임

- Jack Cable, 미국 의회 청문회에서 세 가지 정책 제언 제시:
    1. 신규 코드 작성 단계에서 예방적(secure-by-design) 보안 원칙 철저 적용
    2. 오픈소스 기반 소프트웨어의 전방위적 구조적 개선(재작성, 메모리 안전 언어 전환 등) 필요
    3. 미국산 오픈웨이트(가중치 공개형) AI 모델 생태계 활성화가 기술적·안보적 경쟁력의 핵심
- 이는 최근 미국 정부가 Mythos, Fable 등 첨단 AI 모델의 수출 통제를 강화한 맥락과도 연관
- Anthropic 등 프런티어 모델은 수비수에게도 매우 강력한 도구가 될 수 있는 ‘양면성’을 지님
- 오픈웨이트 모델은 미세조정(fine-tuning) 등 기업별 적응성 및 경쟁 우위 확보에도 필수적
- 오픈소스 소프트웨어가 사이버 공격의 주요 시험대임을 감안할 때, 미국 내 자체 모델 경쟁력 확대가 시급함

### AI 보안 위협에 맞서려면 "기본에 충실한 보안 설계"가 본질임을 강조함

- Frontier 모델의 능력 발전 속도는 매우 빠르지만, 방어의 핵심은 잘 알려지고 검증된 기본 보안 수칙과 원칙 준수임
- 단기적 ‘버그 잡기’보다 구조적 재설계, 근본적 결함 방지 체계에 집중할 때 가장 효과적임
- AI를 적극적으로 사용해 시스템의 복원력(resilience)과 선제적 방어력을 확보해야 함

### 결론: “AI로 인한 보안 위기는 시스템 근본 구조 변화 기회”

- AI 도입은 소프트웨어 개발·보안의 대격변을 일으키나, 잘 알려진 취약점 예방책만 철저히 지켜도 상당수 위협 대응 가능
- 메모리 안전 언어 전환, 체계적 코드 리뷰 자동화, 보안팀과 개발팀 협업 메커니즘 강화가 핵심 과제로 부상
- AI가 실질적 성장 동인인 만큼, 생산성 저해 없는 선제적 보안 체계 마감이 (“secure-by-design”) 중대 목표임
