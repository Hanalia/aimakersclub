## *뛰어난 에이전트 스킬 작성: 누락된 매뉴얼* 핵심 요약

- 영상은 "에이전트 스킬 작성에 있어 진정으로 훌륭한 스킬이란 무엇인가"에 대한 체계적 체크리스트와 프레임워크의 부재를 지적하며, 이를 보완할 구체적 가이드라인을 제공함
- 현재 개발자들과 조직들은 수많은 오픈소스 스킬을 접할 수 있으나, 이들 스킬의 질적 차별성을 구분하거나 조합하는 데 어려움을 겪고 있으며, 이를 '스킬 헬'로 정의함
- Matt PCO(연사의 유명 깃헙 스킬셋)와 Superpowers(모델 인보크형 스킬셋) 등 다양한 스킬 프레임워크의 특징, 한계, 차이점을 상세히 비교함
- 스킬의 '트리거(실행 방식)', '내부 구조', '스티어링(에이전트 행동 유도)', '최소화(불필요 내용 가지치기)'를 점검하는 네 단계 체크리스트를 제시함
- 스킬 실행 방식에서 '유저 인보크(수동)'와 '모델 인보크(자동)' 방식의 트레이드오프(컨텍스트 비용, 사용자의 인지적 부담 등)를 심층적으로 설명함
- 모든 스킬은 '단계별 절차(steps)'와 '참조 정보(reference)'로 구성되며, 불필요한 중복, 분기(material branching), 거대한 스킬 파일 등을 구조적으로 관리해야 함을 강조함
- 에이전트가 사용자의 의도를 제대로 수행하도록 하는 핵심 기법으로 '리딩 워드(Leading Words)' 반복 활용과 필요한 경우 스킬 세분화의 비법을 구체적 사례와 함께 제시함
- 최종적으로 스킬 최적화를 위해 중복, 침전물(irrelevant sediment), 무의미한 문장(no-ops) 등 유지보수상 취약 요소를 지속적으로 제거해야 함을 강조
- 영상 전체에서 직접 작성한 스킬 예시, 실제 구현 코드, 다양한 설계상의 결정 기준까지 구체적으로 설명
- 실천에 도움이 되는 "writing great skills" 스킬 예시와 체크리스트 오픈소스 제공

## 세부 요약 - 주제별 정리

### 개발자와 조직 양자 모두 '스킬 헬'에 빠져 있으며 이는 효율적 스킬 작성을 어렵게 만듦

- 예전에는 '튜토리얼 헬', '프레임워크 헬'과 같은 새로운 학습 지옥이 존재했으나, 현재는 너무 많은 스킬이 난립해 "스킬 헬"이라는 신종 문제로 대두됨
- 스킬을 쉽게 다운받고 시도해볼 순 있으나, 실제 사용에서 어떤 스킬이 좋고, 어떻게 조합해야 하는지 판단하는 것이 어렵다고 언급
- 조직 차원에서도 기존 업무 절차를 에이전트가 구현할 수 있는 스킬로 변환하는 방법 자체에 대한 공통 지식이 부족함
- 스킬들이 약속하는 효과와 실제 결과가 괴리되는 원인이 여기에서 발생한다고 지적
- Matt 자체 스킬셋(PCO skills)을 배포하는 입장에서도 사용자들이 스킬 헬을 벗어날 수 있도록 실질적 가이드가 필요하다고 역설

### 진정한 '좋은 스킬'에 대한 평가 기준과 프레임워크의 부재가 문제임

- 현재 개발 문화에는 스킬의 질을 평가하거나 개선할 체계적 '루브릭' 혹은 체크리스트가 부재함
- 영상의 목적은 어떤 스킬이 좋은지 판단하고 개선할 수 있는 체크리스트와 실천적 프레임워크를 제공하는 것임
- "writing great skills"라는 별도의 스킬까지 오픈소스로 제공, 즉각 실험·적용할 수 있음을 안내

### 스킬 트리거(실행 방식)는 모델 인보크와 유저 인보크의 선택지가 있으며, 각각 장단점이 뚜렷함

- 스킬은 언제나 "수동(user invoked)" 혹은 "자동(model invoked)"으로 실행될 수 있음
    - user invoked: 사용자가 직접 명령하여 에이전트가 스킬을 실행. agent 컨텍스트에 스킬 설명이 항상 포함되지 않음
    - model invoked: 스킬의 설명(description)이 항상 agent 컨텍스트에 포함되어 agent가 필요에 따라 자동 실행
- 모델 인보크 스킬은 agent에 유연성과 자동화된 능력을 부여하지만, 컨텍스트 로드(=토큰 및 인지 부담)가 커짐(예: 100개 추가 시 100개 설명 컨텍스트 상시 추가)
- 반대로 유저 인보크 스킬은 agent의 컨텍스트 부담을 최소화하나, 사용자가 각 스킬을 숙지하고 적시에 호출해야 하므로 인지 부담이 큼
- 대표적 사례로 Superpowers(주로 모델 인보크형)와 본인의 PCO skills(주로 유저 인보크형) 구도를 비교하여 설명
- 선택의 트레이드오프: 자동화의 이점(모델 인보크) vs 예측 가능성과 직접통제(유저 인보크)
- 모델 인보크는 agent가 스킬을 불규칙하게 호출하지 않을 수 있다는 예측 불가 문제가 존재함

### 스킬의 내부 구조 설계에서 '단계별 절차'와 '참조 정보' 두 가지가 핵심 구성 요소임

- 대부분의 스킬은 단계별 절차(steps)와 참조자료(reference)로 구분되어야 하며, 이 둘을 명시적으로 나눠 관리해야 한다고 강조
- 어떤 스킬은 steps만, 어떤 스킬은 reference만을 가질 수도 있음
- 참고 예시: 'to PRD' 스킬 – 3단계(step 1: 관련 컨텍스트 추출 / step 2: 사용자 확인(test seam) / step 3: PRD 작성)로 구성, 여기에 "test seam 설명"과 "PRD 템플릿" 두 가지 reference가 있음
- 이 구조를 통해 스킬을 처음부터 쉽게 설계하고, 확장 및 유지보수도 용이해짐

### 스킬 파일을 작고 간결하게 만드는 것은 유지보수성과 에이전트 효율 모두에 중요함

- 중요 팁: skill.md(스킬 문서)의 크기를 최대한 줄일수록 유지관리가 쉬우며, 요청 단가(토큰 절약)도 감소
- 브랜칭(분기)이 많은 스킬일수록, 개별 branch에서만 사용하는 reference는 skill.md 외부 파일로 이동(컨텍스트 포인터 활용)
- 예시: 'to PRD'는 reference가 항상 필요하므로 skill.md에 포함, 반면 'domain modeling'은 사용 분기별로 only-on-branch 참조 자료를 파일 분리
- 외부 reference 파일: skill.md와 같은 폴더 내 별도 마크다운 파일로 두고, 필요시 컨텍스트 포인터 방식으로 접근
- 이러한 가지치기를 통해 각 branch에 반드시 필요한 정보만 main 파일에 두고, 나머지는 최소화
- 불필요한 내용, branch 관련 중복 등은 skill.md에서 제거 또는 외부화

### 에이전트의 행동을 정확히 유도하고 싶다면 '리딩 워드' 활용과 단계별 분할 기법이 효과적임

- 주된 문제: 스킬에 행동 지시를 명확히 했다고 생각해도, 에이전트가 기대만큼 충실히 따르지 않는 사례가 빈번함
- 해결 기법 1: "리딩 워드(leading words)" 반복 – 의미가 압축된 특정 단어/구절을 여러 번 사용함으로써, 에이전트 reasoning 흐름에 해당 단어가 반복되게 유도(예: "vertical slice"라는 용어 반복 사용으로 단계별 구현 유도)
- 리딩 워드는 실제 reasoning trace 내에서 그 단어의 출현 빈도로 효과를 검증할 수 있음
- 스킬의 모든 단계와 서술에서 이러한 리딩 워드를 불릿/구문 등으로 일관되게 반복 활용할 것 강조
- 해결 기법 2: step 분할 – 에이전트가 특정 단계에서 충분한 '사전작업(leg work)'을 수행하지 않는 문제는, 미래 단계(예: 플랜 세우기)를 완전히 숨기고 현재 단계만 보이게 하여 행동을 유도
- 예시: 기존 "plan mode(clarification→계획수립)"에서, clarification과 plan을 별도 스킬로 분리하여, 에이전트가 'clarifying' 단계에만 집중하도록 설계

### 스킬 가지치기는 중복, 침전물, 무의미한 문구 제거 등 여러 실패 패턴을 예방함

- 실패 패턴 1: 중복(duplication) – 동일 reference(예: PRD 템플릿, test seam 등)를 여러 곳에 반복해 작성하는 것 예방
- 각 자료는 '단일 출처(single source of truth)' 원칙에 따라 한 곳에만 유지되어야 유지관리 비용이 감소함
- 실패 패턴 2: 침전(sediment) – 여러 사람이 하나의 markdown에 내용을 추가만 하고, 불필요한 분기, 구식 정보, branch irrelevant 내용이 쌓임
- 해결: branch relevance에 따라 정리, 불필요한 요소는 삭제
- 실패 패턴 3: noop(no-ops) – 실제로 에이전트의 행동에 영향을 미치지 않는 쓸모없는 구문(예: 의무적 커밋 메시지 작성 독려 문단이 없어도 실제 커밋 메시지 품질은 동일)
- 해결: 삭제 실험(irrelevant 내용 삭제 후 변화 관찰), 리딩 워드 활용, 불필요한 문구의 지속적 제거

### 전체 프로세스는 '트리거', '구조', '스티어링', '가지치기' 네 단계 루브릭으로 깔끔하게 정리됨

- 1. 트리거 점검: 스킬 실행이 올바른 상황에만 이루어지도록 의도와 부담(컨텍스트/인지)을 구조적으로 검토
- 2. 내부 구조 점검: steps와 reference로 백터화(partitioning), 여러 branch 여부에 따라 자료 외부화
- 3. 행동 유도(steering): 리딩 워드 선정 및 반복, 단계별 세분화로 agent 행동 유도 및 reasoning 흐름 관리
- 4. 가지치기(pruning): 최종적으로 불필요한/중복된/의미 없는 정보 없애기

### 공개된 “writing great skills” 오픈소스 스킬과 체크리스트를 즉시 실험 및 적용할 수 있음을 안내함

- 본 과정은 "writing great skills" 스킬로 이미 매뉴얼화 되어 깃헙에 공개, 누구나 바로 실습 가능한 환경 제공
- 커뮤니티에서 유통되는 외부 스킬도 이 체크리스트로 품질 점검 가능
- 덧붙여 AI coding crash course, aihero.dev 뉴스레터 등 추가 리소스 제공 예정임을 안내
