## *프롬프트가 곧 플랫폼이다 - 도미닉 토르노우, Resonate HQ* 핵심 요약

- 2026년 AI 코딩 에이전트들이 최초의 소프트웨어 플랫폼을 조용히 퇴역시킬 예정임을 밝히며, 이는 해당 플랫폼이 나빠서가 아니라 더는 필요치 않기 때문임을 강조함
- Resonate는 최소주의와 단순함을 핵심 가치로 삼는 내구성 있는 실행 플랫폼으로, 이런 속성들이 향후 소프트웨어 엔지니어링의 방향에 중요하게 작용할 것임
- 기존 일반 목적 소프트웨어 구현은 점차 온디맨드로 맞춤 생성되는 특화 구현으로 대체될 전망이며, 이때 핵심 재사용 대상은 구현이 아니라 '명세(specification)'임
- Resonate의 제품 가치는 더 이상 구현에 있지 않고, '프로토콜 명세'라는 추상적 산출물로 이동함; 같은 명세로 여러 곳의 인프라에 맞춘 서버 구현 생성이 가능해짐
- 에이전트가 추상 명세로부터 곧바로 구현에 도달하는 데 한계가 있었기에, 인간이 주도해 '구체 명세(concrete specification)'라는 중간 산출물을 도입해 프로덕션 품질의 시스템 구축을 성공함
- 에이전트가 명세 설계 단계까지 참여하도록 시뮬레이션 환경을 제공하여, 올바른 알고리즘과 시스템 설계를 사전에 검증한 뒤 프로덕션 구현으로 전환하는 새로운 프로세스를 도입함
- 최소주의·단순성은 출발점이 아닌 3년에 걸친 지속적 노력의 산물이며, 결국 내구성 프로토콜은 'durable promise'와 'durable task'라는 두 객체 중심의 매우 작은 구조로 축소됨
- NATs 기반의 분산 환경에 Resonate를 적용하면서 플랫폼 고유의 일관성·버전 관리 문제(신선한 읽기 vs. 오래된 읽기)를 시뮬레이션으로 재현, 반복가능·원인추적 가능한 테스트 도구를 개발함
- 시뮬레이션은 실제 플랫폼이 숨기는 오류 원인(예: 오래된 값으로 인한 불변조건 위반)을 노출하여 에이전트가 설계·디버깅에 적극적으로 기여하게 만듦
- 최종적으로 프롬프트는 플랫폼, 명세는 제품이라는 비전하에, 에이전트가 추상 명세→ 시뮬레이션 구현→ 구체 명세→ 최종 구현 단계까지 설계를 주도할 수 있음을 실증함

## 세부 요약 - 주제별 정리

### 2026년, 소프트웨어 플랫폼의 역할이 불필요해지는 전환이 시작됨

- 2026년 AI 코딩 에이전트들이 처음으로 한 소프트웨어 플랫폼을 퇴역시킬 전망임을 밝힘
- 퇴역의 이유는 플랫폼의 품질 문제가 아닌, 더 이상 플랫폼의 '존재 자체'가 필요하지 않아지기 때문임
- 지금까지 플랫폼은 구현(reimplementation)·라이브러리 등 재사용의 중심이었으나, AI 에이전트가 인프라에 맞춰 즉석에서 맞춤 구현을 생성할 수 있게 되면서 플랫폼의 필요성이 줄어듦
- 소프트웨어 엔지니어링 패러다임이 '구현'의 반복에서 '명세'의 재사용으로 근본적 이동이 이뤄짐을 예고함

### Resonate의 제품 철학은 구현이 아닌 명세와 프로토콜로 이동함

- Resonate는 최소주의(minimalism)와 단순함(simplicity)이라는 기술적 가치를 바탕으로 한 내구성 실행 플랫폼임
- Resonate의 가치와 제품 정의가 ‘구현(implementation)’에서 ‘명세(specification)’ 또는 '프로토콜(protocol)'로 이동함
- 명세만 작성하면, 일반 목적 서버(prior reference implementation) 뿐 아니라 각 인프라 환경(예: 특정 클라우드/미들웨어) 맞춤 서버 구현이 가능함
- 구현이 생성가능(generatable)해지면, 고객과 파트너는 본인 인프라 위에 거의 의존성 없는 내구성 있는 실행 환경을 제공받을 수 있음
- 질문이 “서버를 만들 수 있는가?”에서 “동일한 명세로 신뢰할 수 있는 서버를 반복적으로 합성할 수 있는가?”로 바뀜

### 에이전트가 단순 구현 대신 명세·설계 단계에 올라서야 함을 주장함

- 1차 시도: 추상 명세(abstract specification)를 에이전트에게 바로 주어 특정 구현(예: Rust+PostgreSQL Resonate 서버) 생성을 요청함
- 실패 사례: 에이전트가 베이직 테스트만 통과하는 동작 코드를 만들었으나, 동시성·프로세스 오류·네트워크 장애 등 상황에서 부적합 ⇒ 프로토타입 수준에 머무름
- 원인: 추상 명세와 구체 인프라 간의 간극이 크며, 명세에 구현 구체 정보가 포함되지 않아 에이전트가 적합한 설계를 만들지 못했음

### 인간-에이전트 협업으로 '구체 명세'라는 중간 산출물을 도입하여 성공적인 구현이 가능해짐

- 차별화된 절차 도입: 추상 명세와 구현 사이에 '구체 명세(concrete specification)'라는 중간 단계 설정
- 구체 명세 작성 과정에서 데이터 스키마, 인덱스, SQL 쿼리, 트랜잭션 범위 등 인프라 특화 의사결정을 구체적으로 명시
- 에이전트는 구체 명세를 받아 프로덕션 수준 시스템을 구현하는 데 성공함
- 한계: 이 방식은 에이전트가 '구현'에만 도움을 주고, '설계' 단계에는 기여하지 못함

### 에이전트가 설계 단계까지 참여하려면 시뮬레이션 환경이 필요함을 발견함

- Resonate를 NATs 기반 인프라(Nats.io 등)로 이전하면서 접근법 변화 시도
- 목표를 “에이전트가 프로덕션 시스템을 만들 수 있는가?”에서 “에이전트에게 설계까지 주도하려면 무엇이 필요한가?”로 전환
- 해결책: 에이전트에 결정적(Deterministic) 시뮬레이션 환경을 제공하고, 생산 시스템 대신 '실행가능한 설계 시뮬레이션' 생성을 요구함
- 과정은 추상 명세→시뮬레이터 구현→구체 명세→구현으로 바뀌고, 이 단계에서 에이전트가 설계 전체를 주도함

### 최소주의와 단순성 추구가 설계·구현 난이도와 직결됨

- 최소주의와 단순성은 설계 초기값이 아니라, 3년에 걸친 반복적 개선의 결과임을 강조
- 문제 발생 시마다 "무엇을 제거할 수 있나?" "어떤 추상화를 지울 수 있나?" "어떤 관계를 끊을 수 있나?"를 끊임없이 자문
- 그 결과 Resonate 프로토콜은 'durable promise', 'durable task'라는 두 객체만 중심에 두는 매우 소규모 구조가 됨
- 하지만, 단순한 프로토콜이라도 동시성 있는 분산 환경에선 상태공간이 복잡하기 때문에 설계와 구현 모두 쉽지 않음

### 대상 인프라(NATs)의 특성에 맞는 설계가 필수임을 구체 예시로 설명함

- NATs는 '큐', '키-값 스토어', '지연/스케줄 메시지'라는 기본 요소만 제공
- Resonate 프로토콜을 NATs의 이 제한적 개념으로 표현하려면 새로운 설계가 필요함
- 키-값 스토어는 버전화되어 있고, 읽기 연산 시 최신값(신선한 읽기)이 올 수도 있고 오래된 값(오래된 읽기)이 반환될 수도 있음
- 예시: key의 값이 'foo'(v0)→'bar'(v1)→'baz'(v2)로 변했을 때, 읽기 결과가 v2가 아니더라도 버그가 아닌 인프라 일관성 모델의 합법 결과임
- 쓰기 충돌은, 예를 들어 v0을 읽고 해당 버전으로 쓰기를 시도했더니 이미 더 최신 버전이 존재해 쓰기가 실패하는 케이스로 발생함

### 일관성 모델이 에이전트/인간의 구현 난이도를 높임을 강조함

- 신선한 읽기만 가능한 환경이면 설계가 쉬우나, 일부는 의도적으로 오래된(read staleness) 상태라서 인지·예상·처리 모두 난이도가 상승
- 인간은 물론, AI 에이전트 역시 이 복합 일관성 문제에서 잘못된 의사결정을 쉽게 내릴 수 있음
- 구현의 정확성을 담보하려면 에이전트가 실패 원리를 제대로 이해하고, 문제 추적 및 수정을 해야 함

### 결정적 시뮬레이션 환경이 에이전트 설계·디버깅 능력을 획기적으로 높임

- Python으로 결정적 테스팅 시뮬레이터를 구축함
- 실제 Nats.io처럼 동작하되, 난수 생성기를 통해 새 값 또는 오래된 값 반환을 통제 가능하게 만듦
- 낙관적 동시성(optimistic concurrency) 적용하여 읽은 버전이 최신일 때만 쓰기 성공, 그렇지 않으면 예외(실패)
- 생산 환경과 달리 시뮬레이터는 모든 상태·이벤트를 추적 가능(예: 어떤 읽기가 오래된 읽기였는지, 실제 최신값은 무엇이었는지 등)

### 시뮬레이션 내 '금지된 정보' 노출이 에이전트의 원인-결과 추적과 버그 수정을 돕는 데 핵심적 역할을 함

- 실제 코드는 읽기 결과 그 자체만 알 수 있으나, 시뮬레이션에선 ‘신선한/오래된 읽기 여부'와 그때 숨겨진 최신 값까지 트레이스 이벤트로 기록
- 이런 '금단의 정보(forbidden fruit)'를 활용해 에이전트가 설계한 알고리즘의 실패 원인을 추적·설명할 수 있음
- 예를 들어, 어떤 변수가 불변성과 어긋난 원인이 '오래된 값'으로 인해 잘못된 결정을 내렸기 때문이라는 것을 원인-결과 형태로 파악 및 교정 가능
- 생산 코드는 금단 정보를 사용하지 않지만, 에이전트의 학습·수정 단계에서는 큰 도움이 됨

### 에이전트가 설계·구현 전 과정의 주도자가 되어 추상 명세부터 최종 코드까지 생성해냄을 실증함

- 결정적 시뮬레이션 첫 단계로 PoC(proof of concept)를 검증
- 검증된 알고리즘을 바탕으로 구체 명세(concrete specification)를 에이전트가 도출
- 이후, 구체 명세로부터 최종 구현(production implementation)까지 에이전트가 주도적으로 생성
- 전체 과정: 추상 명세→시뮬레이션 설계·검증→구체 명세→생산 구현 순서가 새 표준으로 자리매김
- 프롬프트(명령어)는 플랫폼이 되고, 최대 재사용 대상은 구현이 아니라 명세라는 새로운 소프트웨어 엔지니어링 패러다임을 실증적으로 보여줌
