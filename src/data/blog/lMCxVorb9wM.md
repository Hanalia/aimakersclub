**영상 링크:** [You Didn't Ship a Bug. You Just Wrote It for a Human. - Ravi Madabhushi, Scalekit](https://www.youtube.com/watch?v=lMCxVorb9wM)  
**채널명:** AI Engineer

## *버그를 배포한 게 아니라 인간을 위해 코드를 작성한 것이다* 핵심 요약

- 발표자 Ravi Madabhushi는 Scalekit 공동창업자로서, 사람(인간) 중심으로 설계된 인증·권한 시스템이 에이전트(Agent) 시대에는 한계를 가진다는 점을 강조함
- 실제 사례로, '유저의 마지막 활동 시간' 갱신 로직이, 인간 사용자 대비 에이전트 접속에서 60배 빈번하게 DB 쓰기 부하를 일으킨 경험을 설명함
- 이런 문제는 단순히 속도나 배치 처리로 해결 가능하지만, 더 근본적으로 인증·권한모델 자체를 재고해야 함을 주장함
- 전통적인 인간/서비스계정 기반 인증 시스템에서는 '인증 주체=행위자'라는 가정이 있지만, 에이전트 환경에서는 성립하지 않음
- 에이전트는 예측불가능(비결정적)하며, 특정 유저를 대신해 행위해야 하므로 세밀한 권한 위임과 인증이 필요함
- 현재 대부분의 API 및 OAuth 권한 부여 모델은 지나치게 포괄적이고, 툴 접근 제어가 충분히 세분화되어 있지 않음(예: Gmail의 broad scope)
- 실제 사례와 함께, 세밀하지 못한 권한 통제가 생산 데이터베이스 삭제 등 위험 사례로 이어질 수 있음을 언급함
- ref.tools 같은 기업은 전적으로 에이전트만이 행위 주체인 시스템을 운영하며, 더 세밀한 OAuth 스코프 및 컨텍스트 분리가 필요함을 실증함
- 궁극적으로, "에이전트에 대해 결정적인 가드레일과 실시간 행동 감시, 최소 권한의 원칙이 필요하며, OAuth보다 더 세밀한 접근제어가 긴급하다"고 강조함

## 세부 요약 - 주제별 정리

### 에이전트 중심 시대에는 사람을 가정한 인증 아키텍처가 확장성 한계를 드러냄

- 기존 시스템의 인증·권한 설계는 인간 사용자를 기본 전제로 구축됨
- 최근 1년 새 API 액세스 패턴의 급격한 변화(에이전트 대량 등장)로, 기존 가정의 한계 노출
- 인간 사용자를 전제로 한 '마지막 활동 시간' 기록 등이 예상치 못하게 자주 갱신되어 인프라에 부담 발생
- 기존의 "인증 주체 = 행위자"라는 가정이 더 이상 항상 성립하지 않음
- 에이전트가 다수 사용자/시스템을 대신하여 다양한 방식으로 활동하며, 이로 인해 새로운 보안·운영 문제가 대두됨

### 인간 및 서비스 계정 기반의 전통적 인증·권한 모델의 한계

- 인간 사용자는 패스워드, 웹 세션 등으로 인증되고, 미리 설정된 권한 범위 내에서만 행동
- 고전적 API 키, 서비스 계정(예: OAuth, SPIFFE) 역시 정해진 권한 내에서 예측 가능하게 동작
- 보안 리뷰 시 코드 검토를 통해 프로그램 행동을 확인·검증 가능(정적, 결정적 특성)
- 더 큰 문제는, 기존 시스템이 '항상 같은 인간, 항상 같은 프로그램'을 가정했다는 점임
- 에이전트 등장으로 "인증 주체"와 "행위자"가 분리되고, 권한 위임과 맥락별 제어 필요성이 대두됨

### 에이전트의 등장으로 인증·권한시스템의 기본 전제가 깨짐

- 에이전트의 핵심 특징: 비결정적(Non-deterministic), 스스로 판단해 다양한 행동 가능
- 인간 개발자가 작성한 API 프로그램은 항상 동일하게 동작했지만, 에이전트는 그렇지 않음
- 에이전트가 사용자 1, 2, 3에 대해 각각 다르게 행동할 수 있으므로, 세밀한 맥락별 권한 제어 필수
- 현재의 OAuth/서비스계정 방식으로는 '누구를 대신해서', '어떤 한계 내에서' 행동하는지 명확 분리가 불가

### 실제 시스템에서 발견된 문제: 지나치게 폭넓은 권한 범위로 인한 위험

- 실제 사례: 글쓰기 트래픽 급증(60배 상승)으로 DB 시스템 부하 발생
- 원인: 사용자가 아닌 에이전트가 단기간 내 반복적으로 '마지막 활동' 기록을 갱신
- 많은 개발자들이 비의도적으로 에이전트에 필요 이상의 권한을 할당하고 있음
- 기존 권한/스코프 프리미티브로는 에이전트에게 최소 권한만 효율적으로 줄 수 없음

### 현재 사용되는 OAuth 및 인증 스코프가 충분히 세밀하지 못함

- Gmail, Salesforce 등 대표적 OAuth 예시: broad scope(ex. 발신권한)만 존재, 세부 설정 불가
- 실질적으로 "이 에이전트가 언제, 누구에게, 어떤 메일만 보낼 수 있는가" 등은 제어 불가
- 에이전트는 권한 내의 모든 행동을 자유롭게 하며, 그 결과로 데이터 위험성 증가

### 비결정적인 에이전트 행동에 대한 가드레일의 시급성

- 과거에는 정적 프로그램의 예측가능성으로 운영 문제가 최소화되었으나, 에이전트는 예외
- 지금도 에이전트로 인해 생산 DB 삭제 등 심각한 운영 사고가 실제로 발생 중
- "오늘의 문제"임을 강조하며 사전적 위험 통제 필요

### ref.tools 사례: 순수 에이전트 기반 플랫폼에서 특화된 권한 체계 도입

- ref.tools는 사람(인간) 없이 오직 에이전트만이 시스템 행위 주체
- 코딩 에이전트 중심 플랫폼으로, 컨텍스트/시나리오별로 세밀한 OAuth 스코프 설계 필요성 실증
- 모든 행위 주체, 엑세스 로그, 인증/권한 부여 내역 등에 대해 실시간 감시 체계 확립

### 최소 권한 및 Just-in-Time 권한 상승 원칙이 필수적으로 적용되어야 함

- 모든 에이전트는 최소 권한(least privileged)만을 기본부여
- 필요시에만, 제한된 범위/시간에 한해 권한(purpose-limited, time-bounded)을 확장
- 사용자, 행위자, 인증 주체, 권한 부여자, 권한 시점 등 모든 이력을 완전하게 추적 가능해야 함

### 복잡해지는 에이전트 환경에서 "기도 말고, 결정적 통제"가 유일 해법임

- "에이전트가 잘못된 행동을 하지 않기를 바라는 것"은 전략이 될 수 없음
- 반드시 프로그래머틱하게 모든 행동에 대해 실시간 tracing과 결정적 허가/거부가 필요

### 미래형 시스템 아키텍처는 OAuth를 넘는, 더 세밀한 컨트롤 구조로 진화해야 함

- OAuth는 출발점일 뿐, 현 수준의 broad scope는 한계가 명확함
- 인가 속성(Attribute-based), 맥락(Context-based), 역할/사용자별로 강하게 분리된 액세스 컨트롤 필요
- 에이전트의 맥락/목적/기한/대상 등에 따라 실시간으로 Permission을 제어하는 구조 도입 필요
- API/시스템 설계는 인간+API만이 아니라, 에이전트의 비결정성·위임·실시간성에 맞춰 재설계돼야 함
