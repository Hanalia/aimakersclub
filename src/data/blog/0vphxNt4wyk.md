## *Eval(평가) 없이 스킬을 배포하지 마세요 — Philipp Schmid, Google DeepMind* 핵심 요약

- 본 영상은 Google DeepMind의 Philipp Schmid가 “스킬(Skill)” 기반 코딩 에이전트 개발 시 평가(Eval) 도입의 중요성을 실제 사례, 데이터, 상세 가이드와 함께 설명함
- 대부분 개발자들이 코딩 에이전트의 스킬을 사용하지만, 정작 Eval을 적용하는 비율은 매우 낮으며, 이로 인해 실제 제품 환경에서 실패 사례가 빈번히 발생하고 있음
- Skill Bench 자료에 따르면 5만 개 이상의 오픈 소스 스킬 중 거의 모두가 Eval 없이 배포되어 있고, AI 생성 스킬은 그 과정에서 품질 저하 및 신뢰성 문제를 일으킬 수 있음
- 스킬은 ‘능력(Capability) 스킬’과 ‘선호(Preference) 스킬’로 나뉘며, 능력 스킬은 향후 모델 발전에 따라 퇴역이 가능하지만 선호 스킬은 도메인 특화 정보로 더욱 견고하게 관리해야 함
- Skill Bench 1.1 기준, 스킬을 적용할 경우 평균 15% 성능 향상을 확인할 수 있지만, AI 생성(자동생성) 스킬은 성능 저하를 일으킬 수 있으니 반드시 검증이 필요함
- 효과적인 스킬 작성법 8가지, 스킬 파일 구조, 디렉티브 사용법, 정보 레이어링, no-op(비효과) 제거, 네거티브 케이스 테스트, 스킬 퇴역 시점 등을 구체적인 예시와 함께 설명
- Google DeepMind 내부에서는 스킬마다 외부 변화가 있을 때마다 자동으로 Eval이 트리거되는 시스템을 도입, 성능이 저하되면 배포가 차단됨
- 실제 Gemini Interactions API용 스킬 개발 시 117개의 케이스를 중심으로 JSON+Python 기반 효율적이고 구체적인 Eval 시스템을 운영해 90% 이상 성능 향상을 이룸
- 스킬은 500줄 이내의 skill.md와 분리된 참고 파일 관리, 사용자의 직접 트리거/모델 트리거 구분, 매번 10~20개 이상의 케이스별 평가 필수, LLM 판정도 활용 가능
- 실전 체크리스트 10가지를 제시하고, 누구나 월요일에 가장 많이 쓰는 스킬을 골라 5개 테스트프롬프트를 만드는 실습을 주문함

## 세부 요약 - 주제별 정리

### 코딩 에이전트의 스킬은 모두 사용하지만, 실제로 Eval을 적용하는 비율은 극히 낮음

- 발표 시작부터 코딩 에이전트(예: anti-gravity, cursor, Claude code 등)를 사용하는 청중 상당수가 손을 듦
- 그러나 “스킬에 Eval을 적용하는가?” 질문에는 일부만 손을 듬
- 현실적으로 대부분의 개발자, 팀, 오픈 소스 커뮤니티가 스킬을 만들면서도, 명시적으로 성능 평가(Eval)를 도입하지 않거나 누락하는 경우가 많다고 지적
- 이로 인해 실제 프로덕션 환경에서 스킬의 성능 저하, 동작 실패, 불안정성 증폭 등 문제가 빈번하게 발생함

### AI가 작성한 오픈 스킬의 99%는 Eval 없이, 신뢰성 검증도 거의 부재함

- Skill Bench 벤치마크는 GitHub 등에서 약 50,000개의 스킬을 수집·분석
- 이들 대부분은 AI가 자동으로 생성한 스킬이며, 테스트되지 않았거나 Eval이 전혀 붙어있지 않음
- 실제로 어떤 스킬이 “효과적인지/비효율적인지”, 그리고 에이전트의 실패 원인이 스킬 때문인지 모델/시나리오 문제인지 구분이 힘듦
- 예시: 실서비스의 고객은 “스킬” 개념을 전혀 모르기 때문에, 내부적으로 적절한 Eval 없는 스킬 배포는 품질관리상 큰 리스크가 됨

### 에이전트 사용자와 개발자는 스킬 인식 및 트리거 방식이 다름을 강조함

- 개발자는 직접 “스킬” 맥락을 이해하고, 명시적 호출(슬래시 명령 등)로 스킬을 테스트 및 활용 가능
- 고객이나 일반 사용자는 스킬 구조나 호출 방식을 전혀 모름. 예를 들어 환불 요청 시 “refund skill”을 직접 호출하는 게 아니라 자연어로만 요청
- 따라서 빌더와 엔드 유저 간의 스킬 사용 경험과 품질 기대치에 큰 차이가 존재함

### 스킬의 구조는 레이어별 정보 분할 및 “progressive disclosure” 방식으로 설계함

- 스킬은 폴더(디렉토리) 내 skill.md 파일과 필요한 참고 자료·에셋으로 이루어짐
- skill.md는 세 단계로 구성:
  - 1단계: 제목(title)과 설명(description, 보통 2~3문장), 모델의 기본 context로 항상 제공됨
  - 2단계: 본문(body, 추가 지시문 및 세부사항, 외부 참조 링크)
  - 3단계: 별도의 참고 파일(예: AWS용/Google Cloud용/특정 워크플로)로 상세 정보 전달
- 설명이 길어질수록 모델 토큰 비용이 높아지고, 간결하지만 핵심을 포함하도록 설계 권장

### 스킬은 ‘능력(Capability)’과 ‘선호(Preference)’ 두 가지 유형으로 나뉨

- 능력 스킬: 모델이 현재 일관되게 못하는 작업을 임시로 지원(예: 로그 추적, 신규 프레임워크 생성 등)
  - 모델 발전과 함께 삭제(retire) 가능. 언제 퇴역시킬지 판단에 Eval 결과가 필수
- 선호 스킬: 조직/팀 고유의 워크플로, 스타일, 도메인 지식 등 보존 목적
  - 도메인 특화이므로 모델 업데이트와 무관하게 오랫동안 보호·유지해야 하며, Eval 기반 성능 모니터링이 더욱 중요

### Skill Bench 데이터와 커뮤니티 사례로 성능 향상 근거 및 AI 스킬의 한계를 입증함

- Skill Bench 1.1 업데이트: 100여가지 코드와 생산성 중심 태스크에서, 개방형·폐쇄형 모델 대상으로 평가
- 결과: 스킬 도입 시 평균 15% 성능 향상 확인
- AI가 자동으로 만든 스킬은 오히려 성능을 저해할 수 있음
  - AI 스킬은 no-op(실질적 변화 없는 명령, 예: “코드를 읽기 쉽게 만드세요”)이 쌓이고 파일이 불필요하게 커짐
  - skill.md는 500줄, 2,000~5,000 토큰 이하로 유지 권장

### 스킬 작성 시 핵심 가이드 8가지를 구체 사례와 함께 소개함

- 가장 중요한 것은 “description” 작성: 모델이 언제, 왜, 어떻게 스킬을 사용할지 명확히 지시
  - “지시문(Directive)”를 사용, 서술적/소극적 설명은 피함(예:“Use the Interactions API if you are working on a chat application”)
  - “언제” 써야 하는지도 구체 명시
- 정보 레이어 구조: description(항상), body(선택), reference(특정 상황/Harness)
- 길고 절차적인 고정 workflow는 스킬 대신 “스크립트”로 분리
- 조건/제약 위주로 목표를 정의: 모델에 너무 구체 명령 필요하면 스킬 부적합, 반복 가능성 있으면 스크립트 활용
- Negative(네거티브) 테스트: “이 스킬을 사용하지 말아야 할 경우”도 반드시 명시·테스트
- 스킬 생성과정에서 10~20개 케이스의 프롬프트로 happy path/negative path 모두 테스트
- 실서비스 추적(postmortem) 데이터도 평가에 넣을 것
- No-op(무의미 명령)은 반드시 걸러야 하며, 평가를 통한 퇴출 필요(출처: Matt, AI 교육자)
- 스킬은 모델이 필요 없을 만큼 발전하면 반드시 Eval 결과에 따라 퇴역 가능성 검토

### 실제 Gemini Interactions API 스킬 개발 사례에서 효율적이고 강력한 평가 프로세스를 도입함

- Gemini Interactions API는 최신 인터페이스로, 모델 공개 시점과 학습 시점에 시간 차 존재
- 스킬을 사용해 모델이 최신 Gemini API를 활용하도록 유도
- 실제 테스트 케이스: 총 117개(실사용자, 합성, 피드백 기반)
- 주요 지표: “정확한 SDK/모델/메서드 사용 여부”, “구버전 패턴 사용 여부”
- 케이스별 프롬프트(JSON), 언어(TypeScript/Python), 트리거 여부, expected check 등으로 구조화
- Python 스크립트에서 Gemini CLI 등 harness를 호출해, 산출물을 분석(e.g. 리턴값 유효성, 정규표현식 체크 등)
- 규칙 기반 평가(assert), LLM Judge(채점 기준표 제공 후 판정)까지 유연하게 활용

### Google DeepMind 내 스킬 배포 시스템은 모든 변화 시 Eval을 필수로 강제함

- 내부 스킬마다 Test/Eval이 포함, 각 변경(diffs)마다 자동으로 전체 테스트 수행
- 워크스페이스/환경 설정, 사전 로드 명령, 정규표현식 기반 평가 스크립트, LLM 판정 항목 등으로 구성
- 테스트 통과하지 못하면 merge/배포 불가. 모든 변경은 성능 개선/테스트 추가 되어야만 반영 허용
- regression(리그레션) 테스트로 기존 스킬 퇴역/성능 평가 자동화

### 실전 스킬 관리 및 평가 체크리스트 10가지를 구체적으로 정리

- 50%의 실패는 description 불량(트리거 부족)에서 비롯됨
- description은 반드시 명령문/직설적으로, 사용자가 인지하지 않아도 명확해야 함
- 네거티브 테스트 누락 주의
- 10~20개 이상의 샘플 테스트 구축(적은 수라도 실제 더 많은 문제 발견 가능)
- 모델이 스킬을 트리거하는 ‘path’보다는 실제 ‘성과(outcome)’로 판단
- 각 시험 케이스별로 초기화된 환경에서 독립적으로 실행(모델이 context/로그을 활용한 ‘cheat’를 방지)
- 에이전트의 비결정성 고려, 한 케이스당 2~6회 반복 실험하여 신뢰도 측정
- 여러 Harness(실행 환경) 교차 평가(CLAUDE, anti-gravity, cursor 등 다수 적용)
- 스킬 업데이트로 불필요해진 eval도 계속 유지, 퇴역 후 모델 성능 감시 가능
- Eval이 항상 스킬 퇴역의 기준이 됨. 모델/환경 변화에 맞춰 주기적 관리 필수

### 사용자는 지금 바로 가장 많이 쓰는 스킬의 Eval 프롬프트 5개를 만들어보기 실습을 권유함

- 월요일에 돌아와 실제 업무에서 자주 쓰는 스킬을 선정, 5개 테스트 프롬프트 생성 권장
- 가장 많이 쓰는 경로/케이스를 코드/에이전트 로그 추적, 추출하여 프롬프트/케이스화
- Eval은 JSON(YAML)+Python 스크립트만으로 손쉽게 작성 가능
- 불필요한 no-op 제거와 ablation test(스킬 포함/미포함 비교실험)로 확실한 성능 증명 가능
- 핵심 결론: “Eval 없이 스킬을 배포하지 마세요(Don’t ship skills without evals)”
