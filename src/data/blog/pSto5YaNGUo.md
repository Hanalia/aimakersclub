## *에이전틱 AI 엔지니어: Mutagent의 자동화된 루프 기반 AI 개발 방법* 핵심 요약

- 본 영상은 Mutagent의 CEO 베네(Bene)와 CTO 부락(Burak)이 에이전틱(Agentic) AI 엔지니어 개념과 루프 기반 소프트웨어 개발법을 중심으로 설명함
- AI 에이전트 개발에서 오프라인 루프(개발·테스트·평가 반복)와 온라인 루프(배포 후 모니터링·진단·최적화)를 모두 자동화하는 중요성 강조
- 전통적인 수작업 루프는 인간의 검토·수정이 병목이 되어 대규모 에이전트 운영에 한계가 발생함
- 이를 극복하기 위해 스펙(요구사항) 정의, 빌드, 성능 평가, 배포, 진단, 최적화 등 모든 단계에 에이전트적 자동화 도입 필요성을 역설
- 성공적인 에이전트 개발을 위해 스펙-구현 분리와 환경 독립적 설계, 그리고 구체적이고 행동 가능한 평가(evaluation) 체계를 구축해야 함
- 반복적 진단과 루트 원인 분석, 자동화된 평가지표 보강 등 “학습된 실패 모드” 관리가 장기적 성능 개선의 핵심임
- Mutagent는 evaluator agent(평가기준 자동 구축), diagnostics agent(자동 진단) 등으로 개발 효율성을 크게 향상시킴
- 실제 데모에서는 오케스트레이터와 각종 연동(코드베이스, 지터브, 티켓·슬랙 이슈) 기반의 자동화된 오류 탐지와 수정 제안 과정을 보여줌
- 이 모든 과정이 코드형/플랫폼형/클라우드형 등 다양한 개발 환경에 유연하게 적용됨
- 결론적으로 반복되는 인간의 디버깅 부담을 AI로 대체해, 더 빠르고 신뢰성 높은 AI 에이전트 개발이 가능함을 실증함

## 세부 요약 - 주제별 정리

### 수작업 기반 루프의 한계 때문에 에이전틱 AI 엔지니어링이 필연적임

- 전통적으로 AI 에이전트 개발/운영은 사람에 의존하는 오프라인-온라인 루프(개발, 테스트, AB 테스트, 배포, 모니터링, 오류 대응)의 느린 반복으로 이루어짐
- 개발자가 직접 결과를 검토하고, 문제 발생 시 다시 개발을 반복해야 하므로 대규모 에이전트나 기능 증가 시 확장성에 한계
- 수작업 리뷰와 반복 작업이 전체 라이프사이클의 병목(bottleneck)이므로 자동화 필요성이 대두됨
- 이를 해결하기 위한 자연스러운 진화로 '에이전틱(Agentic) AI 엔지니어' 개념(에이전트가 루프를 담당하는 자동화 집합체)이 등장함

### 오프라인 루프(에이전트의 개발 및 테스트)와 온라인 루프(운영 중 진단·최적화)의 구조적 구분

- 오프라인 루프: 스펙 작성, 기능 개발, 샘플 생성 및 테스트, 평가, 반복 개선을 의미
- 온라인 루프: 실제 운영(프로덕션) 투입 후 사용 데이터 및 피드백을 기반으로 모니터링, 진단, 실패 원인 규명, 최적화를 반복
- 두 루프 모두 각 단계별 평가와 진단이 이어지며, 반복이 빠를수록 버그 해결과 성능 최적화 속도가 급격히 향상

### 스펙 기반 개발에서 요구사항 정의가 개발 루프의 중심임

- 에이전트 개발의 시작은 명확한 스펙(specification) 작성(역할, 기능, 의사결정 조건, 경계·제약 포함)
- 환경과 도메인에 따라 요구사항이 다르므로, 분명하게 책임과 “하지 않아야 할 일”까지 명확하게 정의
- 추후 코드 구현/플랫폼·프레임워크 변경에 독립적이도록 스펙과 구현(실제 에이전트 코드)을 분리함
- 스펙은 모든 추후 개선·검증과정의 블루프린트(청사진)로서 반복적 참고자료로 활용됨

### 프레임워크와 플랫폼 변경이 빈번하기 때문에 스펙-구현 분리가 중요함

- 에이전트 프레임워크와 러닝타임(예: Hermes, Deep Agents 등)은 빠르게 발전·교체됨
- 스펙 수준에서 플랫폼/프레임워크 의존성을 배제해야 향후 기술 변화에 유연하게 대응 가능
- 실제 경험적으로 harness 교체 필요성(예: 기능 부족, 병목 등)이 자주 발생했음

### 평가(evaluation)는 자동화와 행동가능성을 중심으로 설계해야 함

- 에이전트의 성능 판단 기준(평가 지표, 평가용 데이터셋)은 개발 루프의 핵심 관문임(테스트 없는 개발은 무의미)
- 평가 지표 작성은 도메인 전문가와 데이터 기반으로 진행하나, 특징상 사전 정의만으로 완전한 평가체계가 불가능
- 실전에서 사용자 피드백/운영 실패/에지 케이스 등으로 평가 데이터셋과 기준을 지속적으로 보강해야 함
- 200개 이상 데이터에 대해 수동 평가시 소요 시간/노력이 기하급수적으로 증폭되어 자동화가 필수

### 효과적인 평가는 binary 판단과 낮은 노이즈, 액션 가능한 피드백 제공에 초점을 맞춰야 함

- 단순 점수 기반(LLM as a judge) 평가는 판단 기준이 명확하지 않으면 개선점 도출이 어려움
- 행동 가능한(binary: 통과/실패) 평가 기준이 있으면 문제 발생 시 즉각 조치가 가능
- LLM(대형언어모델) 기반 평가자 사용 시, 평가 결과의 일관성(variance/noise)·칼리브레이션(성능 잡음) 관리 필요
- 실험이 재현 가능하도록, 평가 도구도 신뢰성 있는 방식으로 튜닝되어야 함

### 진단 단계에서는 수집된 실패 모드와 원인 분석, 그리고 평가 체계 강화가 핵심임

- 에이전트 운영 중 다양한 실패(오류) 유형이 누적되어 발생
- 오류 발생의 루트 원인을 분석(예: 프롬프트 문제, 도구(툴) 답변 오류, 환경 요소 등)하고 종류별로 클러스터링
- 진단 결과를 새로운 평가 기준/케이스로 환류시켜 지속적 품질 개선을 달성
- ‘학습된 실패 모드’ 데이터베이스를 축적, 향후 신속 진단과 코드 기반 자동 분류에 활용
- 수백만 개 트레이스(이력 데이터) 전체를 읽는 것은 비효율적이므로, 대표 샘플 추출 및 자동화된 인디케이터 활용

### 최적화는 자동 실험, 피드백 루프, 자동 배포로 완전자동화 가능

- 진단을 통해 확인된 실패 모드별 개선안(코드 변경, 프롬프트 수정 등) 제안·실행
- 새로운 개선안이 평가 루프를 통과하면 자동으로 프로덕션에 배포(updating, auto PR 등)
- 외부 피드백/이슈를 진단에도 반영(티켓, 슬랙 등 연동)
- 전 과정이 반복 구성된 하나의 end-to-end 루프로 운영됨

### Mutagent 플랫폼은 전체 루프를 자동화하는 연구 미리보기 에이전트 세트를 제공함

- evaluator agent: 자동으로 평가 기준 작성, 데이터셋 집합화, 평가 스위트 구성 기능 제공
- diagnostics agent: 생산 환경 트레이스 자동 진단, 주요 오류 탐지, 원인 분석, 개선안 제안 자동화
- 각 에이전트는 orchestrator(오케스트레이터)로 연결되고, 외부 소스(코드, 이슈, 로그 등)와 연동됨

### 실제 제품 데모에서는 orchestrator 기반으로 오류 탐지·분석·수정안 적용까지 시연함

- 코드 환경(예: Code Code)에서 orchestrator가 전체 에이전트 세팅·워크플로우 단계를 대시보드 형태로 표시
- 진단 명령 입력 시 자동으로 트레이스 수집, 대표 샘플링, 다단계 필터링을 통해 주요 실패 모드 분석
- 각 실패 모드별 오류 빈도, 원인(재귀형 why-chain), 가정 리뷰, 구체적 개선안( remedies) 제공
- 제안된 수정사항이 마크다운 태스크 정의로 변환되어, 개발자가 직접 코드 반영 가능(자동 PR 등도 연동)
- 텍스트-to-스피치 기능(Whisper 등) 및 다양한 피드백 수집용 인터페이스 제공

### 전체 개발-운영 사이클이 하나의 완전자동 루프로 연결됨

- 스펙 정의→에이전트 빌드→자동 평가 루프→프로덕션 배포→운영 진단→오류 개선 및 자동 재배포→데이터 축적
- 모든 단계가 에이전트화되어 반복순환(offline+online loop)하며, 기존 인간 개발자의 반복/검토 작업을 AI가 대체
- 마음껏 기능/에이전트 스케일링 가능
- 궁극적으로 AI 엔지니어링에서 “디버깅의 자동화”, “신속한 피드백 주기”, “점진적 품질 향상” 실현 가능
- Mutagent는 향후 관리형(Managed) 서비스도 계획 중임
