## *지금 우리는 무엇을 만들어야 하는가?* 핵심 요약

- 영상은 테오 브라운(Theo Browne)이 AI 모델의 급격한 발전이 소프트웨어 개발 방식과 스케일, 사고방식에 미치는 영향을 사례와 경험을 바탕으로 통찰적으로 설명함
- 소네트 3.5, 오퍼스 4.5, 미토스 등 최신 언어 모델들이 순차적으로 어떻게 업무 자동화와 코드 생산성의 질적 도약을 이루는지 구체적으로 비교
- 최근 AI 모델이 단순 코드 작성은 물론, 테스트, 검증, 자체 모델 스폰까지 이뤄내며 스스로 작업을 분할 및 오케스트레이션할 수 있음을 강조함
- 개발자가 기존에 당연시했던 개발 문화(도구 집착, 환경설정 공유, 언어/프레임워크 정체성 등)에 대한 문제의식을 제기하며, 낡은 습관을 넘어설 필요성을 주장
- 실체험 기반으로 ‘레딧 스크래퍼’ ‘Zoom for Streamers(핑)’ ‘풀스택 클라우드’ 등 본인이 진행한 프로젝트들을 진화하는 AI 역량과 비교하여 등급 구분
- AI 기술 발전으로 인해 기존 ‘스타트업·사이드 프로젝트·비현실적’의 규모 단계가 모두 한 단계씩 내려감(기존 스타트업 아이디어가 이젠 사이드 프로젝트로 가능)
- 마크다운 파일을 활용한 자동화 사례 등 오늘날엔 수많은 ‘스타트업 제품’이 단순 마크다운 또는 몇 줄 프로세스로 대체 가능함을 실제 경험과 함께 시연
- 소프트웨어의 ‘넓이(Breadth)’와 ‘깊이(Depth)’ 스펙트럼을 도식화하며, 이제 소규모 팀도 Vercel·AWS·슬랙·세일즈포스 수준의 플랫폼 경쟁이 현실화되었음을 역설
- 사용자가 스스로 기능을 확장할 수 있는 유연한 아키텍처를 강조하며, 기존에는 불가능하다고 생각했던 스케일의 제품 구상도 이제 도전 가치가 있음을 주장
- 결론적으로 오늘날의 개발자는 기존 상식이나 두려움을 넘어 ‘너무 크게 느껴질 만큼’ 과감하게 시도해야 진정한 기회가 열린다고 독려

## 세부 요약 - 주제별 정리

### AI 모델 진화가 소프트웨어 자동화의 단계 자체를 바꾼다

- 테오 브라운은 AI의 빠른 발전에 따른 "AI 사이코시스"에 공감하는 청중을 환기하며 강연 시작
- 소네트 3.5 모델은 도구 호출 기반으로 실제 코딩 업무에 일관성 있게 활용 가능한 최초의 모델로 인식됨
- 오퍼스 4.5 모델부터는 개별 단계 요구 없이 전체 작업 목표를 지정하면, 모델이 스스로 일련의 복잡한 작업을 추적/분할/완료 가능
- 미토스(그리고 Fable 등 최신 모델)는 코드베이스 이해를 넘어서 스스로를 이해하고, 다른 모델을 스폰/작업 분산/최종 검증까지 가능
- 단순히 Prompt만 달리하면, 이제 자체 툴링이나 커스텀 소프트웨어 없이도 다단계 작업 오케스트레이션이 가능함을 실례로 강조
- 과거에는 개발자가 단계별 개입이 필요했다면, 이젠 AI가 장시간 복잡한 작업도 자율적으로 완결함

### AI 발전 속도가 인간의 숙련 속도를 압도적으로 앞지르고 있다는 자각

- 테오는 한때 “AI 발전이 한계에 다다랐다”고 착각했으나, 모델이 개발자보다 훨씬 빠르게 발전함을 체감
- 자신의 과거 직장에서 처리하던 대부분의 지라(Jira) 티켓이 현재의 Opus 4.5 수준 AI로 trivial하게 해결 가능함을 자평
- “우리가 더 좋아질 수 없다면, 이제는 더 크게 나아가는 수밖에 없다”는 생각의 전환을 주장

### 기존 개발 문화와 도구습관은 더 이상 변혁에 걸림돌이 되고 있다

- 강연자는 10년 이상 소프트웨어를 개발하며 GNU screen, tmux, ssh 등 특정 도구와 방식에 익숙해짐을 솔직히 시인
- 아이폰 UI 변천사(iOS 6→7)를 비유로 들어, 과거는 실물 도구의 스큐어모피즘(실물 구현)에 집착했지만 이후 ‘진짜 효율·편의성 중심’으로 혁신되었음을 설명
- 현재 개발자들은 여전히 터미널, Vim, 언어 등 기존 툴셋에 과도한 집착과 정체성을 부여하는 경향이 있음을 꼬집음
- 개발자들이 환경변수 파일 공유, 언어·프레임워크로 자기 정체성 인증(“저는 자바스크립트 개발자입니다”) 등 불필요한 관행에 매몰됨을 비판

### ‘코드 삭제에 대한 두려움’ 등 조직적 낡은 관행이 혁신을 저해함

- 산업 내 ‘침몰 비용(sunk cost)’ 심리로 인해 비효율적인 코드도 쉽게 지우지 못하는 실상을 예로 듦
- ‘PR(풀리퀘) 죄책감 머지’ 등, 동료의 노고를 생각해 완벽하지 않은 코드를 합치는 경우도 많음을 고백
- 새로운 AI 시대에는 이런 집착에서 벗어나 변화를 수용하는 유연성이 중요하다고 역설

### 본인이 만든 프로젝트를 통해 트렌드 변화를 실감함

- ‘레딧 스크래퍼’(밈 자동 수집기), ‘Zoom for Streamers’(핑, Y Combinator 경험 포함), ‘풀스택 클라우드’ 등 3가지 프로젝트를 본인 사례로 소개
- 불과 1, 2년 전엔 각각 사이드 프로젝트, 스타트업, ‘너무 큰’ 아이디어로 각각 분류됐으나, 지금은 한 단계씩 아래로 내려감
- 예컨대 과거 스타트업 규모가 오늘날에는 간단한 사이드 프로젝트로 구현 가능해짐

### 단순한 ‘마크다운 자동화’만으로도 스타트업 제품을 대체할 수 있다

- PR 리뷰 및 우선순위 지정 등 본인이 과거 독립된 ‘서비스’로 만들었던 기능이, 현재는 단순 마크다운 파일과 cron job(스케줄러)으로 구현됨
- “마크다운 파일을 Codex나 Claude 등 모델에 파이프만 해도 실행 가능하다”는 점을 실제 예시와 함께 강조
- 매일 아침 9시, 크론을 통해 자동 생성된 스태틱 HTML이 S3에 업로드되고 업무 리포트 받는 방식 등, AI 활용 일상의 구체 사례를 공유

### 기술 발전으로 ‘너무 크다’고 여겼던 아이디어의 한계 자체가 불분명해졌다

- 기존 등급 구분(사이드 프로젝트/스타트업/너무 큼)의 ‘너무 큼(Too Big)’이라는 경계가 무너지고 있음을 체감
- 예컨대 ‘독자적 AI 모델 트레이닝’이나 ‘자체 OS 개발’, ‘NPM/Node 대체’ 등 무엇이 진짜 불가능한지 모호해짐
- 불확실성과 두려움 대신, “한계 Push 및 도전이야말로 필요한 자세”로 강조

### ‘더 크게(Bigger)’보다 ‘더 넓게(Wider)’ 사고하는 것이 중요하다

- 개발 규모의 ‘크기’ 개념은 깊이(Depth)와 넓이(Breadth)로 구분 가능
- B2B 예시로서 Vercel(특정 영역에서 깊은 서비스)과 AWS(넓고 다양한 서비스)를 대조 설명
- 전통적으론 소규모 팀이 AWS와 같은 ‘넓이’를 커버 불가했지만, 최신 AI는 다양한 기능 제공이 단기간에 가능케 함
- 데이터베이스 플랫폼 등 주요 기능도 AR 명령어와 휴먼 effort만으로 1~2일 내 구축 가능

### 사용자에게 확장성을 맡기는 플랫폼 아키텍처가 주목받는 시대

- 직접 지원하지 않는 기능은 사용자가 스스로 Slack Bot API 등으로 확장 가능하다면 문제 없음
- Slack의 사례처럼, 기본 제품(채팅) 외에도 플랫폼 확장성이 제품 성공의 열쇠가 됨
- 사용 사례도 예기치 못하게 다변화될 수 있으니, 처음부터 확장 가능성·유연성 중심 설계 필요

### 무모하다 싶은 아이디어일수록 실제로 새로운 시장이 열린다

- “이젠 직접 Slack, AWS, Salesforce와 경쟁할 만큼 크게 생각해야 한다”고 청중 독려
- “내 아이디어가 바보 같아 보이지 않을 경우, 충분히 크지 않다는 뜻”이라는 도전적 멘트로 강연을 마무리
