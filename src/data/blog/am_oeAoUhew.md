## *인간이 방향을 제시하고, 에이전트가 실행하는 소프트웨어 개발 방식: Harness Engineering의 실제 사례* 핵심 요약

- 본 강연은 OpenAI 엔지니어 Ryan Lopopolo가 ‘Harness Engineering’을 주제로, 인간이 방향을 제시하고 AI 에이전트가 소프트웨어 개발을 담당하는 미래형 개발 조직 운영 방식을 소개함
- Ryan은 9개월 동안 오로지 에이전트로만 소프트웨어를 개발하는 팀을 이끌며 에디터를 직접 건드리는 것을 금지하는 실험을 수행함
- AI 모델(예: GPT-5.2/5.4)은 이제 고품질 코드를 무한대로 생산할 수 있는 수준에 도달, 코드 생산은 GPU 용량·토큰 예산만이 제약 조건임
- 인간 엔지니어의 역할은 코드 작성에서 시스템 설계와 에이전트 및 프로세스의 설계‧위임으로 이동함
- 코드 생산의 병목은 더이상 ‘구현’이 아니라, 사람의 시간/집중력, 모델의 컨텍스트 윈도, 주의 할당에 있음
- 코드가 무한대로 생산됨에 따라, 우선순위가 낮던 작업(P3)이 병렬로 모두 착수되어 문제해결 능력이 비약적으로 향상됨
- 소프트웨어 개발의 핵심은 코드 그 자체가 아닌, 요구사항 명세, 문서화, ADR, 가이드라인 등 ‘프롬프트와 가드레일’ 관리를 통한 일관성 보장임
- 리뷰 에이전트, 린트, 맞춤형 테스트 등 다양한 자동화 도구를 통해 코드의 품질, 신뢰성, 보안성을 지속적으로 관리하고 ‘슬롭’(품질 저하)을 조직적으로 제거함
- 저장소/코드베이스를 ‘에이전트 친화적’ 구조로 일관성 있게 설계‧구현하여 에이전트의 컨텍스트 활용을 극대화함
- 에이전트 및 인간 협업은 문서, PR, Markdown 기반으로 투명하게 이루어지며, 수평적 피드백과 고속 병렬처리를 통해 생산성을 극대화함

## 세부 요약 - 주제별 정리

### AI가 소프트웨어 엔지니어의 전체 역할을 수행할 수 있게 된 결정적 전환점이 2025년에 찾아옴

- 최근 6개월간 코딩 에이전트(예: GPT-5.2, GPT-5.4 등)는 소프트웨어 엔지니어의 업무 전체를 해낼 정도로 성능이 급격히 향상됨
- ‘코드는 무료’가 현실이 되었으며, 개발팀에서 에이전트 인력을 할당하는 유일한 제약은 GPU 용량과 토큰 예산임
- 엔지니어 1명이 ‘5명’, ‘50명’, ‘5,000명’의 일을 동시에 감당할 수 있게 되어, 효과적으로 이 자원을 활용하는 시스템 구축이 핵심 과제로 전환됨
- 실제로 Ryan 팀은 인간이 에디터를 사용하는 것을 금지하고 에이전트만으로 구현 전 과정을 해결함
- 이 변화로 엔지니어가 해야 할 일은 코드를 작성하거나 리뷰하는 것이 아니라, 시스템적 사고·설계·위임 및 프로세스 관리로 이동함

### 코드 생산이 무한히 가능해지면서, 구현이 아닌 컨텍스트와 인간 자원이 병목으로 바뀜

- 기존에는 코드 작성이 소프트웨어 개발의 주요 병목(Bottleneck)이었지만, 이제는 인간의 시간, 인간-모델의 주의(Attention), 그리고 모델의 컨텍스트 윈도우가 희소 자원으로 전환됨
- 인간 엔지니어는 단기간(1일~6개월) 미래를 내다보며, 시스템적으로 무제한 코드 생산 능력을 효율적으로 사용하도록 ‘구조’를 설계해야 함
- 우선순위(P0, P1, P2) 이외에 묻혀 있던 중요도 낮은 일거리(P3 등)도 병렬적으로 시도 후 효과적인 것을 채용하는 전략이 가능해짐
- 현업에서 모든 반복적 작업, 내부화(localization/I18N), 유지보수/리팩토링까지 손쉽게 자동화됨

### ‘프롬프트와 가드레일’이 소프트웨어 품질과 생산성의 핵심이 됨

- 코드 그 자체보다 어떻게 명세서, ADR, 퍼소나별 문서, 티켓·코드리뷰 기록 등 ‘의사결정의 이력’이 쌓였는지가 중요해짐
- 이런 각각의 과정이 에이전트에게 요구사항, 품질 기준, ‘좋은 코드란 무엇인지’를 전달하는 핵심 장치(가드레일)가 됨
- 예측 가능한 구조, 반복되는 문서 작성, 로그 축적, 일관적 설계 등은 컨텍스트 활용 극대화와 에이전트의 일관된 행동 유도에 필수적임
- 예: 코드에서 네트워크 요청마다 타임아웃/리트라이를 강제하는 린트, 도큐먼트 등으로 실수를 미연에 방지

### 리뷰 에이전트, 린트, 테스트 자동화를 통해 조직적으로 ‘슬롭’(저품질 코드)을 제거함

- 코드 리뷰의 병목을 줄이기 위해, 매주 금요일 ‘가비지 컬렉션 데이’로 문제되는 코드 품질 저하 원인을 식별하고, 자동화 도구(린트, 테스트, 에이전트 등)로 시스템적으로 일괄 제거
- 에이전트는 반복적으로 실패하는 패턴, 비효율, 기술부채를 빠르게 파악하여 문서화 및 자동화로 해결
- 패치당 500개 이상의 비명시적(NFR) 요구사항이 있을 수 있으나, 에이전트는 트릴리언 이상의 코드 예시를 바탕으로 적응 가능
- 요구사항 미달인 영역을 발견하면, 린트, 테스트, 검토 에이전트, 명시적 프롬프트 추가 등으로 정밀하게 보완

### 에이전트 친화적 코드베이스 구조와 일관성이 조직·대규모 코드 관리의 성패를 좌우함

- 저장소를 750개 패키지로 분리(도메인/계층별)하고, 재사용 유틸리티, 공통 헬퍼를 일관되게 구현 및 사용 강제
- 모든 개발 도구와 로컬 개발 환경을 Codeex/에이전트가 먼저 호출하는 구조로 하여, 인간은 추상화된 skill만 사용
- 패키지 프라이버시, 계층 간 의존성, 파일 길이 제한(예: 350라인 이하) 등 저장소 구조 자체가 ‘프롬프트=명시적 규칙’ 역할
- ‘코드는 텍스트’라는 점을 활용해, 저장소 내 일관성을 강화함으로써 에이전트의 문맥 전이 학습 효율을 극대화함

### 프롬프트/가이드 인젝션은 모델 성능 향상과 긴밀히 연결되며, 운영 조직도 변화하고 있음

- 리뷰 에이전트, 린트 에러 메시지, 프롬프트 파일(.mmd) 등 다양한 방법으로 세부 규칙을 지속적으로 모델에 주입함
- 도메인별(예: 프론트엔드, 백엔드, QA 등)로 좋은 결과 사례, 작성법, 리뷰 기준을 문서화하여 PR 리뷰 에이전트가 항상 참조·적용
- 프롬프트 자체 작성도 에이전트(코덱스)에 위임하여 효율화
- 최근 모델은 인퍼런스 과정에서 컨텍스트 페이징, 오토코(autoco) 컴팩션 등으로 컨텍스트 활용 최적화됨
- 에이전트 SDKS와 검토 테스트, 주기적 문맥 갱신을 통해 방대한 코드/문서의 활용성을 높임

### 수평적이고 고속 병렬적인 인간·에이전트 협업 체계는 기존의 리뷰·커뮤니케이션 관행을 바꿈

- GitHub/Markdown/Pull Request 중심의 허브-스포크 구조로 인간-에이전트가 동등하게 협업
- PR은 각 주체가 피드백을 자율적으로 반영하거나 거부할 수 있으므로 블로킹 없이 빠르게 진행
- 기존보다 훨씬 더 빠른 병렬처리 및 피드백 루프가 가능
- 지나친 피드백 강제는 오히려 모델이 비효율적 ‘괴롭힘’에 노출되어 회피해야 하며, ‘완벽’이 아니라 ‘충분히 좋은’ 상태로 목표를 조정

### 테스트, QA, 의사결정 문서화를 통해 신뢰성을 자동화함

- 에이전트가 직접 QA 플랜 작성법, 실제 테스트 커버리지 산정, 주요 기능 목록화, 유저 여정 경로 등 귀속자료를 기반으로 PR 생성 및 검증
- QA 기준이 문서화되면, QA 리뷰 에이전트가 모든 PR에서 자동으로 해당 기준을 점검함
- 주요 부분은 자동화되어, 인간 엔지니어의 개입 빈도를 점점 줄이고, 더 높은 수준의 의사결정에 집중하게 함

### 새로운 개발 문화에서, 조직 성장과 코드베이스 확장에 필요한 구조적/절차적 대응이 중요해짐

- 저장소 설계, 패키지 구조, 공통 패턴 등에서 일관성을 키우면 에이전트의 컨텍스트 활용 성능이 자연스럽게 향상됨
- 도메인별 유틸/헬퍼를 단일화하고, 구조화된 하위 디렉터리·서브트리를 통해 모듈별 컨텍스트 범위를 한정하는 것이 효과적임
- 팀 내부 ‘딕테이터’ 혹은 총의로 표준을 정하고, 이를 해당 코드로 직접 변환 혹은 전사 이동(migration)시킴

### 코드리뷰, 테스트, 품질관리는 자동화와 세분화된 역할을 통한 병렬 처리로 전환됨

- 병렬 PR 처리(1인당 하루 3~5 PR)은 merge conflict를 최소화하기 위해 코드 트리 구조 개선, PR 오픈 기간 최소화, 자동 리뷰 적용 등으로 대응
- 코드리뷰 피드백은 ‘페르소나 기반 리뷰 에이전트’로 자동화, 각 역할별 기준 문서를 참조하여 품질 기준 충족 여부를 PR마다 독립적으로 검증
- 오프로드 가능한 반복 업무는 지속적으로 자동화·시스템화하며, 인간 집중력이 필요한 고차원 절차로 전환

### 토큰 소비관리는 ‘설계-기획-문서화/리뷰-CI/배포’ 비율로 균형 있게 이뤄지며, 사용 예산 전략이 존재함

- 대규모(백만~수십억 토큰 단위) 토큰 소비는 기획, 문서화, PR, CI 등 3분할로 사용
- 예산·프로플랜($200선)에서 토큰 한도를 신경 쓸 때는, 반복적 블록(코드 전체 복붙)이 아닌 효율적인 사용(부분 변경, 캐시 활용, CI 연계)으로 최적화 가능
- Plan Mode(계획 모드) 사용 시, 계획(PR)을 반드시 직접 읽고 수락하는 절차로 활용하는 것이 권장됨

### 소프트웨어 생산의 미래는 ‘코드=일시적 산출물’, 시스템 설계, 컨텍스트·프로세스 엔지니어링으로 이동함

- 코드 자체는 소모성(disposable) 빌드 산출물이 됨
- LLM(대규모 언어 모델)을 도입한 코드 생성/검증/운영은 ‘스태틱 컴파일러의 최적화 패스’와 유사한 구조로 작동
- 인간 엔지니어의 역할은 비즈니스 목표, 신뢰성 지표, 수개월~1년치 작업 우선순위 설정 등 고차원 프로그래밍으로 이동
- 에이전트는 피드백/유저 피드/운영 등 제품 성장 전 과정에서 활용 가능하며, 명확한 프로세스 문서화 및 표준화가 필수
