## *혼돈에서 안무로: 실제로 효과적인 멀티 에이전트 오케스트레이션 패턴* 핵심 요약

- 발표자 Sandipan Bhaumik는 AWS와 Databricks에서 오랜 기간 분산 데이터 시스템, 특히 금융·헬스케어의 데이터 및 AI 플랫폼을 구축·운영한 경험을 바탕으로 실전 멀티에이전트 AI 시스템 적용 사례와 교훈을 공유함
- 단일 에이전트의 데모 → 멀티에이전트(5개) 확장 시 분산 시스템 문제(레이스 컨디션, 상태 동기화, 장애 전파 등)가 기하급수적으로 증가함
- 실제 금융서비스 신용평가 시스템 사례에서 캐시 무효화 실패로 인해 생산 환경에서 20% 이상의 위험 평가 오류 발생, 원인은 단순 AI 모델이나 프롬프트 문제가 아닌 분산 시스템 설계 미흡
- 멀티에이전트에서 두 가지 핵심 패턴(Choreography: 이벤트 중심, 분산/ Orchestration: 중앙 집중, 통제)을 소개하고, 각 패턴의 장단점 및 사용 적합성을 상세히 분석함
- Orchestration 패턴 도입 시 에이전트 간 직접 통신이 아닌 중앙 오케스트레이터가 모든 과정을 관리해 복잡한 의존성과 트랜잭션 처리, 롤백, 디버깅이 용이
- 상태 관리 측면에서 “불변 상태 스냅샷 버저닝” 방식(append-only, 버전 관리, 불변성)과 데이터 계약(스키마/형상 검증)의 중요성을 강조, 동시성/레이스 컨디션 문제 예방
- 장애·복구 설계에서 Circuit Breaker(에이전트 장애시 빠른 차단/회복)와 Compensation/Saga(분산 트랜잭션 롤백) 패턴을 실제 코드 및 운영 시나리오 예시와 함께 구체적으로 설명
- Databricks에서 실제 멀티에이전트 오케스트레이션 및 상태, 계약, 장애 관리(Delta Lake, Unity Catalog, MLflow 등) 구현 예시로 생산 환경 적용 방안을 제시
- 핵심 메시지는 “멀티에이전트 AI 시스템의 본질은 분산 시스템이며, 제대로 된 인프라/패턴 설계 없이 데모 수준 적용은 불가하며(특히 비즈니스 가치/신뢰성 측면), 반드시 위 패턴들을 실무 아키텍처에 적용해야 함”
- 데모는 쉽지만 프로덕션 시스템은 어렵고, 진정한 시스템 엔지니어링의 중요성을 강조하며 발표를 마무리함

## 세부 요약 - 주제별 정리

### 실제 분산 시스템 경험에서 멀티에이전트의 함정과 본질적 어려움이 드러남

- 발표자 Sandy는 18년간 데이터 시스템을 설계·운영했으며, AWS·Databricks에서 금융/헬스케어 SaaS/AI 플랫폼을 제작한 경험을 바탕으로 발표함
- 지난 2년간 멀티에이전트 AI 시스템을 생산 환경에 도입 및 운영하며 수많은 동일한 실수를 목격했다고 밝힘
- 많은 엔지니어들이 “에이전트 추가=기능 추가”라고 착각하나, 실상은 분산 시스템 학문과 동일한 복잡성 및 함정을 가짐
- 본 발표의 목표는 “실제로 통하는” 멀티에이전트 오케스트레이션 및 상태 관리, 장애 대응 패턴을 공유하는 것임

### 단일 에이전트에서 멀티에이전트 도입 시 분산 시스템 문제가 폭발적으로 증가함

- 단일 LLM 기반 에이전트/프롬프트/도구는 데모에서 매우 잘 동작함
- 하지만 실제 서비스 요청(예: ‘5개 에이전트 추가’) 수락 후, 다음과 같은 복잡성이 생김:
    - 에이전트 간 데이터 의존(Agent A → B, C는 A·B 모두 필요, D는 B의 공유 상태 읽기 등)
    - 한 에이전트(예: E) 장애가 전체 워크플로우 중단시킴
- 이는 더 이상 AI 프롬프트/모델이 아닌 분산 시스템 엔지니어링 과업임을 실감함

### 신용평가 시스템 실전 배포 사례에서 치명적인 분산 아키텍처 실패가 발생함

- 배경: 금융회사의 신용평가 시스템에 처음에는 신용점수 계산 에이전트(Agent 1)만 배포, 2주간 무사고 운영
- 이후 ‘소득 검증·리스크 평가·사기 탐지·최종 승인’ 4개 에이전트 추가, 총 5개로 서비스 확장
- 3일 만에 신용 승인 결과의 20%에서 리스크 평가 오류 및 비정상 승인 발생 (실제 고위험 고객 승인 등)
- 디버깅 결과: 캐시된 데이터가 적시에 무효화되지 않음(Agent 1는 최신 신용점수 750 기록→Agent 2는 캐시된 680을 읽음 → 잘못된 데이터 기반 판단)
- 문제는 데이터베이스나 AI가 아니라 “캐시 계층의 레이스 컨디션/아키텍처 미흡”임
- “분산 시스템 사고 없이 분산 시스템을 구축”한 것이 실패 원인임을 강조

### 멀티에이전트 확장 시 복잡성은 선형이 아니라 기하급수적으로 커짐

- 에이전트 수가 늘수록 연결 개수 및 조정 문제(의존성, 상태 동기화, 장애 전파)가 폭증함
- 예시: 1개→5개 에이전트 확장 시 조정 포인트는 5배가 아닌 최소 10배로(각각의 조합마다 레이스/장애점 존재)
- 단순히 “에이전트 n개를 추가”하는 것이 아니라 n x n개의 “상호 관계 조정, 장애 처리” 문제를 해결해야 함

### 에이전트 조정을 위한 두 가지 패턴(Choreography와 Orchestration)은 장단점과 쓰임새가 명확히 다름

- Choreography(안무): 이벤트 기반, 에이전트 각자가 관심 이벤트만 구독·게시하며 분산·자율적으로 동작
    - 예시: 리서치 완료 이벤트 전달→분석 에이전트가 수신 및 분석→리포트 에이전트가 최종 보고서 생성
    - 장점: 루즈 커플링(느슨한 결합), 에이전트 추가 용이성, 고자율성, 확장성
    - 단점: 장애·버그 추적(이벤트 누락/중복소비) 난이도 급증, 관찰성(observability)이 부족하면 디버깅 불가
    - 사용 적합성: 자연스럽게 이벤트 기반, 자주 에이전트 추가, 자율성 높은 워크플로우에 적합 (단, strong tracing 필수)
- Orchestration(지휘): 중앙 오케스트레이터가 모든 흐름을 직접 관리, 각 에이전트는 바보(단순 입·출력만)
    - 예시: 오케스트레이터가 순차/병렬 호출·상태 관리·장애 처리 등 위임
    - 장점: 전체 트랜잭션 명확함, 장애/인트레이스 용이, 롤백·재시도·모니터링 일원화
    - 사용 적합성: 복잡한 의존성/롤백/상태 추적이 중요한 금융 등, 워크플로우 변경이 드물 때 용이
- 혼합형(Choreography+Saga 등)도 복잡+자율성 양쪽 모두 필요한 경우 등장 가능

### 생산 환경에서의 실제 오케스트레이션 아키텍처는 단일 지휘체제와 불변 상태로 구성됨

- 핵심 구성: 오케스트레이터(워크플로우 엔진+상태 저장+관찰성 관리)가 모든 에이전트 호출·데이터 흐름·로그 추적 담당
- 에이전트끼리는 “직접 소통/공유 상태 조작 불가”하며, 항상 오케스트레이터를 통함
- 실무에서는 Langraph+Mosaic AI Agent Framework(오케스트레이션), Unity Catalog(에이전트 버전관리/스키마), Delta Lake(불변 상태저장), MLflow(트레이스), AI Gateway(서빙-장애정책) 활용이 권장됨

### 상태 관리는 “불변/버전 관리 스냅샷 + 데이터 계약”으로 구현해야 레이스와 버그를 예방함

- 실무에서 가장 흔한 잘못: “여러 에이전트가 동일 DB 레코드에 동시 쓰기/읽기”(shared mutable state) → 마지막 쓰기만 남는 race bug
- 가장 효과적인 방식: “불변 상태 스냅샷(append only, versioning)” 활용
    - 각 에이전트는 상태 v1→v2→v3...형태로 본인이 받은 상태만 신규 버전으로 저장(수정 불가)
    - 각 스냅샷엔 버전, 생성자, 데이터 페이로드 등이 포함
- 코드 예: Python의 frozen(immutable) dataclass, handoff 함수에서 계약(스키마) 검증→버전 증가→새 객체 생성→다음 에이전트 호출
- 장점: 병렬 수정/오래된 데이터 참조를 막고, “버전 히스토리” 덕분에 트레이스/디버깅 용이

### 데이터 계약(Data Contract)은 에이전트 간 단순 값 전달이 아닌 스키마 및 검증 체계를 반드시 요구함

- 예: 리서치 에이전트의 출력(‘findings, confidence, sources, timestamp’ 등)→분석 에이전트의 요구 입력(스키마)이 명확해야 하며, 유효성 기준(예: confidence>0.7)이 충족되지 않으면 즉시 거부
- 각 에이전트의 입력/출력 스키마 명세는 중앙(예: Unity Catalog)에서 버전관리화함
- 계약 덕분에 데이터 품질 문제를 “중간에서 바로 탐지” 가능, 유효하지 않은 데이터가 멀리까지 전파되는 것을 막음

### 장애는 필연적이므로 Circuit Breaker 및 Compensation/Saga 패턴으로 복구 내성을 내장해야 함

- Circuit Breaker:
    - 일정 실패 횟수 이상이면 연결을 “닫고”, 일정 시간 후 재시도, 성공 시 회복, 실패하면 다시 닫기를 반복
    - 장점: 연쇄 장애 전파 차단, 전체 시스템 정지 방지, 불필요한 반복 호출 차단
    - Databricks에서는 모델 서빙/AI Gateway에서 Circuit Breaker 정책(재시도, Rate limit 등) 적용
    - MLflow에서 Circuit Breaker 상태 변화를 로깅해 모니터링
- Compensation/Saga:
    - 에이전트마다 “실행(execute)”과 “보상(compensate)” 메서드 모두 구현 필요
    - 장애 발생시 오케스트레이터가 실행 순서 반대로 보상 메서드 호출→부분 트랜잭션/상태 모두 되돌림
    - 금융/규모 시스템에서는 필수적인 분산 트랜잭션 롤백 방식

### Databricks의 실제 아키텍처 예시는 각 계층별 신뢰성·추적·통제기능이 결합된 형태임

- 오케스트레이터(예: Langraph/Mosaic AI Agent Framework): 멀티에이전트 워크플로우 전체 제어/흐름관리
- 에이전트: SQL/Python 함수 또는 ML 모델로 Unity Catalog에 등록(버전관리/중앙발견/권한관리/데이터계보)
- 서빙 레이어: Model Serving/Function Serving을 통해 Circuit Breaker/재시도 등 정책 적용
- 스토리지: Delta Lake에 상태 스냅샷/버전, 고객 데이터 등 모든 이력 불변 저장, 에이전트 실행 트레이스는 MLflow 로그와 연동
- 전체 워크플로우: 오케스트레이터가 상태 흐름/에이전트 결과/장애 처리(보상 패턴 포함)까지 한 곳에서 트랜잭션 및 로깅

### 모든 멀티에이전트 시스템은 필연적으로 분산 시스템 엔지니어링과 복잡성 곡선을 마주하게 됨

- 에이전트 수가 늘 때마다 혼돈(레이스/상태 이상/실패 모드 등)이 필연적으로 발생, 고민 없이 데모 수준 적용은 실패로 이어짐
- 핵심 패턴(오케스트레이션, 불변 상태, 계약, 장애 복구 등)을 반드시 실무 아키텍처에 도입해야만 실전 시스템 운영이 가능함
- 인프라/패턴 작업은 화려한 데모보다 보이지 않는 영역이지만, 장기적으로 비즈니스 신뢰성과 가치를 만든다고 강조
- 시스템 엔지니어링적 자세의 필요성과, 오늘 소개한 패턴들의 실질적 효과를 끝으로 발표를 마무리함
