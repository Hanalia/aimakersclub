## *최첨단(Frontier) 대형 모델이 아닌 소형 모델 기반 음성 에이전트의 강점* 핵심 요약

- 영상은 Microsoft 소속 Joel Allou와 Ornella Bahidika가 제작한 실시간 AI 음성 튜터 'Ace'의 설계 원칙과 실행 전략을 소개함
- Ace는 거대한 최첨단 언어모델(예: Claude 4.7)이 아닌 작은 언어모델(예: Haiku 4.5)을 의도적으로 사용함
- 음성 인터페이스에서는 약 950ms 이내 응답이 필요한데, 대형 모델들은 수초의 reasoning 과정을 거쳐 실시간성에 부적합함
- 대형 모델의 복잡한 추론(reasoning) 능력이 오히려 사용성(특히 대화 중 응답 속도) 저하로 이어짐을 강조
- Ace는 모델이 직접 추론하는 것이 아니라, 수업 흐름·학생 이해도 관리 등을 코드(상태 머신, 인텔리전트 레이어)에서 처리한 뒤 간단 요약만 모델에 전달하여 신속한 응답을 가능케 함
- 데모 비교: Claude 4.7(대형)에서는 응답까지 수 초가 걸리지만, Haiku 4.5(소형)는 개선된 구조와 함께 900ms 이내에 답변을 시작함
- 모든 "생각"과 추론, 교습 시나리오 설계 등 본질적인 로직은 코드에서 전담하여, 모델은 오직 "말하기"에 집중하도록 만듦
- 이 방식 덕분에 소형 모델도 실시간/고빈도/저비용 음성 어플리케이션에 매우 적합하며, 유사 과제에 동일한 접근을 권장함
- 단, 소형 모델은 “스캐폴딩(구조틀)” 역할을 하는 엄격한 규칙·코드가 필수이며, 이는 초기 구축 시 한 번만 투자하면 됨
- 결론적으로 “가장 빠른 모델을 선택하고, 나머지 복잡성은 코드 스캐폴딩”으로 해결하는 것이 음성 에이전트, 실시간 시스템 구축의 핵심임을 시연

## 세부 요약 - 주제별 정리

### 음성 에이전트에서 실시간성이 가장 중요한 성공 요소임을 강조함

- Ornella는 “음성 통화에서 1초의 침묵도 사용자에게 튜터가 아닌, 시스템이 고장난 것처럼 느껴진다”고 언급
- 실시간 대화에서 “즉각성”이 중요하기 때문에, AI 모델의 응답이 950ms 이내에 시작되어야 함
- 사용자가 1초만 기다려도 뇌는 에이전트가 죽었다고 인식할 만큼 빠른 응답이 필수임
- 음성 에이전트에서는 답변의 질보다 응답의 속도가 먼저 고려되어야 한다는 점을 명확히 함

### 대형 최첨단 모델의 reasoning(추론)은 실시간 인터페이스와 상극임을 사례로 설명함

- Anthropic의 Claude 4.7 등 최첨단(Frontier) 거대 언어모델은 뛰어난 reasoning(추론) 능력이 있음
- 사용자가 교습 관련 질문을 하면 Claude 4.7은 수초 동안 논리적으로 고민하고 답을 생성함
- 하지만 이 과정이 2~3초 이상 걸릴 수 있어, 실질적으로 실시간 인터페이스의 사용자 경험이 현저히 저하됨
- “모델의 뛰어난 사고 과정이 오히려 문제”라는 새로운 시각을 제시함

### 모든 추론, 시나리오 설계 등 “생각하는 부분”을 코드(상태 머신)로 분리하는 구조를 구축함

- Ace 시스템은 레슨의 모든 흐름, 타이밍, 다음 단계 결정, 정보 표시, 학생의 마스터리 추론 등 “생각”하는 모든 과정을 코드로 설계
- 각 Turn(발화)마다 시스템이 모델에게 상황 요약만 제공하고, 모델은 요약을 “말로 표현”하는 역할만 전담
- 이를 위해 상태 머신(state machine), 인텔리전트 레이어, 시나리오별 처리 구조가 개발됨
- 이 구조 덕분에 모델이 오직 텍스트 생성(말하기)에만 집중할 수 있음

### 데모 비교를 통해 소형 모델 구조의 빠른 응답성과 실용성을 실시간으로 입증함

- 첫 번째 데모: “기존 방식”으로 Claude 4.7에 단순 질문 입력 → 수 초 간 답변 기다림 → 느린 응답
- 두 번째 데모: 동일 질문을 개선 구조의 Haiku 4.5(소형 모델)에 입력 → 약 900ms 만에 바로 답변 시작
- 결과적으로 “모든 똑똑한 처리는 미리 코드로 하고, 모델은 말하기만 집중” 구조가 실제 체감 속도를 비약적으로 단축시킴

### 소형 모델 사용 시 발생하는 구조적 어려움과 그 극복 법칙을 솔직하게 공유함

- Joel은 소형 모델(예: Haiku 4.5)은 본질적으로 긴 대화나 논리 구조에서 “붕괴(drift)”될 위험이 높다고 언급
- 일관성을 지키려면 “스캐폴딩(Scaffolding)”이라는 엄격한 규칙 체계와 코드적 구조틀이 반드시 필요함
- 그러나 이 스캐폴딩 구축의 비용은 “한 번”만 투자하면 되고, 회차별 응답에는 부담이 적음
- 무분별하게 최첨단 대형 모델을 사용하는 것보다, 맞는 크기의 모델 선정(속도 기반)을 우선시해야 한다는 전략을 제시

### ‘모든 복잡한 추론·로직은 코드에서 처리, 모델은 말하기’라는 일반화된 전략을 제시함

- Joel은 “라티시(latenct) 예산”에 맞는 가장 빠른 모델을 쓰고, 나머지 인텔리전스는 프로그램적으로 구현하라고 조언
- 상태 머신, 각종 분기·시나리오 케이스, 사용자 반응 처리 로직 등은 모두 코드에서 전담
- 모델은 오직 텍스트 생성(말하기)에 초점을 맞추는 “역할 분리” 전략의 일반적 적용 가능성을 강조
- 이법칙은 음성 에이전트뿐 아니라, 실시간(저지연)·대량 트래픽 AI 어플리케이션 전체에 적용 가능함을 시사함

### 구조적 역할 분배가 소형 모델로도 고품질 실시간 서비스를 가능케 함을 실증 자료로 제시함

- 대형 모델에 전적으로 의존하는 구조는 소규모, 대량 트래픽 환경에서 비용, 지연 등 효율성 한계가 큼
- 본 프로젝트에서는 코드 + 소형 모델 조합만으로 사용자의 실시간 대화 요구, 맥락 유지, 적절한 교습 기능 모두 충족시킴
- 소형 모델 사용이 곧 품질 저하로 이어지는 것이 아니며, 구조화된 역할 분배가 훨씬 효율적임을 실제 예시로 보여줌

### 초기 개발 시 스캐폴딩 구축에 충분한 시간·노력을 투자해야 함을 강조함

- 효과적인 시스템 구현을 위해서는 어떤 상황이 발생할지, 다음엔 어떤 일이 일어나야 하는지, 학생의 반응에 따라 어떻게 로직이 바뀌어야 하는지 등 모든 케이스를 코드로 설계해야 함
- 이는 일회성 구축 부담이 크며, 교습 흐름·시나리오·에러 케이스 처리 등도 모두 미리 설계해야 함을 솔직하게 설명

### 발표를 마치며 Ace 프로젝트의 접근법과 일반적 원칙을 요약함

- Joel과 Ornella는 “모델은 시스템의 가장 작은 부분이 된다”고 표현하며, 전체 음성 에이전트 시스템에서 모델의 역할 최소화를 재차 강조
- 빠른 실행, 효율적인 비용, 높은 품질의 사용자 경험을 동시에 달성할 수 있는 스마트한 접근법임을 정리함
- 영상 마지막에 질문을 받을 수 있음을 고지하며 발표를 마침
