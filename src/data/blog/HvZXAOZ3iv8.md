## *API 이면에 숨겨진 것들 — 벤저민 코웬, 모달* 핵심 요약

- **API 기반 AI 개발이 빠른 성장과 유연성을 제공하지만, 커스터마이징 한계로 인해 점차 파인튜닝이 필수 요소로 부상함**
- **모달(Modal)은 범용 서버리스 컴퓨트 플랫폼으로, 다양한 AI 애플리케이션 구축에 최적화된 환경을 제공함**
- **AI 애플리케이션의 산업적 성장과 함께, 프롬프트 엔지니어링만으로는 한계에 부딪히는 사례가 늘고 있음**
- **대기업 고객의 경우, 지연 시간(latency)·처리량(throughput)·특화 비즈니스 로직 등 맞춤화 요구가 증가하고 있음**
- **파인튜닝 방식을 도입하면 퍼포먼스와 비용 효율성을 동시에 높일 수 있으며, 실제로 인터컴(Intercom)은 1/5의 비용으로 프론티어 API를 능가하는 성과를 냄**
- **최근에는 오픈소스 라이브러리와 서버리스 인프라 덕분에 '직접 모델을 훈련'하는 작업도 어렵지 않게 구현 가능해짐**
- **데이터 수집 및 평가 체계를 잘 갖춘 기업이라면, 비교적 수월하게 커스텀 모델 훈련을 시작할 수 있음**
- **서버리스 환경에서도 대규모 RL(강화학습) 및 하이퍼파라미터 튜닝을 병렬적, 탄력적으로 처리할 수 있음**
- **파인튜닝, RL, 추론 서빙까지 300줄 이내의 파이썬 코드로 빠르게 구축·확장 가능**
- **향후 '언제 내 제품이 도메인 특화로 진입해야 하는가'를 판단하고 준비하는 것이 중요한 전략 요소임**

## 세부 요약 - 주제별 정리

### API 기반 AI 개발은 빠른 확장이 가능하지만, 점차 맞춤화 한계에 부딪히고 있음

- 프론티어 API(frontier API)는 놀라운 속도로 AI 애플리케이션 개발과 혁신을 촉진함
- 개발자들은 누구나 매우 빠른 속도로 다양한 서비스를 구축 가능
- 그러나 API 제공자는 프롬프트 엔지니어링 외에는 커스터마이징에 제약이 큼
- 프롬프트 엔지니어링 예시: LLM에 “원시인처럼 말하라”고 하면 토큰 수를 대폭 줄일 수 있음
- 하지만 이 방식은 서비스가 100배, 1,000배 이상 확장될 때 한계가 명확함
- 대규모 고객을 위한 지연 시간·처리량·고유한 비즈니스 규칙 등 “특화된 수요”에는 대응이 어렵다 판단됨

### 대규모 AI 서비스의 요구는 파인튜닝 및 커스터마이징으로 집중됨

- 인터컴(Intercom), 데카곤(Decagon) 등은 자체 파인튜닝으로 비용과 성능을 크게 개선
- 인터컴: 프론티어 API 대비 1/5 수준의 비용으로 동급 이상의 모델 서비스 제공
- 파인튜닝은 주문형 비즈니스 로직, 특화된 KPI 등을 기존 API보다 더 잘 반영
- “프론티어 랩들은 무엇이든 다 잘하는 모델을 지향하지만, 기업은 비즈니스 특화 모델이 필요하다”는 인용 소개

### 파인튜닝 환경 구성의 복잡성은 신세대 클라우드·서버리스 플랫폼으로 간소화됨

- 전통적 파인튜닝: 대규모 클러스터, 인프라 분리, 전문가 투입이 요구되는 복잡한 과정
- 인프라 엔지니어, AI 엔지니어, 사이언티스트 모두가 인프라까지 신경써야 함
- 현재는 서버리스 컴퓨트, 오픈소스 툴킷의 발달로 세팅 부담이 획기적으로 줄음
- 모달(Modal), 기타 신생 클라우드 제공자들이 이 부분의 장애물을 해소함에 집중

### 오픈소스 라이브러리가 파인튜닝·RL 환경 구축을 혁신적으로 쉽게 만듦

- 최신 오픈소스 라이브러리로 알고리즘 레벨의 수정과 최적화 가능
- 클러스터 관리나 인프라 설계 없이도 커스텀 모델 훈련이 가능함
- 파인튜닝, 강화학습 모두 300줄 내외의 파이썬 코드로 구현 예시 존재
- 개발자는 빠른 반복(iteration) 주기를 그대로 유지하며 자신만의 모델 훈련에 몰두할 수 있음

### 서버리스 환경은 학습뿐만 아니라 대량 실험·튜닝에도 이상적임

- 서버리스를 주로 추론(inference)에만 쓴다고 여기는 오해가 많음
- 하이퍼파라미터 튜닝, RL 롤아웃 등 수만 개의 컨테이너를 병렬 분산 실행에 활용 가능
- 모달에서는 50,000~100,000개 샌드박스의 확장 실례가 최근 분기에 등장
- 비효율적이거나 불필요한 실험 컨테이너는 쉽게 종료할 수 있어 리소스 낭비 최소화

### 제품이 ‘도메인 특화(Custom Domain)’ 단계로 넘어갈 시점 판단법 제안

- 예: 프롬프트 엔지니어링(원시인 모드 등)에도 API 사용 비용이 고객 단가를 상회하면 커스텀 엔드포인트 전환 신호
- 서비스 평가(이벌, evals) 점수가 정체(plateau)되면 파인튜닝을 진지하게 고려할 시기
- 데이터가 부족하거나 평가지표가 미성숙하다면 아직 훈련 시기가 아님
- 훈련의 신호를 감지하고 사전에 데이터 수집·이벌 체계 개발 필요

### 데이터를 꾸준히 쌓았다면 파인튜닝 전환 준비는 이미 이루어진 셈임

- 제품·에이전트 테스트/배포를 하며 실제 훈련 데이터가 쌓임
- 평가 체계가 돌아가고 있다면, 훈련에 필요한 피드백 자료도 확보 가능
- "에이전트 하니스(agent harness)가 이미 구축됐다면 강화학습 훈련도 실현 가능"이라고 강조

### 모달을 비롯한 플랫폼에서는 쉽게 코드 한두 파일로 훈련·배포가 가능함

- 예시 코드: 300줄 내외의 파이썬으로 전형적 파인튜닝 파이프라인 구성
- Modal 계정 생성, 데이터 준비만 하면 즉시 실습 시작 가능
- 예제가 Modal 오픈소스 레포지토리에 공개됨

### 서버리스 환경에서 강화학습(RL)도 대규모 병렬처리로 실무 적용

- RL은 대량의 실험, ‘롤아웃’ 구조의 병렬 검증이 특징
- 모달에서는 GPU 클러스터, 샌드박스의 통합 API 제공으로 사용 편의 증대
- 최근 50,000~100,000 샌드박스 규모로 RL 실험을 실행한 사례 공유
- 도메인에 맞는 강화학습 모델도 누구나 직접 시도 가능함

### 파인튜닝 후에는 추론 서빙도 오픈소스와 서버리스로 손쉽게 가능

- vLLM, SG Lang, Triton Inference Server 등 다양한 오픈소스 추론 서버 활용 가능
- Modal 등 서버리스 환경에서 오토스케일 세팅으로 실시간 트래픽 처리 대응
- 복잡한 추론 워크플로우도 파이썬 파일 한두 개로 구현할 수 있음

### ‘언제 내 모델을 파인튜닝할 것인가’에 사전 대비가 핵심 전략임

- 당장 훈련을 강요하는 메시지가 아니라, “향후 1년~6개월 내 직접 훈련 시점을 고민하라”는 제안
- 본격적인 훈련을 위한 데이터와 이벌(Evaluation) 체계 준비가 중요
- 직접 경험을 통해 경쟁력 있는 특화 모델 확보가 충분히 가능한 시대임
