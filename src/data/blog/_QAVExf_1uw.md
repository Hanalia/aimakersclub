## *에이전트 실패를 활용한 일관된 지식베이스 구축: 수요 기반 컨텍스트 관리 방법론* 핵심 요약

- 동영상 제목은 "*에이전트 실패를 활용한 일관된 지식베이스 구축: 수요 기반 컨텍스트 관리 방법론*"
- 발표자 Raj(IKEA 소프트웨어 엔지니어)가 AI 에이전트의 진화와 컨텍스트 관리의 실제적인 문제를 경험 중심으로 설명
- 엔터프라이즈 환경의 실제 문제: AI가 코드 생성, 인시던트 관리 등 다양한 작업을 수행하지만 실질적인 업무(예: Jira 티켓)는 해결되지 않는 현상
- 맥킨지 2023년 데이터 인용: 88%의 기업이 AI를 도입했지만, 실제로 얻는 가치 창출은 6%에 불과
- 조직 지식은 약 40%가 구전(tribal knowledge)으로 존재하며, 20%는 오래되어 신뢰할 수 없고, 10%는 중복 기록됨
- 전통적인 MCP/RAG 기반의 데이터 검색 계층만으로는 기업 내 지식의 단절과 실무 전달력 한계를 해결하지 못함
- "수요 기반 컨텍스트(demand-driven context)" 방법론을 통해 문제 발생 → 에이전트 실패 → 정보 청구/업데이트 → 지식베이스 진화의 선순환 구조 제안
- 이 방식은 TDD와 유사하게 '의도적 실패'를 통해 조직 내 미기록 지식(미지의 지식)을 찾아내고, AI가 반복적으로 새로운 컨텍스트/지식 블록을 생성, 갱신할 수 있게 함
- 실제 구현 사례와 데모, 자동화 확장법, GitHub 활용 구조, 메타모델 개념 및 한계점 명확히 소개
- 워크숍을 통해 누구나 방법론을 테스트해볼 수 있도록 컨텍스트 갭 스캐너, 오픈 소스 레포 및 실습 프롬프트 제공

## 세부 요약 - 주제별 정리

### AI 에이전트는 작업 자동화에는 능하지만 기업 지식의 실무 전달력에는 한계가 있음

- 최근 AI의 발전으로 자동 코드 생성, 풀스택 앱 빌드, 리뷰, 인시던트 관리 등 기술적 작업은 훌륭하게 수행됨
- 하지만 프로젝트 관리 툴(Jira, Apex)에서 실제 티켓 처리가 진전되지 않는 현상이 빈번
- 이는 기업의 실질적 목표(ROI)와 맞물려 “AI가 많은 일을 하는데 왜 생산성이 대폭 향상되지 않을까?”라는 의문을 유발
- 실제 설문 데이터를 인용해 88%의 기업이 AI를 도입했지만, 가치 창출은 6%에 그친다는 맥킨지 자료 제시

### LLM/에이전트의 한계는 조직만의 고유 지식(기관적 지식) 전달에 있음

- LLM은 오픈/공용 데이터에는 뛰어나고, 가이드 지식(orange)도 비교적 잘 처리하지만, 특정 기업만의 컨텍스트(red)에선 실패함
- 기업 내 의사결정, 실제 업무절차 등은 문서로 남지 않고 다수가 머릿속(tribal knowledge)에만 저장되어 있다는 점을 지적
- 실제 지식 분포는: 20% 오래된 지식, 20% 신뢰 불가, 10% 중복, 40% 구전 지식
- MCP/RAG 서버를 아무리 많이 구축해도 이같은 불완전·폐쇄적·모놀리식(monolithic) 지식 구조가 바뀌지 않으면 실질적 자동화/자율화 구현이 불가

### 기존 "검색 기반" 계층만으로는 엔터프라이즈 지식 갱신/일관성 문제를 해결할 수 없음

- 데이터 검색 계층(RAG/Knowledge Graphs 등)을 활용하면 최대 40% 정확도 향상 가능
- 그러나 실무에서 MCP/RAG 결과는 불확실, 신뢰 불가, 실제 문제 해결에는 한계 발생
- 엔지니어들이 실제로 테스트(evaluation)를 수행하지 않는 문제 및 직접 개입 없이 기대한 완전한 자동화는 실현되지 않음
- 오히려 MCP 서버가 많을수록 반복적으로 질문을 하거나 사람이 직접 정보를 입력(결국 노동 강도 증가)하는 일이 발생

### 조직의 모놀리식(Monolithic) 지식베이스를 "컨텍스트 블록"으로 분해해야 효과적 자동화가 가능함

- 과거 시스템 구조를 모놀리식에서 마이크로서비스로 분리하는 것처럼, 지식베이스도 쪼개야 함
- 문서, 대화, GitHub 등 모든 지식을 작은 단위의 의미 있는 컨텍스트 블록으로 변환하는 작업 필요
- 이는 LLM 업체나 외부에서 대신 해줄 수 없으며, 조직 내에서 자발적으로 직접 수행해야 함
- "수요 기반 컨텍스트"가 이 문제를 해결할 수 있는 프레임워크임을 설득

### 수요 기반 컨텍스트(Demand-Driven Context)는 에이전트 실패를 반복하며 지식 갭을 채우는 방법론임

- 기존 'Push(지식 일괄 주입)' 방식이 아닌 'Pull(문제 중심, 필요시 정보 요청)' 방식을 채택함
- 신입 사원 온보딩 사례와 유사: 업무 지시 후 사원이 질문/필요 데이터 요청 → 점진적 학습
- 에이전트에게 실질적 업무(work item)를 부여, 실패 시 미기록 정보 목록(체크리스트) 도출, 인간이 보완 제공, 에이전트가 지식베이스에 반영
- 이 사이클을 여러 번 반복하면 엔터프라이즈의 미지의/미기록된 지식이 점차 표면화되고 조직 KB가 진화
- 이 과정은 채팅 기록, 기계적 검색 결과에만 의존하지 않고 실제 도메인 문서의 확장을 중시함

### TDD(테스트 주도 개발)와 유사하게 "실패를 전제로 컨텍스트를 구축"하는 점이 핵심임

- TDD는 실패하는 테스트를 먼저 작성하고, 필요한 코드를 점진적으로 채워넣음
- 본 방법론도 마찬가지로 에이전트에게 일부러 실패하는 문제를 주고 ‘무엇이 부족한가’를 명확히 드러냄
- 점진적으로 조직 내 중요한 지식이 채워지고, 많은 사이클 후엔 에이전트의 자율/준자율 작업이 가능해짐

### 구체적인 실사용·자동화 사례를 통해 접근법의 확장성과 한계 모두 공유함

- 터미널 데모: 실제 실패-갭 확인-지식 제공-지식베이스 업데이트 예시 시연
  - 초기에 56개 엔티티 → 문제 1회로 6개 미기록 엔티티 발견, 14회 시도로 1.5→4.4 수준으로 신뢰도/근거 데이터 증가 확인
- 수작업으로는 번거롭지만, 자동화 시(예: 컨텍스트 갭 스캐너) 이미 보유한 Jira/인시던트 로그와 기존 문서 전수로 대규모 갭 탐지·평가 가능
- 기존 각 티켓, 인시던트, 고객 지원 이슈 등 아카이브 데이터를 자동/대량 분석하여 기관 지식의 품질, 누락, 중복, 업데이트 내역을 지도화함
- 주요 결과는 "어떤 시스템/프로세스/구성요소가 반복적으로 미기록/오류/중복되는가"를 도출, 우선순위 정렬(critical, high, medium 등)로 보여줌

### 지식베이스 관리 방식/구조(특히 GitHub 기반 관리)와 메타모델 설계가 체계적 관리에 도움을 줌

- 여러 팀/에이전트가 동시에 파일/지식베이스에 기여할 경우, GitHub의 PR, 리뷰, 머지 등 협업 기능 활용 추천
- 데이터를 GitHub에 저장함으로써 다양한 외부 시스템 슬랙·컨플루언스와 연동/배포도 용이함
- 도메인, 시스템, API, 비즈니스 용어/프로세스 등 구성원 간 관계(메타모델)를 명시적 구조로 설계하면, 에이전트가 지식베이스를 탐색·활용·확장하기 쉬움

### 도입 효과 및 가치: 미기록·미지의 조직 지식 발견, 에이전트의 지식 관리 역할 강화

- 가장 큰 효과로 '알려지지 않은(never documented)' 조직 지식이 표면화되어, 조직의 실질적 지식 갭을 빠르고 정확하게 발견 가능
- 지식 갭이 명확해질수록 에이전트가 단순 정보 소모자에서 적극적 지식 관리자/갭 충족자로 진화
- 기존처럼 사람이 모든 정보를 정제해 주는 상태를 벗어나, AI와 사람이 함께 지식 생태계를 구축·유지하는 체계로 전환

### 수작업 반복은 비효율적이지만 자동화 시 진전 가능하며, 소규모 팀에게 특히 효과적임

- 단, 수많은 사이클을 사람-에이전트 간 반복하는 것은 현실적으로 피로함, 자동화 도구나 프레임워크(예: 컨텍스트 갭 스캐너) 활용이 필수
- 문서/지식 품질이 이미 높은 소규모 조직이나 단일 도메인에는 비용·도입효과 상 제한적일 수 있음
- 대규모 엔터프라이즈보다는 주로 소규모 팀 단위, 범위 축소에 효과적임

### 조직의 실무 맥락 및 기술체계마다 확장 가능하고, 작업 이전에 지식 품질 보완을 선제적으로 수행할 필요가 있음

- 사내 CLI, 플랫폼 도구, 비즈니스 프로세스 등 다양한 작업 툴에도 갭 탐지 및 개선 도입 가능
- 운영 중 실시간 컨텍스트 관리보단, 사전에 최소 품질을 확보하고 난 뒤 운영에 적용하는 것이 효율적임
- 코드베이스와 텍스트가 섞이면 충돌 발생하므로, 소스 우선순위 지정 등 추가 룰/기술적 조치 필요

### 토큰 한계, 업데이트/중복/오류 관리 등 운영 현실 문제도 실제 경험을 바탕으로 논의함

- GPT context window가 확대(1M 토큰)됨에 따라 상당수 도메인은 통합으로도 충분히 입력 가능(평균 96K 토큰 수준)
- 문서의 중복/버전/오류 관리, 최신화 여부는 생성일/업데이트일 기반 필터링 및 경고 구조로 운영
- 비용 측면: 일간 자동 실행해도 토큰 비용이 크지 않음(월단위 산정 시 1$ 이하 소요 예측)
- 오픈 소스 레포와 가이드, 프롬프트 샘플 등 참여·테스트 자원 모두 제공
