## *블라인드 스팟에서 PR 머지까지: 에이전트 기반 지속적 성능 최적화* 핵심 요약

- 이 영상은 Thundra의 공동 창업자이자 CTO인 Mike가 실서비스(프로덕션) 환경의 성능을 에이전트 기반 자동화(AI coding agent)로 지속적으로 최적화하는 워크플로우 구현 사례를 소개함
- 기존 개발팀에서는 성능 저하 이슈가 발생해도 최적화 조사(리서치)에 소요되는 시간과 노력을 예측하기 어려워 주기적이고 선제적인 개선을 거의 수행하지 않음
- Mike의 팀은 GitHub Actions, Cloud Code, 자체 계산 계층(MCP), Slack 연동 등 다양한 도구를 조합해, 반복적이고 자동화된 성능 이슈 탐지 및 개선 시스템을 구축함
- 이 시스템은 실시간 트레이스·쿼리·지연 시간 등 ‘프로덕션 컨텍스트’로부터 주간 리포트를 자동 산출해, ROI가 높은(쉽게 해결 가능하며 효과가 큰) 최적화 후보를 자동 선별함
- 에이전트가 개선점을 코드 수준에서 제안, 자동 수정 및 테스트, 영향 검증, 마지막으로 사람 검토 단계를 거쳐 신뢰도와 실제 가치가 높은 개선만을 추천함
- 도입 초기에는 ‘그럴듯하지만 검증되지 않은’ 제안, 복잡한 쿼리 처리, 피상적 예외처리 등 시행착오를 겪었고, 에이전트의 실질 업무 영향력 강화를 위해 컨텍스트 조직, 질 좋은 데이터, 스킬셋 추가, 반복적 자동화 설계가 중요하다고 설명함
- 각 개선항목은 빈도, 비즈니스 영향도, 위험도 등 다면적으로 평가해, 실제로 엔지니어가 처리할 가치가 있는 소수의 ‘핵심 이슈’만을 표 형태로 제공
- 인간과 에이전트가 하이브리드로 ‘루프’를 완성하도록 하여 신뢰성·효율성 모두를 확보 핵심으로 보며, 정기 자동화의 가치, 팀 업무문화의 변화까지 강조함

## 세부 요약 - 주제별 정리

### 반복되는 성능 이슈 대응의 한계와 자동화 필요성 인식

- 과거와 현재를 막론하고, 제품 담당자/엔지니어가 “어느 페이지가 느립니다, 최적화할 수 있을까요?” 같은 대화를 반복적으로 나눔
- 보통 문제 조사에는 불확실성이 크고(조사만 1시간~1주 소요), 사전 진단 없이 대기하다 심각해지면 급하게 처리하는 “leaky bucket(구멍난 양동이)” 패턴 반복
- 이렇게 성능 이슈의 선제적 개선이 어려운 근본 원인은 조사 단계가 ‘블랙박스’이고, 투자 대비 정확히 무엇을 얻을지 불분명하기 때문임
- “자동화로, 실서비스 맥락에서 정기적으로 데이터를 분석하고 고ROI 기회 자동 선별 → 엔지니어 개입 최소화”라는 솔루션의 필요성이 제기됨

### Thundra가 구축한 에이전트 기반 성능 최적화 워크플로우의 전체 구조

- Thundra는 실서비스 런타임 인텔리전스 계층을 탑재, 코드에이전트가 제대로 된 맥락·포렌식 정보를 바탕으로 성능/안정성 문제를 자동 해결하도록 설계
- GitHub Actions를 활용하여 주간단위로 분석 실행, Cloud Code와 자체 MCP 레이어로 다양한 엔드포인트/함수의 런타임 정보 수집
- 검출된 이슈 리포트는 Slack 등 실제 엔지니어들이 실시간으로 활동하는 채널로 자동 전달되도록 세팅
- 데이터 소스, 에이전트, 실행 환경을 특정 벤더/모델에 종속하지 않도록 추상화 및 유연성 확보
- 인증·권한·툴 콜의 보안, 스케쥴링 또는 웹훅 트리거 등 확장성/유지보수의 용이성 등 실무 요구 반영

### 프로덕션 컨텍스트와 코드 컨텍스트 결합을 통한 실제 문제 타겟팅의 어려움과 해결

- 엔지니어는 보통 CPU, 메모리, SLO 등 시스템 측 매트릭 외에도, 코드 함수 레벨의 맥락이 반드시 필요하나, 이 둘은 흔히 단절되어 있음
- 프로덕션 이벤트(엔드포인트, 크론잡, 외부호출 등)와 함수 레벨의 연결이 약하여, 데이터는 많지만 실제 문제의 원인/위치 추적이 어려웠음
- Thundra는 ‘prod-to-code’ 구조 도입: 실서비스 이벤트(예: “이 엔드포인트가 7초 걸림”)가 함수 수준 컨텍스트와 직접 연결됨
- 요청이 P99 임계값 등 특정 조건(지연 등)에 도달할 때만 법의학 수준의 심층 데이터샘플을 수집, 분석의 효율화 및 실질 사례 기반 해결 가능

### ClickHouse 등 실무 데이터 플랫폼과 고유 쿼리 패턴의 적용

- Thundra는 ClickHouse 기반 컬럼 지향 DB를 활용해, 함수–엔드포인트–포렌식 구조상 조회를 최적화
- HUD Query Language 등 전용 DSL(도메인 특화 언어)으로 복잡한 쿼리를 반복 추출하고, 쿼리 결과의 다양성·일관성을 위해 ‘스킬(skill)’ 모듈을 추가
- 예시: HTTP 500 에러의 발생 원인 추적, 메모리 스파이크 시 어떤 코드/함수가 돌았는지 비교해서 차이분석, dead code 식별 등 구체 업무에 AI 활용
- 정형화된 쿼리/스킬이 자동화 계층(예: dead code 자동 제거, 성능 향상 요소 실시간 탐지 지시 등)의 기반이 됨

### 자동화된 성능 개선 탐지와 올바른 이슈 선별 기준 수립의 중요성

- 자동 오탐의 문제: 에이전트가 ‘그럴듯하지만 실서비스와 무관’한 해결책(예: 피상적 예외처리 등)을 제시하는 경우 빈번했음
- 초기에 지엽적/불필요한 PR 다발(‘80개의 PR 폭탄’)이 엔지니어에게 과부하로 다가오는 경험 발생
- 이를 극복하기 위해, 각 이슈는 빈도(핫패스 경로인지), 비즈니스 민감도(예: 결제 등), 변경 위험도 등 다면적으로 ROI 스코어링
- “이 엔드포인트는 통상 200ms, 특정 상황에서만 45초 이상 지연. Mongo 인덱스 미사용 때문, 간단한 쿼리 변경 제안”과 같이 컨텍스트와 영향, 근거가 명확한 소수의 핵심 리포트만 전달

### 인간+에이전트 하이브리드 루프의 완성 및 최종 신뢰 확보 단계 설계

- 자동화 제안사항은 에이전트가 코드를 자동 수정·테스트·성과 측정까지 모두 실시, 사람이 마지막 리뷰로 역할을 함
- 개선안이 실제로 효과 있었는지, 왜 이 개선이 필요한지 모두 증거 기반으로 제시되어 PR 설득력 및 업무 효율성 증대
- 사람의 개입은 ‘신뢰성 인증’ 및 실제 비즈니스 맥락 최종 판단자로 한정, 잦은 자동 PR로 인한 피로도 최소화

### 성능 개선 자동화 과정에서의 시행착오와 주요 학습 점

- 컨텍스트 과잉/결핍 문제: 데이터가 너무 많아 신호대잡비율이 떨어지거나, 핵심 로그/메트릭이 누락된 경우 모두 대처 필요
- 에이전트가 코드를 다루기엔 함수/파일 레벨로 맥락이 응집되는 구조가 적합하며, 생산성과 효과성 모두 이 구조 전환으로 크게 개선
- 인간/비즈니스 도메인 지식의 개입이 아직 필요하며, 자동화 신뢰성(특히 엔지니어 시간 투자 가치)이 확보되어야 실무 적용이 가능

### 자동화 통해 엔지니어링 문화 및 조직 내 업무방식의 변화

- 에이전트가 “기존에 하지 않던 일(정기적인 저비용·고효율 이슈 탐색)” 자체를 자동화, 단순 반복 대신 실제 성과 창출 구조 확립
- 루틴하게 수행되지 못하던 성능 개선 ‘습관’을 팀 레벨에서 배양하고, 업무 부담 없이 긍정 피드백 루프 유도

### 정기 자동화에 최적화된 유지 보수성, 신뢰성, 확장성 요건 정립

- 에이전트 워크플로우 초기에는 Out-of-the-box가 유용해도, 조직/요구가 진화함에 따라 유지보수성, 유연성, 피드백 루프가 필수
- 스케쥴·트리거, 워크플로우 논리, 에이전트 스킬 집합, 데이터 소스 유연 변경 등 ‘갱신성’에 초점

### 최종적으로 신뢰 기반 에이전트 자동화와 직접 인간 코딩의 차별점 정리

- IDE상 개인용 AI 코딩(Point AI Assist)과, 자율실행되는 에이전트 자동화간에 ‘요구 신뢰수준’의 차이가 명확함(80% 만족으로는 불가)
- 운영 자동화는 80~90% 이상의 신뢰성과 실증이 갖춰져야 실제 조직 내 정착이 가능, 이 점이 가장 높은 진입장벽임

### 자동화 도입을 희망하는 조직을 위한 4개 실질 조언

- 무엇이 중요한지(impact 기준) 명확히 정의 및 스코어링하고, Guardrails 설계로 불필요한 이슈 남발 예방
- 단순 속도 향상이 아닌, ‘아예 실행되지 않던 임무’(예: 정기 low-hanging fruit 탐지)의 자동화가 혁신 포인트임
- 스마트함(모델/알고리즘)이 아니라, 콘텍스트와 전문성 있는 쿼리/스킬/데이터가 실제 실무 효용의 핵심임
- 인간 판별 및 룰 기반 선정, 검증까지 반드시 함께 설계해 ‘엔지니어의 실질 시간/리스크 가치’에 걸맞는 자동화 이어야 함
