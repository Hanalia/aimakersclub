## *컴포넌트를 넘어서: MCP 앱을 위한 생성형 UI 설계* 핵심 요약

- 영상은 2022년 ChatGPT 초기 실험부터 2025년 주요 LLM 모델(예: GPT 5.2, Opus 4.5) 출시까지의 급속한 UI 생성기술 발전사를 서술함
- 최신 LLM 모델들은 단순 코드 생성 수준을 넘어, 고품질·접근성 우수·상상력이 반영된 UI 전체를 즉시 생성할 수 있음을 실례와 함께 설명
- 현재의 생성형 UI 주류 방안은 △정적 컴포넌트(Static Components) △선언적 UI(Declarative UI) △완전 생성 컴포넌트(Generative Components)로 분류됨
- AGUI 프로토콜, Goose(Goose Auto Visualizer), Vercel의 JSON Render 등 주요 구현 예시 및 이들의 기술 구조와 한계점을 구체적으로 소개
- LLM이 완전히 새로운 UI 코드(HTML/CSS/JS)를 실시간 생성하는 실험 사례(포스트맨의 Weather Agent 프로젝트)와 문제점(신뢰성, 보안 등)을 다룸
- 현재 UI 진화의 핵심 과제로 ‘신기술에 맞는 GUI 설계’, ‘MCP 앱 활용’, ‘모델-배포 체계의 조화 찾기’ 등이 제시됨
- MCP 앱은 높은 샌드박스·권한 관리·도구 호출·메시지 전달 등의 탁월한 특징으로 생성형 UI 딜리버리의 표준으로 부상 중임을 역설
- Excalidraw MCP 앱의 인간-에이전트 협업 캔버스 등, 미래 UI는 ‘개인화와 협업 중심 인터랙션’으로 진화할 것이라 전망함
- TV 초창기 비유(라디오쇼의 단순 영상화)처럼, 아직 상상력의 한계에 갇혀 있음도 진솔하게 언급
- 궁극적 답은 아직 없으나, 인간과 AI 에이전트의 적극적 상호작용이 생성형 UI 미래의 핵심 동력이 될 것임을 시사

## 세부 요약 - 주제별 정리

### 2022년 이후 생성형 UI는 LLM 발전에 맞춰 상상 이상의 속도로 진화함

- 2022년 11월 ChatGPT 등장 후, 초창기에는 코드 생성(prompt → 코드 → 붙여넣기 → 수정) 방식이 일반적이었음
- 당시엔 “모델이 개발자보다 코드를 잘 못 쓴다”는 인식이 강했으나, 2024~2025년 사이 GPT 5.2, Opus 4.5 등 고성능 모델 등장으로 큰 전환점을 맞음
- 최신 LLM들은 긴 맥락 이해, 고품질 UI 생성, 빠른 코드 작성 등에서 인간 능력을 뛰어넘기 시작함
- 예시: 블로그 리라이트를 요청했더니 모델이 자동으로 검색박스, 블러 애니메이션, 접근성까지 갖춘 UI를 구현한 실제 경험을 소개
- 3년 만에 모델이 개발자 본인보다 더 나은 프론트엔드 코드를 작성하는 시점에 도달했음을 강조
- 이로 인해, 전통적 정적 UI의 한계를 돌파할 필요성 제기

### 모델이 뛰어난 UI 생성을 제공함에도 불구하고 아직 '혁신적 인터페이스'에 도달하지 못함

- “왜 우리는 아직 부유하는 UI(일명 Jarvis)까지 가지 못했나?”라는 문제 제기
- 인터페이스 언어가 미성숙한 상태로, 아직은 1970년대 터미널과 유사한 상태임을 강조(Andrej Karpathy의 ‘새로운 컴퓨터’ 비유 인용)
- 대부분 SaaS가 단순 챗봇(채팅방)만을 UI로 제공하지만 이것이 최종형태는 아니라는 시각 제시
- 또 다른 극단은 MCP 기반 슈퍼앱(예: ChatGPT, Claude, Gemini) 하나에서 모든 써드파티 UI를 통합하는 방식임
- 이 두 옵션(챗 UI 확산 vs 슈퍼앱) 모두 일시적 진화의 일부이며, 최종 해답은 사용자 선택과 기술 발전에 달렸다고 진단

### 현재 에이전트(Agent) 기반 앱의 대다수는 정적 컴포넌트를 통한 UI 구성을 고수함

- 에이전트는 툴콜(tool call)을 통해 정해진 데이터와 파라미터를 전달, 미리 제작된 컴포넌트를 렌더함
- 구조적으로는 20년 전 전통적 UI와 유사(서버의 데이터→UI 렌더링)
- MCP 앱에서 대표 예시는 AGUI 프로토콜: 클라이언트 툴(React 컴포넌트 매핑) 등록 → 툴콜 시 프롭스/데이터 전달 및 컴포넌트 렌더
- 또 다른 예시, Goose Auto Visualizer: 임의 데이터 입력 → 미리 정의된 여러 시각화 컴포넌트 중 적절한 조합으로 출력
- 이 방식의 한계는 ‘모델의 창의성/유연성 부족’과 ‘디자인 시스템 안으로의 제약’임

### 선언형(declarative) UI는 기존 방식 대비 개인화·동적·유연성을 높이는 중간 단계임

- 선언형 UI는 여전히 정적 컴포넌트와 디자인 시스템에 기반
- 차이점: “모델이 JSON/YAML/Python 등으로 UI 구조·내용을 서술” → 렌더링 엔진이 이를 파싱하여 마치 레고처럼 최종 UI로 조립
- Netflix의 퍼스널라이즈드 UI, Vercel의 JSON Render 같은 실제 활용 예시 언급
- Netflix: 홈 화면 UI가 사용자별로 달라지나, 결국 네트플릭스 고유 컴포넌트 안에서만 조합됨
- JSON Render: JSON과 YAML 방식 모두 지원, 사용자의 데이터/프롭스에 맞춰 역동적인 UI 생성
- 선언형 UI의 장점: 예측가능성, 일관된 디자인 시스템 유지, 빠르고 비용 효율적 구성(LLM 토큰 낭비 최소화)
- 그러나 궁극적으로는 ‘프로그래머가 만든 상자 밖’을 결정적으로 벗어나진 못함

### 완전 생성형(Generative Components) UI는 LLM의 런타임 실시간 코드 생성 역량을 최대한 활용함

- 핵심: LLM에게 HTML, CSS, JS 등 UI 전체 코드를 직접 즉시 생성하도록 위임
- 에이전트가 툴콜(tool call) 또는 역방향 샘플링(reverse sampling)을 통해 실시간 오리지널 UI 생성
- 실험 예시: 포스트맨 Weather Agent
    - 날씨 API 호출 결과, 농담 생성, HTML/CSS/JS 등 프론트엔드 소스코드를 한 번에 생성·렌더링
    - 매번 상상력 넘치고 기발한 UI가 출현(예시: 전혀 없는 컴포넌트, 동적 애니메이션 등)
- 장점: 극단적 유연성/개인화, 개발자 사전설계 없이 무제한 조합 가능
- 크리티컬 문제점: LLM이 생성한 외부 코드는 신뢰·보안상 위험(악성코드, 데이터 유출 등)
- 고로 이런 ‘풀 생성형 UI’는 분배방식(boudary, containment, sandbox 등) 확립이 필수

### MCP 앱은 생성형 UI 딜리버리의 보안성과 확장성을 모두 갖춘 최고의 배포 모델로 평가받음

- MCP 앱은 써드파티 UI를 샌드박스(보안 iFrame), 인증, 도구 호출, UI-에이전트 간 메시지 전달 등 기능을 기본 제공
- iFrame 이중화, 권한 관리 역시 우수. 보안/격리/이벤트 전파 등에 강점
- 앤트로픽(Anthropic)은 자체 UI 렌더링 아키텍처 대신 MCP 앱을 선택(Visualizer), 퍼스트파티 UI마저 MCP로 제공
- 왜냐하면 MCP 앱 프로토콜이 에이전트가 동적으로 생성하는 모든 UI 딜리버리에 가장 최적화되어 있기 때문
- MCP 표준이 써드파티를 넘어 퍼스트파티 UI까지 아우르는 시장 표준으로 자리잡을 가능성 언급

### 지금의 UI 패러다임은 초기 TV 시대와 유사하게 '상상력 부족'의 한계에 직면해 있음

- TV가 처음 등장했을 때, 그저 라디오쇼를 단순 촬영하는 수준에 머물렀다는 역사적 예시 인용
- 마찬가지로 현재 생성형 UI도 '챗봇', '단순 스크립트화' 등, 기술의 잠재력 전체를 활용하지 못하고 있음
- ‘새로운 컴퓨터’를 위한 진정한 인터페이스 혁신은 아직 완성되지 않았고, 향후 발전 가능성이 열려 있다고 강조

### 미래의 생성형 UI는 인간과 에이전트가 상호작용·협업하는 '공동 캔버스'로 진화할 것으로 제시됨

- Excalidraw MCP 앱은 단순 그림 그리기/다이어그램 시각화 도구를 넘어, 에이전트와 사용자가 직접 실시간으로 캔버스에서 공동작업 가능함
- 사용자는 드래그·클릭으로 인터랙션하거나, 에이전트와 대화식으로 변경 요청 가능 → 인간-에이전트 협력의 상징적 모델 제시
- 앞으로의 생성형 UI는 일방향적 자동 생성이 아니라, 고도로 개인화되고 인간-에이전트가 협업하는 동적 경험이 핵심이 될 것
- 사용자의 상상력과 실험의식이 산업혁신 방향을 앞서 규정하게 될 것임을 시사

### 최종적으로 UI와 인간의 상호작용 미래는 아직 완전히 정해지지 않았으며, 함께 그려나가야 할 과제로 남아있음

- 챗 UI, MCP 앱 등 현재의 여러 대안이 ‘최종형’인지 확실치 않다고 솔직히 밝힘
- ‘정답 없음’의 상황에서 개발자, 사용자 모두가 직접 실험하며 새로운 표준과 인터랙션을 만들어가야 한다는 점을 강조
- 미래의 ‘새 컴퓨터’에서 UI는 개인화·협업·유연성·보안성을 모두 갖춘 방식으로 진화할 것임이 전망됨
