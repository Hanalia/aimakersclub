## *Archon의 향후 계획: 라이브 로드맵 세션* 핵심 요약

- 본 영상은 Archon의 로드맵을 직접 라이브로 구성하며, 앞으로의 기능 구현 계획과 핵심 우선순위, 그리고 커뮤니티 참여방식에 대해 논의함
- Archon은 오픈소스 AI 코딩 하니스(harness) 빌더로, 다양한 AI 코딩 프로세스를 반복 가능하고 결정적으로 워크플로우화하여 여러 코드베이스에 병렬 적용할 수 있도록 돕는 도구임
- 사용자는 YAML 포맷으로 워크플로우를 정의하며, 에이전트 기반(비결정적) 노드와 스크립트 기반(결정적) 노드를 혼합 구성이 가능함—코딩 에이전트만으로 구현하기 어려웠던 엄격한 제어가 강점
- 향후 Archon 워크플로우 마켓플레이스 구축이 최우선 과제 중 하나로, 사용자가 직접 만든 워크플로우를 공유/설치할 수 있는 인프라와 절차 설계가 논의됨
- 워크플로우 제출 및 배포 방식(커뮤니티 폴더 PR 방식 vs. 외부 저장소 참조), 보안 및 버전 관리 문제(공급망 공격 방지, 리뷰 부담 등)가 심도 있게 토론
- Archon은 Pi, 코덱스, Claude 등 다양한 코딩 에이전트와의 연동이 가능하며, 각 노드별로 에이전트와 모델을 개별 지정할 수 있음
- 오픈 이슈 조사 결과, 전체 이슈 중 약 40%가 워크플로우 영역에 집중되어 있음—주요 우선순위 개선 사항은 워크플로우 신뢰성, 제어 흐름 강화, 프로젝트 단위 오케스트레이터, 로컬 LLM 지원 등임
- 실시간 실습을 통해 로드맵 페이지와 워크플로우 마켓플레이스 프로토타입을 개발·테스트함—워크플로우 메타데이터 관리 및 featured 워크플로우 지정, CLI 설치 명령 등 세부 설계 논의
- 마무리로 “기능 및 보안 검증된 워크플로우”와 “커뮤니티 실험적 워크플로우”의 구분 필요성과, 미래 확장(로그인, 별점, 자동 보안 스캐나 등)에 대한 장기적 고민 공유

## 세부 요약 - 주제별 정리

### Archon은 에이전트+스크립트 혼합 워크플로우 자동화로 AI 코딩 프로세스의 결정성과 재현성을 극대화함

- Archon은 오픈소스 기반의 AI 코딩 하니스 빌더로 설계됨
- 기존 에이전트(예: anti-gravity 등) 중심의 프로세스를 상위에서 강제/통제할 수 있는 결정적 워크플로우 구현이 가능
- YAML로 개인 혹은 팀의 특정 AI 코딩 프로세스를 도식화하여 다양한 코드베이스에 병렬 적용
- 워크플로우 내 각각의 노드는 에이전트(비결정적, reasoning 담당), 스크립트(결정적, 반복/검증 담당)를 혼합해 설계 가능
- Stripe Minions과 유사하나, 사용자 맞춤 하니스를 자유롭게 정의해 병렬 운영 가능—코드베이스 별 Enforce-ability 강조
- “시스템을 에이전트에 짜넣는 것”이 아니라 “에이전트 노드를 시스템/워크플로우 안에 배치”하는 방식

### Archon 워크플로우 마켓플레이스는 커뮤니티 중심 공유와 간편한 설치를 목표로 설계됨

- Archon 이용자가 직접 정의한 워크플로우를 커뮤니티와 공유, 다른 이용자가 명령 한 줄로 설치 가능하게 하는 것이 목표
- 기존 기본 워크플로우(YAML)가 툴 내에 함께 제공되어 입문/커스터마이즈에 참고 가능
- npm이나 cloud code 플러그인처럼 “명령으로 워크플로우 즉시 설치” 방식 구상
- 워크플로우 등록 초안 프로세스: workflow.md(메타데이터) + YAML, PR로 커뮤니티 폴더에 제출, 마켓플레이스에 등재
- 워크플로우 메타데이터 관리(예: featured 여부, 별점, 저자, 사용 예 등) 설계 논의
- CLI 명령어 archon workflow install {slug} 를 통한 간편 설치 기능 구현(테스트 및 placeholder 단계)

### 워크플로우 제출·배포 프로세스에서 보안, 버전 관리, 운영 효율성을 둘러싼 심층적 공방이 진행됨

- 워크플로우 제출 후 변경(버전업) 절차: 매번 PR(리뷰 통한 검증)/자기 저장소 외부 참조? 등 반복 논의
- 풀리퀘(PR) 방식—보안 검증(공급망 공격 예방, 신뢰성 확인), dead link 방지, 심플한 유지 관리
- 외부 리포지토리 참조—저자가 자유롭게 버전 관리 가능(신속한 업데이트), 워크플로우가 YAML 이상 구성일 때 유리
- “커밋 해시 핀ning” 등 중간적 세부 옵션 제안: PR 승인된 커밋까지만 공식 마켓플레이스에서 설치 가능
- 별점(stars) 기능, 실시간/로그인 인프라 없는 스태틱 웹사이트에서의 데이터 관리 한계 인정
- 장기적으로는 별점·버전 관리 지원을 위해 외부 DB(Supabase, Neon 등) 활용 가능성 검토
- “검증된 공식 워크플로우”와 “실험적 커뮤니티 워크플로우”의 구분 및 위험성 워닝 표기 등 다층적 도입 필요성 제기

### Archon의 우선순위 로드맵과 핵심 개발 과제가 직접 실습 기반으로 상세히 정의됨

- 로드맵 페이지엔 다음 주요 항목 명시:
    - 5분 내 빠른 시작/설치(스타트업 가능)
    - 오픈소스 워크플로우 마켓플레이스
    - 워크플로우 평가·관찰 시스템(Eval)
    - 고도화된 워크플로우 제어흐름(루프, 분기 지원 강화)
    - Persistent Project Orchestrator: 각 코드베이스 별 상태 보존/에이전트 기억
    - 로컬 LLM 지원(OpenAI 호환 Base URL 설정)
    - 워크플로우 실행 신뢰성 보강
    - 미래 과업: 멀티 모델·멀티 저장소/엔터프라이즈 깃헙·프로덕션 배포 등
- 각 항목별로 상세 설명, 관련 이슈·기능 및 예상 시나리오를 링크로 제공

### 오픈 이슈 분석을 통해 워크플로우 개선이 최우선이며, 구체적 요구사항을 세분화하여 로드맵에 반영함

- 현재 Archon 저장소에는 149개 오픈 이슈(버그·제안 등) 존재—커뮤니티 활동 활발, 정량적으로도 증가세
- 하위 에이전트를 사용하여 이슈 자동 분류·패턴 분석 및 유사/중복 이슈 식별을 시도
- 전체 이슈의 40%가 워크플로우 영역 관련(버그+기능 요구)
- 집단화된 우선순위: 워크플로우 루프·제어흐름 신뢰성 문제, 프로젝트 오케스트레이터, 프로바이더 추상화, 멀티 레포·배포·엔터프라이즈 지원 등
- 로드맵에 직접 반영할 신규 항목 제안, 워크플로우별 구체적 태그·이슈 링크 연결 방식 검토

### 워크플로우 메타데이터 및 마켓플레이스 UI·UX 전략이 논의되고 실습을 통해 구현됨

- workflows.json 등 별도 메타데이터 파일 관리 vs. workflow.md 내 front-matter 사용 자동화 검토 및 결정
- featured 워크플로우 노출 기준, 워크플로우별 슬러그·설명·저자 등 업데이트
- 별점(stars)은 별도 관리 필요, 단 로그인 미지원/실시간 업데이트 한계로 현재는 보류
- 마켓플레이스 페이지 내 featured·browse·copy·install 등 섹션 구상 및 코드 구현 시연

### 보안(공급망 공격 등) 우려와 자동 검증 시스템 도입 필요성에 대한 토론이 이어짐

- 레포지토리 기반 관리로만은 워크플로우 수정·버전업 시 무단 악성 변경 가능성 상존
- 기존 Linux/Arch Linux 사례 공유—신뢰 저자가 장기적으로 악성 코드로 업데이트해도 위험
- 커뮤니티 워크플로우 등록·PR/리뷰 및 추후 자동화된 보안 스케너(에이전트 기반 진단) 도입 예정
- “실험적 워크플로우 경고문” 추가(“이 워크플로우는 미검증, 위험에 유의”)와 공식 워크플로우 구분 제안

### 다양한 코딩 에이전트(PI, 코드엑스, Claude 등)와 모델의 조합이 Archon 워크플로우 내 자유롭게 사용됨을 예시로 설명

- providers 필드로 노드별로 모델/에이전트 개별 지정 가능 (예: 구현은 Claude, 검증은 CodeX 등)
- 실제 엔드투엔드 테스트 예시(혼합 provider 테스트) 설명
- 신규 PI 코딩 에이전트 지원 사항(PI 추가, cloud code, codex와의 차이 등) 소개
- 각 에이전트별 특성(엔드포인트/SDK 요구, 활용 제한 등) 및 통합 방법 안내

### 로드맵/마켓플레이스의 구현·테스트 과정과 PR(풀리퀘) 관리 절차를 라이브 코딩・운영 과정과 함께 시연함

- docs/web 내 roadmaps, workflows page 실시간 수정 및 Git 변경사항 분리, PR 생성 절차 공개
- 실시간으로 마크다운, 메타데이터, 커뮤니티 워크플로우 폴더 구조 점검
- PR 분리(로드맵 관련만 우선, 마켓플레이스는 임시 보류)—롤아웃 순차 적용 전략
- 주요 결정은 유지관리자(Maintainer) 간 논의 후 확정 예정, 팀 공통 검토 및 커뮤니티 의견 수렴 강조

### Archon 워크플로우 확장성과 미래 발전 방향: 인증·평가·자동화·커뮤니티 인게이지먼트 등 논의

- 커뮤니티워크플로우가 폭증 시 PR/리뷰 부담 한계 지적—장기적으로는 마켓플레이스 자동화, 가입/별점/버전 운영 등 추진
- 외부 DB 연동(Supabase, Neon 등), 별점·사용 통계 관리, 로그인 플랫폼 검토
- 기본 워크플로우(템플릿)와 실험적/맞춤형 워크플로우의 공존, 참고용·커스터마이즈 전략 권장
- 오픈소스 정신 및 커뮤니티 기여·피드백 중시 철학 강조(컨트리뷰션 장려, 테스트, 자동화 공존)

### 클라우드 코드·AI 에이전트 발전, Archon의 오픈소스 방향 등 개발 문화적 관점 공유

- Archon 개발자 스스로의 성장 배경(스크래치→파이썬→C/Cpp→대학 CS 과정) 공유
- AI 에이전트/코딩 에이전트 시장의 급속 전환(GPT 2/3.5 이력, Pi와 Claude의 대화형 특징 등)
- 오픈소스 First/공유-테스트-확장 철학 및 커뮤니티 주도 개발 기조 재차 강조
- 라이브 스트림/유튜브/다이나무스 커뮤니티와의 적극 소통 의지 표현

### 실시간 Q&A를 통한 추가 정보 제공: 지원 에이전트별 API 현황, 로컬 환경 매핑, 워크플로우 커스터마이징 등

- AWS Kira CLI 연동 불가 사유(프로그램적 SDK 미제공) 설명
- 기본 워크플로우가 어떤 식으로 다양한 기술스택에 맞춰 커스터마이즈 가능한지 예시 시연
- 각 에이전트별 기능(채팅 히스토리/포크/브랜치 등) 및 코딩 -협업 시 실효성에 대해 안내
- 워크플로우간(상위-하위) 연동 및 output-context 전달 설계 관련 팁 제공

### 로드맵 최종 구현 및 향후 과제(마켓플레이스 프로토타입/보안/자동화 등) 명확화하며 라이브 종료

- docs/web 구조 내 각 기능별 페이지 및 커뮤니티워크플로우 폴더 체계 확정, 기본 메타데이터 방식도 완비
- “로드맵 페이지”를 통한 maintainer(관리자) 주도 공공 공개 정책 표방, 추후 라이브 업데이트
- 워크플로우 마켓플레이스는 장기적 보완 예정—단계적 도입 및 점진적 자동화+보안 강화 기획
- 모든 주요 결정은 팀/커뮤니티 내합의 및 실제 사용 피드백 기반으로 지속적으로 발전해 나갈 예정임
