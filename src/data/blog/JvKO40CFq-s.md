## *더 나은 에이전트 인증 — 에이전트 권한/신원 관리 워크숍* 핵심 요약

- 영상은 AI 에이전트 시대에서 보안·인증 문제를 다루고, 기존의 “사용자 인증정보(credential) 공유” 관행의 한계를 지적하며 “에이전트 권한(authz) 위임” 패러다임 필요성을 강조함
- 초기 예시로, 사용자가 “내 이메일 읽어줘” 등 요청 시 실제 기능 실행 방법(에이전트에 내 계정 직접 연결)이 얼마나 취약한지 질문함
- “회사에 신입사원 채용 시 CEO의 전 권한을 바로 주지는 않는다”는 비유를 통해, 에이전트 역시 “전체 계정 접근”이 아니라 “역할에 따라 제한된 권한”만 구체적으로 부여해야 함을 주장
- 에이전트에 필요한 권한만 세분화해 부여하는 ‘capabilities’ 중심의 구조 제시, 구글 OAuth의 ‘scope’(ex: 전체 이메일 읽기)보다 세밀하게 제한/관리 가능
- 모든 에이전트가 개별 신원(Identity, 프라이빗 키 기반)을 갖도록 설계해, “어떤 에이전트가 언제 무엇을 했는지” 감사·추적(log/audit)이 가능하도록 함
- 사용자는 UI(디렉터리, 파인그레인드 권한 승인 등)를 통해 에이전트별 권한 부여·회수·감사 등 쉽게 통제할 수 있고, 실시간 데모(이메일 읽기/전송, 에이전트 차단)로 이를 시연
- 프로토콜은 OIDC, OpenAPI 등 기존 표준 위에서 동작하며, 보안·사용성 밸런스를 유지하는 다양한 정책(호스트별 제한, 시간제한 등) 적용을 지원함
- 오픈소스 프로젝트로 진행 중이며, “AI를 안전하게 쓰는 생태계”를 위해 다양한 제안 환영, 커뮤니티 참여 당부로 영상 마무리

## 세부 요약 - 주제별 정리

### AI 에이전트 시대에는 사용자의 인증정보 공유가 심각한 보안 문제를 초래함

- 실생활에서 “AI 에이전트에게 Gmail, 캘린더 등 내 계정 정보를 자주 주는가?”로 청중에게 질문
- 실제로 “에이전트가 나인 척해서” 내 계정에 로그인하거나, 토큰/MSP 서버 연결 등으로 데이터에 접근하는 사례가 다수임
- 이는 실질적으로 사용자(CEO)의 전 권한을 그대로 넘기는 것과 동일, 오남용/해킹 사고 발생 시 큰 위험 초래
- 과거 인터넷 도입 초기에 모든 데이터가 공개되어 곧 보안 위협 논의가 본격화된 것과 AI 에이전트 도입 초기가 비슷함을 환기함

### 에이전트를 ‘고용’하는 것처럼 제한된 권한과 신원을 부여하는 것이 바람직함

- “회사에 신입을 고용할 때, 이메일/자원 전체 권한을 모두 주지 않는다”는 비유 제시
- AI 에이전트도 사용자 전체 credential(계정정보)을 넘기지 말고, 에이전트 개별 identity(신원)와 authority(권한)를 분리해서 부여 권장
- 권한 범위(authorization)를 업무나 필요에 맞게 세분화하여 부여해야 함
- 신뢰관계가 깨질 경우(에이전트가 이상 행동), 부분적으로 즉시 권한 해지(revoke)해 사고 확산을 막을 수 있음

### “에이전트가 무엇을 할 수 있는지” 자동으로 알 수 있는 디렉터리/발견 시스템의 필요성을 강조함

- 현행 AI 활용 시, 사용자가 수동으로 에이전트에 기능 및 API 정보를 연결·지정해야 하는 비효율 발생
- 이상적인 구조는, 에이전트가 디렉터리(‘폰북’처럼 역할·가능기능이 명시된 자료)에 접근해 ‘내가 할 수 있는 일’을 자동 검색·발견하는 것
- OpenAPI Specification 등 표준을 활용하면, 각 서비스별로 제공하는 기능을 capability 단위로 인식 및 자동 등록 가능
- 실제 구조 예시와 데모를 통해, Notion, Gmail 등 다양한 서비스의 OpenAPI 스펙을 capability로 변환하고 이를 디렉터리에 등록하는 사례 시연

### 기존 OAuth ‘스코프’보다 더 세밀한 ‘capability’ 체계로 기능별 권한 제어가 가능함을 설명함

- OAuth의 ‘scope’는 대부분 전체/폭넓은 접근(ex: read access)이어서 실제 위임하려는 세부 기능 분리에는 한계
- ‘capability’는 “get”, “delete”, “send” 등 API 엔드포인트별로 최소 단위로 세분화됨
- 사용자는 특정 capability만 허용/거부·자동 승인 정책 등 기준을 세밀히 지정할 수 있음
- 실제 디렉터리, 데모 화면을 보여주며 capability 기반 권한 부여·재할당 과정을 구체적으로 설명

### 에이전트별 개별 신원(Identity) 부여를 통해 감시·감사·추적성을 확보할 수 있다고 제시함

- 각 에이전트는 독립된 프라이빗 키·아이디를 갖고, 그 키로 토큰을 서명 및 행동 기록을 남김
- 로그 시스템을 통해 “어떤 에이전트가, 어떤 사용자에 속해, 어느 시각에 무엇을 했는지”를 명확히 추적·감사 가능
- 감시·감사 이슈와 함께, 필요 시 ‘해고’(권한 회수 및 token revoke, host 삭제 등)를 실제 바로 실행하는 데모 시연
- 기존에는 전체 계정 연결해둬서 사고 시 전 해제를 해야 했지만, 새로운 방식에서는 에이전트 단위로 통제가 가능해짐

### 오픈API 및 디렉터리 매칭을 통해 현존 서비스들과의 연동 실현 가능함을 구체적으로 입증함

- Notion, Gmail 등 주요 서비스의 OpenAPI JSON 명세를 capability 디렉터리로 자동 변환(또는 수동 선언)하여 적용
- 데모에서 Notion API를 예시로, 각 엔드포인트별 capability 목록화 및 적용 과정을 보여줌
- 아직 모든 서비스가 완전한 지원을 하진 않으나, 표준 방식+중간 매핑으로 대부분 연동 가능함을 강조

### 권한 범위, 시간, 대상 등 세분화 정책(Policy) 적용을 쉽게 지원함

- capability를 통해 “최대 실행 횟수”, “허용 시간”, “특정 호스트만 허용” 등 각종 제약 가능
- 예를 들어, 에이전트에 이메일 읽기는 자동 승인하지만 전송은 직접 승인/거부하는 방식 등 유연 설정
- 호스트(에이전트가 생성된 환경)가 분리되어, 호스트별로도 정책 달리 설정 가능

### 사용자 경험(UX)과 보안을 동시에 고려한 권한 구조 설계를 시연 및 설명함

- 각 기능별로 “항상 허용/항상 거부/매번 승인” 등 다양한 정책 제공 — 과도한 승인 요구 등 피하기 위함
- 실제 GUI를 통해 Gmail 연동/에이전트의 이메일 읽기·전송 시 실제 권한 요청, 실시간 승인/거부/감사 과정 시연
- 기존에 잘못된 동작 혹은 권한 남용 발생 시, 신속하게 해당 에이전트만 권한 회수 후 기능 정지됨을 확인시킴

### ‘호스트’, ‘에이전트’, ‘사용자’의 신원과 행동을 명확히 구분하며, 감사/라이프사이클 관리가 가능함

- 신원 구조는 사용자-에이전트-호스트로 분리, 각 주체별로 독립적 로그·상태 관리 가능
- 어떤 에이전트가 어떤 사용자로부터, 어떤 호스트(환경)에서 생성되고 어떤 행동을 했는지 확인·통제 가능
- UI에서 “에이전트 상태 조회”, “호스트 삭제”, “정상/이상 행동 감지 및 회수” 등 구현됨

### 프로토콜 구조(플러그인/SDK/서버/디렉터리) 및 개발자 활용 방법 제시

- Agent Auth Protocol v1 데모, v2(진화 중) 안내 — 오픈소스 기반
- 플러그인(Agent Plugin): 서버에서 에이전트 검증·권한 부여 등 실행
- SDK: MCP(연결 지점)에 연동하여 키 생성·에이전트 신원 할당
- 디렉터리: intent(요청의도)-capability(가능행동) 매칭 수행, 폰북 구조의 UI 제공
- 개발자는 각 레이어의 구성 요소 사용, 실시간 감사 및 정책 설정을 자유롭게 시연 가능함

### ‘에이전트에 인증정보가 아닌, 제한된 권한만 줘야 한다’는 결론과 오픈소스 프로젝트 참여 요청

- “에이전트에게 우리 credential(계정정보)가 아니라 권한(authority)만 주자”, “나를 흉내 내려 하지 말고, 내 제한적 역할만 맡으라”는 패러다임 전환 요청
- 오픈소스 및 커뮤니티 주도 프로젝트로, v2에서 기업용·정책 다양화 적용 등 진화 예정
- 개발자·사용자 누구든 아이디어·제안·참여 가능함을 안내하고, 프로젝트 연락처, 디스코드 등 공개하며 마무리
