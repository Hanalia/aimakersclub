## *RAG는 정말 끝났는가? 하이브리드·도구 기반 검색의 부상* 핵심 요약

- **발표자는 Turbo puffer의 배포 엔지니어 Kuba Rogut로, Turbo puffer는 객체 스토리지 위에 구축된 풀 텍스트 및 벡터 검색 DB임**
- **최근 'RAG는 죽었다'는 트윗과 주장들이 소셜 미디어상에 확산, 2025~2026년 검색 트렌드 데이터에서도 이 논쟁이 반영됨**
- **RAG(검색 증강 생성)는 단순 벡터 검색이 아니며, BM25·정규 표현식·필터 등 다양한 검색 도구를 아우르는 개념임**
- **Agentic Search(에이전틱 검색)는 Cloud Code 등에서 볼 수 있듯, 반복적 파일 탐색/grep 수행 등으로 맥락을 점진적으로 찾는 도구적 검색을 의미**
- **Turbo puffer의 주요 고객사 Cursor는 코드베이스에 Merkle tree 도입 및 중복 최소화 전략으로 임베딩 시간과 비용을 극적으로 절감**
- **Cursor에서의 시맨틱 검색 도입은 내부 벤치마크에서 응답 정확도 12.5~24% 향상, 대형 코드베이스에서 2.6% Retention 증가·2.2% 불만족 요청 감소의 실적 기록**
- **Cloud Code는 현재 벡터 검색을 사용하지 않으며, Agentic Search가 반복적 파일 읽기로 인한 토큰 및 비용 증가 문제 내포**
- **반면 Cursor 방식은 선행 인덱싱 후 런타임의 효율적 검색으로 성능·속도·비용 절감 효과 큼. Turbo puffer 내에서도 Cloud Code→Cursor로 내부 이동이 늘어남**
- **최근 AI 고객사들은 '1회 벡터 검색→LLM 입력' 식의 초기 RAG 대신, 에이전트가 여러 검색 도구를 단계적으로 이용하는 복합 하이브리드 검색을 채택 중**
- **구글 제프 딘의 “트릴리온 컨텍스트 윈도우가 아니라, ‘적절한 백만 토큰’을 뽑는 단계적 검색이 중요하다”는 어록이 기술적 방향성에 큰 시사점 제공**

## 세부 요약 - 주제별 정리

### 소셜 미디어 은유와 검색 트렌드를 통해 RAG 논쟁의 현재를 조명함

- 2025~2026년, 트위터(X)에서는 'RAG is dead', '우리에겐 Agentic File Search만 있으면 된다'라는 주장들이 확산됨
- 그러나 2023~2025년 사이 구글 검색량 데이터에서는 RAG 및 관련 검색어가 꾸준히 증가하며, 2025년 중반부터 새로운 급등세 관측
- 단순 소셜 미디어상의 담론 파동과 실제 고객사/기술업계 내 수요·관심의 온도 차를 보여줌

### RAG와 에이전틱 검색 각각의 정의가 현장에서 오해되고 있음

- 많은 사람들이 RAG를 “단순 벡터 검색”으로 오해하나, 본래는 다양한 검색방식(벡터·BM25·grep·이름 기반 glob·정규식·필터 등)과 ‘검색 결과 + LLM’이 결합된 총체적 프로세스
- 에이전틱 검색(agentic search) 역시 “파일 시스템 grep = 에이전틱” 식의 축약된 통용단어이지만, 실제로는 “에이전트가 여러 도구를 반복적으로 적용하며 적합한 맥락을 찾는 점진적 추론·검색”을 의미
- 예) Cloud Code/Codex는 파일 기반 점진적 탐색, 실패할 시 추가 탐색 반복→“해피 스테이트” 도달 시 다음 태스크 진행

### Cursor 사례를 통해 코드베이스 임베딩의 현실적 최적화 방안을 제시함

- Cursor(초기 Turbo puffer 고객)는 신규 코드베이스나 브랜치 오픈 시, 코드 전량을 chunk·임베딩해 시맨틱 검색에 활용
- 대규모 팀(100명)의 실제 작업 패턴 분석: 대다수 개발자가 동일 코드베이스를 반복해서 엽니다→매번 임베딩/업로드하면 비용·시간 부담이 큼
- 이를 해결하기 위해 ‘Merkle Tree(해시 트리)’로 유사 코드베이스 간 변경분만 추출→변경 파일만 파싱·임베딩·업로드→중복 최적
- Turbo puffer DB와의 결합으로 이 과정의 보안성/신속성 강화

### Cursor의 시맨틱 검색 도입이 실제 제품 성능 상승으로 이어짐

- Cursor 내부 ‘context benchmark’ 기준, 시맨틱 검색 도입 시 모델 전체적으로 12.5~13.5% 응답 정확도 상승(특히 Composer 모델에서 24%까지 증가)
- 온라인 AB Test에서 대형 코드베이스 기준 “코드 유지(retention) 2.6% 증가”, “불만족 요청 2.2% 감소” 달성
- 모든 쿼리에 시맨틱 검색이 사용되는 것은 아니므로, 전체 집계 대비 수치가 다소 작아 보임에도 불구하고 유의미한 개선

### Cloud Code의 아키텍처적 한계와 RAG 미활용 배경을 구체적으로 설명함

- Cloud Code는 벡터 검색 미사용(설립자 Boris Cherney 트윗 인용: “초기엔 벡터 DB-RAG 썼으나 효과 미흡”)
- Turbo puffer 내부에선 “임베딩·시맨틱 검색 = 단기 캐시+컴퓨트”로 해석
- Cloud Code 방식: 세션별(에이전트별)로 반복적인 grep, 파일 읽기 필요→동일 질문 반복시 매번 동일 탐색 경로 거침→토큰/비용 모두 누적(예시: 메타데이터 필터링 구조 질문에만 6,000토큰 소비)
- Cursor 방식: 초기 인덱싱 비용은 존재하나, 런타임 탐색 비용·속도·토큰 대폭 절감→팀 내 Turbo puffer 사용자들도 “속도/효율성 때문에 Cursor로 실제 전환 중”

### 하이브리드·도구 기반 검색으로의 패러다임 전환이 본격화됨

- 트위터 논의상 “RAG=1회 벡터 검색→LLM 콘텍스트” 모델은 2023~2024년 초 ‘AI 도입기’에는 유효했으나, 최근 실제 고객사들은 복합적·다단계 검색(벡터+텍스트+파일+필터 조합)을 에이전트가 적시 선택
- 이러한 “하이브리드, 도구 리치, 반복적 맥락 추론형 검색”이 확산되고 있음
- 단순 벡터 검색이 아닌 ‘필요한 만큼, 단계적으로, 다양한 도구로 필요한 컨텍스트만 추출’하는 구조가 성능·서비스 혁신의 기반이 됨

### 구글 제프 딘의 “거대한 컨텍스트 윈도우보다 단계적 올바른 토큰 선택이 더 중요하다”는 견해가 이 패러다임에 의미 부여

- 제프 딘(구글 AI 수석 엔지니어): “트릴리온 사이즈 컨텍스트 윈도우 자체는 중요하지 않다. 정말 필요한 것은 ‘스테이지드 리트리벌’, 즉 수많은 토큰 중에 진짜 필요한 백만 토큰, 십만 토큰, 만 토큰을 빠르고 가볍게 뽑아내는 체계”
- 실제 Turbo puffer 내부, 고객 관심사 모두 단일 대규모 벡터 DB→다단계 축소 검색→필요한 컨텍스트만 LLM 입력이라는 방향에 초점
- 대규모 토큰 자체가 아니라, 그중 ‘적합한 정보’를 효율적으로 가져오는 체계화가 미래 AI 엔지니어링의 핵심임

### Turbo puffer가 추구하는 데이터베이스 플랫폼의 역할과 가치 제시

- 객체 스토리지 기반에 풀텍스트/벡터 검색을 차별화 포인트로 내세움
- Merkle tree 등을 통한 유사성 계산 및 중복 배제, 다양한 검색 도구 통합 제공
- 보안성, 확장성, 비용 효율성을 바탕으로, 하이브리드 검색을 실질적으로 지원하는 제품 전략 강조

### 발표의 맥락 및 참가자 Q&A 권유로 마무리

- 기술적 내용 외에, 발표 세션 후 추가 질문이나 1:1 대화를 통해 Turbo puffer의 상세 특징이나 활용 사례에 대해 더 알아갈 수 있음을 안내하며 마무리
