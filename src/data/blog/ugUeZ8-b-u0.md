## *Rust가 바이브 코딩에 이상적인 언어인 이유* 핵심 요약

- 발표자는 Sentry의 Rust SDK 관리자 Daniel Szoke로, Rust가 ‘바이브 코딩(Vibe-coding, Agentic coding)’에 최적의 언어임을 주장함
- 기존 통념상 바이브 코딩에선 Python, TypeScript, JavaScript가 최적이라고 여겨지며, 실제로 GitHub에서도 AI의 영향으로 이들 언어의 사용 비중이 급증함
- 이러한 언어들이 선택되는 주된 이유는 보편적이고 친숙하며, LLM(대규모 언어 모델)이 쉽게 코드를 작성하고 바로 실행해볼 수 있기 때문임
- 그러나 "모델이 언어를 쉽게 쓸 수 있다는 점"이 정말 최적의 기준인지 의문을 제기하며, 오히려 이런 언어의 유연성 때문에 실수가 쉽게 발생한다는 점을 강조함
- 타입 안정성(type safety)이 강화된 언어라 해도 Python, TypeScript 등은 제한적이며, 궁극적으로 LLM의 근본적 오류 가능성은 사라지지 않음
- 테스트와 코드 리뷰로 오류를 걸러내는 전통적 접근에도 한계가 있고, 에이전트가 직접 생성한 테스트 코드 역시 신뢰성이 부족할 수 있음
- LLM은 인간과 사고 방식이 완전히 달라 예기치 못한 오류가 종종 발생하며, 인간 중심의 오류 방지책만으로는 충분하지 않다고 설명함
- 머피의 법칙(‘잘못될 수 있는 것은 언젠가 반드시 잘못된다’)에 따라, 언어적 결정론적 안전장치가 없는 Python, JS 등은 생산 코드에서 오류가 빈번히 남음
- Rust는 엄격한 컴파일 과정과 타입·메모리·동시성 안정성 등으로 치명적인 버그를 사전에 원천 차단하며, 초보자도 접근 가능한 친절한 에러 메시지 제공 등 학습 허들이 낮음
- Rust 코드는 LLM이 바로 작성하기 어렵지만, 에이전트가 반복적으로 컴파일-에러 수정 구조로 동작할 때 오히려 버그 없는 코드를 만드는 데 유리함
- 마지막으로, Sentry의 무료 3개월 이용권 및 부스 방문을 안내함

## 세부 요약 - 주제별 정리

### Python과 TypeScript 등 기존 언어가 바이브 코딩에 주로 쓰이는 이유가 설명됨

- Python, JavaScript, TypeScript는 개발자와 인공지능 모두에게 친숙한 언어임
- ChatGPT 역시 에이전트 중심 코딩에 가장 적합한 언어로 Python, TypeScript, JavaScript를 추천함
- GitHub는 AI의 영향으로 2023년 기준 기여자 수 면에서 TypeScript가 1위 언어가 됐다고 발표함
- 이들 언어는 프로그래밍 입문자들이 가장 먼저 접하는 경우가 많아 인간에게 학습장벽이 낮음
- LLM 역시 Python, JavaScript, TypeScript를 쉽게 다루며, 코드 실행이 빠르고 피드백 루프가 짧음
- 다양한 프레임워크, 라이브러리, 코드 예시가 많아 개발과 실험이 신속하게 진행됨
- TypeScript와 Python(타입 애노테이션 포함)은 어느 정도 타입 안정성을 제공해 LLM/에이전트의 타입 오류를 부분적으로 예방함
- 전체적으로, LLM의 코드 생성 성공률이 높고 반복성 및 유연성을 제공한다는 점이 선택의 근거임

### 기존 언어의 쉬운 사용성에 대한 맹신이 문제로 지적됨

- LLM이 쉽게 쓸 수 있다는 점이 반드시 장점만은 아님
- JavaScript, Python 등은 동적이고 유연해 코드 작성이 쉽지만, 그만큼 실수 가능성이 증가함
- 이러한 언어들은 명확하지 않은 타입 안정성(any 타입 등)과 제한적 에러 검출 기능 등 한계가 존재
- 간단한 타입 도입으로 오류를 방지하려 해도, 근본적으로 언어가 허술해서 한계가 명확함
- LLM의 비결정론적 특성(같은 입력에도 결과가 달라질 수 있음)이 치명적 오류를 발생시킬 수 있음
- LLM은 영원히 인간처럼 ‘무결점’이 될 수 없으며, 오류 방지 대책이 반드시 필요함

### 테스트 및 코드 리뷰 등 전통적 안전장치의 한계가 구체적으로 드러남

- 에이전트 주도의 코딩에서 테스트 코드는 종종 실제 구현 이후에 생성되어, 의도한 동작 검증에는 미흡할 때가 많음
- 테스트 주도 개발을 해도, 모든 입력 조합을 실질적으로 테스트하는 것은 불가능함
- 테스트 코드와 코드 리뷰 역시 LLM/에이전트에 의해 작성될 경우, 더 미묘하거나 예상치 못한 실수가 있을 수 있음
- 오류를 '증명'하는 데는 도움이 되지만, '안전함'을 완전하게 증명할 수 없음

### LLM의 의사결정 방식이 인간과 다르며 예측 불가한 실패를 유발함

- 《Nexus》라는 책과 유발 하라리(Yuval Noah Harari)의 인용을 통해, LLM의 사고 구조는 인간과 본질적으로 다름을 언급
- 인간은 인과적 사고 및 맥락 기반 추론을 하지만, LLM은 토큰 예측 기반의 ‘이방인(Alien)적’ 추론을 함
- 겉으로 보기엔 훌륭한 코드가 생성됐더라도, 내부적으로 미묘하거나 심각한 버그가 숨겨져 있을 수 있음
- 인간이 생각하지 못한 방식의 오류가 실제로 다수 발생함(예를 들어 어눌한 변수 처리, 누락된 검증 등)

### 결정론적(Deterministic) 안전장치가 없는 언어는 비즈니스 환경에서 위험함

- 머피의 법칙(Anything that can go wrong, will go wrong)은 소프트웨어 개발에서도 그대로 적용됨
- 동적·유연 언어는 에이전트·인간 리뷰 및 테스트 등으로 방어해도, 언젠가는 치명적 오류를 낳을 가능성이 높음
- 결정론적인 언어적 안전장치(Strict compiler, 강제적 타입·메모리 검증 등)가 없는 환경은 대규모 시스템 운영상 큰 리스크가 됨

### Rust는 엄격한 컴파일과 안전성 규약으로 오류를 ‘사전에’ 차단함

- Rust는 컴파일 언어이며, 안전성(safety)과 성능(performance)을 동시에 지향함
- 설계 목표는 C/C++만큼 빠르면서도, 메모리 및 타입 안전성을 극대화하고 동시성 오류도 원천 차단하는 것임
- Rust 컴파일러가 통과된 코드엔 다양한 종류의 치명적인 버그가 거의 존재하지 않음을 어느 정도 보장함
- 타입 안정성(type safety)이 강제되며, 타입 변조나 명시적 무시를 통한 회피가 불가능함
- null 처리 역시 Optional 타입으로 강제되어 Null dereference와 같은 오류 방지가 가능함
- 동시성(Concurrency) 안전도 ‘fearless concurrency’로 요약되듯, 멀티스레드 환경에서 데이터 레이스를 컴파일 타임에 검출함

### Rust의 null 처리, 타입 안정성, 동시성 보장 방식이 구체적 예시와 함께 설명됨

- 모든 Null 가능성은 Option 타입으로 명시적 선언이 필요, 무심코 Null 접근 불가
- 타입 우회(any, unchecked cast)이 불가능해 런타임 오류 대부분 compile-time에 발견
- 멀티스레드 환경 예시에서, TypeScript 등은 동기화 오류가 실행 중에야 드러나지만, Rust는 컴파일 타임에 강제 에러를 출력함
- 예시 코드: 100개의 쓰레드가 동시에 하나의 카운터 값을 증가시킬 때, 적절한 thread-safe 타입이 아니면 러스트 컴파일러가 에러를 보여줌
- 이에 해당 에러 메시지는 문제점을 정확히 진단할 수 있게 안내해, 에이전트 역시 수정 방향을 쉽게 알 수 있음

### 친절한 컴파일 오류 메시지가 초보자·AI 에이전트 모두에게 실질적 도움을 줌

- Rust 컴파일러는 상세하고 구체적인 에러 메시지, 문제 발생 위치, 해결법을 안내함
- 프로그래밍 초보자는 물론 AI 에이전트도 에러 메시지를 기반으로 자동 수정을 시도할 수 있음
- 에이전트가 반복적으로 컴파일-수정 루프를 실행하면, 최종적으로 사람이 발견하기 힘든 버그까지 제거 가능함

### Rust는 LLM·에이전트 기반 개발에서 더 신뢰할 수 있는 프로덕션 코드를 만들어 냄

- Rust의 엄격한 규칙으로 LLM·에이전트가 첫 코드 생성 시 정답을 내기 어렵지만, 실패 후 자동 수정 루프에선 오히려 품질이 높아짐
- 반복적 compile-fix 구조는 인간/AI 공동 개발 시 버그 없는 코드 생산에 적합함
- Rust 컴파일러의 체크 시간은 느릴 수 있으나, 실제로는 인간 코드 리뷰나 LLM 리뷰보다 더 신속하고 신뢰도 높음
- 에이전트 주도의 코딩에서 Rust의 compile error는 '무료 버그 검출기'와 같으며, 생산 코드의 안전성을 대폭 향상시킴

### Rust 사용은 단기 생산성보다 중장기 품질·안전성이라는 근본적 우위를 확보함

- LLM이 쉽다고 Python/JS를 선택하면, 단기적으로는 빠르나 장기적으로 오류·유지보수·신뢰성의 리스크가 내포됨
- Rust는 개발 입문 난이도가 개선되어 있고 강력한 컴파일러 보조가 제공됨
- 에이전트 주도 개발이 성숙할수록 Rust의 방식이 산업·실무에 더 적합하다고 주장함

### 발표자 회사 Sentry의 서비스와 프로모션 안내로 마무리됨

- 발표자는 Sentry의 Rust SDK 관리자로, 회사의 에이전트 모니터링 기능과 무료 3개월 쿠폰(QR 코드)을 안내함
- 발표 또는 Sentry에 대한 질문은 부스에서 자유롭게 가능하다고 홍보함
