**채널명:** AI Engineer

## *MCP는 따분하다(혹은: 왜 우리는 LLM의 반짝임을 잃고 있는가?)* 핵심 요약

- 영상은 LLM(대형 언어 모델)에 대한 현대적 활용 방식이 초기의 혁신성과 호기심을 잃고 ‘MCP’ 같은 지루한 패턴에 갇혔음을 비판적으로 조명함
- ‘MCP(Minus Completion Plus)’란, 텍스트 일부에서 빈칸을 주거나, 단순 교정·보완 형태의 LLM 사용 방식에 대한 풍자적 명칭임
- 현재 LLM 활용이 주로 ‘빈칸 채우기’ 또는 ‘결과물 소폭 수정’ 등에 머물러 있으며, 이는 LLM의 진정한 잠재력을 제한한다고 주장함
- 초창기 LLM 프로젝트들(예: Copilot, GPT-3 Playground)의 실험성과 창의성이 현대에는 ‘프로덕션화’ 과정에서 사라지고 있음을 지적
- 사용자·개발자 모두가 LLM을 일상적, 반복적 도구로만 대하며 새로운 인터페이스나 상상력의 자극이 부족해졌다고 비판함
- LLM의 “스파클”(창의적 반짝임)을 다시 발견하기 위해 패턴화된 MCP 방식을 넘어서야 한다고 제안
- Copilot 및 클론 제품, 각종 LLM 챗봇 등이 보여주는 한계 사례, 현업 AI업계의 MCP 패턴 집중 현상 등 구체적으로 제시
- LLM의 혁신적 적용을 이끄는 ‘상상력 있는 실험’이 현저히 줄어들었음을 전체 예시와 데이터 등으로 설명함

## 세부 요약 - 주제별 정리

### LLM 도입 초기에는 상상력과 실험정신이 두드러졌음을 다양한 예시로 소개함

- 2020년대 초반 GPT-3, Copilot, Playground 등 LLM 관련 초기 프로젝트들은 일반인의 상상력을 자극할 만큼 신선한 느낌을 제공함
- Copilot 등장 시 "AI가 코드를 자동 완성해준다"는 점에서 프로그래머들 사이에 '마법'과 같은 반응을 이끌어냄
- Playground에서는 사용자가 직접 프롬프트를 넣어보고, 예상치 못한 창의적 결과나 실수 자체에 재미를 느낄 수 있었음
- ‘AI Dungeon’ 등 인터랙티브 텍스트 어드벤처, 즉석 단편소설 생성 등 다양한 장르의 실험들이 유행함
- 당시는 정답보다 ‘새로운 상호작용 방식’ 자체가 신기하고 주목받았음

### LLM을 활용하는 방식이 점차 MCP(단순 빈칸 채우기 패턴)로 고착되고 있음을 주장함

- MCP(Minus Completion Plus)는 아주 제한적인 컨텍스트를 주고 LLM이 ‘알아서’ 한 조각을 메꾸도록 하는 방식임
- 가장 흔한 예는 "이 코드를 조금 고쳐줘", "이 부분만 영어로 번역해줘" 등 작은 수정·보완 요청임
- 현재 많은 프로덕션 서비스, 특히 Copilot이나 각종 LLM 플러그인/툴들은 이런 방식(MCP)에 치우쳐 있음
- 유저 인터페이스 역시 광범위하지 않고, 일상의 사소한 반복작업(문장 교정, 줄 바꾸기 등)에 집중됨
- LLM을 단순히 오퍼레이터(명령 응답 도구) 수준으로만 사용하게 됨

### MCP 방식은 LLM의 잠재력 중 극히 일부분만 활용하게 만들었음을 지적함

- LLM의 진가인 ‘상상’, ‘창의성’, ‘지식 추론’은 MCP 방식에서는 드러나기 어려움
- 제품화 과정에서 관리자, PM, 개발자는 “이 기능 하나만 확실하게 돌아가게”에 집중하면서 확장성을 경시함
- 일부 챗봇이나 코딩 보조툴에서 반복적으로 보이는 사례: 사용자는 늘 비슷한 프롬프트(‘~해 줘’)만 입력
- 그 결과, LLM 사용자 경험이 반복적이고 예측 가능하게 수렴하게 됨

### Copilot 및 기타 LLM 기반 서비스도 MCP 패턴에 집중하면서 창의성이 떨어졌음을 설명함

- Copilot, Copilot X, 기타 코드 자동완성 툴들은 초기에는 “코딩을 새로 정의한다”는 평가를 받음
- 그러나 점차 “함수명 자동완성”, “코멘트 생성” 등 예측 가능한 단일 목적 기능에 치중하게 됨
- 예시: Copilot이 ‘다음 줄 코드’나 ‘함수 보완’만 반복적으로 제안하는 상황 발생
- Copilot 클론 제품(예: Amazon CodeWhisperer, Tabnine 등)도 모두 유사한 MCP 패턴 위에 동일한 UX 제공함

### 인터페이스, 프롬프트 디자인 등에서의 상상력 결여가 전반적 문제임을 지적함

- 오늘날 LLM 기반 서비스의 UI는 대부분 ‘텍스트 입력 박스+응답’ 구조에 머무름
- 진입장벽이 낮다는 장점이 있지만, 반대로 새로운 상호작용 방식(예: 멀티모달, 음성, 그래픽 통합 등)에 대한 실험은 드묾
- 초창기 LLM 실험처럼 ‘조건 생성’, ‘서사 확장’, ‘설정 변경’ 등 상상력을 자극하는 인터페이스가 사라졌음
- 결국 사용자는 LLM을 프롬프트의 “자동완성 버튼” 정도로만 여기게 됨

### MCP 패턴이 AI 업계 전체 트렌드가 되었음을 구체적 데이터·사례로 뒷받침함

- 스타트업, 대기업 모두 LLM 관련 신규 제품의 대부분이 텍스트 자동완성, 챗봇, 요약 등 ‘MCP’ 범주에 속함
- 실제 Copilot 등 주요 LLM 툴의 기능 업데이트 내역을 살펴봐도 예외적으로 혁신적인 기능은 거의 없음
- 2024년 기준, 가장 인기 있는 LLM 프론트엔드 제품(예: Notion AI, Jasper 등)도 주로 ‘빈칸 채우기’ 기능 위주임
- AI 개발자 커뮤니티에서도 “새로운 인터페이스/UX” 논의는 줄어들고 ‘기존 기능 정밀화’ 중심으로 이동함

### LLM 사용자 경험이 패턴화되면서, 유저-개발자 모두가 싫증을 내기 시작했다고 분석함

- 초기에는 LLM의 의외성, 기상천외함 자체에 놀라고 즐거워했으나, 반복된 MCP 패턴에 익숙해짐
- 사용자들은 “별다른 반전 없는 챗봇”, “복붙을 줄여주는 툴” 등에 점차 흥미를 잃음
- 일부 개발자는 “LLM으로 더는 새로운 경험을 만들 수 없다”는 자조적 분위기까지 조성
- 실제 LLM 관련 트위터·해커뉴스 논의에서 “LLM은 이미 지루하다”는 평가가 다수 확인됨

### LLM의 스파클을 되찾기 위해 MCP 공식에 머무르지 않는 실험적 시도가 필요함을 강조함

- LLM을 단순 자동완성 기계가 아니라, ‘상상력 엔진’으로 활용할 수 있는 방법 모색 필요
- 영상에서는 “예측 불가능한 프롬프트 조합”, “다중 사용자 협업”, “스토리텔링 확장” 등 예시를 추가로 제안함
- 실제로 LLM 출력 결과를 미리 정해놓지 않은 상태에서의 실험적 UX 도입 가능성을 언급
- 인터페이스, 어플리케이션 레이어, 알고리즘 등 전 단계에서의 새로운 접근이 중요하다고 역설함

### 제작자는 LLM 커뮤니티와 AI 개발자들에게 현실을 돌아보고 더 다양한 실험을 촉구함

- 영상 마지막에는 “우리는 왜 MCP에 만족하며, 왜 더는 실험하지 않는가?”라는 질문을 던짐
- 커뮤니티 안에서 복잡하고 새로운 상호작용 방안, 창의적 활용 사례가 적극적으로 공유·논의되어야 함을 강조
- 초기 LLM 붐 당시의 열정과 실험정신을 되살리는 것이 기술 발전의 열쇠라고 제시
- 끝으로, LLM을 바라보는 시각을 ‘기능적’에서 ‘창의적’으로 되돌릴 필요가 있다고 당부함
