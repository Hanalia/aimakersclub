## *네트워크를 샌드박스 삼아본다면? — Remy Guercio, Tailscale* 핵심 요약

- 영상은 "네트워크를 샌드박스처럼 생각하면 어떤 일이 가능할까?"라는 질문에서 출발하여, Tailscale의 네트워크 레이어에서의 인증/권한 관리 방식 및 이를 활용한 AI 게이트웨이(Aperture) 구현 예시를 소개함
- 기존 샌드박스는 VM이나 컨테이너 기반 등 격리 및 권한 부여 방식(API 키, OAuth, OIDC)으로 운영되지만, 이 모두 '샌드박스 내부'에 인증정보가 존재하는 한계가 있음
- Tailscale은 WireGuard 기반 네트워크에서 각 노드(컨테이너, GPU 서버, 노트북, 모바일 등)에 별도의 ID·태그를 부여하여, 네트워크 레벨에서 인증(authN)·권한(authZ)을 효과적으로 처리함
- AI 게이트웨이 ‘Aperture’는 네트워크의 노드로 배치되어, 자신을 거치는 모든 LLM·AI 액세스의 요청자 신원·그룹·태그·행위 기록을 파악하고, API 키 유출·오남용 위험 없이 세분화된 권한∙예산 통제가 가능함
- Aperture를 통해 GitHub Actions 같은 샌드박스 런너도, 노드 태그 기반 권한 하에 LLM과 통신하며, 요청 로그·실행 명령·비용 등을 상세히 추적·감시함
- 네트워크 레이어에서 모든 트래픽(도구 호출, 명령 실행 등)을 관찰·제어하고, ‘request–response’ 형태로 모든 요청·응답 로그 및 툴 호출이 기록됨
- 실시간 데모에서는 Claude, OpenAI, Gemini 등 주요 LLM의 요청 내역, 지출(토큰·달러 단위), PR 리뷰 봇의 모든 명령 실행 내역 추적, 각종 세부 요청·응답 필드 확인이 시연됨
- Aperture는 무료 플랜으로 제공되며, Tailscale의 오픈소스 라이브러리(TS net)를 이용해 유사한 네트워크 레이어 기반 내부 MCP 서버, API 엔드포인트도 직접 구축 가능함
- 권한 정책은 시각적 에디터 및 JSON, 정책 파일(ACL) 방식, API 등으로 그룹별·사용자별·예산별 세밀하게 지정 가능하며, GitOps 워크플로우와도 통합됨
- 네트워크 계층에서 접근 제어와 투명한 로깅을 동시에 달성하여, AI 개발자와 IT/보안 관리자 모두를 만족시키는 디자인임

## 세부 요약 - 주제별 정리

### Tailscale은 네트워크 자체를 샌드박스처럼 활용하는 방식을 제안함

- 발표자는 "샌드박스의 구성요소는 무엇인가?"라는 질문으로 시작하며, 기존에는 VM, 컨테이너, 그 안의 에이전트 배치 및 권한 모델만을 중시했다고 언급함
- 샌드박스의 기본 요소로 '경계(boundary)'와 '권한(permissions/identity)'를 꼽음
- 기존 방식은 인증키(API key 또는 OAuth/OIDC)가 샌드박스 내부에 존재, 내부에서 키 유출이나 남용 위험이 상존함
- 이에 반해 "네트워크가 곧 샌드박스라면 어떨까?"라는 문제의식을 제시
- WireGuard(암호화 네트워크 프로토콜) 기반으로, Tailscale은 네트워크 차원에서 각 노드에 identity(신원)·권한을 부여 및 인식하도록 설계됨

### 기존 샌드박스의 권한 처리 방식은 API키 혹은 OAuth/OIDC 방식에 의존함

- API 키 모델의 경우, 샌드박스 안에서 키를 저장·관리하므로 유출 시 위험이 큼
- 모델 API에 접근할 수 있는 인증 정보를 샌드박스가 모두 보유하는 형태로, 인증(authN)과 권한(authZ)이 근본적으로 분리되지 않음
- OAuth/OIDC 방식도 결과적으로 키·토큰 등 중요한 정보가 샌드박스 내부에 있음
- 장기간 동작하거나 반복 루프를 도는 모델에서 키 악용 가능성 부각

### Tailscale은 네트워크 레벨에서 인증과 권한을 결합해 처리함

- WireGuard 프로토콜 위에 Tailscale의 identity 시스템이 올라감
- 네트워크상의 모든 노드(컨테이너, 서버, 랩탑, 모바일, GPU서버 등)는 고유 사용자, 소속 그룹, 태그 등의 정보를 갖음
- 예시: PR 리뷰봇은 "이 프로젝트 전용 PR 리뷰봇"이라는 태그가 제공됨
- 네트워크 연결(grpc/http 등)이 생성되는 순간, 양 쪽 모두가 신원 및 태그 정보를 획득
- 기존의 IP 주소 기반 연결과 달리, Tailscale 네트워크에서는 신원 기반 통신이 중심

### 네트워크 상에서 신원과 권한 정보를 이용해 실시간 접근 통제가 가능함

- 모든 노드·연결에 대해 신원(사용자, 그룹, 태그) 정보를 실시간으로 교환·확인
- 예를 들어, 어떤 노드가 필요한 권한을 갖지 않으면 연결 자체가 차단됨
- 반대편 노드도 요청자 신원·권한 정보를 모두 이용해 미세한 접근 제어/로깅 가능
- IP+API 키 방식과 달리, 통신 자체가 보안 및 권한 체계 위에 존재

### Aperture는 AI 게이트웨이로써 네트워크의 정체성과 통제를 모두 실현함

- Aperture는 네트워크상에서 하나의 노드로 배포됨
- 주요 LLM 서비스(Anthropic, OpenAI, Gemini, Vertex, Bedrock 등)의 API 키를 Aperture 한 곳에만 저장
- 이 게이트웨이를 통과하는 모든 요청은 요청자 정보, 소속 그룹·태그 등 세부 신원정보를 포함함
- 예시: GitHub Action Runner가 tailnet에 접속하면 federated OIDC를 통해 태그와 신원 정보 획득
- 샌드박스에서 LLM 쪽으로 호출할 때, 내부에 별도 키가 필요 없고, 키 유출·탈취 위험이 원천적으로 차단됨

### 모든 LLM 요청·비용·행동 내역을 세부적으로 추적 및 로깅할 수 있음

- 발표 데모에서 Aperture의 대시보드 화면을 통해, 각 사용자별 LLM 사용량, 모델별 이용 비용(예: Claude 코드에 단순히 "hello" 요청에 20센트 지출), 모든 요청·응답 로그 확인 방법 시연
- 사용자는 자신의 세션 내역, 요청별 사용 토큰·비용, 요청 헤더·바디 전체, 응답 내용까지 모두 실시간으로 확인 가능
- PR 리뷰봇처럼 자동화된 에이전트의 행위(모든 명령 요청, bash/grep 등 실행, MCP/도구 호출 등)도 30일 단위로 세부 기록 가능
- 모든 도구 호출 및 명령 실행 내역은 반드시 게이트웨이(Aperture)를 통과하므로 100% 추적이 가능함

### 네트워크 레이어에서 API 키 자체의 필요성을 제거하거나 무력화함

- 샌드박스 런너에서 LLM API를 호출할 경우, API 키 없이 연결이 설정됨(테스트 시에도 'API 키 모드'에서 실제 키 입력 필요 없음, URL만 요구)
- Claude Code, Codex, Gemini CLI 등에서도 동일하게 적용됨
- 게이트웨이 내부의 단일 API 키만이 사용되고, 각 호출 주체(사용자, 에이전트, 그룹)는 별도 권한 및 태그로 구분·통제됨

### 예산 제어, 비용 통제, 외부 인터그레이션(웹훅 등)에 대한 지원이 내장됨

- 각 그룹/사용자/에이전트별 허용 예산(예: 하루 $5) 설정 및 관리 가능
- 예산 통제는 모든 제공자(Anthropic, OpenAI, Gemini, 내부 GPU 등) 단위가 아닌 게이트웨이 단위로 일괄적으로 작동
- 도구 호출 등 주요 이벤트시 외부 시스템(webhook)으로 실시간 통지·확장 용이
- 모든 통신·행동이 net 단위로 로깅되어, 예산 초과·비인가 동작 신속 식별 및 차단 가능

### Tailscale과 Aperture는 모두 오픈소스 기반으로 자유롭게 확장/구축 가능함

- Aperture는 Tailscale을 기반으로 하며, 무료 플랜과 함께 제공됨
- underpinning(프리미티브)은 오픈소스 라이브러리 TS net으로 구현되어 있어, 각 조직 맞춤 API 엔드포인트, MCP 서버 등도 자체적으로 네트워크 레벨에서 구현 가능
- 접근 제어, 정책, 신원 전달, 로깅 등 모든 기능을 직접 구현·확장할 수 있음

### 권한 정책은 다양한 방식(시각화·JSON·ACL·API 등)으로 세밀하게 관리됨

- 발급 대상(사용자, 그룹, 태그 등), 모델 접근 허용량, MCP 접근 등 granularity 높은 권한 정책 관리 지원
- 정책은 Aperture 내 'grants'에서 시각 에디터로 설정할 수 있으며, 추후 그룹 등 확장 지원 예정
- 조직의 정책 파일(ACL: Access Control List)에 application capability 적용 가능
- JSON 형식 및 API, GitOps 워크플로우 등 다양한 방식으로 권한을 정의·배포 가능

### 네트워크 기반 접근 제어는 AI 개발자와 보안 담당자 모두의 요구를 만족시킴

- 개발자는 LLM 접근 및 통합이 쉽고, 보안·운영 관리자는 툴 호출, 명령 실행, 비용, 행위 기록 등 운영상 모든 내역을 손쉽게 모니터링/제어 가능
- 보안 및 투명성을 보장하면서도, 개발 생산성도 포기하지 않는 아키텍처임
- 네트워크 차원의 접근 제어 및 로깅을 통해 새로운 형태의 샌드박스·AI 거버넌스 모델 실현 가능

### 향후 기능: 더 정교한 툴 호출/코드 실행 통제 및 보안 장치가 추가 예정임

- Bash 등 도구 명령이 실제로 가장 많이 사용됨(내부 통계 확인)
- 위험 명령(예: rm -rf /) 등은 추후 guardrail(안전장치) 추가 계획
- MCP 방식이나 구조화된 툴 콜이 줄어든 환경에서도, 가능한 모든 네트워크 트래픽을 수집·감시하는 데 초점
- 앞으로 bash 등의 비정형 코드 실행도 보다 세밀하게 제어·감시하는 방안 연구 중임
