**채널명:** AI Engineer

## *Evals는 결함이 많지만 그래도 반드시 활용해야 하는 이유* 핵심 요약

- 본 영상은 "Evals Are Broken, Use Them Anyway"라는 제목 아래, AI 및 코드 에이전트 개발 과정에서 평가(evaluation, 이하 evals)의 한계와 올바른 활용법을 집중적으로 논함
- evals에 대한 대표적 오해를 두 가지 진영(객관적 수치 맹신, 느낌·취향 중심)으로 분류하고, 양 극단 모두가 잘못된 접근임을 지적
- 모델 벤치마크 점수에 대한 맹신을 경계하며, 실제 세계의 사용 경험과 상당히 다를 수 있음을 예시와 트윗 인용으로 설명
- 혁신적·세밀한 evals의 필요성이 심화됨에 따라, 구식 벤치마크(예: OpenAI의 bench verified)는 더 이상 첨단 코드 능력을 평가하지 못함을 증명
- 실제 Coding Agent(대표적으로 Client) 개발 과정에서, 기존 evals로는 실사용 문제를 반영하지 못해 자체 평가 방식을 구축하게 된 여정과 절차를 상세히 설명
- Stanford의 Terminal Bench, Loda Institute의 Harbor 등 현대 코드 평가 인프라 및 오픈 소스 도구들의 구체적 활용법을 언급
- 평가 시 모델, 하네스(조립 및 인터페이스), 문제 세트 각각의 특이점과 개선 포인트를 반복적 힐 클라이밍(hill climbing) 방식으로 찾는 절차 제시
- 점수 최적화 과정에서 드러나는 세 가지 개선 구간(명확한 결함 수정, 미묘한 개선, 과적합 위험)을 실제 수치와 함께 구체적으로 설명
- 벤치마크와 실제 사용 경험 두 가지를 모두 검증해야 한다는 철학을 강조하며, evals의 한계를 인정하면서도 이를 적극적으로 활용할 것을 당부

## 세부 요약 - 주제별 정리

### 객관적 수치에만 의존하는 벤치마크 맹신은 실제 효과를 보장하지 못함

- 많은 사람들이 모델 대시보드에 나열된 점수(예: GPT 5.4 vs Gemini 3.1 Pro, Meta의 벤치마크 최고점 등)만을 신뢰함
- 실제로 유사 점수를 받은 모델 간에도 현저한 성능 차이가 있으며, 벤치마크 결과만 보면 실제 적용 시 전혀 다를 수 있음
- 트위터에서 “메타가 벤치마크만 내세운다”는 비판적 트윗을 예로 들어 실사용과의 괴리를 조명함
- AI 연구자/엔지니어들도 evals 점수를 절대적 진실로 보지 않고, 상당수는 실상 이를 우습게 여길 정도임을 Nikunj의 트윗으로 설명
- 벤치마크 점수는 단순화된 근사치에 불과하며, 맹목적으로 믿는 태도를 경계

### 취향·느낌 중심의 평가 역시 신뢰할 수 있는 기준이 되지 못함

- evals에 대한 또 다른 오해 진영은 “취향 중심” 혹은 "vibes" 진영임
- 이들은 모델의 특성을 인간화하거나, 단순히 대화 경험만을 통해 효용을 판단하는 경향이 있음
- “왜 이걸 좋아하냐” 물으면 상세 근거 없이 “느낌이 좋다”, “대화가 재밌다”고 답하는 방식이 지적됨
- 객관적 수치 맹신, 취향 중심 모두가 불완전하며, 진실은 두 극 사이에 존재함을 분명히 밝힘

### 올바른 evals 활용을 위한 3단계 접근법(레벨별 적용법)이 필요함

- 1단계: 타인이 만든 evals를 적극적으로 참고하여 모델 선택·초기 판단에 활용
- 2단계: evals를 통해 자신이 개발하는 에이전트 및 모델 에셋을 지속적으로 개선
- 3단계: 구체적 실제 사용 사례별 맞춤 evals를 직접 구축함(본 강연에서는 1, 2단계에 집중 설명)
- 단계별로 다른 목적과 구현 난이도를 갖는다는 점을 명확히 강조

### 최신 트렌드를 따라가되, 급진적 ‘얼리어답터’가 될 필요는 없음

- AI 분야 벤치마크 순위와 최신 모델은 한두 달마다 급변하는 경향이 있음(Epochs Index의 변화 사례 제시)
- 변화 속도가 너무 빨라 기존 프론트이어 모델이 금방 뒤처지는 현상이 자주 발생
- 전문가(연사 본인) 수준이 아닌 이상, 바로바로 새 모델을 쫓기보다는 시장 반응이 어느 정도 자리잡은 뒤 검증된 모델을 채택하는 것을 권고

### 구식 또는 표준화된 evals는 현실 문제를 반영하지 못하며, 맞춤형(최신·정확) evals가 요구됨

- OpenAI의 공식 블로그에서도 과거 벤치마크 문제(예: Fibonacci 수열, 행렬 곱셈 등)가 실제 SW 엔지니어링과 무관하다는 점을 인정함
- 코딩 에이전트 개발자 입장에서는 “기존 evals로는 실무 문제를 전혀 판단할 수 없다”는 입장을 밝힘
- 코드 에이전트에 특화한 새로운 벤치마크가 필요하며, ‘최신성’, ‘정확성’, 실제 업무 적합성이 필수적

### 실제 코드 에이전트(클라인) 개발 과정에서 맞춤형 evals 구축의 필요성과 시행착오가 드러남

- Client(클라인) 팀은 1년 전, 기존 evals의 무용성에 좌절하고 “완전히 무시해도 된다”는 입장이었음(동시기 Codex 팀 등 여러 팀 공통)
- 하지만 곧 자체적으로 실제 사용자 코딩 이슈를 반영하는 맞춤형 evals 제작 필요성을 절감
- 수많은 opt-in 사용자 데이터를 현금 보상과 함께 수집, 만여건의 실제 코딩 작업 문제를 정제·정형화
- 문제 세트 구축 과정에서 많은 수작업, 데이터 클리닝, 적합성 검증 등 힘든 과정을 거침

### 하나의 프롬프트나 단답형 질문 기반 evals는 한계가 분명하며, 복잡한 에이전트형 문제에 특화된 평가가 필요함

- 기존 언어모델 단일 질문 평가(예: "고양이 발가락 개수는?")는 이진 답변이 가능해 간단
- 실제 에이전트는 환경 셋업, 파일 탐색, 문서 검색, 환경 설치, 코드 실행 등 복합적 작업을 수행함
- 예시: “MCP 서버 문제를 해결해달라” 요청 시, 에이전트가 환경 전체를 조작하고 복수 단계를 거쳐야 하므로, 평가가 매우 복합화됨
- “어느 정도 작동은 하나, 다른 부분이 망가진다” 등 디테일 이슈까지 세밀하게 검증해야 함

### 최신 오픈 소스 평가 인프라(Terminal Bench, Harbor 등)의 도입과 실행 절차

- Stanford의 Terminal Bench: 실제 프로그래밍 업무를 대변하는 89개 문제로 구성된 코드 평가 패키지
  - 예시: Race condition 디버깅, 데이터베이스 이슈, 인프라 문제 등 다양한 업무 반영
  - CLI(명령줄) 기반으로 코드 에이전트(예: Client, Cloud Code, Codex 등) 직접 연결하여 빠른 평과 가능(최대 30~40분 소요)
- Harbor(Loda Institute): 89개 문제 각각에 대해 독립적 VM 환경 복제 및 병렬 평가 지원
  - 표준화된 인프라 설정(Linux, RAM/CPU 할당 등)으로 환경 자동 초기화
  - Daytona, Docker 등 다양한 머신 활용, Client 팀은 Model사의 인프라 지원 활용
  - 병렬 실행 시 전체 속도는 가장 오래 걸리는 태스크가 제한하게 됨

### 실제 평가 및 시간적 흐름에 따라 개선 포인트(3가지 영역)가 구체적으로 드러남

- 89개 문제 세트 평가 결과를 통해, 최초 점수(예시: 43%)에서 반복적인 failure 분석 및 개선 작업 진행
- Trace 분석: 각 실패 과제에 대해 전체 LM 호출 로그 분석, 실패 이유(예: 툴 에러, 코드 실행 실패 등) 분류 및 원인 타겟팅
- 개선 유형 3가지
  - Zone 1: 명확한 버그, 환경결함 등 '자명한 결함' 교정
  - Zone 2: 모델별 특화 프롬프트 엔지니어링, 미묘한 개선(예: Anthropic 모델은 prompt를 더 길게/짧게 등) – 실질적으로 성능을 가장 크게 개선
  - Zone 3: 과적합(overfitting) 또는 평가 데이터셋 치팅(트윗 등 대외 과시용) – 위험 경계 필요

### 평가 시 세 가지 대상을 구분해 각각 점진적 개선 및 최적화가 요구됨

- (1) 모델 자체의 성능: 일부는 모델이 압도적이라 하네스/에이전트 결함에도 고득점 가능
- (2) 하네스(인터페이스)의 최적화: 동일 모델도 Cloud Code, Cursor, Droid 등 하네스별로 효율 차이 발생
- (3) 문제 세트 구성: 단순한 문제로만 점수 높여봤자 실제 사용자 효용과 일치하지 않으므로, 실제적이고 난이도 있는 문제 필수

### 벤치마크 점수와 사용자 경험(‘vibe check’) 모두를 엄격하게 점검해야 함

- 점수만 높다고 해도, 사용자 경험이 불편하거나, 실제 업무에 도움이 안 되면 무의미함
- 시장에 새 모델·도구 등장 시, 단순 수치가 아니라 실질적 사용성과 벤치마크 성적을 모두 비교해야 함
- Client팀은 특정 모델(Antropic 계열)에선 성능이 우수했던 반면, Gemini 및 Kimini 등에선 취약함을 직접 실험과정에서 발견
- 다양한 모델 지원 시 사용자층 확대 가능하다는 교훈도 실험으로 습득

### 결론적으로 evals의 불완전성에도 불구하고, 맞춤형 평가와 지속적 개선은 필수임을 강조함

- “완벽한 벤치마크는 없다”는 점을 인정하고, 각자 실제 문제에 맞는 평가 환경 및 반복적 개선 프로세스(힐 클라이밍)를 반드시 거칠 것 주문
- 실제 모델 비교·최적화에 관심이 있다면 Twitter DM 등으로 실무 상담 및 채용 연계도 안내
- 강연 내내 evals에 대한 예리한 비판적 관찰, 실전 경험, 구체 도구와 인프라 소개, 반복 개선 노하우를 현장감 있게 전달함
