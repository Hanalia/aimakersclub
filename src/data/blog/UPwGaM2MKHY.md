## *로그가 곧 에이전트다 - 이샨 세갈, 옴나라* 핵심 요약

- 이샨 세갈(Omnara CEO)은 "에이전트의 본질은 모델이나 실행 환경이 아니라, 바로 로그(기록 데이터)"임을 주장함
- 스카이림 게임의 세이브 파일처럼, 에이전트의 정체성과 상태는 로그 데이터에 저장되어 있으며, 실행 환경이 달라져도 로그만 있으면 동일하게 복원 가능함
- 로그는 변경 불가(append-only) 이벤트 이력이며, 사용자 입력, 모델 출력, 도구 호출, 권한 요청 등 모든 상태 전이가 기록됨
- 에이전트의 운영은 로그에서 상태를 재구성하고, 모델이 다음 행동을 제안, 결과를 다시 로그에 추가하는 반복 루프로 설명됨
- 로그를 에이전트의 중심으로 두면 신뢰성, 확장성, 이식성, 분기(포크), 멀티플레이어 환경 등 다양한 시스템 속성이 자연스럽게 확보됨
- 로그 압축(compaction)은 필요하지만 손실이 수반되므로, 원본 로그 보존이 에이전트의 온전한 유지에 필수적임
- 로그에 기록되지 않은 외부 상태(예: 파일 변경, 이메일 전송)는 복원 불가하나, 목적은 에이전트가 인식하고 관리한 내역만 정확히 기록하는 데 있음
- 현 에이전트 인프라는 로그를 부수적 산출물로 간주하여 신뢰성과 관리에 문제 노출(예: 데이터 손실, 일관성 깨짐)
- 로그 소유권이 실질적인 락인(lock-in)의 핵심으로, 공급자가 로그를 소유하면 에이전트 전체를 지배하게 됨
- 옴나라가 개발 중인 매니지드 오픈소스 에이전트 플랫폼은 사용자가 로그를 완전히 통제·소유하도록 설계, 미래 에이전트 생태계의 핵심 가치로 지목
- 결론적으로 “에이전트의 본질은 내구성 있는 작업 이력(로그)”이라는 프레임이 신뢰성, 확장성, 분기, 소유권 등 다양한 강점의 토대가 됨

## 세부 요약 - 주제별 정리

### 에이전트의 정체성은 실행 환경이나 모델이 아니라 로그 데이터에 달려있음

- 대부분의 사람들은 에이전트를 모델 자체나 실행 환경(런타임, 툴 등)으로 오해함
- 실제로 에이전트의 정체성은 모델이나 실행 환경이 아니라, 에이전트의 로그, 데이터에 내재함
- 스카이림(Skyrim) 비디오게임 예시: 캐릭터는 플레이스테이션, 게임 엔진, 컨트롤러로 정의되지 않고, 세이브 파일에 모든 정체성이 담김
- 세이브 파일이 남아 있다면, 기기가 바뀌고 실행 환경이 달라져도 언제든 동일한 상태로 돌아갈 수 있음
- 이와 마찬가지로, 에이전트의 모든 상태, 이력, 정체성은 데이터(로그)에 저장됨

### 로그의 역할과 특성: 변경 불가 내역으로 에이전트의 상태와 역사를 모두 담음

- 로그는 에이전트의 모든 이벤트를 순차적으로 기록하는 변경 불가(append-only) 이력임
- 기록 내용: 사용자 입력, 모델 출력, 호출한 도구 및 결과, 권한 실폐 등 모든 상태 전이
- 에이전트의 상태 복구, 이력 추적, 변화 관리가 로그만으로 가능
- 모델/툴/실행 환경 등은 로그를 해석하고 해석한 결과(새 이벤트)를 다시 로그에 추가하는 '거푸집' 역할에 불과
- 로그만 보존된다면 실행 환경이나 모델이 교체되어도 에이전트 연속성이 보장됨

### 에이전트의 동작 루프는 로그를 중심으로 설계되어 간단하며 유연함

- 에이전트 운영의 기본 루프: 로그에서 현재 상태 복원 → 모델에 상태 전달 → 모델이 다음 행동/출력 제안 → 로그에 결과 추가 반복
- 툴 호출이 필요하면, 툴 실행 결과를 로그에 추가하고 다시 루프 반복 가능
- 이 루프는 본질적으로 '일회용'(disposable)임. 즉, 한 워커(작업자)가 세션을 한 스텝 진행 후 사라지고, 다른 워커가 이어서 진행해도 무방
- 동일한 로그를 활용해 여러 워커, 다양한 환경에서 유연하게 실행 가능

### 데이터베이스 역사를 빗대어 로그 기반 설계의 중요성을 강조함

- 데이터베이스 분야에서도 초기에는 테이블·인덱스 등 불투명(opaque) 구조로 운영되었으나, 핵심은 로그였음
- 진짜 데이터베이스의 본질은 변경 내역의 내구적 순서(로그)이고, 나머지(뷰, 인덱스 등)는 투영(projection, view)임
- 에이전트 시스템도 데이터베이스처럼 ‘로그 역전(inversion)’이 필요하며, 로그가 주체(entity)여야 함

### 모든 시스템 구성요소(모델 입력, UI, 디버깅, 감사 등)는 로그에서 파생되는 투영임

- 모델에 주어지는 컨텍스트도 로그 일부를 투영한 것임
- UI에서 렌더링되는 정보, 디버깅·트레이싱 도구, 사용 내역 감시·감사, 로그 요약(압축), 모두 원본 로그를 기본으로 가공/요약한 뷰임
- 단, 압축(compaction)은 정보를 일부 손실함(손실 요약임)
- 원본 로그를 보존하는 한 다양한 투영/요약/분기가 언제든 생성 가능

### 로그의 크기와 압축(compaction) 이슈 및 그 한계점은 존재함

- 로그는 무한정 커질 수 있으나, 모델의 컨텍스트 윈도우는 유한함
- 따라서 장기적으로는 로그를 더 작은 형태로 요약(압축)해야 함
- 로그 압축은 완벽한 복원이 불가한 '손실' 처리임
- 원본 로그를 유지하지 않고 압축 요약만 남을 경우, 에이전트의 온전한 기록 일부가 영구 소실됨
- 따라서 압축은 "새 로그의 시작점이 되는 독립된 손실 복사본"으로 이해하고, 원본 로그는 별도로 지속 보존하는 게 원칙임

### 로그에 담기지 않는 외부 상태(파일, 이메일 등)는 복원 불가능함을 분명히 밝힘

- 툴 사용이 외부 상태(파일 편집, 이메일 발송 등)를 변화시키면, 이는 로그 내 복원 범위 밖임
- 로그는 에이전트의 ‘세계관’만 관리하는 것이며, 외부 전체 환경을 포함하는 건 아님
- 게임 세이브파일이 게임 모든 자산/엔진 코드를 담지 않고, 오직 플레이어 상태만 기록하는 것과 유사
- “이메일을 보냈다고 해서 과거로 돌아가도 이메일 발송이 취소되진 않음”
- 로그의 역할은 “에이전트가 무엇을 했고, 무엇을 봤으며, 어떻게 이어가야 하는지”만 기록하는 것임

### 로그를 시스템의 근간으로 삼으면 신뢰성, 확장성, 이식성 등 시스템적 효과를 얻을 수 있음

- **신뢰성(Reliability)**: 예) 클라우드 코드에서 권한 승인 대기 중 프로세스가 죽을 경우, 로그 중심이라면 동일 위치에서 아무 워커나 이어갈 수 있음
- **확장성(Scalability)**: 기존에는 1 에이전트당 1 프로세스가 필요하나, 로그 중심이라면 수천 개의 에이전트를 하나의 실행기로 병렬 진척 가능
- sticky session이나 상태 이동/관리의 복잡성이 제거되어 failover, 분산처리, 스케일링이 용이
- **분기(Forking)**: 로그의 특정 시점에서 여러 분기 가능, 분기별로 다른 모델(Claude, GPT, 오픈소스 등) 및 전략 적용 가능
- **멀티플레이어(Multiplayer)**: 에이전트 로그의 공유는 복사/전달이 아니라 권한 부여만으로 실현
- 협업자·매니저·다른 에이전트가 동일 세션 로그에 접근하고, 관찰·참여·수정 가능함
- **마이그레이션(Migration)**: 로그가 에이전트의 온전한 일관성을 보장하므로, 공급자/실행 환경 교체가 단순 엔지니어링 문제로 환원됨

### 현존 에이전트 시스템들은 로그를 부수적인 산출물로 취급해 운영상 한계에 부딪치고 있음

- 현재 클로드 코드, 코덱스 등 주요 에이전트 인프라는 로그를 로컬 디스크의 임시 JSONL, 비신뢰성 모드로 처리
- 기록 실패시 데이터 영구 소실 위험이 상존
- OpenCode 등은 SQLite 파일로 상태 관리, 상태 손상에 대한 깃허브 이슈 다수
- Durable object 등 분산 시스템은 로그가 여러 shard에 분산되어 세션 이력 재구성 및 쿼리가 복잡해짐
- 전반적으로 “로그가 핵심이 아닌 시스템 설계”가 데이터 일관성, 신뢰성, 운영 편의성에 악영향

### 로그 소유권이 에이전트 락인(lock-in)의 본질임을 강조함

- 모델, 툴, API는 교체 및 래핑 가능하므로 깊은 락인은 아님
- 로그를 공급자가 소유하면, 공급자는 에이전트 전체를 장기적으로 통제/소유
- Anthropic(Claude), Google(Gemini) 등 관리형 에이전트 서비스가 로그를 자체 인프라에 저장함으로써 데이터, 워크플로우, 의사결정까지 장악
- 에이전트의 가치는 산출물이 아니라 “활동의 전체 이력(로그)” 그 자체에 있음
- 남의 정책과 시스템에서만 접근 가능한 로그는 결국 사용자의 에이전트 소유권을 침해

### 옴나라(Omnara)가 지향하는 오픈소스 에이전트 플랫폼의 아키텍처와 핵심 철학

- 옴나라가 개발 중인 플랫폼에서는 세션 로그를 시스템의 핵심 데이터로 간주
- 워커(worker)는 루프를 진척할 뿐, 결코 에이전트 자체가 아니며, 각 실행 결과를 로그에만 지속 반영
- 툴이나 모델 호출 등도 로그를 통해 상태 이력에 투명하게 기록
- 워커나 실행 환경의 실패는 무해. 새로운 워커가 동일 상태로 세션 이어 진행 가능
- “사용자가 로그를 완전히 소유·통제·검증 가능한 플랫폼”을 추구
- 이 아키텍처는 오픈소스 관리형 에이전트 생태계의 핵심 기조로 자리매김할 것을 목표로 함

### 마지막 강조: 로그 자체가 에이전트이며, 이를 받아들일 때 다양한 혁신적 시스템 속성이 자연스럽게 구현됨

- “에이전트란 모델 호출, 프롬프트, 루프, 샌드박스 그 자체가 아니라, 내구적 작업 이력, 즉 로그”임을 다시 강조
- 로그를 “시스템의 배출물”이 아니라 “시스템 그 자체”로 대우할 때 신뢰성, 분기, 이식성, 협업성, 소유권, 확장성의 근본적 진보가 가능
- 옴나라가 오픈소스 관리형 에이전트 플랫폼을 곧 공개할 계획임을 덧붙이며, 로그 중심 패러다임이 미래 에이전트의 핵심이 될 것임을 시사
